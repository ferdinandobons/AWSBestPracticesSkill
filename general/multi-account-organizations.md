# Multi-Account & AWS Organizations — Best Practices

## Common scenarios
- Establishing a new landing zone for an enterprise starting its AWS journey        → Account structure, Landing zone automation
- Isolating production from non-production workloads and limiting blast radius        → Account structure, Guardrails
- Centralizing security telemetry, audit trails, and compliance evidence        → Centralized logging & security
- Delegating day-to-day account management away from a single powerful account        → Delegated administration
- Allocating and charging back cloud spend across departments or products        → Billing & cost governance
- Sharing common infrastructure (networking, licenses, images) without duplication        → Resource sharing

## Account structure & organizational units
- **[OU design]** Base organizational units on function or common control requirements (e.g. Security, Infrastructure) — mirroring your company's reporting structure instead makes policies harder to apply consistently. [doc](https://docs.aws.amazon.com/organizations/latest/userguide/orgs_manage_ous_best_practices.html)
- **[OU design]** Create nested OUs to separate production from non-production (SDLC) workloads — this lets policies be applied at the OU level according to lifecycle stage. [doc](https://docs.aws.amazon.com/organizations/latest/userguide/orgs_manage_ous_best_practices.html)
- **[OU design]** Apply policies at the OU level rather than at individual accounts — it simplifies policy management and troubleshooting as the organization scales. [doc](https://docs.aws.amazon.com/organizations/latest/userguide/orgs_manage_ous_best_practices.html)
- **[OU design]** Keep the Security OU limited to the essential security accounts (Log Archive, Audit/security tooling) designated by AWS Control Tower — place additional security-related accounts in a separate OU to keep the core OU clean. [doc](https://docs.aws.amazon.com/whitepapers/latest/organizing-your-aws-environment/organizing-your-aws-environment.html)
- **[OU design]** Factor in SCP quotas per OU and resource-sharing needs (e.g. transit gateways, Route 53 Resolver rules via AWS RAM) when deciding how to split accounts across OUs — this avoids hitting policy limits or awkward re-splits later. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/designing-control-tower-landing-zone/account-structure.html)
- **[Account boundary]** Use the AWS account as the primary isolation boundary for security, billing, and resiliency — separate workloads using accounts rather than relying solely on in-account controls. [doc](https://docs.aws.amazon.com/wellarchitected/latest/management-and-governance-guide/manage-and-govern-with-a-multi-account-point-of-view.html)
- **[Scale]** Avoid creating more accounts than you can operationally manage — periodically review service quotas and deployment latencies when performing actions across many accounts. [doc](https://docs.aws.amazon.com/wellarchitected/latest/management-and-governance-guide/manage-and-govern-with-a-multi-account-point-of-view.html)
- **[Strategy]** Define a multi-account strategy up front that accounts for scale and operational efficiency — start from a foundational set of accounts that support both centralized and decentralized capabilities. [doc](https://docs.aws.amazon.com/wellarchitected/latest/management-and-governance-guide/implementation-priorities.html)
- **[Payer account]** Use a single organization for most use cases — each central billing (payer) account can only belong to one organization, so plan carefully because this choice is hard to change later. [doc](https://aws.amazon.com/blogs/publicsector/migrating-to-a-multi-account-strategy-for-public-sector-customers/)

## Landing zone automation (AWS Control Tower)
- **[Setup]** Adopt AWS Control Tower to automate landing zone setup — it provides best-practice integrations for identity, federated access, central logging, backup, and account structure instead of requiring you to build these manually. [doc](https://aws.amazon.com/controltower/features/)
- **[Region strategy]** Create the landing zone in the AWS Region hosting the majority of workloads — changing the home Region later requires AWS Support assistance and landing zone decommissioning. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/transitioning-to-multiple-aws-accounts/create-landing-zone.html)
- **[Region strategy]** Govern only the Regions where you expect to immediately deploy workloads, then deny access to all ungoverned Regions via an SCP — this keeps workloads within approved Regions. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/transitioning-to-multiple-aws-accounts/create-landing-zone.html)
- **[Naming]** Rename the default Control Tower OUs and shared accounts to reflect their purpose (e.g. Security → Security_Prod, Log Archive → log-archive-prod, Audit → security-tooling-prod) at setup time — some renames aren't possible later. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/transitioning-to-multiple-aws-accounts/create-landing-zone.html)
- **[OU segmentation]** Create nested OUs under the landing zone (e.g. renaming Sandbox to Workloads and adding further OUs beneath it) — this segments production from non-production and infrastructure from application workloads. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/transitioning-to-multiple-aws-accounts/add-organizational-units.html)
- **[Account vending]** Use AWS Control Tower's account factory and IAM Identity Center integration to provision new accounts — this ensures consistency versus manual account creation. [doc](https://docs.aws.amazon.com/controltower/latest/userguide/how-control-tower-works.html)

## Guardrails & service control policies (SCPs)
- **[SCP scope]** Remember that SCPs affect only member accounts — they have no effect on users or roles in the management account, so don't rely on SCPs to restrict management-account activity. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/security-reference-architecture/org-management.html)
- **[SCP strategy]** Combine `Allow` and `Deny` statements deliberately, using `Deny` for restrictions meant to apply broadly — when applied at the root or OU level they affect every account beneath that node. [doc](https://docs.aws.amazon.com/organizations/latest/userguide/orgs_manage_policies_scps_evaluation.html)
- **[SCP strategy]** Use IAM service-last-accessed data to refine SCPs — this restricts access to only the AWS services actually in use, rather than leaving the default `FullAWSAccess` policy unchanged. [doc](https://docs.aws.amazon.com/organizations/latest/userguide/orgs_manage_policies_scps_evaluation.html)
- **[SCP strategy]** Test new or modified SCPs in a separate organization or OU that represents production before deploying broadly — roll out changes gradually from narrow to broad OUs to limit blast radius. [doc](https://docs.aws.amazon.com/organizations/latest/userguide/orgs_manage_policies_scps_examples.html)
- **[SCP strategy]** Remember that SCPs are guardrails, not grants — the effective permission is the intersection of the SCP and the identity/resource-based policy, so IAM permissions must still be granted separately. [doc](https://docs.aws.amazon.com/organizations/latest/userguide/orgs_manage_policies_scps_examples.html)
- **[Root actions]** Attach an SCP that disallows root-user actions across member accounts, applied at the OU or account level as appropriate — this reduces the risk of highly privileged root misuse. [doc](https://aws.amazon.com/blogs/mt/protect-your-aws-resources-from-unauthorized-access-using-aws-organizations-integrations/)
- **[Condition keys]** Use global condition keys such as `aws:PrincipalOrgID`, `aws:PrincipalOrgPaths`, `aws:ResourceOrgId`, and `aws:ResourceOrgPaths` in IAM policies and SCPs — these constrain access based on organization or OU membership, which is especially useful in complex OU structures. [doc](https://aws.amazon.com/blogs/mt/protect-your-aws-resources-from-unauthorized-access-using-aws-organizations-integrations/)
- **[Tag policies]** Use AWS Organizations tag policies to define and enforce a consistent tagging schema across accounts — this supports cost allocation, ABAC, and backup-policy scoping. [doc](https://docs.aws.amazon.com/whitepapers/latest/tagging-best-practices/aws-organizations-tag-policies.html)
- **[Tag policies]** Avoid placing personally identifiable or other sensitive data in tag keys/values — tag data is unencrypted and readable by services like Billing and Cost Management. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/privacy-reference-architecture/org-management-account.html)

## Management account & delegated administration
- **[Management account]** Limit the management account to functions that must be performed there, such as consolidated billing and central configuration of Organizations-wide services — avoid deploying workloads or storing routine resources in it. [doc](https://docs.aws.amazon.com/organizations/latest/userguide/orgs_best-practices_mgmt-acct.html)
- **[Management account]** Limit and regularly review who has access to the management account — privileged operations performed there are not constrained by SCPs. [doc](https://docs.aws.amazon.com/organizations/latest/userguide/orgs_best-practices_mgmt-acct.html)
- **[Delegated admin]** Register member accounts as delegated administrators for AWS services that support it (e.g. CloudTrail, Service Catalog, CloudFormation StackSets) — this reduces reliance on the management account for day-to-day operations. [doc](https://docs.aws.amazon.com/organizations/latest/userguide/orgs_integrate_delegated_admin.html)
- **[Delegated admin]** Store organizational AWS resources in member accounts, keeping them out of the management account — this also simplifies understanding charges on invoices. [doc](https://docs.aws.amazon.com/organizations/latest/userguide/orgs_delegate_policies.html)
- **[Decentralization]** Delegate responsibilities outside the management account wherever possible — this lets teams manage organizational needs from their own accounts instead of requiring management-account access. [doc](https://docs.aws.amazon.com/organizations/latest/userguide/orgs_best-practices_mgmt-acct.html)

## Centralized identity & cross-account access
- **[Identity]** Use AWS IAM Identity Center to centrally manage workforce user access across all accounts in the organization via permission sets — this avoids having to create and reconcile IAM users in each account. [doc](https://docs.aws.amazon.com/singlesignon/latest/userguide/manage-your-accounts.html)
- **[Identity]** Prefer temporary credentials over long-term IAM user credentials — use IAM Identity Center for workforce access to multiple accounts and IAM roles for cross-account or service access. [doc](https://docs.aws.amazon.com/IAM/latest/UserGuide/introduction_identity-management.html)
- **[Identity]** Monitor IAM role quotas (default 1000 roles per account) when using IAM Identity Center at scale — each permission set assignment provisions a corresponding IAM role per account, which can approach the quota. [doc](https://docs.aws.amazon.com/singlesignon/latest/userguide/identity-center-and-iam-roles.html)
- **[Cross-account roles]** Use IAM roles (not shared long-term credentials) to delegate cross-account resource access, adding an `ExternalId` condition when granting access to third parties — this closes the confused-deputy gap. [doc](https://docs.aws.amazon.com/IAM/latest/UserGuide/tutorial_cross-account-with-roles.html)
- **[Cross-account roles]** Remember that IAM roles and resource-based policies only delegate access within a single AWS partition — cross-partition access (e.g. standard `aws` to `aws-cn`) requires a different mechanism. [doc](https://docs.aws.amazon.com/IAM/latest/UserGuide/tutorial_cross-account-with-roles.html)

## Root user & account security
- **[Root user]** Remove root user credentials (password, access keys, MFA) from member accounts using root access management — member accounts then default to secure-by-default with no root sign-in possible. [doc](https://docs.aws.amazon.com/organizations/latest/userguide/orgs_best-practices.html)
- **[Root user]** Prioritize securing the management account's root user first, then member account root users — the management account root has the most privileged access in the organization. [doc](https://docs.aws.amazon.com/wellarchitected/latest/security-pillar/sec_securely_operate_aws_account.html)
- **[Root user]** Enable multi-factor authentication and use a strong, unique password for every root user, and create alerts (via EventBridge, CloudWatch, GuardDuty) for root sign-in — this catches unauthorized or unexpected root use quickly. [doc](https://aws.amazon.com/blogs/mt/protect-your-aws-resources-from-unauthorized-access-using-aws-organizations-integrations/)
- **[Root user]** Use privileged root sessions only for tasks that genuinely require root (e.g. fixing a misconfigured S3 bucket policy that denies all principals) — routine administration should never need root. [doc](https://docs.aws.amazon.com/organizations/latest/userguide/orgs_best-practices.html)
- **[Everyday access]** Create an administrative IAM Identity Center user (or IAM role) for everyday tasks instead of using the root user, in both single-account and multi-account environments — this limits exposure of the highest-privilege credential. [doc](https://docs.aws.amazon.com/IAM/latest/UserGuide/root-user-best-practices.html)

## Centralized logging & auditing
- **[CloudTrail]** Create an organization trail from the management account (or a delegated administrator) — this logs all member accounts' events automatically and consistently, without member accounts able to modify or delete it. [doc](https://docs.aws.amazon.com/organizations/latest/userguide/services-that-can-integrate-cloudtrail.html)
- **[CloudTrail]** Aggregate organization trail logs into a single S3 bucket in a dedicated Log Archive account — separate trail management (Security Tooling account) from log storage privileges for least-privilege access. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/security-reference-architecture/security-tooling.html)
- **[CloudTrail]** Add the `aws:SourceArn` condition key for the organization trail to the resource policy of the S3 bucket (and related KMS keys/SNS topics) — this ensures the bucket accepts data only from that specific trail. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/security-reference-architecture/security-tooling.html)
- **[Access control]** Build granular, dynamic IAM policies that give each member account least-privilege, read-only access to its own slice of the centralized CloudTrail log bucket — avoid broad bucket-wide access. [doc](https://aws.amazon.com/blogs/mt/restrict-access-by-member-account-to-a-centralized-cloudtrail-logging-bucket/)
- **[Regional considerations]** For workloads with data residency requirements, evaluate whether cross-Region log aggregation into a single organization-trail bucket is appropriate — use Region-specific trails with data masking where it is not. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/privacy-reference-architecture/security-tooling-account.html)

## Resource sharing
- **[AWS RAM]** Use AWS Resource Access Manager to share resources (transit gateways, subnets, license configurations, Route 53 Resolver rules, etc.) across accounts or OUs — this avoids duplicating them in every account. [doc](https://docs.aws.amazon.com/whitepapers/latest/aws-overview/security-services.html)
- **[AWS RAM]** Enable AWS RAM integration with AWS Organizations — resources can then be shared within the organization without a manual invitation/acceptance handshake per account. [doc](https://aws.amazon.com/blogs/mt/protect-your-aws-resources-from-unauthorized-access-using-aws-organizations-integrations/)
- **[AWS RAM]** Author fine-grained customer-managed permissions in AWS RAM rather than relying only on AWS-managed permissions — this applies least privilege to shared resources. [doc](https://aws.amazon.com/blogs/security/configure-fine-grained-access-to-your-resources-shared-using-aws-resource-access-manager/)
- **[AWS RAM]** Share resources with an entire OU rather than maintaining manually updated lists of individual account IDs — this reduces operational overhead as accounts are added or removed. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/privacy-reference-architecture/personal-data-account.html)

## Billing & cost governance
- **[Consolidated billing]** Use AWS Organizations consolidated billing to combine usage across member accounts for volume pricing, Reserved Instance, and Savings Plans discounts — you still receive per-account cost detail for reporting. [doc](https://docs.aws.amazon.com/cost-management/latest/userguide/what-is-costmanagement.html)
- **[Cost accountability]** Align account structure with business ownership (department, product line, or team) — account-level cost boundaries then map directly to chargeback and Cost Explorer reporting. [doc](https://aws.amazon.com/blogs/aws-cloud-financial-management/cost-allocation-patterns-in-aws/)
- **[Access control]** Restrict which member accounts can view consolidated billing and Cost Explorer data using IAM — this data can reveal organization-wide volume-discount details. [doc](https://aws.amazon.com/blogs/apn/back-to-basics-guide-for-consolidating-aws-accounts-to-manage-costs-at-scale/)
- **[Reporting]** Use the Cost and Usage Report together with Cost Explorer and Budgets from the management account — this lets you analyze cross-account cost and usage patterns and set proactive budget alerts. [doc](https://aws.amazon.com/blogs/apn/back-to-basics-guide-for-consolidating-aws-accounts-to-manage-costs-at-scale/)
