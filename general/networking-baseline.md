# Networking Baseline — Best Practices

## Common scenarios
- New landing zone or multi-account foundation → VPC & Subnet Design, Multi-VPC Connectivity, IP Address Management
- Securing workload traffic in and out of a VPC → Security & Traffic Control
- Connecting many VPCs and on-premises networks → Multi-VPC Connectivity
- Reducing NAT/internet egress spend and cross-AZ charges → Cost & Data Transfer Optimization
- Making internet-facing endpoints resilient across zones → High Availability & Load Balancing
- Resolving names across hybrid or multi-VPC environments → DNS

## VPC & Subnet Design
- **[Traffic locality]** Keep high-throughput, low-latency traffic flows inside the same VPC — intra-VPC paths are local and avoid the added hops and cost of crossing VPC boundaries. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/designing-hyperscale-amazon-vpc-networks/)
- **[Subnet layout]** Design subnets around routing, Availability Zone distribution, and NACL boundaries rather than treating a VPC like a legacy data-center network — VPCs are software-defined networks, not physical switch fabrics, so traditional subnet-sizing rules of thumb don't apply. [doc](https://aws.amazon.com/blogs/startups/practical-vpc-design/)
- **[Multi-AZ]** Create subnets for your application in multiple Availability Zones — this makes production workloads highly available, fault tolerant, and scalable. [doc](https://docs.aws.amazon.com/vpc/latest/userguide/vpc-security-best-practices.html)
- **[Route tables]** Configure subnet route tables with only the minimum routes needed for connectivity requirements, and ensure resources in each subnet cannot reach subnets or resources they shouldn't. [doc](https://aws.amazon.com/solutions/guidance/network-connectivity-on-aws/)
- **[IP planning]** Size VPC and subnet CIDR blocks to accommodate current workload needs plus future expansion — the largest possible VPC CIDR is /16 and the smallest is /28, and subnet IPv4 CIDRs cannot be resized after creation. [doc](https://docs.aws.amazon.com/wellarchitected/latest/framework/rel_planning_network_topology_ip_subnet_allocation.html)
- **[IP planning]** Reserve unused CIDR space within each VPC for future subnets, and account for IP consumption by load balancers, EC2 Auto Scaling, and per-pod IP allocation in EKS clusters. [doc](https://docs.aws.amazon.com/wellarchitected/latest/framework/rel_planning_network_topology_ip_subnet_allocation.html)
- **[Overlap avoidance]** Never allocate overlapping CIDR blocks across VPCs or on-premises networks that you plan to route between — traffic destined to an overlapping address space is silently dropped. [doc](https://aws.amazon.com/blogs/startups/what-startups-should-know-about-amazon-vpc-part-2/)
- **[IPAM]** Delegate centralized IP address management to a Network account using Amazon VPC IP Address Manager (IPAM) to maintain a unified view of allocations and automate non-overlapping CIDR assignment across the organization. [doc](https://aws.amazon.com/solutions/guidance/network-connectivity-on-aws/)
- **[IPAM]** Configure CloudWatch alerts on IPAM to proactively detect and resolve conflicting IP address issues across the network. [doc](https://aws.amazon.com/solutions/guidance/network-connectivity-on-aws/)

## Security & Traffic Control
- **[Defense in depth]** Use security groups as the primary mechanism for controlling network access, and layer network ACLs as a secondary, coarse-grained control — security groups are stateful and can reference other security groups, while NACLs apply at the subnet level and act as guard rails even if an instance launches without the correct security group. [doc](https://docs.aws.amazon.com/vpc/latest/userguide/infrastructure-security.html)
- **[Private subnets]** Place instances that should not be reached directly from the internet in private subnets, and use a NAT gateway or bastion for outbound internet access. [doc](https://docs.aws.amazon.com/vpc/latest/userguide/infrastructure-security.html)
- **[Least privilege ports]** Restrict administrative ports such as SSH (22) and RDP (3389) in security groups, and prefer AWS Systems Manager Session Manager for remote access instead of exposing inbound management ports. [doc](https://aws.amazon.com/blogs/security/practical-steps-to-minimize-key-exposure-using-aws-security-services/)
- **[Inbound rules]** Restrict inbound security group rules to specific IP ranges rather than allowing traffic from 0.0.0.0/0. [doc](https://docs.aws.amazon.com/vpc/latest/userguide/getting-started-with-amazon-vpc-using-the-aws-cli.html)
- **[Perimeter defense]** Deploy AWS Network Firewall to filter inbound and outbound VPC traffic with intrusion prevention, domain filtering, and deep packet inspection as an additional layer beyond security groups and NACLs. [doc](https://docs.aws.amazon.com/vpc/latest/userguide/infrastructure-security.html)
- **[Centralized policy]** Use AWS Firewall Manager to centrally administer Network Firewall rules and security group policies across accounts, automating cleanup of unused or overly permissive rules as the organization grows. [doc](https://aws.amazon.com/blogs/security/practical-steps-to-minimize-key-exposure-using-aws-security-services/)
- **[Visibility]** Enable VPC Flow Logs to monitor IP traffic to and from a VPC, subnet, or network interface, and feed them into Amazon GuardDuty for threat detection. [doc](https://docs.aws.amazon.com/vpc/latest/userguide/vpc-security-best-practices.html)
- **[Access auditing]** Use Network Access Analyzer to identify unintended network access paths to resources in your VPCs. [doc](https://docs.aws.amazon.com/vpc/latest/userguide/vpc-security-best-practices.html)
- **[IAM integration]** Manage access to VPC resources and APIs using IAM identity federation, users, and roles rather than relying on network controls alone. [doc](https://docs.aws.amazon.com/vpc/latest/userguide/vpc-security-best-practices.html)

## Multi-VPC & Hybrid Connectivity
- **[Hub and spoke]** Use AWS Transit Gateway as a regional hub-and-spoke design to connect thousands of VPCs and on-premises networks without third-party appliances or a VPN overlay. [doc](https://docs.aws.amazon.com/whitepapers/latest/building-scalable-secure-multi-vpc-network-infrastructure/transit-gateway.html)
- **[Single gateway per Region]** Use one Transit Gateway per Region and rely on multiple Transit Gateway route tables to segment routing domains (for example production vs. non-production) — Transit Gateway is highly available by design, so extra gateways are not needed purely for redundancy. [doc](https://docs.aws.amazon.com/whitepapers/latest/building-scalable-secure-multi-vpc-network-infrastructure/transit-gateway.html)
- **[Centralized ownership]** Place the organization's Transit Gateway in a dedicated Network Services account and share it across accounts using AWS Resource Access Manager (RAM) for centralized management. [doc](https://docs.aws.amazon.com/whitepapers/latest/building-scalable-secure-multi-vpc-network-infrastructure/transit-gateway.html)
- **[Traffic isolation]** Use separate Transit Gateway route tables to allow spoke VPCs to reach a shared-services or inspection VPC while disallowing direct spoke-to-spoke access. [doc](https://aws.amazon.com/blogs/architecture/field-notes-how-to-scale-your-networks-on-amazon-web-services/)
- **[Centralized inspection]** Route inter-VPC and egress traffic through a central inspection VPC with AWS Network Firewall deployed behind Transit Gateway, rather than placing firewalls in every spoke VPC. [doc](https://aws.amazon.com/blogs/apn/how-to-import-and-manage-aws-networking-with-terraform-and-controlmonkey/)
- **[Backbone routing]** Prefer Transit Gateway peering or VPC peering over public internet paths for inter-Region or inter-VPC connectivity — traffic stays on the AWS global backbone and never traverses the public internet. [doc](https://docs.aws.amazon.com/whitepapers/latest/aws-vpc-connectivity-options/aws-transit-gateway.html)
- **[Hybrid links]** Use AWS Direct Connect or Site-to-Site VPN to establish private connections between on-premises networks and VPCs instead of routing hybrid traffic over the public internet. [doc](https://docs.aws.amazon.com/vpc/latest/userguide/infrastructure-security.html)
- **[Private AWS service access]** Use interface VPC endpoints (AWS PrivateLink) or gateway endpoints so traffic to supported AWS services stays on the AWS network instead of traversing an internet gateway or NAT device. [doc](https://aws.amazon.com/blogs/architecture/reduce-cost-and-increase-security-with-amazon-vpc-endpoints/)
- **[Endpoint policies]** Attach VPC endpoint policies alongside IAM identity-based policies to tightly control which principals can use a PrivateLink endpoint to reach an AWS service. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/streamline-access-to-most-used-aws-services-using-vpc-endpoints/)
- **[Centralized endpoints]** Centralize interface VPC endpoint access in a shared services VPC rather than deploying redundant endpoints in every VPC, to simplify management and control cost. [doc](https://docs.aws.amazon.com/whitepapers/latest/building-scalable-secure-multi-vpc-network-infrastructure/centralized-access-to-vpc-private-endpoints.html)

## DNS
- **[Hybrid resolution]** Deploy a Route 53 Resolver inbound endpoint and outbound endpoint to bridge DNS resolution between VPCs and on-premises networks over Direct Connect or VPN, since the native VPC resolver is not reachable from on-premises. [doc](https://docs.aws.amazon.com/whitepapers/latest/building-scalable-secure-multi-vpc-network-infrastructure/dns.html)
- **[Centralize resolvers]** Centralize Route 53 Resolver endpoints in a central egress/Network account rather than creating them in every VPC, then share them — this keeps management simpler and cost lower. [doc](https://docs.aws.amazon.com/whitepapers/latest/building-scalable-secure-multi-vpc-network-infrastructure/dns.html)
- **[Selective forwarding]** Create Route 53 Resolver rules that forward only the specific domain names you need to on-premises DNS servers, rather than forwarding all queries. [doc](https://docs.aws.amazon.com/whitepapers/latest/building-scalable-secure-multi-vpc-network-infrastructure/dns.html)
- **[Resolver HA]** Distribute Route 53 Resolver endpoint ENIs across multiple Availability Zones, paired with redundant Site-to-Site VPN or Direct Connect links, for DNS high availability. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/securing-hybrid-workloads-using-amazon-route-53-resolver-dns-firewall/)

## High Availability & Load Balancing
- **[NAT redundancy]** Deploy a NAT gateway in each Availability Zone you use rather than a single shared NAT gateway — an AZ-scoped NAT gateway failure or AZ outage should not take down egress for other zones. [doc](https://docs.aws.amazon.com/whitepapers/latest/building-scalable-secure-multi-vpc-network-infrastructure/using-nat-gateway-for-centralized-egress.html)
- **[Zonal affinity]** Route private subnet traffic to the NAT gateway in the same Availability Zone to avoid unnecessary cross-AZ data transfer, or use a regional NAT gateway that automatically follows workloads across AZs. [doc](https://docs.aws.amazon.com/vpc/latest/userguide/nat-gateways-regional.html)
- **[Cross-zone load balancing]** Plan for enough target capacity in every participating Availability Zone per target group; if you cannot guarantee that, keep cross-zone load balancing enabled so a healthy zone can absorb traffic from an unhealthy one. [doc](https://docs.aws.amazon.com/elasticloadbalancing/latest/application/edit-target-group-attributes.html)
- **[Empty subnets]** Avoid empty subnets behind an Application Load Balancer — ALB exposes zonal IP addresses via DNS for every configured subnet, and an empty AZ subnet with cross-zone balancing off triggers 503 errors. [doc](https://docs.aws.amazon.com/elasticloadbalancing/latest/application/edit-target-group-attributes.html)
- **[Target group parity]** When using multiple target groups on the same load balancer, ensure they all participate in the same set of Availability Zones to avoid unbalanced or empty-zone failures. [doc](https://docs.aws.amazon.com/elasticloadbalancing/latest/application/edit-target-group-attributes.html)
- **[NLB cross-zone]** Understand that Network Load Balancer cross-zone load balancing is disabled by default (unlike ALB, where it's always on) — evaluate whether your workload needs it enabled for even distribution versus keeping it off to avoid inter-AZ data charges. [doc](https://docs.aws.amazon.com/elasticloadbalancing/latest/userguide/how-elastic-load-balancing-works.html)
- **[Health thresholds]** When cross-zone load balancing is off, size DNS failover and routing (fail-open) thresholds based on the actual number of targets registered per Availability Zone. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/using-load-balancer-target-group-health-thresholds-to-improve-availability/)

## Cost & Data Transfer Optimization
- **[Endpoint over NAT]** Use an Amazon S3 gateway VPC endpoint for private-subnet-to-S3 traffic within a Region instead of routing it through a NAT gateway, eliminating both the NAT processing charge and the data transfer path. [doc](https://aws.amazon.com/blogs/industries/analyze-data-transfer-and-adopt-cost-optimized-designs-to-realize-cost-savings/)
- **[NAT placement]** Verify that the resources sending the most traffic through a NAT gateway are in the same Availability Zone as that NAT gateway; if not, add a NAT gateway in the correct AZ to avoid unnecessary cross-AZ data transfer charges. [doc](https://docs.aws.amazon.com/wellarchitected/latest/framework/cost_data_transfer_optimized_components.html)
- **[LB selection]** Prefer Application Load Balancer or Classic Load Balancer where suitable, since they incur no cross-AZ data transfer charge for traffic to targets, unlike Network Load Balancer, which charges for cross-AZ traffic when cross-zone balancing is enabled. [doc](https://aws.amazon.com/blogs/industries/analyze-data-transfer-and-adopt-cost-optimized-designs-to-realize-cost-savings/)
- **[AZ-aware design]** Design applications to be Availability-Zone aware (for example using AZ-affinity features of the data stores and caches you run) to minimize cross-AZ traffic and its associated cost. [doc](https://aws.amazon.com/blogs/industries/analyze-data-transfer-and-adopt-cost-optimized-designs-to-realize-cost-savings/)
- **[Hybrid egress]** Evaluate AWS Direct Connect instead of internet-based transfer for steady, high-volume traffic between AWS and on-premises environments — and avoid routing high-volume hybrid traffic through Transit Gateway, which adds a data processing fee. [doc](https://aws.amazon.com/blogs/industries/analyze-data-transfer-and-adopt-cost-optimized-designs-to-realize-cost-savings/)
- **[Regional boundaries]** Avoid transferring data across AWS Region boundaries unless a deliberate multi-Region architecture requires it, since inter-Region transfer typically incurs charges. [doc](https://docs.aws.amazon.com/wellarchitected/latest/framework/cost_data_transfer_optimized_components.html)
- **[Monitoring spend]** Enable Cost and Usage Reports together with VPC Flow Logs, and analyze them with Cost Explorer or a cost intelligence dashboard, to attribute and track data transfer costs by component. [doc](https://aws.amazon.com/blogs/industries/analyze-data-transfer-and-adopt-cost-optimized-designs-to-realize-cost-savings/)
- **[Unintended paths]** Use Network Access Analyzer to detect unintended traffic paths routed through a NAT gateway that could otherwise use a cheaper, more direct path. [doc](https://aws.amazon.com/blogs/industries/analyze-data-transfer-and-adopt-cost-optimized-designs-to-realize-cost-savings/)
