# Observability — Best Practices

## Instrumentation and Coverage
- **[all workload components]** Turn on logging and metric collection for every component, including managed services, so no part of the workload is a blind spot — gaps in monitoring hide the root cause when incidents occur. [doc](https://docs.aws.amazon.com/wellarchitected/2023-04-10/framework/rel_monitor_aws_resources_monitor_resources.html)
- **[default metrics]** Review the default metrics every AWS service publishes and identify collection gaps before adding custom instrumentation — understanding built-in signals first avoids duplicated effort. [doc](https://docs.aws.amazon.com/wellarchitected/2023-04-10/framework/rel_monitor_aws_resources_monitor_resources.html)
- **[distributed tracing]** Instrument every service in the request path with a single, consistent tracing solution rather than mixing partial or third-party coverage — incomplete tracing (for example, skipping Lambda) hides cold-start and latency root causes. [doc](https://docs.aws.amazon.com/wellarchitected/latest/framework/rel_monitor_aws_resources_end_to_end.html)
- **[client-side visibility]** Configure synthetic canaries and real-user monitoring (RUM) with tracing enabled so client-side interactions are correlated with backend traces — without them, the performance profile is incomplete. [doc](https://docs.aws.amazon.com/wellarchitected/latest/framework/rel_monitor_aws_resources_end_to_end.html)
- **[synthetic testing]** Run synthetic transactions that replay real customer workflows against your endpoints continuously, even without live traffic, to discover failures before customers do. [doc](https://docs.aws.amazon.com/wellarchitected/2023-04-10/framework/rel_monitor_aws_resources_monitor_resources.html)
- **[layered telemetry]** Collect observability data at the infrastructure, application, and business-metric layers together, not just infrastructure, so degradations that affect KPIs and SLAs are visible, not only resource-level failures. [doc](https://aws.amazon.com/solutions/guidance/deep-application-observability-on-aws/)
- **[end-to-end tracing]** Elect one tracing solution for hybrid or multi-vendor environments and fully integrate it, using cloud-native SDKs to instrument non-native components — a partially integrated tracing stack fragments root-cause analysis. [doc](https://docs.aws.amazon.com/wellarchitected/latest/framework/rel_monitor_aws_resources_end_to_end.html)

## Metrics, Alarms, and Dashboards
- **[custom metrics]** Define and publish custom business and application metrics (KPIs), not just infrastructure metrics, so indirect or business-impacting problems are detected even when underlying resources look healthy. [doc](https://docs.aws.amazon.com/wellarchitected/2022-03-31/framework/rel_withstand_component_failures_monitoring_health.html)
- **[alert selection]** Alert on a focused subset of metrics that materially affect reliability rather than every available metric — over-alerting produces false positives and alert fatigue that erode response effectiveness. [doc](https://docs.aws.amazon.com/wellarchitected/2023-04-10/framework/rel_monitor_aws_resources_monitor_resources.html)
- **[alarm response]** Define the recovery process and ownership for every alert before it fires, ideally driving automated workflows from CloudWatch Alarms, so teams can act immediately instead of triaging from scratch. [doc](https://docs.aws.amazon.com/wellarchitected/2023-04-10/framework/rel_monitor_aws_resources_monitor_resources.html)
- **[dynamic thresholds]** Use CloudWatch anomaly detection instead of static thresholds for metrics with seasonal or variable patterns (daily/weekly cycles, organic growth) to reduce false-positive alarms without manual threshold tuning. [doc](https://docs.aws.amazon.com/AmazonCloudWatch/latest/monitoring/CloudWatch_Anomaly_Detection.html)
- **[anomaly detection scope]** Exclude known anomalous periods (deployments, planned load tests) from anomaly-detection model training so the learned baseline reflects genuine normal behavior. [doc](https://docs.aws.amazon.com/AmazonCloudWatch/latest/monitoring/CloudWatch_Anomaly_Detection.html)
- **[operations KPIs]** Establish operations metrics baselines and learn expected activity patterns before configuring alerts, so thresholds are grounded in observed behavior rather than guesses. [doc](https://docs.aws.amazon.com/wellarchitected/2023-04-10/framework/ops-09.html)
- **[dashboards]** Build dashboards to visualize trends and outliers across your metrics so potential problems are surfaced visually before they escalate into incidents. [doc](https://docs.aws.amazon.com/wellarchitected/2022-03-31/framework/rel_withstand_component_failures_monitoring_health.html)
- **[detailed monitoring interval]** Match your metric collection interval (for example, 1-minute detailed monitoring versus 5-minute default) to your recovery time objective, since coarse intervals can delay detection beyond your RTO. [doc](https://docs.aws.amazon.com/wellarchitected/2022-03-31/framework/rel_withstand_component_failures_monitoring_health.html)
- **[periodic review]** Regularly review monitoring scope and metrics as the workload evolves, retiring stale alarms and adding coverage for new components rather than treating the initial setup as permanent. [doc](https://docs.aws.amazon.com/wellarchitected/2025-02-25/framework/rel-06.html)

## Structured Logging and Metric Extraction
- **[embedded metric format]** Emit structured logs using CloudWatch embedded metric format (EMF) so metrics are extracted automatically from log events without maintaining a separate metrics-instrumentation code path. [doc](https://docs.aws.amazon.com/AmazonCloudWatch/latest/monitoring/CloudWatch_Embedded_Metric_Format.html)
- **[dimension cardinality]** Avoid embedding high-cardinality fields (such as `requestId`) as EMF dimensions — each unique dimension combination creates a distinct custom metric, inflating custom-metric usage and cost. [doc](https://docs.aws.amazon.com/AmazonCloudWatch/latest/monitoring/CloudWatch_Embedded_Metric_Format.html)
- **[log-derived alarms]** Alarm on metrics extracted from structured application logs (for example, per-segment latency) to narrow root-cause investigation to the specific dimension driving a regression. [doc](https://aws.amazon.com/blogs/mt/accelerate-troubleshooting-with-structured-logs-in-amazon-cloudwatch/)
- **[Lambda logging format]** Keep embedded-metric-format client libraries current when adopting the Lambda JSON log format, since older EMF libraries and raw `console.log` output may no longer be parsed correctly by CloudWatch. [doc](https://docs.aws.amazon.com/lambda/latest/dg/nodejs-logging.html)

## Distributed Tracing and Sampling
- **[sampling rate]** Tune X-Ray sampling rules to your workload instead of relying on the conservative default, sampling state-changing or transactional calls at a high rate and high-volume read-only calls at a lower rate. [doc](https://docs.aws.amazon.com/xray/latest/devguide/xray-concepts.html)
- **[adaptive sampling]** Configure adaptive sampling to boost trace capture automatically during anomalies (for example, elevated fault rates or latency) while keeping baseline sampling low, preserving root-cause visibility without paying for full sampling at all times. [doc](https://docs.aws.amazon.com/xray/latest/devguide/xray-adaptive-sampling.html)
- **[OpenTelemetry migration]** When migrating from X-Ray SDK instrumentation to OpenTelemetry, choose an explicit sampling strategy (parent-based, trace-ID-ratio, or tail sampling) rather than assuming default behavior carries over unchanged. [doc](https://docs.aws.amazon.com/xray/latest/devguide/xray-sdk-migration.html)

## Security and Access Control for Telemetry
- **[encryption]** Encrypt CloudWatch Logs log groups with AWS KMS and rely on TLS 1.2+ in transit so telemetry data meets the same protection standards as the workload it monitors. [doc](https://docs.aws.amazon.com/AmazonCloudWatch/latest/logs/infrastructure-security.html)
- **[sensitive data masking]** Apply CloudWatch Logs data protection policies to automatically discover and mask sensitive data in log events, rather than depending solely on application code to redact it before logging. [doc](https://docs.aws.amazon.com/AmazonCloudWatch/latest/logs/data-protection.html)
- **[least privilege]** Grant access to logs, dashboards, and alarm configuration through IAM with least-privilege permissions, and route changes to alarm thresholds or automated actions through a change-review process. [doc](https://aws.amazon.com/solutions/guidance/deep-application-observability-on-aws/)
- **[no secrets in telemetry]** Never place confidential or sensitive information in tags or free-form text fields (including log fields) since this data may propagate into billing records or diagnostic logs. [doc](https://docs.aws.amazon.com/AmazonCloudWatch/latest/logs/data-protection.html)
- **[audit trail]** Capture API and user activity with AWS CloudTrail alongside application telemetry so operational monitoring and security auditing draw from a consistent, tamper-evident record. [doc](https://docs.aws.amazon.com/AmazonCloudWatch/latest/logs/data-protection.html)

## Cost Optimization for Observability Data
- **[log retention]** Set an explicit retention period on every CloudWatch Logs log group instead of relying on the indefinite default, since storage costs accrue for logs retained forever. [doc](https://docs.aws.amazon.com/eks/latest/best-practices/cost-opt-observability.html)
- **[tiered retention]** Differentiate retention by environment and log type — shorter retention for development and non-critical operational logs, longer retention for production and compliance-relevant logs. [doc](https://docs.aws.amazon.com/eks/latest/best-practices/cost-opt-observability.html)
- **[archival]** Export logs that must be retained long-term for compliance to Amazon S3 and apply lower-cost storage classes, rather than keeping them in CloudWatch Logs indefinitely. [doc](https://aws.amazon.com/blogs/compute/optimizing-your-aws-lambda-costs-part-2/)
- **[automated governance]** Automate retention-policy enforcement for new and existing log groups (for example, via an EventBridge rule invoking a Lambda function) instead of relying on manual, per-log-group configuration. [doc](https://aws.amazon.com/blogs/infrastructure-and-automation/reduce-log-storage-costs-by-automating-retention-settings-in-amazon-cloudwatch/)
- **[trace sampling cost]** Lower X-Ray sampling rates for high-volume, low-risk endpoints to control tracing costs while preserving enough data to detect issues. [doc](https://docs.aws.amazon.com/eks/latest/best-practices/cost-opt-observability.html)
- **[ingestion review]** When log ingestion costs exceed expectations, reduce retention and reassess whether every source needs full log delivery, since ingestion is priced independently from storage. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/sal-cw-troubleshooting.html)

## Operational Review and Continuous Improvement
- **[KPI ownership]** Identify key performance indicators and assign an owner responsible for each operational outcome so alerts have a clear, accountable recipient. [doc](https://docs.aws.amazon.com/wellarchitected/2023-04-10/framework/ops_operations_health_ops_outcome_alerts.html)
- **[post-incident analysis]** Perform post-incident analysis after alerts fire and feed findings back into KPI and metric definitions, closing the loop between monitoring and continuous improvement. [doc](https://docs.aws.amazon.com/wellarchitected/2023-04-10/framework/ops_operations_health_ops_outcome_alerts.html)
- **[monitoring solution selection]** Match the monitoring tool to the observability need — RUM for client-side performance, X-Ray for cross-component latency, Database Insights for query performance — rather than instrumenting everything with a single generic tool. [doc](https://docs.aws.amazon.com/wellarchitected/latest/performance-efficiency-pillar/perf_process_culture_use_monitoring_solutions.html)
- **[metric retention alignment]** Configure metric and log retention policies to match both security requirements and operational goals, reviewing them as part of regular workload assessments rather than a one-time setup. [doc](https://docs.aws.amazon.com/wellarchitected/latest/performance-efficiency-pillar/perf_compute_hardware_collect_compute_related_metrics.html)

## Common scenarios
- New service onboarding: enable default AWS service metrics and structured application logging from day one, add distributed tracing before the service takes production traffic, and set log retention explicitly rather than accepting the indefinite default.
- Debugging elevated latency: use end-to-end X-Ray traces correlated with EMF-derived latency metrics to isolate the failing component, then drill into CloudWatch Logs Insights for the detailed root cause.
- Reducing observability spend: apply log-group retention policies, export compliance logs to S3, lower trace sampling on high-volume low-risk routes, and use anomaly detection instead of proliferating static alarms.
- Preparing for an incident review: verify every workload component publishes metrics, every alert has a documented owner and runbook, and dashboards reflect current architecture rather than a stale initial build.
