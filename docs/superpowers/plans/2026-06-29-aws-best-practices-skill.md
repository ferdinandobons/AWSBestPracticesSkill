# AWSBestPracticesSkill Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build a Claude/Codex skill that collects ONLY AWS best practices for every AWS service, navigable by use case, sourced from official AWS docs.

**Architecture:** A `SKILL.md` router points to `services/<category>/<service>.md` files. Each service file holds only best practices, organized by the 6 Well-Architected pillars, each practice carrying a `[when-it-applies]` context tag and an official AWS source link. `catalog.json` is the machine-readable source of truth; `scripts/check.py` validates structure/coverage/freshness and diffs regenerations. Content is generated by querying the AWS Knowledge MCP, in two stages: a few exemplar services to lock the format, then a fan-out Workflow over all services.

**Tech Stack:** Markdown content; Python 3.8+ stdlib for `scripts/check.py` (no deps); GitHub Actions for CI; AWS Knowledge MCP (`mcp__plugin_deploy-on-aws_awsknowledge__*`) and AWS Pricing MCP (`get_pricing_service_codes`) for sourcing; the Workflow tool for mass generation.

## Global Constraints

- **Only best practices** in `services/**`: no service descriptions, overviews, pricing/cost estimates, tutorials, getting-started, or extended code samples. Cost *optimization practices* are allowed; prices are not.
- **Content language: English** for all service file content and `SKILL.md`.
- **Every best-practice bullet** = imperative action + `[context tag]` + short rationale + `[doc](official AWS URL)`.
- **Allowed H2 sections** in a service file: `Scenari comuni`→ rendered as **`Common scenarios`** (English) + the 6 pillars: `Security`, `Reliability`, `Performance Efficiency`, `Cost Optimization`, `Operational Excellence`, `Sustainability`. Include only pillars with real practices.
- **`scripts/check.py`**: Python stdlib only, exit ≠ 0 on errors (CI gate).
- **Sources**: official AWS domains only (`docs.aws.amazon.com`, `aws.amazon.com/architecture`, `wa.aws.amazon.com`).
- **Release discipline**: CI green before any version bump/tag; README is part of the release surface; tag only when stable.
- The spec is the companion reference: `docs/superpowers/specs/2026-06-29-aws-best-practices-skill-design.md`.

> **Note on H2 naming:** the spec template drafted `Scenari comuni` (Italian) in mockups, but per the resolved language decision the actual section header in files is **`Common scenarios`**. `check.py`'s `ALLOWED_H2` must use `Common scenarios`.

## Addendum — added scope (2026-06-29)

Per user request: a `/goal` command that drives recursive/parallel generation, and general (cross-service) best-practice docs. This modifies Tasks 2, 4, 5, 6 and adds 4b/4c.

- **check.py `type` support (Task 2)**: each catalog entry has `type: "service" | "general"` (default `service`). `iter_entries` yields it; `check_conformance(path, is_general)` skips the pillar / `Common scenarios` requirement when general, but still requires H1, ≥1 H2 section, and every bullet sourced. Build this into `check.py` from the start.

- **Task 4b — General docs**: create the `general/` files listed in spec §14.1 via AWS Knowledge MCP research (cross-service best practices only). Add a `general` category to `catalog.json` with `type: "general"` entries (no `aws_service_code`; `pillars` may be empty). One general doc (`general/security-baseline.md`) is part of the Task 5 exemplar checkpoint.

- **Task 4c — `/goal` command**: create
  - `.claude/commands/goal.md` — the slash command prompt (per spec §14.3): instructs the model to compute the work-list via `python3 scripts/check.py --json`, launch the parent Workflow with `args`, then update `catalog.json` + run `check.py`. Supports `$ARGUMENTS` = `<category> | --stale | --all`.
  - `.claude/workflows/goal.js` — parent workflow: `parallel(categories.map(c => () => workflow({scriptPath: '.claude/workflows/goal-category.js'}, {category, services})))`.
  - `.claude/workflows/goal-category.js` — child workflow: `pipeline(services, generate, verify)` with loop-until-dry on failures; the `generate` agent queries the MCP and Writes the service/general file from the template; `verify` rejects descriptions/pricing/tutorials and confirms sources. Returns `{path, pillars, sources, status}` per service.

- **Task 6 (revised)**: mass generation is performed BY invoking `/goal --all` (the recursive parallel workflow), not a one-off pipeline. Task 6 becomes: run `/goal --all`, let it loop until coverage is complete, update `catalog.json` from returned metadata, then `python3 scripts/check.py` + `--check-links` green.

**Workflow constraints recap:** JS scripts have no filesystem access → the work-list passes via `args`; the spawned agents (full subagents) do the file Writes. Workflow nesting is one level only (parent→category-child; the child uses agents, not further `workflow()` calls). `catalog.json` is updated once by the main loop after the workflow returns, to avoid concurrent-write races.
