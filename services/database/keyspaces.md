# Amazon Keyspaces — Best Practices

## Common scenarios
- Migrating existing Apache Cassandra workloads to a managed, serverless service        → Reliability, Operational Excellence
- High-scale NoSQL applications needing single-digit-millisecond reads/writes        → Performance Efficiency, Cost Optimization
- Multi-Region, active-active applications requiring low-latency global access        → Reliability, Performance Efficiency
- Multi-tenant SaaS data isolation using IAM-scoped keyspaces and tables        → Security

## 🔒 Security
- **[encryption at rest]** Rely on the default AWS owned key for encryption at rest, or switch to a customer managed KMS key when you need control over key rotation and access auditing. [doc](https://docs.aws.amazon.com/keyspaces/latest/devguide/best-practices-security-preventative.html)
- **[authentication]** Use IAM roles instead of long-term credentials for applications and EC2 instances accessing Amazon Keyspaces, since roles provide temporary, automatically rotated access keys. [doc](https://docs.aws.amazon.com/keyspaces/latest/devguide/best-practices-security-preventative.html)
- **[authorization]** Attach least-privilege IAM policies (AWS managed or customer managed) that scope permissions to specific Amazon Keyspaces APIs and resources rather than granting broad access. [doc](https://docs.aws.amazon.com/keyspaces/latest/devguide/best-practices-security-preventative.html)
- **[fine-grained access]** Use IAM policy conditions to restrict access to specific keyspaces or tables, for example granting read-only access to some users and write access to others based on identity. [doc](https://docs.aws.amazon.com/keyspaces/latest/devguide/best-practices-security-preventative.html)
- **[tenant isolation]** In multi-tenant designs, scope IAM identity-based policies to each tenant's keyspace so tenants cannot access or modify another tenant's data or schema. [doc](https://aws.amazon.com/blogs/apn/storing-multi-tenant-saas-data-in-a-serverless-environment-with-amazon-keyspaces/)
- **[client-side encryption]** Encrypt sensitive or confidential data as close to its origin as possible so plaintext is never exposed to a third party throughout its lifecycle. [doc](https://docs.aws.amazon.com/keyspaces/latest/devguide/best-practices-security-preventative.html)
- **[transport security]** Require TLS 1.2 (and prefer TLS 1.3) with cipher suites that support perfect forward secrecy (DHE or ECDHE) for all client driver connections. [doc](https://docs.aws.amazon.com/keyspaces/latest/devguide/infrastructure-security.html)
- **[network isolation]** Use interface VPC endpoints (AWS PrivateLink) to keep traffic between your VPC and Amazon Keyspaces off the public internet, and combine with resource policies to restrict access to specific VPCs or source IPs. [doc](https://docs.aws.amazon.com/keyspaces/latest/devguide/infrastructure-security.html)
- **[key usage auditing]** Monitor customer managed KMS key usage via AWS CloudTrail to track when the key was used, by whom, and from which IP address. [doc](https://docs.aws.amazon.com/keyspaces/latest/devguide/best-practices-security-detective.html)
- **[DDL auditing]** Enable AWS CloudTrail to capture Amazon Keyspaces data definition language operations (creating/altering keyspaces and tables) for compliance and change tracking. [doc](https://docs.aws.amazon.com/keyspaces/latest/devguide/best-practices-security-detective.html)
- **[resource tagging]** Tag keyspaces and tables by purpose, environment, or confidentiality level, and use tag-based IAM conditions to enforce access and security controls at scale. [doc](https://docs.aws.amazon.com/keyspaces/latest/devguide/best-practices-security-detective.html)

## 🛡️ Reliability
- **[durability]** Rely on Amazon Keyspaces' automatic three-way replication across Availability Zones within a Region for built-in durability and high availability, without managing infrastructure. [doc](https://docs.aws.amazon.com/keyspaces/latest/devguide/disaster-recovery-resiliency.html)
- **[backups]** Enable point-in-time recovery (PITR) so tables can be continuously backed up and restored, protecting against accidental writes or deletes. [doc](https://docs.aws.amazon.com/keyspaces/latest/devguide/disaster-recovery-resiliency.html)
- **[multi-Region resilience]** Use multi-Region replication for applications that need protection from single-Region degradation, near-zero RTO, and single-digit-second RPO through active-active reads and writes. [doc](https://docs.aws.amazon.com/keyspaces/latest/devguide/multiRegion-replication_how-it-works.html)
- **[failover]** Route application traffic to a healthy Region using Amazon Route 53 during a Regional failure, since multi-Region keyspaces require no database promotion or failover procedure. [doc](https://docs.aws.amazon.com/keyspaces/latest/devguide/multiRegion-replication_how-it-works.html)
- **[conflict resolution]** Design applications with the understanding that concurrent multi-Region updates are resolved using cell-level timestamps and last-writer-wins semantics, and avoid relying on strict cross-Region write ordering. [doc](https://docs.aws.amazon.com/keyspaces/latest/devguide/multiRegion-replication.html)
- **[PITR for multi-Region]** Enable PITR on all replicas of a multi-Region table before relying on it for restore, since a successful restore requires PITR enabled across every replicated Region. [doc](https://docs.aws.amazon.com/keyspaces/latest/devguide/PointInTimeRecovery_HowItWorks.html)
- **[connection resiliency]** Configure client driver retry policies appropriately so transient connection or throughput errors are retried instead of surfacing as application failures. [doc](https://docs.aws.amazon.com/keyspaces/latest/devguide/connections.html)
- **[partition design]** Design partition keys for uniform read/write activity distribution to avoid hot partitions that can degrade availability under sustained load. [doc](https://docs.aws.amazon.com/keyspaces/latest/devguide/bp-partition-key-design.html)

## ⚡ Performance Efficiency
- **[partition keys]** Design partition keys so read and write activity is spread evenly across partitions, aligning the key design with your application's actual access patterns. [doc](https://docs.aws.amazon.com/keyspaces/latest/devguide/bp-partition-key-design.html)
- **[write sharding]** Use write sharding — appending a random or calculated value to a compound partition key — to spread heavy write throughput across more partitions when a single logical key would become a bottleneck. [doc](https://docs.aws.amazon.com/keyspaces/latest/devguide/bp-partition-key-design.html)
- **[per-partition throughput]** Keep per-partition traffic within the supported throughput limits (3,000 RCUs and 1,000 WCUs) and rely on burst capacity for short-term spikes rather than designing around sustained overages. [doc](https://docs.aws.amazon.com/keyspaces/latest/devguide/bp-partition-key-design.html)
- **[connection pooling]** Configure the client driver to establish multiple connections per endpoint — targeting roughly 500 CQL requests per second per connection — instead of relying on default single-connection-per-endpoint behavior. [doc](https://docs.aws.amazon.com/keyspaces/latest/devguide/connections.html)
- **[connection sizing]** Calculate the number of connections per endpoint from your application's expected queries-per-second, instance count, and available endpoints to avoid `PerConnectionRequestExceeded` errors. [doc](https://docs.aws.amazon.com/keyspaces/latest/devguide/connections.html)
- **[read consistency]** Use eventually consistent reads by default and reserve strongly consistent reads for access patterns that specifically require them, since eventual consistency is both cheaper and lower-latency. [doc](https://docs.aws.amazon.com/keyspaces/latest/devguide/CostOptimization_TableUsagePatterns.html)
- **[bulk loading]** When bulk-loading data (for example with Spark), randomize write order across partition keys and tune batch/retry settings to take advantage of built-in load balancing across partitions. [doc](https://aws.amazon.com/blogs/big-data/how-william-hill-migrated-nosql-workloads-at-scale-to-amazon-keyspaces/)

## 💰 Cost Optimization
- **[capacity mode]** Choose on-demand capacity mode for unpredictable or spiky workloads to avoid over- or under-provisioning, and switch to provisioned mode with Application Auto Scaling once traffic patterns stabilize. [doc](https://docs.aws.amazon.com/keyspaces/latest/devguide/CostOptimization_TableCapacityMode.html)
- **[auto scaling tuning]** Regularly review Application Auto Scaling target utilization settings on provisioned tables to identify low-utilization tables and adjust for seasonal or spiky demand patterns. [doc](https://docs.aws.amazon.com/keyspaces/latest/devguide/best-practices.html)
- **[read consistency]** Use eventually consistent reads wherever your access pattern allows, since strongly consistent reads cost roughly double in read capacity. [doc](https://docs.aws.amazon.com/keyspaces/latest/devguide/CostOptimization_TableUsagePatterns.html)
- **[data lifecycle]** Enable Time to Live (TTL) to automatically expire and delete data you no longer need, simplifying application logic and reducing storage costs. [doc](https://docs.aws.amazon.com/keyspaces/latest/devguide/CostOptimization_TableUsagePatterns.html)
- **[unused resources]** Periodically identify and clean up unused tables and stale point-in-time recovery backups to eliminate costs from resources no longer in active use. [doc](https://docs.aws.amazon.com/keyspaces/latest/devguide/best-practices.html)
- **[cost visibility]** Apply consistent table tags and use Cost Explorer's table-level cost views to attribute Amazon Keyspaces spend to teams, applications, or environments. [doc](https://docs.aws.amazon.com/keyspaces/latest/devguide/best-practices.html)

## ⚙️ Operational Excellence
- **[monitoring]** Use the preconfigured Amazon Keyspaces console dashboard and Amazon CloudWatch metrics to track latency and errors across tables, establishing a performance baseline over time. [doc](https://docs.aws.amazon.com/keyspaces/latest/devguide/monitoring-overview.html)
- **[alarms]** Set CloudWatch alarms on key metrics such as `SystemErrors`, `ConsumedReadCapacityUnits`, and `ConsumedWriteCapacityUnits` to detect anomalies and trigger automated responses like Application Auto Scaling actions. [doc](https://docs.aws.amazon.com/keyspaces/latest/devguide/monitoring-cloudwatch.html)
- **[capacity comparison]** Regularly compare provisioned versus consumed read/write capacity using CloudWatch metric math to spot under- or over-provisioned tables. [doc](https://docs.aws.amazon.com/keyspaces/latest/devguide/monitoring-cloudwatch.html)
- **[audit logging]** Enable AWS CloudTrail across all Regions to capture Amazon Keyspaces API activity for troubleshooting operational issues and demonstrating compliance. [doc](https://docs.aws.amazon.com/keyspaces/latest/devguide/best-practices-security-detective.html)
- **[connection monitoring]** Monitor client driver connection health and error rates so misconfigured connection pools (leading to `PerConnectionRequestExceeded`) are caught before they affect application availability. [doc](https://docs.aws.amazon.com/keyspaces/latest/devguide/connections.html)
