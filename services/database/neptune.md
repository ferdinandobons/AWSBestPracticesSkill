# Amazon Neptune — Best Practices

## Common scenarios
- Multi-hop relationship traversal and graph exploration (fraud detection, knowledge graphs, recommendations)        → Performance Efficiency, Cost Optimization
- High-availability transactional graph workloads requiring automatic failover        → Reliability
- Applications needing IAM-authenticated, encrypted access to sensitive relationship data        → Security
- Multi-Region or globally distributed graph applications        → Reliability, Performance Efficiency

## 🔒 Security
- **[data hygiene]** Keep confidential or sensitive information out of cluster names, tags, parameter groups, IAM roles, and other metadata, since this data can appear in billing or diagnostic logs. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/security-pillar.html)
- **[encryption at rest]** Enable storage encryption on Neptune DB clusters using the AWS-managed KMS key or a customer-managed key to protect data at rest and meet compliance requirements. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/security-pillar.html)
- **[query injection]** Use parameterized queries or prepared statements for SPARQL and openCypher instead of string concatenation with user-supplied input, and use Gremlin Language Variants instead of raw string-based Gremlin scripts, to prevent injection attacks. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/security-pillar.html)
- **[network isolation]** Keep production clusters and clusters with sensitive data private to the VPC, avoid enabling public accessibility, and restrict inbound security group rules to known IP traffic if public endpoints are ever used in non-production environments. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/security-pillar.html)
- **[encryption in transit]** Rely on Neptune's enforced SSL/TLS connections for all instance and cluster endpoints, and terminate SSL on any proxy or load balancer rather than attempting SSL passthrough. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/security-pillar.html)
- **[IAM authentication]** Enable IAM database authentication and sign requests with AWS Signature Version 4 so that every connection to the cluster is authenticated, rather than relying on open access within the VPC. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/security-pillar.html)
- **[least privilege]** Grant IAM principals only the Neptune management and data-access permissions required for their duties, use IAM groups to manage permissions for multiple users, and never use the AWS account root user to manage Neptune resources. [doc](https://docs.aws.amazon.com/neptune/latest/userguide/best-practices-general-security.html)
- **[credential lifecycle]** Prefer temporary credentials over persistent ones, and rotate IAM credentials regularly. [doc](https://docs.aws.amazon.com/neptune/latest/userguide/best-practices-general-security.html)

## 🛡️ Reliability
- **[read replica placement]** Deploy at least two Neptune read replicas in different Availability Zones from the writer so a healthy instance can always serve reads and take over as primary during a failover. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/reliability-pillar.html)
- **[uniform instance sizing]** Size read replicas at least as large as the primary writer instance to keep replication lag in check and avoid replica restarts, and keep all instances the same size to facilitate failovers. [doc](https://docs.aws.amazon.com/neptune/latest/userguide/best-practices-general-basic.html)
- **[global database]** Use a Neptune global database when your application has a worldwide footprint or needs multi-Region failover, since it enables low-latency global reads and fast recovery from a Region-wide outage. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/reliability-pillar.html)
- **[auto-scaling]** Use Neptune auto-scaling to add or remove read replicas based on CPU utilization, but account for instance startup taking up to 15 minutes, and schedule known scale-out events at least 15 minutes ahead of demand. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/reliability-pillar.html)
- **[backups]** Configure a backup retention period that matches your recovery point objective, and take manual snapshots when you need to retain data beyond the retention window. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/reliability-pillar.html)
- **[failover testing]** Regularly test failover (including forced and reboot-with-failover scenarios) to understand how long the process takes and confirm that client applications reconnect automatically afterward. [doc](https://docs.aws.amazon.com/neptune/latest/userguide/best-practices-general-basic.html)
- **[client retry logic]** Design clients to create a new connection after failover and implement retry logic with exponential backoff to ride out the brief interruption during primary promotion. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/reliability-pillar.html)
- **[co-location]** Run client applications in the same Region and VPC as the Neptune cluster, since cross-region connections and VPC peering introduce latency that undermines single-digit-millisecond query response times. [doc](https://docs.aws.amazon.com/neptune/latest/userguide/best-practices-general-basic.html)
- **[version upgrades]** Clone your DB cluster and test your application against a new major engine version before upgrading production. [doc](https://docs.aws.amazon.com/neptune/latest/userguide/best-practices-general-basic.html)

## ⚡ Performance Efficiency
- **[graph modeling]** Use granular labels, store only properties needed for path determination or filtering, and use deterministic node IDs where possible instead of relying on random GUIDs, since accessing nodes by ID is the most efficient method. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/performance-efficiency-pillar.html)
- **[high-degree nodes]** Avoid modeling patterns that navigate across nodes with an abnormally high number of same-label edges, since these add significant compute and data complexity; use slow-query logs to detect them. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/performance-efficiency-pillar.html)
- **[query diagnostics]** Use the profile and explain APIs for Gremlin, openCypher, and SPARQL to identify inefficient query plans, redundant filters, and unoptimized steps before they cause production slowdowns. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/performance-efficiency-pillar.html)
- **[slow-query logging]** Enable slow-query logging and review it regularly to catch queries causing excessive I/O or latency. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/performance-efficiency-pillar.html)
- **[OSGP index]** Enable the OSGP index if your query pattern frequently accesses property values without specifying property labels, since the default access strategy becomes inefficient for a large number of distinct edges. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/performance-efficiency-pillar.html)
- **[analytical workloads]** Use Neptune Analytics instead of the transactional Neptune database for analytical, investigatory, or algorithmic workloads that evaluate the entire graph, since Neptune is optimized for queries that fan out from a starting node. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/performance-efficiency-pillar.html)
- **[right-sizing]** Monitor the `BufferCacheHitRatio` (target at or above 99.9 percent) and `MainRequestQueuePendingRequests` (target near zero) CloudWatch metrics, and move to an instance class with more RAM or vCPU when they degrade. [doc](https://docs.aws.amazon.com/neptune/latest/userguide/best-practices-general-basic.html)
- **[read scaling]** Load-balance read traffic across read replicas and reuse Gremlin client objects across threads rather than creating new clients per request. [doc](https://docs.aws.amazon.com/neptune/latest/userguide/best-practices.html)
- **[engine upgrades]** Keep the Neptune engine version current and consider upgrading to 1.4.5.0 or later to take advantage of newer instance families such as Graviton4 `r8g` for better price-performance. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/performance-efficiency-pillar.html)

## 💰 Cost Optimization
- **[right-sizing instances]** Test application performance across instance sizes and compare `BufferCacheHitRatio` and `MainRequestQueuePendingRequests` before committing, since instance costs typically make up around 85 percent of total Neptune spend. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/cost-optimization-pillar.html)
- **[storage configuration]** Choose Neptune Standard for low-to-moderate I/O workloads and Neptune I/O-Optimized when I/O spend exceeds roughly 25 percent of total Neptune cost, and re-evaluate periodically since you can switch the storage type once every 30 days. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/cost-optimization-pillar.html)
- **[serverless vs. provisioned]** Use provisioned instances for stable, predictable workloads and Neptune Serverless for workloads with a few hours of high usage per day, comparing NCU-hour costs against equivalent provisioned instance costs before deciding. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/cost-optimization-pillar.html)
- **[avoid T-family for load testing]** Avoid `db.t3`/`db.t4g` instances for performance or load testing and for workloads using openCypher, Graph Explorer, or generative AI integrations, since their lower RAM-to-vCPU ratio disables DFE engine statistics and produces unrepresentative results. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/cost-optimization-pillar.html)
- **[workload fit]** Confirm your data model has genuine graph structure and multi-hop query needs before choosing Neptune, and consider pairing it with purpose-built databases like DynamoDB for access patterns that don't require graph traversal. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/cost-optimization-pillar.html)
- **[newer instance generations]** Upgrade to Neptune 1.4.5.0 or later and use `r8g` Graviton4 instances to get better write throughput at lower cost than older-generation `r7g`/`r6g` instances. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/cost-optimization-pillar.html)

## ⚙️ Operational Excellence
- **[infrastructure as code]** Deploy Neptune clusters using CloudFormation, AWS CDK, or Terraform, and automate operational procedures such as resizing instances, adding replicas, or performing manual failovers. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/operational-excellence-pillar.html)
- **[externalized connections]** Store connection strings outside the client (for example, in AWS Secrets Manager or DynamoDB) so they can change dynamically to support blue/green deployments and near-zero-downtime migrations. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/operational-excellence-pillar.html)
- **[CI/CD testing]** Test infrastructure and application queries against an ephemeral, non-production Neptune cluster through a CI/CD pipeline before deploying to production. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/operational-excellence-pillar.html)
- **[monitoring and alarms]** Build CloudWatch dashboards and alarms on CPU, memory, `BufferCacheHitRatio`, and `MainRequestQueuePendingRequests` to detect deviations from expected behavior before they affect users. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/operational-excellence-pillar.html)
- **[audit and slow-query logging]** Enable Neptune audit logging and slow-query logging, and publish both to CloudWatch Logs for long-term retention and analysis, since on-instance log storage is limited and gets overwritten. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/operational-excellence-pillar.html)
- **[control-plane logging]** Enable AWS CloudTrail to log Neptune control-plane API calls for audit and compliance purposes. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/operational-excellence-pillar.html)
- **[resource tagging]** Tag Neptune resources with metadata to track ownership, environment, and usage. [doc](https://docs.aws.amazon.com/neptune/latest/userguide/tagging.html)
- **[bulk load hygiene]** Avoid repeated restarts during bulk loading and consider a temporarily larger writer instance to speed up initial data loads. [doc](https://docs.aws.amazon.com/neptune/latest/userguide/best-practices.html)

## 🌱 Sustainability
- **[region selection]** Choose AWS Regions with lower published carbon intensity or proximity to renewable energy projects, cross-referenced against Regions where Neptune is available. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/sustainability-pillar.html)
- **[demand-based right-sizing]** Monitor `CPUUtilization`, `MainRequestQueuePendingRequests`, and `TotalRequestsPerSec` to right-size cluster resources to actual demand, and use Neptune Serverless when traffic patterns vary unpredictably rather than provisioning for peak. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/sustainability-pillar.html)
- **[stop idle environments]** Automate stopping non-production Neptune environments during hours when they are not in use. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/sustainability-pillar.html)
- **[shared resources]** Have developers share a single Neptune cluster and Jupyter notebook instance through multi-tenancy partitioning rather than provisioning separate clusters per developer. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/sustainability-pillar.html)
- **[query and model efficiency]** Optimize queries and graph models to minimize the compute resources required per query, and use the Gremlin results cache to avoid recomputing frequently recurring or paginated queries. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/sustainability-pillar.html)
- **[stay current]** Keep Neptune engine versions up to date to benefit from support for more efficient instance families like Graviton and from query optimization improvements that reduce resource consumption. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-well-architected-framework/sustainability-pillar.html)
