# Amazon ElastiCache — Best Practices

## Common scenarios
- Caching layer in front of a relational or NoSQL database        → Performance Efficiency, Cost Optimization
- Session store for web/mobile applications        → Reliability, Security
- Real-time leaderboards, counters, and pub/sub messaging        → Performance Efficiency
- Low-latency feature store for ML inference        → Performance Efficiency, Reliability

## 🔒 Security
- **[network access]** Restrict inbound rules on the cluster's security group to the specific clients/security groups that need access instead of broad CIDR ranges — limits the network attack surface. [doc](https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/SecurityPillar.html)
- **[Lambda access]** Grant AWS Lambda functions access to ElastiCache via an IAM execution role with the `AWSLambdaVPCAccessExecutionRole` permission rather than embedding credentials — keeps access managed through IAM. [doc](https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/SecurityPillar.html)
- **[authentication]** Enable Role-Based Access Control (RBAC) with defined users, user groups, and access strings on Redis OSS 6.x+/Valkey 7.2+ clusters, and require in-transit encryption to use it — enforces least privilege at the data-plane level. [doc](https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/SecurityPillar.html)
- **[authentication]** On engine versions that don't support RBAC, set a strong AUTH token and maintain a rotation policy — ElastiCache supports up to two concurrent tokens, letting you rotate without downtime. [doc](https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/SecurityPillar.html)
- **[command safety]** Rename or disable disruptive commands (e.g. `FLUSHALL`, `KEYS`) using the `rename-commands` parameter or RBAC's `-@dangerous` category — prevents accidental data loss from mistaken production commands. [doc](https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/SecurityPillar.html)
- **[encryption at rest]** Enable at-rest encryption at cluster creation time and use AWS KMS customer-managed keys for stricter requirements — at-rest encryption cannot be added to an existing cluster later. [doc](https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/SecurityPillar.html)
- **[encryption in transit]** Enable in-transit (TLS) encryption at cluster creation and use clients that support TLS 1.2 or later (TLS 1.3+ recommended) with perfect-forward-secrecy cipher suites — protects data at its most vulnerable point, benchmark first since encryption adds processing overhead. [doc](https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/infrastructure-security.html)
- **[IAM]** Scope control-plane actions (create/modify/delete clusters) with least-privilege IAM identity-based policies rather than broad managed policies — keeps a clear line of accountability for who can change resources. [doc](https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/SecurityPillar.html)
- **[detection]** Monitor the `AuthenticationFailures`, `KeyAuthorizationFailures`, and `CommandAuthorizationFailures` CloudWatch metrics and alarm on them — surfaces unauthorized access attempts as they happen. [doc](https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/SecurityPillar.html)
- **[audit]** Log ElastiCache API calls with AWS CloudTrail and use the Redis/Valkey `ACL LOG` command — gives you the detail needed to investigate authorization failures. [doc](https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/SecurityPillar.html)

## 🛡️ Reliability
- **[high availability]** Use Multi-AZ with automatic failover for all production Valkey/Redis OSS replication groups (cluster mode enabled sets this up automatically) — Memcached has no native replication, so treat it as ephemeral-only. [doc](https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/ReliabilityPillar.html)
- **[high availability]** Deploy at least two replicas per shard spread across Availability Zones, and use a minimum of three shards for cluster-mode-enabled workloads — provides read scalability and lets the cluster maintain quorum during failover. [doc](https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/ReliabilityPillar.html)
- **[failover testing]** Validate client-side reconnection behavior using the ElastiCache Test Failover API before a real event — confirms your application recovers correctly ahead of an actual production failover. [doc](https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/ReliabilityPillar.html)
- **[backup/RPO]** Document your RPO, schedule automatic backups during low-usage windows against read replicas, and periodically test restores — avoids surprises about how much data loss is acceptable and confirms recovery actually works. [doc](https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/ReliabilityPillar.html)
- **[disaster recovery]** Use Multi-AZ across at least 3 AZs and enable Global Datastore for cross-region replication — protects against AZ failure locally and regional failure globally with low-latency cross-region reads. [doc](https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/ReliabilityPillar.html)
- **[monitoring]** Track `ReplicationLag`, `BytesUsedForCache`, and `DatabaseMemoryUsagePercentage` — surfaces replication health issues and memory pressure before they cause unresponsiveness. [doc](https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/ReliabilityPillar.html)
- **[node failures]** Distribute nodes across multiple Availability Zones and enable Multi-AZ so ElastiCache automatically fails over to a replica if a primary node fails — minimizes the impact of individual node or AZ failures. [doc](https://docs.aws.amazon.com/whitepapers/latest/reactive-systems-on-aws/resilient.html)
- **[durable data]** For data that cannot tolerate loss (transactional/purchase data), also store it in a durable database such as Amazon RDS or DynamoDB — Redis OSS replication is asynchronous, so some data loss can occur during failover. [doc](https://docs.aws.amazon.com/whitepapers/latest/scale-performance-elasticache/elasticache-for-redis.html)

## ⚡ Performance Efficiency
- **[monitoring]** Baseline performance with load tests against a subset of your real workload while watching CloudWatch metrics — gives you a reference point for detecting future regressions. [doc](https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/PerformanceEfficiencyPillar.html)
- **[query analysis]** Analyze the Slow Log (`slowlog get`) and benchmark with `redis-benchmark`/`valkey-benchmark` under representative load — pinpoints commands exceeding latency thresholds so you can optimize them. [doc](https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/PerformanceEfficiencyPillar.html)
- **[connection routing]** Connect clients to the configuration endpoint (cluster mode enabled) or the primary/reader endpoints (cluster mode disabled) rather than individual node endpoints — ensures correct load distribution and easier scaling. [doc](https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/PerformanceEfficiencyPillar.html)
- **[read scaling]** Offload read traffic to replica nodes via the reader endpoint or `READONLY` command — reduces CPU and network pressure on the primary node. [doc](https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/OperationalExcellencePillar.html)
- **[hot keys]** Detect hot keys (e.g. with `--hotkeys` under an LFU maxmemory policy) and replicate them across nodes or use server-assisted client-side caching — avoids overloading a single node with disproportionate access. [doc](https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/PerformanceEfficiencyPillar.html)
- **[cache effectiveness]** Track the cache-hit ratio (hits / (hits + misses)) continuously and investigate the cause of misses — a ratio close to 1 confirms your cache is actually reducing load on the backing store. [doc](https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/PerformanceEfficiencyPillar.html)
- **[sizing]** Choose node type and shard count based on whether the workload is memory-bound, CPU-bound, or network-bound, starting with a modest instance and scaling from there — avoids paying for capacity the workload doesn't need. [doc](https://aws.amazon.com/blogs/database/best-practices-for-sizing-your-amazon-elasticache-for-redis-clusters/)
- **[node families]** Prefer current-generation node families (e.g. M6g/R6g Graviton-based) over older generations — delivers more memory per vCPU, better networking bandwidth, and improved CPU performance. [doc](https://docs.aws.amazon.com/whitepapers/latest/scale-performance-elasticache/elasticache-for-memcached.html)
- **[auto scaling]** Use ElastiCache Auto Scaling (target tracking or scheduled policies on shards/replicas) instead of manually resizing — automatically matches capacity to CPU/memory utilization as load changes. [doc](https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/AutoScaling-Policies.html)

## 💰 Cost Optimization
- **[reserved nodes]** Purchase Reserved Nodes (No/Partial/All Upfront, 1- or 3-year terms) for steady-state baseline capacity once workload sizing is known — reservations are size-flexible within a node family and Region so you can still right-size later. [doc](https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/CacheNodes.Reserved.html)
- **[data tiering]** For datasets where only a subset (up to ~20%) is accessed regularly and added latency is tolerable, use data-tiering-capable node types (R6gd and later) — moves less-active data to SSD and lowers memory-only node costs. [doc](https://aws.amazon.com/elasticache/features/)
- **[cost tracking]** Apply cost allocation tags to ElastiCache resources and use AWS Cost Explorer — lets you track spend by node type, environment (dev/test/prod), and usage pattern. [doc](https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/CostOptimizationPillar.html)
- **[governance]** Restrict who can create ElastiCache resources with IAM roles/policies and establish a recurring review of node types, environments, and backup retention — keeps spend tied to accountable teams. [doc](https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/CostOptimizationPillar.html)
- **[right-sizing]** Continuously monitor utilization (CPU, memory, network) with CloudWatch — identifies over-provisioned clusters you can resize or scale in without compromising availability or performance targets. [doc](https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/CostOptimizationPillar.html)

## ⚙️ Operational Excellence
- **[alerting]** Configure Amazon SNS notifications for critical ElastiCache events (failover, node replacement, scaling, snapshot failures) — lets you respond or automate reactions instead of discovering issues after impact. [doc](https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/OperationalExcellencePillar.html)
- **[scaling operations]** Monitor CPU, memory, and network utilization to decide when and how to scale, offloading reads to replicas first — matches the environment to workload needs instead of over- or under-provisioning. [doc](https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/OperationalExcellencePillar.html)
- **[trend monitoring]** Review CloudWatch metrics over extended time ranges (data points retained up to 455 days) — surfaces gradual workload growth that a point-in-time check would miss. [doc](https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/OperationalExcellencePillar.html)
- **[infrastructure as code]** Manage ElastiCache resources through CloudFormation rather than manual console changes — preserves operational consistency and avoids configuration drift. [doc](https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/OperationalExcellencePillar.html)
- **[connection management]** Reuse existing client connections (connection pooling) instead of creating new ones per request — avoids the overhead of establishing new connections and reduces the risk of hitting per-node connection limits. [doc](https://aws.amazon.com/blogs/database/use-amazon-elasticache-for-redis-as-a-near-real-time-feature-store/)
- **[monitoring setup]** Set CloudWatch alarms on CPUUtilization, EngineCPUUtilization, SwapUsage, Evictions, CurrConnections, Memory, Network, Latency, and Replication — these are the metrics AWS recommends watching so you can act before performance degrades. [doc](https://docs.aws.amazon.com/AmazonElastiCache/latest/dg/CacheMetrics.WhichShouldIMonitor.html)
