# Amazon Aurora DSQL — Best Practices

## Common scenarios
- Always-available, multi-Region transactional applications        → Reliability, Security
- Serverless PostgreSQL-compatible OLTP with unpredictable/bursty traffic        → Performance Efficiency, Cost Optimization
- Multi-tenant SaaS backends needing strong consistency without failover management        → Reliability, Operational Excellence
- Migrating PostgreSQL workloads that need horizontal scale and zero-downtime maintenance        → Performance Efficiency, Operational Excellence

## 🔒 Security
- **[authentication]** Use IAM roles rather than long-term credentials to authenticate to Aurora DSQL, obtaining temporary access keys instead of storing static credentials in applications or on EC2 instances. [doc](https://docs.aws.amazon.com/aurora-dsql/latest/userguide/best-practices-security-preventative.html)
- **[authorization]** Attach least-privilege IAM policies scoped to specific Aurora DSQL API operations and resources, and avoid using the database `admin` role for everyday operations — create custom database roles instead. [doc](https://docs.aws.amazon.com/aurora-dsql/latest/userguide/best-practices-security-preventative.html)
- **[connection security]** Configure clients with `sslmode=verify-full` in production so the server certificate chain and hostname are validated, not just that TLS is used. [doc](https://docs.aws.amazon.com/aurora-dsql/latest/userguide/best-practices-security-preventative.html)
- **[client hygiene]** Keep PostgreSQL client libraries up to date (AWS recommends PostgreSQL 17) to pick up security fixes and compatibility improvements. [doc](https://docs.aws.amazon.com/aurora-dsql/latest/userguide/best-practices-security-preventative.html)
- **[network access]** Prefer AWS PrivateLink VPC interface endpoints over the public endpoint to keep cluster traffic within your VPC, and attach a VPC endpoint policy so only intended principals can use the endpoint. [doc](https://aws.amazon.com/blogs/database/amazon-aurora-dsql-connections-drivers-strings-and-best-practices/)
- **[network egress]** Restrict outbound traffic from application hosts with security group egress rules, VPC network ACLs, and VPC Flow Logs, since a compromised application without egress controls could exfiltrate data. [doc](https://aws.amazon.com/blogs/database/amazon-aurora-dsql-connections-drivers-strings-and-best-practices/)
- **[schema permissions]** Manage authorization through schema-level grants — have admins create schemas and grant `USAGE` to specific roles rather than letting non-admin users operate in the public schema. [doc](https://docs.aws.amazon.com/aurora-dsql/latest/userguide/considerations.html)
- **[encryption]** Use a customer managed AWS KMS key instead of the AWS owned key when you need direct control over key rotation, access policy, and auditing of Aurora DSQL cluster encryption. [doc](https://docs.aws.amazon.com/aurora-dsql/latest/userguide/data-encryption.html)
- **[monitoring]** Create CloudWatch alarms on key Aurora DSQL metrics and route them to Amazon SNS so sustained threshold breaches trigger notification or automated response. [doc](https://docs.aws.amazon.com/aurora-dsql/latest/userguide/best-practices-security-detective.html)
- **[auditing]** Enable AWS CloudTrail to capture Aurora DSQL API calls, including caller identity and source IP, for security investigation and compliance. [doc](https://docs.aws.amazon.com/aurora-dsql/latest/userguide/monitoring-overview.html)
- **[resource tagging]** Tag Aurora DSQL clusters by security, confidentiality, and environment attributes to support grouped access controls and automation. [doc](https://docs.aws.amazon.com/aurora-dsql/latest/userguide/best-practices-security-detective.html)

## 🛡️ Reliability
- **[connection lifecycle]** Design clients to handle the fixed 60-minute maximum connection lifetime — Aurora DSQL closes connections regardless of activity so the application must reconnect and refresh IAM tokens gracefully. [doc](https://aws.amazon.com/blogs/database/amazon-aurora-dsql-connections-drivers-strings-and-best-practices/)
- **[retry logic]** Implement retry with exponential backoff and jitter for both `SerializationFailure` (OCC conflicts) and transient `OperationalError` conditions, since transient connection failures are a normal part of operating against a distributed system. [doc](https://aws.amazon.com/blogs/database/amazon-aurora-dsql-connections-drivers-strings-and-best-practices/)
- **[concurrency conflicts]** Treat OCC commit-time conflicts (`OC000`) as expected and retry the transaction, rather than treating them as application errors. [doc](https://docs.aws.amazon.com/aurora-dsql/latest/userguide/working-with-concurrency-control.html)
- **[schema changes]** Expect and retry on `OC001` catalog-version conflicts after DDL changes — retry from the same session so it refreshes its cached catalog version. [doc](https://docs.aws.amazon.com/aurora-dsql/latest/userguide/troubleshooting.html)
- **[multi-Region design]** Use a multi-Region cluster with two strongly consistent regional endpoints for workloads that require the highest resiliency, since both endpoints support concurrent, zero-replication-lag reads and writes. [doc](https://docs.aws.amazon.com/aurora-dsql/latest/userguide/disaster-recovery-resiliency.html)
- **[backup strategy]** Protect clusters with AWS Backup, using continuous backups for point-in-time recovery and scheduled backup plans for longer retention. [doc](https://docs.aws.amazon.com/aurora-dsql/latest/userguide/backup-aurora-dsql.html)
- **[multi-Region backup]** Create a cross-Region copy of your backup in the peer Region ahead of time, since a multi-Region cluster restore requires an identical recovery point already present in that Region. [doc](https://docs.aws.amazon.com/aws-backup/latest/devguide/restore-auroradsql.html)
- **[connectivity failures]** On rare multi-Region linked-cluster recovery events, retry the transaction or reconnect rather than treating a transient error as a hard failure, since typically only a fraction of the workload is affected. [doc](https://docs.aws.amazon.com/aurora-dsql/latest/userguide/considerations.html)

## ⚡ Performance Efficiency
- **[primary key design]** Choose primary keys with high cardinality (for example, random UUIDs or composite keys) and avoid hot key writes and hot key ranges, since Aurora DSQL's optimistic concurrency model causes concentrated writes to retry. [doc](https://aws.amazon.com/blogs/database/everything-you-dont-need-to-know-about-amazon-aurora-dsql-part-2-shallow-view/)
- **[key selection]** Prefer natural keys matching your dominant access pattern over synthetic surrogate keys when uniqueness and stability allow it, because the primary key is the table's full covering index in Aurora DSQL's storage model. [doc](https://aws.amazon.com/blogs/database/improve-query-performance-with-explain-plans-in-amazon-aurora-dsql/)
- **[index creation]** Use `CREATE INDEX ASYNC` instead of `CREATE INDEX` for non-blocking, zero-downtime index builds on existing tables. [doc](https://docs.aws.amazon.com/aurora-dsql/latest/userguide/working-with-create-index-async.html)
- **[index design]** Rely on the automatically created index backing a `UNIQUE` or `PRIMARY KEY` constraint instead of also creating an explicit secondary index on the same column, to avoid doubling write cost and consuming index quota for no benefit. [doc](https://aws.amazon.com/blogs/database/user-authentication-and-session-management-with-amazon-aurora-dsql/)
- **[connection pooling]** Use connection pooling with a `max_lifetime` set below Aurora DSQL's one-hour connection limit so the pool proactively recycles connections instead of surfacing timeout errors. [doc](https://docs.aws.amazon.com/aurora-dsql/latest/userguide/SECTION_program-with-dsql-connector-for-python.html)
- **[large tables]** Query the system catalog for row counts on large tables instead of running `COUNT(*)`, which can be significantly slower at scale. [doc](https://docs.aws.amazon.com/aurora-dsql/latest/userguide/considerations.html)
- **[query tuning]** Use `EXPLAIN ANALYZE VERBOSE` to compare Total DPU before and after changes (adding an index, restructuring a query) to quantify the performance and cost impact of an optimization. [doc](https://aws.amazon.com/blogs/database/improve-query-performance-with-explain-plans-in-amazon-aurora-dsql/)
- **[client networking]** Configure clients on IPv4-only hosts to use IPv4 explicitly, since dual-stack clients that fall back to IPv6 after IPv4 throttling can return `NetworkUnreachable` errors. [doc](https://docs.aws.amazon.com/aurora-dsql/latest/userguide/considerations.html)

## 💰 Cost Optimization
- **[query efficiency]** Reduce DPU consumption by optimizing queries (adding targeted indexes, restructuring joins) since Aurora DSQL bills for compute, read, and write DPUs directly tied to work performed. [doc](https://aws.amazon.com/blogs/database/improve-query-performance-with-explain-plans-in-amazon-aurora-dsql/)
- **[idle workloads]** Take advantage of automatic scale-to-zero behavior for intermittent workloads, since Aurora DSQL incurs no DPU charges while a cluster is idle. [doc](https://aws.amazon.com/rds/aurora/dsql/pricing/)
- **[cost monitoring]** Monitor the ComputeDPU, ReadDPU, WriteDPU, StreamDPU, and MultiRegionWriteDPU CloudWatch metrics to attribute spend to specific activity types and catch inefficient patterns early. [doc](https://docs.aws.amazon.com/aurora-dsql/latest/userguide/billing-metering.html)
- **[multi-Region cost]** Account for the extra DPU charge on replicated writes when sizing a multi-Region cluster, since each peer Region incurs additional MultiRegion Write DPU cost proportional to originating writes. [doc](https://docs.aws.amazon.com/aurora-dsql/latest/userguide/billing-metering.html)
- **[commitment discounts]** Apply Database Savings Plans to consistent baseline Aurora DSQL usage to reduce costs when usage is predictable over a committed term. [doc](https://aws.amazon.com/rds/aurora/dsql/faqs/)

## ⚙️ Operational Excellence
- **[connection management]** Use the Aurora DSQL Connectors for your language/driver so IAM token generation and refresh are handled automatically rather than through custom token-management code. [doc](https://aws.amazon.com/blogs/database/amazon-aurora-dsql-connections-drivers-strings-and-best-practices/)
- **[DDL/DML separation]** Structure deployments so each transaction contains only one DDL statement and never mixes DDL with DML, since Aurora DSQL rejects transactions that combine both. [doc](https://docs.aws.amazon.com/aurora-dsql/latest/userguide/working-with-ddl.html)
- **[permission propagation]** After granting or revoking schema permissions, establish a new connection to pick up the change immediately, since existing connections may not see the update for up to the one-hour connection lifetime. [doc](https://docs.aws.amazon.com/aurora-dsql/latest/userguide/considerations.html)
- **[application-layer integrity]** Implement referential integrity checks in the application layer with consistent validation and transaction boundaries, since this pattern avoids the performance bottlenecks of cascading operations in a distributed database. [doc](https://docs.aws.amazon.com/aurora-dsql/latest/userguide/working-with-postgresql-compatibility-migration-guide.html)
- **[monitoring]** Monitor Aurora DSQL with Amazon CloudWatch metrics and set alarms on thresholds like commit latency, bytes read, and bytes written to catch issues before they affect availability. [doc](https://docs.aws.amazon.com/aurora-dsql/latest/userguide/monitoring-overview.html)
- **[storage reclamation]** Expect a delay before storage limit calculations reflect space freed by `DROP TABLE`, and request a cluster storage quota increase proactively if you need additional capacity in the interim. [doc](https://docs.aws.amazon.com/aurora-dsql/latest/userguide/considerations.html)
- **[backup automation]** Manage Aurora DSQL backup schedules, retention, and cross-Region/cross-account copies centrally through AWS Backup rather than ad hoc processes, since Aurora DSQL backup/restore operations are only available through AWS Backup. [doc](https://docs.aws.amazon.com/aws-backup/latest/devguide/restore-auroradsql.html)
