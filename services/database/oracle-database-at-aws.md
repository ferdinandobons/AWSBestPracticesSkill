# Oracle Database@AWS — Best Practices

## Common scenarios
- Migrating on-premises Oracle Exadata workloads to AWS with minimal changes        → Reliability, Performance Efficiency
- Running mission-critical Oracle databases needing HA/DR across AZs and Regions        → Reliability
- Low-latency integration between Oracle databases and AWS-hosted applications        → Performance Efficiency
- Unifying Oracle transactional data with AWS analytics, ML, and generative AI services        → Cost Optimization, Operational Excellence

## 🔒 Security
- **[encryption]** Rely on Transparent Data Encryption (TDE) for all databases and backups — it is mandatory and enabled by default for Autonomous Database and Exadata Database, so data files and every backup destination stay encrypted automatically. [doc](https://aws.amazon.com/blogs/database/navigating-backup-and-recovery-options-for-oracle-databaseaws/)
- **[key management]** Choose customer-managed keys (AWS KMS for ExaDB-D, or OCI Vault/Oracle Key Vault) instead of Oracle-managed keys — this gives you direct control over key rotation, disabling, and deletion for compliance and auditability. [doc](https://aws.amazon.com/blogs/database/navigating-backup-and-recovery-options-for-oracle-databaseaws/)
- **[IAM policies]** Start from AWS managed policies and move toward least-privilege customer-managed policies — this scopes access down to the specific Oracle Database@AWS actions and resources each user actually needs. [doc](https://docs.aws.amazon.com/odb/latest/UserGuide/security_iam_id-based-policy-examples.html)
- **[IAM conditions]** Add IAM policy conditions, such as requiring SSL/TLS or restricting access to a specific AWS service — this further narrows when and how Oracle Database@AWS actions can be performed. [doc](https://docs.aws.amazon.com/odb/latest/UserGuide/security_iam_id-based-policy-examples.html)
- **[policy validation]** Validate identity-based policies with IAM Access Analyzer before attaching them — it flags overly permissive grants and confirms policies follow IAM best practices. [doc](https://docs.aws.amazon.com/odb/latest/UserGuide/security_iam_id-based-policy-examples.html)
- **[MFA]** Require multi-factor authentication for IAM users and root accounts that can manage Oracle Database@AWS resources — these actions can incur cost and expose sensitive data, so a single factor isn't enough protection. [doc](https://docs.aws.amazon.com/odb/latest/UserGuide/security_iam_id-based-policy-examples.html)
- **[network isolation]** Use peered CIDRs as a network-level access control list on the ODB network, scoped to specific subnets rather than entire VPC ranges — this limits which resources can reach your Oracle databases. [doc](https://aws.amazon.com/blogs/database/implement-network-connectivity-patterns-for-oracle-databaseaws/)
- **[audit logging]** Enable AWS CloudTrail for Oracle Database@AWS API activity — it records who made changes, from where, and when, which is essential for security investigations. [doc](https://docs.aws.amazon.com/odb/latest/UserGuide/monitoring-overview.html)
- **[backup immutability]** Apply protection policies with retention lock on backups stored in Autonomous Recovery Service — this prevents early deletion of backups, even by a privileged user, for the duration of the retention period. [doc](https://aws.amazon.com/blogs/database/navigating-backup-and-recovery-options-for-oracle-databaseaws/)

## 🛡️ Reliability
- **[high availability]** Use Oracle Data Guard cross-AZ configurations between ExaDB-D or ADB-D VM clusters — this protects against local hardware and Availability Zone failures, following Oracle Maximum Availability Architecture (MAA) guidance. [doc](https://aws.amazon.com/blogs/database/well-architected-design-for-resiliency-with-oracle-databaseaws/)
- **[disaster recovery]** Add Oracle Data Guard cross-Region configurations on top of cross-AZ configurations — this creates defense-in-depth protection that also covers AWS Region-wide outages. [doc](https://aws.amazon.com/blogs/database/well-architected-design-for-resiliency-with-oracle-databaseaws/)
- **[network design]** Choose Transit Gateway or VCN peering deliberately for Data Guard connectivity between ODB networks — the right choice differs in cost and topology between cross-AZ and cross-Region deployments. [doc](https://aws.amazon.com/blogs/database/well-architected-design-for-resiliency-with-oracle-databaseaws/)
- **[backups]** Configure automated backups for production environments rather than relying only on manual or user-managed backups — this improves operational efficiency and consistency. [doc](https://aws.amazon.com/blogs/database/navigating-backup-and-recovery-options-for-oracle-databaseaws/)
- **[backup scheduling]** Schedule ExaDB-D backup windows during non-business hours — this minimizes the performance impact of backup operations on production workloads. [doc](https://aws.amazon.com/blogs/database/navigating-backup-and-recovery-options-for-oracle-databaseaws/)
- **[near-zero RPO]** Use Autonomous Recovery Service real-time data protection for critical databases — it continuously transfers transactions and redo logs, reducing potential data loss to under one second. [doc](https://aws.amazon.com/blogs/database/navigating-backup-and-recovery-options-for-oracle-databaseaws/)
- **[cross-region DR]** Enable cross-region backups to OCI Object Storage for Autonomous AI Databases — this provides a low-cost disaster recovery mechanism. [doc](https://aws.amazon.com/blogs/database/navigating-backup-and-recovery-options-for-oracle-databaseaws/)
- **[retention review]** Periodically review and adjust backup retention periods — business and compliance requirements change, so initial defaults can become stale. [doc](https://aws.amazon.com/blogs/database/navigating-backup-and-recovery-options-for-oracle-databaseaws/)
- **[monitoring]** Set Amazon CloudWatch alarms on Oracle Database@AWS metrics — this detects degraded reliability and availability conditions and triggers timely notifications or automated responses. [doc](https://docs.aws.amazon.com/odb/latest/UserGuide/monitoring-overview.html)

## ⚡ Performance Efficiency
- **[network connectivity]** Establish ODB peering between your Amazon VPC and the ODB network — this gives application servers on AWS low-latency, near-local access to Oracle Exadata databases. [doc](https://docs.aws.amazon.com/odb/latest/UserGuide/what-is-odb.html)
- **[connectivity pattern selection]** Use direct VPC-to-ODB-network peering for single-VPC application patterns, then move to a Transit Gateway or Cloud WAN transit-VPC pattern as you scale — this keeps connectivity simple until multi-VPC or multi-Region access is actually needed. [doc](https://aws.amazon.com/blogs/database/implement-network-connectivity-patterns-for-oracle-databaseaws/)
- **[VPC Lattice access]** Use Amazon VPC Lattice for direct connectivity from your VPC to individual VMs in the ODB network and to AWS services such as Amazon S3 — this avoids extra network hops that add latency. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/oracle-databaseaws-network-connectivity-using-amazon-vpc-lattice/)
- **[DNS configuration]** Configure Route 53 Resolver rules for Oracle domain name resolution — this ensures application traffic reliably resolves and reaches the ODB network without manual host-file management. [doc](https://aws.amazon.com/blogs/database/implement-network-connectivity-patterns-for-oracle-databaseaws/)
- **[data unification]** Use zero-ETL integrations, such as with Amazon Redshift, to unify Oracle transactional data with AWS analytics and ML services — this enables near real-time analysis without building and maintaining custom pipelines. [doc](https://docs.aws.amazon.com/odb/latest/UserGuide/what-is-odb.html)

## 💰 Cost Optimization
- **[backup destination choice]** Choose Amazon S3, OCI Object Storage, or Autonomous Recovery Service for backups based on required durability, RPO, and cost — defaulting to one option for every workload can overspend or under-protect. [doc](https://aws.amazon.com/blogs/database/navigating-backup-and-recovery-options-for-oracle-databaseaws/)
- **[standby backup deduplication]** Disable backups on the Data Guard standby instance and back up only the primary, keeping the same backup destination on both — this avoids paying for duplicate backup storage. [doc](https://aws.amazon.com/blogs/database/navigating-backup-and-recovery-options-for-oracle-databaseaws/)
- **[backup cleanup]** Periodically review manual and user-managed backups on ExaDB-D and delete those no longer needed — this avoids paying for unnecessary retained storage. [doc](https://aws.amazon.com/blogs/database/navigating-backup-and-recovery-options-for-oracle-databaseaws/)
- **[non-production refresh]** Use the built-in clone feature (full or metadata clone) to refresh Autonomous AI Database non-production environments — this avoids provisioning and backing up separate full databases just for testing. [doc](https://aws.amazon.com/blogs/database/navigating-backup-and-recovery-options-for-oracle-databaseaws/)
- **[commercial benefits]** Consolidate Oracle Database@AWS usage onto AWS Marketplace billing — usage then counts toward existing AWS commitments and Oracle Support Rewards. [doc](https://docs.aws.amazon.com/odb/latest/UserGuide/what-is-odb.html)

## ⚙️ Operational Excellence
- **[monitoring integration]** Use Amazon CloudWatch, which receives Oracle Exadata metric data from OCI — this lets you build dashboards and alarms for Oracle Database@AWS alongside the rest of your AWS resources. [doc](https://docs.aws.amazon.com/odb/latest/UserGuide/monitoring-cloudwatch.html)
- **[event automation]** Use Amazon EventBridge to respond automatically to Oracle Database@AWS events, such as availability issues or resource state changes — this removes reliance on manual checks. [doc](https://docs.aws.amazon.com/odb/latest/UserGuide/monitoring-overview.html)
- **[unified tooling]** Provision and manage Oracle Database@AWS resources with familiar AWS APIs, CLI, or SDKs, which call the corresponding OCI APIs — this keeps operations consistent with the rest of your AWS estate. [doc](https://docs.aws.amazon.com/odb/latest/UserGuide/what-is-odb.html)
- **[infrastructure as code]** Provision the ODB network, Exadata infrastructure, VM clusters, and Autonomous VM clusters with Terraform or AWS CloudFormation — this makes environments repeatable, reviewable, and consistent. [doc](https://aws.amazon.com/blogs/database/provision-oracle-databaseaws-resources-using-terraform/)
