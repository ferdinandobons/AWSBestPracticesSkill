# Amazon Neptune Analytics — Best Practices

## Common scenarios
- Ephemeral graph analytics (fraud detection, social network analysis)        → Performance Efficiency, Cost Optimization
- Running graph algorithms (PageRank, WCC, centrality, similarity) on data loaded from S3 or Neptune Database        → Performance Efficiency, Reliability
- Vector similarity search / GraphRAG for generative AI applications        → Performance Efficiency, Security
- Combining Neptune Database (OLTP) with Neptune Analytics (OLAP) pipelines        → Cost Optimization, Operational Excellence

## 🔒 Security
- **[data protection]** Encrypt the graph at rest with the default key or a customer-managed AWS KMS key, and encrypt snapshots and S3 export data — protects sensitive graph data from unauthorized access. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/security-pillar.html)
- **[data protection]** Never put confidential or sensitive information in graph names, tags, or free-form metadata fields — this data can surface in billing or diagnostic logs. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/security-pillar.html)
- **[query safety]** Use parameterized queries instead of string concatenation with user-supplied input in openCypher — prevents injection-style attacks. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/security-pillar.html)
- **[network security]** Keep public connectivity disabled unless required, and require IAM authentication for callers to enforce least-privilege access to graph operations. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/security-pillar.html)
- **[network security]** Create private graph endpoints scoped to specific VPCs, subnets, and security groups when applications must reach the graph privately. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/security-pillar.html)
- **[data in transit]** Rely on Neptune Analytics' enforced SSL/HTTPS connections and use TLS 1.2 or later with cipher suites supporting perfect forward secrecy for all clients. [doc](https://docs.aws.amazon.com/neptune-analytics/latest/userguide/infrastructure-security.html)
- **[IAM]** Grant callers only the read, write, or delete permissions on specific graphs that their queries require, and use `aws:ResourceTag` condition keys to further restrict access by tag. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/security-pillar.html)
- **[IAM]** Use an AWS SDK (for example, the Boto3 Neptune Graph client) so Signature Version 4 request signing is handled for you rather than implemented manually. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/security-pillar.html)
- **[data loading]** Scope the IAM role used for bulk import or batch load to only read/list the specific S3 prefix staging the input data, with the correct trust relationship for Neptune Analytics to assume it. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/security-pillar.html)

## 🛡️ Reliability
- **[deployment planning]** Decide upfront on seeding source, capacity, replica count, connectivity, and backup strategy before creating a graph — these decisions are hard to change after the fact. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/reliability-pillar.html)
- **[capacity]** Specify maximum provisioned memory at graph creation time to bound graph size, and adjust provisioned capacity later if requirements change. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/reliability-pillar.html)
- **[availability]** Provision a replica as a warm standby when fast recovery from graph failure matters — a graph with a replica recovers faster than one without. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/reliability-pillar.html)
- **[failure handling]** Implement a retry-with-backoff pattern in client applications to handle "service unavailable" errors that occur for in-flight requests during failover. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/reliability-pillar.html)
- **[backup and recovery]** Take snapshots on a schedule matched to how often the graph changes, since Neptune Analytics does not take automatic snapshots, and delete older snapshots you no longer need for point-in-time restoration. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/reliability-pillar.html)
- **[disaster recovery]** Share and restore snapshots across accounts or AWS Regions when your recovery time and recovery point objectives require cross-Region DR. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/reliability-pillar.html)
- **[quotas]** Review Service Quotas for the number of graphs/snapshots, maximum provisioned memory, and API request rates, and reduce active graph count (by combining graphs or deleting ephemeral ones) if you approach quota limits. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/reliability-pillar.html)
- **[scaling]** Monitor `NumQueuedRequestsPerSec`, `GraphStorageUsagePercent`, `GraphSizeBytes`, and `CPUUtilization` in CloudWatch to decide when to vertically scale m-NCU capacity. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/reliability-pillar.html)

## ⚡ Performance Efficiency
- **[graph modeling]** Reshape transactional graph models for analytics workloads — trim inessential node/edge types so algorithms like weakly connected components traverse fewer edges and a smaller diameter. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/performance-efficiency-pillar.html)
- **[query optimization]** Use parameterized queries so the query plan cache (enabled by default) can reuse prepared plans for queries completing in 100ms or less. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/performance-efficiency-pillar.html)
- **[query optimization]** Run an explain plan on slow queries to identify bottlenecks before tuning. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/performance-efficiency-pillar.html)
- **[query optimization]** Follow documented openCypher best practices — flatten maps in UNWIND clauses, specify edge labels explicitly, place restrictive filters early, and avoid unnecessary WITH clauses or named paths. [doc](https://docs.aws.amazon.com/neptune-analytics/latest/userguide/bestPractices.html)
- **[graph algorithms]** Narrow the input node set with a `MATCH` clause and filter on node/edge labels before invoking a graph algorithm, and use bounding parameters such as `maxDepth` to limit result size. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/performance-efficiency-pillar.html)
- **[graph algorithms]** Tune the `concurrency` parameter and test both parallel and single-threaded execution — single-threaded can be faster for smaller inputs where parallelism adds overhead without benefit. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/performance-efficiency-pillar.html)
- **[vector search]** Choose the smallest embedding dimension that still yields accurate similarity results to speed up both search and load. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/performance-efficiency-pillar.html)
- **[writes]** Use bulk import for source data over 50 GB and batch load for smaller datasets, reserving transactional openCypher create/update/delete operations for small incremental changes only. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/performance-efficiency-pillar.html)
- **[writes]** Load vector embeddings up front using bulk import or batch load with concurrency set to 1, rather than inserting them through transactional queries. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/performance-efficiency-pillar.html)
- **[writes]** Use mutate algorithm variants (for example, degree mutate) to persist computed results as properties so you avoid recomputation when the underlying graph hasn't changed. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/performance-efficiency-pillar.html)
- **[writes]** Use the graph reset administrative action instead of an openCypher drop query to clear a large graph — reset completes in roughly constant time regardless of data size. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/performance-efficiency-pillar.html)
- **[right-sizing]** Monitor `NumQueuedRequestsPerSec`, `NumOpenCypherRequestsPerSec`, `GraphStorageUsagePercent`, `GraphSizeBytes`, and `CPUUtilization` to determine whether provisioned m-NCU capacity matches actual graph size and load. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/performance-efficiency-pillar.html)

## 💰 Cost Optimization
- **[workload fit]** Confirm your workload genuinely needs multi-hop graph traversal before adopting Neptune Analytics — single-hop or blob-heavy data patterns are often cheaper modeled elsewhere. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/cost-optimization-pillar.html)
- **[data loading]** Prefer bulk import or batch load from data already staged in Amazon S3 over populating from a Neptune Database cluster/snapshot, since the latter incurs extra clone, restore, and S3 export costs. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/cost-optimization-pillar.html)
- **[capacity]** Bound minimum and maximum provisioned capacity at graph creation and adjust it later rather than over-provisioning up front. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/cost-optimization-pillar.html)
- **[lifecycle]** Snapshot and delete graphs that are no longer needed, then restore from snapshot when required again — analytics workloads are frequently ephemeral. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/cost-optimization-pillar.html)
- **[lifecycle]** Stop a graph instead of deleting it when it is temporarily idle — a stopped graph preserves all data and configuration while billing at a reduced rate. [doc](https://docs.aws.amazon.com/neptune-analytics/latest/userguide/managing-stopping.html)
- **[replicas]** Minimize the replica count (0, 1, or 2) to match your actual availability requirement, since each replica incurs the full m-NCU rate of the primary graph. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/cost-optimization-pillar.html)
- **[monitoring]** Track Neptune Analytics spend in AWS Billing and Cost Management and correlate it with CloudWatch metrics to confirm provisioned capacity isn't oversized for observed usage. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/cost-optimization-pillar.html)
- **[notebooks]** Share a single Neptune notebook instance across users with separate folders, and shut it down when not in use to avoid idle compute charges. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/cost-optimization-pillar.html)

## ⚙️ Operational Excellence
- **[IaC]** Provision Neptune Analytics graphs, private endpoints, and notebook instances through AWS CloudFormation for consistent, repeatable environment configuration. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/operational-excellence-pillar.html)
- **[automation]** Automate resizing, snapshotting, deleting, restoring, and resetting graphs through the Neptune Analytics API via the AWS CLI or SDKs rather than performing these as manual console actions. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/operational-excellence-pillar.html)
- **[configuration]** Store connection strings externally (for example, in AWS Secrets Manager or Amazon DynamoDB) so they can be updated dynamically without redeploying clients. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/operational-excellence-pillar.html)
- **[organization]** Tag Neptune Analytics resources to organize them, track cost allocation, and drive IAM access conditions based on resource tags. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/operational-excellence-pillar.html)
- **[environment separation]** Maintain separate graphs for development, test, and production, and separate graphs for user groups with distinct SLAs or usage patterns. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/operational-excellence-pillar.html)
- **[change management]** Store IaC templates in source control (never AWS credentials) and deploy through a CI/CD pipeline that tests changes in a non-production graph before promoting to production. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/operational-excellence-pillar.html)
- **[observability]** Build CloudWatch dashboards and alarms on `NumQueuedRequestsPerSec`, `NumOpenCypherRequestsPerSec`, `GraphStorageUsagePercent`, `GraphSizeBytes`, and `CPUUtilization` to catch capacity issues before they affect users. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/operational-excellence-pillar.html)
- **[audit logging]** Enable AWS CloudTrail logging of Neptune Analytics control-plane API calls, and optionally data-plane query executions, to monitor for unauthorized or anomalous activity. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/operational-excellence-pillar.html)

## 🌱 Sustainability
- **[ephemeral workloads]** Snapshot and delete graphs once analytics runs are complete instead of leaving them provisioned indefinitely, restoring from the snapshot only when needed again. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/sustainability-pillar.html)
- **[scheduling]** For flexible, non-urgent ephemeral workloads, run analytics during off-peak electricity demand periods for your Region where possible. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/sustainability-pillar.html)
- **[snapshot hygiene]** Skip re-snapshotting read-only or unchanged data before deletion to avoid redundant storage and processing. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/sustainability-pillar.html)
- **[idle resources]** Stop Neptune notebooks when they are not in use to avoid unnecessary resource consumption. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/sustainability-pillar.html)
- **[right-sizing]** Monitor CloudWatch metrics such as `GraphStorageUsagePercent` and `CPUUtilization` to identify oversized graphs that could run on a smaller, more efficient capacity. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/neptune-analytics-well-architected-framework/sustainability-pillar.html)
