# AWS Security Incident Response — Best Practices

## Common scenarios
- Centralizing detection, triage, and 24/7 expert response for GuardDuty and Security Hub findings across an organization        → Security, Operational Excellence
- Establishing an incident response team, escalation paths, and case management for security events        → Security, Operational Excellence
- Granting AWS experts scoped access to investigate and contain active security incidents on your behalf        → Security, Reliability
- Integrating incident data with SIEM, ITSM (Jira, ServiceNow), and automation tooling via EventBridge        → Operational Excellence, Security

## 🔒 Security
- **[delegated administrator]** Designate a dedicated security tooling account (not the AWS Organizations management account) as the delegated administrator for Security Incident Response — follows the principle of least privilege and keeps incident response administration separate from organization management. [doc](https://docs.aws.amazon.com/security-ir/latest/userguide/considerations_important.html)
- **[least privilege]** Start with AWS managed policies like `AWSSecurityIncidentResponseCaseFullAccess` and `AWSSecurityIncidentResponseReadOnlyAccess` for development and testing, then move to custom IAM policies scoped to specific case actions (such as `CreateCaseComment` or `GetCase`) for production — ensures team members only get the access level their role requires. [doc](https://aws.amazon.com/blogs/security/aws-security-incident-response-the-customers-journey-to-accelerating-the-incident-response-lifecycle/)
- **[policy hygiene]** Apply least-privilege permissions in identity-based policies, add IAM policy conditions (such as requiring SSL) to further restrict access, and validate policies with IAM Access Analyzer — reduces the risk of overly broad access to case and investigation data. [doc](https://docs.aws.amazon.com/security-ir/latest/userguide/policy-best-practices.html)
- **[MFA]** Require multi-factor authentication for IAM users or roles that can access Security Incident Response, especially those with case write access — adds a control against compromised credentials being used to tamper with active investigations. [doc](https://docs.aws.amazon.com/security-ir/latest/userguide/policy-best-practices.html)
- **[pre-provisioned access]** Provision least-privilege, incident-response-specific access to AWS accounts in advance of an event, using approval flows or a credential vault, and document and test that access before it's needed — avoids delays caused by requesting emergency access under pressure. [doc](https://docs.aws.amazon.com/whitepapers/latest/aws-security-incident-response-guide/prepare-access-to-aws-accounts.html)
- **[privileged credential hygiene]** Restrict and monitor use of any privileged credentials created specifically for incident response, and never use them for daily activities — limits the blast radius if these high-privilege credentials are misused or compromised. [doc](https://docs.aws.amazon.com/whitepapers/latest/aws-security-incident-response-guide/prepare-access-to-aws-accounts.html)
- **[tagging]** Do not store personally identifiable information (PII) or other confidential/sensitive data in Security Incident Response tags — tags are used for administrative purposes only and are not intended for private or sensitive data. [doc](https://docs.aws.amazon.com/security-ir/latest/userguide/aws-managed-policies.html)
- **[containment authorization]** Explicitly decide, in advance, whether to grant AWS Security Incident Response engineers permission to take containment actions on your behalf, and align that decision with your escalation framework — ensures containment authority matches your organization's risk tolerance rather than being decided ad hoc during an active event. [doc](https://docs.aws.amazon.com/security-ir/latest/userguide/monitoring-and-investigation.html)
- **[findings sources]** Enable Amazon GuardDuty and AWS Security Hub across your organization so Security Incident Response can ingest and triage the widest possible set of security findings — the service's automated triage and escalation quality depends directly on the breadth of finding sources feeding it. [doc](https://aws.amazon.com/security-incident-response/faqs/)

## 🛡️ Reliability
- **[organization-wide coverage]** Enable AWS Security Incident Response for your full AWS Organization (or targeted OUs) so coverage automatically adjusts as accounts are added or removed — eliminates manual per-account onboarding and prevents newly created accounts from being left unprotected. [doc](https://docs.aws.amazon.com/security-ir/latest/userguide/deploy-configure.html)
- **[single delegated administrator]** Use the same delegated administrator account across all Regions — you cannot designate different delegated administrators per Region — and register accounts and OUs through AWS Organizations so coverage stays consistent as your organization evolves. [doc](https://docs.aws.amazon.com/security-ir/latest/userguide/considerations_important.html)
- **[membership team size]** Define at least two incident response team members (primary and secondary contacts) when creating your membership, with clear communication preferences — ensures notifications and escalations reach someone even if one contact is unavailable. [doc](https://docs.aws.amazon.com/security-ir/latest/userguide/enable-sir-using-cli.html)
- **[account lifecycle awareness]** Understand that removing the delegated administrator account cancels the subscription and removes coverage for all associated member accounts immediately — plan any delegated administrator account changes carefully to avoid an unplanned coverage gap. [doc](https://docs.aws.amazon.com/security-ir/latest/userguide/considerations_important.html)
- **[escalation framework]** Develop a clear escalation framework that defines thresholds for self-managed, AWS-supported, and proactive cases, and who has authority to determine case type based on severity and impact — ensures incidents are routed and escalated consistently instead of relying on ad hoc judgment during an event. [doc](https://aws.amazon.com/blogs/security/optimize-security-operations-with-aws-security-incident-response/)

## ⚙️ Operational Excellence
- **[incident response plan]** Define and maintain an incident response plan, and build runbooks and playbooks for common scenarios, before an event occurs — the Well-Architected Framework identifies preparation, operations, and post-incident activity as the three foundations of a successful cloud incident response program. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/security-controls-by-caf-capability/incident-response-recommendations.html)
- **[playbooks as code]** Implement playbooks as code (scripts invoked via AWS Systems Manager Automation, Lambda, or EventBridge) rather than relying solely on manual written procedures — ensures consistent, repeatable investigation steps and reduces errors from manual execution during high-pressure events. [doc](https://docs.aws.amazon.com/wellarchitected/2023-10-03/framework/rel_testing_resiliency_playbook_resiliency.html)
- **[EventBridge automation]** Use Amazon EventBridge rules or pipes to react to Security Incident Response case and membership events, routing them to ITSM tools (Jira, ServiceNow), paging systems, or analytics pipelines — keeps your existing operational tooling in sync with incident state without manual handoffs. [doc](https://docs.aws.amazon.com/security-ir/latest/userguide/eventbridge.html)
- **[notification review]** Regularly review the email distribution list or alias configured for Security Incident Response notifications, and plan to extend notification mechanisms (PagerDuty, Jira, and others) via EventBridge as your operational tooling matures — prevents alerts from silently going to stale or unmonitored recipients. [doc](https://aws.amazon.com/blogs/security/aws-security-incident-response-the-customers-journey-to-accelerating-the-incident-response-lifecycle/)
- **[CloudTrail auditability]** Enable an organization CloudTrail trail (not just 90-day Event history) so Security Incident Response API calls — who made a case change, from what IP, and when — are retained and queryable for as long as your compliance needs require. [doc](https://docs.aws.amazon.com/security-ir/latest/userguide/logging-using-cloudtrail.html)
- **[analytics strategy]** Decide up front whether to use native AWS analytics (Amazon Athena, Amazon OpenSearch, Amazon Detective) or integrate with an existing SIEM to enrich incident investigations with contextual data — avoids scrambling to stand up analysis tooling mid-incident. [doc](https://aws.amazon.com/blogs/security/optimize-security-operations-with-aws-security-incident-response/)
- **[playbook ownership]** Assign a clear owner to each playbook and runbook and validate them continuously (not just at creation) — keeps response procedures accurate as your environment and threat landscape change. [doc](https://docs.aws.amazon.com/wellarchitected/2023-10-03/framework/ops_ready_to_support_use_playbooks.html)
