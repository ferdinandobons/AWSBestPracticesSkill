# AWS WAF — Best Practices

## Common scenarios
- Filtering and blocking common web exploits (SQLi, XSS, OWASP Top 10) on CloudFront, ALB, API Gateway, or AppSync        → Security
- Mitigating Layer 7 HTTP flood and bot-driven DDoS attacks against public-facing applications        → Security, Reliability
- Centrally deploying and enforcing consistent WAF rules across many accounts and resources        → Operational Excellence, Security
- Tuning rule scope and evaluation order to minimize false positives and control inspection cost        → Cost Optimization, Performance Efficiency

## 🔒 Security
- **[threat modeling]** Complete a threat-modeling exercise first to determine which protections you actually need (for example, skip SQL injection rules if you have no SQL backend) — reduces false positives from unnecessary rules. [doc](https://docs.aws.amazon.com/whitepapers/latest/guidelines-for-implementing-aws-waf/requirements.html)
- **[managed rules]** Start with AWS Managed Rules baseline rule groups (such as the Core rule set and Amazon IP reputation list), then add application-specific managed rule groups matching your workload's profile — provides broad OWASP Top 10 coverage without hand-writing every rule. [doc](https://docs.aws.amazon.com/whitepapers/latest/guidelines-for-implementing-aws-waf/requirements.html)
- **[rule scoping]** Scope each rule to only the paths or components of your application where it's actually needed, for example enabling SQL injection inspection only on paths that talk to a database — reduces false positives and unnecessary rule invocation cost. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/accelerate-and-protect-your-websites-using-amazon-cloudfront-and-aws-waf/)
- **[bot mitigation]** Add the Bot Control managed rule group to gain visibility and control over pervasive bot traffic (scrapers, scanners, crawlers) that can consume resources, skew metrics, or cause downtime. [doc](https://docs.aws.amazon.com/waf/latest/developerguide/aws-managed-rule-groups-bot.html)
- **[account takeover]** Add the Account Takeover Prevention (ATP) managed rule group in front of login endpoints to detect and block credential-stuffing and other malicious login attempts. [doc](https://docs.aws.amazon.com/waf/latest/developerguide/aws-managed-rule-groups-atp.html)
- **[geo restriction]** Use geographic match rule statements (or CloudFront geo-restriction) to block or rate-limit traffic from countries where you do not expect legitimate users — narrows the attack surface. [doc](https://docs.aws.amazon.com/whitepapers/latest/aws-best-practices-ddos-resiliency/aws-waf.html)
- **[defense in depth]** Layer AWS WAF with VPC security controls to address ingress protection at Layer 7, and separately apply security best practices for VPC egress traffic, since AWS WAF only controls ingress — avoids a false sense of complete network coverage. [doc](https://docs.aws.amazon.com/whitepapers/latest/guidelines-for-implementing-aws-waf/guidelines-for-implementing-aws-waf.html)
- **[centralized governance]** Use AWS Firewall Manager with AWS Organizations to enforce a baseline WAF policy across all accounts automatically, while still allowing application teams to add their own rules via retrofitting — ensures no resource is left unprotected as accounts and resources are added. [doc](https://aws.amazon.com/blogs/containers/protecting-your-amazon-eks-web-apps-with-aws-waf/)

## 🛡️ Reliability
- **[rate limiting]** Add a rate-based rule to every web ACL protecting internet-facing resources, tuned above your normal per-IP traffic rate, as a baseline defense against HTTP flood (Layer 7 DDoS) events — automatically blocks offending IPs until their request rate drops. [doc](https://docs.aws.amazon.com/waf/latest/developerguide/ddos-app-layer-rbr.html)
- **[rate limiting]** Use multiple rate-based rules with different thresholds for different parts of your application, such as a high general threshold plus a lower threshold scoped to sensitive endpoints like `/login.html` — mitigates targeted abuse without over-restricting normal traffic. [doc](https://docs.aws.amazon.com/waf/latest/developerguide/ddos-app-layer-rbr.html)
- **[DDoS protection]** Associate a web ACL with your resource and enable Shield Advanced automatic application-layer DDoS mitigation for business-critical, internet-facing applications — lets Shield Advanced dynamically manage rule responses during an active attack. [doc](https://docs.aws.amazon.com/waf/latest/developerguide/ddos-automatic-app-layer-response-config.html)
- **[capacity planning]** Keep total web ACL WCU (Web ACL Capacity Unit) consumption comfortably within limits and monitor consumption as you add rules or rule groups — prevents deployment failures when updating rules under time pressure. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/accelerate-and-protect-your-websites-using-amazon-cloudfront-and-aws-waf/)
- **[false positives]** Establish a methodology to detect and manage false positives (for example, testing in Count mode before switching to Block) before enforcing new or updated rules in production — avoids inadvertently blocking legitimate traffic. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/accelerate-and-protect-your-websites-using-amazon-cloudfront-and-aws-waf/)

## ⚡ Performance Efficiency
- **[rule ordering]** Order rules so that less-expensive terminating rules (such as IP reputation and rate-based rules) run before more expensive rules like Bot Control — limits execution of costly rules and reduces overall inspection latency. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/accelerate-and-protect-your-websites-using-amazon-cloudfront-and-aws-waf/)
- **[allow-listing]** Place rules that allow trusted sources of traffic (such as known search engine crawlers) ahead of other rules — avoids unnecessarily evaluating expensive rules against trusted requests. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/accelerate-and-protect-your-websites-using-amazon-cloudfront-and-aws-waf/)
- **[scope-down statements]** Use scope-down statements to narrow the set of requests a rule evaluates to only the relevant subset of traffic — reduces the processing cost of rule evaluation. [doc](https://docs.aws.amazon.com/waf/latest/developerguide/how-aws-waf-works.html)
- **[cache optimization]** Review sampled requests for patterns like random query strings and configure your cache (for example, CloudFront) to forward only the query string parameters relevant to your application — mitigates cache-busting patterns that increase origin load. [doc](https://docs.aws.amazon.com/whitepapers/latest/aws-best-practices-ddos-resiliency/aws-waf.html)

## 💰 Cost Optimization
- **[WCU budgeting]** Avoid adding WAF rules unrelated to your application's actual threat model (for example, SQL injection rules with no SQL backend) — every additional WCU beyond the default allocation increases per-request charges. [doc](https://docs.aws.amazon.com/waf/latest/developerguide/aws-waf-capacity-units.html)
- **[rule scoping]** Scope expensive managed rule groups such as Bot Control to only the paths that need them rather than applying them web-ACL-wide — reduces both false positives and the WCU-driven request cost of invoking those rules. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/accelerate-and-protect-your-websites-using-amazon-cloudfront-and-aws-waf/)
- **[rate-based rules]** When using custom aggregation keys on rate-based rules, use only the number of keys you actually need, since each additional custom aggregation key adds significant WCU usage — keeps rule cost proportional to the protection value. [doc](https://aws.amazon.com/blogs/security/discover-the-benefits-of-aws-waf-advanced-rate-based-rules/)
- **[rule consolidation]** Reuse the same inspection logic across multiple rules using labels instead of duplicating transformations or statements — lets AWS WAF combine rule behavior and can lower overall WCU consumption. [doc](https://docs.aws.amazon.com/waf/latest/developerguide/how-aws-waf-works.html)

## ⚙️ Operational Excellence
- **[monitoring]** Enable Amazon CloudWatch metrics (AllowedRequests, BlockedRequests, PassedRequests) for every rule and build dashboards or alarms on abnormal rule activity — gives near-real-time visibility into how the web ACL is protecting the application. [doc](https://docs.aws.amazon.com/whitepapers/latest/guidelines-for-implementing-aws-waf/monitoring-and-visibility.html)
- **[logging]** Enable full AWS WAF logging (via Amazon Data Firehose to S3 or other destinations) for production workloads to capture raw request headers and the exact rule match details — provides the depth needed for debugging false positives and investigating incidents that CloudWatch metrics alone cannot. [doc](https://docs.aws.amazon.com/whitepapers/latest/guidelines-for-implementing-aws-waf/monitoring-and-visibility.html)
- **[incident response]** Use AWS CloudTrail logs alongside AWS WAF traffic logs and CloudWatch alarms as part of your incident response process — correlates who changed WAF configuration with what traffic was seen. [doc](https://docs.aws.amazon.com/waf/latest/developerguide/waf-incident-response.html)
- **[deployment model]** Choose and standardize a deployment model — console, AWS Firewall Manager for multi-account rollout, or infrastructure-as-code tools such as AWS CloudFormation — deliberately rather than ad hoc, based on how many accounts and teams manage rules. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/accelerate-and-protect-your-websites-using-amazon-cloudfront-and-aws-waf/)
- **[centralized management]** Use AWS Firewall Manager with a designated administrator account and AWS Config enabled across the organization to automatically detect and remediate newly created resources that lack the required WAF protection — keeps growing estates compliant without manual tracking. [doc](https://aws.amazon.com/blogs/containers/protecting-your-amazon-eks-web-apps-with-aws-waf/)
- **[rule review]** Periodically review sampled requests and WAF logs to identify emerging attack patterns and refine or add custom rules based on observed traffic — turns operational visibility into continuously improving protection. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/building-your-first-aws-waf-web-acl-to-protect-against-evolving-threats/)
