# AWS Security Hub — Best Practices

## Common scenarios
- Centralized cloud security posture management (CSPM) across multiple accounts and Regions        → Security, Operational Excellence
- Aggregating findings from GuardDuty, Inspector, Macie, and partner tools into one dashboard        → Security, Operational Excellence
- Continuous compliance checks against CIS, PCI DSS, NIST, and AWS Foundational Security Best Practices        → Security, Operational Excellence
- Automated triage, suppression, and remediation of security findings at scale        → Security, Cost Optimization

## 🔒 Security
- **[enablement]** Enable Security Hub (and Security Hub CSPM) in every Region where you have authorized activity, not only your primary Regions, and use Service Control Policies to restrict activity in Regions you don't use — ensures continuous monitoring instead of blind spots in unused Regions. [doc](https://aws.amazon.com/blogs/security/nine-aws-security-hub-best-practices/)
- **[standards]** Leave the AWS Foundational Security Best Practices standard and CIS AWS Foundations checks enabled after enabling Security Hub, and ensure AWS Config is recording all supported resources (including global resources) in every account and Region where Security Hub runs — these checks depend on service-linked Config rules to evaluate compliance. [doc](https://aws.amazon.com/blogs/security/nine-aws-security-hub-best-practices/)
- **[complementary services]** Enable both Security Hub and Security Hub CSPM together, along with Amazon GuardDuty, Amazon Inspector, and Amazon Macie — Security Hub CSPM findings feed into Security Hub, where they're correlated with findings from other services to surface the most critical risks and exposures. [doc](https://docs.aws.amazon.com/securityhub/latest/userguide/what-are-securityhub-services.html)
- **[access control]** Grant broad access to Security Hub List/Read actions so teams can view findings, but restrict Write actions (archive, resolve, remediate) to a small, authorized group using AWS managed or customer managed IAM policies — prevents unauthorized tampering with findings while preserving visibility. [doc](https://aws.amazon.com/blogs/security/nine-aws-security-hub-best-practices/)
- **[multi-account]** Use AWS Organizations integration to designate a delegated administrator account for Security Hub CSPM, configure automatic enrollment of new accounts, and only share account IDs/root emails with trusted accounts when inviting members manually — centralizes governance and avoids manual per-account onboarding gaps. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/transitioning-to-multiple-aws-accounts/security-incident-response.html)
- **[access boundaries]** Remember that Security Hub administrator accounts can view and manage findings from all member accounts in administered Regions, but member accounts cannot view other members' data — design your administrator account assignment around who genuinely needs that cross-account visibility. [doc](https://docs.aws.amazon.com/securityhub/latest/userguide/finding-aggregation.html)
- **[remediation]** Build custom, automated remediation playbooks (for example using EventBridge, Lambda, and Systems Manager Automation) for findings that don't require human judgment, and use custom actions to route specific findings to external tools for additional investigation — shortens mean time to remediation for high-volume, well-understood finding types. [doc](https://aws.amazon.com/blogs/security/nine-aws-security-hub-best-practices/)

## 🛡️ Reliability
- **[cross-Region aggregation]** Enable cross-Region finding aggregation to a single designated aggregation Region, and check "Link future Regions" so newly released AWS Regions are automatically included — gives a consistent, complete view of findings without manual reconfiguration as AWS expands. [doc](https://aws.amazon.com/blogs/security/best-practices-for-cross-region-aggregation-of-security-findings/)
- **[multi-administrator design]** When using multiple administrator accounts with cross-Region aggregation, give each administrator a distinct set of member accounts, keep the same member accounts consistent across Regions for a given administrator, and use a different home Region per administrator — avoids conflicting or overlapping aggregation configurations. [doc](https://docs.aws.amazon.com/securityhub/latest/userguide/finding-aggregation.html)
- **[dependency on administrator]** Be aware that if an administrator account is removed or disassociated from a member account, cross-Region aggregation stops for that member — plan administrator account lifecycle changes carefully to avoid unexpected gaps in aggregated visibility. [doc](https://docs.aws.amazon.com/securityhub/latest/userguide/finding-aggregation.html)
- **[downstream integration]** Send Security Hub findings to a SIEM or ticketing system (such as Amazon OpenSearch Service, Jira, or ServiceNow) from a single, centralized aggregation point rather than wiring up separate integrations per Region — reduces operational overhead and the risk of inconsistent or missed findings across integration points. [doc](https://aws.amazon.com/blogs/security/best-practices-for-cross-region-aggregation-of-security-findings/)

## 💰 Cost Optimization
- **[finding noise]** Enable consolidated control findings so Security Hub CSPM generates a single finding per security check even when a control applies to multiple enabled standards, instead of one finding per standard — reduces finding volume and the downstream cost of processing, storing, and triaging duplicates. [doc](https://docs.aws.amazon.com/securityhub/latest/userguide/controls-findings-create-update.html)
- **[global resources]** Record global resources (such as IAM) in AWS Config in only one Region and disable AWS Foundational Security Best Practices periodic controls tied to global resources in other Regions — avoids paying for and reviewing duplicate evaluations of the same global resource in every Region. [doc](https://aws.amazon.com/blogs/security/best-practices-for-cross-region-aggregation-of-security-findings/)
- **[GuardDuty findings]** Auto-archive GuardDuty findings associated with global IAM resources that are not Region-specific when aggregating across Regions — prevents paying attention to and reprocessing the same global-resource finding multiple times. [doc](https://aws.amazon.com/blogs/security/best-practices-for-cross-region-aggregation-of-security-findings/)
- **[centralized remediation]** Implement automatic remediation from a single, central Region rather than duplicating remediation automation per Region — reduces the operational and infrastructure cost of maintaining redundant response pipelines. [doc](https://aws.amazon.com/blogs/security/best-practices-for-cross-region-aggregation-of-security-findings/)

## ⚙️ Operational Excellence
- **[tagging]** Apply tags to your SecurityHub::Hub resource for access control and cost allocation purposes — supports attribute-based access control and clearer cost/usage attribution across accounts and Regions. [doc](https://aws.amazon.com/blogs/security/nine-aws-security-hub-best-practices/)
- **[automation rules]** Use automation rules to automatically update finding fields (such as severity or workflow status) or suppress low-priority findings based on defined criteria, and filter automation criteria by control ID rather than title or description since IDs remain stable while titles/descriptions can change — keeps automations reliable as Security Hub content evolves. [doc](https://docs.aws.amazon.com/securityhub/latest/userguide/automations.html)
- **[automation governance]** Restrict creation, editing, and deletion of automation rules to the Security Hub administrator account, give each rule a clear description of its purpose, and understand rule execution order (lower `RuleOrder` values apply first) — prevents conflicting or opaque automations as rule count grows toward the per-account limit. [doc](https://aws.amazon.com/blogs/security/streamline-security-response-at-scale-with-aws-security-hub-automation/)
- **[response automation]** Use EventBridge rules for actions outside Security Hub (invoking Lambda, notifying SNS/SQS, relaying to a SIEM or ticketing tool) and remember automation rules run before EventBridge rules see the finding — sequence internal finding updates ahead of external notifications/remediation. [doc](https://docs.aws.amazon.com/securityhub/latest/userguide/securityhub-v2-eventbridge-automations.html)
- **[insights]** Customize Security Hub's default managed insights as templates for your own insights to prioritize which resources and findings your team acts on first — focuses operational attention on the highest-value findings instead of raw finding volume. [doc](https://aws.amazon.com/blogs/security/nine-aws-security-hub-best-practices/)
- **[usage patterns]** Choose a deliberate usage pattern — read-only dashboards for application owners, a single-pane-of-glass for security teams, or centralized routing to an existing SIEM — and configure permissions and integrations to match, rather than mixing ad hoc access models — keeps findings ownership and remediation responsibility clear across stakeholders. [doc](https://aws.amazon.com/blogs/security/three-recurring-security-hub-usage-patterns-and-how-to-deploy-them/)
