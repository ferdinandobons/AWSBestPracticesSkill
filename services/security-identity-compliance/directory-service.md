# AWS Directory Service — Best Practices

## Common scenarios
- Providing Active Directory for Windows/Linux workloads, EC2 domain join, and AD-aware apps        → Security, Reliability, Operational Excellence
- Extending or trusting an on-premises Active Directory into AWS        → Security, Reliability
- Enabling AD authentication for AWS services (WorkSpaces, RDS for SQL Server, FSx for Windows File Server, IAM Identity Center)        → Security, Operational Excellence
- Running Active Directory across multiple AWS Regions or multiple AWS accounts        → Reliability, Cost Optimization

## 🔒 Security
- **[directory type]** Choose the directory type (AWS Managed Microsoft AD, AD Connector, or Simple AD) based on user scale, trust requirements, and whether AD is self-managed on-premises rather than defaulting to one option. — Matches the security and feature set (schema extensions, trusts, MFA) to the actual use case. [doc](https://docs.aws.amazon.com/directoryservice/latest/admin-guide/ms_ad_best_practices.html)
- **[network security]** Do not attempt to open additional inbound ports on the directory's managed security group beyond what AWS provisions. — The managed security group already opens only the ports required for Active Directory communication; widening it decreases directory security. [doc](https://docs.aws.amazon.com/directoryservice/latest/admin-guide/ms_ad_best_practices.html)
- **[network security]** Do not attach the directory's managed security group to other EC2 instances, especially any with public IP addresses. — AWS may modify this security group without notice, and reusing it on internet-facing instances exposes AD ports to the internet. [doc](https://docs.aws.amazon.com/directoryservice/latest/admin-guide/ms_ad_best_practices.html)
- **[network security]** When tighter isolation is needed, restrict the directory security group to a narrower CIDR range (specific subnets or hosts) rather than the full VPC range, using separate customer/client and domain-controller CIDR scenarios as a guide. — Reduces the network path available to reach domain controllers. [doc](https://docs.aws.amazon.com/directoryservice/latest/admin-guide/ms_ad_network_security.html)
- **[account isolation]** Deploy AWS Managed Microsoft AD into a separate, dedicated AWS account and share it with other accounts rather than hosting it alongside workloads. — Creates an IAM boundary that limits who can administer the directory and requires explicit sharing for access. [doc](https://aws.amazon.com/blogs/industries/fsi-service-spotlight-aws-managed-microsoft-active-directory/)
- **[IAM]** Grant IAM identity-based policies for Directory Service following least privilege (for example scoping to `ds:Describe*` for read-only needs) instead of broad wildcard actions. — Limits the blast radius of a compromised or misconfigured principal. [doc](https://docs.aws.amazon.com/directoryservice/latest/admin-guide/IAM_Auth_Access_Overview.html)
- **[predefined objects]** Do not move, delete, or otherwise alter the predefined organizational unit, groups, or administrative users that AWS creates with the directory. — Altering these objects can make the directory inaccessible to both you and AWS. [doc](https://docs.aws.amazon.com/directoryservice/latest/admin-guide/ms_ad_best_practices.html)
- **[MFA]** Enable multi-factor authentication for AWS Managed Microsoft AD to strengthen sign-in security for AWS applications such as WorkSpaces and QuickSight. — Adds a one-time passcode requirement on top of AD credentials, reducing the impact of credential theft. [doc](https://aws.amazon.com/blogs/security/introducing-aws-directory-service-for-microsoft-active-directory-standard-edition/)
- **[encryption in transit]** Enable LDAP over SSL/TLS (LDAPS) so that LDAP communications between applications and the directory are encrypted end to end. — Protects directory data from interception when accessed over untrusted networks. [doc](https://docs.aws.amazon.com/directoryservice/latest/admin-guide/ms_ad_security.html)
- **[protocol hardening]** Edit directory security settings to disable legacy ciphers (such as RC4 or DES) and legacy protocols (such as SSL 2.0/3.0 and TLS 1.0/1.1) on the secure channel configuration. — Removes weak cryptographic options across all domain controllers without added operational effort. [doc](https://aws.amazon.com/directoryservice/features/)
- **[SMB hardening]** Ensure clients accessing SYSVOL and NETLOGON shares use SMB 2.x or newer, and update any remaining SMBv1 clients. — AWS Managed Microsoft AD only supports SMBv2+, and SMBv1 is a known-insecure protocol that AWS and Microsoft recommend disabling. [doc](https://docs.aws.amazon.com/directoryservice/latest/admin-guide/ms_ad_best_practices.html)
- **[trusts]** When configuring a trust relationship, verify the trust type matches on both sides, the trust direction is set correctly for one-way trusts, and FQDNs/NetBIOS names are unique between forests. — Misconfigured trusts can silently fail or create unintended access paths between directories. [doc](https://docs.aws.amazon.com/directoryservice/latest/admin-guide/ms_ad_best_practices.html)
- **[password policy]** Apply Active Directory password policies (including user-specific fine-grained policies for administrators versus general users) rather than relying on a single default policy. — Lets you enforce stronger password and lockout requirements for privileged accounts. [doc](https://docs.aws.amazon.com/directoryservice/latest/admin-guide/ms_ad_security.html)
- **[logging]** Enable log forwarding of domain controller security event logs to Amazon CloudWatch Logs. — Provides transparency and supports security monitoring, audit, and log retention requirements. [doc](https://aws.amazon.com/directoryservice/features/)

## 🛡️ Reliability
- **[domain controllers]** Deploy additional domain controllers beyond the two AWS creates by default across Availability Zones. — Increases fault resiliency during patching and other events that make a domain controller unreachable, and improves scale-out performance. [doc](https://docs.aws.amazon.com/directoryservice/latest/admin-guide/ms_ad_best_practices.html)
- **[multi-Region]** Use AWS Managed Microsoft AD Enterprise Edition multi-Region replication when Active Directory-aware workloads span multiple AWS Regions. — Provides automated cross-Region replication, local performance, and disaster recovery if all domain controllers in one Region become unavailable. [doc](https://docs.aws.amazon.com/directoryservice/latest/admin-guide/ms_ad_configure_multi_region_replication.html)
- **[load balancing]** Do not place an external network load balancer in front of AWS Managed Microsoft AD endpoints. — Active Directory's own DC-locator algorithm already finds the most responsive domain controller, and external load balancers can misroute traffic to a DC that is not yet ready. [doc](https://docs.aws.amazon.com/directoryservice/latest/admin-guide/ms_ad_best_practices.html)
- **[application design]** Program applications to use the Windows DC locator service or DDNS instead of hard-coding a specific domain controller address. — Avoids hot-spotting a single DC and ensures applications automatically fail over to another DC during patching or recovery. [doc](https://docs.aws.amazon.com/directoryservice/latest/admin-guide/ms_ad_best_practices.html)
- **[backups]** Take manual snapshots of the directory and back up any instance before manually joining it to the domain (especially Linux instances). — Snapshots support point-in-time restore, and pre-join backups protect against instances becoming unreachable if the join procedure fails. [doc](https://docs.aws.amazon.com/directoryservice/latest/admin-guide/ms_ad_best_practices.html)
- **[alerting]** Set up Amazon SNS notifications for directory status changes, and re-associate a directory to a new SNS topic before deleting the old one. — Ensures you are alerted promptly when a directory moves from Active to Impaired or Inoperable. [doc](https://docs.aws.amazon.com/directoryservice/latest/admin-guide/ms_ad_best_practices.html)
- **[capacity planning]** Load test with representative object counts and request volumes before rolling out to production, adding domain controllers as needed. — Confirms the directory scales to the application's real workload and avoids surprises from under-provisioned capacity. [doc](https://docs.aws.amazon.com/directoryservice/latest/admin-guide/ms_ad_best_practices.html)

## ⚡ Performance Efficiency
- **[LDAP queries]** Use efficient, narrowly scoped LDAP queries instead of broad queries across tens of thousands of objects. — Broad queries can consume significant CPU on a single domain controller and cause hot-spotting that affects other applications sharing that DC. [doc](https://docs.aws.amazon.com/directoryservice/latest/admin-guide/ms_ad_best_practices.html)
- **[schema extensions]** Plan schema extensions carefully, indexing only for important and frequent queries rather than over-indexing. — Indexes consume directory space, and rapidly changing indexed values can cause performance problems. [doc](https://docs.aws.amazon.com/directoryservice/latest/admin-guide/ms_ad_best_practices.html)
- **[scaling]** Track domain controller performance metrics in CloudWatch and scale out (add domain controllers) or scale up (move to Enterprise Edition) based on observed utilization. — Right-sizes the directory to sustained and peak load rather than guessing at capacity. [doc](https://aws.amazon.com/blogs/security/explore-scaling-options-for-aws-directory-service-for-microsoft-active-directory/)
- **[local access]** For multi-Region deployments, rely on multi-Region replication so applications authenticate against a local, in-Region directory copy. — Reduces authentication latency by avoiding cross-Region calls for AD-aware applications. [doc](https://docs.aws.amazon.com/directoryservice/latest/admin-guide/ms_ad_configure_multi_region_replication.html)

## 💰 Cost Optimization
- **[directory type]** Match the directory type to actual needs — AWS Managed Microsoft AD for larger deployments needing trusts, AD Connector to reuse an existing self-managed AD, or Simple AD for small, low-cost, basic-compatibility needs — instead of over-provisioning a feature-rich option. — Avoids paying for capabilities (for example, trusts or schema extensions) the workload does not use. [doc](https://docs.aws.amazon.com/directoryservice/latest/admin-guide/simple_ad_best_practices.html)
- **[sharing]** Share a single AWS Managed Microsoft AD directory across multiple AWS accounts within a Region instead of creating a separate directory per account. — Reduces the number of directories to manage and pay for while still allowing per-account domain join. [doc](https://aws.amazon.com/blogs/architecture/running-hybrid-active-directory-service-with-aws-managed-microsoft-active-directory/)
- **[right-sizing]** Use CloudWatch domain controller utilization metrics to decide when to add or remove domain controllers rather than statically over-provisioning. — Lets you define the number of domain controllers needed to meet performance and resilience targets without paying for unused capacity. [doc](https://aws.amazon.com/blogs/security/top-2021-aws-service-launches-security-professionals-should-review-part-2/)

## ⚙️ Operational Excellence
- **[monitoring]** Monitor domain controller performance counters (CPU, memory) and directory status in CloudWatch, and configure alarms to trigger automated responses to high utilization. — Surfaces capacity and health issues before they affect AD-aware applications. [doc](https://aws.amazon.com/directoryservice/features/)
- **[centralized logging]** Forward security event logs to a centralized CloudWatch Logs group (optionally in a dedicated logging account) and integrate with tools like OpenSearch or Splunk for analysis. — Enables correlation of authentication and directory-change events for troubleshooting and compliance reporting. [doc](https://aws.amazon.com/blogs/apn/how-to-monitor-and-analyze-aws-managed-microsoft-ad-security-logs-using-amazon-cloudwatch-and-splunk/)
- **[DHCP]** Create a DHCP options set and assign it to the directory's VPC so instances automatically point to the directory's DNS servers. — Ensures domain name resolution works correctly for all instances in the VPC without manual DNS configuration per instance. [doc](https://docs.aws.amazon.com/directoryservice/latest/admin-guide/ms_ad_best_practices.html)
- **[domain join]** Join instances to the domain automatically at launch time by selecting the directory as the domain join directory, rather than joining manually afterward. — Reduces manual configuration steps and the chance of join errors. [doc](https://docs.aws.amazon.com/directoryservice/latest/admin-guide/ms_ad_best_practices.html)
- **[conditional forwarders]** Enable persistent conditional forwarder storage in Active Directory (replicated) when configuring conditional forwarders. — Ensures the forwarder setting survives domain controller replacement due to infrastructure failure. [doc](https://docs.aws.amazon.com/directoryservice/latest/admin-guide/ms_ad_best_practices.html)
- **[application cleanup]** Remove all AWS Enterprise applications (WorkSpaces, WorkDocs, WorkMail, RDS, etc.) associated with a directory before deleting it. — Prevents deletion failures and orphaned application state. [doc](https://docs.aws.amazon.com/directoryservice/latest/admin-guide/ms_ad_best_practices.html)
