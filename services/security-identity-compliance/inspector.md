# Amazon Inspector — Best Practices

## Common scenarios
- Continuous vulnerability scanning of EC2 instances, ECR container images, and Lambda functions        → Security, Operational Excellence
- Centralized, organization-wide vulnerability management across a multi-account AWS estate        → Security, Operational Excellence
- Prioritizing and triaging thousands of CVE findings at scale        → Security, Operational Excellence, Cost Optimization
- Network reachability analysis to detect unintended internet exposure        → Security, Reliability

## 🔒 Security
- **[multi-account]** Designate a dedicated Security Tooling account (not the AWS Organizations management account) as the Amazon Inspector delegated administrator — centralizes findings and scan management while keeping the management account free of operational duties. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/vulnerability-management/amazon-inspector.html)
- **[multi-account]** Enable Amazon Inspector on all accounts in the organization and integrate it with EventBridge and Security Hub CSPM to drive reporting and notifications — closes coverage gaps left by manually onboarded accounts. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/aws-security-controls/detective-controls.html)
- **[multi-account]** Use AWS Organizations policies to enforce Amazon Inspector enablement across all accounts, and designate the delegated administrator before creating the policy — ensures member accounts are automatically associated for centralized findings visibility rather than left to opt in individually. [doc](https://docs.aws.amazon.com/inspector/latest/user/getting_started_tutorial.html)
- **[IAM]** Grant only least-privilege permissions for Amazon Inspector actions, starting from AWS managed policies and narrowing with customer-managed policies as your use case stabilizes — reduces blast radius of compromised credentials with Inspector access. [doc](https://docs.aws.amazon.com/inspector/v1/userguide/security_iam_id-based-policy-examples.html)
- **[IAM]** Add IAM policy conditions (such as requiring SSL/TLS for all requests) and validate policies with IAM Access Analyzer before attaching them — catches overly permissive statements before they reach production. [doc](https://docs.aws.amazon.com/inspector/v1/userguide/security_iam_id-based-policy-examples.html)
- **[prioritization]** Triage findings using the Amazon Inspector risk score, which contextualizes the CVSS base score against your environment (network reachability, exploitability), rather than raw CVE severity alone — focuses remediation effort on the vulnerabilities that pose real risk. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/aws-security-controls/detective-controls.html)
- **[network exposure]** Review Amazon Inspector network reachability findings regularly to catch overly permissive security groups, NACLs, or internet gateways that expose instances (including behind load balancers) to the internet — surfaces unintended exposure that vulnerability scanning alone would miss. [doc](https://aws.amazon.com/blogs/security/practical-steps-to-minimize-key-exposure-using-aws-security-services/)
- **[code security]** Use Amazon Inspector code vulnerability scanning for Lambda functions and code repositories to catch injection flaws, weak cryptography, and data leaks in application code — extends coverage beyond infrastructure and package vulnerabilities to the code itself. [doc](https://aws.amazon.com/blogs/security/practical-steps-to-minimize-key-exposure-using-aws-security-services/)
- **[SBOM]** Centrally manage software bill of materials (SBOM) exports for all monitored resources to shift security earlier in the development lifecycle — gives teams visibility into software composition before vulnerabilities are exploited. [doc](https://aws.amazon.com/inspector/)
- **[integration]** Integrate Amazon Inspector with AWS Security Hub so findings are correlated and enriched with organization-wide context — surfaces critical risks that are only apparent when analyzed across the full environment. [doc](https://aws.amazon.com/inspector/)

## 🛡️ Reliability
- **[architecture]** Treat Amazon Inspector as a Regional service and designate a delegated administrator, add member accounts, and activate scan types independently in every Region you operate in — a configuration in one Region does not extend coverage to others. [doc](https://aws.amazon.com/blogs/security/amazon-inspector-suppression-rules-best-practices-for-aws-organizations/)
- **[continuous assessment]** Rely on Amazon Inspector's automatic rescanning behavior (triggered by new package installs, patches, or newly published CVEs) rather than scheduling manual point-in-time scans — ensures resources are reassessed throughout their lifecycle without operational effort. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/vulnerability-management/amazon-inspector.html)
- **[EC2 coverage]** Use hybrid scan mode (the default for new accounts) so Amazon Inspector falls back to agentless scanning for EC2 instances that lack an SSM Agent — avoids coverage gaps for instances where the agent-based method cannot run. [doc](https://docs.aws.amazon.com/inspector/latest/user/scanning-ec2.html)
- **[monitoring]** Route Amazon Inspector findings through Amazon EventBridge to durable, redundant targets (Lambda, SNS, Security Hub) rather than relying on manual console review — ensures new findings and state changes are not missed when operators are unavailable. [doc](https://docs.aws.amazon.com/inspector/latest/user/monitoring-overview.html)
- **[ECR coverage]** For multi-architecture container images, configure re-scan duration based on image pull or push events instead of last-in-use date, since last-in-use tracking is not supported for multi-arch images — prevents images from silently dropping out of continuous scanning coverage. [doc](https://docs.aws.amazon.com/inspector/latest/user/scanning_resources_configure_duration_setting_ecr.html)

## 💰 Cost Optimization
- **[ECR re-scan duration]** Tune the ECR re-scan duration to match your build cadence — a shorter duration for frequently rebuilt images and a longer duration for long-lived images — since rescans are billed per image and unnecessary continual scanning of stale images adds cost without security value. [doc](https://docs.aws.amazon.com/inspector/latest/user/scanning_resources_configure_duration_setting_ecr.html)
- **[ECR lifecycle]** Let Amazon Inspector automatically stop monitoring (mark inactive) container images that fall outside the configured push/last-in-use window — avoids paying to rescan images that are no longer deployed or pulled. [doc](https://docs.aws.amazon.com/inspector/latest/user/scanning_resources_configure_duration_setting_ecr.html)
- **[trial period]** Use the 15-day free trial when first enabling Amazon Inspector or a new scan type to estimate ongoing cost before committing at scale — avoids budget surprises once continual scanning is fully enabled. [doc](https://aws.amazon.com/blogs/containers/container-scanning-updates-in-amazon-ecr-private-registries-using-amazon-inspector/)
- **[findings triage]** Use suppression rules to archive findings that match agreed-upon low-risk criteria (informational severity, non-production tags, low Amazon Inspector score) so downstream tools like Security Hub, S3, or a SIEM are not paying to store and process low-value findings — but re-evaluate rules regularly to avoid over-suppression that hides real risk. [doc](https://aws.amazon.com/blogs/security/amazon-inspector-suppression-rules-best-practices-for-aws-organizations/)
- **[tagging strategy]** Tag resources with environment and risk-exposure attributes (for example EnvironmentName, RiskExposureScore) so suppression rules can be centrally managed at scale instead of relying on account-specific resource, subnet, or VPC IDs that drift over time — keeps rules maintainable across large multi-account estates. [doc](https://aws.amazon.com/blogs/security/amazon-inspector-suppression-rules-best-practices-for-aws-organizations/)

## ⚙️ Operational Excellence
- **[multi-account]** Use the Amazon Inspector delegated administrator model to centrally activate/deactivate scans, view aggregated findings, and manage suppression rules for the entire organization from one place — avoids fragmented, per-account manual configuration. [doc](https://aws.amazon.com/blogs/security/amazon-inspector-suppression-rules-best-practices-for-aws-organizations/)
- **[suppression rules]** Build suppression rules on the Amazon Inspector score together with resource tags rather than ad hoc severity filtering alone, and cap awareness at the delegated administrator's 500-rule-per-Region limit — keeps large-scale triage manageable without hitting hard service limits. [doc](https://aws.amazon.com/blogs/security/amazon-inspector-suppression-rules-best-practices-for-aws-organizations/)
- **[suppression rules]** Re-evaluate suppression rules on a regular cadence rather than treating them as set-and-forget — vulnerability landscapes and resource risk profiles change, and stale rules can hide newly relevant findings. [doc](https://aws.amazon.com/blogs/security/amazon-inspector-suppression-rules-best-practices-for-aws-organizations/)
- **[automation]** Create EventBridge rules that filter on Amazon Inspector finding attributes (such as `inspectorScore` and `severity`) to trigger automated notifications or remediation workflows for critical findings — shortens response time versus manual console monitoring. [doc](https://aws.amazon.com/blogs/security/perform-continuous-vulnerability-scanning-of-aws-lambda-functions-with-amazon-inspector/)
- **[auditing]** Enable AWS CloudTrail to log all Amazon Inspector API calls (for example `CreateFindingsReport`, `UpdateOrganizationConfiguration`) — supports auditing of who changed scan configuration and when. [doc](https://docs.aws.amazon.com/inspector/latest/user/logging-using-cloudtrail.html)
- **[AMI scanning]** For golden AMI pipelines, tag AMIs for on-demand Amazon Inspector scanning and use EventBridge to detect scan completion before promoting the AMI — integrates vulnerability gating directly into image-build automation. [doc](https://aws.amazon.com/blogs/security/how-to-scan-ec2-amis-using-amazon-inspector/)
