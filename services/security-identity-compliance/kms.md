# AWS KMS — Best Practices

## Common scenarios
- Centralizing creation and control of cryptographic keys for encryption at rest        → Security, Operational Excellence
- Enabling server-side encryption for S3, EBS, RDS, and other AWS services        → Security, Cost Optimization
- Client-side (envelope) encryption in custom applications via the AWS Encryption SDK        → Security, Performance Efficiency
- Multi-Region and multi-account key strategies for disaster recovery and centralized governance        → Reliability, Operational Excellence

## 🔒 Security
- **[key policy]** Apply least-privilege permissions in key policies and IAM policies, delegating only the minimum actions required rather than granting `kms:*` broadly — reduces blast radius if a principal is compromised. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/aws-kms-best-practices/access.html)
- **[key policy]** Prefer key policies over broad IAM policies for sensitive permissions such as `kms:PutKeyPolicy` and `kms:ScheduleKeyDeletion`, and specify exact key ARNs in the `Resource` element of IAM statements — limits which principals can manage or delete a given key. [doc](https://docs.aws.amazon.com/kms/latest/developerguide/iam-policies-best-practices.html)
- **[key creation]** Restrict `kms:CreateKey` to only the principals who need it, since key creators can set the initial key policy and grant themselves further access — prevents unauthorized privilege escalation. [doc](https://docs.aws.amazon.com/kms/latest/developerguide/iam-policies-best-practices.html)
- **[access control]** Use separate customer managed KMS keys per data classification and enforce separation of duties for key management tasks (creation, rotation, policy changes) — limits impact of a single compromised role. [doc](https://aws.amazon.com/blogs/security/strengthen-the-devops-pipeline-and-protect-data-with-aws-secrets-manager-aws-kms-and-aws-certificate-manager/)
- **[grants]** Scope grants tightly to a specific grantee principal and only the operations needed, use encryption context constraints (prefer `EncryptionContextEqual`) to bind grants to their intended purpose, and use `SourceArn` for AWS service-created grants — prevents grant misuse for unintended data. [doc](https://docs.aws.amazon.com/kms/latest/developerguide/grant-best-practices.html)
- **[grants]** Retire or revoke grants as soon as their permission is no longer needed, since grants do not expire automatically — unretired grants are latent security risk. [doc](https://docs.aws.amazon.com/kms/latest/developerguide/grant-best-practices.html)
- **[cross-service access]** Use AWS KMS encryption context and the global `aws:SourceArn`/`aws:SourceAccount` condition keys when granting AWS services permission to use a key on your behalf — ensures the permission is valid only for the resources the service should access. [doc](https://docs.aws.amazon.com/kms/latest/developerguide/least-privilege.html)
- **[authentication]** Require multi-factor authentication for IAM users (or use IAM roles instead) before performing privileged KMS actions such as scheduling key deletion — mitigates risk from compromised long-term credentials. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/aws-kms-best-practices/access.html)
- **[key deletion]** Treat `ScheduleKeyDeletion` as destructive: set an appropriate waiting period (7–30 days) and monitor for any use of the key during that window before it is permanently deleted — deleted key material and the data it protected are unrecoverable. [doc](https://docs.aws.amazon.com/kms/latest/developerguide/deleting-keys.html)
- **[governance]** Choose a centralized model (keys managed in one or a few accounts, other accounts only permitted to use them) when you need to minimize risk of unintended deletion or privilege escalation by delegated administrators, versus a decentralized per-account model for simpler local delegation — align the model with your organizational control requirements. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/aws-kms-best-practices/key-management.html)

## 🛡️ Reliability
- **[key rotation]** Enable automatic annual rotation for symmetric encryption KMS keys so new key material is generated yearly while old material is retained for decrypting existing data — reduces long-term exposure of a single key without breaking access to previously encrypted data. [doc](https://docs.aws.amazon.com/kms/latest/developerguide/multi-region-keys-overview.html)
- **[disaster recovery]** Use multi-Region keys (not the same key manually recreated) when encrypted data or digital signatures must move across Regions, such as for disaster recovery, global tables, or globally distributed signature verification — replica keys share key material and key IDs with the primary for interoperability. [doc](https://aws.amazon.com/kms/features/)
- **[multi-Region keys]** Perform automatic and on-demand rotation only on the primary key of a multi-Region key set and let AWS KMS synchronize rotation status and key material to replicas — avoids inconsistent key material across Regions. [doc](https://docs.aws.amazon.com/kms/latest/developerguide/multi-region-keys-overview.html)
- **[throttling]** Implement a backoff-and-retry strategy for `ThrottlingException` responses, and avoid bursts of state-changing calls (such as repeated `EnableKey`/`DisableKey`) against the same "hot" key — reduces request failures under load. [doc](https://docs.aws.amazon.com/kms/latest/developerguide/throttling.html)
- **[key deletion]** Before finalizing deletion of a KMS key, configure a CloudWatch alarm to detect any attempted use during the pending-deletion waiting period, and use `CancelKeyDeletion` if the key is still needed — prevents irreversible loss of access to encrypted data. [doc](https://docs.aws.amazon.com/kms/latest/developerguide/deleting-keys.html)

## ⚡ Performance Efficiency
- **[client-side encryption]** Use data key caching with the AWS Encryption SDK for applications that perform frequent client-side encryption operations — reduces the number of calls to AWS KMS and improves application throughput. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/aws-kms-best-practices/cost.html)
- **[grants]** Use a grant token immediately after creating a grant if the grantee must use the permission right away, since AWS KMS grants follow an eventual consistency model and may take up to several minutes to propagate — avoids transient `AccessDeniedException` errors. [doc](https://docs.aws.amazon.com/kms/latest/developerguide/using-grant-token.html)
- **[capacity planning]** Monitor AWS KMS API usage against Service Quotas using CloudWatch metrics and request quota increases proactively before high-traffic events — prevents request throttling as usage grows. [doc](https://aws.amazon.com/blogs/security/manage-your-aws-kms-api-request-rates-using-service-quotas-and-amazon-cloudwatch/)

## 💰 Cost Optimization
- **[S3 encryption]** Enable Amazon S3 Bucket Keys for SSE-KMS encrypted buckets that generate high request volumes — can reduce AWS KMS request costs by up to 99% compared to per-object KMS calls. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/aws-kms-best-practices/cost.html)
- **[client-side encryption]** Use AWS Encryption SDK data key caching in high-throughput client-side encryption workloads — reduces the number of billed AWS KMS API requests. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/aws-kms-best-practices/cost.html)
- **[budgeting]** Create an AWS Budgets usage budget scoped to KMS keys or requests to get alerted when spend exceeds expected thresholds — improves visibility into key storage and request costs. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/aws-kms-best-practices/cost.html)
- **[logging]** Manage CloudTrail data event logging scope for high-volume KMS-related operations and use Amazon Athena for ad-hoc log analysis instead of always-on heavy querying — controls the cost of auditing large volumes of cryptographic API calls. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/aws-kms-best-practices/cost.html)
- **[key strategy]** In multi-tenant or multi-service architectures, evaluate a shared customer managed key per tenant instead of one key per tenant per service — reduces the number of billed keys while preserving tenant isolation through policy and encryption context. [doc](https://aws.amazon.com/blogs/architecture/simplify-multi-tenant-encryption-with-a-cost-conscious-aws-kms-key-strategy/)

## ⚙️ Operational Excellence
- **[monitoring]** Enable a CloudTrail trail (or organization trail) to retain AWS KMS API activity beyond the default 90-day event history — supports auditing and reconstructing a KMS key's lifecycle. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/aws-kms-best-practices/monitoring.html)
- **[monitoring]** Set CloudWatch alarms on sensitive KMS operations such as `PutKeyPolicy`, `DisableKey`, and `ScheduleKeyDeletion`, and route CloudTrail events to EventBridge for automated response — enables faster detection and remediation of misconfigurations or misuse. [doc](https://docs.aws.amazon.com/kms/latest/developerguide/security-logging-monitoring.html)
- **[monitoring]** Use per-key CloudWatch usage metrics with CloudWatch Metrics Insights queries to identify which KMS keys consume the most request quota or drive the most cost — supports capacity planning and cost attribution. [doc](https://aws.amazon.com/blogs/security/aws-kms-cloudwatch-metrics-help-you-better-track-and-understand-how-your-kms-keys-are-being-used/)
- **[governance]** Decide deliberately between a centralized and decentralized multi-account key management model based on your IaC/automation maturity and administrative boundaries, and manage AWS accounts as a single AWS Organizations organization — keeps key governance predictable as the estate grows. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/aws-kms-best-practices/key-management.html)
- **[grants]** Use the `Name` parameter on `CreateGrant` requests and periodically call `ListGrants` to detect duplicate grants — avoids needlessly consuming the grants-per-key quota. [doc](https://docs.aws.amazon.com/kms/latest/developerguide/grant-best-practices.html)
