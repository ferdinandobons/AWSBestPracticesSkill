# AWS IAM Identity Center — Best Practices

## Common scenarios
- Centralized workforce sign-in (SSO) to multiple AWS accounts        → Security, Operational Excellence
- Federating an external IdP (Okta, Entra ID, Active Directory) to AWS        → Security, Reliability
- Fine-grained, scalable permission management across an AWS Organization        → Security, Operational Excellence
- Delegating day-to-day identity administration away from the management account        → Security, Operational Excellence

## 🔒 Security
- **[access management]** Use IAM Identity Center for centralized workforce access instead of creating long-lived IAM users. — Eliminates standing credentials and centralizes lifecycle management across accounts. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/patterns/use-user-ids-iam-policies-access-control-automation.html)
- **[permission sets]** Design permission sets around least privilege, starting from AWS managed job-function policies and tailoring with customer managed policies or permissions boundaries as needed. — Limits blast radius of any single credential. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/patterns/manage-aws-iam-identity-center-permission-sets-as-code-by-using-aws-codepipeline.html)
- **[management account]** Create dedicated permission sets for the Organizations management account and assign them directly to individual users rather than groups. — Reduces the number of principals with access to the most sensitive account. [doc](https://aws.amazon.com/blogs/security/getting-started-with-aws-sso-delegated-administration/)
- **[delegated administration]** Register a member account as the IAM Identity Center delegated administrator instead of managing identities from the Organizations management account. — Minimizes the number of people who need access to the management account. [doc](https://docs.aws.amazon.com/singlesignon/latest/userguide/delegated-admin.html)
- **[MFA]** Enforce multi-factor authentication for all users and encourage registering multiple backup authenticators (FIDO2 security keys, biometrics, or TOTP apps). — Protects sign-in from credential compromise and prevents lockout from lost devices. [doc](https://docs.aws.amazon.com/singlesignon/latest/userguide/mfa-configure.html)
- **[MFA]** Avoid the "one-time password sent by email" MFA option when users must sign in to the AWS access portal to read that same email. — Prevents users from being locked out because they cannot retrieve the verification code. [doc](https://docs.aws.amazon.com/singlesignon/latest/userguide/mfa-configure.html)
- **[access lifecycle]** Synchronize users and groups from your authoritative identity source using SCIM so access is automatically updated or revoked as roles change or employees leave. — Keeps entitlements aligned with the current workforce lifecycle. [doc](https://docs.aws.amazon.com/wellarchitected/latest/framework/sec_permissions_lifecycle.html)
- **[ABAC]** Use attribute-based access control with attributes sourced from your identity provider to centralize and scale authorization instead of creating many narrowly scoped permission sets. — Reduces permission-set sprawl and lets access adapt automatically as tagged attributes change. [doc](https://docs.aws.amazon.com/singlesignon/latest/userguide/abac.html)
- **[group governance]** If groups are used to grant access to the management account, restrict who can modify group membership in the identity source and log/review changes to that membership. — Prevents privilege escalation through uncontrolled group changes. [doc](https://aws.amazon.com/blogs/security/getting-started-with-aws-sso-delegated-administration/)
- **[emergency access]** Establish a break-glass or emergency access process (for example, direct IAM federation to a dedicated emergency operations account) that works independently of IAM Identity Center and the external IdP. — Preserves the ability to administer AWS during an identity provider or IAM Identity Center disruption. [doc](https://docs.aws.amazon.com/singlesignon/latest/userguide/emergency-access.html)

## 🛡️ Reliability
- **[multi-Region]** Replicate the IAM Identity Center instance to additional AWS Regions when continued account access is required during a Regional disruption. — Maintains authentication and already-provisioned permissions if the primary Region is impaired. [doc](https://docs.aws.amazon.com/singlesignon/latest/userguide/resiliency-regional-behavior.html)
- **[Region selection]** Choose the primary Region deliberately before enabling IAM Identity Center. — The primary Region cannot be changed later without deleting and reconfiguring the entire instance. [doc](https://docs.aws.amazon.com/help-panel/IAM/latest/console/hp_enable_idc_regions_info.html)
- **[emergency access]** Set up and periodically test emergency access even when the instance is replicated to multiple Regions. — Provides continued access to the AWS Management Console during an external IdP or IAM Identity Center disruption. [doc](https://docs.aws.amazon.com/singlesignon/latest/userguide/emergency-access.html)

## ⚙️ Operational Excellence
- **[permission sets as code]** Manage permission sets and assignments as code (for example, via a pipeline or Terraform) rather than manually in the console. — Enables review, versioning, and repeatable rollout of access changes across accounts. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/patterns/manage-aws-iam-identity-center-permission-sets-as-code-by-using-aws-codepipeline.html)
- **[planning]** Plan permission sets, their target accounts/OUs, and assigned principals before deployment. — Permission set names, association IDs, and principal types/IDs cannot be changed after deployment. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/patterns/manage-aws-iam-identity-center-permission-sets-as-code-by-using-aws-codepipeline.html)
- **[session duration]** Configure user interactive session duration deliberately based on your organization's security requirements rather than leaving the default. — Balances user friction against the risk window of a long-lived session token. [doc](https://docs.aws.amazon.com/singlesignon/latest/userguide/user-interactive-sessions.html)
- **[logging]** Create an organization CloudTrail trail that captures IAM Identity Center, Identity Store, OIDC, and SCIM API events across all Regions and accounts. — Provides a durable audit record of sign-ins, assignment changes, and directory integration activity for investigation and rollback. [doc](https://docs.aws.amazon.com/singlesignon/latest/userguide/security-logging-and-monitoring.html)
- **[monitoring]** Send CloudTrail events for IAM Identity Center to CloudWatch Logs (or CloudTrail Lake) and configure alarms for sensitive actions. — Enables near-real-time detection of unexpected administrative changes. [doc](https://aws.amazon.com/blogs/security/how-to-use-multiple-instances-of-aws-iam-identity-center/)
