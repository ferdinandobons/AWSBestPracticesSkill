# AWS Firewall Manager — Best Practices

## Common scenarios
- Centrally enforcing a baseline of AWS WAF rules, security groups, or Network Firewall policies across every account in an AWS Organization        → Security, Operational Excellence
- Automatically protecting newly created resources (VPCs, ALBs, CloudFront distributions) as they are provisioned across accounts        → Security, Reliability
- Auditing and remediating overly permissive security groups (open SSH/RDP/SMB to the internet) org-wide        → Security
- Monitoring AWS Shield Advanced DDoS protection compliance and centralizing findings across multiple accounts        → Reliability, Operational Excellence

## 🔒 Security
- **[prerequisites]** Enable AWS Organizations with all features turned on and enable AWS Config in every member account before onboarding Firewall Manager — Firewall Manager depends on AWS Config to detect new and changed resources across the organization. [doc](https://docs.aws.amazon.com/waf/latest/developerguide/fms-prereq.html)
- **[administrator account]** Designate a dedicated Security Tooling account (not the Organizations management account) as the Firewall Manager administrator — separates security policy administration from organization management duties. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/security-reference-architecture/security-tooling.html)
- **[least privilege]** Use multi-admin support to grant additional Firewall Manager administrators only restricted administrative scope (specific accounts, OUs, Regions, or policy types) rather than full scope — enforces least-privilege access as responsibilities are delegated across teams. [doc](https://docs.aws.amazon.com/waf/latest/developerguide/fms-administrators.html)
- **[layered policies]** Combine a centrally managed baseline policy (for example an IP reputation and rate-limiting rule group) with application-team-managed rule groups in the same web ACL — lets the security team enforce org-wide protections while application teams retain flexibility for app-specific rules. [doc](https://docs.aws.amazon.com/whitepapers/latest/guidelines-for-implementing-aws-waf/requirements.html)
- **[security group audit]** Create a security group content-audit policy using a managed application list (such as the default list denying public access to SSH, RDP, NFS, SMB, and NetBIOS) to continuously find and flag high-risk exposure across all accounts. [doc](https://aws.amazon.com/blogs/security/how-to-continuously-audit-and-limit-security-groups-with-aws-firewall-manager/)
- **[IAM policies]** Start from AWS managed policies for Firewall Manager and move toward customer-managed, least-privilege policies, adding IAM conditions (such as requiring SSL or MFA) to further restrict who can create or modify policies. [doc](https://docs.aws.amazon.com/waf/latest/developerguide/fms-security_iam_id-based-policy-examples.html)
- **[network firewall governance]** Use a Network Firewall Firewall Manager policy to enforce mandatory inspection of VPC traffic across the organization, combined with AWS RAM resource sharing so centrally managed firewalls and rule groups can be applied to member-account VPCs. [doc](https://aws.amazon.com/blogs/security/enforce-your-aws-network-firewall-protections-at-scale-with-aws-firewall-manager/)

## 🛡️ Reliability
- **[automatic remediation]** After validating a policy in identify-only (manual remediation) mode, switch to automatic remediation so Firewall Manager continuously detects and fixes configuration drift on in-scope resources without manual intervention. [doc](https://aws.amazon.com/firewall-manager/faqs/)
- **[compliance monitoring]** Keep AWS Config recording set to continuous (the default) for all protected resources and avoid rapidly toggling Firewall Manager protections — frequent changes can cause AWS Config to lose track of state and produce incorrect compliance or remediation status. [doc](https://docs.aws.amazon.com/waf/latest/developerguide/fms-compliance.html)
- **[DDoS resilience]** Create a Shield Advanced Firewall Manager policy to enforce DDoS protection compliance across all in-scope accounts and automatically protect newly created resources as they come into policy scope. [doc](https://docs.aws.amazon.com/waf/latest/developerguide/ddos-viewing-multiple-accounts.html)
- **[centralized visibility]** Integrate Firewall Manager with AWS Security Hub CSPM to get a single dashboard of DDoS events and policy compliance findings across every protected account, rather than checking each account individually. [doc](https://docs.aws.amazon.com/waf/latest/developerguide/fms-findings.html)

## ⚙️ Operational Excellence
- **[centralized logging]** Enable centralized AWS WAF logging in your Firewall Manager policy, directing logs to a Kinesis Data Firehose delivery stream into a single account (such as your Security Tooling account) — consolidates logs from web ACLs across the whole organization for analysis. [doc](https://aws.amazon.com/blogs/security/centrally-manage-aws-waf-api-v2-and-aws-managed-rules-at-scale-with-firewall-manager/)
- **[monitoring]** Use Amazon CloudWatch alarms on Firewall Manager and related metrics, and enable AWS CloudTrail logging for Firewall Manager API calls (PutPolicy, DeletePolicy, AssociateAdminAccount, etc.) — provides the audit trail and alerting needed to debug drift or unauthorized policy changes. [doc](https://docs.aws.amazon.com/waf/latest/developerguide/fms-incident-response.html)
- **[event-driven automation]** Match Firewall Manager CloudTrail events in Amazon EventBridge (source `aws.fms`) to trigger automated notifications or workflows on specific administrative API calls. [doc](https://docs.aws.amazon.com/eventbridge/latest/ref/events-ref-fms.html)
- **[compliance findings]** Route Firewall Manager compliance and attack findings into AWS Security Hub CSPM and filter by Product Name "Firewall Manager" to track noncompliant resources and attacks alongside other security findings in one place. [doc](https://docs.aws.amazon.com/waf/latest/developerguide/fms-findings.html)
- **[policy rollout]** Start new policies in manual/identify-only remediation mode to inventory noncompliant resources before enabling automatic remediation — avoids unexpected changes to production resources when a new policy is first deployed. [doc](https://aws.amazon.com/blogs/security/how-to-continuously-audit-and-limit-security-groups-with-aws-firewall-manager/)
- **[delegated administration]** Use tag-based or OU-based scoping in policies, and delegate administrator responsibilities by OU, account, Region, or policy type, so that policy management scales cleanly as the organization adds accounts and teams. [doc](https://aws.amazon.com/blogs/security/enable-multi-admin-support-to-manage-security-policies-at-scale-with-aws-firewall-manager/)
