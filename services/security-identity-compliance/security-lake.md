# Amazon Security Lake — Best Practices

## Common scenarios
- Centralizing security logs and findings from AWS, SaaS, and custom sources into one data lake        → Security, Operational Excellence
- Multi-account, multi-Region log aggregation for compliance and regional data residency        → Security, Reliability
- Long-term retention of security data for investigations and SIEM/analytics integration        → Cost Optimization, Operational Excellence
- Enabling subscribers (Athena, OpenSearch, SIEM/XDR) to query normalized OCSF security data        → Security, Performance Efficiency

## 🔒 Security
- **[IAM]** Grant Security Lake users the minimum permissions needed (for example, list sources without permission to create sources or subscribers) and move from AWS managed policies toward custom least-privilege policies as usage matures — limits blast radius if a principal is compromised. [doc](https://docs.aws.amazon.com/security-lake/latest/userguide/best-practices-overview.html)
- **[IAM]** Validate identity-based policies with IAM Access Analyzer and add explicit conditions (such as requiring SSL) to further restrict access to Security Lake actions and resources — catches overly permissive policies before they're attached. [doc](https://docs.aws.amazon.com/security-lake/latest/userguide/security_iam_id-based-policy-examples.html)
- **[IAM]** Require multi-factor authentication (MFA) for any IAM users or root user scenarios that can access Security Lake, and add MFA conditions to sensitive policies — adds a control against credential compromise. [doc](https://docs.aws.amazon.com/security-lake/latest/userguide/security_iam_id-based-policy-examples.html)
- **[multi-account]** Use a dedicated Log Archive account (not the Organizations management account) as the Security Lake delegated administrator, and restrict access to the underlying S3 buckets to only the Security Lake service and IAM roles it manages for sources/subscribers — keeps log storage isolated and tightly controlled. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/security-reference-architecture/log-archive.html)
- **[multi-account]** Have security teams and tools access Security Lake data only through a separate Security Tooling account (as data-access or query-access subscribers) rather than directly in the Log Archive account — preserves separation of duties between log custody and log analysis. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/security-reference-architecture/log-archive.html)
- **[encryption]** Use server-side encryption with a customer managed AWS KMS key (SSE-KMS) instead of the default S3-managed key when you need direct control over the key policy and auditability of key usage, and attach a key policy that grants only `kms:CreateGrant`, `kms:DescribeKey`, and `kms:GenerateDataKey` to the roles that need it — adds a managed, auditable layer of access control beyond IAM alone. [doc](https://docs.aws.amazon.com/security-lake/latest/userguide/data-protection.html)
- **[encryption]** Note that Security Lake does not require `kms:Decrypt` permissions for itself, but subscribers consuming source data do need `Decrypt` permissions on the key — grant that permission narrowly to subscriber roles only. [doc](https://docs.aws.amazon.com/security-lake/latest/userguide/data-protection.html)
- **[data protection]** Use TLS 1.2 at minimum (prefer TLS 1.3) for all communication with Security Lake and related AWS resources, and enable AWS CloudTrail to log API and user activity against Security Lake — provides an audit trail of who accessed or changed the data lake configuration. [doc](https://docs.aws.amazon.com/security-lake/latest/userguide/data-protection.html)
- **[data protection]** Never place confidential or sensitive information (such as customer email addresses) into Security Lake tags or free-form name fields — this data can surface in billing or diagnostic logs. [doc](https://docs.aws.amazon.com/security-lake/latest/userguide/data-protection.html)
- **[integration]** Integrate Security Lake with AWS Security Hub CSPM so CSPM findings from AWS and partner services flow into the data lake — gives a consolidated view of compliance posture alongside raw log data. [doc](https://docs.aws.amazon.com/security-lake/latest/userguide/best-practices-overview.html)
- **[coverage]** Enable Security Lake in all supported AWS Regions, even ones you don't actively use, so activity tied to global services or unused Regions is still collected — reduces blind spots from unauthorized or unusual activity in dormant Regions. [doc](https://aws.amazon.com/security-lake/faqs/)
- **[CloudTrail]** Maintain at least one multi-Region CloudTrail organization trail with logging enabled for read and write management events before relying on Security Lake to collect CloudTrail management events — Security Lake's CloudTrail coverage depends on this trail existing. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/security-reference-architecture/log-archive.html)

## 🛡️ Reliability
- **[storage]** Rely on the S3 object integrity, versioning, and Lifecycle rules under Security Lake's single-tenant buckets, and apply S3 best practices (such as Object Lock where appropriate) for the underlying buckets — protects log durability and availability for incident response. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/security-reference-architecture/log-archive.html)
- **[rollup Regions]** Configure one or more rollup Regions to consolidate logs from contributing Regions, and create the required IAM roles for data replication and AWS Glue partition registration up front when using the API/CLI — ensures cross-Region consolidation keeps working outside the console's automatic role creation. [doc](https://docs.aws.amazon.com/security-lake/latest/userguide/add-rollup-region.html)
- **[custom sources]** Follow the documented S3 partition data path format for custom sources — replication to a rollup Region cannot be performed on data that doesn't follow this format, breaking cross-Region consolidation. [doc](https://docs.aws.amazon.com/security-lake/latest/userguide/add-rollup-region.html)
- **[encryption]** Be aware that replication from a customer-managed-key (CMK) encrypted regional data lake to an S3-managed-encryption regional data lake is not supported — plan consistent encryption choices across Regions before enabling rollup replication. [doc](https://docs.aws.amazon.com/security-lake/latest/userguide/add-rollup-region.html)
- **[operations]** When removing a Lambda function associated with Security Lake, delete it directly rather than disabling it first — disabling before deletion can interfere with data querying and other functionality. [doc](https://docs.aws.amazon.com/security-lake/latest/userguide/best-practices-overview.html)

## 💰 Cost Optimization
- **[lifecycle]** Configure lifecycle transitions and expiration per Region (for example, transition to `ONEZONE_IA` after a set number of days, expire objects after your retention window) rather than keeping all data on standard storage indefinitely — automated tiering reduces storage costs while meeting compliance retention needs. [doc](https://docs.aws.amazon.com/security-lake/latest/userguide/lifecycle-management.html)
- **[retention]** Choose which log sources to retain long-term in Security Lake, which to forward to analytics tools, and which to query in place, instead of defaulting to uniform retention across all sources — lets you extend retention only where compliance or investigation value justifies the cost. [doc](https://aws.amazon.com/security-lake/pricing/)
- **[format]** Take advantage of Security Lake's automatic conversion of ingested data to Apache Parquet (and Apache Iceberg table support) rather than querying raw log formats — reduces storage footprint and improves query cost-efficiency for subscriber analytics. [doc](https://aws.amazon.com/security-lake/pricing/)

## ⚙️ Operational Excellence
- **[monitoring]** Monitor Security Lake with Amazon CloudWatch metrics, which are collected at one-minute intervals and retained for 15 months, and configure alarms on relevant thresholds — provides near real-time and historical visibility into data lake health. [doc](https://docs.aws.amazon.com/security-lake/latest/userguide/cloudwatch-metrics.html)
- **[monitoring]** Review the Security Lake console Summary page regularly for issues from the last 14 days affecting the service or its underlying S3 buckets — surfaces problems early so you can investigate and mitigate before they affect data collection. [doc](https://docs.aws.amazon.com/security-lake/latest/userguide/best-practices-overview.html)
- **[auditing]** Enable AWS CloudTrail logging for Security Lake API calls to track who performed which actions, from where, and when — supports operational troubleshooting and security auditing of changes to the data lake configuration. [doc](https://docs.aws.amazon.com/security-lake/latest/userguide/best-practices-overview.html)
- **[multi-account]** Use AWS Organizations to designate a delegated administrator for Security Lake and enable log collection organization-wide, rather than configuring each account individually — reduces operational overhead of onboarding new accounts and keeps coverage consistent. [doc](https://aws.amazon.com/blogs/big-data/generate-security-insights-from-amazon-security-lake-data-using-amazon-opensearch-ingestion/)
