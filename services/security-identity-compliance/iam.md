# AWS IAM — Best Practices

## Common scenarios
- Granting human users console/API access to AWS accounts        → Security, Operational Excellence
- Giving workloads (EC2, Lambda, containers) permission to call AWS APIs        → Security, Reliability
- Delegating permissions management to teams or third parties        → Security, Operational Excellence
- Enforcing organization-wide permission guardrails across accounts        → Security, Operational Excellence

## 🔒 Security
- **[human access]** Require human users to federate through an identity provider (e.g. IAM Identity Center) and assume roles for temporary credentials instead of creating IAM users — removes long-lived passwords/keys and centralizes access management. [doc](https://docs.aws.amazon.com/IAM/latest/UserGuide/best-practices.html)
- **[workload access]** Require workloads to use IAM roles (instance profiles, Lambda execution roles, IAM Roles Anywhere, ECS/EKS integrations) instead of distributing long-term access keys — temporary credentials are automatically rotated and reduce standing risk. [doc](https://docs.aws.amazon.com/IAM/latest/UserGuide/best-practices.html)
- **[authentication]** Require multi-factor authentication for the root user and any IAM users, preferring phishing-resistant MFA such as passkeys or security keys — adds a second factor beyond a compromised password. [doc](https://docs.aws.amazon.com/IAM/latest/UserGuide/best-practices.html)
- **[long-term credentials]** Where IAM user access keys are unavoidable (e.g. workloads that can't use roles, some third-party clients), rotate and update them regularly using access-key last-used information — limits the exposure window of a leaked key. [doc](https://docs.aws.amazon.com/IAM/latest/UserGuide/best-practices.html)
- **[root user]** Avoid using the root user for everyday or administrative tasks; lock away its credentials, enable MFA, don't create root access keys, and for multi-account setups remove root credentials from member accounts entirely — the root user has unrestrictable full access to the account. [doc](https://docs.aws.amazon.com/IAM/latest/UserGuide/root-user-best-practices.html)
- **[permissions]** Apply least-privilege permissions by defining only the actions, resources, and conditions a task actually requires, starting from AWS managed policies and narrowing to customer managed policies as your use case matures — minimizes blast radius of a compromised identity. [doc](https://docs.aws.amazon.com/IAM/latest/UserGuide/best-practices.html)
- **[least privilege]** Use IAM Access Analyzer to generate fine-grained policies from a role's actual CloudTrail activity, test them, then deploy to production — right-sizes permissions to observed need instead of guesswork. [doc](https://docs.aws.amazon.com/IAM/latest/UserGuide/best-practices.html)
- **[unused access]** Regularly review and remove unused users, roles, permissions, policies, and credentials using IAM last-accessed information and IAM Access Analyzer's unused-access findings — shrinks the set of standing permissions an attacker could exploit. [doc](https://docs.aws.amazon.com/IAM/latest/UserGuide/access-analyzer-concepts.html)
- **[conditions]** Add IAM policy conditions (e.g. requiring TLS, restricting by source service or network) to further restrict when a granted permission actually applies — narrows access even within an otherwise-necessary permission. [doc](https://docs.aws.amazon.com/IAM/latest/UserGuide/best-practices.html)
- **[cross-account/public access]** Use IAM Access Analyzer to verify and continuously monitor public and cross-account access to resources before and after granting it — surfaces unintended external exposure. [doc](https://docs.aws.amazon.com/IAM/latest/UserGuide/best-practices.html)
- **[policy validation]** Validate IAM policies with IAM Access Analyzer policy validation (100+ automated checks) before saving them — catches insecure or non-functional policy statements early. [doc](https://docs.aws.amazon.com/IAM/latest/UserGuide/best-practices.html)
- **[multi-account guardrails]** Use AWS Organizations service control policies (SCPs) and resource control policies (RCPs) to set organization-wide permission guardrails for principals and resources, on top of (not instead of) identity- and resource-based policies — enforces a security baseline no account-level policy can override. [doc](https://docs.aws.amazon.com/IAM/latest/UserGuide/best-practices.html)
- **[delegation]** Use permissions boundaries to cap the maximum permissions an IAM entity can grant when delegating role/policy creation to others, such as letting developers manage their own workload roles — prevents privilege escalation through delegated administration. [doc](https://docs.aws.amazon.com/IAM/latest/UserGuide/best-practices.html)
- **[identity strategy]** Distinguish and separately manage human identities (workforce users) and machine identities (workloads), defining principals, policies, and credential handling appropriate to each — aligns access design with the AWS Well-Architected security pillar's identity guidance. [doc](https://docs.aws.amazon.com/wellarchitected/latest/framework/sec-iam.html)

## ⚙️ Operational Excellence
- **[access management]** Manage human access centrally with AWS IAM Identity Center rather than per-account IAM users, assigning permission sets across accounts — gives one place to grant, change, and audit access as roles change. [doc](https://docs.aws.amazon.com/IAM/latest/UserGuide/best-practices.html)
- **[lifecycle]** Treat joiner/mover/leaver processes as time-boxed role assignments through IAM Identity Center and temporary credentials so access expires automatically when no longer needed — avoids manual cleanup of stale permissions. [doc](https://docs.aws.amazon.com/wellarchitected/latest/framework/sec_permissions_lifecycle.html)
- **[monitoring]** Enable AWS CloudTrail to log IAM and API activity, and feed it into IAM Access Analyzer's policy generation and last-accessed analysis — gives the visibility needed to safely tighten permissions over time. [doc](https://docs.aws.amazon.com/IAM/latest/UserGuide/best-practices.html)
- **[governance]** Establish preventive and detective guardrails for every account using AWS Organizations (SCPs/RCPs) together with account-level controls — bakes governance into account provisioning rather than relying on manual review. [doc](https://docs.aws.amazon.com/IAM/latest/UserGuide/best-practices.html)
