# AWS CloudHSM — Best Practices

## Common scenarios
- Generating and storing cryptographic keys in single-tenant, FIPS-validated hardware for regulatory compliance        → Security, Reliability
- Offloading SSL/TLS or database (e.g., Oracle TDE) encryption key operations to dedicated HSMs        → Security, Performance Efficiency
- Serving as a custom key store for AWS KMS to meet key-control or compliance requirements        → Security, Operational Excellence
- Code signing and PKI root/issuing certificate authority key protection        → Security, Reliability

## 🔒 Security
- **[network isolation]** Launch HSMs and client instances in private VPC subnets rather than public subnets — limits access to HSMs from outside your network. [doc](https://docs.aws.amazon.com/cloudhsm/latest/userguide/bp-cluster-management.html)
- **[network isolation]** Use VPC endpoints to reach the AWS CloudHSM API instead of routing client instances through the internet — removes the need for internet access on instances that only need the control-plane API. [doc](https://docs.aws.amazon.com/cloudhsm/latest/userguide/bp-cluster-management.html)
- **[security groups]** Do not delete or modify the preconfigured TCP rule (ports 2223-2225) in the auto-created cluster security group, and ensure only trusted administrators can access instances placed in it — anyone reaching an instance in that security group can reach the HSMs, and an unauthenticated zeroize operation can irrecoverably destroy key material. [doc](https://docs.aws.amazon.com/cloudhsm/latest/userguide/configure-sg.html)
- **[credentials]** Protect HSM user (crypto officer/admin and crypto user) credentials as securely as any root secret — AWS CloudHSM has no access to these credentials and cannot help recover a lost or locked-out account. [doc](https://docs.aws.amazon.com/cloudhsm/latest/userguide/bp-hsm-user-management.html)
- **[user management]** Maintain at least two admin (crypto officer) users in the cluster — prevents a lost or locked admin password from locking you out entirely. [doc](https://docs.aws.amazon.com/cloudhsm/latest/userguide/bp-hsm-user-management.html)
- **[user management]** Enable quorum authentication requiring a minimum number of admin approvals for user management operations — limits the blast radius if a single admin credential is compromised. [doc](https://docs.aws.amazon.com/cloudhsm/latest/userguide/bp-hsm-user-management.html)
- **[user management]** Create multiple crypto users with distinctly limited, separated responsibilities instead of one user with full control — enforces separation of duties over key generation, sharing, and use. [doc](https://docs.aws.amazon.com/cloudhsm/latest/userguide/bp-hsm-user-management.html)
- **[user management]** When adding or removing users or changing passwords, refresh the cluster configuration file (Configure tool with `-m`) and connect to every HSM in the cluster before making changes — keeps HSMs synchronized and prevents authentication failures from desynchronized credentials. [doc](https://aws.amazon.com/blogs/security/cloudhsm-best-practices-to-maximize-performance-and-avoid-common-configuration-pitfalls/)
- **[key attributes]** Generate keys with the minimum set of attributes required for their task (e.g., an encryption key should not also be wrap-capable) — reduces what a compromised key or credential can be used for. [doc](https://docs.aws.amazon.com/cloudhsm/latest/userguide/bp-application-integration.html)
- **[key wrapping]** Mark keys non-extractable unless export is explicitly required, and use trusted key wrapping to restrict crypto users to wrap/unwrap only keys an admin has explicitly marked as trusted — non-extractable keys can never be exported once set, so choose deliberately. [doc](https://docs.aws.amazon.com/cloudhsm/latest/userguide/bp-hsm-key-management.html)

## 🛡️ Reliability
- **[high availability]** Provision at least two HSMs per cluster, spread across different Availability Zones, so no single HSM is a point of failure — high availability and load balancing are automatic once multiple HSMs exist in a cluster. [doc](https://docs.aws.amazon.com/cloudhsm/latest/userguide/cluster-high-availability-load-balancing.html)
- **[capacity redundancy]** Size clusters with at least +1 redundancy over your peak-load requirement — absorbs scheduled HSM maintenance or replacement without service impact. [doc](https://docs.aws.amazon.com/cloudhsm/latest/userguide/bp-cluster-management.html)
- **[subnet planning]** Include a subnet from every Availability Zone in the Region when creating a cluster, even if CloudHSM is not yet available in all of them — additional subnets cannot be added to a cluster after creation. [doc](https://aws.amazon.com/blogs/security/cloudhsm-best-practices-to-maximize-performance-and-avoid-common-configuration-pitfalls/)
- **[key durability]** Use at least three HSMs spread across different Availability Zones — required for applications that need strong durability guarantees for newly generated keys. [doc](https://docs.aws.amazon.com/cloudhsm/latest/userguide/bp-cluster-management.html)
- **[client resilience]** Bootstrap the Client SDK using the `--cluster-id` parameter rather than hardcoding individual HSM IP addresses — keeps client configuration resilient to HSM maintenance or an Availability Zone outage. [doc](https://docs.aws.amazon.com/cloudhsm/latest/userguide/bp-application-integration.html)
- **[retries]** Implement client-side retry logic on cluster operations, since AWS may replace an HSM for operational reasons and in-flight operations on that HSM can fail — retries are expected to succeed against the replacement. [doc](https://docs.aws.amazon.com/cloudhsm/latest/userguide/bp-application-integration.html)
- **[disaster recovery]** Copy cluster backups across AWS Regions and be prepared to restore a new cluster from backup — clusters are Regional resources that cannot span or be extended across Regions. [doc](https://docs.aws.amazon.com/cloudhsm/latest/userguide/regions.html)
- **[disaster recovery]** Use VPC peering to reach a cluster from another account or Region, or connect to multiple active clusters from the same application (Client SDK 5) — lets traffic shift away from an impaired cluster or Region. [doc](https://docs.aws.amazon.com/cloudhsm/latest/userguide/bp-application-integration.html)
- **[deployment]** Stagger client application deployments and restarts using progressive wave-based, one-box, or rolling deployment strategies — limits the impact of a bad deployment while preserving capacity to serve production traffic. [doc](https://docs.aws.amazon.com/cloudhsm/latest/userguide/bp-application-integration.html)

## ⚡ Performance Efficiency
- **[load testing]** Load test the cluster against your anticipated peak traffic, using published performance estimates as a starting point — confirms the architecture is resilient and correctly sized before production use. [doc](https://docs.aws.amazon.com/cloudhsm/latest/userguide/bp-cluster-management.html)
- **[key type]** Use token keys instead of session keys when you need requests load-balanced across all HSMs in the cluster — a session key's throughput is limited to the single HSM where it resides. [doc](https://docs.aws.amazon.com/cloudhsm/latest/userguide/bp-hsm-key-management.html)
- **[key lookup]** Cache key objects in application memory after an initial key-find operation instead of re-querying for the same key on every operation — key find operations query every HSM in the cluster and are expensive. [doc](https://docs.aws.amazon.com/cloudhsm/latest/userguide/bp-application-integration.html)
- **[multi-threading]** Initialize the PKCS #11 library or JCE provider only once per application and give each thread its own session — sharing sessions or SPI objects across threads is not supported and can degrade performance or correctness. [doc](https://docs.aws.amazon.com/cloudhsm/latest/userguide/bp-application-integration.html)
- **[throttling]** Ensure the cluster is scaled for peak traffic and carries +1 redundancy for maintenance events, and handle throttling errors with backoff — undersized clusters or reduced HSM counts during an Availability Zone outage can trigger HSM throttling. [doc](https://docs.aws.amazon.com/cloudhsm/latest/userguide/bp-application-integration.html)
- **[key storage limits]** Use trusted key wrapping to store keys in an external data store and unwrap them into the HSM as session keys only when needed, or distribute keys across multiple clusters — works around per-HSM limits on the number of stored token/session keys. [doc](https://docs.aws.amazon.com/cloudhsm/latest/userguide/bp-hsm-key-management.html)

## 💰 Cost Optimization
- **[elasticity]** Scale a cluster down to zero HSMs when the workload does not require continuous availability (e.g., periodic signing jobs), then restore from the automatic periodic backup when the HSMs are needed again — avoids paying for idle HSM hours. [doc](https://docs.aws.amazon.com/cloudhsm/latest/userguide/bp-cluster-management.html)

## ⚙️ Operational Excellence
- **[client logging]** On ephemeral platforms such as Amazon ECS or AWS Lambda, configure the Client SDK (and any third-party integration built on it) to log to the console rather than to a file — ephemeral compute makes file-based logs hard to collect, while console output is captured automatically into CloudWatch Logs. [doc](https://docs.aws.amazon.com/cloudhsm/latest/userguide/bp-monitoring.html)
- **[audit logging]** Monitor HSM audit logs published to Amazon CloudWatch and trigger alarms on unexpected management commands — supports detection of unauthorized or unintended operations on the HSM. [doc](https://docs.aws.amazon.com/cloudhsm/latest/userguide/bp-monitoring.html)
- **[API auditing]** Use AWS CloudTrail, which captures all AWS CloudHSM control-plane API calls from the console and SDK — enables auditing of account activity to detect unwanted actions. [doc](https://docs.aws.amazon.com/cloudhsm/latest/userguide/bp-monitoring.html)
- **[metrics and alarms]** Configure CloudWatch alarms on cluster metrics for approaching key, session, or user count limits per HSM, for differences in HSM user/key counts, and for unhealthy HSMs — surfaces synchronization issues and capacity limits before they affect the application. [doc](https://docs.aws.amazon.com/cloudhsm/latest/userguide/bp-monitoring.html)
- **[credential handling]** Authenticate using implicit login where supported (e.g., JCE) and store HSM credentials securely, for example in AWS Secrets Manager, rather than embedding them in application code — reduces re-authentication failures and credential exposure. [doc](https://docs.aws.amazon.com/cloudhsm/latest/userguide/bp-application-integration.html)
