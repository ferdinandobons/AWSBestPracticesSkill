# AWS IAM Access Analyzer — Best Practices

## Common scenarios
- Detecting unintended public or cross-account access to S3, IAM roles, KMS keys, Lambda, and SQS        → Security
- Right-sizing IAM roles and users by removing unused permissions, keys, and passwords        → Security, Cost Optimization
- Validating and testing IAM policies before deployment, including in CI/CD pipelines        → Security, Operational Excellence
- Monitoring access continuously across an AWS Organization from a central account        → Security, Operational Excellence

## 🔒 Security
- **[external access]** Create an external access analyzer with the appropriate zone of trust (account or organization) in every Region you use, so resource-based policies for supported resource types are continuously monitored for access from outside that trust boundary — surfaces unintended public or cross-account exposure as soon as a policy changes. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/security-reference-architecture/checklist.html)
- **[internal access]** Configure an internal access analyzer for sensitive, explicitly-selected resources to confirm only intended principals within your organization or account can reach them — reinforces least privilege for high-value resources beyond the external-access check. [doc](https://docs.aws.amazon.com/IAM/latest/UserGuide/access-analyzer-delegated-administrator.html)
- **[unused access]** Create an unused access analyzer to continuously surface unused IAM roles, unused access keys and passwords for IAM users, and unused service/action permissions for active principals — guides ongoing right-sizing toward least privilege instead of a one-time review. [doc](https://aws.amazon.com/blogs/security/customize-the-scope-of-iam-access-analyzer-unused-access-analysis/)
- **[scope tuning]** Scope unused-access analysis by excluding specific accounts, roles, or users (for example sandbox accounts or dedicated security/audit roles) using account IDs or tags — reduces noise so reviewers focus on findings that matter. [doc](https://aws.amazon.com/blogs/security/customize-the-scope-of-iam-access-analyzer-unused-access-analysis/)
- **[least privilege]** Generate fine-grained IAM policies from a role's actual CloudTrail access activity instead of hand-authoring broad permissions, then test the generated policy before deploying it — grants only what the workload has demonstrably used. [doc](https://aws.amazon.com/iam/resources/best-practices/)
- **[policy validation]** Validate every new and existing IAM policy with IAM Access Analyzer policy validation (100+ checks covering security warnings, errors, and best-practice suggestions) before saving — catches issues like overly broad `iam:PassRole` with `Resource: *` early. [doc](https://aws.amazon.com/iam/access-analyzer/features/)
- **[custom policy checks]** Use `CheckNoNewAccess` to block policy changes that grant access beyond a defined reference policy, and `CheckAccessNotGranted` to confirm specific sensitive actions or resources are never allowed — enforces organization-specific guardrails that generic validation can't express. [doc](https://docs.aws.amazon.com/IAM/latest/UserGuide/access-analyzer-custom-policy-checks.html)
- **[public access]** Run `CheckNoPublicAccess` against resource policies before deployment to confirm a specific resource type cannot be made public — prevents accidental public exposure from ever reaching production. [doc](https://aws.amazon.com/blogs/security/how-to-implement-iam-policy-checks-with-visual-studio-code-and-iam-access-analyzer/)
- **[triage]** Create archive rules that automatically archive findings matching known, intended access patterns (for example a cross-account audit role) so active findings stay focused on unreviewed or unintended access — keeps the finding queue actionable as your environment scales. [doc](https://aws.amazon.com/blogs/security/new-streamline-existing-iam-access-analyzer-findings-using-archive-rules/)

## ⚙️ Operational Excellence
- **[organization-wide coverage]** Enable IAM Access Analyzer in every member account and the management account, and designate a delegated administrator (typically a dedicated security/tooling account) to create and manage analyzers with the organization as the zone of trust — centralizes findings review without granting analyzer management rights to the management account itself. [doc](https://docs.aws.amazon.com/IAM/latest/UserGuide/access-analyzer-delegated-administrator.html)
- **[dashboard triage]** Use the delegated administrator's centralized dashboard to review findings across the organization and prioritize which accounts to investigate based on finding volume and type — turns organization-wide monitoring into a manageable, risk-ranked workflow. [doc](https://aws.amazon.com/blogs/security/customize-the-scope-of-iam-access-analyzer-unused-access-analysis/)
- **[automation]** Send IAM Access Analyzer findings to Amazon EventBridge and route new, changed, or deleted findings to notification or remediation workflows (for example SNS alerts or Lambda-based auto-remediation) — closes the loop from detection to action without manual polling. [doc](https://docs.aws.amazon.com/IAM/latest/UserGuide/access-analyzer-eventbridge.html)
- **[CI/CD integration]** Integrate IAM Access Analyzer policy validation and custom policy checks into CI/CD pipelines (e.g. via the CloudFormation/Terraform policy validator or IDE tooling) so noncompliant policies are caught before merge or deployment — shifts policy review left instead of catching issues in production. [doc](https://aws.amazon.com/blogs/security/how-to-implement-iam-policy-checks-with-visual-studio-code-and-iam-access-analyzer/)
- **[posture management]** Integrate IAM Access Analyzer findings with AWS Security Hub CSPM to correlate external-access and unused-access findings against broader security posture and standards — gives a single place to prioritize identity risk alongside other security findings. [doc](https://aws.amazon.com/iam/access-analyzer/features/)
