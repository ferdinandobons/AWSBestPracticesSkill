# Amazon Verified Permissions — Best Practices

## Common scenarios
- Externalizing fine-grained authorization (RBAC/ABAC) out of application code        → Security, Operational Excellence
- Policy decision point (PDP) for multi-tenant SaaS APIs and microservices        → Security, Reliability
- UI- and API-level permission checks tied to Amazon Cognito or another OIDC IdP        → Security, Performance Efficiency
- High-volume authorization for web/mobile apps needing low-latency decisions        → Performance Efficiency, Cost Optimization

## 🔒 Security
- **[Identifiers]** Use immutable, unique values such as UUIDs for principal and resource identifiers instead of human-readable names, so that reused names (e.g., a departed user's username) never inherit stale access. — [doc](https://docs.aws.amazon.com/verifiedpermissions/latest/apireference/Welcome.html)
- **[Identifiers]** Never embed personally identifying, confidential, or sensitive information inside principal/resource identifiers, since identifiers are captured in log entries. — [doc](https://docs.aws.amazon.com/verifiedpermissions/latest/apireference/Welcome.html)
- **[Error handling]** Return 403 Forbidden rather than 404 Not Found for requests referencing an entity that doesn't match any policy, to avoid revealing whether a resource exists. — [doc](https://docs.aws.amazon.com/verifiedpermissions/latest/userguide/design-authz-strategy.html)
- **[Guardrails]** Use `forbid` statements as permission guardrails to make certain actions never allowed (for example, requiring MFA before a destructive action), on top of the default-deny behavior of `permit` policies. — [doc](https://aws.amazon.com/blogs/security/get-the-best-out-of-amazon-verified-permissions-by-using-fine-grained-authorization-methods/)
- **[Namespaces]** Use Cedar namespaces to disambiguate identically named actions across different roles or subsystems (for example, a user `deletePhoto` versus an admin `deletePhoto`). — [doc](https://aws.amazon.com/blogs/security/get-the-best-out-of-amazon-verified-permissions-by-using-fine-grained-authorization-methods/)
- **[Resource-centric modeling]** Model permissions around underlying resources rather than individual API operations when multiple APIs can expose the same data, so one abstract permission (e.g., `viewGroupMembership`) consistently governs every code path that can reach it. — [doc](https://docs.aws.amazon.com/verifiedpermissions/latest/userguide/design-authz-strategy.html)
- **[Encryption]** Encrypt policy stores with an AWS KMS customer managed key instead of the default AWS owned key when you need to control and audit who can access your authorization policy data. — [doc](https://docs.aws.amazon.com/verifiedpermissions/latest/userguide/security-data-protection-cmk.html)
- **[KMS key policy]** Grant Verified Permissions only the minimum required key permissions (`kms:Encrypt`, `kms:GenerateDataKeyWithoutPlaintext`, `kms:DescribeKey`, `kms:ReEncryptTo`, `kms:ReEncryptFrom`, `kms:Decrypt`) in the customer managed key policy. — [doc](https://docs.aws.amazon.com/verifiedpermissions/latest/userguide/security-data-protection-cmk.html)
- **[IAM least privilege]** Start from AWS managed policies for Verified Permissions and then scope down to customer managed, least-privilege IAM policies for who can create, read, or delete policy stores, policies, and identity sources. — [doc](https://docs.aws.amazon.com/verifiedpermissions/latest/userguide/security_iam_id-based-policy-examples.html)
- **[IAM conditions]** Add IAM policy conditions (such as requiring SSL/TLS or restricting calls to a specific calling AWS service) to further restrict who and what can manage Verified Permissions resources. — [doc](https://docs.aws.amazon.com/verifiedpermissions/latest/userguide/security_iam_id-based-policy-examples.html)
- **[IAM validation]** Validate IAM policies that grant Verified Permissions access with IAM Access Analyzer before deployment to catch overly permissive or non-functional statements. — [doc](https://docs.aws.amazon.com/verifiedpermissions/latest/userguide/security_iam_id-based-policy-examples.html)
- **[MFA]** Require multi-factor authentication for IAM users or roles that can administer Verified Permissions policy stores, and add MFA conditions to sensitive policy-management API calls. — [doc](https://docs.aws.amazon.com/verifiedpermissions/latest/userguide/security_iam_id-based-policy-examples.html)
- **[Token validation]** Configure client application ID (Cognito) or audience (OIDC) validation on identity sources so `IsAuthorizedWithToken`/`BatchIsAuthorizedWithToken` reject tokens issued to unintended app clients. — [doc](https://docs.aws.amazon.com/verifiedpermissions/latest/userguide/cognito-validation.html)
- **[Multi-tenant isolation]** Use per-tenant policy stores when strict isolation of tenant policies or per-tenant IAM access control is required, since a shared policy store needs tenant identifiers embedded in every policy and request instead. — [doc](https://docs.aws.amazon.com/verifiedpermissions/latest/userguide/design-authz-strategy.html)
- **[Zero Trust]** Use Verified Permissions as the policy decision point to continually re-verify each request against centralized Cedar policies rather than relying on standing/session-based access, aligning with Zero Trust principles. — [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/saas-multitenant-api-access-authorization/avp.html)

## 🛡️ Reliability
- **[Retries]** Implement retry with exponential backoff for Verified Permissions API calls so that transient throttling under per-second RPS quotas doesn't surface as user-facing failures. — [doc](https://docs.aws.amazon.com/verifiedpermissions/latest/userguide/quotas.html)
- **[Quota planning]** Review adjustable RPS quotas (for example, `BatchIsAuthorized`, `CreatePolicy`) and request increases proactively ahead of expected traffic growth for a policy store. — [doc](https://docs.aws.amazon.com/verifiedpermissions/latest/userguide/quotas.html)
- **[Multi-tenant quotas]** Factor per-policy-store resource and request-rate quotas into your shared-vs-per-tenant policy store decision, since a shared store means tenants compete for the same quota. — [doc](https://docs.aws.amazon.com/verifiedpermissions/latest/userguide/design-authz-strategy.html)
- **[KMS availability]** Ensure the IAM principal and Verified Permissions itself retain access to any customer managed KMS key used for a policy store, since losing key access can make authorization decisions stale or block reads/updates entirely. — [doc](https://docs.aws.amazon.com/verifiedpermissions/latest/userguide/security-data-protection-cmk.html)

## ⚡ Performance Efficiency
- **[Batch authorization]** Use `BatchIsAuthorized`/`BatchGetPolicy`-style batch calls to resolve multiple authorization decisions (compound authorization or UI permission checks) in a single API round trip instead of one call per action. — [doc](https://aws.amazon.com/blogs/security/use-amazon-verified-permissions-for-fine-grained-authorization-at-scale/)
- **[Response caching]** Cache authorization responses at the policy enforcement point (for example, API Gateway or a Lambda authorizer) for requests with high cache-hit potential, but avoid caching when policies must take effect immediately or are time-dependent. — [doc](https://aws.amazon.com/blogs/security/use-amazon-verified-permissions-for-fine-grained-authorization-at-scale/)
- **[Entity grouping]** Group related principals or resources into entities referenced with `in` statements instead of writing one policy per entity, reducing the number of policies evaluated and easing scale-out. — [doc](https://aws.amazon.com/blogs/security/get-the-best-out-of-amazon-verified-permissions-by-using-fine-grained-authorization-methods/)
- **[Policy templates]** Use policy templates to avoid duplicating near-identical policies across many principals or resources, centralizing updates to one place. — [doc](https://aws.amazon.com/blogs/security/get-the-best-out-of-amazon-verified-permissions-by-using-fine-grained-authorization-methods/)
- **[Context vs. external data]** Prefer passing additional authorization data via the request `context` parameter (or JWT claims) over fetching external data mid-decision, since context/claims avoid extra lookups and latency. — [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/saas-multitenant-api-access-authorization/best-practices.html)

## 💰 Cost Optimization
- **[Batch authorization]** Combine multiple authorization checks into batch API calls rather than issuing many individual requests, reducing overall request volume for compound and UI-permission use cases. — [doc](https://aws.amazon.com/blogs/security/use-amazon-verified-permissions-for-fine-grained-authorization-at-scale/)
- **[Response caching]** Cache authorization decisions at the enforcement layer for predictable, repeated access patterns to reduce redundant calls to the service. — [doc](https://aws.amazon.com/blogs/security/use-amazon-verified-permissions-for-fine-grained-authorization-at-scale/)
- **[Managed policy engine]** Use Verified Permissions instead of building and operating a custom authorization/policy engine, since the managed service removes the operational burden of running that infrastructure yourself. — [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/saas-multitenant-api-access-authorization/best-practices.html)

## ⚙️ Operational Excellence
- **[CI/CD]** Provision and update policy stores and policies through a CI/CD pipeline using the AWS SDK rather than manual console changes, to reduce operator error and drift. — [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/saas-multitenant-api-access-authorization/best-practices.html)
- **[Observability]** Route access-attempt logs to AWS CloudTrail, Amazon CloudWatch log groups, S3, or Amazon Data Firehose so security and audit teams can respond quickly to incidents and compliance requests. — [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/saas-multitenant-api-access-authorization/best-practices.html)
- **[Audit trail]** Enable AWS CloudTrail for Verified Permissions to capture console and API activity (including `IsAuthorized` and policy-management calls), and deliver the trail to an S3 bucket for durable audit history. — [doc](https://docs.aws.amazon.com/verifiedpermissions/latest/userguide/monitoring-cloudtrail.html)
- **[Design first]** Design the authorization model (principals, resources, actions, and their relationships) from the desired user experience before writing Cedar policy statements, rather than starting from the policy language itself. — [doc](https://docs.aws.amazon.com/verifiedpermissions/latest/userguide/design-authz-strategy.html)
- **[Schema and identity sources]** When connecting an Amazon Cognito or OIDC identity source, update the policy store schema so Cedar policies can reference the token attributes/claims your identity source maps into principals and groups. — [doc](https://docs.aws.amazon.com/verifiedpermissions/latest/userguide/cognito-create.html)
