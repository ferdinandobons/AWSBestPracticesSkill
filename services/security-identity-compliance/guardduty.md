# Amazon GuardDuty — Best Practices

## Common scenarios
- Continuous threat detection across CloudTrail, VPC Flow Logs, and DNS logs        → Security, Operational Excellence
- Centralized, organization-wide threat monitoring across a multi-account AWS estate        → Security, Operational Excellence
- Runtime and malware protection for EC2, EKS, ECS, S3, RDS, and Lambda workloads        → Security, Reliability
- Automated triage and remediation of security findings        → Operational Excellence, Cost Optimization

## 🔒 Security
- **[multi-account]** Designate a dedicated security-tooling account (not the AWS Organizations management account) as the GuardDuty delegated administrator, following least-privilege principles — centralizes findings while avoiding unnecessary privileges on the management account. [doc](https://docs.aws.amazon.com/help-panel/guardduty/latest/console/settings-admin.html)
- **[multi-account]** Integrate GuardDuty with AWS Organizations and configure auto-enable so new member accounts are automatically protected as they are created — closes coverage gaps from manually onboarded accounts. [doc](https://docs.aws.amazon.com/guardduty/latest/ug/guardduty_organizations.html)
- **[coverage]** Enable GuardDuty in every supported AWS Region, even Regions with no active workloads, since findings can surface unauthorized activity anywhere in the account — reduces blind spots for account-level threats such as compromised credentials. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/transitioning-to-multiple-aws-accounts/security-incident-response.html)
- **[protection plans]** Enable the protection plans matched to your workload profile — S3 Protection and Malware Protection for EC2/S3 for storage and compute, EKS Protection and Runtime Monitoring for containers, RDS Protection for databases, and Lambda Protection for serverless — foundational logs alone do not cover these attack surfaces. [doc](https://aws.amazon.com/guardduty/features/)
- **[IAM]** Grant `AmazonGuardDutyFullAccess` only to administrators who configure the service, and `AmazonGuardDutyReadOnlyAccess` to security/audit staff who only need to view findings — applies least privilege to who can change detection configuration. [doc](https://aws.amazon.com/blogs/mt/provisioning-access-to-security-and-audit-teams-in-an-aws-multi-account-environment-created-by-aws-control-tower/)
- **[IAM]** Scope custom IAM policies for GuardDuty to specific actions and resources rather than wildcard permissions, and validate policies with IAM Access Analyzer — limits blast radius if a principal with GuardDuty permissions is compromised. [doc](https://docs.aws.amazon.com/guardduty/latest/ug/security_iam_id-based-policy-examples.html)
- **[threat lists]** Prefer entity lists over legacy IP address lists for trusted and threat sources, since entity lists use direct S3 access with a single IAM permission and avoid per-Region IAM policy size growth — simplifies and hardens list management at scale. [doc](https://docs.aws.amazon.com/guardduty/latest/ug/guardduty_upload-lists.html)
- **[integration]** Integrate GuardDuty with AWS Security Hub CSPM so findings are aggregated with other security services (Config, IAM Access Analyzer, Network Firewall) into one normalized view — improves signal correlation across your security posture. [doc](https://docs.aws.amazon.com/guardduty/latest/ug/securityhub-integration.html)
- **[healthcare/regulated]** Enable GuardDuty in accounts that host or process sensitive data (for example PHI) to add intelligent threat detection as part of infrastructure protection controls — supports compliance-driven monitoring requirements. [doc](https://docs.aws.amazon.com/wellarchitected/latest/healthcare-industry-lens/infrastructure-protection.html)
- **[network]** Require TLS 1.2 (prefer TLS 1.3) and cipher suites with perfect forward secrecy (such as ECDHE) for clients calling the GuardDuty API — protects API traffic in transit. [doc](https://docs.aws.amazon.com/guardduty/latest/ug/infrastructure-security.html)

## 🛡️ Reliability
- **[architecture]** Treat GuardDuty as a Regional service and enable it independently in each Region you operate in or want visibility into, relying on the underlying Availability Zone architecture for durability — a single Region's configuration does not protect other Regions. [doc](https://docs.aws.amazon.com/guardduty/latest/ug/disaster-recovery-resiliency.html)
- **[multi-account]** Confirm Region-specific feature availability before relying on a given finding type or protection plan (for example RDS Protection is unavailable in some Regions and partitions), and design detection coverage accordingly — avoids assuming uniform capability across all Regions. [doc](https://docs.aws.amazon.com/guardduty/latest/ug/guardduty_regions.html)
- **[findings pipeline]** Route findings through Amazon EventBridge to durable, redundant targets (Lambda, SNS, Security Hub) instead of relying solely on console review — ensures findings are not lost if operators are unavailable to check the console. [doc](https://docs.aws.amazon.com/guardduty/latest/ug/guardduty_findings_eventbridge.html)
- **[aggregation]** Aggregate findings cross-Region using EventBridge, S3, or Security Hub's cross-Region aggregation capability — provides a resilient, single view of findings even if you operate across many Regions. [doc](https://aws.amazon.com/guardduty/faqs/)

## 💰 Cost Optimization
- **[Regions]** Enable GuardDuty broadly across Regions despite the fact that pricing is based on analyzed event volume — GuardDuty remains an effective and cost-efficient detection tool even in low-traffic Regions. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/transitioning-to-multiple-aws-accounts/security-incident-response.html)
- **[free trial]** Take advantage of the 30-day free trial granted per account/per Region when enabling GuardDuty or a new protection plan, and use the console's estimated usage cost view during the trial to forecast ongoing spend before committing — avoids cost surprises after trial expiration. [doc](https://docs.aws.amazon.com/guardduty/latest/ug/guardduty-pricing.html)
- **[Runtime Monitoring]** Enable GuardDuty Runtime Monitoring instead of leaving standalone VPC Flow Log analysis running on the same EC2 instances — GuardDuty does not double-charge for VPC Flow Log analysis on instances where the runtime security agent is active. [doc](https://aws.amazon.com/guardduty/faqs/)
- **[Malware Protection for Backup]** Use tiered scanning frequency (full vs. incremental scans) based on data criticality for GuardDuty Malware Protection for AWS Backup — incremental scans only process changed data since the previous backup, reducing scanned-data costs versus scanning full backups every time. [doc](https://aws.amazon.com/blogs/storage/scan-backups-for-malware-with-amazon-guardduty-malware-protection-for-aws-backup/)
- **[findings noise]** Build suppression rules reactively, only for finding types you have repeatedly confirmed as false positives (such as authorized vulnerability scanners or bastion hosts) — reduces downstream processing/storage cost in Security Hub, S3, or a SIEM without discarding genuine threats. [doc](https://docs.aws.amazon.com/guardduty/latest/ug/findings_suppression-rule.html)
- **[cross-Region noise]** Suppress duplicate findings tied to global resources (such as IAM-related findings) in every Region except your designated Security Hub aggregation Region — avoids paying to process and store redundant copies of the same finding. [doc](https://aws.amazon.com/blogs/security/best-practices-for-cross-region-aggregation-of-security-findings/)

## ⚙️ Operational Excellence
- **[multi-account]** Use the GuardDuty delegated administrator model with AWS Organizations so one team can view findings and manage protection plans across all member accounts from a single place — avoids fragmented, per-account manual configuration. [doc](https://docs.aws.amazon.com/guardduty/latest/ug/guardduty_organizations.html)
- **[automation]** Connect GuardDuty findings to Amazon EventBridge rules that trigger Lambda functions for automated remediation (for example, tightening a security group after a probing finding) — shortens response time for well-understood finding types. [doc](https://aws.amazon.com/guardduty/faqs/)
- **[automation]** Integrate high-severity GuardDuty findings with AWS Systems Manager Incident Manager (or an equivalent incident response tool) via EventBridge — ensures critical findings trigger a structured response process rather than depending on manual monitoring. [doc](https://aws.amazon.com/blogs/security/how-to-automate-incident-response-to-security-events-with-aws-systems-manager-incident-manager/)
- **[aggregation]** Enable the automatic Security Hub CSPM integration so GuardDuty findings feed a unified security posture view alongside other services, instead of monitoring GuardDuty in isolation — simplifies triage workflows and reporting for security teams. [doc](https://aws.amazon.com/blogs/mt/centrally-detect-and-investigate-security-findings-with-aws-organizations-integrations/)
- **[EKS operations]** Let GuardDuty manage deployment and updates of its security agent across EKS clusters (enable `EKS_ADDON_MANAGEMENT`) rather than manually installing and updating agents per cluster — reduces operational overhead of keeping runtime monitoring current. [doc](https://docs.aws.amazon.com/guardduty/latest/ug/eks-runtime-monitoring-configuration-standalone-acc.html)
- **[investigation]** Combine GuardDuty findings with Amazon Detective for deeper investigation of malware and related findings on EC2 and container workloads — speeds up root-cause analysis beyond what a single finding provides. [doc](https://aws.amazon.com/blogs/publicsector/detect-and-investigate-amazon-ec2-malware-with-amazon-guardduty-and-amazon-detective/)
