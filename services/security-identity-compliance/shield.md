# AWS Shield — Best Practices

## Common scenarios
- Protecting internet-facing web applications from network/transport-layer DDoS attacks        → Security, Reliability
- Guaranteeing availability and getting 24/7 expert response for business-critical applications        → Reliability, Operational Excellence
- Mitigating application-layer (L7) HTTP flood attacks in combination with AWS WAF        → Security, Reliability
- Centrally enforcing DDoS protection across many accounts and resources in an organization        → Security, Operational Excellence

## 🔒 Security
- **[architecture]** Front internet-facing applications with Amazon CloudFront and Amazon Route 53 wherever possible, since traffic to them is inspected by a fully inline DDoS mitigation system with no observable latency — provides the fastest detection and mitigation of layer 3/4 DDoS events. [doc](https://docs.aws.amazon.com/waf/latest/developerguide/ddos-event-detection-infrastructure.html)
- **[attack surface]** Restrict origin access to only your CloudFront distribution (using the CloudFront managed prefix list in security groups, or by keeping firewall allow-lists updated with CloudFront's IP ranges for on-premises origins) — prevents attackers from bypassing edge protections by hitting the origin directly. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/how-to-enhance-cloudfront-origin-security-of-on-premise-web-servers-using-third-party-firewalls/)
- **[application layer]** Associate an AWS WAF web ACL with Shield Advanced protected resources and add a rate-based rule to limit requests from any single IP address — provides a baseline defense against HTTP request floods. [doc](https://docs.aws.amazon.com/waf/latest/developerguide/ddos-automatic-app-layer-response-config.html)
- **[application layer]** Enable Shield Advanced automatic application layer DDoS mitigation so Shield Advanced can create, evaluate, and deploy rules in your web ACL to count or block requests it determines are part of an attack — reduces application downtime without requiring manual intervention during an event. [doc](https://docs.aws.amazon.com/waf/latest/developerguide/ddos-automatic-app-layer-response.html)
- **[subscription]** Subscribe to Shield Advanced for applications that require guaranteed availability, rapid access to DDoS mitigation experts, or predictable costs during an attack — Shield Standard's automatic protections cover only the most common network/transport-layer events. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/security-controls-by-caf-capability/infrastructure-controls.html)
- **[governance]** Use AWS Firewall Manager to create a Shield Advanced security policy across an AWS Organizations structure so new and existing in-scope resources are automatically protected and audited for compliance — prevents unprotected resources from being deployed by individual teams. [doc](https://docs.aws.amazon.com/waf/latest/developerguide/shield-policies.html)

## 🛡️ Reliability
- **[incident response]** Enable proactive engagement and configure Route 53 health checks on protected resources so the Shield Response Team (SRT) contacts you directly, and can start troubleshooting, as soon as a health check goes unhealthy during a suspected DDoS event — cuts response time versus waiting for you to open a support case. [doc](https://docs.aws.amazon.com/waf/latest/developerguide/ddos-srt-proactive-engagement.html)
- **[health checks]** Ensure Route 53 health checks associated with Shield Advanced protections accurately reflect the true health of the application (not just a shallow ping) — the SRT and Shield Advanced use health check state to decide whether an event qualifies for proactive engagement. [doc](https://docs.aws.amazon.com/waf/latest/developerguide/ddos-srt-proactive-engagement.html)
- **[architecture]** Design DDoS-resilient reference architectures per the AWS Best Practices for DDoS Resiliency whitepaper, layering CloudFront, Global Accelerator, and Route 53 in front of regional resources — gives access to globally distributed edge capacity that can absorb large volumetric attacks. [doc](https://docs.aws.amazon.com/whitepapers/latest/aws-best-practices-ddos-resiliency/aws-best-practices-ddos-resiliency.html)
- **[cost protection]** Enable Shield Advanced DDoS cost protection on Amazon EC2, ELB, CloudFront, Global Accelerator, and Route 53 so you can request service credits if these resources scale up in response to a DDoS attack — avoids unplanned cost impact from attack-driven autoscaling. [doc](https://aws.amazon.com/shield/faqs/)
- **[support]** Maintain a Business or Enterprise Support plan for any application protected by Shield Advanced so you can escalate to the Shield Response Team during an active event — SRT engagement is gated on support tier. [doc](https://docs.aws.amazon.com/waf/latest/developerguide/ddos-srt-proactive-engagement.html)

## ⚙️ Operational Excellence
- **[monitoring]** Create CloudWatch alarms on the `DDoSDetected` metric for each Shield Advanced protected resource, and build dashboards from Shield Advanced's detection, mitigation, and top-contributor metrics — gives early, actionable notice of a possible attack in progress. [doc](https://docs.aws.amazon.com/waf/latest/developerguide/shield-metrics.html)
- **[monitoring]** Use Shield Advanced attack flow logs (delivered to S3, CloudWatch Logs, or Data Firehose) to capture traffic metadata during events — supports post-event forensics, verifying mitigation effectiveness, and feeding existing analysis pipelines. [doc](https://aws.amazon.com/blogs/security/gain-visibility-into-ddos-attacks-with-flow-logs-in-aws-shield-advanced/)
- **[centralized visibility]** Aggregate Shield Advanced findings into AWS Security Hub and use Firewall Manager auto-remediation to bring noncompliant resources into protection across multiple accounts — simplifies tracking DDoS events and protection coverage organization-wide. [doc](https://aws.amazon.com/blogs/security/set-up-centralized-monitoring-for-ddos-events-and-auto-remediate-noncompliant-resources/)
- **[contacts]** Configure and keep current the proactive engagement contact list (up to ten contacts, with notes for the SRT) so the right people can be reached quickly during an event — stale contact details delay SRT response when it matters most. [doc](https://docs.aws.amazon.com/waf/latest/developerguide/ddos-srt-proactive-engagement.html)
- **[situational awareness]** Review the Shield Advanced global threat environment dashboard periodically to understand DDoS trends and top attack vectors against CloudFront, ELB, and Route 53 — informs architecture decisions that improve resiliency over time. [doc](https://aws.amazon.com/blogs/security/now-you-can-monitor-ddos-attack-trends-with-aws-shield-advanced/)
