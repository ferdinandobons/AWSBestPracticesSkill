# Amazon Managed Blockchain — Best Practices

## Common scenarios
- Multi-party permissioned ledger for supply chain / trade provenance (Hyperledger Fabric)        → Security, Reliability
- Reading and submitting transactions to public Ethereum or Bitcoin without running own node infrastructure (AMB Access / Query)        → Reliability, Performance Efficiency, Cost Optimization
- Consortium networks where members join/leave and enforce endorsement policies        → Security, Operational Excellence
- Decentralized applications (dapps) needing low-latency blockchain data queries        → Performance Efficiency

## 🔒 Security
- **[Identity]** Require human users to federate through an identity provider (for example IAM Identity Center) and use temporary credentials rather than long-lived IAM user keys to access AMB resources. — Reduces the blast radius of leaked credentials. [doc](https://docs.aws.amazon.com/managed-blockchain/latest/hyperledger-fabric-dev/security-iam.html)
- **[Identity]** Avoid using the AWS account root user for day-to-day AMB operations; create IAM users/roles with least-privilege policies scoped to specific network, member, and node actions. — Limits what any single compromised credential can do. [doc](https://docs.aws.amazon.com/managed-blockchain/latest/hyperledger-fabric-dev/security-iam.html)
- **[Data protection]** Enable multi-factor authentication (MFA) on all accounts that can manage Managed Blockchain networks. — Adds a second factor against credential theft. [doc](https://docs.aws.amazon.com/managed-blockchain/latest/hyperledger-fabric-dev/managed-blockchain-data-protection.html)
- **[Data protection]** Enforce TLS 1.2 (prefer TLS 1.3) for all client-to-network communication with peer nodes, ordering service, and CA endpoints. — Protects transaction and certificate data in transit. [doc](https://docs.aws.amazon.com/managed-blockchain/latest/hyperledger-fabric-dev/managed-blockchain-data-protection.html)
- **[Data protection]** Use a customer managed AWS KMS key (instead of the default AWS owned key) for members and peer nodes that hold sensitive or regulated ledger data, and monitor key usage. — Gives full control over key rotation, access policy, and auditability of decryption grants. [doc](https://docs.aws.amazon.com/managed-blockchain/latest/hyperledger-fabric-dev/managed-blockchain-encryption-at-rest.html)
- **[Data protection]** Never place confidential or sensitive information (customer PII, credentials) in tags, free-form name fields, or URLs passed to the API/console. — This data can surface in billing and diagnostic logs. [doc](https://docs.aws.amazon.com/managed-blockchain/latest/hyperledger-fabric-dev/managed-blockchain-data-protection.html)
- **[Network]** Access Hyperledger Fabric network components only through interface VPC endpoints, and scope security groups tightly: allow inbound only from known member client IP ranges/security groups on the specific ordering-service, CA, and peer-event ports, with SSH restricted to trusted clients. — Keeps ledger and CA traffic off the public internet and limits lateral access between members. [doc](https://docs.aws.amazon.com/managed-blockchain/latest/hyperledger-fabric-dev/managed-blockchain-security-sgs.html)
- **[Governance]** Define a clear governance model (endorsement policies, member onboarding/removal process, working group for data/API standards) before inviting other organizations into a consortium network. — A permissioned blockchain still needs operational governance; it is not self-managing. [doc](https://aws.amazon.com/blogs/industries/improve-aircraft-parts-provenance-using-amazon-managed-blockchain/)
- **[Auditability]** Enable AWS CloudTrail trails for Managed Blockchain management (and, for Ethereum, data) events and deliver logs to a dedicated S3 bucket. — Provides an immutable audit record of who created/deleted networks, members, and nodes. [doc](https://docs.aws.amazon.com/managed-blockchain/latest/hyperledger-fabric-dev/logging-using-cloudtrail.html)

## 🛡️ Reliability
- **[Availability]** Distribute peer nodes (Hyperledger Fabric) or Ethereum nodes for a member across multiple Availability Zones rather than concentrating them in one AZ. — Design the blockchain application for resiliency against AZ-level failures. [doc](https://docs.aws.amazon.com/managed-blockchain/latest/hyperledger-fabric-dev/managed-blockchain-hyperledger-peer-nodes.html)
- **[Availability]** Provision at least one peer node per member, and add additional peer nodes for members that must keep operating (query, endorse, submit transactions) if one node fails. — A new node can restore ledger data from existing peers, but a member with zero healthy nodes cannot transact. [doc](https://docs.aws.amazon.com/managed-blockchain/latest/hyperledger-fabric-dev/network-components.html)
- **[Capacity]** Monitor peer node CPU/memory/storage utilization and resize the AMB Access instance type (or add nodes) before a node reaches capacity and is marked Failed/added to the deny list. — Nodes that exhaust memory or storage stop participating in the network. [doc](https://docs.aws.amazon.com/managed-blockchain/latest/hyperledger-fabric-dev/managed-blockchain-view-peer-node.html)
- **[Ethereum]** Size Ethereum nodes at `bc.t3.xlarge` or larger for production workloads. — AWS recommends this minimum size for optimal performance and minimal degradation. [doc](https://docs.aws.amazon.com/managed-blockchain/latest/ethereum-dev/ethereum-nodes.html)
- **[Transactions]** Keep individual transaction payloads under 4 MB. — Transactions at or above this size are rejected by the network. [doc](https://docs.aws.amazon.com/managed-blockchain/latest/hyperledger-fabric-dev/network-components.html)

## ⚡ Performance Efficiency
- **[Instance sizing]** Choose the AMB Access instance family/size (bc.t3, bc.m5, bc.c5) based on measured transaction throughput and CPU/memory needs rather than defaulting to the smallest type, and scale up when metrics show sustained pressure. — Matches compute/memory capacity to actual chaincode and endorsement workload. [doc](https://aws.amazon.com/managed-blockchain/features/)
- **[Scaling]** Add peer nodes to a member when transaction volume grows instead of over-provisioning a single large node upfront. — Managed Blockchain APIs let you add nodes on demand to meet changing throughput needs. [doc](https://aws.amazon.com/managed-blockchain/features/)
- **[Query workloads]** For read-heavy public-blockchain use cases (looking up balances, transaction history, UTXOs), use AMB Query instead of operating dedicated nodes. — Delivers blockchain data with sub-second latency without managing node infrastructure. [doc](https://aws.amazon.com/about-aws/whats-new/2024/03/amb-query-listing-utxos-bitcoin-blockchain-networks/)

## 💰 Cost Optimization
- **[Right-sizing]** Continuously monitor peer/Ethereum node CPU and memory utilization and downsize or consolidate nodes that are consistently underutilized. — Avoids paying for larger instance types than the workload requires. [doc](https://docs.aws.amazon.com/managed-blockchain/latest/hyperledger-fabric-dev/managed-blockchain-hyperledger-peer-nodes.html)
- **[Access pattern]** Prefer AMB Access/Query multi-tenant API operations over dedicated nodes for workloads that only need periodic reads or standard JSON-RPC calls to public networks. — Shared, serverless node infrastructure avoids the cost of running and maintaining always-on dedicated nodes for light workloads. [doc](https://docs.aws.amazon.com/managed-blockchain/latest/hyperledger-fabric-dev/what-is-managed-blockchain.html)
- **[Governance]** Agree on a transparent cost/membership model (who pays for membership, peer nodes, storage, data written) with consortium members before onboarding them. — Membership, node, storage, and data-transfer charges apply per member, so unclear cost allocation causes friction as the network scales. [doc](https://aws.amazon.com/blogs/industries/improve-aircraft-parts-provenance-using-amazon-managed-blockchain/)

## ⚙️ Operational Excellence
- **[Monitoring]** Enable peer node, chaincode, and certificate authority (CA) logging to Amazon CloudWatch Logs, and create metric filters/alarms on error patterns. — Lets you debug rejected proposals, endorsement failures, and certificate/enrollment issues, and alerts you to problems automatically. [doc](https://docs.aws.amazon.com/managed-blockchain/latest/hyperledger-fabric-dev/monitoring-cloudwatch-logs.html)
- **[Auditing]** Create an AWS CloudTrail trail covering all Regions for Managed Blockchain so every `CreateNode`, `GetNode`, `DeleteNetwork`, and similar API call is captured with caller identity. — Gives an ongoing, queryable record of administrative activity across the network. [doc](https://docs.aws.amazon.com/managed-blockchain/latest/hyperledger-fabric-dev/logging-using-cloudtrail.html)
- **[Resource organization]** Tag networks, members, and nodes consistently (for example CostCenter, Environment, Project) using AWS tagging best practices. — Improves resource identification, cost allocation reporting, and enables tag-based IAM access control. [doc](https://docs.aws.amazon.com/managed-blockchain/latest/hyperledger-fabric-dev/tagging.html)
- **[Infrastructure as code]** Manage networks, members, and nodes with the AWS CLI and AWS CloudFormation rather than manual console changes. — Managed Blockchain's support for CLI/CloudFormation makes network setup and changes repeatable and auditable. [doc](https://aws.amazon.com/blogs/industries/improve-aircraft-parts-provenance-using-amazon-managed-blockchain/)
