# AWS Ground Station — Best Practices

## Common scenarios
- Downlinking Earth observation / SAR imagery to S3 or EC2 for near-real-time processing        → Performance Efficiency, Reliability
- Telemetry, tracking, and command (TT&C) of a satellite fleet across multiple ground stations        → Reliability, Operational Excellence
- Uplinking commands and validating them via uplink echo before transmission        → Security, Reliability
- Testing mission software and scheduling workflows before production satellite contacts        → Operational Excellence, Cost Optimization

## 🔒 Security
- **[IAM]** Start from AWS managed policies and narrow down to least-privilege customer-managed policies scoped to specific `groundstation:` actions and resource ARNs — reduces blast radius of over-broad access. [doc](https://docs.aws.amazon.com/ground-station/latest/ug/security_iam_id-based-policy-examples.html)
- **[IAM]** Use IAM Access Analyzer to validate identity-based policies for AWS Ground Station and require MFA for users who can schedule or modify contacts — catches overly permissive statements before they're attached. [doc](https://docs.aws.amazon.com/ground-station/latest/ug/security_iam_id-based-policy-examples.html)
- **[IAM]** Scope the trust policy on the dataflow endpoint service role (assumed by `groundstation.amazonaws.com`) down to the specific dataflow endpoint group ARN once it exists, rather than leaving a wildcard in place permanently — limits which AWS Ground Station resources can assume the role. [doc](https://docs.aws.amazon.com/ground-station/latest/ug/how-it-works.dataflow-endpoint-group.html)
- **[S3 delivery]** Restrict the IAM role AWS Ground Station assumes to write to your S3 bucket with an `aws:SourceAccount` and `aws:SourceArn` condition scoped to your account's `s3-recording` configs — prevents confused-deputy access from other accounts. [doc](https://docs.aws.amazon.com/ground-station/latest/ug/examples.pbs-to-s3.html)
- **[Encryption]** Use a customer-managed KMS key instead of the AWS-owned default key for ephemeris data and azimuth/elevation tracking data when you need control over key policies, rotation, and auditability. [doc](https://docs.aws.amazon.com/ground-station/latest/ug/security.encryption-at-rest.html)
- **[Encryption]** Implement end-to-end encryption for satellite command and telemetry data in addition to the transport-level encryption AWS provides, per AWS Ground Station security guidance. [doc](https://docs.aws.amazon.com/ground-station/latest/ug/security.html)
- **[Network]** Scope the two dataflow-endpoint security groups tightly: the customer-side security group should allow only UDP traffic from the AWS Ground Station-side security group on the exact ports declared in the dataflow endpoint group. [doc](https://docs.aws.amazon.com/ground-station/latest/ug/dataflows.vpc-configuration.html)
- **[Network]** Place dataflow-endpoint ENIs in subnets that are mutually routable if you span multiple Availability Zones, and size subnets with enough free IPs for one address per dataflow — misconfiguration here causes contact setup failures, not just cost. [doc](https://docs.aws.amazon.com/ground-station/latest/ug/dataflows.vpc-configuration.html)
- **[Access control]** Rely on AWS Ground Station's contact-level isolation — access to the antenna gateway during a contact is limited to the dataflow endpoints you configured — and avoid widening dataflow endpoint security groups beyond what a contact requires. [doc](https://aws.amazon.com/ground-station/faqs/)

## 🛡️ Reliability
- **[Scheduling]** Treat `ReserveContact` as asynchronous (result available within 5 minutes, typically under 1) and handle `FAILED_TO_SCHEDULE` explicitly in mission software instead of assuming success — antenna and dataflow-endpoint availability are evaluated at request time. [doc](https://docs.aws.amazon.com/ground-station/latest/ug/how-it-works.html)
- **[Scheduling]** Because the antenna network is a shared, time-sliced resource, build retry/backoff and alternate-contact logic into mission planning software to handle contention with other customers instead of assuming a requested window is always available. [doc](https://aws.amazon.com/blogs/publicsector/test-and-integrate-ground-segment-with-aws-ground-station-digital-twin/)
- **[Config changes]** Remember that `ReserveContact` snapshots the mission profile and config resources at reservation time — updating a `MissionProfile` or `Config` afterward will not affect an already-scheduled contact; cancel and re-reserve if the change must apply. [doc](https://docs.aws.amazon.com/ground-station/latest/ug/contacts.html)
- **[Compute readiness]** Launch and fully initialize the EC2 receiver instance (with DataDefender or the AWS Ground Station Agent, and the SDR configured) with sufficient lead time before pre-pass so it is ready before data starts arriving from the antenna. [doc](https://aws.amazon.com/blogs/publicsector/test-and-integrate-ground-segment-with-aws-ground-station-digital-twin/)
- **[Resilience testing]** Use AWS Fault Injection Service to inject faults into ground segment components and validate resiliency and availability before relying on the path for production contacts. [doc](https://aws.amazon.com/blogs/publicsector/test-and-integrate-ground-segment-with-aws-ground-station-digital-twin/)
- **[Geographic diversity]** Take advantage of the global network of antenna locations to schedule contacts across multiple sites for the same satellite, reducing single-site dependency for critical passes. [doc](https://docs.aws.amazon.com/ground-station/latest/ug/aws-ground-station-antenna-locations.html)
- **[Tracking]** Set `Autotrack` to `PREFERRED` (autotracking with fallback to program track) or `REQUIRED` in the TrackingConfig based on mission tolerance for signal loss, instead of leaving tracking behavior to defaults. [doc](https://aws.amazon.com/blogs/publicsector/creating-aws-cloudformation-templates-for-aws-ground-station-digif-workloads/)

## ⚡ Performance Efficiency
- **[Data delivery]** Use cross-region data delivery to operate multiple simultaneous contacts from a single AWS Region without paying extra inter-region transfer for antenna access, leveraging AWS's low-latency global backbone. [doc](https://aws.amazon.com/ground-station/features/)
- **[Data delivery]** Choose EC2 (VITA-49, real-time) delivery over S3 (PCAP, delivered every 30 seconds) when your workload needs to begin processing within seconds of reception at the antenna; use S3 delivery when asynchronous batch/post-processing is sufficient. [doc](https://docs.aws.amazon.com/ground-station/latest/ug/getting-started.step2.html)
- **[Networking]** Keep dataflow-endpoint subnets in the same Availability Zone where possible when multiple subnets are involved, since cross-AZ placement of ENIs can add data-transfer overhead. [doc](https://docs.aws.amazon.com/ground-station/latest/ug/how-it-works.dataflow-endpoint-group.html)

## 💰 Cost Optimization
- **[Scheduling mode]** Use On-Demand scheduling for variable or exploratory workloads, and switch to Reserved scheduling (12-month monthly commitment) once usage is predictable to get a discounted rate and improved scheduling priority. [doc](https://docs.aws.amazon.com/ground-station/latest/ug/contacts.billing.html)
- **[Cancellations]** Avoid cancelling or rescheduling contacts close to the contact start time, since late cancellations can still incur cost. [doc](https://docs.aws.amazon.com/ground-station/latest/ug/contacts.html)
- **[Testing]** Validate scheduling logic, configs, and error handling using the digital twin feature instead of consuming production antenna capacity for integration testing and CI/CD regression runs. [doc](https://docs.aws.amazon.com/ground-station/latest/ug/digital-twin.html)
- **[Compute]** Start EC2 receiver/processing instances only shortly before a scheduled contact and stop them afterward (automated via EventBridge) instead of running them continuously. [doc](https://docs.aws.amazon.com/ground-station/latest/ug/monitoring.automating-events.html)

## ⚙️ Operational Excellence
- **[Automation]** Create EventBridge rules on AWS Ground Station events (Contact State Change, Dataflow Endpoint Group State Change, Ephemeris Status Change, S3 Upload Complete, Dataflow Endpoint Group Health Status Change) to drive Lambda automation, SNS notifications, or auditing instead of polling. [doc](https://docs.aws.amazon.com/eventbridge/latest/ref/events-ref-groundstation.html)
- **[Automation]** Use EventBridge-triggered Lambda functions to start EC2 receiver instances ahead of a contact and stop them afterward, and to verify that expected receiver instances still exist. [doc](https://aws.amazon.com/blogs/publicsector/creating-aws-cloudformation-templates-for-aws-ground-station-digif-workloads/)
- **[Change management]** Run changes to mission control software, IaC, or AWS Ground Station configs through the digital twin environment in an isolated non-production account before promoting to production, since the digital twin emits the same EventBridge events and API responses as production. [doc](https://docs.aws.amazon.com/ground-station/latest/ug/digital-twin.html)
- **[IaC]** Define dataflow endpoint groups, configs (tracking, antenna downlink/uplink, S3 recording), IAM roles, and security groups as CloudFormation templates so ground-segment infrastructure is reproducible and reviewable. [doc](https://aws.amazon.com/blogs/publicsector/creating-aws-cloudformation-templates-for-aws-ground-station-digif-workloads/)
- **[Observability]** Correlate contact state-change events with AWS Cost and Usage Report data (for example via Glue, Athena, and QuickSight) to give operators a single view of contact status alongside billing. [doc](https://aws.amazon.com/blogs/publicsector/simplify-satellite-contact-reporting-with-the-aws-ground-station-contacts-dashboard/)
