# Amazon WorkSpaces Secure Browser — Best Practices

## Common scenarios
- Isolating access to internal websites and SaaS applications without a full virtual desktop        → Security, Cost Optimization
- Providing kiosk-style, single-app browser access from thin clients or unmanaged devices        → Security, Operational Excellence
- Safe, disposable browsing of external/untrusted websites to contain malware and phishing risk        → Security, Reliability
- Giving contact-center or BYOD users controlled access to corporate web apps with DLP controls        → Security, Cost Optimization

## 🔒 Security
- **[IAM]** Implement least-privilege access and create specific IAM roles (Full Access or Read Only) scoped to WorkSpaces Secure Browser actions rather than granting broad permissions. [doc](https://docs.aws.amazon.com/workspaces-web/latest/adminguide/security-best-practices.html)
- **[IAM]** Start from AWS managed policies and progressively narrow to customer-managed, least-privilege policies scoped to specific actions, resources, and conditions. [doc](https://docs.aws.amazon.com/workspaces-web/latest/adminguide/security_iam_service-with-iam-policy-best-practices.html)
- **[IAM]** Use IAM Access Analyzer to validate identity-based policies for WorkSpaces Secure Browser against IAM best practices before deploying them. [doc](https://docs.aws.amazon.com/workspaces-web/latest/adminguide/security_iam_service-with-iam-policy-best-practices.html)
- **[IAM]** Require multi-factor authentication (MFA) for IAM users or the root user in accounts that manage WorkSpaces Secure Browser resources. [doc](https://docs.aws.amazon.com/workspaces-web/latest/adminguide/security_iam_service-with-iam-policy-best-practices.html)
- **[Credentials]** Be cautious about how, when, and to whom you share web portal domains and user credentials — anyone can reach the portal URL, but a session cannot start without a valid credential. [doc](https://docs.aws.amazon.com/workspaces-web/latest/adminguide/security-best-practices.html)
- **[Encryption]** Provide a customer-managed KMS key so WorkSpaces Secure Browser encrypts portal customization data, browser policy, and session logs at rest with keys you control. [doc](https://docs.aws.amazon.com/workspaces-web/latest/adminguide/data-encryption.html)
- **[Network access]** Use IP access control groups to restrict portal access to trusted networks — WorkSpaces Secure Browser checks the user's IP before authentication and terminates the session if the IP changes mid-session. [doc](https://docs.aws.amazon.com/workspaces-web/latest/adminguide/ip-access-controls.html)
- **[Data exfiltration]** Restrict clipboard, file transfer, and local printing permissions in user settings to the minimum needed (for example, paste-into-session only, upload-only) to reduce the risk of sensitive data leaving through the session. [doc](https://docs.aws.amazon.com/workspaces-web/latest/adminguide/user-settings.html)
- **[Website access]** Use URL filtering (allowlists, blocklists, and content-category filtering) to control which internal sites, SaaS apps, or external destinations a session can reach. [doc](https://docs.aws.amazon.com/workspaces-web/latest/adminguide/web-content-filtering.html)
- **[Detective controls]** Use AWS CloudTrail and Amazon CloudWatch together to detect and track access history, process logs, and identify anomalies as part of your detective-control strategy. [doc](https://docs.aws.amazon.com/workspaces-web/latest/adminguide/security-best-practices.html)
- **[Logging]** Turn on user activity logging (Session Logger to S3, or User Access Logging via Kinesis Data Streams) to record session start/stop and URL-visit events for compliance and investigation. [doc](https://docs.aws.amazon.com/workspaces-web/latest/adminguide/monitoring-logging.html)
- **[Transport security]** Require clients to support TLS 1.2 (and prefer TLS 1.3) with cipher suites offering perfect forward secrecy, such as ECDHE, when connecting to WorkSpaces Secure Browser API endpoints. [doc](https://docs.aws.amazon.com/workspaces-web/latest/adminguide/infrastructure-security.html)
- **[Data protection]** Never place confidential or sensitive information (such as customer emails) into free-form fields like portal or resource names, since this data can surface in billing or diagnostic logs. [doc](https://docs.aws.amazon.com/workspaces-web/latest/adminguide/data-protection.html)
- **[Trust store]** Add required private certificate authorities or site-specific certificates to the portal's trust store so users can reach internal sites without certificate errors, rather than disabling certificate validation. [doc](https://docs.aws.amazon.com/help-panel/workspaces-web/latest/console/hp-network-trust.html)

## 🛡️ Reliability
- **[VPC/subnets]** Configure at least two (ideally three) private subnets in different Availability Zones per portal to support high availability, since WorkSpaces Secure Browser requires this minimum and recommends more for resilience. [doc](https://docs.aws.amazon.com/workspaces-web/latest/adminguide/network-settings.html)
- **[VPC/subnets]** Size subnet CIDR ranges to provide enough client IP addresses for peak concurrent sessions plus headroom for growth, since each session consumes a unique IP from the configured subnets. [doc](https://docs.aws.amazon.com/workspaces-web/latest/adminguide/vpc-setup-recommendations.html)
- **[VPC/ENIs]** Track your account's elastic network interface (ENI) limit against the maximum concurrent-session capacity you configure, since each session consumes its own ENI and the default Region limit is 5000. [doc](https://docs.aws.amazon.com/workspaces-web/latest/adminguide/vpc-setup-recommendations.html)
- **[Service quotas]** Proactively request service quota increases for web portals and maximum concurrent sessions per instance type before you need the capacity, since default quotas (for example, 25 concurrent sessions for standard.regular) can throttle user sign-in during growth. [doc](https://docs.aws.amazon.com/workspaces-web/latest/adminguide/request-service-quota.html)
- **[VPC connectivity]** Ensure every subnet used by the portal has a stable connection to whatever internal content (VPC-hosted, on-premises, or internet) users need to reach. [doc](https://docs.aws.amazon.com/workspaces-web/latest/adminguide/vpc-reqs.html)
- **[Monitoring]** Monitor `SessionAttempt`, `SessionSuccess`, and `SessionFailure` CloudWatch metrics per portal to detect session-establishment problems before they affect many users. [doc](https://docs.aws.amazon.com/workspaces-web/latest/adminguide/monitoring-cloudwatch.html)
- **[Backup awareness]** Account for the fact that cross-AZ/cross-region content backup, encrypted backups, and automatic backups are not provided by the service — design your own backup/DR approach for anything session-dependent. [doc](https://docs.aws.amazon.com/workspaces-web/latest/adminguide/disaster-recovery-resiliency.html)

## ⚡ Performance Efficiency
- **[Instance sizing]** Choose `standard.regular` for static text/image-heavy sites, `standard.large` when workloads need more memory (multi-tab/window switching, remote dataset queries), and `standard.xlarge` for real-time audio/video, screen sharing, or large file processing. [doc](https://aws.amazon.com/blogs/desktop-and-application-streaming/announcing-new-larger-instance-types-for-amazon-workspaces-web/)
- **[Monitoring]** Track `GlobalCpuPercent`, `GlobalMemoryPercent`, `DisplayLatency`, and `InputLatency` CloudWatch metrics per portal/user to catch under-provisioned instances or degraded streaming experience. [doc](https://docs.aws.amazon.com/workspaces-web/latest/adminguide/monitoring-cloudwatch.html)
- **[Networking]** Provision a NAT gateway in a public subnet when sessions need general internet access, since streaming instances themselves are never assigned public IPs. [doc](https://docs.aws.amazon.com/workspaces-web/latest/adminguide/vpc-setup-recommendations.html)
- **[Capacity planning]** Capture usage data and hardware requirements from your existing environment (or a proof of concept) before selecting instance type, rather than guessing at sizing. [doc](https://docs.aws.amazon.com/wellarchitected/latest/end-user-computing-lens/euccost05-bp02.html)

## 💰 Cost Optimization
- **[Service selection]** Choose WorkSpaces Secure Browser instead of a full persistent desktop or application-streaming fleet when users only need access to web-based/SaaS applications, since it is the lower-cost option for that use case. [doc](https://docs.aws.amazon.com/wellarchitected/latest/end-user-computing-lens/euccost05-bp02.html)
- **[Rightsizing]** Rightsize instance type and maximum concurrent sessions per portal using captured usage metrics (CloudWatch, WorkSpaces dashboards) instead of defaulting to larger instance types, and adjust as workload patterns change. [doc](https://docs.aws.amazon.com/wellarchitected/latest/end-user-computing-lens/euccost05-bp03.html)
- **[Portal design]** Segment users across multiple portals by workload profile (for example, a static-content portal on `standard.regular` versus an audio/video portal on `standard.xlarge`) so each portal's instance type matches its actual needs instead of over-provisioning one portal for the heaviest use case. [doc](https://docs.aws.amazon.com/workspaces-web/latest/adminguide/limit-example.html)
- **[Cost allocation]** Tag WorkSpaces Secure Browser resources to allocate cost to business owners, departments, or functions, and establish a tagging strategy before deployment. [doc](https://docs.aws.amazon.com/wellarchitected/latest/end-user-computing-lens/cost-effective-resources.html)

## ⚙️ Operational Excellence
- **[Monitoring]** Use CloudWatch metrics (`ActiveSession`, `SessionIdleDisconnect`, latency and utilization metrics) together with CloudWatch Logs and CloudTrail as the core toolset for observing portal health and API activity. [doc](https://docs.aws.amazon.com/workspaces-web/latest/adminguide/monitoring-overview.html)
- **[Logging]** Configure alarms on `SessionLoggerTargetNotFoundError` and `SessionLoggerAccessDeniedError` to catch accidental deletion of the log-delivery destination or its permissions. [doc](https://docs.aws.amazon.com/workspaces-web/latest/adminguide/session-events-session-logger.html)
- **[Patching]** Rely on WorkSpaces Secure Browser to patch and update the underlying Chrome and Linux platform automatically, but still monitor usage and configure the service according to your own policies and guidelines, since that responsibility remains yours. [doc](https://docs.aws.amazon.com/workspaces-web/latest/adminguide/vulnerability-analysis-and-management.html)
- **[Quota management]** Identify and request increases for the two quotas most likely to block growth — number of web portals and maximum concurrent sessions per instance type per Region — ahead of anticipated demand. [doc](https://docs.aws.amazon.com/workspaces-web/latest/adminguide/request-service-quota.html)
- **[Auditability]** Enable AWS CloudTrail to capture WorkSpaces Secure Browser API calls, including caller identity and source IP, for operational troubleshooting and audit history. [doc](https://docs.aws.amazon.com/workspaces-web/latest/adminguide/monitoring-overview.html)
