# Amazon WorkSpaces — Best Practices

## Common scenarios
- Providing persistent, cloud-hosted Windows/Linux desktops for remote or hybrid employees        → Security, Reliability, Cost Optimization
- Extending secure desktop access to contractors, BPO agents, or other external users        → Security, Operational Excellence
- Supporting graphics- or compute-intensive desktop workloads (CAD, dev workstations)        → Performance Efficiency, Cost Optimization
- Building business-continuity/disaster-recovery desktop capacity across Regions        → Reliability, Sustainability

## 🔒 Security
- **[Identity]** Rely on a centralized authentication system (SAML 2.0 IdP or AWS Directory Service/Active Directory) and enforce multi-factor authentication for all users rather than relying on a single credential factor. [doc](https://docs.aws.amazon.com/wellarchitected/latest/end-user-computing-lens/eucsec06-bp01.html)
- **[Identity]** Manage entitlements through Active Directory or IdP groups instead of per-user grants, so access reviews and role changes propagate consistently and reduce operational error. [doc](https://docs.aws.amazon.com/wellarchitected/latest/end-user-computing-lens/eucsec06-bp01.html)
- **[Identity]** Restrict user permissions on the WorkSpace itself to the minimum required for the role, and disable local administrator rights unless there is a specific, reviewed business need. [doc](https://docs.aws.amazon.com/wellarchitected/latest/end-user-computing-lens/security-foundations.html)
- **[Network]** Define IP access control groups so WorkSpaces can only be reached from trusted CIDR ranges (e.g. corporate VPN egress IPs), and route NAT traffic through a static IP so the group rule remains accurate. [doc](https://docs.aws.amazon.com/workspaces/latest/adminguide/amazon-workspaces-ip-access-control-groups.html)
- **[Network]** For external or less-trusted users, layer IP access control groups with device-trust certificates (client and root certificates) so only authorized, company-issued devices can establish a session. [doc](https://aws.amazon.com/blogs/desktop-and-application-streaming/designing-deploying-secure-environments-using-amazon-workspaces/)
- **[Encryption]** Enable root- and user-volume encryption using AWS KMS when you launch each WorkSpace — encryption can only be set at launch time and cannot be changed afterward. [doc](https://docs.aws.amazon.com/workspaces/latest/adminguide/encrypt-workspaces.html)
- **[Encryption]** Use TLS 1.2 (and prefer TLS 1.3) for all API/console communication with WorkSpaces resources, and use a FIPS endpoint if FIPS 140-3 validated cryptography is required. [doc](https://docs.aws.amazon.com/workspaces/latest/adminguide/data-protection.html)
- **[Detection]** Monitor and aggregate user access logs (successful and unsuccessful sign-ins, including smart-card events) in a central location using AWS CloudTrail, so anomalous access patterns can be investigated. [doc](https://docs.aws.amazon.com/wellarchitected/latest/end-user-computing-lens/detection.html)
- **[Detection]** Install endpoint protection software on WorkSpaces instances to detect unexpected behavior, and periodically verify instances remain configured to your baseline to catch configuration drift. [doc](https://docs.aws.amazon.com/wellarchitected/latest/end-user-computing-lens/detection.html)
- **[Governance]** Identify external stakeholders and their regulatory/compliance requirements up front, and identify distinct user risk-profile groups so security controls (device trust, MFA, network restrictions) match each group's risk. [doc](https://docs.aws.amazon.com/wellarchitected/latest/end-user-computing-lens/security-foundations.html)
- **[Data protection]** Never place confidential or sensitive information (customer emails, credentials) in tags or free-form name fields, since this data can surface in billing or diagnostic logs. [doc](https://docs.aws.amazon.com/workspaces/latest/adminguide/data-protection.html)

## 🛡️ Reliability
- **[Architecture]** Add redundancy and remove single points of failure by assuming failures will occur — WorkSpaces already deploys across a minimum of two Availability Zones by default within a Region. [doc](https://docs.aws.amazon.com/wellarchitected/latest/end-user-computing-lens/eucrel01-bp01.html)
- **[Disaster recovery]** Use cross-Region redirection with Route 53 DNS failover routing and CloudWatch alarm health checks to redirect users to WorkSpaces in a secondary Region when the primary Region is unreachable. [doc](https://docs.aws.amazon.com/workspaces/latest/adminguide/business-continuity.html)
- **[Disaster recovery]** For workloads that need automated standby capacity, use WorkSpaces Multi-Region Resilience to provision standby WorkSpaces in a secondary Region from the latest primary bundle configuration, rather than building custom failover automation. [doc](https://aws.amazon.com/blogs/desktop-and-application-streaming/advancing-business-continuity-with-amazon-workspaces-multi-region-resilience/)
- **[Disaster recovery]** Replicate custom WorkSpace images/bundles and directory-level security configuration (internet access, local admin, IP access controls) to the standby Region as soon as they change in the primary Region, so failover doesn't degrade your security posture. [doc](https://aws.amazon.com/blogs/desktop-and-application-streaming/temp-building-for-business-continuity-with-amazon-workspaces-and-aws-directory-services/)
- **[Testing]** Regularly test failure-recovery procedures (e.g. with AWS Fault Injection Service) and validate that backup/DR plans actually restore operations within your required recovery time. [doc](https://docs.aws.amazon.com/wellarchitected/latest/end-user-computing-lens/eucrel01-bp01.html)
- **[Monitoring]** Set up CloudWatch monitoring and alarms on WorkSpaces health and performance so degraded instances are detected and remediated automatically rather than through user reports. [doc](https://docs.aws.amazon.com/wellarchitected/latest/end-user-computing-lens/eucrel01-bp01.html)

## ⚡ Performance Efficiency
- **[Protocol selection]** Choose the streaming protocol (WSP/DCV vs. PCoIP) based on your decision criteria — bidirectional audio/video needs, zero-client endpoints, OS of the local device, and smart-card requirements — rather than defaulting to one protocol. [doc](https://docs.aws.amazon.com/whitepapers/latest/best-practices-deploying-amazon-workspaces/design-considerations.html)
- **[Protocol selection]** Keep round-trip network latency under 100 ms (max 250 ms) for PCoIP and under 250 ms for DCV; connections degrade significantly between 250–400 ms and PCoIP disconnects above 375 ms, so deploy WorkSpaces in the Region closest to end users when latency is consistently degraded. [doc](https://docs.aws.amazon.com/workspaces/latest/adminguide/bundle-options.html)
- **[Bundle sizing]** Test candidate bundles in a representative environment running your users' actual applications before standardizing on a bundle, rather than sizing from specs alone. [doc](https://docs.aws.amazon.com/workspaces/latest/adminguide/bundle-options.html)
- **[Bundle sizing]** For video-conferencing-heavy use cases, use Power, PowerPro, GeneralPurpose.4xlarge, or GeneralPurpose.8xlarge bundles for the best experience. [doc](https://docs.aws.amazon.com/workspaces/latest/adminguide/amazon-workspaces-networking.html)
- **[Right-sizing]** Capture baseline resource-utilization metrics from an existing reference environment before choosing bundles, then continually monitor and adjust bundle/instance type as usage patterns change. [doc](https://docs.aws.amazon.com/wellarchitected/latest/end-user-computing-lens/euccost05-bp03.html)
- **[Monitoring]** Monitor WorkSpaces using the CloudWatch automatic dashboard and CloudWatch metrics to identify unhealthy instances, track resource utilization over time, and catch anomalies before they affect users. [doc](https://docs.aws.amazon.com/workspaces/latest/adminguide/cloudwatch-dashboard.html)
- **[Monitoring]** Establish baseline KPIs and routinely re-measure performance against them, since without a baseline you cannot tell whether the environment is meeting objectives as demand changes. [doc](https://docs.aws.amazon.com/wellarchitected/latest/end-user-computing-lens/process-and-culture.html)
- **[Testing]** Conduct realistic end-to-end performance testing aligned with your organization's actual use cases and constraints before and after major changes, instead of relying on synthetic benchmarks alone. [doc](https://docs.aws.amazon.com/wellarchitected/latest/end-user-computing-lens/process-and-culture.html)

## 💰 Cost Optimization
- **[Running mode]** Choose AutoStop for WorkSpaces used intermittently and AlwaysOn for WorkSpaces used as a full-time primary desktop — as a rule of thumb, AlwaysOn becomes more cost-effective once usage exceeds roughly 80 hours per month for a given bundle. [doc](https://aws.amazon.com/blogs/desktop-and-application-streaming/how-to-manage-amazon-workspaces-cost-optimization-at-scale/)
- **[Automation]** Deploy Cost Optimizer for Amazon WorkSpaces to analyze usage data and automatically (or via dry-run reporting) convert WorkSpaces to the most cost-effective running mode. [doc](https://docs.aws.amazon.com/wellarchitected/latest/end-user-computing-lens/optimize-over-time.html)
- **[Right-sizing]** Right-size bundles to actual hardware requirements instead of over-provisioning, using captured usage metrics, and revisit sizing continually as application and OS requirements evolve. [doc](https://docs.aws.amazon.com/wellarchitected/latest/end-user-computing-lens/euccost05-bp03.html)
- **[Visibility]** Enable AWS Cost and Usage Reports with resource IDs for WorkSpaces so you can analyze spend at the level of an individual WorkSpace, and build a cost/usage dashboard (e.g. via the Cloud Intelligence Dashboards) for ongoing visibility. [doc](https://docs.aws.amazon.com/wellarchitected/latest/end-user-computing-lens/euccost02-bp01.html)
- **[Lifecycle]** Identify and terminate idle or unused WorkSpaces — since AutoStop WorkSpaces retain a fixed monthly infrastructure cost until deleted, stopping the instance alone doesn't eliminate that cost. [doc](https://docs.aws.amazon.com/workspaces/latest/adminguide/running-mode.html)
- **[Licensing]** Use open-source operating systems where use cases allow, or bring your own OS licenses where available, to reduce per-desktop licensing costs. [doc](https://docs.aws.amazon.com/wellarchitected/latest/end-user-computing-lens/design-principles.html)

## ⚙️ Operational Excellence
- **[Monitoring]** Establish and monitor service metrics and KPIs for the WorkSpaces environment, and monitor WorkSpaces Personal-specific CloudWatch metrics as part of routine operations. [doc](https://docs.aws.amazon.com/wellarchitected/latest/end-user-computing-lens/process-and-culture.html)
- **[Monitoring]** Use CloudWatch Internet Monitor with WorkSpaces directories to get visibility into ISP/city-level internet performance issues affecting end users, beyond what CloudWatch metrics alone show. [doc](https://docs.aws.amazon.com/workspaces/latest/adminguide/amazon-workspaces-monitoring.html)
- **[Auditability]** Use CloudTrail to log WorkSpaces API calls (who made a request, from what IP, and when) to support troubleshooting and audit requirements, including sign-in events for smart-card users. [doc](https://docs.aws.amazon.com/workspaces/latest/adminguide/amazon-workspaces-monitoring.html)
- **[Change management]** Track and version-control changes to your EUC solution design, and implement automated processes so environment/image updates can be deployed and rolled back repeatably. [doc](https://docs.aws.amazon.com/wellarchitected/latest/end-user-computing-lens/evolve.html)
- **[Continuous improvement]** Follow AWS EUC news/release sources and allocate time for your team to evaluate new WorkSpaces features so the environment benefits from performance and capability improvements over time. [doc](https://docs.aws.amazon.com/wellarchitected/latest/end-user-computing-lens/process-and-culture.html)
- **[Sizing changes]** Align instance type/bundle size to fleet-wide workload needs on an ongoing basis, and where appropriate enable self-service WorkSpaces management so users can request changes without a full operations ticket cycle. [doc](https://docs.aws.amazon.com/wellarchitected/latest/end-user-computing-lens/process-and-culture.html)

## 🌱 Sustainability
- **[Running mode]** Choose AutoStop mode below the usage threshold where it's more efficient — stopped instances lower the carbon footprint associated with running WorkSpaces compared to AlwaysOn — and use Cost Optimizer for Amazon WorkSpaces to set the appropriate mode based on actual usage. [doc](https://docs.aws.amazon.com/wellarchitected/latest/end-user-computing-lens/eucsus01-bp02.html)
- **[Bundle selection]** Select the instance type/bundle to match the software requirements and persona of each user group, rather than standardizing everyone on the largest bundle, to avoid over-provisioning compute that isn't used. [doc](https://docs.aws.amazon.com/wellarchitected/latest/end-user-computing-lens/alignment-to-demand.html)
- **[Idle resources]** Adapt AutoStop and idle-disconnect timeout settings so disconnected sessions release compute resources promptly instead of running unnecessarily. [doc](https://docs.aws.amazon.com/wellarchitected/latest/end-user-computing-lens/alignment-to-demand.html)
- **[Image management]** Optimize machine image creation, copying, and sharing across environments (development, testing, production) to avoid duplicating image-build resource consumption unnecessarily. [doc](https://docs.aws.amazon.com/wellarchitected/latest/end-user-computing-lens/software-and-architecture.html)
