# Amazon WorkSpaces Applications — Best Practices

## Common scenarios
- Streaming line-of-business or ISV desktop applications to any device        → Security, Performance Efficiency
- Delivering GPU/CPU-intensive applications (CAD, engineering, creative tools) without local hardware        → Performance Efficiency, Cost Optimization
- Providing occasional or bursty access to specialized apps for a subset of a workforce        → Cost Optimization, Reliability
- Enabling federated, agentless application access for contractors or external users        → Security, Operational Excellence

## 🔒 Security
- **[Endpoint hardening]** Avoid installing full antivirus suites that scan ephemeral system volumes on streaming instances — this adds overhead without improving security since instances are purged after each session. [doc](https://docs.aws.amazon.com/appstream2/latest/developerguide/security-best-practices.html)
- **[Persistent data]** Keep home folders and application settings storage in the same AWS Region as your fleets, and rely on the default Amazon S3-managed KMS encryption at rest, restricting administrative access to that bucket. [doc](https://docs.aws.amazon.com/appstream2/latest/developerguide/securing-persistent-data.html)
- **[Persistent data]** When requirements demand more control over user state than S3-backed home folders provide, use an SMB file share (for example FSx for Windows File Server) with encryption at rest and in transit enabled. [doc](https://docs.aws.amazon.com/appstream2/latest/developerguide/securing-persistent-data.html)
- **[IAM]** Follow least-privilege principles when attaching IAM policies to roles used by streaming instances, granting only the specific API actions and resources applications or scripts require. [doc](https://docs.aws.amazon.com/appstream2/latest/developerguide/best-practices-for-using-iam-role-with-streaming-instances.html)
- **[IAM]** Create a distinct IAM role per WorkSpaces Applications resource so permissions can be adjusted for one resource without affecting others. [doc](https://docs.aws.amazon.com/appstream2/latest/developerguide/best-practices-for-using-iam-role-with-streaming-instances.html)
- **[IAM]** Use IAM policy conditions (for example source IP ranges) to limit where streaming-instance role credentials can be used from. [doc](https://docs.aws.amazon.com/appstream2/latest/developerguide/best-practices-for-using-iam-role-with-streaming-instances.html)
- **[IAM]** Restrict administrator access to the Amazon S3 buckets used for home folders and application settings persistence with dedicated IAM policies. [doc](https://docs.aws.amazon.com/appstream2/latest/developerguide/using-services.html)
- **[Network]** Deploy streaming instances in private subnets when they do not need to be reachable from the internet, and provide outbound access through a NAT gateway. [doc](https://docs.aws.amazon.com/appstream2/latest/developerguide/control-network-traffic.html)
- **[Network]** Use security groups to restrict traffic to streaming instances (for example allowing only corporate network ranges), and to isolate traffic between instances and WorkSpaces Applications management services. [doc](https://docs.aws.amazon.com/appstream2/latest/developerguide/control-network-traffic.html)
- **[Network]** Use an interface VPC endpoint to stream from, and call WorkSpaces Applications API operations from, your VPC without traversing the public internet. [doc](https://docs.aws.amazon.com/appstream2/latest/developerguide/control-network-traffic.html)
- **[Network]** Never block the WorkSpaces Applications management network range (198.19.0.0/16) or the required management/streaming ports (8300, 3128, 8000, 8443, 53) and link-local service addresses in firewall, proxy, or antivirus configuration, or streaming and instance management will break. [doc](https://docs.aws.amazon.com/appstream2/latest/developerguide/network-exclusions.html)
- **[Network]** Place a web proxy or content-filtering solution in the egress path when users need public internet access from inside a streaming session, to reduce data-loss risk. [doc](https://docs.aws.amazon.com/appstream2/latest/developerguide/controlling-egress-traffic.html)
- **[Transport security]** Require clients to support TLS 1.2 (TLS 1.3 recommended) with perfect-forward-secrecy cipher suites (DHE/ECDHE) when calling WorkSpaces Applications APIs. [doc](https://docs.aws.amazon.com/appstream2/latest/developerguide/infrastructure-security.html)
- **[Authentication]** Use SAML 2.0 federated sign-in (or certificate-based authentication for AD-joined deployments) so users authenticate with existing corporate credentials, and use user pools only for small deployments under 100 users. [doc](https://docs.aws.amazon.com/appstream2/latest/developerguide/control-network-traffic.html)
- **[Image hygiene]** Never bake secrets, credentials, or other sensitive configuration items into a WorkSpaces Applications image. [doc](https://docs.aws.amazon.com/whitepapers/latest/workspaces-linux-best-practices/appendix-a.html)

## 🛡️ Reliability
- **[Multi-AZ]** Configure fleets across a minimum of two Availability Zones (one subnet per AZ), and use more than two AZs to optimize Fleet Auto Scaling headroom and IP address space for growth. [doc](https://docs.aws.amazon.com/whitepapers/latest/best-practices-for-deploying-amazon-appstream-2/availability-zones.html)
- **[Redundancy]** Assume failures will occur: add redundancy at every architecture layer (network paths, storage, streaming instances) and use Fleet Auto Scaling so instance counts track demand and absorb spikes without degrading the user experience. [doc](https://docs.aws.amazon.com/wellarchitected/latest/end-user-computing-lens/eucrel01-bp01.html)
- **[Disaster recovery]** For regional-level resilience beyond multi-AZ, deploy WorkSpaces Applications in multiple AWS Regions by copying images between Regions and establishing separate fleets. [doc](https://docs.aws.amazon.com/appstream2/latest/developerguide/disaster-recovery-resiliency.html)
- **[Testing]** Regularly test failure-recovery procedures, for example using AWS Fault Injection Service to simulate failure scenarios and validate recovery strategies. [doc](https://docs.aws.amazon.com/wellarchitected/latest/end-user-computing-lens/eucrel01-bp01.html)
- **[Monitoring]** Set up Amazon CloudWatch alarms on fleet and session metrics to detect and automatically remediate issues promptly. [doc](https://docs.aws.amazon.com/appstream2/latest/developerguide/logging-monitoring-alerting.html)

## ⚡ Performance Efficiency
- **[Image optimization]** Run the Image Assistant Optimize step against every application, and supplement it with an optimization manifest for files (plug-ins, components) not automatically detected, to reduce first-launch latency for users. [doc](https://docs.aws.amazon.com/appstream2/latest/developerguide/optimizing-app-launch-performance.html)
- **[Instance selection]** Match instance family (general purpose, compute, memory, or graphics-optimized) to actual application resource needs rather than defaulting to one type, testing candidate types against real workloads such as video conferencing. [doc](https://aws.amazon.com/workspaces/applications/faqs/)
- **[Fleet type]** Choose between single-session and multi-session fleets based on workload profile: single-session for ISV/resource-intensive applications needing dedicated resources, multi-session for consolidation efficiency where the noisy-neighbor effect is acceptable. [doc](https://aws.amazon.com/blogs/desktop-and-application-streaming/key-considerations-for-single-versus-multi-session-fleets-on-amazon-appstream-2-0/)
- **[Right-sizing]** Capture baseline utilization metrics (CPU, RAM, disk, network) from an existing reference environment before choosing an instance type, then continually monitor fleet, instance, and session performance metrics in the WorkSpaces Applications console and CloudWatch to adjust sizing over time. [doc](https://docs.aws.amazon.com/wellarchitected/latest/end-user-computing-lens/euccost05-bp03.html)

## 💰 Cost Optimization
- **[Fleet scaling]** Use Fleet Auto Scaling with scheduled, step, or target-tracking policies so instance counts track real concurrent usage instead of provisioning for peak headcount, avoiding over-provisioning during nights, weekends, and remote-work variance. [doc](https://docs.aws.amazon.com/whitepapers/latest/best-practices-for-deploying-amazon-appstream-2/fleet-auto-scaling-strategies.html)
- **[Idle resource control]** Deploy the Cost Optimizer for Amazon WorkSpaces Applications (or an equivalent tool) to monitor and automatically stop image builders and app blocks that are left running past a defined threshold, since these bill continuously until explicitly stopped. [doc](https://docs.aws.amazon.com/wellarchitected/latest/end-user-computing-lens/euccost07-bp01.html)
- **[Service selection]** Reserve WorkSpaces Applications for CPU/RAM/GPU-intensive or occasionally used applications, and compare estimated costs against alternatives (a lighter WorkSpaces bundle, or WorkSpaces Secure Browser for purely web-based apps) using real usage patterns before committing to a configuration. [doc](https://docs.aws.amazon.com/wellarchitected/latest/end-user-computing-lens/euccost05-bp02.html)
- **[Continuous optimization]** Periodically review fleet utilization after go-live and adjust scaling policies and buffer capacity as usage patterns, applications, or organizational needs change, terminating unused resources. [doc](https://docs.aws.amazon.com/wellarchitected/latest/end-user-computing-lens/optimize-over-time.html)
- **[Image efficiency]** Install only the applications required into a base image and build custom bundles at the lowest viable disk storage capacity to minimize per-image overhead. [doc](https://docs.aws.amazon.com/whitepapers/latest/workspaces-linux-best-practices/appendix-a.html)

## ⚙️ Operational Excellence
- **[Automation]** Script and automate image configuration end-to-end rather than relying on manual builder steps, to enable repeatability and scale as you add or update images. [doc](https://docs.aws.amazon.com/whitepapers/latest/workspaces-linux-best-practices/well-architected-principles-for-image-management.html)
- **[Patching]** Use Managed WorkSpaces Applications Image Updates to keep the operating system, drivers, and WorkSpaces Applications agent current, and select "always use the latest agent version" in Image Assistant so new streaming instances pick up agent updates automatically. [doc](https://docs.aws.amazon.com/appstream2/latest/developerguide/keep-image-updated-managed-image-updates.html)
- **[Patching]** Treat application and dependency patching, Windows OS patching, and WorkSpaces Applications software-component updates as separate, explicit responsibilities and schedule each on its own cadence. [doc](https://docs.aws.amazon.com/whitepapers/latest/best-practices-for-deploying-amazon-appstream-2/image-updates.html)
- **[Image design]** Abstract applications from images and align the number of images to identified user personas so each image serves a discrete, well-understood application set rather than growing unmanageable monolithic images. [doc](https://docs.aws.amazon.com/whitepapers/latest/workspaces-linux-best-practices/well-architected-principles-for-image-management.html)
- **[Logging]** Enable a CloudTrail trail covering WorkSpaces Applications management API calls (fleet, image, stack, session actions) to support auditing and incident investigation. [doc](https://docs.aws.amazon.com/appstream2/latest/developerguide/logging-using-cloudtrail.html)
- **[Monitoring]** Use CloudWatch alarms alongside AWS Trusted Advisor checks (S3 bucket logging, open-access permissions, and versioning for home-folder/app-settings buckets) to maintain visibility into reliability, availability, and security posture. [doc](https://docs.aws.amazon.com/appstream2/latest/developerguide/logging-monitoring-alerting.html)
