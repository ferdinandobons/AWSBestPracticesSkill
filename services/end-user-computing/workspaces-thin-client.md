# Amazon WorkSpaces Thin Client — Best Practices

## Common scenarios
- Provisioning low-cost endpoint devices for call-center and back-office staff        → Security, Cost Optimization
- Centrally managing and patching a fleet of thin client devices        → Operational Excellence, Reliability
- Providing secure, no-local-storage access to WorkSpaces, WorkSpaces Applications, or WorkSpaces Web        → Security
- Planning device lifecycle and migration ahead of AWS service end-of-support        → Operational Excellence, Reliability

## 🔒 Security
- **[identity]** Set up individual users with IAM or IAM Identity Center and grant only the permissions required for their job instead of sharing root credentials. — Limits blast radius if a credential is compromised. [doc](https://docs.aws.amazon.com/workspaces-thin-client/latest/ag/data-protection.html)
- **[identity]** Start from AWS managed policies and then move to least-privilege customer managed policies scoped to specific `thinclient:*` actions and resources. — Reduces the risk of unintended environment or device changes. [doc](https://docs.aws.amazon.com/workspaces-thin-client/latest/ag/security_iam_id-based-policy-examples.html)
- **[identity]** Add IAM policy conditions (for example requiring SSL/TLS or restricting by source service) to further scope WorkSpaces Thin Client access. — Narrows access to only the intended call context. [doc](https://docs.aws.amazon.com/workspaces-thin-client/latest/ag/security_iam_id-based-policy-examples.html)
- **[identity]** Validate IAM policies with IAM Access Analyzer before attaching them. — Catches overly permissive or non-functional statements before they reach production. [doc](https://docs.aws.amazon.com/workspaces-thin-client/latest/ag/security_iam_id-based-policy-examples.html)
- **[identity]** Require multi-factor authentication (MFA) for accounts and add MFA conditions to policies for sensitive API calls. — Adds a second factor against stolen credentials. [doc](https://docs.aws.amazon.com/workspaces-thin-client/latest/ag/data-protection.html)
- **[data protection]** Use TLS 1.2 at minimum (TLS 1.3 recommended) for all communication with AWS resources. — Protects data in transit between devices, the service, and the console. [doc](https://docs.aws.amazon.com/workspaces-thin-client/latest/ag/data-protection.html)
- **[data protection]** Add a customer managed AWS KMS key as a second encryption layer on top of the default AWS owned key encryption for environment and device data. — Gives you control over key policies, rotation, and revocation of access to sensitive fleet data. [doc](https://docs.aws.amazon.com/workspaces-thin-client/latest/ag/encryption-rest.html)
- **[data protection]** Scope customer managed key policies to the specific `thinclient:` service principal and account, and grant only the `DescribeKey`, `GenerateDataKey`, and `Decrypt` actions it needs. — Prevents unrelated principals or services from using the key. [doc](https://docs.aws.amazon.com/workspaces-thin-client/latest/ag/encryption-rest.html)
- **[data protection]** Never place confidential or sensitive information (such as customer emails) into tags or free-form name fields. — Tag and name data can surface in billing and diagnostic logs. [doc](https://docs.aws.amazon.com/workspaces-thin-client/latest/ag/data-protection.html)
- **[data protection]** Enforce least-privilege roles specifically for WorkSpaces Thin Client actions and protect environment activation codes and user credentials from unnecessary sharing. — Physical access to a device alone cannot start a session without a valid activation code and credentials, but leaked codes still enable unauthorized enrollment. [doc](https://docs.aws.amazon.com/workspaces-thin-client/latest/ag/data-encryption.html)
- **[monitoring]** Enable AWS CloudTrail logging for WorkSpaces Thin Client API calls and route logs to a dedicated Amazon S3 bucket or CloudTrail Lake event data store. — Provides an auditable record of who created, modified, or deleted environments and devices. [doc](https://docs.aws.amazon.com/workspaces-thin-client/latest/ag/logging-using-cloudtrail.html)
- **[patching]** Track WorkSpaces Thin Client's cross-integration with the update management processes of the underlying desktop service (WorkSpaces, WorkSpaces Applications, or WorkSpaces Web) since patching responsibility is shared across them. — Ensures vulnerability management coverage extends to the full connection chain, not just the thin client device. [doc](https://docs.aws.amazon.com/workspaces-thin-client/latest/ag/vulnerability-analysis-and-management.html)

## 🛡️ Reliability
- **[resilience]** Rely on the underlying AWS Regions and Availability Zones design for WorkSpaces Thin Client's backend, and pair it with cross-Region resilience features on the connected desktop service. — Availability Zones provide automatic fail-over that a single data center cannot match. [doc](https://docs.aws.amazon.com/workspaces-thin-client/latest/ag/disaster-recovery-resiliency.html)
- **[business continuity]** For business continuity, configure WorkSpaces Cross-Region Redirection with DNS routing policies, or provision a standby WorkSpace via WorkSpaces Multi-Region Resilience, and create a connection alias in the Region used by WorkSpaces Thin Client. — Ensures end users can keep working if the primary desktop Region is disrupted. [doc](https://docs.aws.amazon.com/workspaces-thin-client/latest/ag/disaster-recovery.html)
- **[lifecycle planning]** Plan and execute migration to a supported alternative thin client before March 31, 2027, when AWS ends support for Amazon WorkSpaces Thin Client (new customer onboarding already stopped April 20, 2026). — Devices stop working and the management console becomes inaccessible after the end-of-support date, so migrating early avoids a forced, unplanned cutover. [doc](https://docs.aws.amazon.com/workspaces-thin-client/latest/ag/workspacesthinclient-end-of-support.html)
- **[lifecycle planning]** Inventory all currently managed devices, select a supported alternative thin client vendor, and deregister and securely decommission WorkSpaces Thin Client devices once the transition is complete. — A documented transition plan prevents orphaned devices and gaps in end-user access. [doc](https://docs.aws.amazon.com/workspaces-thin-client/latest/ag/workspacesthinclient-end-of-support.html)

## ⚙️ Operational Excellence
- **[fleet management]** Use the device management service's inventory view to track compliance status, registration state, and active duration across the fleet. — Gives administrators a single source of truth for fleet health. [doc](https://aws.amazon.com/workspaces/thin-client/)
- **[patch management]** Enable "Always keep software up-to-date" on environments and configure a system-wide or custom maintenance window so updates apply automatically on a predictable schedule. — Avoids devices drifting to unsupported software versions between manual pushes. [doc](https://docs.aws.amazon.com/workspaces-thin-client/latest/ag/entering-environment-details.html)
- **[patch management]** Prefer updating software during the scheduled maintenance window rather than pushing updates immediately, unless a critical fix is required. — Immediate updates interrupt active user sessions, while maintenance-window updates minimize disruption. [doc](https://docs.aws.amazon.com/workspaces-thin-client/latest/ag/device-software-update.html)
- **[monitoring]** Monitor WorkSpaces Thin Client device metrics in Amazon CloudWatch and set alarms on key thresholds. — Surfaces device-level issues (connectivity, health) proactively instead of relying on user-reported problems. [doc](https://docs.aws.amazon.com/workspaces-thin-client/latest/ag/cloudwatch-metrics.html)
- **[monitoring]** Combine CloudTrail event history with CloudWatch metrics as complementary monitoring tools — CloudTrail for who-did-what auditing, CloudWatch for fleet health and performance. — Together they cover both governance and operational visibility needs. [doc](https://docs.aws.amazon.com/workspaces-thin-client/latest/ag/monitoring-overview.html)
- **[device retirement]** Deregister unused or decommissioned devices promptly rather than leaving them registered. — Reduces the managed fleet's attack surface and keeps inventory and billing records accurate. [doc](https://docs.aws.amazon.com/workspaces-thin-client/latest/ag/workspacesthinclient-end-of-support.html)
