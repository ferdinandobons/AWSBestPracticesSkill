# AWS Deadline Cloud — Best Practices

## Common scenarios
- Cloud-based render farm for 2D/3D animation, VFX, and video rendering        → Cost Optimization, Performance Efficiency
- Multi-artist studio with shared queues, fleets, and job attachments        → Security, Operational Excellence
- Burst rendering with service-managed or customer-managed EC2 fleets        → Cost Optimization, Reliability
- Hybrid on-premises and cloud render nodes with usage-based licensing        → Reliability, Operational Excellence

## 🔒 Security
- **[credentials]** Protect AWS account credentials, set up individual IAM users with least privilege, and require MFA on every account that can access Deadline Cloud. [doc](https://docs.aws.amazon.com/deadline-cloud/latest/developerguide/security-best-practices.html)
- **[job run-as user]** Specify a "Job run as user" (and group) for every queue so jobs run with least-privileged OS permissions, since a queue without one runs as the agent user, which can impersonate any queue user and escalate privileges. [doc](https://docs.aws.amazon.com/deadline-cloud/latest/developerguide/security-best-practices.html)
- **[queue isolation]** Give each queue its own S3 bucket-and-root-prefix combination for job attachments; queues that share a bucket/prefix can let one queue's users read another queue's data. [doc](https://docs.aws.amazon.com/deadline-cloud/latest/developerguide/security-best-practices.html)
- **[farm boundaries]** Treat the farm as your primary security boundary — fleets, queues, and storage profiles can't be shared across farms — and only associate a fleet with queues in the same security boundary, since worker hosts can retain data or running processes visible to other queues on that fleet. [doc](https://docs.aws.amazon.com/deadline-cloud/latest/developerguide/security-best-practices.html)
- **[worker hosts]** Restrict write access to worker agent program files to root (Linux) or Administrator (Windows) only, avoid reusing the same `jobRunAsUser` across queues in different security boundaries, and regularly patch the OS and all installed submitters, adaptors, and worker agents. [doc](https://docs.aws.amazon.com/deadline-cloud/latest/developerguide/security-best-practices.html)
- **[Windows credentials]** Use strong passwords for the Windows queue `jobRunAsUser`, rotate them regularly, and restrict access to (and delete unused) Windows password secrets. [doc](https://docs.aws.amazon.com/deadline-cloud/latest/developerguide/security-best-practices.html)
- **[scheduled task denial]** Deny queue `jobRunAsUser` accounts access to `cron`/`at` on Linux and Task Scheduler on Windows so jobs can't schedule persistent or delayed commands. [doc](https://docs.aws.amazon.com/deadline-cloud/latest/developerguide/security-best-practices.html)
- **[workstation hygiene]** Secure persisted AWS credentials on artist workstations, require federation with an identity provider for temporary credentials, only install trusted software, and only use submitters/adaptors obtained through Deadline Cloud Monitor. [doc](https://docs.aws.amazon.com/deadline-cloud/latest/developerguide/security-best-practices.html)
- **[DNS and routing]** Restrict permissions on local DNS override files (`/etc/hosts`, `C:\Windows\system32\etc\hosts`) and on route tables for both workstations and worker hosts to prevent traffic interception or redirection. [doc](https://docs.aws.amazon.com/deadline-cloud/latest/developerguide/security-best-practices.html)
- **[software integrity]** Verify the authenticity of downloaded Deadline Cloud installers and packages (signatures/checksums) before installing, to protect against file tampering. [doc](https://docs.aws.amazon.com/deadline-cloud/latest/developerguide/security-best-practices.html)
- **[private connectivity]** Use an AWS PrivateLink interface endpoint to access Deadline Cloud from your VPC when you need to keep traffic off the public internet. [doc](https://docs.aws.amazon.com/deadline-cloud/latest/developerguide/security.html)
- **[transport security]** Require TLS 1.2 at minimum (TLS 1.3 recommended) for all communication with AWS resources used by Deadline Cloud. [doc](https://docs.aws.amazon.com/deadline-cloud/latest/developerguide/security-best-practices.html)
- **[free-form fields]** Never place sensitive identifying information (customer account numbers, credentials in URLs) into free-form fields like job or resource names, since this data can be picked up in diagnostic logs. [doc](https://docs.aws.amazon.com/deadline-cloud/latest/developerguide/security-best-practices.html)

## 🛡️ Reliability
- **[resource limits]** Create farm-level *limits* for constrained shared resources (floating licenses, shared file servers) and associate them with queues so new workers don't fail when a resource is already fully claimed by other jobs. [doc](https://docs.aws.amazon.com/deadline-cloud/latest/developerguide/build-job-limits.html)
- **[fleet-to-step matching]** Ensure a queue's associated fleets have a minimum worker specification that meets or exceeds every step's requirements, since a step won't schedule to a fleet whose minimum spec falls short. [doc](https://aws.amazon.com/blogs/media/leveraging-aws-deadline-clouds-customer-managed-fleets/)
- **[customer-managed fleet scaling]** For customer-managed fleets using event-based auto scaling, enable Auto Scaling group scale-in protection, since the ASG has no knowledge of which instance is working on which job. [doc](https://aws.amazon.com/blogs/media/leveraging-aws-deadline-clouds-customer-managed-fleets/)
- **[worker lifecycle]** Configure ASG instances to terminate (not stop) when Deadline Cloud shuts down the worker OS, avoiding instances left in a stopped-but-billable or inconsistent state. [doc](https://aws.amazon.com/blogs/media/leveraging-aws-deadline-clouds-customer-managed-fleets/)
- **[monitoring plan]** Build a monitoring plan covering CloudWatch metrics/alarms, CloudWatch Logs, EventBridge events, and CloudTrail so multi-point failures across the farm, fleets, and queues can be diagnosed quickly. [doc](https://docs.aws.amazon.com/deadline-cloud/latest/developerguide/monitoring-overview.html)
- **[quota awareness]** Review Deadline Cloud service quotas (farms per Region, fleets per farm, on-demand vCPUs per Region, etc.) and request increases proactively for quotas your workload will approach. [doc](https://docs.aws.amazon.com/deadline-cloud/latest/userguide/deadline-cloud-quotas.html)

## ⚡ Performance Efficiency
- **[instance sizing]** Match EC2 instance type to actual job demands — smaller instances cost less per minute but take longer per job, while larger instances finish faster at a higher per-minute cost — and validate against real workload profiling. [doc](https://docs.aws.amazon.com/deadline-cloud/latest/userguide/cost-management.html)
- **[OS choice for render nodes]** Prefer Linux worker instances over Windows when your application and renderer support it, since Windows instances carry a fixed per-vCPU OS charge and are billed hourly rather than by the second. [doc](https://aws.amazon.com/blogs/media/optimizing-computing-performance-with-aws-thinkbox-deadline-part-3/)
- **[standby capacity]** Use a standby worker count to keep a warm pool of idle workers when jobs need to start processing immediately, trading idle-time cost for reduced start latency. [doc](https://docs.aws.amazon.com/deadline-cloud/latest/userguide/auto-scaling-configuration.html)
- **[scale-out tuning]** Tune the scale-out rate and worker idle duration for your fleet to balance how quickly capacity ramps up against how long idle workers linger before shutting down. [doc](https://docs.aws.amazon.com/deadline-cloud/latest/userguide/auto-scaling-configuration.html)

## 💰 Cost Optimization
- **[Spot Instances]** Use EC2 Spot Instances for fleets whenever your jobs can tolerate interruption, since Spot pricing can be up to 90% cheaper than On-Demand while On-Demand remains the choice for workloads that can't be interrupted. [doc](https://docs.aws.amazon.com/deadline-cloud/latest/userguide/cost-management.html)
- **[fleet sizing]** Set a maximum worker count on service-managed fleets so they can't scale past a defined ceiling regardless of queued work, and only set a minimum worker count above zero when reduced start latency is worth paying for idle time. [doc](https://docs.aws.amazon.com/deadline-cloud/latest/userguide/cost-management.html)
- **[Windows instance churn]** Avoid frequently creating and terminating Windows Spot instances for small jobs, since Windows is billed hourly — leaving a fleet running to absorb multiple jobs within the same hour is cheaper than repeated start/stop cycles. [doc](https://aws.amazon.com/blogs/media/optimizing-computing-performance-with-aws-thinkbox-deadline-part-3/)
- **[CloudWatch Logs retention]** Set a retention policy on the `/aws/deadline/*` log groups instead of relying on the default of never expiring, and export logs you need to keep to S3, which is cheaper than long-term CloudWatch Logs storage. [doc](https://docs.aws.amazon.com/deadline-cloud/latest/userguide/cost-management.html)
- **[S3 lifecycle]** Apply S3 Lifecycle rules to automatically delete unused job attachments, assets, and output, and only retain assets currently in use or needed shortly. [doc](https://docs.aws.amazon.com/deadline-cloud/latest/userguide/cost-management.html)
- **[PrivateLink endpoints]** Remove AWS PrivateLink interface endpoints (Deadline Cloud, license endpoints) when not in use, since each is billed hourly and Deadline Cloud typically requires three to five endpoints. [doc](https://docs.aws.amazon.com/deadline-cloud/latest/userguide/cost-management.html)
- **[budget alerts]** Create Deadline Cloud budgets per farm and subscribe to budget-threshold EventBridge events to get notified as spend crosses configurable percentage thresholds, so you can act before a budget is exhausted. [doc](https://docs.aws.amazon.com/deadline-cloud/latest/userguide/budget-threshold-events.html)

## ⚙️ Operational Excellence
- **[monitoring goals]** Define a monitoring plan up front — what to monitor, how often, which tools, and who gets notified — before relying on CloudWatch, CloudWatch Logs, EventBridge, and CloudTrail to track farm health. [doc](https://docs.aws.amazon.com/deadline-cloud/latest/developerguide/monitoring-overview.html)
- **[log volume awareness]** Log only the minimum data required to monitor tasks and workers, since high-volume job output to CloudWatch Logs increases both cost and API throttling risk. [doc](https://docs.aws.amazon.com/deadline-cloud/latest/userguide/cost-management.html)
- **[GetLogEvents quota]** Request a `GetLogEvents` quota increase in CloudWatch before onboarding many artists, and close log-tailing views in Deadline Cloud Monitor when not actively debugging, to avoid hitting read limits. [doc](https://docs.aws.amazon.com/deadline-cloud/latest/developerguide/monitoring-cloudwatch.html)
- **[host configuration scripts]** Treat host configuration script changes as high-risk operational changes — an incorrect configuration can make a worker unstable or stop it from working, and debugging that failure is your responsibility. [doc](https://docs.aws.amazon.com/deadline-cloud/latest/developerguide/security-best-practices.html)
- **[queue-fleet governance]** Document which AWS resources and security boundaries each queue and fleet combination is meant to span before creating them, since farm structure decisions are hard to change safely afterward. [doc](https://docs.aws.amazon.com/deadline-cloud/latest/developerguide/security-best-practices.html)
