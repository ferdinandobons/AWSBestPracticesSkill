# Amazon QuickSight — Best Practices

## Common scenarios
- Self-service BI dashboards shared across business teams        → Security, Cost Optimization
- Embedded analytics inside customer-facing SaaS applications        → Security, Performance Efficiency
- Large-scale, high-concurrency reporting on datasets imported into SPICE        → Performance Efficiency, Cost Optimization
- Governed, auditable BI with row/column-level access control        → Security, Operational Excellence

## 🔒 Security
- **[network access]** Restrict access to HTTPS and secure WebSockets (wss://) only, and if QuickSight must reach an on-premises or non-AWS database, open that server's firewall to the applicable QuickSight IP range rather than to the whole internet. [doc](https://docs.aws.amazon.com/quick/latest/userguide/best-practices-security.html)
- **[data-in-transit]** Require SSL/TLS (certificates signed by a publicly-recognized CA) for all connections from QuickSight to databases, especially over public networks. [doc](https://docs.aws.amazon.com/quick/latest/userguide/best-practices-security.html)
- **[private connectivity]** Use Enterprise edition VPC connections (via an elastic network interface) to reach AWS data sources and on-premises systems (paired with Direct Connect) instead of exposing databases publicly. [doc](https://docs.aws.amazon.com/quick/latest/userguide/best-practices-security.html)
- **[data at rest]** Store imported data in SPICE with encryption at rest, and use customer-managed AWS KMS keys for SPICE and for underlying S3/CloudWatch log encryption when you need auditability and the ability to revoke access immediately. [doc](https://aws.amazon.com/blogs/business-intelligence/best-practices-for-amazon-quicksight-spice-and-direct-query-mode/)
- **[identity federation]** Integrate Active Directory or IAM Identity Center for authentication instead of managing standalone QuickSight credentials, and enable multi-factor authentication where supported. [doc](https://docs.aws.amazon.com/quick/latest/userguide/best-practices-security.html)
- **[fine-grained access]** Apply row-level and column-level security so users and groups only see the rows and fields they're authorized for, sourcing the security rules from the underlying data store (views/tables) rather than ad hoc UI uploads so they stay under change management. [doc](https://aws.amazon.com/blogs/big-data/applying-row-level-and-column-level-security-on-amazon-quicksight-dashboards/)
- **[IP restrictions]** Turn on IP-based access restrictions to limit which networks can reach the QuickSight UI, mobile app, and embedded pages. [doc](https://aws.amazon.com/blogs/big-data/use-ip-restrictions-to-control-access-to-amazon-quicksight/)
- **[least privilege]** Grant only the specific `quicksight:*` asset-management actions (describe/list/search/update permissions) needed for admin console tasks, rather than broad account-level access. [doc](https://aws.amazon.com/blogs/big-data/govern-and-manage-permissions-of-amazon-quicksight-assets-with-the-new-centralized-asset-management-console/)
- **[asset governance]** Organize dashboards, analyses, and datasets into shared folders with owner/viewer/contributor roles (including restricted folders) instead of managing per-user permissions on every asset individually. [doc](https://aws.amazon.com/blogs/big-data/organize-and-share-your-content-with-folders-in-amazon-quicksight/)
- **[embedded apps]** For embedding, use registered-user or anonymous-identity embedding with least-privilege IAM policies and JWT-based authentication (e.g., via Amazon Cognito), and only allow-list the specific domains that need to render QuickSight content. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/patterns/embed-quick-sight-visual-components-into-web-apps-cognito-iac.html)

## 🛡️ Reliability
- **[managed HA]** Rely on QuickSight's AWS-managed infrastructure — Multi-AZ redundancy within its Region and automatic patching — rather than building custom failover for the service itself. [doc](https://docs.aws.amazon.com/quick/latest/userguide/disaster-recovery-resiliency.html)
- **[SPICE durability]** Understand that SPICE stores and encrypts imported data with high availability through redundant copies across multiple Availability Zones in the QuickSight Region. [doc](https://aws.amazon.com/blogs/machine-learning/implement-a-backup-strategy-for-amazon-quick-sight-bi-assets/)
- **[DR planning]** Design an explicit disaster-recovery strategy if you need protection beyond a single Region, since QuickSight's identity/user data lives in one designated Region and both that Region and the dashboard's Region must be available for user access to succeed. [doc](https://aws.amazon.com/blogs/machine-learning/implement-a-backup-strategy-for-amazon-quick-sight-bi-assets/)
- **[backup strategy]** Back up BI assets (data sources, datasets, analyses, dashboards, templates) via the QuickSight APIs/CloudFormation so critical assets can be restored or redeployed into a secondary Region if needed. [doc](https://aws.amazon.com/blogs/machine-learning/implement-a-backup-strategy-for-amazon-quick-sight-bi-assets/)
- **[refresh continuity]** Expect dashboards on SPICE datasets to keep serving the previous snapshot during a scheduled refresh so users aren't interrupted mid-refresh. [doc](https://aws.amazon.com/blogs/business-intelligence/best-practices-for-amazon-quicksight-spice-and-direct-query-mode/)

## ⚡ Performance Efficiency
- **[storage engine]** Use SPICE whenever possible instead of Direct Query — it auto-scales and caches results, giving much higher concurrency and consistently fast response times without customer-managed infrastructure. [doc](https://aws.amazon.com/blogs/big-data/tips-and-tricks-for-high-performant-dashboards-in-amazon-quicksight/)
- **[metadata hygiene]** Remove unused datasets from analyses so unnecessary metadata isn't fetched on every dashboard load. [doc](https://aws.amazon.com/blogs/big-data/tips-and-tricks-for-high-performant-dashboards-in-amazon-quicksight/)
- **[RLS/CLS performance]** Keep row-level and column-level security rule datasets in SPICE (not direct query) and limit the number/complexity of rules, since these lookups run before every visual query; for embedded dashboards, use session tags for row-level security with an anonymous identity to avoid extra lookups. [doc](https://aws.amazon.com/blogs/big-data/tips-and-tricks-for-high-performant-dashboards-in-amazon-quicksight/)
- **[calculated fields]** Move calculations into the data prep stage (rather than the analysis) so non-aggregated, non-parameterized logic is materialized into SPICE, reducing runtime computation on every query. [doc](https://aws.amazon.com/blogs/big-data/tips-and-tricks-for-high-performant-dashboards-in-amazon-quicksight/)
- **[table/pivot design]** Keep tables and pivot tables to a minimal set of rows, columns, metrics, and table calculations, and use the Values field well thoughtfully for raw table views, to reduce query and rendering overhead. [doc](https://aws.amazon.com/blogs/big-data/tips-and-tricks-for-high-performant-dashboards-in-amazon-quicksight/)
- **[embedding performance]** In embedded scenarios, keep user-management flows off the critical rendering path, push heavier workloads to backend services rather than the client, avoid tearing down the embedding iframe on in-app navigation, and use `navigateToDashboard()`/`navigateToSheet()` instead of remounting. [doc](https://aws.amazon.com/blogs/big-data/tips-and-tricks-for-high-performant-dashboards-in-amazon-quicksight/)
- **[chart cardinality]** Use "Hide the other category" only when a dimension's cardinality is below the visual's cutoff limit, to avoid misleading aggregation and unnecessary query overhead. [doc](https://aws.amazon.com/blogs/big-data/tips-and-tricks-for-high-performant-dashboards-in-amazon-quicksight/)

## 💰 Cost Optimization
- **[SPICE vs. direct query]** Prefer SPICE for datasets that are queried repeatedly across many users/dashboards — you pay once for capacity (GB) rather than per query, shifting load off the source data store and cutting both cost and latency. [doc](https://aws.amazon.com/blogs/business-intelligence/best-practices-for-amazon-quicksight-spice-and-direct-query-mode/)
- **[capacity right-sizing]** Regularly review purchased SPICE capacity against actual dataset sizes per Region, and release unused capacity since it's shared across all users in that Region and account. [doc](https://docs.aws.amazon.com/quick/latest/userguide/spice.html)
- **[auto-purchase]** Enable SPICE capacity auto-purchase so QuickSight acquires needed capacity automatically instead of over-provisioning "just in case" or failing ingestion due to under-provisioning. [doc](https://docs.aws.amazon.com/quick/latest/userguide/managing-spice-capacity.html)
- **[license/usage governance]** Use usage and adoption monitoring (active users, sign-in frequency, inactive accounts) to right-size Author/Reader license counts and remove unused accounts. [doc](https://aws.amazon.com/blogs/business-intelligence/create-centralized-monitoring-for-amazon-quick-suite-administration/)
- **[refresh scheduling]** Scope SPICE dataset refreshes to only the data that changed (incremental refresh) and choose refresh frequency based on actual freshness requirements, since both ingestion and any underlying query engine (e.g., Athena) costs scale with refresh volume and frequency. [doc](https://aws.amazon.com/blogs/messaging-and-targeting/deploy-amazon-quicksight-dashboard-for-amazon-pinpoint-engagement-events/)

## ⚙️ Operational Excellence
- **[IaC deployment]** Automate dataset, data source, folder, and permission deployment with AWS CloudFormation to keep configurations consistent and repeatable across environments instead of manual console changes. [doc](https://aws.amazon.com/blogs/big-data/automate-your-validated-dataset-deployment-using-amazon-quicksight-and-aws-cloudformation/)
- **[centralized governance]** Use the QuickSight admin asset management console (and its corresponding least-privilege IAM policy) to audit and update permissions across analyses, dashboards, datasets, and data sources at scale. [doc](https://aws.amazon.com/blogs/big-data/govern-and-manage-permissions-of-amazon-quicksight-assets-with-the-new-centralized-asset-management-console/)
- **[monitoring]** Track per-visual CloudWatch metrics such as `VisualLoadTime` and `VisualLoadErrorCount` (under `AWS/QuickSight/Visual Metrics`) to catch slow or failing dashboard visuals before users report them. [doc](https://docs.aws.amazon.com/quick/latest/userguide/monitoring-quicksight.html)
- **[usage analytics]** Use the built-in analytics/admin dashboards to monitor adoption, dashboard usage, and security-relevant events (blocked messages, failed queries) to guide governance and capacity decisions. [doc](https://aws.amazon.com/blogs/business-intelligence/create-centralized-monitoring-for-amazon-quick-suite-administration/)
- **[change management]** Source row-level and column-level security rules exclusively from the underlying data store (not console uploads or custom SQL) so they're covered by the same change-management process as the rest of your data. [doc](https://aws.amazon.com/blogs/big-data/automate-your-validated-dataset-deployment-using-amazon-quicksight-and-aws-cloudformation/)
