# Amazon OpenSearch Service — Best Practices

## Common scenarios
- Log analytics and observability pipelines ingesting high-volume time-series data        → Performance Efficiency, Cost Optimization
- Full-text and application search for websites and internal tools        → Performance Efficiency, Reliability
- Vector search and retrieval-augmented generation (RAG) for AI applications        → Performance Efficiency, Cost Optimization
- Security information and event management (SIEM) and audit log storage        → Security, Reliability

## 🔒 Security
- **[Access control]** Enable fine-grained access control on every domain to define role-based, index-, document-, and field-level permissions instead of relying on a single domain-wide policy — restricts sensitive data to only authorized users. [doc](https://docs.aws.amazon.com/opensearch-service/latest/developerguide/fgac.html)
- **[Network]** Deploy domains within a VPC rather than with a public endpoint — keeps traffic on the AWS network and adds an extra layer of isolation controlled by security groups. [doc](https://docs.aws.amazon.com/opensearch-service/latest/developerguide/bp.html)
- **[Access policy]** Apply a restrictive resource-based access policy following least privilege, and avoid the anonymous principal (`"Principal": {"AWS": "*"}`) even for VPC-only domains — enforces security in layers beyond network isolation. [doc](https://docs.aws.amazon.com/opensearch-service/latest/developerguide/bp.html)
- **[Encryption]** Enable encryption at rest using AWS KMS (AES-256) — protects sensitive data stored on domain volumes and snapshots from unauthorized access. [doc](https://docs.aws.amazon.com/opensearch-service/latest/developerguide/bp.html)
- **[Encryption]** Enable node-to-node encryption (TLS) — protects data replicated and distributed between cluster nodes, not just client-to-domain traffic. [doc](https://docs.aws.amazon.com/opensearch-service/latest/developerguide/bp.html)
- **[Posture management]** Monitor domain configuration with AWS Security Hub CSPM controls for OpenSearch Service — continuously evaluates resource configuration against security standards and compliance frameworks. [doc](https://docs.aws.amazon.com/opensearch-service/latest/developerguide/bp.html)
- **[Identity federation]** Use fine-grained access control together with IAM master users or Amazon Cognito–federated identities for Dashboards sign-in — avoids relying solely on static internal-database credentials. [doc](https://aws.amazon.com/opensearch-service/features/security/)

## 🛡️ Reliability
- **[High availability]** Deploy production domains using Multi-AZ with Standby, which spreads three dedicated master nodes and two replica shards per index across three Availability Zones (two active, one standby) — raises availability to 99.99%. [doc](https://docs.aws.amazon.com/opensearch-service/latest/developerguide/bp.html)
- **[Master nodes]** Enable three dedicated master nodes sized to match your shard count — offloads cluster management from data nodes, improving stability and enabling some configuration changes without downtime. [doc](https://docs.aws.amazon.com/opensearch-service/latest/developerguide/bp.html)
- **[Replicas]** Configure at least one replica shard per index (two in a three-AZ deployment) — ensures primary and replica shards land on different data nodes and Availability Zones for redundancy. [doc](https://docs.aws.amazon.com/opensearch-service/latest/developerguide/bp.html)
- **[Disaster recovery]** Implement a disaster recovery strategy for OpenSearch Service — combine automated/manual snapshots, Index State Management (ISM) snapshot policies for critical indices, and cross-cluster replication for active-passive or active-active multi-Region resilience. [doc](https://docs.aws.amazon.com/wellarchitected/latest/amazon-opensearch-service-lens/workload-architecture.html)
- **[Multi-Region]** For availability beyond a single Region's SLA, use cross-cluster replication (active-passive, leader/follower) for small or slowly changing datasets, or dual-write ingest pipelines (active-active) for large, fast-changing datasets — test failover as part of your DR process. [doc](https://docs.aws.amazon.com/opensearch-service/latest/developerguide/bp.html)
- **[Quotas]** Regularly review OpenSearch Service service quotas and limits to avoid scaling constraints that interrupt operations as workloads grow. [doc](https://docs.aws.amazon.com/wellarchitected/latest/amazon-opensearch-service-lens/workload-architecture.html)
- **[Sizing]** Use production-appropriate, reliable instance types and sizing for business-critical workloads rather than minimal dev/test configurations. [doc](https://docs.aws.amazon.com/wellarchitected/latest/amazon-opensearch-service-lens/workload-architecture.html)
- **[Ingest flow]** Build flow control/buffering into upstream ingest pipelines and batch writes via the `_bulk` API instead of single-document requests, so traffic spikes don't overwhelm the cluster. [doc](https://docs.aws.amazon.com/opensearch-service/latest/developerguide/bp.html)
- **[Version currency]** Keep domains current with OpenSearch service software updates and engine version upgrades, validating each upgrade on a dedicated development/test domain before applying it to production. [doc](https://docs.aws.amazon.com/opensearch-service/latest/developerguide/bp.html)

## ⚡ Performance Efficiency
- **[Shard sizing]** Size shards to 10–30 GiB for search workloads or 30–50 GiB for log workloads (50 GiB max), and make shard count a multiple of the data node count to keep distribution even and avoid hot nodes. [doc](https://docs.aws.amazon.com/opensearch-service/latest/developerguide/bp.html)
- **[Shard sizing]** Keep shards per node under roughly 25 per GiB of JVM heap, and budget about 1.5 vCPU per active shard when sizing data nodes. [doc](https://docs.aws.amazon.com/opensearch-service/latest/developerguide/bp.html)
- **[Bulk ingestion]** Start bulk request size at 3–5 MiB and tune upward, and enable gzip compression on requests/responses to cut payload size and improve indexing throughput. [doc](https://docs.aws.amazon.com/opensearch-service/latest/developerguide/bp.html)
- **[Refresh interval]** Set `refresh_interval` to 30 seconds or higher on write-heavy indexes — the default 1-second refresh trades indexing throughput for near-real-time search visibility that most workloads don't need. [doc](https://docs.aws.amazon.com/opensearch-service/latest/developerguide/bp.html)
- **[Auto-Tune]** Enable Auto-Tune on all domains so queue sizes, cache sizes, and JVM settings are continuously adjusted based on real cluster metrics, and review its recommendations on a recurring maintenance window. [doc](https://docs.aws.amazon.com/opensearch-service/latest/developerguide/bp.html)
- **[Mappings]** Define explicit index mappings and templates (with `dynamic: strict` for search workloads) instead of relying on dynamic mapping, to prevent unwanted field explosion and centralize settings like shard count and refresh interval. [doc](https://docs.aws.amazon.com/opensearch-service/latest/developerguide/bp.html)
- **[Vector search]** Tune the OpenSearch vector engine's algorithm (HNSW/IVF/exact k-NN), engine (FAISS/Lucene/NMSLIB), and compression technique (scalar, binary, product quantization) to balance search speed, recall quality, and memory cost for your specific workload. [doc](https://aws.amazon.com/blogs/big-data/opensearch-vector-engine-is-now-disk-optimized-for-low-cost-accurate-vector-search/)
- **[Vector search]** Enable GPU-acceleration and Auto-optimize together during vector ingestion to reduce index build time and automatically discover better latency/quality/memory trade-offs for large-scale vector databases. [doc](https://docs.aws.amazon.com/opensearch-service/latest/developerguide/serverless-vector-ingestion.html)
- **[Response size]** Use the `filter_path` parameter to exclude unnecessary fields from bulk responses, reducing payload size without dropping fields needed to identify or retry failures. [doc](https://docs.aws.amazon.com/opensearch-service/latest/developerguide/bp.html)

## 💰 Cost Optimization
- **[Instance selection]** Use the latest-generation instance types for both data and master nodes, and avoid T2/`t3.small` instances in production, which become unstable under sustained load. [doc](https://docs.aws.amazon.com/opensearch-service/latest/developerguide/bp.html)
- **[Storage]** Use the latest Amazon EBS gp3 volumes instead of gp2 — higher baseline IOPS/throughput at lower cost, independently provisionable IOPS and throughput, and larger per-node volume size limits. [doc](https://docs.aws.amazon.com/opensearch-service/latest/developerguide/bp.html)
- **[Storage tiering]** Move time-series/log data to UltraWarm and cold storage tiers (backed by Amazon S3) once a domain approaches roughly 2.5 TiB, and automate the migration with Index State Management. [doc](https://docs.aws.amazon.com/opensearch-service/latest/developerguide/bp.html)
- **[Reserved capacity]** After at least 14 days of stable, observed usage, review AWS Cost Management reservation recommendations and purchase Reserved Instances for steady-state capacity. [doc](https://docs.aws.amazon.com/opensearch-service/latest/developerguide/bp.html)
- **[Index lifecycle]** Regularly identify and remove unused indexes (snapshotting them to S3 first) to reduce shard count and improve resource balance, or automate this with an ISM policy. [doc](https://docs.aws.amazon.com/opensearch-service/latest/developerguide/bp.html)
- **[Cost visibility]** Apply cost allocation tags to OpenSearch Service resources for detailed cost tracking, and actively monitor usage and cost trends to catch inefficiencies before they compound. [doc](https://docs.aws.amazon.com/wellarchitected/latest/amazon-opensearch-service-lens/optimize-over-time.html)
- **[Estimation]** Use the AWS Pricing Calculator to estimate instance and storage costs, including the Amazon S3 cost of manual domain snapshots, before committing to a configuration. [doc](https://docs.aws.amazon.com/wellarchitected/latest/amazon-opensearch-service-lens/optimize-over-time.html)

## ⚙️ Operational Excellence
- **[Monitoring]** Configure recommended CloudWatch alarms on cluster and instance metrics, and regularly review them against actual workload performance. [doc](https://docs.aws.amazon.com/opensearch-service/latest/developerguide/bp.html)
- **[Logging]** Enable search slow logs, indexing slow logs, and error log publishing to CloudWatch Logs for all production domains, and configure logging thresholds — otherwise no logs are captured. [doc](https://docs.aws.amazon.com/opensearch-service/latest/developerguide/bp.html)
- **[Auditing]** Enable audit logs (available once fine-grained access control is on) to track user activity for security and compliance review. [doc](https://docs.aws.amazon.com/opensearch-service/latest/developerguide/bp.html)
- **[Index lifecycle]** Use Index State Management to automate rollovers, snapshots, storage-tier migrations, and deletion of old indexes instead of managing index lifecycle manually. [doc](https://docs.aws.amazon.com/opensearch-service/latest/developerguide/bp.html)
- **[Backups]** Implement a regular snapshot cadence via ISM policies to ensure consistent, automated backups of index data. [doc](https://docs.aws.amazon.com/wellarchitected/latest/amazon-opensearch-service-lens/operational-excellence.html)
- **[Alerting]** Build comprehensive monitoring and alerting that includes log analysis, performance tracking, and real-time notifications for slow queries, security events, and critical errors. [doc](https://docs.aws.amazon.com/wellarchitected/latest/amazon-opensearch-service-lens/operational-excellence.html)
