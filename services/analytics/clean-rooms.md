# AWS Clean Rooms — Best Practices

## Common scenarios
- Multi-party data collaboration for audience overlap and campaign measurement        → Security, Operational Excellence
- Cross-company analytics without copying or exposing raw underlying data        → Security, Reliability
- Privacy-preserving lookalike modeling and identity resolution between partners        → Security, Performance Efficiency
- Regulated or sensitive data-sharing collaborations requiring audit trails        → Security, Operational Excellence

## 🔒 Security
- **[analysis rules]** Configure aggregation, list, or custom analysis rules on every configured table to restrict query shape and output, and specify only the columns and functions actually required for the agreed use case rather than allowlisting broadly. [doc](https://docs.aws.amazon.com/clean-rooms/latest/userguide/best-practices.html)
- **[table scoping]** Create separate configured tables per query use case (e.g., audience planning vs. attribution) instead of one broad table, and use additional configured tables built on the same underlying AWS Glue table to customize which columns/functions are exposed per collaboration. [doc](https://docs.aws.amazon.com/clean-rooms/latest/userguide/best-practices.html)
- **[aggregation thresholds]** Add aggregation constraints on any column that is sensitive at the row level, including columns shared with other members' tables and columns not exposed in the analysis rule, to reduce the risk of re-identification via correlation with outside data. [doc](https://docs.aws.amazon.com/clean-rooms/latest/userguide/best-practices.html)
- **[collaboration governance]** Conduct due diligence and put legal agreements in place with collaborators before joining a collaboration, and periodically review collaborators' configured tables and analysis rules to confirm they still match what was agreed. [doc](https://docs.aws.amazon.com/clean-rooms/latest/userguide/best-practices.html)
- **[testing]** Create test collaborations to validate that analysis rules enforce the intended restrictions before using them with real, sensitive production data. [doc](https://docs.aws.amazon.com/clean-rooms/latest/userguide/best-practices.html)
- **[differential privacy]** Use AWS Clean Rooms Differential Privacy with custom analysis rules to add mathematically-calibrated noise and protect against user re-identification, without requiring in-house differential privacy expertise. [doc](https://aws.amazon.com/clean-rooms/faqs/)
- **[cryptographic computing]** Use Cryptographic Computing for Clean Rooms so data stays encrypted even while queries are processed, for workloads with stringent data-handling requirements. [doc](https://docs.aws.amazon.com/clean-rooms/latest/userguide/what-is.html)
- **[least privilege IAM]** Start from AWS managed policies for AWS Clean Rooms and narrow to custom, resource-scoped policies, applying least-privilege permissions and IAM conditions (e.g., requiring SSL) to control who can create, query, or manage collaborations. [doc](https://docs.aws.amazon.com/clean-rooms/latest/userguide/security_iam_id-based-policy-examples.html)
- **[policy validation]** Validate identity-based policies with IAM Access Analyzer before applying them, and require MFA for users and roles that can manage collaborations or configured tables. [doc](https://docs.aws.amazon.com/clean-rooms/latest/userguide/security_iam_id-based-policy-examples.html)
- **[query/output separation]** Designate distinct collaboration roles (query runner vs. query result receiver) so the member permitted to run queries is not automatically the one who receives results, when that separation fits the use case. [doc](https://aws.amazon.com/clean-rooms/faqs/)
- **[encryption at rest]** Use customer-managed AWS KMS keys (instead of relying solely on AWS-owned keys) to encrypt Clean Rooms ML models, lookalike segment jobs, and ID mapping table contents when you need direct control over key policies and revocation. [doc](https://docs.aws.amazon.com/clean-rooms/latest/userguide/data-protection.html)
- **[encryption in transit]** Require TLS 1.2 (prefer TLS 1.3) with perfect-forward-secrecy cipher suites (e.g., ECDHE) for all client connections to the AWS Clean Rooms API. [doc](https://docs.aws.amazon.com/clean-rooms/latest/userguide/infrastructure-security.html)
- **[sensitive data hygiene]** Never place confidential or sensitive information (e.g., customer email addresses) in tags or free-form name fields, since this content can surface in billing or diagnostic logs. [doc](https://docs.aws.amazon.com/clean-rooms/latest/userguide/data-protection.html)
- **[account protection]** Protect AWS account credentials, provision individual users through IAM Identity Center or IAM, and enable multi-factor authentication on every account used to access AWS Clean Rooms. [doc](https://docs.aws.amazon.com/clean-rooms/latest/userguide/data-protection.html)

## 🛡️ Reliability
- **[analysis rule change management]** Whenever you add or update an analysis rule on a configured table, review every collaboration that table is associated with to confirm none are left relying on an obsolete or unintended rule configuration. [doc](https://docs.aws.amazon.com/clean-rooms/latest/userguide/best-practices.html)
- **[fully managed architecture]** Favor the fully managed, serverless nature of AWS Clean Rooms (no infrastructure to patch or scale) to get elastic, highly available processing for collaboration queries and jobs. [doc](https://aws.amazon.com/solutions/guidance/building-queries-clean-rooms/)
- **[event-driven monitoring]** Use Amazon EventBridge to react to AWS Clean Rooms events in near real time instead of polling, enabling automated detection of failures or anomalous collaboration activity. [doc](https://docs.aws.amazon.com/clean-rooms/latest/userguide/monitoring-overview.html)

## ⚡ Performance Efficiency
- **[collaboration sizing]** Account for the number of collaborators in a collaboration channel when designing the collaboration, since a higher collaborator count increases processing time. [doc](https://docs.aws.amazon.com/wellarchitected/latest/video-streaming-advertising-lens/advperf04-bp06.html)
- **[data partitioning]** Partition large datasets effectively before bringing them into a collaboration, since unpartitioned large datasets directly degrade query performance. [doc](https://docs.aws.amazon.com/wellarchitected/latest/video-streaming-advertising-lens/advperf04-bp06.html)
- **[query reuse]** Optimize queries first, then package them as analysis templates with parameters so proven, efficient queries can be reused across a collaboration instead of being rewritten each time. [doc](https://docs.aws.amazon.com/wellarchitected/latest/video-streaming-advertising-lens/advperf04-bp06.html)
- **[compute sizing]** Choose the Spark compute size (CR.1X vs. the larger CR.4X) that matches your workload's data volume and query complexity, scaling up only when processing large datasets or complex analytics requires more vCPU/memory. [doc](https://aws.amazon.com/clean-rooms/faqs/)
- **[Spark tuning]** Tune supported Spark properties (e.g., shuffle partitions, broadcast join thresholds, adaptive query execution settings) for SQL queries to fine-tune performance and memory usage for complex queries or large datasets. [doc](https://docs.aws.amazon.com/clean-rooms/latest/userguide/running-sql-queries.html)
- **[query hints]** Use SQL hints to guide the Spark analytics engine's execution plan for analyses, improving query performance and reducing compute cost. [doc](https://docs.aws.amazon.com/clean-rooms/latest/sql-reference/sql-commands-hints-spark.html)
- **[cryptographic overhead]** Weigh the added processing time of cryptographic computing operations against the privacy benefit when deciding whether to use it for a given secure-computation workload. [doc](https://docs.aws.amazon.com/wellarchitected/latest/video-streaming-advertising-lens/advperf04-bp06.html)

## ⚙️ Operational Excellence
- **[query logging]** Turn on the query logging feature so all queries run in a collaboration are captured, then regularly review the logs to confirm queries match the use cases and columns agreed upon by collaborators. [doc](https://docs.aws.amazon.com/clean-rooms/latest/userguide/best-practices.html)
- **[example query review]** Review the example query the console generates after configuring an analysis rule to understand what queries the rule actually permits, since other queries beyond the example remain possible. [doc](https://docs.aws.amazon.com/clean-rooms/latest/userguide/best-practices.html)
- **[API auditing]** Enable AWS CloudTrail to capture all AWS Clean Rooms API calls, including requester identity, source IP, and timestamp, and deliver logs to an S3 bucket via a trail for ongoing audit support. [doc](https://docs.aws.amazon.com/clean-rooms/latest/userguide/logging-using-cloudtrail.html)
- **[log analysis]** Route AWS Clean Rooms CloudTrail and query logs to Amazon CloudWatch Logs so you can monitor, store, and alert on log content, and archive log data for durable long-term retention. [doc](https://docs.aws.amazon.com/clean-rooms/latest/userguide/monitoring-overview.html)
- **[cost tracking]** Apply cost allocation tags to collaboration memberships so spend can be tracked and categorized per collaboration in AWS Billing and Cost Management. [doc](https://docs.aws.amazon.com/clean-rooms/latest/userguide/monitoring-overview.html)
