# AWS Data Pipeline — Best Practices

## Common scenarios
- Orchestrating scheduled ETL between S3, RDS, DynamoDB, Redshift, and EMR        → Reliability, Operational Excellence
- Moving on-premises data into AWS on a recurring schedule using Task Runner        → Security, Reliability
- Running multi-stage batch jobs with dependency checking and automatic retries        → Reliability, Cost Optimization
- Migrating data between AWS accounts (e.g., DynamoDB table copies) via managed pipelines        → Security, Operational Excellence

## 🔒 Security
- **[IAM roles]** Define a distinct pipeline role (`role` field) and EC2 instance/resource role (`resourceRole` field) rather than reusing one role for both, so pipeline-management permissions are separated from the permissions granted to applications running on EC2/EMR resources. [doc](https://docs.aws.amazon.com/datapipeline/latest/DeveloperGuide/dp-iam-roles.html)
- **[Least privilege]** Start from the example pipeline-role and `AmazonEC2RoleforDataPipelineRole` policies and remove allow-statements for AWS services your pipeline's objects don't actually use (e.g., drop DynamoDB or SNS actions if no `DynamoDBDataNode` or `SnsAlarm` is defined). [doc](https://docs.aws.amazon.com/datapipeline/latest/DeveloperGuide/dp-iam-roles.html)
- **[Legacy default role]** Review any pipeline still using `DataPipelineDefaultRole` with the deprecated `AWSDataPipelineRole` managed policy attached, and replace or scope down that policy to match current security requirements. [doc](https://docs.aws.amazon.com/datapipeline/latest/DeveloperGuide/dp-iam-roles.html)
- **[Credentials]** Set up individual users with IAM Identity Center or IAM instead of sharing account root credentials, granting each user only the permissions needed for their job duties, and require MFA on all accounts. [doc](https://docs.aws.amazon.com/datapipeline/latest/DeveloperGuide/data-protection.html)
- **[Transport security]** Ensure clients calling the AWS Data Pipeline API support TLS 1.2 or later with cipher suites that provide perfect forward secrecy (such as DHE or ECDHE). [doc](https://docs.aws.amazon.com/datapipeline/latest/DeveloperGuide/infrastructure-security.html)
- **[Temporary credentials]** Sign AWS Data Pipeline API requests using AWS STS temporary security credentials rather than long-lived IAM user access keys where possible. [doc](https://docs.aws.amazon.com/datapipeline/latest/DeveloperGuide/infrastructure-security.html)
- **[IMDSv2]** Configure EC2 and EMR resources launched by pipelines to require IMDSv2 (supported on EMR 5.23.1, 5.27.1, 5.32+, or 6.2+) instead of allowing IMDSv1. [doc](https://docs.aws.amazon.com/datapipeline/latest/DeveloperGuide/data-protection.html)
- **[Sensitive data]** Never place confidential or sensitive information, such as customer email addresses, in pipeline tags or free-form Name fields — this data can be captured in billing or diagnostic logs. [doc](https://docs.aws.amazon.com/datapipeline/latest/DeveloperGuide/data-protection.html)
- **[Data discovery]** Use Amazon Macie to discover and help secure sensitive data stored in the Amazon S3 buckets that pipelines read from or write to. [doc](https://docs.aws.amazon.com/datapipeline/latest/DeveloperGuide/data-protection.html)
- **[Access sharing]** Use IAM policies based on pipeline tags and worker groups to share individual pipelines with specific users (e.g., read-only for auditors) instead of granting broad account-wide AWS Data Pipeline permissions. [doc](https://docs.aws.amazon.com/datapipeline/latest/DeveloperGuide/dp-control-access.html)
- **[On-premises Task Runner]** When polling for tasks from a Task Runner installed inside your own network, scope its `workerGroup` narrowly and secure the local resource independently, since AWS Data Pipeline does not manage that host's OS-level security. [doc](https://docs.aws.amazon.com/datapipeline/latest/DeveloperGuide/dp-how-task-runner-user-managed.html)

## 🛡️ Reliability
- **[Preconditions]** Attach preconditions (e.g., checking that an S3 key or database table exists) to data nodes and activities so downstream processing does not start against incomplete or missing input. [doc](https://docs.aws.amazon.com/datapipeline/latest/DeveloperGuide/dp-concepts-preconditions.html)
- **[Cascading failures]** Set `failureAndRerunMode` to `cascade` on activities, data nodes, and preconditions so a failed dependency propagates to its consumers instead of leaving them waiting indefinitely. [doc](https://docs.aws.amazon.com/datapipeline/latest/DeveloperGuide/dp-manage-cascade-failandrerun.html)
- **[Backfill caution]** Test cascade-enabled pipelines with short schedule time ranges first, since cascade failures combined with large backfills can rapidly create and tear down EMR/EC2 resources without doing useful work. [doc](https://docs.aws.amazon.com/datapipeline/latest/DeveloperGuide/dp-manage-cascade-failandrerun.html)
- **[Rerun from point of failure]** Model long-running ETL as discrete activities with explicit dependencies so that rerunning a failed object only re-executes it and its downstream consumers, instead of requiring the whole pipeline to restart from the beginning. [doc](https://docs.aws.amazon.com/datapipeline/latest/DeveloperGuide/dp-manage-cascade-failandrerun.html)
- **[Termination handling]** Use the Terminate action to cancel pending or unfinished activities, resources, or data nodes that exceed their `lateAfterTimeout`, preventing stuck tasks from blocking a pipeline indefinitely. [doc](https://docs.aws.amazon.com/datapipeline/latest/DeveloperGuide/dp-object-terminate.html)
- **[Pause instead of delete]** Use SetStatus with `PAUSE` on individual components to temporarily halt a pipeline instead of deleting it, since deleting a pipeline and its run history cannot be undone. [doc](https://docs.aws.amazon.com/sdk-for-ruby/v2/api/Aws/DataPipeline/Client.html)

## ⚙️ Operational Excellence
- **[API audit logging]** Enable AWS CloudTrail so every AWS Data Pipeline API call (such as CreatePipeline) is recorded with identity information, letting you determine who made each request and from where. [doc](https://docs.aws.amazon.com/datapipeline/latest/DeveloperGuide/dp-cloudtrail-logging.html)
- **[Event-driven automation]** Build EventBridge rules matching the `aws.datapipeline` source and `datapipeline.amazonaws.com` event source to trigger automated alerting or remediation workflows on specific pipeline API actions. [doc](https://docs.aws.amazon.com/eventbridge/latest/ref/events-ref-datapipeline.html)
- **[Task Runner logs]** Rely on AWS Data Pipeline publishing Task Runner logs to Amazon S3 before terminating managed EC2/EMR resources, and centralize these logs for post-run troubleshooting. [doc](https://docs.aws.amazon.com/datapipeline/latest/DeveloperGuide/dp-how-task-runner-dp-managed.html)
- **[Worker groups]** When running Task Runner on self-managed resources (on-premises servers or existing EC2 instances), assign a distinct `workerGroup` value per logical resource pool so activities are routed to the correct runner. [doc](https://docs.aws.amazon.com/datapipeline/latest/DeveloperGuide/dp-how-task-runner-user-managed.html)
- **[Role governance]** Periodically review which pipelines use `DataPipelineDefaultRole`/`DataPipelineDefaultResourceRole` versus custom roles, and document ownership so role permission changes don't unexpectedly affect unrelated pipelines. [doc](https://docs.aws.amazon.com/datapipeline/latest/DeveloperGuide/dp-get-setup.html)
