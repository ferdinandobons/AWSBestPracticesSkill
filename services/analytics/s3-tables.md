# Amazon S3 Tables — Best Practices

## Common scenarios
- Fully managed Apache Iceberg data lake for high-frequency analytics ingestion        → Performance Efficiency, Cost Optimization
- Centralized, governed tabular datasets queried by Athena, Redshift, and EMR/Spark        → Security, Operational Excellence
- Streaming or transactional data (e.g., CDC, IoT, clickstream) requiring high TPS writes        → Performance Efficiency, Reliability
- Multi-Region or multi-account analytics with regulatory data-residency requirements        → Reliability, Security

## 🔒 Security
- **[Access control]** Use table bucket policies for bucket/namespace-wide permissions and table policies for per-table grants, always including `GetTableMetadataLocation` and `UpdateTableMetadataLocation` in any policy that allows writes — omitting these breaks write/delete operations. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/s3-tables-resource-based-policies.html)
- **[IAM]** Grant IAM identities only the `s3tables:*` actions they need on specific table/bucket ARNs rather than account-wide access — by default no identity can create, read, or modify table buckets or tables without explicit grants. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/s3-tables-setting-up.html)
- **[Governance]** Integrate table buckets with AWS Lake Formation when you need column-level or row-level fine-grained access control, or when multiple third-party query engines need credential vending — plain IAM only supports table/namespace/bucket-level grants. [doc](https://docs.aws.amazon.com/lake-formation/latest/dg/create-s3-tables-catalog.html)
- **[Governance]** Use tags on tables and table buckets for attribute-based access control (ABAC) so permissions scale with new tables without constant policy rewrites. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/table-bucket-tagging.html)
- **[Encryption]** Configure server-side encryption with AWS KMS customer-managed keys (SSE-KMS) instead of the SSE-S3 default when you need audit trails or regulatory control over encryption keys, and grant the `maintenance.s3tables.amazonaws.com` service principal, your S3 Tables integration role, and your client access role permission to use the KMS key so automatic maintenance and analytics access keep working. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/s3-tables-kms-encryption.html)
- **[Encryption]** Set default SSE-KMS encryption at the table bucket level so newly created tables inherit it automatically, and specify a per-table KMS key at creation time for tables that need isolation from the bucket default — existing tables' encryption cannot be changed in place. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/s3-tables-kms-specify.html)
- **[Network]** Require HTTPS and AWS Signature Version 4 for all S3 Tables requests — HTTP requests are auto-redirected and public access policies cannot be configured, since S3 Block Public Access is always enforced on table buckets and cannot be disabled. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/s3-tables-restrictions.html)
- **[Governance]** Apply AWS Organizations service control policies (SCPs) against the `s3tables` namespace to enforce organization-wide guardrails on table bucket and table operations, in addition to identity- and resource-based policies. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/s3-tables-security-overview.html)

## 🛡️ Reliability
- **[Monitoring]** Collect CloudWatch metrics and set alarms on table bucket and table activity so you can detect and debug multipoint failures before they affect downstream consumers. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/s3-tables-monitoring-overview.html)
- **[Auditing]** Enable AWS CloudTrail management and data event logging for table buckets to capture both table/data-level API requests and automatic maintenance operations, giving you a full record of who changed what and when. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/s3-tables-logging.html)
- **[Disaster recovery]** Configure S3 Tables replication to maintain read-only, continuously synchronized replicas of tables (with full data, metadata, and snapshot history) across Regions or accounts to support regulatory data-residency, centralized analytics, or resilient test/dev environments. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/s3-tables-replication-tables.html)
- **[Disaster recovery]** Choose bucket-level replication when all tables in a bucket need the same destinations, and table-level replication for selective, per-table replication needs — you can replicate to up to five destination table buckets per source. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/s3-tables-replication-setting-up.html)
- **[Snapshot management]** Keep automatic snapshot management enabled (or tune `minSnapshotsToKeep`/`maxSnapshotAgeHours`) so you retain enough Iceberg snapshot history for time-travel and recovery without letting metadata grow unbounded. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/s3-tables-maintenance-overview.html)

## ⚡ Performance Efficiency
- **[Compaction]** Keep automatic compaction enabled and choose a compaction strategy that matches query patterns — `binpack` for general small-file consolidation, `sort` when queries filter on specific columns, and `z-order` when queries filter across multiple dimensions simultaneously; `auto` (default) picks sort or binpack based on whether a sort order is defined. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/s3-tables-maintenance.html)
- **[Compaction]** Tune the compaction target file size (64 MB–512 MB, default 512 MB) via `PutTableMaintenanceConfiguration` to match your data access patterns and query engine characteristics. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/s3-tables-maintenance.html)
- **[Naming]** Use all-lowercase letters for table bucket names, namespaces, table names, and column names — capitalized identifiers are not visible to AWS Glue Data Catalog, Lake Formation, or query engines like Athena and cause federation errors. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/s3-tables-buckets-naming.html)
- **[Storage tiering]** Use S3 Tables Intelligent-Tiering so infrequently accessed data files move to lower-cost tiers, while compaction continues to operate on frequently accessed (FA-tier) files — understand that delete files on colder-tier data aren't compacted until the underlying data is accessed again. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/tables-intelligent-tiering.html)
- **[Integration]** Enable AWS analytics service integration (Glue Data Catalog) when creating table buckets in the console so query engines such as Athena, Redshift, and EMR can automatically discover and query tables without manual catalog registration. [doc](https://docs.aws.amazon.com/help-panel/AmazonS3/latest/console/hp-table-buckets.html)

## 💰 Cost Optimization
- **[Maintenance]** Rely on built-in unreferenced file removal and compaction to clean up orphaned files and consolidate small objects automatically, reducing both storage footprint and per-request costs from scanning many small files. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/s3-tables-maintenance-overview.html)
- **[Compaction strategy]** Default to `binpack` compaction unless query patterns justify `sort` or `z-order` — the latter two incur higher compaction costs, so reserve them for tables with clear, recurring filter patterns. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/s3-tables-maintenance.html)
- **[Storage tiering]** Enable S3 Tables Intelligent-Tiering to automatically move cold data files to lower-cost tiers while maintenance continues in the background, reducing storage spend for tables with uneven access patterns. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/tables-intelligent-tiering.html)
- **[Record expiration]** Configure record expiration settings on tables that only need data retained for a bounded period so Amazon S3 automatically removes expired records instead of accumulating them indefinitely. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/s3-tables-maintenance-overview.html)
- **[Encryption]** When using SSE-KMS, rely on S3 Tables' table-level key model, which minimizes the number of KMS API calls compared to per-object encryption — this keeps SSE-KMS more cost-effective for tables than for general-purpose buckets. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/s3-tables-kms-encryption.html)

## ⚙️ Operational Excellence
- **[Monitoring]** Use Amazon CloudWatch alarms on S3 Tables metrics to get notified of threshold breaches, and pair them with CloudTrail logs to determine the request, source IP, and identity behind any event. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/s3-tables-monitoring-overview.html)
- **[Maintenance visibility]** Track automated maintenance job status and history (compaction, snapshot management, unreferenced file removal) through CloudTrail management events so you can confirm maintenance is running as configured. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/s3-tables-maintenance.html)
- **[Catalog integration]** Decide up front whether to manage S3 Tables access via IAM alone or via Lake Formation grants — both are supported and migration between the two is possible, but plan the transition to avoid access disruptions during cutover. [doc](https://docs.aws.amazon.com/lake-formation/latest/dg/manage-s3tables-catalog-integration.html)
- **[Programmatic setup]** When creating table buckets outside the console (SDK, CLI, or REST API), explicitly configure the AWS Glue Data Catalog and Lake Formation integration steps yourself — unlike console-created buckets, this integration is not enabled automatically. [doc](https://aws.amazon.com/blogs/big-data/getting-started-with-amazon-s3-tables-in-amazon-sagemaker-unified-studio/)
- **[Capacity planning]** Track table bucket, namespace, and table counts against default quotas (10 table buckets per Region per account, 10,000 namespaces per bucket, 10,000 tables per bucket) and request quota increases proactively for accounts approaching these limits. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/s3-tables-regions-quotas.html)
