# AWS Lake Formation — Best Practices

## Common scenarios
- Centralizing fine-grained access control over a Glue Data Catalog / S3 data lake        → Security, Operational Excellence
- Scaling data permissions across hundreds/thousands of tables and principals        → Security, Operational Excellence
- Sharing data lake databases and tables across AWS accounts or organizations        → Security, Reliability
- Migrating existing IAM-based Glue/S3 permissions to Lake Formation governance        → Security, Reliability

## 🔒 Security
- **[access model]** Prefer the fine-grained Lake Formation permissions model over the default "use only IAM access control" method — it lets you grant least-privilege, table/column/row-level access instead of leaving access solely to broad IAM policies. [doc](https://docs.aws.amazon.com/lake-formation/latest/dg/access-control-fine-grained.html)
- **[tag-based access]** Use Lake Formation tag-based access control (LF-TBAC) instead of naming individual databases/tables in grants once you have many Data Catalog objects — LF-Tag expressions scale permission management and reduce the number of grants needed. [doc](https://docs.aws.amazon.com/lake-formation/latest/dg/lf-tag-considerations.html)
- **[row/column security]** Apply data filters to restrict access to specific combinations of columns and rows — this cell-level security keeps sensitive data such as PII exposed only to authorized principals. [doc](https://docs.aws.amazon.com/lake-formation/latest/dg/data-filtering.html)
- **[least privilege admin]** Avoid designating an existing IAM `AdministratorAccess` user as the data lake administrator; instead grant a dedicated principal only the data-lake-admin permissions it needs — this keeps the ability to grant/revoke all Lake Formation permissions out of a general admin's hands. [doc](https://docs.aws.amazon.com/lake-formation/latest/dg/initial-lf-config.html)
- **[encryption]** Enforce S3 server-side encryption (SSE-S3 or SSE-KMS) for data at rest in the data lake — Lake Formation relies on S3's own encryption capabilities rather than providing a separate encryption layer. [doc](https://docs.aws.amazon.com/lake-formation/latest/dg/encryption-at-rest.html)
- **[direct bucket access]** Shut down direct IAM/bucket-policy access paths to the underlying S3 buckets once a location is registered with Lake Formation — otherwise principals can bypass Lake Formation's access controls entirely by reading S3 directly. [doc](https://docs.aws.amazon.com/lake-formation/latest/dg/access-control-underlying-data.html)
- **[cross-account sharing]** Use direct cross-account grants to specific IAM principals rather than account-wide shares when sharing Data Catalog resources — this retains control over exactly who in the external account can access the data. [doc](https://docs.aws.amazon.com/lake-formation/latest/dg/cross-account-permissions.html)
- **[transport security]** Require TLS 1.2 or later with forward-secrecy cipher suites and sign requests with IAM credentials or short-lived AWS STS tokens — this protects Lake Formation API calls in transit and avoids long-lived static credentials. [doc](https://docs.aws.amazon.com/lake-formation/latest/dg/infrastructure-security.html)
- **[IAM hygiene]** Follow IAM security best practices when creating the identities Lake Formation grants permissions to — Lake Formation permissions are enforced on top of IAM, so weak IAM hygiene undermines the whole access model. [doc](https://docs.aws.amazon.com/lake-formation/latest/dg/what-is-lake-formation.html)

## 🛡️ Reliability
- **[migration safety]** Use hybrid access mode to incrementally opt in specific principals when moving from IAM-based Glue Data Catalog permissions to the Lake Formation model — flipping default settings for all users at once risks breaking existing automation and ETL jobs. [doc](https://docs.aws.amazon.com/lake-formation/latest/dg/hybrid-access-mode.html)
- **[migration safety]** Determine users' existing IAM permissions and replicate them explicitly as Lake Formation grants before revoking `IAMAllowedPrincipals`/default IAM access on existing databases and tables — skipping this step can cause automation that creates databases and tables to fail. [doc](https://docs.aws.amazon.com/lake-formation/latest/dg/upgrade-glue-lake-formation.html)
- **[cross-account resilience]** Grant Lake Formation permissions to AWS Organizations or organizational units (or use LF-Tags) instead of many individual named-resource grants — named-resource cross-account grants are bounded by AWS RAM capacity, while an org-level grant counts as a single grant. [doc](https://docs.aws.amazon.com/lake-formation/latest/dg/cross-account-notes.html)
- **[resource links]** Create resource links for shared databases/tables, or use the "All tables" wildcard for a whole database — this gives consuming accounts a stable, queryable reference to shared resources via Athena and Redshift Spectrum. [doc](https://docs.aws.amazon.com/lake-formation/latest/dg/cross-data-sharing-lf.html)

## ⚙️ Operational Excellence
- **[delegated tag management]** Delegate LF-Tag creation and lifecycle management to designated "LF-Tag creators" instead of routing every tag change through the data lake administrator — this keeps tag ontology decisions close to the data engineers and analysts who understand the data. [doc](https://docs.aws.amazon.com/lake-formation/latest/dg/managing-tags.html)
- **[audit logging]** Enable and review AWS CloudTrail logs for Lake Formation API calls such as `PutDataLakeSettings`, `GrantPermissions`, `RevokePermissions`, and `GetDataAccess` — this shows which principals accessed what data, through which service, and when. [doc](https://docs.aws.amazon.com/lake-formation/latest/dg/logging-using-cloudtrail.html)
- **[audit logging]** Plan monitoring and audit coverage with the knowledge that high-volume data-plane operations like `GetTableObjects`, `UpdateTableObjects`, and `GetWorkUnitResults` are not logged to CloudTrail by default — assuming full data-access logging from CloudTrail alone would leave a monitoring gap. [doc](https://docs.aws.amazon.com/lake-formation/latest/dg/monitoring-overview.html)
- **[permission auditing]** Periodically list and review the Lake Formation permissions granted to principals on catalogs, databases, and tables, especially before a hybrid-access-mode or IAM-model migration — this confirms the intended access model is actually in effect before changing it. [doc](https://docs.aws.amazon.com/lake-formation/latest/dg/hybrid-access-mode-update.html)
