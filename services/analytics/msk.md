# Amazon MSK — Best Practices

## Common scenarios
- Real-time event streaming and pub/sub messaging between microservices        → Reliability, Performance Efficiency
- Ingesting high-volume clickstream, IoT, or log data for downstream analytics        → Performance Efficiency, Cost Optimization
- Change-data-capture pipelines feeding data lakes or data warehouses        → Reliability, Security
- Cross-Region replication and disaster recovery for Kafka-based architectures        → Reliability, Operational Excellence

## 🔒 Security
- **[Network]** Keep clients connecting to brokers privately within your Amazon VPC so traffic never leaves the AWS network, and use security group rules to control which protocols and ports are allowed on brokers. [doc](https://docs.aws.amazon.com/msk/latest/developerguide/port-info.html)
- **[Authentication]** Use IAM access control for both authentication and authorization of Apache Kafka actions — it defaults to least-privilege access and is the easiest and most secure option compared to mTLS or SASL/SCRAM. [doc](https://docs.aws.amazon.com/msk/latest/developerguide/iam-access-control.html)
- **[Authentication]** When mutual TLS is required, use independent AWS Private CAs per MSK cluster so certificates signed by one PCA cannot authenticate to a different cluster. [doc](https://docs.aws.amazon.com/msk/latest/developerguide/msk-authentication-cluster.html)
- **[IAM policies]** Start from AWS managed policies and narrow toward least-privilege customer-managed policies scoped to specific cluster/topic resources, and validate policies with IAM Access Analyzer. [doc](https://docs.aws.amazon.com/msk/latest/developerguide/security_iam_service-with-iam-policy-best-practices.html)
- **[Cross-account access]** Attach a cluster policy to define exactly which cross-account IAM principals can set up private connectivity, and use it together with IAM client authentication to granularly scope Kafka data-plane permissions. [doc](https://docs.aws.amazon.com/msk/latest/developerguide/security_iam_service-with-iam-resource-based-policies.html)
- **[Encryption at rest]** Encrypt data at rest with a customer-managed AWS KMS key rather than relying only on the AWS-managed default key, for tighter control over key policy and rotation. [doc](https://docs.aws.amazon.com/msk/latest/developerguide/msk-encryption.html)
- **[Encryption in transit]** Enable TLS encryption for both broker-to-broker and client-to-broker communication — this is enabled by default but should be verified and not disabled to plaintext for production clusters. [doc](https://docs.aws.amazon.com/msk/latest/developerguide/bestpractices.html)
- **[ZooKeeper]** Restrict access to Apache ZooKeeper nodes with dedicated security groups so cluster metadata cannot be modified out-of-band. [doc](https://docs.aws.amazon.com/msk/latest/developerguide/zookeeper-security.html)
- **[Logging]** Stream broker logs to CloudWatch Logs, Amazon S3, or Amazon Data Firehose, and enable AWS CloudTrail for MSK API calls to support troubleshooting, auditing, and compliance. [doc](https://docs.aws.amazon.com/msk/latest/developerguide/msk-logging.html)

## 🛡️ Reliability
- **[Cluster topology]** Deploy a three-AZ cluster with a replication factor of at least 3 — a replication factor of 1 can cause offline partitions during rolling updates and a factor of 2 risks data loss. [doc](https://docs.aws.amazon.com/msk/latest/developerguide/bestpractices.html)
- **[Durability]** Set minimum in-sync replicas (minISR) to at most RF-1 (for example, minISR=2 with RF=3) so the cluster stays writable and available during a rolling update or single-broker failure. [doc](https://docs.aws.amazon.com/msk/latest/developerguide/bestpractices.html)
- **[Client resilience]** Configure client connection strings with at least one broker from each Availability Zone so producers and consumers can fail over when a broker is temporarily offline for maintenance. [doc](https://docs.aws.amazon.com/msk/latest/developerguide/bestpractices.html)
- **[Producer config]** Set `retries` high (integer max or similar), `acks=all`, and a `delivery.timeout.ms` of 60 seconds or more so producers survive broker failover without breaking Kafka's high-availability guarantees. [doc](https://docs.aws.amazon.com/msk/latest/developerguide/bestpractices-kafka-client.html)
- **[Consumer config]** Set `session.timeout.ms` above application startup time, use sticky partition assignors (`StickyAssignor` or `CooperativeStickyAssignor`), and implement a clean shutdown hook so consumer group rebalances stay predictable. [doc](https://docs.aws.amazon.com/msk/latest/developerguide/bestpractices-kafka-client.html)
- **[Capacity headroom]** Keep broker CPU utilization (CPU User + CPU System) under 60% so the cluster retains headroom to absorb broker failures, patching, and rolling upgrades without cascading load. [doc](https://docs.aws.amazon.com/msk/latest/developerguide/bestpractices.html)
- **[Disk monitoring]** Alarm on the `KafkaDataLogsDiskUsed` CloudWatch metric at 85% and respond with automatic storage scaling, adjusted retention, or deleting unused topics before the cluster runs out of disk space. [doc](https://docs.aws.amazon.com/msk/latest/developerguide/bestpractices.html)
- **[Cluster hygiene]** Never add brokers directly through ZooKeeper commands on ZooKeeper-based clusters — brokers added this way are invisible to MSK's cluster management and can cause data loss. [doc](https://docs.aws.amazon.com/msk/latest/developerguide/bestpractices.html)
- **[Connection storms]** Stagger producer/consumer restarts across application deployments and add random jitter before creating clients and on shutdown, so mass reconnects don't overload the cluster and trigger cascading failures. [doc](https://docs.aws.amazon.com/msk/latest/developerguide/bestpractices-kafka-client.html)

## ⚡ Performance Efficiency
- **[Broker sizing]** Right-size the number of partitions per broker to the documented recommendations for your instance type, and validate higher partition counts with your own performance testing before relying on them in production. [doc](https://docs.aws.amazon.com/msk/latest/developerguide/bestpractices.html)
- **[Thread tuning]** On m5.4xlarge, m7g.4xlarge, or larger brokers, tune `num.io.threads` and `num.network.threads` to the documented values for the instance size (increase io.threads first, then network.threads) to fully use available CPU cores. [doc](https://docs.aws.amazon.com/msk/latest/developerguide/bestpractices.html)
- **[Rebalancing]** Use Cruise Control to continuously manage partition assignment and load distribution rather than manually reassigning partitions, and limit any manual `kafka-reassign-partitions` call to 10 partitions at a time for safe operation. [doc](https://docs.aws.amazon.com/msk/latest/developerguide/bestpractices.html)
- **[Cluster type]** Start with MSK Express brokers for most workloads since they eliminate storage and network throughput capacity planning; use Standard brokers when you need manual storage control, tiered storage, or custom replication quotas. [doc](https://aws.amazon.com/blogs/big-data/best-practices-for-right-sizing-your-apache-kafka-clusters-to-optimize-performance-and-cost/)
- **[Producer tuning]** Increase `linger.ms` (25ms for most use cases) and `batch.size` (64–128KB), and use `lz4` or `zstd` compression to reduce per-request overhead and improve throughput. [doc](https://docs.aws.amazon.com/msk/latest/developerguide/bestpractices-kafka-client.html)
- **[Consumer tuning]** Raise `fetch.min.bytes` and size the consumer group so the number of consumers is at least equal to the number of partitions for better parallelism and resiliency. [doc](https://docs.aws.amazon.com/msk/latest/developerguide/bestpractices-kafka-client.html)
- **[Network cost/latency]** Set `client.rack` to the AZ ID so consumers read from the nearest replica, reducing cross-AZ network traffic and latency. [doc](https://aws.amazon.com/blogs/big-data/reduce-network-traffic-costs-of-your-amazon-msk-consumers-with-rack-awareness/)
- **[Log recovery]** Increase the number of log-recovery threads (up to the number of CPU cores) so brokers with many partitions restart faster after an unclean shutdown. [doc](https://docs.aws.amazon.com/msk/latest/developerguide/bestpractices.html)
- **[Monitoring granularity]** Use a Prometheus scrape interval of 60 seconds or higher when open monitoring is enabled — shorter intervals can drive high broker CPU usage. [doc](https://docs.aws.amazon.com/msk/latest/developerguide/bestpractices.html)

## 💰 Cost Optimization
- **[Storage]** Use tiered storage to move older log segments to a low-cost storage tier while retaining recent data on performance-optimized storage, cutting both broker count and provisioned EBS storage needed for long retention. [doc](https://aws.amazon.com/blogs/big-data/retain-more-for-less-with-tiered-storage-for-amazon-msk/)
- **[Retention]** Set retention time or size at the cluster or topic level (`retention.ms`, `retention.bytes`) so Kafka removes inactive log segments automatically instead of accumulating unbounded storage. [doc](https://docs.aws.amazon.com/msk/latest/developerguide/bestpractices.html)
- **[Scaling strategy]** Prefer adding partitions/brokers (horizontal scale-out) over broker-size upgrades when producers use round-robin writes, since it gives more granular cost and resource control than resizing brokers. [doc](https://docs.aws.amazon.com/msk/latest/developerguide/bestpractices.html)
- **[Sizing]** Use the MSK sizing guidance and performance-testing methodology to right-size broker count and type for your throughput, availability, and latency needs instead of over-provisioning by default. [doc](https://aws.amazon.com/blogs/big-data/best-practices-for-right-sizing-your-apache-kafka-clusters-to-optimize-performance-and-cost/)
- **[Monitoring level]** Choose the lowest CloudWatch monitoring level (DEFAULT) that meets your operational needs, since PER_BROKER, PER_TOPIC_PER_BROKER, and PER_TOPIC_PER_PARTITION levels add cost as granularity increases. [doc](https://docs.aws.amazon.com/msk/latest/developerguide/metrics-details.html)
- **[Broker type]** Evaluate MSK Express brokers for new workloads to avoid paying for over-provisioned storage and network capacity that Standard brokers require you to plan manually. [doc](https://aws.amazon.com/blogs/big-data/best-practices-for-right-sizing-your-apache-kafka-clusters-to-optimize-performance-and-cost/)

## ⚙️ Operational Excellence
- **[Monitoring]** Set CloudWatch alarms on core health signals — CPU (User+System) above 60%, `KafkaDataLogsDiskUsed` above 85%, and `HeapMemoryAfterGC` above 60% — to catch degradation before it affects availability. [doc](https://docs.aws.amazon.com/msk/latest/developerguide/bestpractices.html)
- **[Observability]** Extend default CloudWatch metrics with open monitoring via Prometheus (and Amazon Managed Grafana) for deeper broker, topic, and partition-level visibility beyond what DEFAULT-level CloudWatch metrics provide. [doc](https://aws.amazon.com/blogs/big-data/enhance-operational-insights-for-amazon-msk-using-amazon-managed-service-for-prometheus-and-amazon-managed-grafana/)
- **[Client monitoring]** Track client-side producer and consumer metrics (byte-rate, record-error-rate, records-lag-max, request-latency) alongside broker metrics, and include them when raising AWS support cases to speed root-cause analysis. [doc](https://docs.aws.amazon.com/msk/latest/developerguide/bestpractices-kafka-client.html)
- **[Automation]** Automate scaling responses to alarms — for example, trigger storage auto scaling or broker-size updates — rather than relying on manual intervention during a capacity event. [doc](https://docs.aws.amazon.com/msk/latest/developerguide/bestpractices.html)
- **[Client versioning]** Use Kafka AdminClient version 2.8.0 or later (or the `--bootstrap-servers` flag instead of `--zookeeper`) for admin operations to avoid topic ID mismatch errors on modern Kafka versions. [doc](https://docs.aws.amazon.com/msk/latest/developerguide/bestpractices.html)
