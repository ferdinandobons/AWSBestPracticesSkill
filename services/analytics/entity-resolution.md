# AWS Entity Resolution — Best Practices

## Common scenarios
- Building a unified customer profile by matching records across CRM, ad, and commerce systems        → Security, Performance Efficiency
- Deduplicating and linking healthcare, genomic, or patient records across data stores        → Security, Reliability
- Matching product or business codes (SKUs, UPCs) across catalogs and business units        → Performance Efficiency, Cost Optimization
- Running near real-time fraud or identity matching against a growing reference dataset        → Reliability, Performance Efficiency

## 🔒 Security
- **[credentials]** Set up individual IAM Identity Center or IAM users instead of using root credentials, and enable multi-factor authentication on every account that manages AWS Entity Resolution — this limits the blast radius of a compromised credential. [doc](https://docs.aws.amazon.com/entityresolution/latest/userguide/data-protection.html)
- **[encryption]** Provide your own AWS KMS customer managed key (CMK) for schema mappings and matching workflow output instead of relying on the default service key — this gives you control over key policies, rotation, and revocation for sensitive matched data. [doc](https://docs.aws.amazon.com/entityresolution/latest/userguide/data-protection.html)
- **[data in transit]** Require TLS 1.2 (and prefer TLS 1.3) for all communication with AWS Entity Resolution and its API endpoints — this protects schema, workflow, and match data while in transit. [doc](https://docs.aws.amazon.com/entityresolution/latest/userguide/data-protection.html)
- **[sensitive fields]** Never place confidential or sensitive information, such as customer email addresses, into workflow names, tags, or other free-form text fields — these values can surface in billing records and diagnostic logs. [doc](https://docs.aws.amazon.com/entityresolution/latest/userguide/data-protection.html)
- **[least privilege]** Start from AWS managed policies for AWS Entity Resolution and then narrow them into customer managed policies scoped to specific schema mappings, workflows, and S3/Glue resources — apply least-privilege permissions rather than broad service-wide access. [doc](https://docs.aws.amazon.com/entityresolution/latest/userguide/security_iam_id-based-policy-examples.html)
- **[conditions]** Add IAM policy conditions (for example requiring SSL/TLS or restricting access to calls made through a specific AWS service) to further restrict when AWS Entity Resolution actions are permitted — this narrows access beyond what actions and resources alone can express. [doc](https://docs.aws.amazon.com/entityresolution/latest/userguide/security_iam_id-based-policy-examples.html)
- **[policy validation]** Validate identity-based policies with IAM Access Analyzer before attaching them — it checks policies against IAM best practices and flags overly permissive grants. [doc](https://docs.aws.amazon.com/entityresolution/latest/userguide/security_iam_id-based-policy-examples.html)
- **[cross-account access]** Use resource-based policies or IAM cross-account roles to grant another account access to a matching workflow or schema mapping instead of sharing credentials — scope the principal explicitly rather than broadening account-wide trust. [doc](https://docs.aws.amazon.com/entityresolution/latest/userguide/security_iam_service-with-iam.html)
- **[private connectivity]** Create a VPC interface endpoint (AWS PrivateLink) for AWS Entity Resolution, using the FIPS endpoint variant if you require FIPS 140-3 validated cryptography, so traffic to the service stays off the public internet. [doc](https://docs.aws.amazon.com/entityresolution/latest/userguide/vpc-interface-endpoints.html)
- **[data residency]** Rely on AWS Entity Resolution's built-in regionalization, which processes and stores data in the same Region you invoke it from, when you have data sovereignty or regional health/privacy compliance requirements. [doc](https://aws.amazon.com/blogs/industries/aws-entity-resolution-for-health-data/)
- **[sensitive data discovery]** Pair AWS Entity Resolution with Amazon Macie on the underlying S3 buckets to discover and classify sensitive data before and after matching workflows run. [doc](https://docs.aws.amazon.com/entityresolution/latest/userguide/data-protection.html)

## 🛡️ Reliability
- **[quota awareness]** Design workflow orchestration around the account's concurrent job quotas (one concurrent matching job and one concurrent ID mapping job per Region by default) rather than assuming jobs run in parallel — queue or schedule jobs to avoid throttling. [doc](https://docs.aws.amazon.com/entityresolution/latest/userguide/quotas.html)
- **[quota increases]** Request quota increases in advance for adjustable limits, such as the number of matching workflows, ID mapping workflows, or records per ML-based matching workflow, before scaling a workload into production. [doc](https://docs.aws.amazon.com/entityresolution/latest/userguide/quotas.html)
- **[incremental processing]** Use near real-time or incremental matching (via the GenerateMatchId API) for continuously arriving records instead of rerunning full batch jobs — this keeps matches current without reprocessing your entire dataset on every update. [doc](https://docs.aws.amazon.com/entityresolution/latest/apireference/API_GenerateMatchId.html)
- **[retry handling]** Implement throttling-aware retries with backoff around AWS Entity Resolution API calls (for example GenerateMatchId and GetMatchId), since these have per-second rate quotas that can trigger throttling errors under bursty load. [doc](https://docs.aws.amazon.com/entityresolution/latest/userguide/quotas.html)

## ⚡ Performance Efficiency
- **[matching technique choice]** Choose rule-based matching for deterministic, explainable exact/fuzzy matches and reserve ML-based matching for records with variable or missing name/address/email/phone/date-of-birth fields — using the wrong technique for the data shape reduces match quality. [doc](https://docs.aws.amazon.com/entityresolution/latest/userguide/create-matching-workflow.html)
- **[rule type selection]** Pick the Advanced rule type when you need fuzzy matching, AND/OR logic, or one-to-one schema mappings, and the Simple rule type when you need real-time or ID mapping workflows — the rule type cannot be changed after the workflow is created, so match it to the use case up front. [doc](https://docs.aws.amazon.com/entityresolution/latest/userguide/creating-matching-workflow-rule-based.html)
- **[normalization]** Leave AWS Entity Resolution's built-in data normalization enabled (lowercasing, special-character and whitespace removal) unless your data input is already normalized — consistent normalization materially improves match accuracy across noisy source systems. [doc](https://docs.aws.amazon.com/entityresolution/latest/userguide/what-is-service.html)
- **[rule prioritization]** Order match rules from most to least precise and review the rule number attached to each output match group — this lets you understand and tune the fidelity of matches a rule-based workflow produces. [doc](https://docs.aws.amazon.com/entityresolution/latest/userguide/glossary.html)
- **[confidence scoring]** Use the confidence score returned by ML-based matching workflows to filter or prioritize match groups downstream instead of treating every match as equally reliable. [doc](https://docs.aws.amazon.com/entityresolution/latest/userguide/glossary.html)

## 💰 Cost Optimization
- **[incremental over full reprocessing]** Configure automatic incremental processing on matching workflows so only new or changed records are matched against existing data, instead of re-running full batch matching workflows on the entire dataset. [doc](https://docs.aws.amazon.com/entityresolution/latest/apireference/API_CreateMatchingWorkflow.html)
- **[right-size ML workloads]** Scope ML-based matching workflows to the records and fields that actually need probabilistic matching, and use simpler rule-based workflows for exact-match use cases — this avoids paying for ML-based processing where deterministic rules are sufficient. [doc](https://docs.aws.amazon.com/entityresolution/latest/userguide/create-matching-workflow.html)

## ⚙️ Operational Excellence
- **[activity logging]** Create an AWS CloudTrail trail covering AWS Entity Resolution so every API call is captured with the identity, source IP, and time of the request, and route logs to Amazon S3 for retention. [doc](https://docs.aws.amazon.com/entityresolution/latest/userguide/logging-using-cloudtrail.html)
- **[workflow monitoring]** Send AWS Entity Resolution matching workflow logs to Amazon CloudWatch Logs and set alarms on thresholds (such as job failures or unusually low match rates) to catch problems automatically rather than by manual review. [doc](https://docs.aws.amazon.com/entityresolution/latest/userguide/monitoring-overview.html)
- **[event-driven automation]** Use Amazon EventBridge rules matching on `aws.entityresolution` events to trigger automated downstream actions (such as notifications or pipeline steps) when specific AWS Entity Resolution API calls occur. [doc](https://docs.aws.amazon.com/eventbridge/latest/ref/events-ref-entityresolution.html)
- **[job status tracking]** Track matching workflow job status and metrics (records processed, matches found) exposed by the service to confirm workflows completed as expected before consuming their output downstream. [doc](https://docs.aws.amazon.com/entityresolution/latest/userguide/what-is-service.html)
