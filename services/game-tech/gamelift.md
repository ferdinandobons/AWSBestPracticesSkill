# Amazon GameLift Servers — Best Practices

## Common scenarios
- Hosting dedicated multiplayer game servers at scale        → Reliability, Performance Efficiency, Cost Optimization
- Global low-latency matchmaking and game session placement        → Performance Efficiency, Reliability
- Reducing game hosting infrastructure spend        → Cost Optimization
- Operating and patching fleets over the game's lifecycle        → Security, Operational Excellence

## 🔒 Security
- **[Fleet lifecycle]** Replace managed fleets (EC2 and container) that are older than 30 days, and at minimum once a year for production, to pick up the latest AMI and avoid running outdated, vulnerable dependencies — updating fleet attributes does not refresh the AMI, only creating a new fleet does. [doc](https://docs.aws.amazon.com/gameliftservers/latest/developerguide/security-best-practices.html)
- **[Fleet lifecycle]** Automate fleet replacement using AWS CloudFormation for repeatable fleet configurations, aliases to abstract fleet IDs, and blue/green deployment to migrate player traffic with zero downtime and instant rollback capability. [doc](https://docs.aws.amazon.com/gameliftservers/latest/developerguide/security-best-practices.html)
- **[Fleet lifecycle]** Test server builds against the latest AMI version before deployment and request fleet quota increases ahead of time so replacement fleets can be created without delay. [doc](https://docs.aws.amazon.com/gameliftservers/latest/developerguide/security-best-practices.html)
- **[Networking]** Never open inbound ports to 0.0.0.0/0 (for example RDP/SSH); use UpdateFleetPortSettings to scope InboundPermissionAuthorizations to specific IP addresses or ranges. [doc](https://docs.aws.amazon.com/gameliftservers/latest/developerguide/security-best-practices.html)
- **[Networking]** Use Amazon GameLift Servers VPC isolation and policies to restrict access to specific VPCs or VPC endpoints, and require clients to use TLS 1.2+ (TLS 1.3 recommended) with PFS cipher suites for API calls. [doc](https://docs.aws.amazon.com/gameliftservers/latest/developerguide/infrastructure-security.html)
- **[IAM]** Grant only the minimum IAM permissions needed for players, operators, and services to interact with fleets, queues, and builds, following the identity-based policy examples provided for Amazon GameLift Servers. [doc](https://docs.aws.amazon.com/gameliftservers/latest/developerguide/security_iam_id-based-policy-examples.html)
- **[IAM]** Create a dedicated IAM service role (with a trust policy scoped to Amazon GameLift Servers) when your game servers need to access other AWS resources, instead of embedding long-term credentials in the build. [doc](https://docs.aws.amazon.com/gameliftservers/latest/developerguide/setting-up-role.html)
- **[Patch management]** Keep customer-managed software current — update the game server build/container image and your SDKs (AWS SDK, Server SDK, Client SDK) regularly, since AWS only patches the OS/AMI when a new fleet is created. [doc](https://docs.aws.amazon.com/gameliftservers/latest/developerguide/vulnerability-analysis-management.html)
- **[Auditing]** Turn on AWS CloudTrail to capture all Amazon GameLift Servers API calls for security analysis and operational troubleshooting. [doc](https://aws.amazon.com/gamelift/servers/faqs/)

## 🛡️ Reliability
- **[Availability]** Run production fleets with multiple instances distributed across the fleet so traffic can shift away from an unhealthy or unresponsive instance. [doc](https://docs.aws.amazon.com/gameliftservers/latest/developerguide/disaster-recovery-resiliency.html)
- **[Multi-region]** Use game session queues that span multiple AWS Regions and locations so placements can fail over automatically during a regional slowdown or outage. [doc](https://docs.aws.amazon.com/gameliftservers/latest/developerguide/disaster-recovery-resiliency.html)
- **[Queues]** Populate each queue with fleets that vary in fleet type, location, and instance type to maximize placement flexibility and resilience against localized capacity shortages. [doc](https://docs.aws.amazon.com/gameliftservers/latest/developerguide/queues-intro.html)
- **[Queues]** Set a realistic queue timeout and, when supplying player latency data, configure player latency policies so placements don't stall waiting for latency targets that aren't achievable for all players in the match. [doc](https://aws.amazon.com/blogs/gametech/development-phase-steps-for-successful-launches-on-amazon-gamelift-servers/)
- **[Auto scaling]** Use target-based (target-tracking) auto scaling as the primary scaling mechanism, supplementing with rule-based policies only for specific edge cases, to keep hosting capacity aligned with player demand automatically. [doc](https://docs.aws.amazon.com/gameliftservers/latest/developerguide/fleets-autoscaling.html)
- **[Region selection]** Balance the number of Regions/locations you deploy to against matchmaking pool fragmentation — spreading a game across too many Regions can thin player pools and increase match times for diminishing latency gains. [doc](https://aws.amazon.com/blogs/gametech/fine-tuning-player-latency-with-amazon-gamelift-servers/)

## ⚡ Performance Efficiency
- **[Runtime configuration]** Configure concurrent server processes per instance to run as many game sessions as your instance's CPU/memory allows while keeping performance stable, and tune game session activation limits to avoid resource contention during simultaneous session starts. [doc](https://docs.aws.amazon.com/gameliftservers/latest/developerguide/fleets-multiprocess.html)
- **[Compute selection]** Match Amazon EC2 instance family, type, and size to your game server's actual CPU, memory, and network requirements rather than defaulting to a single instance type across all fleets. [doc](https://docs.aws.amazon.com/gameliftservers/latest/developerguide/gamelift-pricing-cost-optimization.html)
- **[Placement]** Use game session queues with the FleetIQ algorithm to place sessions based on player latency, cost, and location instead of targeting a single fleet directly. [doc](https://docs.aws.amazon.com/gameliftservers/latest/developerguide/queues-intro.html)
- **[Observability]** Use server telemetry metrics (published to Amazon Managed Service for Prometheus/Grafana or CloudWatch) to track server tick rate, tick time, packet loss, and CPU/memory usage, and act on regressions before they affect players. [doc](https://docs.aws.amazon.com/gameliftservers/latest/developerguide/monitoring-cloudwatch.html)

## 💰 Cost Optimization
- **[Instances]** Use AWS Graviton instance types where supported for better price-performance and lower cost than comparable x86 On-Demand instances. [doc](https://docs.aws.amazon.com/gameliftservers/latest/developerguide/gamelift-pricing-cost-optimization.html)
- **[Instances]** Combine Spot and On-Demand fleets behind the same queue (roughly 50-85% Spot usage recommended) to reduce cost while preserving availability, and monitor InstanceInterruptions/GameServerInterruptions to gauge Spot viability. [doc](https://docs.aws.amazon.com/gameliftservers/latest/developerguide/gamelift-pricing-cost-optimization.html)
- **[Instances]** Build and run game servers on Linux rather than Windows for lower-cost instances where your engine supports it. [doc](https://docs.aws.amazon.com/gameliftservers/latest/developerguide/gamelift-pricing-cost-optimization.html)
- **[Capacity]** Scale fleets to zero when not in use (development/test) and rely on target-based auto scaling in production to avoid over-provisioning idle capacity. [doc](https://docs.aws.amazon.com/gameliftservers/latest/developerguide/gamelift-pricing-cost-optimization.html)
- **[Capacity]** Maintain a right-sized buffer of idle game servers via target tracking to absorb demand spikes without over-provisioning for steady-state load. [doc](https://docs.aws.amazon.com/gameliftservers/latest/developerguide/gamelift-pricing-cost-optimization.html)
- **[Placement]** Use queue priority settings to weigh hosting cost alongside latency, and monitor placement metrics to find further optimization opportunities. [doc](https://docs.aws.amazon.com/gameliftservers/latest/developerguide/gamelift-pricing-cost-optimization.html)
- **[Multi-region strategy]** Deploy hosting to your largest player markets first and use secondary locations for overflow capacity during peak times, rather than spreading capacity evenly across many Regions. [doc](https://docs.aws.amazon.com/gameliftservers/latest/developerguide/gamelift-pricing-cost-optimization.html)
- **[Cost tracking]** Apply cost allocation tags to fleets and other resources, and use AWS Cost Explorer and AWS Budgets to review Amazon GameLift Servers usage and spend on a regular schedule. [doc](https://docs.aws.amazon.com/gameliftservers/latest/developerguide/gamelift-pricing-cost-optimization.html)
- **[Data transfer]** Track client-server data transfer and use compression to reduce bandwidth usage and improve latency. [doc](https://docs.aws.amazon.com/gameliftservers/latest/developerguide/gamelift-pricing-cost-optimization.html)

## ⚙️ Operational Excellence
- **[Monitoring]** Monitor core CloudWatch metrics — PercentIdleInstances, PercentAvailableGameSessions, AverageWaitTime, HealthyServerProcesses, CPUUtilization — to size fleets correctly and catch capacity or health issues early. [doc](https://docs.aws.amazon.com/gameliftservers/latest/developerguide/monitoring-cloudwatch.html)
- **[Monitoring]** Set up Amazon CloudWatch alarms on key thresholds (queue wait time, idle instance percentage, unhealthy instance replacement) to get proactive notification instead of discovering issues from player reports. [doc](https://docs.aws.amazon.com/gameliftservers/latest/developerguide/monitoring-overview.html)
- **[Observability]** Use server telemetry metrics through Amazon Managed Service for Prometheus and Amazon Managed Grafana for detailed server-side timing, network, memory, and CPU telemetry beyond what CloudWatch alone provides. [doc](https://docs.aws.amazon.com/gameliftservers/latest/developerguide/monitoring-cloudwatch.html)
- **[Events]** Subscribe to game session placement events via Amazon SNS or Amazon EventBridge (and process them with AWS Lambda) instead of polling Describe APIs, for scalable placement tracking. [doc](https://aws.amazon.com/blogs/gametech/development-phase-steps-for-successful-launches-on-amazon-gamelift-servers/)
- **[Auditing]** Enable AWS CloudTrail for Amazon GameLift Servers to record API activity for operational troubleshooting and change tracking. [doc](https://docs.aws.amazon.com/gameliftservers/latest/developerguide/monitoring-overview.html)
- **[IaC]** Manage fleets, queues, and related resources as AWS CloudFormation (or AWS CDK) templates so deployments and fleet replacements are repeatable and auditable. [doc](https://docs.aws.amazon.com/gameliftservers/latest/developerguide/gamelift-roadmap-hybrid.html)
