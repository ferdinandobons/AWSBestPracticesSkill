# AWS Verified Access — Best Practices

## Common scenarios
- Providing VPN-less zero-trust access to internal corporate web applications        → Security, Operational Excellence
- Extending secure access to non-HTTP(S) applications (SSH, RDP, Git, databases) over TCP        → Security, Reliability
- Consolidating access policies for many applications with similar security requirements        → Operational Excellence, Cost Optimization
- Migrating from a Client VPN or perimeter-firewall model to identity- and device-based access control        → Security, Reliability

## 🔒 Security
- **[access policies]** Write Verified Access policies in Cedar with the scope left undefined and express all restrictions in the condition clause using identity and device trust context — this is the only supported policy structure for Verified Access. [doc](https://docs.aws.amazon.com/verified-access/latest/ug/auth-policies-policy-statement-struct.html)
- **[default deny]** Rely on the default-deny behavior of Verified Access groups and endpoints — all requests are blocked until you explicitly define a policy, so treat an undefined policy as a deployment blocker, not an open pass-through. [doc](https://docs.aws.amazon.com/verified-access/latest/ug/how-it-works.html)
- **[trust providers]** Attach both an identity trust provider (IAM Identity Center or an OIDC-compatible IdP) and one or more device trust providers to each Verified Access instance so that access decisions combine user identity with device security posture rather than identity alone. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/security-reference-architecture/network.html)
- **[network isolation]** Configure backing applications (ALB, NLB, or ENI targets) to accept traffic only from the security groups of their Verified Access endpoints, so the application cannot be reached even if an attacker discovers its IP address. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/building-zero-trust-access-across-multi-account-aws-environments/)
- **[application-layer protection]** Associate an AWS WAF web ACL with each Verified Access instance to filter common exploits such as SQL injection and cross-site scripting before requests reach protected HTTP(S) applications. [doc](https://docs.aws.amazon.com/verified-access/latest/ug/waf-integration.html)
- **[policy grouping]** Group Verified Access endpoints with similar security requirements into a shared Verified Access Group so baseline policy is enforced consistently, and layer stricter endpoint-level policies only where an individual application needs them. [doc](https://docs.aws.amazon.com/whitepapers/latest/building-scalable-secure-multi-vpc-network-infrastructure/centralized-access-to-vpc-private-endpoints.html)
- **[multi-account governance]** Use Verified Access Group sharing across accounts in an OU so that consistent access policies automatically extend to new accounts instead of being re-authored per account. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/building-zero-trust-access-across-multi-account-aws-environments/)
- **[IAM least privilege]** Grant administrators only the specific IAM permissions needed to manage Verified Access instances, trust providers, groups, and endpoints, using individual IAM Identity Center or IAM users rather than shared root/account credentials. [doc](https://docs.aws.amazon.com/verified-access/latest/ug/data-protection.html)
- **[transport security]** Require TLS 1.2 at minimum (TLS 1.3 recommended) for all communication with Verified Access endpoints and backing applications. [doc](https://docs.aws.amazon.com/verified-access/latest/ug/data-protection.html)
- **[sensitive data hygiene]** Never place confidential or sensitive information such as customer email addresses into Verified Access resource tags or free-form name fields, since this data can surface in billing or diagnostic logs. [doc](https://docs.aws.amazon.com/verified-access/latest/ug/data-protection.html)
- **[application-side authorization]** Treat Verified Access policies as the authentication/initial-authorization layer and keep fine-grained in-app authorization (specific actions/resources per user) inside the application itself using the signed identity claims Verified Access forwards. [doc](https://aws.amazon.com/blogs/security/a-walk-through-aws-verified-access-policies/)

## 🛡️ Reliability
- **[timeout awareness]** Design backing applications to respond within the 60-second Verified Access connection idle timeout — requests exceeding it return an HTTP 504 gateway timeout to the client. [doc](https://docs.aws.amazon.com/verified-access/latest/ug/verified-access-quotas.html)
- **[quota planning]** Check Verified Access account-level quotas (instances, groups, trust providers, endpoints) in each target Region before large-scale or multi-account rollouts, and request increases ahead of deployment where needed. [doc](https://docs.aws.amazon.com/verified-access/latest/ug/verified-access-quotas.html)
- **[capacity validation]** Monitor performance under expected load and authenticated-traffic bursts before cutting production users over, since Verified Access endpoints sit directly in the request path to every protected application. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/building-zero-trust-access-across-multi-account-aws-environments/)
- **[migration continuity]** When migrating from Client VPN or another access method, onboard new applications to Verified Access incrementally while keeping the existing solution live for unmigrated applications, rather than cutting all applications over at once. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/aws-client-vpn-and-aws-verified-access-migration-and-interoperability-patterns/)

## ⚡ Performance Efficiency
- **[same-VPC connectivity]** Deploy Verified Access endpoints directly in the same VPC as the target application where possible — endpoints connect natively within VPC subnets without requiring Transit Gateway or VPC peering, and support shared VPCs via subnet sharing. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/building-zero-trust-access-across-multi-account-aws-environments/)
- **[protocol fit]** Use HTTP(S) endpoints for browser-based applications and TCP endpoints (SSH, RDP, and other TCP protocols) for non-HTTP(S) applications such as Git repositories, databases, and EC2 instance groups, matching the endpoint type to the actual application protocol. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/security-reference-architecture/network.html)

## 💰 Cost Optimization
- **[policy consolidation]** Consolidate applications with common security requirements under shared Verified Access Groups instead of provisioning redundant per-application trust providers and duplicated policy logic, reducing management and quota overhead. [doc](https://docs.aws.amazon.com/whitepapers/latest/building-scalable-secure-multi-vpc-network-infrastructure/centralized-access-to-vpc-private-endpoints.html)
- **[WAF at the instance level]** Apply AWS WAF once at the Verified Access instance level for shared protection across endpoints, migrating existing per-ALB WAF policies to the instance rather than duplicating them on both the ALB and Verified Access. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/security-reference-architecture/network.html)

## ⚙️ Operational Excellence
- **[access logging]** Enable Verified Access logs and send them to CloudWatch Logs, Amazon S3, or Amazon Data Firehose so every access attempt is centrally recorded for audits and incident response. [doc](https://docs.aws.amazon.com/verified-access/latest/ug/access-logs.html)
- **[trust context in logs]** Enable Verified Access trust context in logging so policy evaluations capture the full user and device claims from your trust providers, making policy troubleshooting and validation far faster than reconstructing context from multiple sources. [doc](https://aws.amazon.com/blogs/security/how-to-use-aws-verified-access-logs-to-write-and-troubleshoot-access-policies/)
- **[log format]** Use the OCSF (Open Cybersecurity Schema Framework) log format for Verified Access access logs to simplify ingestion into supported SIEM and observability tooling. [doc](https://aws.amazon.com/verified-access/features/)
- **[API auditing]** Enable AWS CloudTrail to capture Verified Access API calls, including source IP and calling identity, so configuration changes to instances, groups, and policies are auditable. [doc](https://docs.aws.amazon.com/verified-access/latest/ug/monitoring-overview.html)
- **[policy testing]** Validate and iterate on Cedar policies using the Verified Access Cedar Playground before deploying changes to production groups or endpoints. [doc](https://aws.amazon.com/blogs/security/a-walk-through-aws-verified-access-policies/)
- **[WAF association status]** Check that Verified Access endpoint status has returned to `active` (not `updating`) after associating or disassociating an AWS WAF web ACL before considering the change complete. [doc](https://docs.aws.amazon.com/verified-access/latest/ug/waf-integration.html)
