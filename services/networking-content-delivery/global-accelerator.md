# AWS Global Accelerator — Best Practices

## Common scenarios
- Providing fixed global static IP addresses as a stable entry point for internet-facing apps        → Reliability, Operational Excellence
- Improving performance for latency-sensitive TCP/UDP workloads (gaming, VoIP, IoT, financial apps)        → Performance Efficiency
- Building multi-Region, multi-endpoint architectures with fast automatic failover        → Reliability
- Shielding origins (ALB, NLB, EC2) from direct internet exposure while reducing DDoS attack surface        → Security

## 🔒 Security
- **[origin exposure]** Keep Application Load Balancer, Network Load Balancer, and EC2 instance endpoints in private subnets and rely on Global Accelerator to front them, since it doesn't require public IP addresses or an internet gateway route on the endpoint to accept internet traffic. [doc](https://docs.aws.amazon.com/global-accelerator/latest/dg/secure-vpc-connections.html)
- **[attack surface reduction]** Configure listeners for only the specific ports and protocols your application needs, since Global Accelerator only accepts traffic on configured listeners, reducing the attack surface at the edge. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/well-architecting-online-applications-with-cloudfront-and-aws-global-accelerator/)
- **[security groups]** Use private subnets for EC2 and load balancer endpoints and rely on security groups to control traffic, since any public IP or Elastic IP directly assigned to an instance ENI remains reachable outside of Global Accelerator otherwise. [doc](https://docs.aws.amazon.com/global-accelerator/latest/dg/secure-vpc-connections.html)
- **[least privilege]** Apply least-privilege IAM permissions for Global Accelerator actions and move toward AWS customer managed policies scoped to specific use cases rather than broad managed policies. [doc](https://docs.aws.amazon.com/global-accelerator/latest/dg/security_iam_id-based-policy-examples.html)
- **[accidental deletion]** Restrict accelerator delete permissions (for example using tag-based/ABAC IAM policies) since deleting an accelerator permanently releases its static IP addresses and they cannot be recovered. [doc](https://docs.aws.amazon.com/global-accelerator/latest/dg/about-accelerators.html)
- **[federated access]** Require human users to sign in through federation with an identity provider (such as IAM Identity Center) and use temporary credentials instead of long-term IAM user credentials to manage accelerators. [doc](https://docs.aws.amazon.com/global-accelerator/latest/dg/security-iam.html)
- **[policy validation]** Validate identity-based policies with IAM Access Analyzer and add conditions (such as requiring SSL/TLS) to further restrict who can create, modify, or delete Global Accelerator resources. [doc](https://docs.aws.amazon.com/global-accelerator/latest/dg/security_iam_id-based-policy-examples.html)
- **[transport security]** Ensure clients calling the Global Accelerator API support TLS 1.2 at minimum (TLS 1.3 recommended) with cipher suites that provide perfect forward secrecy. [doc](https://docs.aws.amazon.com/global-accelerator/latest/dg/infrastructure-security.html)

## 🛡️ Reliability
- **[multi-Region failover]** Add endpoints in multiple AWS Regions to a standard accelerator so that Global Accelerator's continuous health checks automatically redirect traffic away from an unhealthy Region within seconds, without any DNS dependency. [doc](https://docs.aws.amazon.com/global-accelerator/latest/dg/disaster-recovery-resiliency.html)
- **[network zone resiliency]** Rely on Global Accelerator's fault-isolating design, where each of the accelerator's static IP addresses is served by an independent network zone, so clients can retry on the healthy IP if the other becomes unreachable due to blocking or network disruption. [doc](https://docs.aws.amazon.com/global-accelerator/latest/dg/disaster-recovery-resiliency.html)
- **[health check tuning]** Configure custom health check settings (interval, thresholds, path) on endpoints so Global Accelerator's health monitoring accurately reflects real endpoint health for your workload. [doc](https://docs.aws.amazon.com/global-accelerator/latest/dg/introduction-how-it-works.html)
- **[connection collision avoidance]** Configure port overrides so a single load balancer used as an endpoint by more than one accelerator (or reachable both through the accelerator and directly from the internet) receives traffic on distinct destination ports, avoiding 5-tuple hash collisions that drop connections. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/best-practices-for-deployment-with-aws-global-accelerator/)
- **[cross-zone collisions]** Disable cross-zone load balancing on Network Load Balancer endpoints when client IP address preservation is enabled, to prevent duplicate-connection errors and TCP delays caused by the client being routed to the same target from different accelerator IP addresses. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/best-practices-for-deployment-with-aws-global-accelerator/)
- **[replicate infrastructure]** Prefer replicating infrastructure with distinct accelerator/endpoint paths (rather than relying solely on port overrides) when migrating existing architectures onto Global Accelerator, to structurally avoid connection collisions. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/best-practices-for-deployment-with-aws-global-accelerator/)
- **[traffic shifting]** Use endpoint weights and the traffic dial to gradually shift traffic during blue/green deployments, stack upgrades, or performance testing instead of cutting over all traffic at once. [doc](https://docs.aws.amazon.com/global-accelerator/latest/dg/introduction-how-it-works.html)
- **[static entry point]** Use Global Accelerator's static IP addresses as allow-listed entry points for clients that are not DNS-aware or that cache DNS aggressively, so endpoint changes or Regional failover don't require client-side updates. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/well-architecting-online-applications-with-cloudfront-and-aws-global-accelerator/)

## ⚡ Performance Efficiency
- **[edge acceleration]** Use Global Accelerator to route user traffic over the AWS global network from the nearest edge location via Anycast, maximizing the portion of the path on AWS's congestion-free backbone instead of the public internet. [doc](https://docs.aws.amazon.com/global-accelerator/latest/dg/introduction-how-it-works.html)
- **[TCP optimization]** Take advantage of Global Accelerator's edge-terminated TCP connections and tuned receive-window/congestion-window settings for latency-sensitive TCP/UDP workloads (gaming, VoIP, IoT, financial trading), rather than routing directly over the public internet. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/well-architecting-online-applications-with-cloudfront-and-aws-global-accelerator/)
- **[client affinity]** Set client affinity to Source IP for stateful applications that need a given client consistently routed to the same endpoint, understanding this uses a 2-tuple hash instead of the default 5-tuple hash. [doc](https://docs.aws.amazon.com/global-accelerator/latest/dg/about-listeners-client-affinity.html)
- **[weighted routing]** Use endpoint weights to control the proportion of traffic sent to each endpoint within a group, supporting canary releases and load distribution across unevenly sized endpoint fleets. [doc](https://docs.aws.amazon.com/global-accelerator/latest/dg/about-endpoints-endpoint-weights.html)
- **[protocol fit]** Choose Global Accelerator (rather than an HTTP-layer CDN) for non-HTTP TCP/UDP workloads or non-cacheable APIs where its Layer 4 acceleration provides more benefit than content caching. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/well-architecting-online-applications-with-cloudfront-and-aws-global-accelerator/)

## 💰 Cost Optimization
- **[accelerator consolidation]** Consolidate endpoints for a single application under one accelerator where possible, since each provisioned accelerator (enabled or disabled) incurs its own fixed hourly fee. [doc](https://docs.aws.amazon.com/global-accelerator/latest/dg/introduction-pricing.html)
- **[decommission unused accelerators]** Delete accelerators that are no longer routing production traffic instead of leaving them disabled, since the fixed hourly fee is charged for every hour an accelerator is provisioned regardless of its enabled state. [doc](https://docs.aws.amazon.com/global-accelerator/latest/dg/introduction-pricing.html)
- **[traffic direction awareness]** Account for the Data Transfer-Premium fee, which is charged only on the dominant traffic direction (inbound or outbound) each hour, when modeling costs for high-throughput workloads. [doc](https://aws.amazon.com/global-accelerator/pricing/)
- **[IP address costs]** Factor in standard public IPv4 address charges for the static IPv4 addresses associated with your accelerators when estimating total cost. [doc](https://aws.amazon.com/global-accelerator/pricing/)

## ⚙️ Operational Excellence
- **[metrics and alarms]** Monitor Global Accelerator CloudWatch metrics (including RST metrics for TCP resets) in real time and create alarms to detect production issues, noting that metrics and logs must be viewed in the US West (Oregon) Region. [doc](https://docs.aws.amazon.com/global-accelerator/latest/dg/monitoring-global-accelerator.html)
- **[flow logs]** Enable Global Accelerator flow logs to capture detailed 5-tuple traffic records for troubleshooting reachability issues, diagnosing overly restrictive security groups, and supporting security audits. [doc](https://docs.aws.amazon.com/global-accelerator/latest/dg/monitoring-global-accelerator.flow-logs.html)
- **[flow log analysis]** Analyze and visualize flow log data with Amazon Athena and Amazon QuickSight to troubleshoot reachability, identify security vulnerabilities, and understand end-user access patterns. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/analyzing-and-visualizing-aws-global-accelerator-flow-logs-using-amazon-athena-and-amazon-quicksight/)
- **[API auditing]** Enable AWS CloudTrail to capture all Global Accelerator API calls, including console and SDK/CLI actions, for governance, change tracking, and incident investigation. [doc](https://docs.aws.amazon.com/global-accelerator/latest/dg/logging-and-monitoring.html)
