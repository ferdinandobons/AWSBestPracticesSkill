# Amazon Application Recovery Controller (ARC) — Best Practices

## Common scenarios
- Automated multi-Region failover using highly available routing controls        → Reliability
- Continuous recovery-readiness auditing of standby resources before an incident        → Reliability, Operational Excellence
- Automatic or manual traffic shifting away from an impaired Availability Zone        → Reliability
- Guarding against operator error during high-pressure failover events        → Reliability, Security

## 🔒 Security
- **[IAM]** Start with AWS managed policies and progressively narrow to least-privilege customer-managed policies scoped to the specific ARC actions (for example `route53-recovery-cluster:UpdateRoutingControlState`) a role actually needs. [doc](https://docs.aws.amazon.com/r53recovery/latest/dg/security_iam_id-based-policy-examples-routing.html)
- **[IAM]** Restrict who can call `UpdateRoutingControlState`/`UpdateRoutingControlStates` to a small, dedicated set of principals (users, roles, or automation) so failover actions can't be triggered by unauthorized identities. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/running-recovery-oriented-applications-with-amazon-route-53-application-recovery-controller-aws-ci-cd-tools-and-terraform/)
- **[IAM]** Add IAM policy conditions (such as requiring SSL/TLS) to further restrict access to ARC actions and resources beyond basic allow/deny. [doc](https://docs.aws.amazon.com/r53recovery/latest/dg/security_iam_id-based-policy-examples-routing.html)
- **[IAM]** Validate ARC identity-based policies with IAM Access Analyzer to catch overly permissive grants and confirm they follow IAM policy best practices. [doc](https://docs.aws.amazon.com/r53recovery/latest/dg/security_iam_id-based-policy-examples-routing.html)
- **[IAM]** Require multi-factor authentication for IAM principals that can change routing control states, given the operational and availability impact of an unauthorized failover. [doc](https://docs.aws.amazon.com/r53recovery/latest/dg/security_iam_id-based-policy-examples-routing.html)
- **[auditing]** Enable a CloudTrail trail covering ARC so that every routing control state change, cluster operation, and readiness-check API call is captured with the calling identity for audit and incident forensics. [doc](https://docs.aws.amazon.com/r53recovery/latest/dg/cloudtrail-routing.html)

## 🛡️ Reliability
- **[routing controls]** Rely on ARC routing controls for failover rather than control-plane API calls — the cluster runs on a quorum of five independent Regional endpoints, so failover still works even if up to two Regional endpoints are unavailable. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/creating-disaster-recovery-mechanisms-using-amazon-route-53/)
- **[safety rules]** Define assertion and gating safety rules (for example, requiring at least N of the zonal routing controls to stay ON) to prevent an operator from accidentally disabling all cells at once and causing an unintended fail-open. [doc](https://docs.aws.amazon.com/r53recovery/latest/dg/routing-control.safety-rules.html)
- **[safety rules]** Treat safety-rule overrides as exceptional and auditable — only bypass a safety rule when you've explicitly confirmed the override is correct for the situation. [doc](https://aws.amazon.com/blogs/database/best-practices-for-amazon-dynamodb-global-tables-part-2-failover-strategies/)
- **[readiness checks]** Set up readiness checks across recovery groups, cells, and resource sets so ARC continuously audits that standby capacity, configuration, and quotas are consistent and ready before you actually need to fail over. [doc](https://docs.aws.amazon.com/r53recovery/latest/dg/recovery-readiness.html)
- **[readiness checks]** Specify a DNS target resource (a Route 53 record or Network Load Balancer) for each recovery group so ARC can give more detailed architecture recommendations, including whether you have the optimal number of cells. [doc](https://docs.aws.amazon.com/r53recovery/latest/dg/recovery-readiness.evaluate-arch.html)
- **[zonal shift/autoshift]** Prescale resource capacity for static stability (N+1 across Availability Zones) and validate it with manual zonal shifts before enabling zonal autoshift, so the remaining AZs can absorb traffic when one is shifted away. [doc](https://docs.aws.amazon.com/r53recovery/latest/dg/arc-zonal-autoshift.considerations.html)
- **[zonal shift/autoshift]** Configure and regularly review weekly practice runs for zonal autoshift, with alarms attached, to continuously confirm your capacity is still scaled to tolerate the loss of one Availability Zone. [doc](https://docs.aws.amazon.com/r53recovery/latest/dg/arc-zonal-autoshift.considerations.html)
- **[zonal shift/autoshift]** Lower client keepalive duration (for example to around 300 seconds on an Application Load Balancer) so pre-existing connections reconnect quickly and actually move away from an impaired Availability Zone during a shift. [doc](https://docs.aws.amazon.com/r53recovery/latest/dg/arc-zonal-autoshift.considerations.html)
- **[architecture]** For applications siloed by Availability Zone, prefer zonal shift or zonal autoshift for AZ-level recovery — no additional ARC setup is required, unlike routing controls and readiness checks. [doc](https://docs.aws.amazon.com/r53recovery/latest/dg/recovery-readiness.html)
- **[testing]** Regularly exercise failover using routing controls (not only during real incidents) so the team is confident the mechanism, dependencies, and runbooks behave correctly under pressure. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/building-highly-resilient-applications-using-amazon-route-53-application-recovery-controller-part-1-single-region-stack/)

## ⚙️ Operational Excellence
- **[monitoring]** Publish and monitor ARC CloudWatch metrics for readiness checks, and create alarms on readiness status so degraded recovery posture is detected before an actual disaster. [doc](https://docs.aws.amazon.com/r53recovery/latest/dg/cloudwatch-readiness.html)
- **[monitoring]** Use CloudTrail, CloudWatch, and EventBridge together for ARC readiness checks to analyze patterns, trigger automated responses, and troubleshoot issues. [doc](https://docs.aws.amazon.com/r53recovery/latest/dg/monitoring-readiness.html)
- **[regional endpoint]** Remember that readiness checks and recovery control configuration must be managed in the US West (Oregon) Region endpoint, and specify that Region explicitly in CLI/SDK calls to avoid confusing operational errors. [doc](https://docs.aws.amazon.com/recovery-cluster/latest/api/what-is-recovery-control.html)
- **[runbooks]** Automate failover workflows (for example via CI/CD or Systems Manager runbooks) so that turning routing controls on/off is a tested, repeatable procedure rather than an ad hoc manual action during an incident. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/running-recovery-oriented-applications-with-amazon-route-53-application-recovery-controller-aws-ci-cd-tools-and-terraform/)
