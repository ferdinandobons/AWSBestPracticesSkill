# AWS Transit Gateway — Best Practices

## Common scenarios
- Hub-and-spoke connectivity across hundreds or thousands of VPCs        → Reliability, Operational Excellence, Cost Optimization
- Centralized traffic inspection or egress for multiple spoke VPCs        → Security, Reliability
- Hybrid connectivity to on-premises networks over Direct Connect or Site-to-Site VPN        → Reliability, Performance Efficiency
- Multi-account, multi-Region network segmentation for production/non-production isolation        → Security, Operational Excellence

## 🔒 Security
- **[data in transit]** Enable Transit Gateway Encryption Support to enforce end-to-end encryption for traffic between VPC attachments in Enforce mode, especially for inter-VPC traffic that would otherwise be unencrypted. [doc](https://docs.aws.amazon.com/vpc/latest/tgw/tgw-encryption-support.html)
- **[access control]** Use IAM policies to control which users and roles can create, modify, or delete transit gateways, attachments, and route tables. [doc](https://aws.amazon.com/transit-gateway/features/)
- **[traffic segmentation]** Use multiple transit gateway route tables with distinct associations and propagations to isolate routing domains (for example, production vs. non-production) so they cannot communicate with each other. [doc](https://docs.aws.amazon.com/whitepapers/latest/aws-vpc-connectivity-options/aws-transit-gateway.html)
- **[centralized inspection]** Route spoke-to-spoke and egress traffic through a centralized inspection VPC using dedicated hub and spoke transit gateway route tables, and use black-hole routes to block direct spoke-to-spoke communication where it isn't allowed. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/securing-egress-using-ids-ips-leveraging-transit-gateway/)
- **[cross-account guardrails]** Use AWS Organizations service control policies and IAM policies to prevent unauthorized modification of transit gateway and VPC configurations, particularly where cross-Region or cross-border data flows must be controlled. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/privacy-reference-architecture/network-account.html)
- **[controlled sharing]** When sharing a transit gateway across accounts with AWS RAM, remember the acceptor account cannot create, modify, or delete route tables, propagations, or associations, keeping routing control with the owner. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/integrate-third-party-services/architecture-3-1.html)

## 🛡️ Reliability
- **[no extra gateways for HA]** Do not deploy additional transit gateways purely for high availability — a transit gateway is highly available by design within a Region. [doc](https://docs.aws.amazon.com/vpc/latest/tgw/tgw-best-design-practices.html)
- **[per-AZ attachment subnets]** Attach at least one subnet per Availability Zone to each VPC attachment, since resources in an AZ without an attachment subnet cannot reach the transit gateway. [doc](https://docs.aws.amazon.com/vpc/latest/tgw/tgw-vpc-attachments.html)
- **[route propagation]** Enable route propagation for Direct Connect gateway and BGP Site-to-Site VPN attachments so routes update automatically instead of relying on static routes. [doc](https://docs.aws.amazon.com/vpc/latest/tgw/tgw-best-design-practices.html)
- **[BGP multipath]** Use BGP-based Site-to-Site VPN connections and enable multipath on the customer gateway device or firewall when supported, for more resilient hybrid connectivity. [doc](https://docs.aws.amazon.com/vpc/latest/tgw/tgw-best-design-practices.html)
- **[non-overlapping CIDRs]** Ensure VPC and on-premises CIDR blocks never overlap, since a transit gateway does not support routing between attachments with identical or overlapping CIDRs. [doc](https://docs.aws.amazon.com/vpc/latest/tgw/tgw-vpc-attachments.html)
- **[MTU migration care]** When migrating from VPC peering to a transit gateway, update both VPCs at the same time to avoid MTU size mismatches that can drop jumbo packets on asymmetric traffic. [doc](https://docs.aws.amazon.com/vpc/latest/tgw/tgw-best-design-practices.html)
- **[flow symmetry for appliances]** Enable Transit Gateway appliance mode on attachments to centralized appliance VPCs (such as those behind a Gateway Load Balancer) to maintain flow symmetry when attachment Availability Zones are misaligned across VPCs. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/centralized-inspection-architecture-with-aws-gateway-load-balancer-and-aws-transit-gateway/)
- **[route table scope]** Limit the number of transit gateway route tables to what your segmentation design genuinely requires, since unnecessary route tables add operational complexity without a reliability benefit. [doc](https://docs.aws.amazon.com/vpc/latest/tgw/tgw-best-design-practices.html)
- **[disaster recovery]** Use a single transit gateway per Region for disaster recovery designs, and use unique Autonomous System Numbers (ASNs) plus inter-Region peering when multiple transit gateways are required. [doc](https://docs.aws.amazon.com/vpc/latest/tgw/tgw-best-design-practices.html)
- **[global network design]** Ensure VPC CIDR blocks don't overlap, design for no single point of failure, and avoid building regional dependencies into application architectures when connecting transit gateways across Regions. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/using-the-aws-cdk-and-aws-transit-gateway-inter-region-peering-to-build-a-global-network/)

## ⚡ Performance Efficiency
- **[dedicated attachment subnets]** Use a separate, small-CIDR subnet (for example, `/28`) for each transit gateway VPC attachment so more addresses remain available for EC2 resources in the workload subnets. [doc](https://docs.aws.amazon.com/vpc/latest/tgw/tgw-best-design-practices.html)
- **[open ACLs on attachment subnets]** Keep network ACLs on transit gateway attachment subnets open (inbound and outbound) and apply more restrictive ACLs on workload subnets instead, to avoid unexpected traffic drops at the attachment layer. [doc](https://docs.aws.amazon.com/vpc/latest/tgw/tgw-best-design-practices.html)
- **[consistent route tables]** Associate the same VPC route table with all subnets tied to a transit gateway attachment unless your design needs multiple route tables, such as a middlebox VPC routing through several NAT gateways. [doc](https://docs.aws.amazon.com/vpc/latest/tgw/tgw-best-design-practices.html)
- **[high-throughput hybrid links]** Use Transit Gateway Connect attachments (up to 20 Gbps, 8500-byte MTU) instead of Site-to-Site VPN (up to 1.25 Gbps, 1500-byte MTU) when integrating SD-WAN appliances that need higher bandwidth and jumbo frame support. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/migrating-sd-wan-appliances-to-aws-transit-gateway-connect/)
- **[per-AZ bandwidth awareness]** Design attachments with the per-Availability-Zone bandwidth ceiling of a transit gateway VPC attachment in mind, spreading heavy traffic flows across AZs where needed. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/saas-network-access-options/options-aws.html)

## 💰 Cost Optimization
- **[chargeback attribution]** Use Flexible Cost Allocation metering policies to assign transit gateway data-processing charges to the source or destination attachment owner (rather than the default sender-based model) so centralized inspection or egress accounts aren't stuck absorbing costs incurred by spoke workloads. [doc](https://docs.aws.amazon.com/vpc/latest/tgw/metering-policy.html)
- **[traffic-based chargeback]** Use Transit Gateway Flow Logs to analyze data processing volumes per attachment and build accurate chargeback reports in multi-account environments. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/using-aws-transit-gateway-flow-logs-to-chargeback-data-processing-costs-in-a-multi-account-environment/)
- **[attachment lifecycle]** Delete unused transit gateway attachments promptly, since VPC, VPN, Direct Connect, and peering attachments are billed hourly for as long as they remain attached. [doc](https://aws.amazon.com/transit-gateway/pricing/)
- **[peering vs. Connect cost]** Prefer Transit Gateway Connect or Private IP VPN attachments over standalone VPN/Direct Connect where applicable, since these don't carry additional data-processing charges beyond the underlying attachment. [doc](https://aws.amazon.com/transit-gateway/pricing/)

## ⚙️ Operational Excellence
- **[centralized ownership]** Place your organization's transit gateway in a dedicated Network Services account and share it with other accounts using AWS RAM (manually or via AWS Organizations sharing) for centralized management by your network team. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/patterns/centralize-network-connectivity-using-aws-transit-gateway.html)
- **[route table organization]** Create dedicated transit gateway route tables per routing domain (for example, Prod, Staging, Network Services) and manage associations/propagations explicitly rather than relying on defaults. [doc](https://aws.amazon.com/blogs/architecture/field-notes-working-with-route-tables-in-aws-transit-gateway/)
- **[monitoring and observability]** Enable Transit Gateway Flow Logs and CloudWatch metrics (bytes, packets, drop count) to monitor traffic patterns and troubleshoot connectivity issues, and use CloudTrail to audit control-plane API calls. [doc](https://docs.aws.amazon.com/vpc/latest/tgw/transit-gateway-monitoring.html)
- **[global visibility]** Register transit gateways in AWS Network Manager to centrally monitor topology, routing, and status across accounts, Regions, and on-premises locations, and use CloudWatch Events for change notifications. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/achieving-traffic-segmentation-in-multi-aws-region-environments-using-aws-transit-gateway-and-aws-cloud-wan/)
- **[quota awareness]** Track transit gateway quotas (such as attachments per transit gateway and peering attachments) and request limit increases proactively before they block new attachments at scale. [doc](https://docs.aws.amazon.com/vpc/latest/tgw/transit-gateway-quotas.html)
