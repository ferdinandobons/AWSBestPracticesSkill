# AWS Direct Connect — Best Practices

## Common scenarios
- Hybrid connectivity between on-premises data centers and AWS VPCs        → Reliability, Performance Efficiency
- Reducing reliance on public internet for large, consistent data transfers        → Performance Efficiency, Cost Optimization
- Meeting compliance/security requirements for private network paths        → Security, Reliability
- Connecting multiple VPCs/accounts across Regions to a single on-premises network        → Operational Excellence, Cost Optimization

## 🔒 Security
- **[data in transit]** Encrypt sensitive traffic with MACsec on 10/100/400 Gbps dedicated connections at supported locations — Direct Connect does not encrypt traffic by default, and MACsec provides point-to-point Layer 2 confidentiality, integrity, and origin authenticity. [doc](https://docs.aws.amazon.com/directconnect/latest/UserGuide/MACsec.html)
- **[data in transit]** Combine Direct Connect with an IPsec Site-to-Site VPN when end-to-end encryption is required — MACsec is hop-by-hop only, so layer an IPsec overlay for full encryption between your network and AWS. [doc](https://docs.aws.amazon.com/directconnect/latest/UserGuide/encryption-in-transit.html)
- **[identity and access]** Grant least-privilege IAM permissions for Direct Connect actions and move from AWS managed policies toward customer-managed, scoped policies — limits blast radius of compromised credentials or misconfigured automation. [doc](https://docs.aws.amazon.com/directconnect/latest/UserGuide/security_iam_id-based-policy-examples.html)
- **[identity and access]** Add IAM policy conditions (e.g., require SSL/TLS, restrict source IP or VPC endpoint) and validate policies with IAM Access Analyzer — further restricts who can create, modify, or delete Direct Connect resources. [doc](https://docs.aws.amazon.com/directconnect/latest/UserGuide/security_iam_id-based-policy-examples.html)
- **[identity and access]** Require multi-factor authentication (MFA) for accounts that manage Direct Connect resources — protects against credential compromise for a service that controls network-level access to your environment. [doc](https://docs.aws.amazon.com/directconnect/latest/UserGuide/data-protection.html)
- **[routing hygiene]** Ensure BGP prefixes advertised over a public virtual interface are never re-advertised beyond your network boundary (e.g., onto the public internet) — prevents route leaks and unintended exposure. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/salesforce-hyperforce-connectivity-dx-megaport/vpn-workforce.html)
- **[client access]** Use TLS 1.2 or later (TLS 1.3 recommended) with perfect-forward-secrecy cipher suites for all API calls to Direct Connect, and sign requests with IAM credentials or temporary STS credentials — protects management-plane traffic. [doc](https://docs.aws.amazon.com/directconnect/latest/UserGuide/infrastructure-security.html)
- **[auditability]** Enable AWS CloudTrail logging for Direct Connect API calls and monitor log files in near real time — supports incident response and detection of unauthorized configuration changes. [doc](https://docs.aws.amazon.com/directconnect/latest/UserGuide/dc-incident-response.html)

## 🛡️ Reliability
- **[connection redundancy]** Use the AWS Direct Connect Resiliency Toolkit connection wizard to select a resiliency model (maximum resiliency, high resiliency, or development/test) matched to your SLA requirements — ensures the right number and placement of connections before you order. [doc](https://docs.aws.amazon.com/directconnect/latest/UserGuide/resiliency_toolkit.html)
- **[maximum resiliency]** For critical workloads, provision separate connections that terminate on separate devices in more than one Direct Connect location — protects against device, connectivity, and complete location failures, and is required for the 99.99% SLA. [doc](https://docs.aws.amazon.com/directconnect/latest/UserGuide/disaster-recovery-resiliency.html)
- **[maintenance readiness]** Configure connections for maximum resiliency so that planned maintenance events drain traffic to redundant connections automatically — AWS never schedules maintenance that would take down all redundant paths simultaneously. [doc](https://docs.aws.amazon.com/directconnect/latest/UserGuide/dx-maintenance.html)
- **[failover validation]** Regularly test failover using the Direct Connect Resiliency Toolkit failover test feature before relying on redundant paths in production — proactively surfaces misconfigurations that would otherwise only appear during a real outage. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/testing-aws-direct-connect-resiliency-with-resiliency-toolkit-failover-testing/)
- **[backup path]** Configure a backup IPsec VPN connection for Direct Connect so VPC-bound traffic fails over automatically if the dedicated connection is lost — without a backup path, VPC traffic is dropped on failure. [doc](https://aws.amazon.com/directconnect/faqs/)
- **[capacity planning]** Provision each redundant connection with enough capacity to absorb full traffic if its peer connection fails, and avoid collapsing multiple diverse LAGs into a single higher-speed connection without preserving redundancy — prevents a resiliency regression when upgrading link speed. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/upgrading-aws-direct-connect-to-100-gbps-in-5-steps/)
- **[continuous monitoring]** Use Amazon CloudWatch Network Monitor to actively measure packet loss, latency, and the AWS Network Health Indicator on hybrid paths — pinpoints whether degradation originates within the AWS network or on-premises. [doc](https://docs.aws.amazon.com/directconnect/latest/UserGuide/dx-maintenance.html)
- **[operational events]** When you receive a Direct Connect operational event notification, verify application traffic, check CloudWatch connection metrics, and confirm redundant connections have absorbed the shift — confirms resiliency configurations are working as designed. [doc](https://docs.aws.amazon.com/directconnect/latest/UserGuide/dx-operational-notifications.html)

## ⚡ Performance Efficiency
- **[throughput scaling]** Use Link Aggregation Groups (LAGs) to bundle multiple physical connections of the same bandwidth into a single logical, LACP-managed interface — increases throughput while presenting one BGP session to manage. [doc](https://docs.aws.amazon.com/directconnect/latest/APIReference/API_CreateLag.html)
- **[packet size]** Enable jumbo frames (MTU 9001 for private virtual interfaces, 8500 for transit virtual interfaces) where end-to-end support exists — reduces per-packet overhead and improves throughput for large data transfers. [doc](https://docs.aws.amazon.com/directconnect/latest/UserGuide/create-vif.html)
- **[traffic engineering]** Use BGP local preference communities and longest-prefix-match advertisements to influence which path inbound and outbound traffic takes across multiple connections — gives predictable active/passive or load-shared behavior instead of relying on default path selection. [doc](https://docs.aws.amazon.com/directconnect/latest/UserGuide/routing-and-bgp.html)
- **[multi-path load sharing]** Rely on equal-cost multi-path (ECMP) routing across multiple transit or private virtual interfaces with matching AS_PATH length and BGP attributes to share load across connections — increases effective throughput without manual traffic splitting. [doc](https://docs.aws.amazon.com/directconnect/latest/UserGuide/routing-and-bgp.html)
- **[global reach]** Use AWS Direct Connect gateway to connect a single set of connections to VPCs and virtual private gateways across multiple AWS Regions — avoids provisioning separate physical connections per Region. [doc](https://aws.amazon.com/directconnect/faqs/)
- **[site-to-site connectivity]** Use Direct Connect SiteLink to establish private, end-to-end paths directly between your own Direct Connect locations without routing through an AWS Region — reduces latency for site-to-site traffic that doesn't need to traverse AWS Regional infrastructure. [doc](https://aws.amazon.com/directconnect/features/)

## 💰 Cost Optimization
- **[connection consolidation]** Use Direct Connect gateway to share private virtual interfaces across up to 20 VPCs and to attach transit virtual interfaces to multiple Transit Gateways — reduces the number of physical connections and BGP sessions needed. [doc](https://aws.amazon.com/directconnect/faqs/)
- **[architecture choice]** Route multi-VPC traffic through a Transit Gateway attached via a transit virtual interface rather than provisioning direct connections per VPC — consolidates data-transfer-out billing and simplifies attachment management, though standard Transit Gateway attachment pricing applies. [doc](https://aws.amazon.com/blogs/architecture/overview-of-data-transfer-costs-for-common-architectures/)
- **[cost attribution]** In multi-account architectures, understand that Direct Connect port-hour charges are billed to the connection owner while data-transfer-out charges follow the account owning the Transit Gateway/VPC attachment sending the traffic — design account and attachment ownership deliberately to align cost with the consuming team. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/understanding-aws-direct-connect-multi-account-pricing/)
- **[right-sizing]** Size LAG bundles and connection speed to actual sustained throughput needs rather than defaulting to the largest available port, and re-evaluate when upgrading to higher-speed connections — avoids paying for unused capacity while still preserving redundancy. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/upgrading-aws-direct-connect-to-100-gbps-in-5-steps/)

## ⚙️ Operational Excellence
- **[monitoring]** Monitor Direct Connect connections and virtual interfaces with Amazon CloudWatch metrics (e.g., ConnectionState, ConnectionBpsEgress/Ingress, ConnectionPpsEgress/Ingress) and create alarms on state changes — provides automated detection of connectivity or throughput anomalies. [doc](https://docs.aws.amazon.com/directconnect/latest/UserGuide/monitoring-cloudwatch.html)
- **[automation]** Prefer automated monitoring tools (CloudWatch Alarms, CloudTrail log monitoring) over manual console checks, and reserve manual dashboard review for what alarms don't cover — reduces mean time to detect issues. [doc](https://docs.aws.amazon.com/directconnect/latest/UserGuide/monitoring-overview.html)
- **[incident response]** Establish a runbook for Direct Connect operational event notifications that includes checking CloudWatch metrics, validating Network Monitor packet loss/latency, and confirming failover to redundant connections — speeds up triage and separates AWS-side from on-premises-side issues. [doc](https://docs.aws.amazon.com/directconnect/latest/UserGuide/dx-operational-notifications.html)
- **[change auditing]** Enable AWS CloudTrail for Direct Connect and share/monitor log files across accounts in near real time — supports root-cause analysis and change tracking for network configuration. [doc](https://docs.aws.amazon.com/directconnect/latest/UserGuide/dc-incident-response.html)
- **[proactive testing]** Periodically exercise the Direct Connect Resiliency Toolkit failover test rather than waiting for an unplanned outage to discover configuration gaps — builds confidence in documented resiliency posture ahead of real maintenance or failure events. [doc](https://docs.aws.amazon.com/directconnect/latest/UserGuide/dx-maintenance.html)
