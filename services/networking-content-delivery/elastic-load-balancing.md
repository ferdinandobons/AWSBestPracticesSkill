# Elastic Load Balancing — Best Practices

## Common scenarios
- Distributing HTTP/HTTPS traffic across microservices or containers        → Performance Efficiency, Reliability
- Terminating TLS/SSL centrally in front of EC2, ECS, or EKS targets        → Security, Performance Efficiency
- Providing a highly available, multi-AZ entry point for internet-facing apps        → Reliability, Operational Excellence
- Load balancing high-throughput or latency-sensitive TCP/UDP traffic        → Performance Efficiency, Cost Optimization

## 🔒 Security
- **[Listener encryption]** Use HTTPS/TLS listeners with a predefined security policy and ACM-managed or IAM-managed certificates instead of plaintext HTTP/TCP listeners — encrypts data in transit and centralizes certificate management. [doc](https://docs.aws.amazon.com/elasticloadbalancing/latest/userguide/infrastructure-security.html)
- **[Traffic redirects]** Enforce redirects from HTTP to HTTPS on Application Load Balancer listeners — prevents clients from inadvertently sending sensitive data unencrypted. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/patterns/migrate-windows-ssl-certificates-to-an-application-load-balancer-using-acm.html)
- **[Load balancer security groups]** Restrict the load balancer's security group to accept inbound traffic only from expected clients and only on the listener ports it needs — reduces the exposed attack surface. [doc](https://docs.aws.amazon.com/elasticloadbalancing/latest/userguide/infrastructure-security.html)
- **[Target security groups]** Configure target (EC2/ECS) security groups to accept traffic only from the load balancer's security group, on the listener and health-check ports — ensures targets are reachable exclusively through the load balancer and can't be bypassed directly. [doc](https://docs.aws.amazon.com/elasticloadbalancing/latest/application/load-balancer-update-security-groups.html)
- **[Private targets]** Deploy Application Load Balancer targets in private subnets without public or Elastic IP addresses — prevents direct internet access to targets that bypasses the load balancer. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/security-best-practices-when-using-alb-authentication/)
- **[ALB authentication]** When using ALB built-in authentication, validate the JWT signature and confirm the `signer` header matches the ALB's ARN on the target — prevents forged authentication headers from reaching your application. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/security-best-practices-when-using-alb-authentication/)
- **[Web application protection]** Attach AWS WAF web ACLs to internet-facing Application Load Balancers to block or monitor requests using IP, header, and URI-based rules — guards against common web exploits and unwanted bots. [doc](https://docs.aws.amazon.com/elasticloadbalancing/latest/userguide/infrastructure-security.html)
- **[Least privilege IAM]** Scope IAM policies for ELB separately for service users (day-to-day target/listener changes) and service administrators (load balancer creation, security policy changes) — limits blast radius of compromised credentials. [doc](https://aws.amazon.com/blogs/industries/fsi-services-spotlight-featuring-elastic-load-balancing-elb-2/)
- **[Access log encryption]** Enable server-side encryption (SSE-S3 or SSE-KMS) on the S3 bucket that stores ELB access logs — protects request-level log data containing client IPs and paths at rest. [doc](https://aws.amazon.com/blogs/industries/fsi-services-spotlight-featuring-elastic-load-balancing-elb-2/)
- **[Path MTU discovery]** Allow inbound ICMP traffic in target security groups to support Path MTU Discovery — avoids silent packet-drop issues that look like security misconfigurations. [doc](https://docs.aws.amazon.com/elasticloadbalancing/latest/application/load-balancer-update-security-groups.html)

## 🛡️ Reliability
- **[Multi-AZ deployment]** Deploy each load balancer across at least two Availability Zones with healthy, evenly distributed targets in each — enables automatic failover if an entire AZ becomes unavailable. [doc](https://docs.aws.amazon.com/awssupport/latest/user/fault-tolerance-checks.html)
- **[Health checks]** Configure application-layer health checks with an appropriate threshold and interval on every target group — ensures traffic is routed only to targets that can actually serve requests. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/patterns/migrate-windows-ssl-certificates-to-an-application-load-balancer-using-acm.html)
- **[Connection draining / deregistration delay]** Set a deregistration delay of at least 120 seconds (longer for slow requests) so in-flight requests complete before a target is fully deregistered — avoids dropped connections during scale-in, deployments, or unhealthy-instance replacement. [doc](https://docs.aws.amazon.com/elasticloadbalancing/latest/network/edit-target-group-attributes.html)
- **[Cross-zone load balancing]** Enable cross-zone load balancing when you need traffic distributed evenly across all healthy targets regardless of AZ, not just within each node's own AZ — improves resiliency when target counts are uneven across AZs. [doc](https://docs.aws.amazon.com/elasticloadbalancing/latest/network/load-balancer-target-groups.html)
- **[DNS name usage]** Point application/domain configuration at the load balancer's DNS name rather than resolved IP addresses — load balancer nodes and IPs can change, and DNS is how ELB fails over between AZs. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/patterns/migrate-windows-ssl-certificates-to-an-application-load-balancer-using-acm.html)
- **[Target group health thresholds]** Tune DNS failover and routing failover healthy-host thresholds on target groups to match your workload's tolerance for partial AZ failure — controls when the load balancer shifts traffic away from a degraded AZ versus keeps routing to remaining healthy targets there. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/using-load-balancer-target-group-health-thresholds-to-improve-availability/)
- **[Idle timeout tuning]** Set the load balancer's connection idle timeout lower than any upstream client/proxy timeout, and ensure your application's own idle timeout is longer than the load balancer's — mismatched timeouts cause spurious HTTP 502 errors. [doc](https://docs.aws.amazon.com/elasticloadbalancing/latest/application/edit-load-balancer-attributes.html)

## ⚡ Performance Efficiency
- **[Load balancer type selection]** Choose Application Load Balancer for HTTP/HTTPS/WebSocket workloads needing advanced request routing, and Network Load Balancer for ultra-low-latency TCP/UDP/TLS workloads with millions of requests per second — matching the load balancer to protocol and traffic characteristics avoids unnecessary overhead. [doc](https://docs.aws.amazon.com/wellarchitected/latest/framework/perf_networking_load_balancing_distribute_traffic.html)
- **[TLS offload]** Offload SSL/TLS termination to the load balancer instead of terminating on each target — frees target CPU from handshake overhead and centralizes certificate/cipher management. [doc](https://docs.aws.amazon.com/wellarchitected/latest/framework/perf_networking_load_balancing_distribute_traffic.html)
- **[HTTP/2]** Enable HTTP/2 on Application Load Balancer listeners so it can serve HTTP/2 to clients even when targets only speak HTTP/1.1 — improves client response times through more efficient TCP connection use. [doc](https://docs.aws.amazon.com/wellarchitected/latest/framework/perf_networking_load_balancing_distribute_traffic.html)
- **[Front every workload]** Place a load balancer in front of internet-facing workloads rather than exposing compute directly, and route only the traffic that needs it through existing load balancers — matches Well-Architected guidance against exposing workloads without load balancing or overloading a single load balancer with unrelated traffic. [doc](https://docs.aws.amazon.com/wellarchitected/latest/framework/perf_networking_load_balancing_distribute_traffic.html)

## 💰 Cost Optimization
- **[Cross-zone charges]** Understand that enabling cross-zone load balancing on Network/Gateway Load Balancer incurs standard inter-AZ data transfer charges — weigh the availability benefit against the added cost for your traffic volume. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/best-practices-for-deploying-gateway-load-balancer/)
- **[Balanced AZ usage]** Avoid enabling Availability Zones with no registered targets and avoid large imbalances in target counts across enabled AZs — unused or imbalanced AZ configuration still accrues load balancer charges without a fault-tolerance benefit. [doc](https://docs.aws.amazon.com/awssupport/latest/user/fault-tolerance-checks.html)
- **[Deregistration delay tuning]** Reduce deregistration delay for short-lived, fast-response services (while keeping it higher for long-lived requests like uploads or streaming) — shortens the window a draining target keeps consuming capacity during frequent deployments. [doc](https://docs.aws.amazon.com/AmazonECS/latest/developerguide/load-balancer-connection-draining.html)

## ⚙️ Operational Excellence
- **[Access logs]** Enable Elastic Load Balancing access logs to an S3 bucket for every production load balancer — provides request-level detail (client IP, latency, path, response code) for traffic analysis and troubleshooting. [doc](https://docs.aws.amazon.com/elasticloadbalancing/latest/application/load-balancer-access-logs.html)
- **[CloudWatch metrics]** Monitor ELB CloudWatch metrics (request counts, error counts/types, latency for ALB; active/new flow counts and processed bytes for NLB) and alarm on anomalies — verifies the load balancer and its targets are performing as expected. [doc](https://docs.aws.amazon.com/elasticloadbalancing/latest/classic/elb-monitor-logs.html)
- **[API auditing]** Enable AWS CloudTrail for Elastic Load Balancing API calls — tracks who changed listener, target group, or security configuration and when, supporting incident investigation. [doc](https://docs.aws.amazon.com/elasticloadbalancing/latest/classic/elb-monitor-logs.html)
- **[Reverse proxy log parity]** If you run a self-managed reverse proxy behind or alongside ELB, forward its access logs to CloudWatch (e.g., via the unified CloudWatch agent) so you retain the same visibility as native ELB access logs. [doc](https://aws.amazon.com/blogs/security/logging-strategies-for-security-incident-response/)
