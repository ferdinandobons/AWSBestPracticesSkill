# AWS Cloud WAN — Best Practices

## Common scenarios
- Connecting VPCs, data centers, and branch offices into one global network        → Reliability, Operational Excellence
- Segmenting traffic between production, development, and business units        → Security
- Centralizing firewall/IDS-IPS inspection for east-west and internet-bound traffic        → Security, Reliability
- Building multi-Region, highly available hybrid connectivity with Direct Connect and VPN        → Reliability, Performance Efficiency, Cost Optimization

## 🔒 Security
- **[network segmentation]** Design a clear segmentation strategy up front, isolating traffic (for example production, development, or business units) into separate Cloud WAN segments that align with your security and compliance requirements. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/designing-for-global-scale-xm-cybers-22-region-aws-cloud-wan-implementation/)
- **[segment isolation]** Enable `isolate-attachments` on segments so attachments in the same segment cannot communicate directly with each other, forcing traffic through required inspection or shared-services paths. [doc](https://docs.aws.amazon.com/network-manager/latest/cloudwan/cloudwan-policy-examples-service-insertion.html)
- **[least-privilege inter-segment access]** Use segment actions (`share`, `send-via`, `send-to`) to explicitly and narrowly define which segments can reach each other, rather than allowing broad default connectivity between segments. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/inspecting-network-traffic-between-amazon-vpcs-with-aws-cloud-wan/)
- **[centralized inspection]** Route intra-segment, cross-segment, and internet-bound traffic through a centralized inspection VPC or network function group using AWS Network Firewall, Gateway Load Balancer, or third-party appliances defined in the core network policy. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/simplify-global-security-inspection-with-aws-cloud-wan-service-insertion/)
- **[stateful inspection symmetry]** Turn on appliance mode for VPC attachments hosting stateful security appliances so forward and return traffic for a flow is pinned to the same Availability Zone and appliance. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/inspecting-network-traffic-between-amazon-vpcs-with-aws-cloud-wan/)
- **[least-privilege IAM]** Create IAM policies that grant only the specific Cloud WAN and Network Manager actions and resources each user or role needs, rather than broad global-network access. [doc](https://docs.aws.amazon.com/network-manager/latest/cloudwan/cloudwan-security.html)
- **[global network access awareness]** Remember that granting access to a global network grants access to all core network edges across every AWS Region it spans, and scope IAM policies accordingly. [doc](https://docs.aws.amazon.com/network-manager/latest/cloudwan/cloudwan-security.html)
- **[controlled cross-account sharing]** Share core networks across accounts or an AWS Organization using AWS RAM with least-privilege resource-share permissions, and require sign-off from the network owner rather than open sharing. [doc](https://docs.aws.amazon.com/network-manager/latest/cloudwan/cloudwan-share-network.html)
- **[attachment acceptance controls]** Require attachment acceptance on segments and network function groups so new VPC, VPN, or Transit Gateway attachments cannot join a segment without explicit approval. [doc](https://docs.aws.amazon.com/network-manager/latest/cloudwan/cloudwan-policy-examples-service-insertion.html)
- **[automated attachment governance]** Automate attachment-to-segment mapping and validation (for example with EventBridge and Lambda) to prevent cross-network domain leakage and remove manual, error-prone configuration steps. [doc](https://aws.amazon.com/solutions/guidance/attachment-management-to-aws-cloud-wan/)
- **[periodic segmentation audits]** Regularly audit segment isolation and network access logs, and apply least-privilege principles when defining inter-segment communication rules. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/designing-for-global-scale-xm-cybers-22-region-aws-cloud-wan-implementation/)

## 🛡️ Reliability
- **[per-Region hybrid redundancy]** Maintain a Transit Gateway in every Region where you have Direct Connect-based hybrid connectivity through Cloud WAN, so connectivity does not depend on a remote Region being available. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/achieve-optimal-routing-with-aws-cloud-wan-for-multi-region-networks/)
- **[redundant Direct Connect]** Provision redundant Direct Connect connections from at least two separate colocation facilities to avoid a single physical point of failure in hybrid connectivity. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/achieve-optimal-routing-with-aws-cloud-wan-for-multi-region-networks/)
- **[built-in HA]** Rely on Cloud WAN's built-in high availability (core network edges are highly available by design) rather than provisioning extra Transit Gateways or core network edges per Region purely for redundancy. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/achieve-optimal-routing-with-aws-cloud-wan-for-multi-region-networks/)
- **[non-overlapping IP planning]** Conduct thorough CIDR planning before rollout to avoid overlapping address spaces across AWS Regions and on-premises networks. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/designing-for-global-scale-xm-cybers-22-region-aws-cloud-wan-implementation/)
- **[policy change safety]** Create and validate new core network policy versions (and compare them against the current version) before deploying, and keep the ability to restore an out-of-date policy version if a change causes issues. [doc](https://docs.aws.amazon.com/network-manager/latest/cloudwan/cloudwan-create-policy-version.html)
- **[staged rollout]** Use infrastructure as code for policies and attachments, validate changes with pilot deployments in non-critical Regions first, and maintain backup connectivity and rollback procedures during migrations. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/designing-for-global-scale-xm-cybers-22-region-aws-cloud-wan-implementation/)
- **[multi-AZ attachments]** Deploy Cloud WAN VPC attachments across multiple Availability Zones, since single-AZ attachments fall under a lower SLA tier than multi-AZ deployments. [doc](https://aws.amazon.com/cloud-wan/sla/)
- **[segment quota awareness]** Account for per-core-network quotas on segments and other resources when designing segment-splitting or multi-Region routing architectures, and request quota increases proactively where needed. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/achieve-optimal-routing-with-aws-cloud-wan-for-multi-region-networks/)
- **[egress independence per Region]** Provision an Egress VPC in each Region that has workloads needing centralized internet egress, to avoid taking a cross-Region dependency for internet access. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/achieve-optimal-routing-with-aws-cloud-wan-for-multi-region-networks/)

## ⚡ Performance Efficiency
- **[attachment CIDR sizing]** Use a small, dedicated CIDR (for example a /28) for each core network attachment subnet so remaining VPC address space stays available for other resources. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/achieve-optimal-routing-with-aws-cloud-wan-for-multi-region-networks/)
- **[open attachment ACLs]** Keep inbound and outbound network ACLs on core network attachment subnets open, since access control for these paths is intended to be enforced at the segment/policy level, not the subnet. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/achieve-optimal-routing-with-aws-cloud-wan-for-multi-region-networks/)
- **[fine-grained routing control]** Use Cloud WAN Routing Policy features such as route filtering, summarization, and BGP attribute manipulation to optimize traffic paths and troubleshoot complex, multi-path global networks. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/aws-cloud-wan-routing-policy-fine-grained-controls-for-your-global-network-part-1/)
- **[segment splitting for optimal routing]** Use segment splitting to steer traffic onto optimal paths in multi-Region designs, while tracking the per-core-network segment quota this consumes. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/achieve-optimal-routing-with-aws-cloud-wan-for-multi-region-networks/)
- **[dual-stack readiness]** Design segments and attachments to support both IPv4 and IPv6 where workloads require it, since Cloud WAN supports dual-stack traffic across core network edges. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/achieve-optimal-routing-with-aws-cloud-wan-for-multi-region-networks/)

## 💰 Cost Optimization
- **[minimize core network edges]** Enable core network edges only in the AWS Regions you actually need, since each core network edge incurs its own hourly charge. [doc](https://aws.amazon.com/blogs/apn/expanding-vmware-cloud-on-aws-multi-region-connectivity-using-aws-cloud-wan/)
- **[peering over duplicate processing]** Prefer Transit Gateway peering connections into Cloud WAN where applicable, since traffic over peered connections between a core network edge and Transit Gateway avoids additional data processing charges. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/aws-cloud-wan-and-aws-transit-gateway-migration-and-interoperability-patterns/)
- **[incremental migration]** Migrate from Transit Gateway to Cloud WAN incrementally, keeping both running side by side during transition, to control cost and risk rather than a disruptive big-bang cutover. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/aws-cloud-wan-and-aws-transit-gateway-migration-and-interoperability-patterns/)
- **[track attachment and data-processing spend]** Monitor the number of attachments, peering connections, and data processing volume per core network edge, since each is billed separately and scales with network growth. [doc](https://aws.amazon.com/cloud-wan/pricing/)

## ⚙️ Operational Excellence
- **[centralized policy management]** Define segments, routing, and security posture in a single versioned core network policy so network configuration stays consistent across all Regions and accounts. [doc](https://docs.aws.amazon.com/whitepapers/latest/building-scalable-secure-multi-vpc-network-infrastructure/aws-cloud-wan.html)
- **[policy version control]** Create a new core network policy version for every change, compare it against the previous version before deploying, and keep the option to restore an out-of-date version if needed. [doc](https://docs.aws.amazon.com/network-manager/latest/cloudwan/cloudwan-create-policy-version.html)
- **[continuous monitoring]** Monitor core network and attachment health using Amazon CloudWatch metrics under the `AWS/NetworkManager` namespace, and set alarms on key thresholds such as latency, packet loss, and throughput. [doc](https://docs.aws.amazon.com/network-manager/latest/cloudwan/cloudwan-cloudwatch-metrics.html)
- **[event-driven visibility]** Onboard CloudWatch Logs Insights and use Amazon EventBridge to capture near-real-time core network events (such as policy change-set execution or edge-location changes) for troubleshooting and automation. [doc](https://docs.aws.amazon.com/network-manager/latest/cloudwan/cloudwan-events-metrics.html)
- **[automated attachment workflows]** Use EventBridge, SQS, and Lambda to automatically process Network Manager events and configure attachments, including a dead-letter queue for malformed events, to reduce manual oversight. [doc](https://aws.amazon.com/solutions/guidance/attachment-management-to-aws-cloud-wan/)
- **[documentation and change management]** Maintain up-to-date network diagrams, routing policies, and troubleshooting guides, and establish a clear change-management process for network modifications. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/designing-for-global-scale-xm-cybers-22-region-aws-cloud-wan-implementation/)
