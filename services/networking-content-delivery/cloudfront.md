# Amazon CloudFront — Best Practices

## Common scenarios
- Global delivery of static assets (images, video, downloads) from S3 or on-prem origins        → Performance Efficiency, Cost Optimization
- Accelerating and protecting dynamic content and APIs served from ALB, EC2, or Lambda origins        → Performance Efficiency, Security
- Serving private/paid content to authenticated or entitled users        → Security
- Building highly available, DDoS-resilient public-facing web applications        → Reliability, Security

## 🔒 Security
- **[origin access]** Use Origin Access Control (OAC) instead of the legacy Origin Access Identity (OAI) to restrict Amazon S3 origins — OAC supports all Regions, SSE-KMS encryption, and dynamic PUT/DELETE requests that OAI cannot handle. [doc](https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/private-content-restricting-access-to-s3.html)
- **[origin bypass prevention]** Configure a custom origin header with a secret rotated via AWS Secrets Manager (validated by the origin) or restrict origin security groups to CloudFront's managed IP prefix list — prevents traffic from bypassing CloudFront and hitting the origin directly. [doc](https://aws.amazon.com/blogs/security/how-to-enhance-amazon-cloudfront-origin-security-with-aws-waf-and-aws-secrets-manager/)
- **[perimeter protection]** Attach AWS WAF web ACLs (rate-based rules, Bot Control, ATP) to distributions in front of dynamic or API workloads, and add AWS Shield Advanced where extra DDoS protection is needed — stops unwanted traffic at the edge before it reaches the origin. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/accelerate-protect-and-make-dynamic-workloads-delivery-cost-efficient-with-amazon-cloudfront/)
- **[private content]** Serve private or entitled content using CloudFront signed URLs or signed cookies rather than exposing origin URLs directly — ensures only authorized viewers can retrieve objects. [doc](https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/SecurityAndPrivateContent.html)
- **[transport encryption]** Require HTTPS between viewers and CloudFront and between CloudFront and the origin, and select a security policy of at least TLSv1.2_2021 — controls the minimum TLS version and cipher suite CloudFront accepts. [doc](https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/DownloadDistValuesGeneral.html)
- **[geographic restriction]** Apply CloudFront geographic restrictions (geo-blocking) when content must be limited to specific countries — enforces licensing or regulatory access requirements. [doc](https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/SecurityAndPrivateContent.html)
- **[sensitive fields]** Enable field-level encryption for specific sensitive fields such as payment or personal data — keeps those fields encrypted throughout the entire request lifecycle, not just in transit. [doc](https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/SecurityAndPrivateContent.html)
- **[data protection]** Avoid putting confidential or sensitive information, such as customer email addresses, into distribution tags or free-form name fields — this data can surface in billing or diagnostic logs. [doc](https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/data-protection-summary.html)
- **[continuous compliance]** Use AWS Config managed rules (for example CloudFrontOriginAccessIdentityEnabled, CloudFrontSniEnabled, CloudFrontViewerPolicyHttps, CloudFrontAssociatedWithWaf) or AWS Security Hub CSPM CloudFront controls — continuously evaluates distributions against security best practices. [doc](https://docs.aws.amazon.com/whitepapers/latest/secure-content-delivery-amazon-cloudfront/configuration-management.html)

## 🛡️ Reliability
- **[origin failover]** Configure an origin group with a primary and secondary origin — CloudFront automatically fails over to the secondary when the primary is unavailable or returns configured failure status codes (500, 502, 503, 504). [doc](https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/high_availability_origin_failover.html)
- **[failover tuning]** Adjust origin connection attempts and timeouts (connection attempts, connection timeout, response timeout) to match workload latency tolerance — shorter timeouts speed up failover for latency-sensitive use cases like live streaming. [doc](https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/high_availability_origin_failover.html)
- **[DDoS resiliency]** Rely on CloudFront's inline L3/L4 DDoS mitigation and AWS Shield Standard, and follow the AWS Best Practices for DDoS Resiliency guidance for the rest of the architecture — reduces the blast radius of volumetric attacks. [doc](https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/SecurityAndPrivateContent.html)
- **[health monitoring]** Pair Amazon Route 53 health checks with CloudFront metrics to monitor origin and application health — surfaces significant availability changes so you can react quickly. [doc](https://docs.aws.amazon.com/whitepapers/latest/secure-content-delivery-amazon-cloudfront/logging-and-monitoring-in-cloudfront.html)

## ⚡ Performance Efficiency
- **[cache hit ratio]** Enable CloudFront Origin Shield in the AWS Region closest to the origin — consolidates requests from all edge locations and regional edge caches into a single origin request path, improving cache hit ratio and reducing origin load. [doc](https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/origin-shield.html)
- **[cache key design]** Forward only the headers, cookies, and query string parameters the origin actually needs into the cache key — forwarding everything multiplies cached variants and lowers the cache hit ratio. [doc](https://aws.amazon.com/cloudfront/getting-started/S3/)
- **[connection reuse]** Take advantage of CloudFront's persistent connections to origins to avoid repeated TCP/TLS handshakes, and verify reuse with the Server-Timing header's cdn-upstream-connect metric — reduces latency for both static and dynamic content. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/accelerate-protect-and-make-dynamic-workloads-delivery-cost-efficient-with-amazon-cloudfront/)
- **[compression]** Enable automatic object compression for eligible text-based responses — reduces transferred bytes and improves load times. [doc](https://aws.amazon.com/cloudfront/getting-started/S3/)
- **[traffic spikes]** Rely on CloudFront's collapsed forwarding during traffic spikes — merges simultaneous cache-miss requests for the same object into a single origin request, protecting the origin from overload. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/dynamic-whole-site-delivery-with-amazon-cloudfront/)

## 💰 Cost Optimization
- **[origin cost reduction]** Enable Origin Shield to reduce the number of simultaneous origin requests — lowers origin compute costs (for example just-in-time packaging or image transformation) and data-transfer-out charges. [doc](https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/origin-shield.html)
- **[regional edge caches]** Use CloudFront's regional edge caches, included at no additional cost, before adding Origin Shield — reduces origin operating costs at no extra charge. [doc](https://aws.amazon.com/cloudfront/features/)
- **[free origin fetches]** Prefer AWS origins such as Amazon S3, EC2, or Elastic Load Balancing — data transferred from these origins to CloudFront edge locations incurs no data transfer charge. [doc](https://aws.amazon.com/cloudfront/features/)
- **[price class selection]** Choose a lower CloudFront price class when viewers are concentrated in specific regions — limits the set of edge locations used for billing while still serving other viewers from a farther edge location. [doc](https://aws.amazon.com/cloudfront/getting-started/S3/)

## ⚙️ Operational Excellence
- **[monitoring]** Monitor the six default CloudWatch operational metrics per distribution and enable additional metrics where needed, alarming on signals like 4xx error rate or request volume — detects issues such as DDoS activity early. [doc](https://docs.aws.amazon.com/whitepapers/latest/secure-content-delivery-amazon-cloudfront/logging-and-monitoring-in-cloudfront.html)
- **[access logging]** Enable standard access logs (delivered to S3) or real-time logs (delivered to Kinesis Data Streams) — supports security audits, troubleshooting, and traffic analysis. [doc](https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/logging-and-monitoring.html)
- **[API auditing]** Enable AWS CloudTrail to log CloudFront API calls — captures who made each request, from which IP address, and when, for governance and incident investigation. [doc](https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/logging-and-monitoring.html)
- **[configuration tracking]** Use AWS Config to record configuration changes to distributions (state, price class, origins, geo restrictions, Lambda@Edge) and evaluate them with Config Rules — provides an auditable history and continuous compliance checks. [doc](https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/TrackingChanges.html)
- **[edge function logs]** Send CloudFront Functions and Lambda@Edge logs to Amazon CloudWatch Logs — CloudFront does not store edge function logs itself, so this is required to troubleshoot edge logic. [doc](https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/logging-and-monitoring.html)
