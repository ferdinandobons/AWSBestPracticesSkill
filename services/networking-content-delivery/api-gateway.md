# Amazon API Gateway — Best Practices

## Common scenarios
- Expose REST/HTTP APIs for serverless (Lambda) or containerized backends        → Security, Performance Efficiency, Cost Optimization
- Secure and throttle public-facing APIs for third-party or partner consumers        → Security, Reliability
- Provide private APIs for internal microservices within a VPC        → Security, Reliability
- Build real-time, two-way communication apps (chat, streaming dashboards) with WebSocket APIs        → Reliability, Performance Efficiency

## 🔒 Security
- **[access control]** Implement least-privilege IAM policies for creating, reading, updating, or deleting APIs, and choose an appropriate per-API authorization mechanism (IAM, Cognito user pools, Lambda authorizer, or resource policies) — limits blast radius of compromised credentials and matches authorization to consumer type. [doc](https://docs.aws.amazon.com/apigateway/latest/developerguide/security-best-practices.html)
- **[internal/private consumers]** Use AWS_IAM authorization with least-privileged IAM roles for callers that already operate inside your AWS environment — avoids managing separate credentials for trusted internal callers. [doc](https://docs.aws.amazon.com/wellarchitected/latest/serverless-applications-lens/identity-and-access-management.html)
- **[custom auth logic]** Use a Lambda authorizer when you need custom token validation or integration with an existing identity provider — keeps token-to-identity mapping logic isolated from backend services. [doc](https://aws.amazon.com/blogs/compute/building-well-architected-serverless-applications-controlling-serverless-api-access-part-1/)
- **[public/user-facing APIs]** Use Amazon Cognito user pools or OIDC/JWT authorizers for end-user authentication, leveraging OAuth scopes for granular authorization — avoids building custom user management and token validation. [doc](https://docs.aws.amazon.com/wellarchitected/latest/serverless-applications-lens/identity-and-access-management.html)
- **[private APIs]** Attach a resource policy restricting invocation to specific VPCs or VPC endpoints using `aws:SourceVpc`/`aws:SourceVpce` conditions rather than allowing all VPCs — private APIs require a resource policy and this prevents unintended cross-VPC access. [doc](https://docs.aws.amazon.com/apigateway/latest/developerguide/apigateway-private-api-create.html)
- **[public APIs]** Apply a resource policy to restrict invocation by source IP/CIDR or AWS account as an additional layer beyond authorizers — reduces exposure even if an authorizer is misconfigured. [doc](https://docs.aws.amazon.com/apigateway/latest/developerguide/apigateway-resource-policies.html)
- **[edge protection]** Enable AWS WAF on the API stage to inspect incoming requests for SQL injection, cross-site scripting, and to enforce IP allow/deny lists — combine with authorizers and resource policies for defense in depth. [doc](https://aws.amazon.com/blogs/architecture/things-to-consider-when-you-build-rest-apis-with-amazon-api-gateway/)
- **[transport security]** Enforce TLS 1.2 at minimum (TLS 1.3 recommended) and select hardened TLS security policies (e.g. disabling CBC ciphers or enforcing FIPS-aligned suites) on REST API endpoints and custom domains — protects data in transit and meets compliance requirements such as PCI DSS. [doc](https://docs.aws.amazon.com/apigateway/latest/developerguide/infrastructure-security.html)
- **[auditability]** Enable AWS CloudTrail to record management-plane actions and API invocations, including caller identity and source IP — essential for forensic investigation and compliance evidence. [doc](https://docs.aws.amazon.com/apigateway/latest/developerguide/security-best-practices.html)
- **[configuration compliance]** Enable AWS Config rules to continuously evaluate API Gateway resource configuration against your desired security baseline and alert on drift. [doc](https://docs.aws.amazon.com/apigateway/latest/developerguide/security-best-practices.html)
- **[posture management]** Use AWS Security Hub CSPM controls for API Gateway to continuously evaluate resources against security standards and compliance frameworks. [doc](https://docs.aws.amazon.com/apigateway/latest/developerguide/security-best-practices.html)
- **[sensitive data]** Never place confidential or sensitive information (such as customer emails) in tags, free-form name fields, or embedded credentials in backend URLs — this data can surface in billing or diagnostic logs. [doc](https://docs.aws.amazon.com/apigateway/latest/developerguide/data-protection.html)

## 🛡️ Reliability
- **[traffic protection]** Configure account-level, stage-level, and method-level throttling (steady-state rate and burst) so traffic spikes are shed at the gateway instead of overwhelming backend systems — size limits to what your downstream architecture (e.g., relational databases) can actually sustain. [doc](https://docs.aws.amazon.com/apigateway/latest/developerguide/api-gateway-request-throttling.html)
- **[per-client fairness]** Use usage plans and API keys to assign distinct throttling and quota targets per client or tenant — prevents one noisy client from starving others and lets you scale limits with validated backend capacity. [doc](https://aws.amazon.com/blogs/architecture/things-to-consider-when-you-build-rest-apis-with-amazon-api-gateway/)
- **[client behavior]** Design clients to handle `429 Too Many Requests` with retry, exponential backoff, and jitter rather than immediate resubmission — smooths bursts and avoids retry storms against a throttled API. [doc](https://aws.amazon.com/blogs/architecture/throttling-a-tiered-multi-tenant-rest-api-at-scale-using-api-gateway-part-1/)
- **[safe rollout]** Use canary release deployments to shift a percentage of production traffic to a new API version before full promotion, with separate canary-specific logs and metrics — lets you validate changes and roll back with minimal blast radius. [doc](https://docs.aws.amazon.com/apigateway/latest/developerguide/canary-release.html)
- **[disaster recovery]** For business-critical APIs, implement multi-Region failover using Route 53 health checks and DNS failover to a secondary Region, relying on the data plane rather than control-plane operations during recovery — avoids dependency on the primary Region's control plane during an outage. [doc](https://aws.amazon.com/blogs/compute/implementing-multi-region-failover-for-amazon-api-gateway/)

## ⚡ Performance Efficiency
- **[API type selection]** Choose HTTP APIs over REST APIs for proxy-style serverless workloads that don't need usage plans, API keys, or request/response transformation — HTTP APIs offer lower latency for the majority of use cases. [doc](https://aws.amazon.com/api-gateway/features/)
- **[response latency]** Enable API caching on REST API stages with an appropriate TTL and cache key so repeated requests are served from cache instead of invoking the backend — reduces both latency and backend load; monitor `CacheHitCount`/`CacheMissCount` to tune capacity. [doc](https://docs.aws.amazon.com/apigateway/latest/developerguide/monitoring-cloudwatch.html)
- **[payload size]** Enable payload compression (`minimumCompressionSize`) for REST APIs and test against real traffic patterns to pick an optimal threshold — compressing very small payloads can increase size, so validate before enabling broadly. [doc](https://docs.aws.amazon.com/apigateway/latest/developerguide/api-gateway-gzip-compression-decompression.html)
- **[capacity planning]** Monitor `Latency` and `IntegrationLatency` CloudWatch metrics to distinguish gateway overhead from backend responsiveness, and validate that downstream systems (e.g., relational databases) can sustain the request rate API Gateway and its integrations can scale to. [doc](https://docs.aws.amazon.com/apigateway/latest/developerguide/monitoring-cloudwatch.html)

## 💰 Cost Optimization
- **[API type selection]** Default to HTTP APIs instead of REST APIs when you don't need API management features like usage plans or API keys — HTTP APIs can reduce cost substantially compared to REST APIs for the same workload. [doc](https://aws.amazon.com/api-gateway/features/)
- **[data transfer]** Enable response payload compression for large JSON payloads to reduce data transfer volume and improve latency, after validating the size threshold against real traffic. [doc](https://aws.amazon.com/blogs/compute/optimizing-network-footprint-in-serverless-applications/)
- **[reduce backend invocations]** Enable REST API caching so repeat requests are served from the API Gateway cache rather than re-invoking backend compute and downstream data stores. [doc](https://docs.aws.amazon.com/apigateway/latest/developerguide/monitoring-cloudwatch.html)

## ⚙️ Operational Excellence
- **[observability]** Enable CloudWatch Logs (execution and/or access logs) for every stage and set an appropriate logging verbosity to support debugging of request execution and client access issues. [doc](https://docs.aws.amazon.com/apigateway/latest/developerguide/security-monitoring.html)
- **[alerting]** Create CloudWatch alarms on key metrics (4xx/5xx error rates, Latency, IntegrationLatency) so you are notified when a metric breaches a threshold rather than discovering incidents from user reports. [doc](https://docs.aws.amazon.com/apigateway/latest/developerguide/monitoring-cloudwatch.html)
- **[distributed tracing]** Enable AWS X-Ray on REST APIs to construct a service map of requests across integrated services, helping identify bottlenecks and failure points across a multi-service call chain. [doc](https://docs.aws.amazon.com/apigateway/latest/developerguide/security-monitoring.html)
- **[change validation]** Use canary release deployments with canary-specific CloudWatch Logs and metrics to validate new API versions against real traffic before promoting them to the full production stage. [doc](https://docs.aws.amazon.com/apigateway/latest/developerguide/canary-release.html)
