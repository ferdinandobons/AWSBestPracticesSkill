# AWS PrivateLink — Best Practices

## Common scenarios
- Private access from a VPC to AWS service APIs without traversing the internet        → Security, Reliability
- Exposing an internal or SaaS service to consumer VPCs across accounts        → Security, Operational Excellence
- Centralizing endpoint access for many spoke VPCs in a hub-and-spoke network        → Cost Optimization, Operational Excellence
- Cross-account/cross-VPC connectivity with overlapping CIDR ranges        → Reliability, Security

## 🔒 Security
- **[endpoint policies]** Attach a least-privilege VPC endpoint policy to every interface/gateway endpoint instead of leaving the default full-access policy — endpoint policies add a resource-level control layer independent of IAM identity policies and service-specific policies. [doc](https://docs.aws.amazon.com/whitepapers/latest/aws-vpc-connectivity-options/aws-privatelink.html)
- **[data perimeter]** Scope endpoint policies with `aws:PrincipalOrgID` and `aws:ResourceOrgID` conditions (specifying your org ID explicitly on both sides, not `${aws:PrincipalOrgID}`) to enforce that only trusted identities reach trusted resources through the endpoint. [doc](https://aws.amazon.com/blogs/security/how-to-use-interface-vpc-endpoints-to-meet-your-security-objectives/)
- **[endpoint service permissions]** Grant `Allow principals` only to the specific AWS accounts, roles, or users that should connect to your endpoint service — avoid the wildcard `*` principal, which makes the underlying load balancer effectively public even without a public IP. [doc](https://docs.aws.amazon.com/vpc/latest/privatelink/configure-endpoint-service.html)
- **[endpoint service acceptance]** Require manual acceptance (`AcceptanceRequired`) for endpoint connection requests when you need to vet consumers individually, rather than auto-accepting all permitted principals. [doc](https://docs.aws.amazon.com/vpc/latest/privatelink/configure-endpoint-service.html)
- **[network access]** Associate restrictive security groups with interface endpoint ENIs to control which VPC resources and ports can reach the endpoint, in addition to endpoint policies. [doc](https://docs.aws.amazon.com/whitepapers/latest/aws-vpc-connectivity-options/aws-privatelink.html)
- **[organization governance]** Use a Service Control Policy with `ec2:VpceServiceName` / `ec2:VpceServiceOwner` conditions to restrict which endpoint services (AWS, internal, or approved third-party) accounts in your organization are allowed to connect to. [doc](https://aws.amazon.com/blogs/security/governing-and-securing-aws-privatelink-service-access-at-scale-in-multi-account-environments/)
- **[data in transit]** Do not assume encryption in transit — PrivateLink itself provides no default encryption, so terminate TLS at the service/load balancer or encrypt at the application layer for sensitive data. [doc](https://aws.amazon.com/privatelink/faqs/)
- **[sensitive workloads]** Use AWS PrivateLink to keep traffic that carries personal or regulated data off the public internet entirely, avoiding internet gateways, NAT devices, or public IPs for in-scope services. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/privacy-reference-architecture/personal-data-account.html)

## 🛡️ Reliability
- **[high availability]** Deploy interface endpoints across a minimum of three Availability Zones (at minimum multiple AZs) so traffic automatically fails over to healthy endpoint network interfaces if an AZ becomes unavailable. [doc](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/privatelink-interface-endpoints.html)
- **[endpoint service HA]** Back your endpoint service with a Network Load Balancer spanning multiple Availability Zones so consumer connections in any AZ have a healthy target to reach. [doc](https://aws.amazon.com/blogs/awsforsap/secure-sap-hanacloud-connectivity-using-aws-privatelink/)
- **[DNS resolution]** Enable Private DNS (or configure Route 53 private hosted zones for centralized/shared-VPC architectures) so clients transparently resolve standard service hostnames to endpoint private IPs without application changes. [doc](https://docs.aws.amazon.com/vpc/latest/privatelink/privatelink-access-aws-services.html)
- **[event notifications]** Configure an SNS notification (with a topic policy permitting the endpoint service to publish) to alert on endpoint accept/reject/connect/delete events so you can react quickly to connectivity changes. [doc](https://docs.aws.amazon.com/vpc/latest/privatelink/create-notification-endpoint-service.html)
- **[NLB association]** Remember that for a given Availability Zone, an interface endpoint establishes a connection with only one Network Load Balancer instance, even if the endpoint service is associated with multiple NLBs — design AZ-level capacity accordingly. [doc](https://docs.aws.amazon.com/whitepapers/latest/aws-privatelink/deploying-aws-privatelink.html)

## ⚡ Performance Efficiency
- **[bandwidth planning]** Provision endpoint network interfaces across multiple subnets/AZs — each interface endpoint supports up to 10 Gbps per Availability Zone and can automatically scale to 100 Gbps, so size subnet/ENI distribution to expected throughput. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/designing-hyperscale-amazon-vpc-networks/)
- **[client IP visibility]** Enable Proxy Protocol v2 on the Network Load Balancer when the service needs the real consumer source IP and VPC endpoint ID, since PrivateLink normally presents the NLB's private IP as the source. [doc](https://docs.aws.amazon.com/whitepapers/latest/aws-privatelink/deploying-aws-privatelink.html)
- **[IPv6]** Enable dual-stack on the destination VPC, subnets, Network Load Balancer, and DNS names before enabling IPv6 on the endpoint service configuration, since PrivateLink requires all of these to support IPv6 consistently. [doc](https://docs.aws.amazon.com/whitepapers/latest/aws-vpc-connectivity-options/aws-privatelink.html)

## 💰 Cost Optimization
- **[endpoint type choice]** Use gateway endpoints (no hourly or data-processing charges) instead of interface endpoints for Amazon S3 and DynamoDB access from within a single VPC; use interface endpoints only when you need cross-VPC, cross-Region, on-premises, or Transit Gateway/VPN reachability. [doc](https://docs.aws.amazon.com/eks/latest/best-practices/cost-opt-networking.html)
- **[centralize endpoints]** Consolidate interface endpoints in a shared/hub VPC accessed by spoke VPCs (via Transit Gateway or VPC peering) rather than provisioning a duplicate interface endpoint per VPC, reducing per-AZ hourly and data-processing charges. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/designing-hyperscale-amazon-vpc-networks/)
- **[cleanup]** Delete VPC endpoints that are unused, rejected by the service owner, or whose backing service no longer exists — hourly billing continues until the endpoint is explicitly deleted. [doc](https://aws.amazon.com/blogs/apn/design-considerations-using-vpc-endpoints-for-amazon-s3-with-vmware-cloud-on-aws/)
- **[data transfer awareness]** Account for inter-AZ or inter-VPC data-processing charges when routing spoke-VPC traffic to centralized interface endpoints through Transit Gateway or peering, since these add cost on top of PrivateLink's own per-GB data-processing fee. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/designing-hyperscale-amazon-vpc-networks/)

## ⚙️ Operational Excellence
- **[usage monitoring]** Publish and monitor `AWS/PrivateLinkServices` CloudWatch metrics (`ActiveConnections`, `NewConnections`, `BytesProcessed`, `EndpointsCount`, `RstPacketsSent`) to detect unhealthy targets, capacity issues, or anomalous consumer behavior. [doc](https://docs.aws.amazon.com/vpc/latest/privatelink/privatelink-cloudwatch-metrics.html)
- **[anomaly detection]** Use CloudWatch Contributor Insights and anomaly-detection alarms on per-VPC-endpoint metrics to identify a misbehaving or compromised consumer endpoint and reject its connection to protect the service for all other consumers. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/gain-usage-insights-with-amazon-cloudwatch-metrics-for-aws-privatelink/)
- **[DNS operations]** When centralizing endpoints in a shared VPC, disable auto-created private DNS and instead manually manage Route 53 private hosted zones associated to all consumer VPCs, since the automatically managed private hosted zone only resolves within the VPC that owns the endpoint. [doc](https://docs.aws.amazon.com/whitepapers/latest/building-scalable-secure-multi-vpc-network-infrastructure/centralized-access-to-vpc-private-endpoints.html)
- **[change management]** Track that removing endpoint service permissions does not tear down already-accepted connections — plan explicit deletion of the associated VPC endpoint when decommissioning a consumer. [doc](https://docs.aws.amazon.com/vpc/latest/privatelink/configure-endpoint-service.html)
