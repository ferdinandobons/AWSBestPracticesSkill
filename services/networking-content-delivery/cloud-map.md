# AWS Cloud Map — Best Practices

## Common scenarios
- Service discovery for dynamically scaling microservices (ECS, EKS, Lambda)        → Reliability, Operational Excellence
- Health-aware routing so traffic only reaches healthy service instances        → Reliability
- Cross-account or cross-team shared service registries        → Security, Operational Excellence
- CI/CD deployments across environments, Regions, and versions without hardcoded endpoints        → Operational Excellence, Reliability

## 🔒 Security
- **[IAM]** Start from AWS managed policies and move toward least-privilege customer-managed policies that grant only the `servicediscovery` actions (for example `CreateService`, `RegisterInstance`, `DiscoverInstances`) a given role actually needs. [doc](https://docs.aws.amazon.com/cloud-map/latest/dg/security_iam_id-based-policy-examples.html)
- **[IAM]** Add IAM policy conditions (such as requiring SSL/TLS transport or restricting access to calls made through a specific AWS service like CloudFormation) to further scope down who can create, modify, or delete namespaces and services. [doc](https://docs.aws.amazon.com/cloud-map/latest/dg/security_iam_id-based-policy-examples.html)
- **[IAM]** Validate identity-based policies with IAM Access Analyzer before attaching them, to catch overly permissive grants on AWS Cloud Map resources. [doc](https://docs.aws.amazon.com/cloud-map/latest/dg/security_iam_id-based-policy-examples.html)
- **[access control]** Scope who can create namespaces or register/deregister instances, since these actions directly affect which endpoints your applications discover and can incur account costs. [doc](https://docs.aws.amazon.com/cloud-map/latest/dg/what-is-cloud-map.html)
- **[sharing]** When sharing a namespace across accounts with AWS RAM, use the narrowest managed or customer-managed permission needed by the consumer accounts rather than granting full namespace control. [doc](https://docs.aws.amazon.com/ram/latest/userguide/shareable.html)
- **[auditing]** Enable a multi-Region AWS CloudTrail trail to capture `servicediscovery` API calls (who registered/deregistered instances, created/deleted namespaces) so unauthorized or unexpected registry changes can be investigated. [doc](https://docs.aws.amazon.com/cloud-map/latest/dg/logging-using-cloudtrail.html)

## 🛡️ Reliability
- **[health checks]** Configure a health check (Route 53 or custom) on every service so AWS Cloud Map stops routing discovery responses to unhealthy instances instead of returning all registered instances regardless of status. [doc](https://docs.aws.amazon.com/cloud-map/latest/dg/services-health-checks.html)
- **[custom health checks]** For resources not reachable over the public internet (for example instances inside a VPC), use a custom health check with a third-party health checker in the same VPC, and submit `UpdateInstanceCustomHealthStatus` promptly on status changes. [doc](https://docs.aws.amazon.com/cloud-map/latest/dg/services-health-checks.html)
- **[custom health checks]** Design your health-status reporting to re-affirm `HEALTHY` regularly — if no update arrives within 30 seconds of an `UNHEALTHY` report, AWS Cloud Map treats the instance as confirmed unhealthy and stops routing to it. [doc](https://docs.aws.amazon.com/cloud-map/latest/dg/services-health-checks.html)
- **[Route 53 health checks]** Monitor Route 53 health check status with CloudWatch alarms or the Route 53 console for namespaces discovered via API calls, since there is no DNS record to visually associate the health check with. [doc](https://docs.aws.amazon.com/cloud-map/latest/dg/services-health-checks.html)
- **[discovery pattern]** Query AWS Cloud Map by service name and/or endpoint rather than caching or hardcoding the IP address of the backing compute (Lambda, container, or EC2 instance), so failover and rescheduling of instances doesn't break discovery. [doc](https://aws.amazon.com/blogs/architecture/new-application-integration-with-aws-cloud-map-for-service-discovery/)
- **[DNS caching]** Account for client- and JVM-level DNS caching when using DNS-based discovery — if your compute stack caches resolved IPs for the record TTL, prefer API-based discovery (`DiscoverInstances`) so instance changes are reflected without waiting on DNS cache expiry. [doc](https://aws.amazon.com/blogs/architecture/new-application-integration-with-aws-cloud-map-for-service-discovery/)
- **[scale]** Plan around the `Instances per service` (1,000) and `Instances per namespace` (2,000) quotas when designing services with many dynamically scaling tasks, since these quotas cap how large a single service or namespace can grow. [doc](https://docs.aws.amazon.com/cloud-map/latest/dg/cloud-map-limits.html)
- **[ECS at scale]** For Amazon ECS services using Cloud Map–based service discovery with a large number of tasks, be aware that `RegisterInstance`/`DeregisterInstance`/`ListInstances`/`GetInstancesHealthStatus` calls can be throttled during simultaneous large deployments, and plan deployment concurrency accordingly. [doc](https://docs.aws.amazon.com/AmazonECS/latest/developerguide/operating-at-scale-service-quotas-best-practice.html)

## ⚡ Performance Efficiency
- **[discovery type]** Choose DNS-based discovery for simple name resolution into existing infrastructure, and API-based discovery (`DiscoverInstances`) when you need lower-latency updates, richer metadata (custom attributes), or discovery of URL/ARN-based resources that DNS can't represent. [doc](https://aws.amazon.com/cloud-map/faqs/)
- **[TTL]** Set an appropriate DNS TTL for each service's DNS records — lower TTLs propagate instance changes to clients faster but increase query volume, while higher TTLs reduce query load at the cost of slower convergence after registration changes. [doc](https://docs.aws.amazon.com/cloud-map/latest/api/API_DnsRecord.html)
- **[routing policy]** Pick the routing policy to match traffic distribution needs: `WEIGHTED` for proportional traffic shifting (for example blue/green or canary), `MULTIVALUE` for simple multi-endpoint answers with health filtering. [doc](https://aws.amazon.com/blogs/containers/load-balancing-amazon-ecs-services-with-a-kubernetes-ingress-controller-style-approach/)
- **[filtering]** Use custom attributes on registered instances (such as deployment stage or version) so clients can filter `DiscoverInstances` results to exactly the matching resources instead of resolving all instances and filtering client-side. [doc](https://aws.amazon.com/blogs/aws/aws-cloud-map-easily-create-and-maintain-custom-maps-of-your-applications/)
- **[throttling]** Design clients to handle `RequestLimitExceeded` from `DiscoverInstances` with backoff/retry, and cache results where appropriate, since the API enforces per-account, per-Region steady and burst rate quotas. [doc](https://docs.aws.amazon.com/cloud-map/latest/dg/throttling.html)

## 💰 Cost Optimization
- **[quotas]** Increasing DNS-namespace instance counts requires raising the Route 53 records-per-hosted-zone quota, which carries its own Route 53 charges — factor this into capacity planning before requesting quota increases. [doc](https://docs.aws.amazon.com/cloud-map/latest/dg/cloud-map-limits.html)
- **[health checks]** Prefer custom health checks over per-instance Route 53 health checks where you already operate your own health-checking infrastructure (for example inside a VPC), avoiding duplicate health-check charges. [doc](https://docs.aws.amazon.com/cloud-map/latest/dg/services-health-checks.html)
- **[shared namespaces]** Use AWS RAM to share a single namespace across accounts/teams rather than provisioning duplicate namespaces (and duplicate Route 53 hosted zones) per account. [doc](https://docs.aws.amazon.com/cloud-map/latest/dg/sharing-namespaces.html)

## ⚙️ Operational Excellence
- **[tagging]** Tag namespaces and services at creation time (rather than with a follow-up script) so cost allocation, ownership, and governance metadata is never missing for a resource. [doc](https://docs.aws.amazon.com/cloud-map/latest/dg/using-tags.html)
- **[multi-account]** Use AWS RAM to share namespaces owned by a central account with other accounts or an entire AWS Organization, so teams get consistent cross-account service discovery without duplicating namespace management. [doc](https://docs.aws.amazon.com/cloud-map/latest/dg/sharing-namespaces.html)
- **[monitoring]** Define a monitoring plan for AWS Cloud Map covering what to monitor, how often, which tools, and who is notified, rather than relying on ad hoc checks. [doc](https://docs.aws.amazon.com/cloud-map/latest/dg/monitoring-overview.html)
- **[auditing]** Use CloudTrail Event history (or a configured trail for retention beyond 90 days) to review who created namespaces, registered instances, or changed services, supporting change tracking and incident investigation. [doc](https://docs.aws.amazon.com/cloud-map/latest/dg/logging-using-cloudtrail.html)
- **[migration]** When migrating ECS services from Cloud Map–based service discovery to ECS Service Connect, verify old and new namespaces side by side in the console before decommissioning the original service-discovery namespace. [doc](https://aws.amazon.com/blogs/containers/migrate-existing-amazon-ecs-services-from-service-discovery-to-amazon-ecs-service-connect/)
- **[quotas]** Periodically check current usage against account-level quotas (namespaces per Region, instances per namespace/service) using Service Quotas, and request increases proactively before scaling events hit the limit. [doc](https://docs.aws.amazon.com/cloud-map/latest/dg/service-quotas-manage.html)
