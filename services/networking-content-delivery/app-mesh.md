# AWS App Mesh — Best Practices

## Common scenarios
- Zero-downtime rolling deployments across ECS/Kubernetes microservices        → Reliability
- Service-to-service encryption and identity verification with mTLS        → Security
- End-to-end observability (metrics, logs, traces) across service hops        → Operational Excellence
- Controlled traffic shifting and retries during scale-out/scale-in events        → Reliability, Performance Efficiency

## 🔒 Security
- **[IAM]** Start from AWS managed policies for App Mesh and move toward least-privilege customer-managed policies that grant only the specific `appmesh` actions a role needs, rather than broad `appmesh:*` access. [doc](https://docs.aws.amazon.com/app-mesh/latest/userguide/security_iam_id-based-policy-examples.html)
- **[IAM]** Add IAM policy conditions — such as requiring SSL/TLS transport or scoping calls to a specific AWS service like CloudFormation — to further restrict who can create, modify, or delete meshes, virtual nodes, and routes. [doc](https://docs.aws.amazon.com/app-mesh/latest/userguide/security_iam_id-based-policy-examples.html)
- **[IAM]** Validate identity-based policies with IAM Access Analyzer before attaching them to catch overly permissive grants on App Mesh resources. [doc](https://docs.aws.amazon.com/app-mesh/latest/userguide/security_iam_id-based-policy-examples.html)
- **[mTLS]** Enforce mutual TLS (mTLS) between Envoy proxies for service-to-service authentication and encryption, using `STRICT` TLS mode as the steady-state target rather than leaving connections in plaintext. [doc](https://aws.amazon.com/blogs/containers/using-mtls-with-spiffe-spire-in-app-mesh-on-eks/)
- **[mTLS rollout]** Run listeners in `PERMISSIVE` mode only temporarily during migration to mTLS, so services can be verified and onboarded to the mesh without breaking existing plaintext clients, then switch to `STRICT` once all mTLS-required services are confirmed onboarded. [doc](https://aws.amazon.com/blogs/containers/three-things-to-consider-when-implementing-mutual-tls-with-aws-app-mesh/)
- **[certificate authority]** Define an organization-wide certificate authority (self-managed or AWS Certificate Manager Private CA) before starting an mTLS deployment, and use shorter validity periods for internal service-to-service certificates than for external end-user-facing certificates to limit exposure from a leaked certificate. [doc](https://aws.amazon.com/blogs/containers/three-things-to-consider-when-implementing-mutual-tls-with-aws-app-mesh/)
- **[certificate lifecycle]** Plan certificate delivery to the Envoy proxy, rotation, and revocation before enabling mTLS in production — manual certificate lifecycle handling significantly increases operational overhead if not automated. [doc](https://aws.amazon.com/blogs/containers/three-things-to-consider-when-implementing-mutual-tls-with-aws-app-mesh/)
- **[TLS listeners]** Configure TLS directly on virtual node and virtual gateway listeners (via ACM certificate, customer-provided certificate files, or an SDS endpoint) so Envoy — not the application — negotiates and terminates encrypted connections. [doc](https://docs.aws.amazon.com/cdk/api/v2/docs/aws-cdk-lib.aws_appmesh-readme.html)
- **[client policy]** Set a backend client policy with TLS validation (trust chain) on virtual nodes so outbound connections to a virtual service enforce certificate validation rather than trusting any upstream by default. [doc](https://aws.amazon.com/blogs/containers/enable-traffic-encryption-between-services-in-aws-app-mesh-using-aws-certificate-manager-or-customer-provided-certificates/)

## 🛡️ Reliability
- **[retries]** Configure all virtual routers with a default retry policy on every route (minimum `maxRetries` of 2), specifying the relevant retry events per protocol — TCP: `connection-error`; HTTP/HTTP2: `stream-error` and `gateway-error`; gRPC: `cancelled` and `unavailable` — to increase the odds a retried request reaches a healthy destination. [doc](https://docs.aws.amazon.com/app-mesh/latest/userguide/best-practices.html)
- **[retries]** Tune `maxRetries` and `perRetryTimeout` deliberately, weighing maximum request latency (`maxRetries` × `perRetryTimeout`) against the improved success rate of additional retries, and only enable retries for non-idempotent operations after careful case-by-case review. [doc](https://docs.aws.amazon.com/app-mesh/latest/userguide/best-practices.html)
- **[deployment velocity]** Reduce rolling-deployment velocity instead of using ECS defaults (100% minimum healthy / 200% maximum) — target 100% minimum healthy with 125% maximum (150% only for very small desired counts of 2-3 tasks) to reduce Envoy dataplane divergence from the orchestrator's view. [doc](https://docs.aws.amazon.com/app-mesh/latest/userguide/best-practices.html)
- **[deployment velocity]** On Kubernetes, set the deployment update strategy to `maxUnavailable: 0%` and `maxSurge: 25%` to control how much the visible set of destinations churns during a rollout. [doc](https://docs.aws.amazon.com/app-mesh/latest/userguide/best-practices.html)
- **[scaling]** Scale out new tasks/pods before scaling in old ones, for both Amazon ECS and Kubernetes, to minimize the percentage of scaled-in destinations visible to Envoy at any one time and reduce failed-retry probability. [doc](https://docs.aws.amazon.com/app-mesh/latest/userguide/best-practices.html)
- **[health checks]** On Amazon ECS, configure container health checks and container dependency ordering so the Envoy proxy container is confirmed running and healthy before any application container that needs outbound connectivity starts. [doc](https://docs.aws.amazon.com/app-mesh/latest/userguide/best-practices.html)
- **[DNS resolution]** Set the IP protocol preference for DNS-based service discovery (`IPv4_PREFERRED`, `IPv6_PREFERRED`, `IPv4_ONLY`, `IPv6_ONLY`) to match what your infrastructure actually supports, rather than relying on the default `IPv6_PREFERRED`, which can degrade performance if your environment isn't truly IPv6-capable; set this at the mesh level and override per virtual node as needed. [doc](https://docs.aws.amazon.com/app-mesh/latest/userguide/best-practices.html)

## ⚡ Performance Efficiency
- **[DNS resolution]** Choose the DNS IP-version preference (`IPv4_PREFERRED`, `IPv6_PREFERRED`, `IPv4_ONLY`, `IPv6_ONLY`) deliberately at the mesh or virtual node level instead of accepting the `IPv6_PREFERRED` default, since an unnecessary IPv6 lookup-then-fallback sequence adds latency in IPv4-only environments. [doc](https://docs.aws.amazon.com/app-mesh/latest/userguide/best-practices.html)
- **[retry tuning]** Account for the fact that a retried connection to a no-longer-present endpoint consumes the full `perRetryTimeout`, and size `perRetryTimeout` accordingly so retries don't inflate end-to-end request latency more than necessary. [doc](https://docs.aws.amazon.com/app-mesh/latest/userguide/best-practices.html)

## ⚙️ Operational Excellence
- **[observability]** Configure App Mesh/Envoy to export metrics, access logs, and traces (for example to Amazon CloudWatch and AWS X-Ray, or compatible tools like Prometheus, Datadog, or Jaeger) so you get consistent per-hop visibility across every microservice without instrumenting application code directly. [doc](https://aws.amazon.com/blogs/containers/getting-started-with-app-mesh-and-eks/)
- **[logging]** Use FireLens as a container log router to collect Envoy and application logs from Amazon ECS/Fargate tasks into CloudWatch alongside App Mesh metrics. [doc](https://docs.aws.amazon.com/app-mesh/latest/userguide/envoy-logs.html)
- **[migration planning]** Plan a migration off AWS App Mesh before September 30, 2026, when AWS discontinues support for the service and removes console/API access to App Mesh resources; for ECS workloads, evaluate Amazon ECS Service Connect as the recommended replacement. [doc](https://docs.aws.amazon.com/app-mesh/latest/userguide/best-practices.html)
- **[Kubernetes readiness]** Be aware that the App Mesh controller for Kubernetes does not factor liveness/readiness probe results into AWS Cloud Map instance registration/deregistration, and design health-check-driven traffic shifting accordingly rather than assuming probe status alone controls mesh routing. [doc](https://docs.aws.amazon.com/app-mesh/latest/userguide/best-practices.html)
- **[auditing]** Track who created, modified, or deleted meshes, virtual nodes, and routes by reviewing `appmesh` API activity, since these actions directly affect production traffic routing and can incur costs. [doc](https://docs.aws.amazon.com/app-mesh/latest/userguide/security_iam_id-based-policy-examples.html)
