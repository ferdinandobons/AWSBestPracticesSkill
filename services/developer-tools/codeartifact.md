# AWS CodeArtifact — Best Practices

## Common scenarios
- Centralized private package repository for internal libraries        → Security, Operational Excellence
- Proxying and caching public packages (npm, PyPI, Maven, NuGet)        → Reliability, Cost Optimization
- Protecting build pipelines from software supply-chain attacks        → Security
- Sharing approved packages across teams and accounts        → Security, Operational Excellence

## 🔒 Security
- **[IAM]** Grant least-privilege permissions and start from AWS managed policies before narrowing to custom policies scoped to specific domains/repositories — reduces blast radius of compromised credentials. [doc](https://docs.aws.amazon.com/codeartifact/latest/ug/security_iam_id-based-policy-examples.html)
- **[IAM]** Add IAM policy conditions (e.g., require SSL, restrict to specific services) and validate policies with IAM Access Analyzer — catches overly permissive or non-functional statements before they reach production. [doc](https://docs.aws.amazon.com/codeartifact/latest/ug/security_iam_id-based-policy-examples.html)
- **[Authentication]** Limit authorization token lifetime using the `sts:DurationSeconds` condition key instead of relying on the 12-hour default — shortens the window a leaked token remains usable. [doc](https://docs.aws.amazon.com/codeartifact/latest/ug/security_iam_id-based-policy-examples.html)
- **[Supply chain]** Configure package origin controls (Publish/Upstream settings) so a package version can never be both directly published and ingested from a public repository — closes the primary vector for dependency substitution (dependency confusion) attacks. [doc](https://docs.aws.amazon.com/codeartifact/latest/ug/dependency-substitution-attacks.html)
- **[Supply chain]** Use package group origin controls to apply Publish/Upstream restrictions in bulk across a namespace of internal packages rather than one at a time — ensures new internal packages are protected by default and scales governance. [doc](https://docs.aws.amazon.com/codeartifact/latest/ug/package-group-origin-controls.html)
- **[Repository policies]** Apply a resource-based repository policy that explicitly grants only the intended principals access, since by default only the repository owner has access — avoids accidental cross-account exposure. [doc](https://docs.aws.amazon.com/codeartifact/latest/ug/repo-policies.html)
- **[Domain policy]** Use domain permission policies to control which accounts can create repositories or configure external connections at the domain level — centralizes governance instead of relying on per-repository policies alone. [doc](https://docs.aws.amazon.com/codeartifact/latest/ug/domain-overview.html)
- **[Encryption]** Use a customer-managed AWS KMS key for domains that require custom key policies, rotation control, or auditing beyond the AWS managed key default — gives explicit control over who can decrypt package assets. [doc](https://docs.aws.amazon.com/codeartifact/latest/ug/security-encryption.html)
- **[Encryption]** Use only symmetric KMS keys for CodeArtifact domains and account for added latency/availability dependency if using an external key store (XKS) — an asymmetric key is unsupported and XKS key loss causes loss of all domain assets. [doc](https://docs.aws.amazon.com/codeartifact/latest/ug/domain-overview.html)
- **[Network]** Access CodeArtifact from within a VPC using interface VPC endpoints (AWS PrivateLink) for both the API and repositories endpoints, plus an S3 gateway endpoint for asset retrieval — keeps package traffic off the public internet. [doc](https://docs.aws.amazon.com/codeartifact/latest/ug/vpc-endpoints.html)
- **[Network]** Scope VPC endpoint policies to the specific principals and actions needed (e.g., `GetAuthorizationToken`, `ReadFromRepository`) rather than allowing all actions — limits what a compromised instance in the VPC can do against CodeArtifact. [doc](https://docs.aws.amazon.com/codeartifact/latest/ug/create-vpc-endoint-policy.html)
- **[Monitoring]** Enable a CloudTrail trail for CodeArtifact API activity, including calls made by package manager clients, and deliver events to S3 — provides an audit trail of who accessed or published which packages and when. [doc](https://docs.aws.amazon.com/codeartifact/latest/ug/codeartifact-information-in-cloudtrail.html)

## 🛡️ Reliability
- **[Upstreams]** Configure a one-to-one mapping between a repository and each external public repository (e.g., npmjs, Maven Central), and have other repositories use it as an upstream rather than each connecting directly — increases cache hits and preserves availability if a package is later removed upstream. [doc](https://docs.aws.amazon.com/codeartifact/latest/ug/external-connection.html)
- **[Upstreams]** Rely on package retention in intermediate repositories: once a package version is pulled from an upstream, a reference is retained even if the upstream repository, connection, or package version is later deleted or changed — protects builds from upstream outages or unpublish events. [doc](https://docs.aws.amazon.com/codeartifact/latest/ug/repo-upstream-behavior.html)
- **[IaC]** Set `DeletionPolicy: Retain` and `UpdateReplacePolicy: Retain` on CodeArtifact domain and repository resources in CloudFormation — prevents accidental loss of hard-to-recreate dependencies if a stack is updated or deleted. [doc](https://docs.aws.amazon.com/codeartifact/latest/ug/cloudformation-codeartifact.html)
- **[Quotas]** Review CodeArtifact service quotas (e.g., requests per second per API, upstreams per repository, domains per account) and request increases proactively for high-throughput CI/CD pipelines — avoids throttling during build spikes. [doc](https://docs.aws.amazon.com/codeartifact/latest/ug/service-limits.html)
- **[Domain design]** Use a single production domain per organization (with an optional separate sandbox domain for experimentation) rather than many ad hoc domains — since deduplication and copying only work within a domain, fragmenting domains reduces cache efficiency and availability of shared packages. [doc](https://docs.aws.amazon.com/codeartifact/latest/ug/domain-overview.html)

## ⚡ Performance Efficiency
- **[Caching]** Maintain a single external connection per public repository per domain instead of multiple repositories each connecting independently to the same public source — increases cache hit rate and reduces the time to resolve dependencies. [doc](https://aws.amazon.com/blogs/devops/integrating-aws-codeartifact-package-mgmt-flow/)
- **[Networking]** Use VPC endpoints for build environments (e.g., CI runners, SageMaker notebooks) that fetch packages frequently — avoids the latency and reliability variance of routing through an internet gateway or NAT device. [doc](https://docs.aws.amazon.com/codeartifact/latest/ug/vpc-endpoints.html)

## 💰 Cost Optimization
- **[Domain design]** Consolidate repositories under a single organizational domain since assets are deduplicated at the domain level — storing the same package across multiple domains (e.g., unnecessary sandbox domains) duplicates storage costs. [doc](https://aws.amazon.com/blogs/devops/integrating-aws-codeartifact-package-mgmt-flow/)
- **[Requests]** Minimize redundant external connections to the same public repository across repositories in a domain — each fetched asset is billed as a request, so a one-to-one mapping with caching reduces repeated external package resolution charges. [doc](https://aws.amazon.com/blogs/devops/integrating-aws-codeartifact-package-mgmt-flow/)
- **[Cost allocation]** Tag domains and repositories to allocate storage costs (tied to domains) and request costs (tied to repositories/packages) — enables chargeback and cost visibility per team or product line. [doc](https://docs.aws.amazon.com/codeartifact/latest/ug/tag-resources.html)

## ⚙️ Operational Excellence
- **[Automation]** Subscribe to CodeArtifact events via Amazon EventBridge (delivered through CloudTrail) to trigger downstream automation such as starting a CodePipeline execution, invoking a Lambda function, or notifying teams when a package changes — enables automated dependency and approval workflows. [doc](https://docs.aws.amazon.com/codeartifact/latest/ug/working-with-service-events.html)
- **[Monitoring]** Use Amazon CloudWatch metrics (Usage/By AWS Resource namespace) to track CodeArtifact request volume by operation and account — supports capacity planning and anomaly detection ahead of quota limits. [doc](https://docs.aws.amazon.com/codeartifact/latest/ug/working-with-service-events.html)
- **[Governance]** Use the package origin control toolkit (or the `PutPackageOriginConfiguration` API) to retroactively apply origin restrictions in bulk to packages created before the feature existed, with dry-run and revert support — brings legacy repositories up to current security posture without manual per-package edits. [doc](https://aws.amazon.com/blogs/devops/tighten-your-package-security-with-codeartifact-package-origin-control-toolkit/)
- **[IaC]** Provision domains, repositories, upstream relationships, and external connections declaratively with AWS CloudFormation (`AWS::CodeArtifact::Domain`, `AWS::CodeArtifact::Repository`) — ensures repeatable, auditable environment setup across accounts. [doc](https://docs.aws.amazon.com/AWSCloudFormation/latest/TemplateReference/aws-resource-codeartifact-repository.html)
