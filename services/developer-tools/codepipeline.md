# AWS CodePipeline — Best Practices

## Common scenarios
- Automating build, test, and deploy release workflows        → Operational Excellence, Reliability
- Continuous delivery of container, serverless, or EC2-based applications        → Reliability, Performance Efficiency
- Multi-account / multi-region release pipelines        → Security, Reliability
- Gating releases with manual approvals and automated rollback        → Reliability, Operational Excellence

## 🔒 Security
- **[secrets]** Store tokens, passwords, and API keys in AWS Secrets Manager and reference them in action/pipeline configuration instead of entering them directly — plaintext secrets in action configuration or variable defaults are exposed in logs. [doc](https://docs.aws.amazon.com/codepipeline/latest/userguide/security-best-practices.html)
- **[artifact encryption]** Configure server-side encryption (SSE-KMS) for the S3 artifact bucket, using a customer managed symmetric KMS key when you need to control or share access (for example cross-account pipelines) — the default AWS managed key can't be shared or rotated by you. [doc](https://docs.aws.amazon.com/codepipeline/latest/userguide/S3-artifact-encryption.html)
- **[artifact bucket policy]** Add an S3 bucket policy that denies `PutObject` without the `x-amz-server-side-encryption` header and denies non-TLS requests, to enforce encryption and secure transport for pipeline artifacts. [doc](https://docs.aws.amazon.com/codepipeline/latest/userguide/S3-artifact-encryption.html)
- **[least privilege]** Grant IAM users and the CodePipeline service role only the specific `codepipeline:*` actions and resource ARNs they need (for example scoping approval permissions to a single pipeline/stage/action) rather than the broad `AWSCodePipeline_FullAccess` managed policy. [doc](https://docs.aws.amazon.com/codepipeline/latest/userguide/security_iam_id-based-policy-examples.html)
- **[iam:PassRole]** Restrict who can pass a service role to CodePipeline — `AWSCodePipeline_FullAccess` grants `iam:PassRole`, so only apply it to trusted administrators. [doc](https://docs.aws.amazon.com/codepipeline/latest/userguide/getting-started-codepipeline.html)
- **[Jenkins integration]** When using a Jenkins build/test provider, run Jenkins on its own EC2 instance with a dedicated instance profile scoped only to the permissions Jenkins needs (for example S3 read access), rather than reusing a broad role. [doc](https://docs.aws.amazon.com/codepipeline/latest/userguide/security-best-practices.html)
- **[Commands action secrets]** For the Commands action, reference secrets via the `SECRETS_MANAGER` environment variable type instead of `PLAINTEXT`, and grant the pipeline service role only `secretsmanager:GetSecretValue` on the specific secret ARN — plaintext values are otherwise displayed in the console/CLI. [doc](https://docs.aws.amazon.com/codepipeline/latest/userguide/action-reference-Commands.html)
- **[auditing]** Enable AWS CloudTrail to capture all CodePipeline API calls (who made a request, from what IP, and when) for security analysis and operational troubleshooting. [doc](https://docs.aws.amazon.com/codepipeline/latest/userguide/monitoring-cloudtrail-logs.html)

## 🛡️ Reliability
- **[pipeline structure]** Group related test, deploy, and approval actions together within the same stage (rather than splitting them across stages) so a test action is guaranteed to validate the same change that was deployed. [doc](https://docs.aws.amazon.com/codepipeline/latest/userguide/concepts-how-it-works.html)
- **[stage granularity]** Balance stage granularity — too many fine-grained stages allows too many concurrent in-flight changes, while overly coarse stages with many actions can slow down releases; size stages around environments (e.g., per application environment, Region, or Availability Zone). [doc](https://docs.aws.amazon.com/codepipeline/latest/userguide/concepts-how-it-works.html)
- **[rollback]** Configure automatic rollback on stage failure (V2 pipelines) so a failed deployment is automatically reverted to the last successful execution's artifacts and variables, reducing manual recovery time. [doc](https://docs.aws.amazon.com/codepipeline/latest/userguide/stage-rollback.html)
- **[execution mode]** Choose the execution mode deliberately — use QUEUED when every triggered change must run in order without being skipped, PARALLEL when independent branch pipelines must not block each other, and the default SUPERSEDED only when only the latest change through a stage matters. [doc](https://docs.aws.amazon.com/help-panel/codepipeline/latest/helppanel/hp-execution-modes.html)
- **[stopping executions]** Disable a stage's inbound transition before stopping an in-progress execution, so the stage doesn't immediately accept and lock in a subsequent execution once it's unlocked. [doc](https://docs.aws.amazon.com/codepipeline/latest/userguide/concepts-how-it-works.html)
- **[stage conditions]** Use stage-level conditions (gates) to check rules before entering or after exiting a stage and automatically roll back when a condition fails, catching problems before they propagate further through the pipeline. [doc](https://aws.amazon.com/blogs/devops/enhance-release-control-with-aws-codepipeline-stage-level-conditions/)
- **[cross-region artifacts]** For cross-region actions, provision a replication bucket (with its own encryption key) in each target Region so artifacts can be reliably replicated to and from the Region where each action runs. [doc](https://docs.aws.amazon.com/cdk/api/v2/docs/aws-cdk-lib.aws_codepipeline-readme.html)

## ⚡ Performance Efficiency
- **[parallel actions]** Model independent build, test, and deployment actions to run in parallel within a stage (using `runOrder`) instead of serially, to reduce overall release time. [doc](https://aws.amazon.com/codepipeline/features/)
- **[execution mode]** Use PARALLEL execution mode for multi-branch/monorepo pipelines (e.g., GitFlow feature branches) so unrelated branch executions don't wait on each other, and QUEUED mode when ordered processing of every change matters more than throughput. [doc](https://aws.amazon.com/blogs/devops/aws-codepipeline-adds-support-for-branch-based-development-and-monorepos/)
- **[event-based triggers]** Use event-based change detection (CloudWatch Events/EventBridge) instead of polling for source changes — pipelines inactive for more than 30 days have polling disabled, and event-based triggers react faster with less overhead. [doc](https://docs.aws.amazon.com/codepipeline/latest/userguide/pipeline-requirements.html)

## 💰 Cost Optimization
- **[stage granularity]** Avoid overly granular pipeline stages/actions that create unnecessary duplicate build or deploy steps — group related actions per environment to minimize redundant action executions. [doc](https://docs.aws.amazon.com/codepipeline/latest/userguide/concepts-how-it-works.html)
- **[execution supersession]** Rely on the default SUPERSEDED execution mode for high-frequency-commit pipelines where only the latest change needs to reach production, so older in-flight executions are stopped early instead of consuming pipeline and downstream build resources to completion. [doc](https://docs.aws.amazon.com/codepipeline/latest/userguide/concepts.html)

## ⚙️ Operational Excellence
- **[monitoring]** Monitor pipeline, stage, and action state changes with Amazon EventBridge (or CloudWatch Events) and route them to targets such as Lambda or SNS to trigger notifications or corrective action automatically. [doc](https://docs.aws.amazon.com/codepipeline/latest/userguide/detect-state-changes-cloudwatch-events.html)
- **[notifications]** Configure CodeStar/pipeline notifications (via Amazon SNS) for key events such as pipeline execution started, succeeded, failed, and manual approval needed, so teams are alerted without polling the console. [doc](https://aws.amazon.com/codepipeline/faqs/)
- **[metrics and alarms]** Track the `PipelineDuration` and `FailedPipelineExecutions` CloudWatch metrics at the pipeline and account level, and set alarms on them to detect degraded or failing releases early. [doc](https://docs.aws.amazon.com/codepipeline/latest/userguide/metrics-dimensions.html)
- **[variables]** Use pipeline-level and action output variables (V2 pipelines) to pass values such as commit IDs between actions instead of hardcoding them, keeping pipeline definitions reusable and reducing manual configuration drift. [doc](https://docs.aws.amazon.com/codepipeline/latest/userguide/reference-variables.html)
- **[investigating failures]** Use the pipeline execution history and the `FailedPipelineExecutionId` trigger reference to trace a rollback or failure back to the specific execution and revision that caused it. [doc](https://aws.amazon.com/blogs/devops/de-risk-releases-with-aws-codepipeline-rollbacks/)
- **[auditing]** Enable AWS CloudTrail for CodePipeline so operational and security teams can reconstruct the timeline of pipeline changes and API activity during incident response. [doc](https://docs.aws.amazon.com/codepipeline/latest/userguide/incident-response.html)
