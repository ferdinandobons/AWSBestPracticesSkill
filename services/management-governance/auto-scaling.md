# AWS Auto Scaling — Best Practices

## Common scenarios
- Automatically matching EC2 fleet capacity to variable or unpredictable traffic        → Reliability, Cost Optimization, Performance Efficiency
- Maintaining application availability across Availability Zone failures        → Reliability
- Blending On-Demand, Spot, and Reserved/Savings Plans capacity in one group        → Cost Optimization
- Rolling out AMI/launch template updates without downtime        → Operational Excellence, Reliability

## 🔒 Security
- **[IAM]** Grant only the minimum Amazon EC2 Auto Scaling API actions needed (e.g., `Describe*` for read-only roles), and scope create/update/delete permissions using resource-level and tag-based conditions — limits blast radius if credentials are compromised. [doc](https://docs.aws.amazon.com/autoscaling/ec2/userguide/security-iam.html)
- **[IAM]** Restrict who can modify launch templates and which AMI they reference, since the launch template determines the security posture (OS, patches, agents) of every instance the group creates — treat launch template changes as a controlled, audited change-management action. [doc](https://aws.amazon.com/blogs/publicsector/documenting-the-use-of-amazon-ec2-auto-scaling-groups-in-dod/)
- **[IAM]** Create a dedicated IAM role with least-privilege permissions and attach it via an instance profile in the launch template for applications running on Auto Scaling instances — avoids embedding long-lived credentials on instances. [doc](https://docs.aws.amazon.com/autoscaling/ec2/userguide/us-iam-role.html)
- **[Instance metadata]** Require IMDSv2 on the launch template (`MetadataHttpTokens=required`), and optionally enforce this account-wide with an IAM policy condition that denies `ec2:RunInstances` unless IMDSv2 is required — mitigates SSRF-style credential theft from the instance metadata service. [doc](https://docs.aws.amazon.com/autoscaling/ec2/userguide/ec2-auto-scaling-launch-template-permissions.html)
- **[Data protection]** Encrypt EBS volumes attached to Auto Scaling instances using AWS KMS customer managed keys (configured via the launch template), or enable account-level EBS encryption by default — protects data at rest without relying on AMI-level configuration alone. [doc](https://docs.aws.amazon.com/autoscaling/ec2/userguide/ec2-auto-scaling-data-protection.html)
- **[Baseline hygiene]** Keep AMIs referenced by launch templates patched and under configuration management (via IaC), rather than patching individual running instances — Auto Scaling replaces instances continuously, so drift-free security posture depends on the base image, not on ad hoc instance changes. [doc](https://aws.amazon.com/blogs/publicsector/documenting-the-use-of-amazon-ec2-auto-scaling-groups-in-dod/)

## 🛡️ Reliability
- **[High availability]** Span every Auto Scaling group across multiple Availability Zones and maintain at least one instance in each enabled zone — protects the workload from a single-AZ failure. [doc](https://docs.aws.amazon.com/autoscaling/ec2/userguide/disaster-recovery-resiliency.html)
- **[Load balancing]** Attach a load balancer that distributes traffic across the same set of Availability Zones, and keep cross-zone load balancing enabled on Application Load Balancers — ensures each instance receives a similar share of traffic and limits the impact of a failover. [doc](https://docs.aws.amazon.com/autoscaling/ec2/userguide/disaster-recovery-resiliency.html)
- **[Health checks]** Enable ELB health checks on the Auto Scaling group (not just EC2 status checks) so unhealthy instances are detected and replaced based on actual application responsiveness, not just instance-level state. [doc](https://docs.aws.amazon.com/autoscaling/ec2/userguide/disaster-recovery-resiliency.html)
- **[Health checks]** Set the health check grace period high enough for the application to finish initializing and pass its first ELB health check, but not needlessly high — set it too high and truly unhealthy instances stay in service too long; use lifecycle hooks (grace period can be 0) if you need guaranteed pre-`InService` initialization instead. [doc](https://docs.aws.amazon.com/autoscaling/ec2/userguide/health-check-grace-period.html)
- **[Scaling policy]** Use dynamic scaling (target tracking, preferred) instead of manual/static capacity, and validate the scaling metric actually increases or decreases proportionally with instance count — an unsuitable metric (e.g., CPU for a queue-depth-bound workload) causes both under- and over-scaling. [doc](https://docs.aws.amazon.com/wellarchitected/latest/framework/perf_compute_hardware_scale_compute_resources_dynamically.html)
- **[Testing]** Test scale-in behavior explicitly (not just scale-out), including effects on sticky sessions and in-flight requests, and review Auto Scaling activity history to confirm scaling activities behave as expected. [doc](https://docs.aws.amazon.com/wellarchitected/latest/framework/perf_compute_hardware_scale_compute_resources_dynamically.html)
- **[AZ rebalancing]** Let Amazon EC2 Auto Scaling automatically rebalance instances across Availability Zones after an AZ recovers from an outage — avoid manual interventions (detaching instances, standby) that can leave the group permanently unbalanced across zones. [doc](https://docs.aws.amazon.com/autoscaling/ec2/userguide/as-instance-termination.html)

## ⚡ Performance Efficiency
- **[Scaling metrics]** Choose a scaling metric that reflects actual workload utilization for the specific application (e.g., queue depth for a transcoding pipeline rather than CPU) — a mismatched metric under- or over-provisions capacity relative to real demand. [doc](https://docs.aws.amazon.com/wellarchitected/latest/performance-efficiency-pillar/perf_compute_hardware_scale_compute_resources_dynamically.html)
- **[Scaling policy]** Prefer target tracking scaling policies over simple/step scaling for continuous, self-adjusting capacity that follows the demand curve closely, reserving step scaling for cases needing more advanced multi-threshold behavior. [doc](https://aws.amazon.com/blogs/mt/create-amazon-ec2-auto-scaling-policy-memory-utilization-metric-windows/)
- **[Predictive scaling]** Use predictive scaling for workloads with recurring, predictable demand patterns to provision capacity ahead of anticipated load instead of reacting only after utilization crosses a threshold — reduces both latency spikes and unnecessary overprovisioning. [doc](https://docs.aws.amazon.com/wellarchitected/latest/framework/perf_compute_hardware_scale_compute_resources_dynamically.html)
- **[Instance warm-up]** Enable default instance warmup so newly launched instances finish initializing before their metrics are folded into the group's aggregated CloudWatch metrics — prevents scaling policies from reacting to skewed data from not-yet-warmed instances. [doc](https://docs.aws.amazon.com/cdk/api/v2/python/aws_cdk.aws_autoscaling/README.html)
- **[Warm pools]** For applications with long boot times, use a warm pool to pre-initialize instances (optionally in a stopped state) so they can enter service faster during scale-out events than launching cold instances on demand. [doc](https://docs.aws.amazon.com/autoscaling/ec2/userguide/ec2-auto-scaling-warm-pools.html)

## 💰 Cost Optimization
- **[Right-sizing]** Start with a small baseline capacity and let dynamic scaling policies add instances as needed rather than over-provisioning a fixed fleet up front — avoids paying for capacity that isn't consistently used. [doc](https://aws.amazon.com/blogs/startups/how-to-optimize-costs-for-aws-services/)
- **[Purchase options]** Use a mixed instances policy to combine Reserved Instances/Savings Plans for baseline steady-state capacity with On-Demand and Spot Instances for the variable portion of a group's capacity — matches purchase model to how predictable each unit of capacity actually is. [doc](https://aws.amazon.com/blogs/startups/how-to-optimize-costs-for-aws-services/)
- **[Spot allocation]** When mixing Spot capacity into a group, explicitly review and set the Spot allocation strategy, number of Spot pools, and multiple compatible instance types/sizes rather than relying purely on defaults — diversification reduces the chance of simultaneous Spot interruptions across the fleet. [doc](https://docs.aws.amazon.com/autoscaling/ec2/APIReference/API_InstancesDistribution.html)
- **[Warm pools]** Keep warm-pool instances in a stopped state when idle rather than running, to reduce the cost of pre-initialized standby capacity while still cutting scale-out latency. [doc](https://aws.amazon.com/blogs/compute/scaling-your-applications-faster-with-ec2-auto-scaling-warm-pools/)
- **[Scale-in]** Configure aggressive-enough scale-in policies (paired with reliable health/utilization metrics) to release unneeded capacity promptly during low-traffic periods, instead of only scaling out — Auto Scaling only saves cost if the group actually shrinks when demand drops. [doc](https://docs.aws.amazon.com/wellarchitected/latest/sustainability-pillar/sus_sus_user_a2.html)

## ⚙️ Operational Excellence
- **[IaC]** Manage launch templates, AMI versions, and Auto Scaling group configuration as infrastructure as code rather than through ad hoc console edits — keeps the base image and scaling configuration consistent, versioned, and auditable. [doc](https://aws.amazon.com/blogs/publicsector/documenting-the-use-of-amazon-ec2-auto-scaling-groups-in-dod/)
- **[Rolling updates]** Use instance refresh to roll out new launch template versions (updated AMI, IMDSv2 settings, instance types) across a group with controlled batching instead of manually terminating instances — an instance refresh also updates instances waiting in a warm pool. [doc](https://docs.aws.amazon.com/autoscaling/ec2/userguide/instance-refresh-overview.html)
- **[Lifecycle hooks]** Use lifecycle hooks to run and confirm application initialization (or graceful deregistration/log flush on termination) before an instance transitions to `InService` or is fully terminated — makes health check grace period behavior more predictable and prevents traffic from hitting a not-yet-ready instance. [doc](https://docs.aws.amazon.com/autoscaling/ec2/userguide/health-check-grace-period.html)
- **[Instance maintenance]** Set the health check grace period high enough for ELB health checks to reliably determine target readiness, but not so high that it delays detection of genuinely unhealthy instances during maintenance operations like instance refresh. [doc](https://docs.aws.amazon.com/autoscaling/ec2/userguide/instance-maintenance-policy-overview-and-considerations.html)

## 🌱 Sustainability
- **[Elasticity]** Use dynamic (not manual) scaling with target tracking so capacity matches real-time demand and scales down quickly during periods of low load — reduces the number of idle instances running unnecessarily. [doc](https://docs.aws.amazon.com/wellarchitected/latest/sustainability-pillar/sus_sus_user_a2.html)
- **[Metric selection]** Validate that scaling metrics genuinely reflect utilization for the workload type before relying on them to drive scale-in decisions — an inappropriate metric can keep excess instances running (or scale in too aggressively), wasting or starving resources respectively. [doc](https://docs.aws.amazon.com/wellarchitected/latest/sustainability-pillar/sus_sus_user_a2.html)
- **[Predictive scaling]** Use predictive scaling for workloads with predictable demand patterns to avoid the resource waste of static overprovisioning used as a safety margin. [doc](https://docs.aws.amazon.com/wellarchitected/latest/sustainability-pillar/sus_sus_user_a2.html)
