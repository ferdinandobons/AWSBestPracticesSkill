# Amazon Managed Service for Prometheus — Best Practices

## Common scenarios
- Centralized, multi-cluster metrics monitoring for Amazon EKS/ECS/Kubernetes workloads        → Reliability, Performance Efficiency
- High-cardinality container and microservices observability without managing Prometheus servers        → Performance Efficiency, Cost Optimization
- Cross-account or cross-Region metrics aggregation for a centralized observability platform        → Reliability, Security
- Alerting and automated incident detection on infrastructure and application metrics        → Operational Excellence, Cost Optimization

## 🔒 Security
- **[Ingestion]** Avoid exposing Prometheus scrape endpoints to the public internet and secure any self-managed Prometheus agent or ADOT collector configuration — you are responsible for the security of customer-managed collectors under the shared responsibility model. [doc](https://docs.aws.amazon.com/prometheus/latest/userguide/AMP-ingest-with-prometheus.html)
- **[Authentication]** Use IAM and AWS Signature Version 4 (SigV4) signing to authenticate and authorize every request to ingest or query metrics — all Amazon Managed Service for Prometheus API calls, including remote_write, are authorized by IAM. [doc](https://docs.aws.amazon.com/prometheus/latest/userguide/AMP-secure-metric-ingestion.html)
- **[Least privilege]** Grant IAM roles for service accounts (IRSA) or collector IAM roles only the specific `aps:RemoteWrite` and query permissions needed, scoped to the workspace ARN — avoid broad or workspace-unscoped policies. [doc](https://docs.aws.amazon.com/prometheus/latest/userguide/set-up-irsa.html)
- **[Least privilege]** Start from AWS managed policies and move toward customer managed, least-privilege policies for humans and workloads accessing workspaces — use IAM Access Analyzer to validate policies. [doc](https://docs.aws.amazon.com/prometheus/latest/userguide/security_iam_id-based-policy-examples.html)
- **[Networking]** Deploy collectors and scrapers in private subnets and use interface VPC endpoints (AWS PrivateLink) for both the `aps` and `aps-workspaces` endpoints — this keeps ingestion and query traffic off the public internet. [doc](https://docs.aws.amazon.com/prometheus/latest/userguide/AMP-and-interface-VPC.html)
- **[Networking]** Attach a VPC endpoint policy to restrict which AWS principals and source IPs can use the endpoint to reach Amazon Managed Service for Prometheus — this adds a network-layer control beyond IAM identity policies. [doc](https://aws.amazon.com/blogs/mt/amazon-managed-service-for-prometheus-now-offers-vpc-endpoint-policy-support/)
- **[Encryption in transit]** Require TLS 1.2 (recommend TLS 1.3) with cipher suites that support perfect forward secrecy (DHE/ECDHE) for all clients communicating with the service. [doc](https://docs.aws.amazon.com/prometheus/latest/userguide/infrastructure-security.html)
- **[Sensitive data]** Never put confidential or sensitive information such as customer emails into tags, labels, or free-form text fields — this data can surface in billing or diagnostic logs. [doc](https://docs.aws.amazon.com/prometheus/latest/userguide/data-protection.html)
- **[Governance]** Set up cross-account ingestion and governance controls (e.g. per-tenant scraper roles) when centralizing metrics from multiple AWS accounts into a shared workspace, rather than granting broad account-wide access. [doc](https://aws.amazon.com/blogs/mt/optimizing-metrics-ingestion-with-amazon-managed-service-for-prometheus/)
- **[Auditing]** Enable AWS CloudTrail for Amazon Managed Service for Prometheus and deliver logs to a trail (Event history alone only retains 90 days) — this gives an ongoing, searchable record of workspace management API calls for security analysis. [doc](https://docs.aws.amazon.com/prometheus/latest/userguide/logging-using-cloudtrail.html)

## 🛡️ Reliability
- **[High availability]** Configure `cluster` and `__replica__` external labels when running redundant collector replicas so Amazon Managed Service for Prometheus automatically de-duplicates data — without this you risk gaps if one replica fails and pay for duplicate samples if you don't. [doc](https://docs.aws.amazon.com/prometheus/latest/userguide/AMP-ingest-dedupe.html)
- **[High availability]** Run multiple Prometheus agents or ADOT collectors sending the same metrics with a shared cluster name so a single collector failure doesn't create a monitoring gap. [doc](https://docs.aws.amazon.com/prometheus/latest/userguide/AMP-ingest-high-availability.html)
- **[Disaster recovery]** Send metrics to a second workspace in another AWS Region for workloads that need access to metrics even during a Region-level disruption. [doc](https://docs.aws.amazon.com/prometheus/latest/userguide/AMP-send-to-multiple-workspaces.html)
- **[Managed collectors]** Prefer the AWS-managed scraper over self-managed Prometheus agents where possible — it removes the operational burden of scaling, patching, and monitoring collector availability yourself. [doc](https://aws.amazon.com/blogs/mt/simplifying-prometheus-metrics-collection-across-your-aws-infrastructure/)
- **[Native reliability]** Rely on Amazon Managed Service for Prometheus's built-in replication across Availability Zones for the stored data itself, and add cross-Region or deduplication safeguards only for your own ingestion-path resiliency needs. [doc](https://docs.aws.amazon.com/prometheus/latest/userguide/disaster-recovery-resiliency.html)

## ⚡ Performance Efficiency
- **[Query design]** Include as many label selectors as possible in PromQL queries to minimize the number of time series scanned, and choose a step interval aligned with the query time range. [doc](https://aws.amazon.com/blogs/opensource/best-practices-for-migrating-self-hosted-prometheus-on-amazon-eks-to-amazon-managed-service-for-prometheus/)
- **[Recording rules]** Precompute frequently used or computationally expensive PromQL expressions as recording rules — querying the precomputed series is much faster than re-running the original expression on every query. [doc](https://docs.aws.amazon.com/prometheus/latest/userguide/AMP-Ruler.html)
- **[Query governance]** Configure query throttling thresholds (warning and error) on Query Samples Processed (QSP) to catch expensive queries before they scan billions of samples. [doc](https://docs.aws.amazon.com/prometheus/latest/userguide/query-insights-control.html)
- **[Diagnostics]** Enable query logging to CloudWatch Logs to identify and analyze queries that exceed QSP thresholds, so you can target optimization at the queries that matter most. [doc](https://docs.aws.amazon.com/prometheus/latest/userguide/query-insights-control.html)
- **[Scale planning]** Monitor active-series and ingestion-rate usage against service quotas (e.g. via `prometheus_tsdb_head_series` and CloudWatch usage metrics) and request quota increases proactively before hitting throttling limits. [doc](https://docs.aws.amazon.com/prometheus/latest/userguide/AMP_quotas.html)

## 💰 Cost Optimization
- **[Ingestion]** Increase the scrape/collection interval (e.g. from 30s to 60s) where the extra resolution isn't needed — ingestion volume, not storage, drives the majority of cost for most customers. [doc](https://docs.aws.amazon.com/prometheus/latest/userguide/AMP-costs.html)
- **[Ingestion]** Use `relabel_config` to filter and drop unneeded metrics before they are sent to the workspace, reducing the number of active series ingested. [doc](https://docs.aws.amazon.com/prometheus/latest/userguide/AMP-costs.html)
- **[Deduplication]** Always configure cluster/replica labels for high-availability collector setups — without deduplication you are billed for every duplicate sample ingested and stored. [doc](https://docs.aws.amazon.com/prometheus/latest/userguide/AMP-ingest-dedupe.html)
- **[Alerting]** Use Amazon Managed Service for Prometheus's native alerting rather than external alerting systems that re-query metrics for their own high availability — external HA alerting can multiply query costs several-fold. [doc](https://docs.aws.amazon.com/prometheus/latest/userguide/AMP-costs.html)
- **[Alerting]** Right-size alert evaluation intervals and PromQL lookback windows (e.g. reduce a 10-minute `rate()` window to 5 minutes) to cut the data scanned per alert evaluation. [doc](https://docs.aws.amazon.com/prometheus/latest/userguide/AMP-costs.html)
- **[Query governance]** Set QSP warning/error thresholds so runaway or unintentionally expensive queries are flagged or rejected before they drive up query costs. [doc](https://docs.aws.amazon.com/prometheus/latest/userguide/query-insights-control.html)

## ⚙️ Operational Excellence
- **[Alerting rules]** Build alerting rules on top of recording rules and well-structured metrics rather than raw complex expressions — this keeps alert definitions readable and maintainable as they grow. [doc](https://aws.amazon.com/blogs/mt/alerting-best-practices-with-amazon-managed-service-for-prometheus/)
- **[Rule organization]** Organize recording and alerting rules into logically named rule groups and namespaces so existing Prometheus rules files can be imported and managed without restructuring. [doc](https://docs.aws.amazon.com/prometheus/latest/userguide/AMP-Ruler.html)
- **[Quota monitoring]** Set CloudWatch alarms on Prometheus usage metrics (active series, ingestion rate) at a threshold like 80% of quota so you can request an increase before requests start getting throttled. [doc](https://aws.amazon.com/blogs/opensource/best-practices-for-migrating-self-hosted-prometheus-on-amazon-eks-to-amazon-managed-service-for-prometheus/)
- **[Auditing]** Use AWS CloudTrail to track workspace creation, deletion, and configuration changes for operational troubleshooting and change tracking across teams. [doc](https://docs.aws.amazon.com/prometheus/latest/userguide/logging-using-cloudtrail.html)
- **[Governance]** Establish per-tenant scraper roles and query attribution when running a shared, centralized observability workspace across multiple teams or accounts, so usage and cost can be attributed and unexpected spikes traced back to their source. [doc](https://aws.amazon.com/blogs/mt/optimizing-metrics-ingestion-with-amazon-managed-service-for-prometheus/)
