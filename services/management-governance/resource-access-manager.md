# AWS Resource Access Manager — Best Practices

## Common scenarios
- Sharing VPC subnets, transit gateways, or other network resources across accounts        → Reliability, Operational Excellence, Cost Optimization
- Centrally governing which accounts/OUs can access shared resources        → Security, Operational Excellence
- Granting least-privilege, fine-grained access to shared resources        → Security
- Auditing and monitoring cross-account resource sharing activity        → Security, Operational Excellence

## 🔒 Security
- **[Organizations integration]** Enable AWS RAM integration with AWS Organizations and share resources with OUs or the whole organization rather than individual accounts — this keeps access automatically in sync as accounts move in and out of the org/OU, so resources are never shared with an account you didn't intend. [doc](https://docs.aws.amazon.com/wellarchitected/2023-10-03/framework/sec_permissions_lifecycle.html)
- **[condition keys]** Use global condition keys such as `aws:PrincipalOrgID`, `aws:PrincipalOrgPaths`, `aws:ResourceOrgId`, and `aws:ResourceOrgPaths` in IAM policies and SCPs to constrain resource access to your organization or OU structure — this is especially valuable in large organizations with complex OU hierarchies that need strict organizational access boundaries. [doc](https://aws.amazon.com/blogs/mt/protect-your-aws-resources-from-unauthorized-access-using-aws-organizations-integrations/)
- **[least privilege]** Prefer customer managed permissions over the default AWS managed permission when the default doesn't provide least-privilege access for your sharing use case — author precise actions and conditions (for example, letting developers assign IPAM addresses without viewing other accounts' ranges) instead of granting broader default access. [doc](https://docs.aws.amazon.com/ram/latest/userguide/create-customer-managed-permissions.html)
- **[permission conditions]** Add conditions to managed permissions to require sensitive actions to originate from an approved IP range or from principals authenticated with MFA — conditions add an extra layer of control on top of the allowed actions for a shared resource. [doc](https://docs.aws.amazon.com/ram/latest/userguide/security-ram-permissions.html)
- **[external sharing]** Restrict or disable sharing with principals outside your AWS organization using a service control policy that denies `ram:CreateResourceShare`/`ram:UpdateResourceShare` when `ram:RequestedAllowsExternalPrincipals` is true — this prevents resource shares from being extended to accounts outside your trust boundary. [doc](https://docs.aws.amazon.com/ram/latest/userguide/security-scp.html)
- **[IAM access]** Follow least privilege when granting IAM principals permission to manage AWS RAM — use the provided AWS managed policies where they fit, and author custom IAM policies with the RAM-specific condition keys (`ram:Principal`, `ram:PermissionArn`, `ram:RequestedResourceType`, etc.) when you need finer control. [doc](https://docs.aws.amazon.com/ram/latest/userguide/security-iam-policies.html)
- **[data protection]** Protect account credentials with MFA, require TLS 1.2+ (prefer TLS 1.3) for API communication, and never place confidential or sensitive information in RAM tags or free-form name fields — tag and text-field content can surface in billing or diagnostic logs. [doc](https://docs.aws.amazon.com/ram/latest/userguide/data-protection.html)
- **[network sharing]** When sharing data-plane connections such as VPC subnets or transit gateway attachments with other accounts, establish a process to verify connections are made only to preapproved Regions and comply with your data residency requirements. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/privacy-reference-architecture/network-account.html)
- **[VPC sharing]** Use a service control policy that denies `ram:*` for accounts or OUs that should never participate in resource sharing (for example, accounts that must not receive shared VPC subnets) — this enforces the sharing boundary at the organization level rather than relying on manual review. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/vpc-sharing-key-considerations-and-best-practices/)

## 🛡️ Reliability
- **[lifecycle management]** Share resources through AWS RAM instead of manually re-creating them or maintaining per-account access lists — access is automatically granted or revoked as accounts join or leave the organization/OU, keeping shared access consistent with your intended account topology at all times. [doc](https://docs.aws.amazon.com/wellarchitected/2023-10-03/framework/sec_permissions_lifecycle.html)
- **[centralized resources]** Create foundational infrastructure (VPC subnets, transit gateways, IPAM pools, private CAs) once in an owning account and share it via AWS RAM rather than provisioning duplicates in every account — this avoids configuration drift between duplicated resources across accounts. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/rehost-multi-account-architecture-interface-endpoints/solution-3.html)
- **[large-scale sharing]** When associating a large number of resources (50+) to a resource share, add the consuming account/principal to the share first, then add resources in batches of 10 or fewer, verifying each batch reaches `ASSOCIATED` status before continuing — adding resources before the principal exists can trigger RAM API throttling and result in resources with empty or corrupted resource policies. [doc](https://aws.amazon.com/blogs/messaging-and-targeting/global-sms-strategy-for-gaming-lessons-from-migrating-2fa-across-100-countries/)
- **[verification]** After a batch share operation, confirm shared resources are visible from the consuming account using the relevant `describe-*`/`list-*` CLI commands with the shared-resource owner filter, and keep CloudTrail logs available to troubleshoot any associations that don't complete as expected. [doc](https://aws.amazon.com/blogs/messaging-and-targeting/global-sms-strategy-for-gaming-lessons-from-migrating-2fa-across-100-countries/)

## ⚙️ Operational Excellence
- **[single management point]** Manage resource sharing centrally through AWS RAM instead of resource-specific mechanisms where possible — RAM provides one consistent console, API, and permission model across the different resource types you share, standardizing the sharing experience and reducing operational overhead. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/security-reference-architecture/network.html)
- **[visibility]** Use AWS RAM's visibility into which specific IAM principals have access to each shared resource (rather than relying solely on resource-based policies) so resource owners and IAM principals can both directly retrieve accurate shared-resource access information. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/privacy-reference-architecture/personal-data-account.html)
- **[monitoring]** Enable AWS CloudTrail logging for AWS RAM API calls and configure Amazon EventBridge rules to react to resource-share change events — this gives you an audit trail of who created, modified, or accepted resource shares and lets you automate responses to specific sharing events. [doc](https://docs.aws.amazon.com/ram/latest/userguide/security-monitoring.html)
- **[event-driven ops]** Build EventBridge rules that match on `source: aws.ram` and `eventSource: ram.amazonaws.com` to detect and act on specific AWS RAM API activity, such as new resource shares or invitation acceptance. [doc](https://docs.aws.amazon.com/eventbridge/latest/ref/events-ref-ram.html)
- **[migrating existing shares]** If a resource is currently shared using a resource-based policy (for example an IAM or S3 bucket policy) and you want to manage it through AWS RAM instead, use the `PromoteResourceShareCreatedFromPolicy` API to bring it under RAM management rather than recreating the share. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/privacy-reference-architecture/personal-data-account.html)
- **[delegated ownership]** Invoke and manage AWS RAM from the account that owns the shared resource (for example, a dedicated network account for VPC subnets and transit gateways, or a data lake account for Lake Formation resources) to keep separation of duties between resource owners and consumers. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/security-reference-architecture/network.html)

## 💰 Cost Optimization
- **[avoid duplication]** Create a resource once in an owning account and share it with other accounts through AWS RAM instead of provisioning duplicate resources per account — this eliminates redundant infrastructure and its ongoing management cost across every consuming account. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/rehost-multi-account-architecture-interface-endpoints/solution-3.html)
- **[centralized permission management]** Manage access to shared resources through a single set of managed permissions rather than duplicating and synchronizing identical IAM/resource policies across every account that consumes the resource — this reduces the ongoing operational effort (and associated cost) of keeping per-account policies aligned. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/rehost-multi-account-architecture-interface-endpoints/solution-3.html)
- **[centralized governance]** Centrally govern shareable resources such as private certificate authorities so certificate issuance is managed from one place across multiple accounts — this reduces the cost and operational burden of running and maintaining duplicate infrastructure per account. [doc](https://aws.amazon.com/ram/)
