# AWS Resilience Hub — Best Practices

## Common scenarios
- Defining RTO/RPO-based resiliency policies and assessing applications against them        → Reliability, Operational Excellence
- Improving recovery posture before production release via CI/CD-gated assessments        → Reliability, Operational Excellence
- Validating recovery behavior with fault-injection experiments (AWS FIS)        → Reliability
- Detecting and reacting to resilience drift as application architecture changes        → Reliability, Operational Excellence

## 🔒 Security
- **[Permissions]** Use the AWS-managed policies for AWS Resilience Hub as the basis for the Invoker role rather than hand-rolled permissions — this keeps access aligned as new Resilience Hub capabilities are released. [doc](https://aws.amazon.com/blogs/mt/using-permissions-to-unlock-resilience-with-aws-resilience-hub/)
- **[Cross-account access]** When application resources span accounts, set up the dedicated `AwsResilienceHubExecutorAccountRole` in each resource account rather than reusing broad admin credentials — this scopes what Resilience Hub can read in each account. [doc](https://docs.aws.amazon.com/resilience-hub/latest/userguide/security-iam-resilience-hub-permissions.html)
- **[Scheduled assessments]** Only grant the `AwsResilienceHubPeriodicAssessmentRole` to applications that actually use scheduled or drift-detection assessments — avoid attaching it where daily automated access isn't needed. [doc](https://docs.aws.amazon.com/resilience-hub/latest/userguide/security-iam-resilience-hub-permissions.html)
- **[VPC endpoints]** If workload resources sit behind VPC endpoints, ensure the endpoint policies grant Resilience Hub read-only access to those resources — otherwise assessments silently miss resources it cannot reach. [doc](https://docs.aws.amazon.com/resilience-hub/latest/userguide/security-iam-resilience-hub-permissions.html)
- **[Least privilege]** Write custom identity-based policies scoped to specific Resilience Hub actions and resource ARNs (e.g. restrict who can start assessments or delete applications) instead of granting account-wide Resilience Hub access. [doc](https://docs.aws.amazon.com/resilience-hub/latest/userguide/security_iam_id-based-policy-examples.html)
- **[Auditability]** Enable a CloudTrail trail that captures Resilience Hub API calls (console and programmatic) — this gives you who ran or modified assessments, resiliency policies, and applications, and when. [doc](https://docs.aws.amazon.com/resilience-hub/latest/userguide/integrate-cloudtrail.html)

## 🛡️ Reliability
- **[Policy definition]** Define resiliency policies with explicit RTO/RPO targets for application, infrastructure, Availability Zone, and (for multi-Region workloads) Region disruption types — assessing against a single blended target misses failure modes specific to each layer. [doc](https://docs.aws.amazon.com/resilience-hub/latest/userguide/setup-resiliency-policy.html)
- **[Policy targets]** Avoid setting RTO/RPO to zero in a resiliency policy — this forces the compliance status to "Policy breached" by definition since no real workload achieves a true zero recovery time or data loss. [doc](https://docs.aws.amazon.com/resilience-hub/latest/userguide/setup-resiliency-policy.html)
- **[Assessment cadence]** Re-run a resiliency assessment whenever the application's components or configuration change — recommendations, alarms, and SOP guidance are generated against the current architecture and go stale as the app evolves. [doc](https://docs.aws.amazon.com/resilience-hub/latest/userguide/resil-assessments.html)
- **[Alarms]** Deploy the CloudWatch alarms recommended by an assessment, and re-run the assessment after infrastructure changes so alarm recommendations stay matched to the current resource set. [doc](https://docs.aws.amazon.com/resilience-hub/latest/userguide/alarms.html)
- **[Existing alarms]** Let Resilience Hub detect and incorporate already-configured CloudWatch alarms into its assessment, or explicitly mark a recommended alarm as "Already implemented" — this keeps the resilience score accurate instead of double-counting or ignoring existing monitoring. [doc](https://docs.aws.amazon.com/resilience-hub/latest/userguide/alarms.html)
- **[Fault injection testing]** Run the AWS FIS experiments and scenarios that Resilience Hub recommends (or your own) to actually validate recovery — an assessment score alone doesn't prove an application recovers within its RTO/RPO. [doc](https://docs.aws.amazon.com/resilience-hub/latest/userguide/arh-testing.html)
- **[Guardrails]** Configure AWS FIS stop conditions and rollback guardrails before running fault-injection experiments in production — this bounds blast radius if an experiment reveals a real weakness. [doc](https://docs.aws.amazon.com/resilience-hub/latest/userguide/testing.html)
- **[Drift detection]** Enable resilience drift detection with daily scheduled assessments and Amazon SNS notifications — this proactively surfaces when an application no longer meets its resiliency policy instead of discovering it during an actual outage. [doc](https://aws.amazon.com/blogs/mt/identifying-resilience-drift-using-aws-resilience-hub/)
- **[Drift detection setup]** Ensure the application's IAM role (or the `AwsResilienceHubPeriodicAssessmentRole`) and the target SNS topic's resource policy are both correctly configured — drift notifications silently fail to deliver without these. [doc](https://docs.aws.amazon.com/help-panel/resilience-hub/latest/console/application-drifts-compliance-gated.html)
- **[Iterative improvement]** Treat recommendations as an iterative loop: implement the suggested component, alarm, SOP, or test changes, re-assess, and compare against the previous report until the estimated workload RTO/RPO meet policy targets. [doc](https://aws.amazon.com/blogs/apn/best-practices-from-eviden-for-leveraging-aws-resilience-hub-to-protect-applications/)

## ⚙️ Operational Excellence
- **[CI/CD integration]** Integrate Resilience Hub assessments into your CI/CD pipeline so every build is validated before release, and gate promotion to later environments when resilience issues are found — this catches resilience regressions before they reach production. [doc](https://aws.amazon.com/blogs/architecture/continually-assessing-application-resilience-with-aws-resilience-hub-and-aws-codepipeline/)
- **[Application definition]** Describe applications using a resource-collection method that matches how you already manage infrastructure (CloudFormation stacks, Terraform state, resource groups, or AppRegistry) — this keeps the Resilience Hub application definition in sync with your actual deployed resources. [doc](https://aws.amazon.com/blogs/apn/best-practices-from-eviden-for-leveraging-aws-resilience-hub-to-protect-applications/)
- **[SOPs]** Deploy the recommended standard operating procedures (SOPs) for common recovery actions (e.g. failover, restore from backup) alongside your application — these give operators a tested, repeatable recovery script instead of ad hoc manual steps during an incident. [doc](https://docs.aws.amazon.com/resilience-hub/latest/userguide/resil-assessments.html)
- **[Continuous tracking]** After production deployment, keep tracking resiliency posture over time via Resilience Hub rather than treating the initial assessment as a one-time checkbox — architectures and dependencies change continuously. [doc](https://aws.amazon.com/blogs/apn/best-practices-from-eviden-for-leveraging-aws-resilience-hub-to-protect-applications/)
- **[Governance reporting]** Consolidate resilience findings, recommendations, and compliance status across accounts into a single Resilience Hub view for stakeholder and audit reporting instead of manually gathering data from each team. [doc](https://aws.amazon.com/resilience-hub/)
- **[Monitoring]** Use the CloudWatch metrics, CloudTrail events, and EventBridge integration for Resilience Hub to track assessment activity and automate responses to findings. [doc](https://docs.aws.amazon.com/resilience-hub/latest/userguide/next-gen-monitoring.html)
