# AWS Fault Injection Service — Best Practices

## Common scenarios
- Running chaos engineering experiments to validate application resilience        → Reliability
- Testing recovery procedures for EC2, ECS, EKS, RDS, and other AWS resource failures        → Reliability, Operational Excellence
- Validating steady-state behavior and stop-condition guardrails before production rollout        → Reliability, Security
- Scaling fault injection experimentation across accounts and Regions in an organization        → Security, Operational Excellence

## 🔒 Security
- **[IAM]** Grant the AWS FIS experiment role only the permissions required for the specific actions and target resource types used in the experiment template, following least privilege. — Overly broad experiment roles let FIS (or anyone who can start an experiment) impact resources beyond the intended blast radius. [doc](https://docs.aws.amazon.com/fis/latest/userguide/security-iam.html)
- **[IAM]** Use resource tags (for example `Purpose=Test`) in identity-based policies to restrict which experiment templates a principal can start or modify. — Tag-based conditions prevent unauthorized use of templates intended for other environments. [doc](https://docs.aws.amazon.com/fis/latest/userguide/security_iam_id-based-policy-examples.html)
- **[Organization]** Standardize separate IAM roles for template authoring (for example an orchestrator role) versus experiment execution (an executor role assumed only by the FIS service), and enforce them with SCPs at the OU level. — Separating "who can define experiments" from "what the experiment is allowed to touch" limits misuse of the powerful execution role. [doc](https://aws.amazon.com/blogs/mt/scaling-aws-fault-injection-service-across-your-organization-using-account-controls/)
- **[IAM]** When experiments invoke Systems Manager documents, explicitly scope the SSM automation role with PassRole and least-privilege statements rather than granting broad `ssm:*` or `iam:*` access. — FIS-to-SSM handoffs multiply the effective permissions of an experiment if the automation role is over-permissioned. [doc](https://aws.amazon.com/blogs/devops/securely-validate-business-application-resilience-with-aws-fis-and-iam/)
- **[Auditing]** Use AWS CloudTrail to capture AWS FIS API calls and the downstream service API calls made during experiments. — CloudTrail logs provide who ran what action, on which resources, and when, supporting audit and incident investigation. [doc](https://docs.aws.amazon.com/fis/latest/userguide/monitoring-experiments.html)

## 🛡️ Reliability
- **[Planning]** Start experiments in a pre-production or test environment before ever targeting production, and progress to production only as confidence builds. — Fault injection performs real actions on real resources, so an unvalidated experiment can cause unintended customer impact. [doc](https://docs.aws.amazon.com/fis/latest/userguide/getting-started-planning.html)
- **[Planning]** Define the steady-state behavior of the system (latency, error rate, retries, etc.) and form an explicit hypothesis of expected impact before running an experiment. — A clear hypothesis turns an experiment into a falsifiable test of resilience rather than an unstructured disruption. [doc](https://docs.aws.amazon.com/fis/latest/userguide/getting-started-planning.html)
- **[Safety]** Attach stop conditions backed by CloudWatch alarms to every experiment template so the experiment automatically halts if steady-state thresholds are breached. — Stop conditions bound the blast radius of an experiment and prevent it from cascading into a real outage. [doc](https://docs.aws.amazon.com/fis/latest/userguide/stop-conditions.html)
- **[Targeting]** Use resource tags and filters (for example `environment:prod` or a `chaos-ready` tag) to scope experiment targets precisely instead of broad selectors. — Fine-grained targeting keeps experiments limited to intended resources and dimensions, reducing risk of unintended impact. [doc](https://aws.amazon.com/fis/features/)
- **[Program]** Review outage history, identify the services with the largest customer impact, and consult with the team to prioritize which failure hypotheses to test. — Prioritizing experiments based on real risk and past incidents produces more actionable resilience findings than arbitrary fault injection. [doc](https://docs.aws.amazon.com/fis/latest/userguide/getting-started-planning.html)
- **[Program]** Run chaos engineering experiments regularly, both inside CI/CD pipelines and independently of deployments, and treat experiment definitions as code maintained through the development lifecycle. — Continuous, repeatable experimentation (rather than one-off tests) is needed to keep confidence in resilience current as the system evolves. [doc](https://docs.aws.amazon.com/wellarchitected/latest/reliability-pillar/rel_testing_resiliency_failure_injection_resiliency.html)
- **[Cross-account/Region]** When running multi-account or cross-Region scenarios, validate target resource selection with the FIS target preview feature and verify that execution roles can assume every cross-account role involved before starting the experiment. — Multi-account experiments have a wider blast radius, so confirming scope and role assumption in advance avoids touching unintended accounts. [doc](https://aws.amazon.com/blogs/mt/scaling-aws-fault-injection-service-across-your-organization-and-regions/)

## ⚙️ Operational Excellence
- **[Monitoring]** Establish comprehensive monitoring and alerting (CloudWatch metrics/alarms) before running any experiment, since without it you cannot measure or understand experiment impact. — Monitoring is the mechanism that lets you observe steady-state deviation and validate or refute your hypothesis. [doc](https://docs.aws.amazon.com/fis/latest/userguide/getting-started-planning.html)
- **[Observability]** Enable AWS FIS experiment logging in addition to CloudWatch metrics and EventBridge state-change notifications to capture detailed experiment execution data. — Combining logs, metrics, and events gives full visibility into what happened at each stage of an experiment for post-experiment review. [doc](https://docs.aws.amazon.com/fis/latest/userguide/monitoring-experiments.html)
- **[Automation]** Integrate FIS experiments into CI/CD pipelines (for example with CodePipeline or Step Functions) so resilience is verified automatically with every deployment. — Automated, repeated fault injection catches resilience regressions early instead of relying on infrequent manual testing. [doc](https://aws.amazon.com/blogs/architecture/chaos-testing-with-aws-fault-injection-simulator-and-aws-codepipeline/)
- **[Architecture review]** Review the full application architecture, including all components, dependencies, and recovery procedures, against the AWS Well-Architected Framework before designing experiments. — Understanding dependencies ensures experiments target meaningful failure modes and that recovery procedures are actually exercised. [doc](https://docs.aws.amazon.com/fis/latest/userguide/getting-started-planning.html)
- **[Scenario library]** Use AWS FIS pre-built scenarios (for example Availability Zone power interruption or cross-Region connectivity disruption) as a starting point instead of building every experiment template from scratch. — Managed scenarios reduce undifferentiated setup work and come with suggested metrics to monitor. [doc](https://aws.amazon.com/fis/features/)
