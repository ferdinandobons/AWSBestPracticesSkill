# AWS Systems Manager — Best Practices

## Common scenarios
- Remote shell/RDP access to EC2 and hybrid nodes without open SSH/RDP ports        → Security, Reliability
- Centralized OS and application patch management across fleets and accounts        → Security, Operational Excellence
- Centralized configuration/secrets storage (Parameter Store) for apps and IaC        → Security, Operational Excellence
- Automated runbooks and drift remediation for routine operational tasks        → Operational Excellence, Reliability

## 🔒 Security
- **[IAM access]** Grant only the Systems Manager permissions required for each role or task using IAM policies and permissions boundaries — reduces blast radius from compromised credentials or human error. [doc](https://docs.aws.amazon.com/systems-manager/latest/userguide/security-best-practices.html)
- **[Parameter Store]** Store secrets and sensitive configuration as `SecureString` parameters encrypted with a customer managed KMS key rather than the AWS managed key — gives you fine-grained IAM/key-policy control over who can decrypt each secret. [doc](https://docs.aws.amazon.com/systems-manager/latest/userguide/security-best-practices.html)
- **[SSM documents]** Define `allowedValues` and `allowedPattern` on document parameters, and turn on block public sharing for documents — prevents invalid or malicious input and unintended public exposure of runbooks. [doc](https://docs.aws.amazon.com/systems-manager/latest/userguide/security-best-practices.html)
- **[Network access]** Use VPC interface endpoints (AWS PrivateLink) for Systems Manager so managed nodes don't need public IP addresses or internet egress — keeps management traffic on the AWS network. [doc](https://docs.aws.amazon.com/systems-manager/latest/userguide/security-best-practices.html)
- **[Session Manager]** Restrict Session Manager to interactive-commands or specific session documents, and scope access with IAM conditions on resource tags (e.g., ABAC) — limits what commands and which nodes a given user can reach. [doc](https://docs.aws.amazon.com/systems-manager/latest/userguide/getting-started-restrict-access-examples.html)
- **[Just-in-time access]** Require operators to request time-bound access to managed nodes before starting a Session Manager or RDP session, and record session activity to S3 — moves toward zero standing privileges and improves auditability. [doc](https://docs.aws.amazon.com/systems-manager/latest/userguide/systems-manager-just-in-time-node-access.html)
- **[Automation permissions]** Grant temporary, scoped permissions inside an Automation runbook instead of adding broad permissions to the node's instance profile — access is automatically removed once the workflow completes. [doc](https://docs.aws.amazon.com/systems-manager/latest/userguide/security-best-practices.html)
- **[Agent hardening]** Keep SSM Agent, the AWS CLI, the Session Manager plugin, and the CloudWatch agent updated on a recurring cadence (at least every two weeks), and verify the SSM Agent signature during updates — closes known vulnerabilities in the management plane. [doc](https://docs.aws.amazon.com/systems-manager/latest/userguide/security-best-practices.html)
- **[Monitoring]** Enable CloudTrail for Systems Manager API activity and AWS Config to track resource configuration and compliance over time — supports incident investigation and continuous audit. [doc](https://docs.aws.amazon.com/systems-manager/latest/userguide/security-best-practices.html)

## 🛡️ Reliability
- **[Host management]** Enable Default Host Management Configuration at the organization level so EC2 instances automatically get the IAM role and settings needed to stay managed, with automatic drift remediation if a node loses managed status. [doc](https://docs.aws.amazon.com/systems-manager/latest/userguide/settings-overview.html)
- **[State Manager]** Use State Manager associations to continuously enforce desired configuration (patch level, domain join, agent config) on nodes and automatically remediate drift rather than relying on one-time setup. [doc](https://docs.aws.amazon.com/systems-manager/latest/userguide/state-manager-about.html)
- **[Patch scheduling]** Patch through maintenance windows scoped by patch groups, with one maintenance window per environment/Availability Zone, so patching doesn't interrupt production availability. [doc](https://docs.aws.amazon.com/systems-manager/latest/userguide/sysman-patch-scheduletasks.html)
- **[Rollout safety]** Use patch baseline auto-approval delays and staged patch groups (dev → test → prod) so a patch is validated in lower environments before it reaches production. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/patch-management-hybrid-cloud/process.html)
- **[Automation at scale]** Use Automation rate controls (concurrency and error-threshold settings) when running disruptive operations across a fleet — stops a bad runbook from cascading across all targets at once. [doc](https://docs.aws.amazon.com/systems-manager/latest/userguide/systems-manager-automation.html)

## ⚙️ Operational Excellence
- **[Patching]** Centralize patching with Patch Manager Quick Setup patch policies so a single org-wide policy defines schedules and baselines across all accounts and Regions, instead of per-account configuration. [doc](https://docs.aws.amazon.com/systems-manager/latest/userguide/quick-setup-patch-manager.html)
- **[Hybrid coverage]** Register on-premises servers and other-cloud machines via Systems Manager hybrid activations so the same patch policy, schedule, and reporting apply consistently outside AWS. [doc](https://docs.aws.amazon.com/systems-manager/latest/userguide/hybrid-activation-managed-nodes.html)
- **[Runbooks]** Codify recurring operational and incident-response tasks as Automation runbooks (including custom logic via `aws:executeScript`) instead of ad hoc scripts — improves consistency and reduces operator error. [doc](https://docs.aws.amazon.com/systems-manager/latest/userguide/systems-manager-automation.html)
- **[Playbooks]** Build investigation playbooks for common incidents and progressively automate them into Systems Manager Automations as your operations practice matures, integrating with Incident Manager for coordinated response. [doc](https://docs.aws.amazon.com/wellarchitected/latest/operational-excellence-pillar/ops_ready_to_support_use_playbooks.html)
- **[Parameter organization]** Organize Parameter Store values using hierarchies (paths), tags, and labels instead of a flat namespace — makes it easier to find the right parameter and manage per-environment configuration versions safely. [doc](https://docs.aws.amazon.com/systems-manager/latest/userguide/sysman-paramstore-hierarchies.html)
- **[AMI management]** Reference AWS or self-managed "golden AMI" parameters in Parameter Store from launch configurations instead of hardcoding AMI IDs — ensures new instances always launch from the current approved, patched image. [doc](https://docs.aws.amazon.com/systems-manager/latest/userguide/systems-manager-parameter-store.html)
- **[Inventory and visibility]** Enable Quick Setup inventory collection (every 30 minutes) across managed nodes to maintain an up-to-date view of installed applications, agents, and configuration for audits and troubleshooting. [doc](https://docs.aws.amazon.com/systems-manager/latest/userguide/quick-setup-host-management.html)
