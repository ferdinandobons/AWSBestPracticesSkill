# AWS Compute Optimizer — Best Practices

## Common scenarios
- Rightsizing EC2 instances and Auto Scaling groups to cut waste        → Cost Optimization, Performance Efficiency
- Identifying idle EC2, EBS, RDS, NAT Gateway, and other resources for cleanup        → Cost Optimization
- Multi-account, organization-wide compute governance and reporting        → Operational Excellence, Security
- Automating rightsizing and cleanup actions at scale        → Operational Excellence, Cost Optimization

## 🔒 Security
- **[IAM]** Grant only the minimum Compute Optimizer and related read permissions (for example `ec2:DescribeInstances`, `ec2:DescribeVolumes`, `autoscaling:Describe*`, `lambda:ListFunctions`) needed to view recommendations — avoid broad wildcard access. — least-privilege access to recommendation data limits exposure of utilization and configuration details. [doc](https://docs.aws.amazon.com/compute-optimizer/latest/ug/security-iam.html)
- **[Organizations]** Configure a delegated administrator account for Compute Optimizer instead of using the management account directly. — lets you exercise least privilege by minimizing access to the organization's management account while still enabling org-wide visibility. [doc](https://docs.aws.amazon.com/compute-optimizer/latest/ug/delegate-administrator-account.html)
- **[Organizations]** Rely on AWS Organizations trusted access, enabled automatically when opting in from the management account, and re-enable it explicitly if it is ever disabled. — trusted access controls whether Compute Optimizer can analyze and generate recommendations for member accounts. [doc](https://docs.aws.amazon.com/compute-optimizer/latest/ug/security-iam.html)
- **[Automation]** Scope Compute Optimizer Automation permissions separately (full vs. read-only access policies) for standalone accounts and organization management accounts. — separating automation permissions from read-only recommendation viewing reduces the risk of unintended changes to resources. [doc](https://docs.aws.amazon.com/compute-optimizer/latest/ug/security-iam.html)

## ⚡ Performance Efficiency
- **[EC2]** Enable memory utilization metrics via the unified CloudWatch Agent or external metrics ingestion (Datadog, Dynatrace, Instana, New Relic). — CloudWatch does not collect memory by default, and full CPU-plus-memory visibility produces higher-confidence recommendations for memory-intensive workloads like databases and JVM applications. [doc](https://docs.aws.amazon.com/compute-optimizer/latest/ug/external-metrics-ingestion.html)
- **[EC2]** Tune CPU utilization threshold (P90/P95/P99.5) and CPU/memory headroom preferences to match each workload's risk tolerance rather than relying only on defaults. — production workloads that can't afford to miss peaks should use P99.5, while dev/test workloads can use P90 for smaller, cheaper recommendations. [doc](https://docs.aws.amazon.com/compute-optimizer/latest/ug/rightsizing-preferences.html)
- **[EC2]** Evaluate Graviton (ARM64) recommendations with a structured process — check binary/library compatibility, verify third-party agent support, and load-test in a non-production environment before migrating. — a CPU architecture shift carries more validation risk than a same-architecture resize, even though it can deliver significant price-performance gains. [doc](https://aws.amazon.com/blogs/compute/optimize-ec2-costs-with-aws-compute-optimizer-right-sizing/)
- **[EC2]** Extend the metrics lookback period (32 or 93 days via Enhanced Infrastructure Metrics) for workloads with monthly or seasonal utilization patterns instead of relying on the 14-day default. — a longer history captures recurring peaks that a short lookback window would miss, improving recommendation relevance. [doc](https://docs.aws.amazon.com/compute-optimizer/latest/ug/enhanced-infrastructure-metrics.html)

## 💰 Cost Optimization
- **[FinOps]** Enable AWS Cost Optimization Hub so Compute Optimizer savings estimates reflect existing Savings Plans, Reserved Instances, and other discounts rather than On-Demand rates only. — organizations with significant discount coverage can otherwise see inflated on-demand savings estimates that don't match realized savings. [doc](https://aws.amazon.com/blogs/compute/optimize-ec2-costs-with-aws-compute-optimizer-right-sizing/)
- **[EC2]** Review idle resource recommendations (EC2, EBS, RDS, NAT Gateway, ElastiCache, DynamoDB, and more) regularly and act on findings to delete, stop, or downsize unused resources. — idle resources generate ongoing cost with no workload benefit and are flagged automatically from utilization data. [doc](https://docs.aws.amazon.com/compute-optimizer/latest/ug/view-idle-recommendations.html)
- **[EC2]** Set preferred instance type constraints in rightsizing preferences to align recommendations with procurement commitments (Reserved Instances, Savings Plans) or approved instance families. — restricting the recommendation pool prevents suggestions that would strand existing purchase commitments or violate governance standards. [doc](https://docs.aws.amazon.com/compute-optimizer/latest/ug/rightsizing-preferences.html)
- **[Organizations]** Opt in at the organization level rather than per-account when managing many accounts, so rightsizing becomes a recurring, centralized activity instead of a one-time, per-account exercise. — organization-wide enrollment centralizes multi-account, multi-Region recommendation data into a single view. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/optimize-costs-microsoft-workloads/rightsize.html)
- **[EBS]** Use Compute Optimizer Automation rules to continuously snapshot-and-delete long-unattached EBS volumes and upgrade outdated volume types on a recurring schedule, scoped by Region and resource tags. — automation turns one-time cleanup into a sustained, low-effort savings process and includes rollback if an action needs to be reversed. [doc](https://docs.aws.amazon.com/compute-optimizer/latest/ug/automation-rules.html)

## ⚙️ Operational Excellence
- **[Tagging]** Tag resources consistently (at minimum Environment, and ideally Application/Workload and Owner/Team) so recommendations can be filtered and routed to the right team. — Compute Optimizer's console, exports, and API all support tag-based filtering, making tagged fleets far more actionable at scale. [doc](https://aws.amazon.com/blogs/compute/optimize-ec2-costs-with-aws-compute-optimizer-right-sizing/)
- **[Process]** Establish a regular rightsizing review cadence (weekly or bi-weekly) with FinOps or cloud operations, prioritizing over-provisioned instances with high savings and low performance risk first. — a structured, repeatable workflow converts one-off recommendations into a continuous optimization loop rather than a point-in-time cleanup. [doc](https://aws.amazon.com/blogs/compute/optimize-ec2-costs-with-aws-compute-optimizer-right-sizing/)
- **[Process]** Validate recommendations with application owners before acting, since metrics alone may not reveal seasonal traffic or scheduled batch jobs. — owner context reduces the risk of a rightsizing change causing an unexpected production incident. [doc](https://aws.amazon.com/blogs/compute/optimize-ec2-costs-with-aws-compute-optimizer-right-sizing/)
- **[Automation]** Automate recommendation ingestion and rightsizing actions using Amazon EventBridge, AWS Step Functions, and AWS Lambda for EC2, in addition to native Compute Optimizer Automation rules for EBS. — automation scales optimization beyond manual review cycles across large fleets and many accounts. [doc](https://aws.amazon.com/blogs/aws-cloud-financial-management/optimize-costs-by-automating-aws-compute-optimizer-recommendations/)
- **[Organizations]** Track realized savings after implementing changes (for example with AWS Cost Explorer) to close the loop between recommendations and actual outcomes. — measuring realized savings validates the rightsizing program and informs future preference tuning. [doc](https://aws.amazon.com/blogs/compute/optimize-ec2-costs-with-aws-compute-optimizer-right-sizing/)
