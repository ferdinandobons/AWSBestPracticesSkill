# AWS Service Catalog — Best Practices

## Common scenarios
- Curating a self-service catalog of approved IT products for end users        → Security, Operational Excellence
- Enforcing compliant, standardized infrastructure across teams and accounts        → Security, Reliability
- Deploying approved products consistently across multiple accounts and Regions        → Reliability, Operational Excellence
- Tracking and controlling spend for provisioned catalog products        → Cost Optimization

## 🔒 Security
- **[launch constraints]** Attach an IAM role to each product through a launch constraint instead of granting end users direct provisioning permissions — this lets users provision resources through the catalog without granting them permissions to perform unapproved operations directly. [doc](https://docs.aws.amazon.com/servicecatalog/latest/adminguide/security-best-practices.html)
- **[template constraints]** Define template constraints to limit the parameter values end users can enter when launching a product — Service Catalog always enforces the most restrictive constraint across the portfolio and product, which keeps deployments (for example, allowed EC2 instance types) within approved bounds. [doc](https://docs.aws.amazon.com/servicecatalog/latest/adminguide/security-best-practices.html)
- **[access control]** Grant portfolio access using IAM policies scoped to users, groups, and roles rather than broad account-wide permissions — this limits who can browse, view, and modify products and portfolios to only those who need it. [doc](https://docs.aws.amazon.com/servicecatalog/latest/adminguide/controlling_access.html)
- **[ABAC]** Use attribute-based access control with tags on Service Catalog resources and principals to build dynamic, tag-based IAM permission policies — this scales more gracefully than static per-resource policies as the organization and catalog grow. [doc](https://aws.amazon.com/blogs/mt/secure-cloud-assets-using-aws-service-catalogs-attribute-based-access-control/)
- **[network access]** Access AWS Service Catalog APIs privately from your VPC using VPC endpoints powered by AWS PrivateLink — this keeps traffic between your VPC and Service Catalog off the public internet without requiring an internet gateway, NAT gateway, or VPN. [doc](https://docs.aws.amazon.com/servicecatalog/latest/adminguide/data-protection.html)
- **[transport security]** Require clients to use TLS 1.2 or later (TLS 1.3 recommended) with cipher suites that support perfect forward secrecy when calling Service Catalog APIs — this protects API traffic in transit per the service's infrastructure security guidance. [doc](https://docs.aws.amazon.com/servicecatalog/latest/adminguide/infrastructure-security.html)
- **[data residency]** Make portfolios and products available only in the Regions you intend to use, and use the CopyProduct API to explicitly replicate products to other Regions — this gives you control over where product data is stored. [doc](https://docs.aws.amazon.com/servicecatalog/latest/adminguide/infrastructure-security.html)

## 🛡️ Reliability
- **[multi-account/Region deployment]** Use a stack set constraint to deploy a product consistently across multiple designated accounts and Regions, with end users controlling the target accounts and deployment order — this standardizes provisioning at scale instead of relying on manual per-account deployments. [doc](https://docs.aws.amazon.com/servicecatalog/latest/adminguide/constraints-stackset.html)
- **[self-service actions]** Enable Service Catalog self-service actions so end users can perform approved operational tasks (backup and restore, troubleshooting, running approved commands) without administrator involvement — this reduces administrative maintenance load while keeping operations within compliance and security guardrails. [doc](https://docs.aws.amazon.com/servicecatalog/latest/adminguide/disaster-recovery-resiliency.html)
- **[versioning]** Test product template updates in CloudFormation (or the Terraform engine) before publishing a new product version — new versions are automatically distributed to every user with portfolio access, so validating changes first avoids propagating a broken template. [doc](https://docs.aws.amazon.com/servicecatalog/latest/adminguide/productmgmt-update.html)

## ⚙️ Operational Excellence
- **[tagging]** Define TagOptions on portfolios and products to give end users a consistent, constrained set of tag keys and values at launch time — this enforces a standard taxonomy for provisioned resources instead of relying on free-text tags. [doc](https://docs.aws.amazon.com/cdk/api/v2/docs/aws-cdk-lib.aws_servicecatalog-readme.html)
- **[versioning]** Manage products through explicit versions rather than overwriting templates in place — each new version is tracked separately so end users can choose which version to launch and update existing provisioned products deliberately. [doc](https://docs.aws.amazon.com/servicecatalog/latest/adminguide/managing-versions.html)
- **[auditability]** Enable an organization-wide CloudTrail trail to capture all Service Catalog API calls (for example CreatePortfolio, CreateProduct, UpdateProvisionedProduct) — the logged identity information lets you determine who made each change and whether it came from a user, role, or another AWS service. [doc](https://docs.aws.amazon.com/servicecatalog/latest/adminguide/logging-using-cloudtrail.html)
- **[notifications]** Configure notification constraints to publish Amazon SNS notifications about stack events for provisioned products — this gives administrators and stakeholders visibility into provisioning activity as it happens. [doc](https://docs.aws.amazon.com/servicecatalog/latest/adminguide/logging-and-monitoring.html)
- **[monitoring plan]** Before relying on Service Catalog in production, define a monitoring plan covering goals, resources to watch, monitoring frequency, tooling, and escalation owners, then collect CloudWatch metrics against it — Service Catalog metric data is delivered to CloudWatch in 1-minute periods, supporting this kind of proactive plan. [doc](https://docs.aws.amazon.com/servicecatalog/latest/adminguide/service-catalog-monitoring.html)
- **[CI/CD governance]** Deliver Service Catalog products through a CI/CD pipeline (for example using the AWS CDK) rather than manual console changes — this reduces the operational complexity of managing shared accounts, roles, and portfolios as the number of products and teams grows. [doc](https://aws.amazon.com/blogs/devops/govern-ci-cd-best-practices-via-aws-service-catalog/)

## 💰 Cost Optimization
- **[budgets]** Associate AWS Budgets with Service Catalog portfolios using the portfolio tag as the budget's tracking dimension — this lets you set cost thresholds per portfolio and receive alerts based on actual or forecasted spend. [doc](https://aws.amazon.com/blogs/aws-cloud-financial-management/launch-aws-service-catalog-integration-with-aws-budgets/)
- **[cost allocation tagging]** Automate propagation and activation of cost-allocation tags (via TagOptions and account-level tags) for resources provisioned through Service Catalog — this ensures spend from provisioned products is attributable and visible in Cost Explorer and the Cost and Usage Report. [doc](https://aws.amazon.com/blogs/aws-cloud-financial-management/automating-tagging-for-resources-created-by-aws-service-catalog/)
- **[constrained instance types]** Use template constraints to restrict end users to approved, cost-appropriate resource sizes (for example specific EC2 instance types) rather than allowing arbitrary selections — this prevents oversized or non-standard resources from being provisioned through the catalog. [doc](https://docs.aws.amazon.com/servicecatalog/latest/adminguide/security-best-practices.html)
