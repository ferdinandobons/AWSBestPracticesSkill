# AWS Control Tower — Best Practices

## Common scenarios
- Setting up a secure, multi-account landing zone for a new organization        → Security, Operational Excellence
- Enforcing preventive and detective guardrails across many AWS accounts        → Security, Operational Excellence
- Standardizing and automating new-account provisioning at scale        → Operational Excellence, Reliability
- Centralizing logging, backup, and compliance monitoring across an organization        → Reliability, Security

## 🔒 Security
- **[resource integrity]** Do not modify or delete resources that AWS Control Tower created in the management account, shared accounts, or member accounts — changes can force a landing zone update or OU re-registration and can produce inaccurate compliance reporting. [doc](https://docs.aws.amazon.com/controltower/latest/userguide/getting-started-guidance.html)
- **[Config recorder]** Keep the AWS Config recorder active in every governed account — if it is deleted, detective controls can no longer detect and report drift, and non-compliant resources may be reported as compliant due to insufficient information. [doc](https://docs.aws.amazon.com/controltower/latest/userguide/getting-started-guidance.html)
- **[execution role]** Never delete the `AWSControlTowerExecution` role from member accounts, even unenrolled ones — without it you cannot enroll the account or register its parent OU with AWS Control Tower. [doc](https://docs.aws.amazon.com/controltower/latest/userguide/getting-started-guidance.html)
- **[Region restrictions]** Do not disallow AWS Regions using SCPs or AWS STS directly — this can put AWS Control Tower into an undefined state; instead use the built-in Region deny control at the landing zone or OU level. [doc](https://docs.aws.amazon.com/controltower/latest/userguide/getting-started-guidance.html)
- **[FullAWSAccess SCP]** Keep the AWS Organizations `FullAWSAccess` SCP attached and do not merge it with other SCPs — detaching or modifying it is not reported as drift but can unpredictably break AWS Control Tower functionality, such as Config recorder or CloudTrail access. [doc](https://docs.aws.amazon.com/controltower/latest/userguide/getting-started-guidance.html)
- **[security roles]** Do not modify or delete the IAM roles that AWS Control Tower creates in the shared accounts of the Security OU — modification can require a landing zone update and can weaken the audit/log-archive security boundary. [doc](https://docs.aws.amazon.com/controltower/latest/userguide/getting-started-guidance.html)
- **[control selection]** Assess each workload and OU's actual requirements before enabling controls, and align control selection to a recognized IT compliance framework (such as NIST 800-53, CIS AWS Benchmark, or PCI-DSS) that AWS Control Tower groups controls by — this produces a consistent, auditable governance baseline instead of ad hoc control choices. [doc](https://aws.amazon.com/blogs/mt/best-practices-for-applying-controls-with-aws-control-tower/)
- **[control types]** Understand how each control is implemented before enabling it — preventive controls (SCPs) disallow actions outright, detective controls (AWS Config rules) flag violations after the fact, and proactive controls (CloudFormation hooks/Guard rules) block non-compliant resources before they are provisioned — and combine all three rather than relying on one type alone. [doc](https://aws.amazon.com/blogs/mt/best-practices-for-applying-controls-with-aws-control-tower/)
- **[layered controls]** Pair AWS Control Tower preventive controls with AWS Security Hub CSPM detective findings against the Foundational Security Best Practices standard — the two are mutually reinforcing and together give both upfront prevention and continuous detection of drift from a secure baseline. [doc](https://aws.amazon.com/controltower/faqs/)
- **[service access]** Do not use the AWS Organizations `DisableAWSServiceAccess` API to remove AWS Control Tower's trusted access from the organization — doing so can break drift-detection messaging and prevent accurate compliance reporting for OUs, accounts, and controls. [doc](https://docs.aws.amazon.com/controltower/latest/userguide/getting-started-guidance.html)

## 🛡️ Reliability
- **[landing zone foundation]** Start new multi-account environments with an AWS Control Tower landing zone rather than a hand-built one — it establishes a well-architected baseline with AWS Config, CloudTrail, and guardrails already wired together in every governed Region. [doc](https://docs.aws.amazon.com/wellarchitected/latest/management-and-governance-guide/implementation-priorities.html)
- **[drift detection]** Monitor AWS Control Tower drift notifications (via EventBridge in the management account for LZ 4.0+, or SNS for earlier versions) and resolve drift promptly — unresolved drift on OUs, SCPs, or accounts can disrupt AWS Control Tower's ability to manage governance and blocks subsequent landing zone operations. [doc](https://docs.aws.amazon.com/controltower/latest/userguide/governance-drift.html)
- **[auto-enrollment]** Enable auto-enrollment for accounts so that accounts moved between registered OUs are automatically brought under the new OU's governance — this avoids the "moved member account" drift condition and the need for manual re-registration. [doc](https://docs.aws.amazon.com/controltower/latest/userguide/governance-drift.html)
- **[backup]** Apply the AWS Control Tower backup plan across your landing zone — built on AWS Backup, it gives every governed account consistent, best-practice backup rules, retention, and scheduling. [doc](https://docs.aws.amazon.com/controltower/latest/userguide/with-backup.html)
- **[OU sizing]** Keep each registered OU to at most around 680–1000 accounts — registration and re-registration time scales with the number of accounts multiplied by governed Regions, and oversized OUs slow down or block routine operations like adding a new Region. [doc](https://docs.aws.amazon.com/controltower/latest/userguide/getting-started-guidance.html)
- **[nested OU partitioning]** Partition large, fast-growing OUs (such as Workloads or Sandbox) into fixed-size nested OUs (for example, capped at 100 accounts each) instead of one unbounded OU — this keeps registration and update operations predictable as the environment scales into the hundreds or thousands of accounts. [doc](https://aws.amazon.com/blogs/mt/organizing-your-aws-control-tower-landing-zone-with-nested-ous/)

## ⚙️ Operational Excellence
- **[OU design]** Organize accounts into purpose-based OUs (Security, Infrastructure, Workloads, Sandbox, Suspended, Policy Staging) that mirror your multi-account strategy rather than reporting hierarchy — a clean, either flat or nested OU structure lets you apply controls consistently and predictably as you grow. [doc](https://docs.aws.amazon.com/controltower/latest/userguide/aws-multi-account-landing-zone.html)
- **[policy staging]** Maintain a dedicated Policy Staging OU to test new or modified controls and SCPs on non-critical accounts before rolling them out broadly — this limits the blast radius if a policy change has unintended effects. [doc](https://docs.aws.amazon.com/controltower/latest/userguide/aws-multi-account-landing-zone.html)
- **[account provisioning]** Use Account Factory (or Account Factory for Terraform / Account Factory Customization for GitOps-style workflows) to provision new accounts with standardized, pre-approved baseline configuration instead of creating accounts manually — this keeps every new account consistent and audit-ready from day one. [doc](https://docs.aws.amazon.com/controltower/latest/userguide/what-is-control-tower.html)
- **[knowledge transfer]** As the management-account administrator, document and communicate to member-account owners which preventive controls apply to their OU — this lets application teams plan their architecture around guardrails instead of discovering them as deployment failures. [doc](https://docs.aws.amazon.com/controltower/latest/userguide/best-practices.html)
- **[access auditing]** Use IAM access advisor, the IAM policy simulator, and centralized CloudTrail logs to determine the actual scope of user and role access within the landing zone — CloudTrail logs for member-account activity are viewable in the shared Log Archive account. [doc](https://docs.aws.amazon.com/controltower/latest/userguide/best-practices.html)
- **[concurrency limits]** Plan account and control operations around AWS Control Tower's single-action-at-a-time model — only optional-control deployment and up to ten concurrent Account Factory actions can run in parallel, so batch large rollouts accordingly to avoid failed operations. [doc](https://docs.aws.amazon.com/controltower/latest/userguide/getting-started-guidance.html)
- **[conformance packs]** Package detective controls as AWS Config conformance packs, including the sample pack aligned to the Well-Architected Security Pillar — this lets you apply them consistently and repeatably every time you create or update an account. [doc](https://docs.aws.amazon.com/wellarchitected/latest/management-and-governance-guide/implementation-priorities.html)
