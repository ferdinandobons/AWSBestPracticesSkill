# AWS Health — Best Practices

## Common scenarios
- Getting personalized, near-real-time alerts about issues and planned changes affecting your AWS resources        → Operational Excellence, Reliability
- Aggregating health events across every account in an AWS Organization for centralized visibility        → Operational Excellence, Reliability
- Automating incident response and change management workflows (Jira, ServiceNow, Slack, Chime) from health events        → Operational Excellence
- Auditing who can view or manage AWS Health data and organizational view settings        → Security

## 🔒 Security
- **[least privilege]** Grant AWS Health users only the minimum permissions they need — for example, allow access to view the Health Dashboard without granting rights to enable or disable AWS Organizations integration. [doc](https://docs.aws.amazon.com/health/latest/ug/security-best-practices.html)
- **[policy design]** Start from AWS managed policies for common AWS Health use cases, then author customer-managed policies scoped to your specific needs to move toward least-privilege permissions. [doc](https://docs.aws.amazon.com/health/latest/ug/security_iam_id-based-policy-examples.html)
- **[conditions]** Add IAM policy conditions (such as requiring SSL/TLS or restricting calls to specific AWS services) to further restrict how AWS Health permissions can be used. [doc](https://docs.aws.amazon.com/health/latest/ug/security_iam_id-based-policy-examples.html)
- **[policy validation]** Validate new and existing AWS Health IAM policies with IAM Access Analyzer to catch overly permissive grants and confirm policies follow IAM best practices before deployment. [doc](https://docs.aws.amazon.com/health/latest/ug/security_iam_id-based-policy-examples.html)
- **[delegated admin]** Use the Delegated Administrator feature to delegate AWS Health organizational view access to a member account instead of operating directly from the management account, keeping the management account restricted while giving operational teams the visibility they need. [doc](https://docs.aws.amazon.com/health/latest/ug/aggregate-events.html)
- **[auditability]** Enable AWS CloudTrail to capture AWS Health API calls, including caller identity, source IP, and call time, so you can audit who accessed or changed Health configuration. [doc](https://docs.aws.amazon.com/health/latest/ug/monitoring-logging-health-events.html)

## 🛡️ Reliability
- **[event visibility]** Use AWS Health endpoints (the AWS Health API, the `aws.health` EventBridge source, and the AWS Health Dashboard) as the authoritative, most detailed and real-time sources of information about events and changes that could affect your workloads, rather than relying only on email notifications. [doc](https://docs.aws.amazon.com/wellarchitected/latest/financial-services-industry-lens/fsiops5.html)
- **[organization-wide aggregation]** Enable organizational view so a user in the management account (or delegated administrator) can aggregate AWS Health events across every account in the AWS Organization, rather than checking each account individually. [doc](https://docs.aws.amazon.com/health/latest/ug/aggregate-events.html)
- **[event retention]** Remember that organizational view events are retained for only 90 days and that events occurring before organizational view was enabled are not backfilled — export or forward events you need to retain longer. [doc](https://docs.aws.amazon.com/health/latest/ug/aggregate-events.html)
- **[proactive maintenance]** Check the account-specific health view regularly for upcoming maintenance and required upgrades on resources such as Amazon EC2, Amazon RDS, and Aurora, and act before AWS-driven maintenance windows arrive. [doc](https://aws.amazon.com/blogs/database/increase-visibility-of-performance-and-events-on-amazon-rds-and-amazon-aurora/)
- **[region fallback]** Configure Amazon EventBridge rules in US East (N. Virginia) in addition to your primary Region, since AWS Health events for which the Region can't be determined are delivered to US East (N. Virginia) by default. [doc](https://docs.aws.amazon.com/health/latest/ug/security-best-practices.html)
- **[event-driven automation]** Create Amazon EventBridge rules on the `aws.health` source to automatically capture events, trigger corrective actions, or send notifications rather than depending on manual dashboard checks. [doc](https://docs.aws.amazon.com/eventbridge/latest/ref/events-ref-health.html)

## ⚙️ Operational Excellence
- **[multi-channel notification]** Use AWS User Notifications to deliver AWS Health alerts through email, chat, and push channels to the AWS Management Console mobile app, and configure aggregation rules to reduce notification volume for stakeholders who don't need endpoint-level detail. [doc](https://docs.aws.amazon.com/wellarchitected/latest/financial-services-industry-lens/fsiops5.html)
- **[ITSM integration]** Integrate AWS Health events with Jira and ServiceNow through the AWS Service Management Connector so operational and account-impacting events automatically create or update tickets and incidents. [doc](https://docs.aws.amazon.com/wellarchitected/latest/financial-services-industry-lens/fsiops5.html)
- **[chat integration]** Integrate AWS Health with Amazon Chime or Slack so teams receive real-time notifications about events in the channels they already monitor. [doc](https://docs.aws.amazon.com/health/latest/ug/security-best-practices.html)
- **[targeted event rules]** Filter EventBridge rules by category (for example, focus on the `issue` category for incident response, `ACTION_REQUIRED` in the Actionability field for change management, or the `SECURITY` persona and Abuse events for security workflows) so each stakeholder group only receives events relevant to them. [doc](https://docs.aws.amazon.com/health/latest/ug/creating-event-bridge-events-rule-for-aws-health.html)
- **[centralized dashboard review]** Check the Health Dashboard often — both the public Service health page and the account-specific Your account health page — to catch resource issues, upcoming changes, and other notifications before they cause impact. [doc](https://docs.aws.amazon.com/health/latest/ug/security-best-practices.html)
- **[long-term email archive]** Retain AWS Health event emails sent to the account's associated address in a searchable folder so historical events remain available beyond the 90-day organizational view retention window. [doc](https://aws.amazon.com/blogs/database/increase-visibility-of-performance-and-events-on-amazon-rds-and-amazon-aurora/)
- **[monitoring integration]** Combine Amazon CloudWatch, Amazon EventBridge, and AWS CloudTrail to monitor AWS Health itself — track metrics and dashboards, trigger automated actions on events, and log API activity for operational visibility. [doc](https://docs.aws.amazon.com/health/latest/ug/monitoring-logging-health-events.html)
