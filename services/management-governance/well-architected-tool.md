# AWS Well-Architected Tool — Best Practices

## Common scenarios
- Running periodic architecture reviews against the AWS Well-Architected Framework at major milestones        → Operational Excellence
- Centralizing governance by sharing workloads across teams, accounts, and an AWS Organization        → Operational Excellence, Security
- Auto-populating review evidence from Trusted Advisor checks and AppRegistry application resources        → Operational Excellence, Cost Optimization, Reliability, Security
- Tracking architectural improvement over time using milestones and custom lenses/profiles        → Operational Excellence

## 🔒 Security
- **[access control]** Start from AWS managed policies for AWS WA Tool and move toward least-privilege customer managed policies scoped to the specific workload, lens, profile, or review template actions each user needs. [doc](https://docs.aws.amazon.com/wellarchitected/latest/userguide/security_iam_id-based-policy-examples.html)
- **[sharing granularity]** When only specific individuals need access to a workload, share it directly with those IAM users rather than with the entire AWS account, since an account-level share grants access to every user in that account. [doc](https://docs.aws.amazon.com/wellarchitected/latest/userguide/sharing-considerations.html)
- **[conditions]** Add IAM policy conditions (for example, requiring SSL/TLS on requests) to further restrict how AWS WA Tool actions can be invoked beyond what identity-based policies alone provide. [doc](https://docs.aws.amazon.com/wellarchitected/latest/userguide/security_iam_id-based-policy-examples.html)
- **[policy validation]** Validate identity-based policies for AWS WA Tool with IAM Access Analyzer before deploying them, to catch overly permissive grants and confirm policies match IAM best practices. [doc](https://docs.aws.amazon.com/wellarchitected/latest/userguide/security_iam_id-based-policy-examples.html)
- **[MFA]** Require multi-factor authentication for IAM users or the root user who can create, modify, or share AWS WA Tool workloads, given the sensitive architectural and account information workloads can contain. [doc](https://docs.aws.amazon.com/wellarchitected/latest/userguide/security_iam_id-based-policy-examples.html)
- **[audit trail]** Enable an AWS CloudTrail trail that captures AWS WA Tool API calls (CreateWorkload, DeleteWorkload, CreateWorkloadShare, and others) so you retain a durable, cross-Region record of who created, modified, or shared workloads. [doc](https://docs.aws.amazon.com/wellarchitected/latest/userguide/logging-using-cloudtrail.html)
- **[Trusted Advisor role]** When activating the Trusted Advisor integration for a workload, review the auto-created IAM service role's permissions and trust relationship before accepting, since it grants AWS WA Tool read access to Trusted Advisor check results for the specified accounts. [doc](https://docs.aws.amazon.com/wellarchitected/latest/userguide/activate-ta-for-workload.html)

## ⚙️ Operational Excellence
- **[review cadence]** Perform Well-Architected workload reviews at major milestones in the development cycle (design, pre-launch, post-launch changes) rather than only once, so the review reflects the workload's current state. [doc](https://aws.amazon.com/well-architected-tool/faqs/)
- **[milestones]** Save a milestone after completing or meaningfully updating a review to capture an immutable snapshot of the workload's risk posture, giving you a trail of architectural decisions across the workload's lifecycle. [doc](https://docs.aws.amazon.com/wellarchitected/latest/userguide/details-review.html)
- **[centralized governance]** Create workloads in a central account and share them with application owners so a central architectural authority can pre-answer centrally-governed questions and review results across teams, instead of every team starting from scratch. [doc](https://aws.amazon.com/blogs/mt/scaling-well-architected-reviews-with-the-aws-well-architected-tool/)
- **[Organizations sharing]** Enable AWS Organizations support in AWS WA Tool settings before sharing workloads or custom lenses with an organization or specific organizational units, to streamline distribution instead of sharing account-by-account. [doc](https://docs.aws.amazon.com/wellarchitected/latest/userguide/sharing-considerations.html)
- **[standardization]** Use review templates and profiles to pre-populate common answers and prioritize the questions most relevant to your business goals, driving consistency across multiple workloads and reducing repetitive manual entry. [doc](https://aws.amazon.com/well-architected-tool/features/)
- **[custom lenses]** Define custom lenses for technology- or organization-specific best practices that extend beyond the AWS Well-Architected Framework, so governance requirements specific to your business are measured alongside AWS guidance. [doc](https://docs.aws.amazon.com/wellarchitected/latest/userguide/intro.html)
- **[automated discovery]** Activate the Trusted Advisor and AWS Service Catalog AppRegistry integrations for a workload so check results and application resources are surfaced automatically during a review, shortening the time needed to gather evidence manually. [doc](https://aws.amazon.com/blogs/architecture/accelerating-well-architected-framework-reviews-using-integrated-aws-trusted-advisor-insights/)
- **[API-driven workflows]** Use the AWS WA Tool APIs to integrate best-practice measurement, milestones, and lens data directly into existing architecture governance processes and CI/CD workflows, rather than treating reviews as a manual, disconnected activity. [doc](https://docs.aws.amazon.com/wellarchitected/latest/userguide/intro.html)
