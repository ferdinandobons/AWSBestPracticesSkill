# Amazon DevOps Guru — Best Practices

## Common scenarios
- Enabling ML-powered anomaly detection across an account, CloudFormation stacks, or tagged applications        → Operational Excellence, Reliability
- Investigating an operational insight's timeline, metrics, and events to find root cause        → Operational Excellence, Reliability
- Routing insight notifications to on-call and ticketing systems (SNS, OpsCenter)        → Operational Excellence
- Scoping and estimating the cost of resource analysis coverage before enabling broadly        → Cost Optimization

## 🔒 Security
- **[IAM policies]** Start from the AWS managed policies for DevOps Guru and progressively narrow to least-privilege customer-managed policies scoped to specific actions and resources rather than leaving broad managed access in place long-term. [doc](https://docs.aws.amazon.com/devops-guru/latest/userguide/security_iam_id-based-policy-examples.html)
- **[Conditions]** Add IAM policy conditions (for example requiring SSL/TLS, or restricting calls to when made through a specific AWS service like CloudFormation) to further constrain how DevOps Guru permissions can be used. [doc](https://docs.aws.amazon.com/devops-guru/latest/userguide/security_iam_id-based-policy-examples.html)
- **[Policy validation]** Validate identity-based policies for DevOps Guru with IAM Access Analyzer before attaching them — it checks JSON policy language and flags overly permissive statements. [doc](https://docs.aws.amazon.com/devops-guru/latest/userguide/security_iam_id-based-policy-examples.html)
- **[MFA]** Require multi-factor authentication for IAM users or the root user that can manage DevOps Guru resources, since enabling/disabling analysis and changing coverage can affect cost and monitoring posture. [doc](https://docs.aws.amazon.com/devops-guru/latest/userguide/security_iam_id-based-policy-examples.html)
- **[SNS resource policy]** Scope the resource policy on the Amazon SNS topic used for DevOps Guru notifications with `aws:SourceArn`/`aws:SourceAccount` conditions so only your DevOps Guru channel can publish to it. [doc](https://aws.amazon.com/blogs/devops/automate-container-anomaly-monitoring-of-amazon-elastic-kubernetes-service-clusters-with-amazon-devops-guru/)
- **[Notification encryption]** Enable encryption at rest on the SNS topic used for DevOps Guru insight notifications — it is not enabled by default when the topic is created. [doc](https://aws.amazon.com/blogs/devops/automate-container-anomaly-monitoring-of-amazon-elastic-kubernetes-service-clusters-with-amazon-devops-guru/)
- **[Private connectivity]** Use an interface VPC endpoint (AWS PrivateLink) for DevOps Guru API calls so traffic between your VPC and DevOps Guru stays on the AWS network instead of traversing the internet. [doc](https://docs.aws.amazon.com/devops-guru/latest/userguide/vpc-interface-endpoints.html)
- **[Auditing]** Enable a CloudTrail trail that captures DevOps Guru API activity so you can determine who changed resource coverage, notification settings, or insight feedback, and from where. [doc](https://docs.aws.amazon.com/devops-guru/latest/userguide/logging-using-cloudtrail.html)

## 🛡️ Reliability
- **[Insight triage]** Start investigation from the highlighted metrics in an insight's timeline view — they are usually the key indicators of the underlying problem. [doc](https://docs.aws.amazon.com/devops-guru/latest/userguide/best-practices.html)
- **[Root cause]** Cross-reference CloudWatch metrics just before the first highlighted anomaly to pinpoint exactly when and how behavior changed, rather than only looking at the anomaly itself. [doc](https://docs.aws.amazon.com/devops-guru/latest/userguide/best-practices.html)
- **[RDS insights]** For Amazon RDS resources, correlate Performance Insights counter metrics with database load to get detailed diagnosis of database performance anomalies. [doc](https://docs.aws.amazon.com/devops-guru/latest/userguide/best-practices.html)
- **[Multi-dimensional metrics]** Inspect all dimensions of a graphed metric, not just the aggregate — multiple dimensions of the same metric can be anomalous simultaneously and reveal a more precise scope. [doc](https://docs.aws.amazon.com/devops-guru/latest/userguide/best-practices.html)
- **[Correlated events]** Check the events section of an insight for deployment or infrastructure changes that happened around the same time — this often explains the causal trigger for the anomaly. [doc](https://docs.aws.amazon.com/devops-guru/latest/userguide/best-practices.html)
- **[Recommendations]** Read and follow the links inside an insight's recommendations — they frequently include troubleshooting steps that speed up diagnosis and remediation. [doc](https://docs.aws.amazon.com/devops-guru/latest/userguide/best-practices.html)
- **[Resolved insights]** Review new insights daily even after they resolve on their own, and look for recurring patterns — an unresolved systemic issue left unaddressed can cause more serious incidents later. [doc](https://docs.aws.amazon.com/devops-guru/latest/userguide/best-practices.html)
- **[Application boundary]** Define application boundaries with CloudFormation stacks or AWS resource tags rather than the entire account when resources belong to distinct logical applications — this lets DevOps Guru group related anomalies and produce more meaningful, targeted recommendations. [doc](https://aws.amazon.com/blogs/devops/define-application-boundary-using-aws-resources-tags-in-amazon-devops-guru/)

## ⚙️ Operational Excellence
- **[Coverage boundary]** Choose an explicit resource analysis coverage boundary (account, specific CloudFormation stacks, or tagged resource groups) instead of leaving it unscoped, so coverage expands and contracts predictably as you add or remove resources. [doc](https://aws.amazon.com/devops-guru/faqs/)
- **[Notification severity]** Configure SNS notification settings to match your operational tolerance — choose all severity levels or only High/Medium, and choose whether to be notified on every update or only on open/close/severity-escalation events. [doc](https://docs.aws.amazon.com/devops-guru/latest/userguide/update-notifications.html)
- **[Ticketing integration]** Enable automatic OpsItem creation in AWS Systems Manager OpsCenter (or route SNS notifications into an incident tool such as PagerDuty) so insights flow directly into your existing incident management workflow instead of relying on manual console checks. [doc](https://aws.amazon.com/devops-guru/faqs/)
- **[Baselining]** Expect an initial baselining period (minutes to about an hour depending on resource count) after enabling DevOps Guru or adding resources before insights start appearing — don't conclude coverage isn't working during this window. [doc](https://aws.amazon.com/devops-guru/faqs/)
- **[Log anomaly detection]** Enable log-based anomaly detection alongside metric analysis so recommendations draw on contextualized log snippets, not just metrics — this shortens root-cause time for issues that only surface in application logs. [doc](https://aws.amazon.com/blogs/aws/new-detect-and-resolve-issues-quickly-with-log-anomaly-detection-and-recommendations-from-amazon-devops-guru/)
- **[Monitoring DevOps Guru itself]** Use CloudWatch to monitor DevOps Guru and CloudTrail to log its API calls as part of your standard operational tooling, the same as any other production service dependency. [doc](https://docs.aws.amazon.com/devops-guru/latest/userguide/monitoring-overview.html)

## 💰 Cost Optimization
- **[Cost estimation before rollout]** Run the DevOps Guru cost estimator against your intended coverage (account, a CloudFormation stack, or tags) before enabling broad analysis, since you are billed per active analyzed resource-hour. [doc](https://docs.aws.amazon.com/devops-guru/latest/userguide/cost-estimate.html)
- **[Scope to active resources]** Keep the resource coverage boundary aligned to resources you actually operate and want monitored — DevOps Guru stops analyzing and billing resources as soon as they are removed from the account, stack, or tag boundary, so prune stale coverage promptly. [doc](https://aws.amazon.com/devops-guru/faqs/)
- **[Realistic utilization]** Adjust the cost estimator's utilization percentage per service to reflect actual expected usage instead of the default 100% assumption, to avoid over-estimating (or under-planning for) monthly charges. [doc](https://docs.aws.amazon.com/devops-guru/latest/userguide/cost-estimate.html)
