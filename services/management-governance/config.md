# AWS Config — Best Practices

## Common scenarios
- Continuous audit of resource configuration and drift detection        → Security, Operational Excellence
- Compliance-as-code against internal policies or industry standards (CIS, PCI-DSS, HIPAA)        → Security, Operational Excellence
- Multi-account, multi-Region governance and centralized compliance reporting        → Operational Excellence, Reliability
- Cost control of recording volume and rule evaluations at scale        → Cost Optimization

## 🔒 Security
- **[Delivery channel]** Confirm your S3 delivery channel is properly configured and verify AWS Config is recording successfully — an unconfirmed or broken channel silently stops your audit trail. [doc](https://docs.aws.amazon.com/config/latest/developerguide/security-best-practices.html)
- **[S3 bucket]** Use a secure, non-public Amazon S3 bucket for configuration history and snapshot files, and turn on the S3 best-practices conformance pack to catch accidental public access — this bucket holds sensitive configuration details for every recorded resource. [doc](https://aws.amazon.com/blogs/mt/aws-config-best-practices/)
- **[IAM permissions]** Use the AWS Config service-linked role (AWSServiceRoleForConfig) or the AWS-managed `AWS_ConfigRole` policy rather than a hand-rolled role — AWS updates the managed policy automatically as new resource types are supported. [doc](https://docs.aws.amazon.com/config/latest/developerguide/security-iam.html)
- **[SNS topic]** Restrict SNS topic permissions so only AWS Config can publish configuration and compliance notifications — an overly permissive topic policy lets other principals inject or read sensitive change events. [doc](https://aws.amazon.com/blogs/mt/aws-config-best-practices/)
- **[Tagging]** Tag AWS Config resources to make them easier to manage, search for, and filter, especially across large or multi-team environments. [doc](https://docs.aws.amazon.com/config/latest/developerguide/security-best-practices.html)
- **[Access control]** Scope IAM identity-based policies tightly for read-only vs. full access to AWS Config, and separately control access to aggregated cross-account data, since aggregators can expose configuration details from many accounts. [doc](https://docs.aws.amazon.com/config/latest/developerguide/security_iam_id-based-policy-examples.html)
- **[IAM auditing]** Record IAM resource configuration changes (users, groups, roles, policies) so you can reconstruct exactly which permissions were in effect at any point in time for security investigations. [doc](https://aws.amazon.com/blogs/security/how-to-record-and-govern-your-iam-resource-configurations-using-aws-config/)

## 🛡️ Reliability
- **[Coverage]** Enable AWS Config in all accounts and Regions you operate in — this is a CIS-recommended baseline and can be rolled out at scale with CloudFormation StackSets. [doc](https://aws.amazon.com/blogs/mt/aws-config-best-practices/)
- **[Resource scope]** Record configuration changes for all supported resource types (not a hand-picked subset) so new resource types are automatically captured going forward and no blind spots exist in your audit trail. [doc](https://aws.amazon.com/blogs/mt/aws-config-best-practices/)
- **[Snapshots]** Turn on periodic snapshots at a minimum frequency of once per day to guarantee a daily backup of the latest configuration state for every recorded resource. [doc](https://aws.amazon.com/blogs/mt/aws-config-best-practices/)
- **[Central storage]** For multi-account environments, deliver configuration history and snapshots to an S3 bucket in a central account, and stream notifications to a central SNS topic, so a security/IT team has one place to consume data from the whole organization. [doc](https://aws.amazon.com/blogs/mt/aws-config-best-practices/)
- **[Aggregation]** Create an AWS Organizations-based aggregator to collect configuration and compliance data from every account and Region automatically, including accounts that join or leave the organization later. [doc](https://aws.amazon.com/blogs/mt/aws-config-best-practices/)

## ⚡ Performance Efficiency
- **[Rule triggers]** Prefer change-triggered custom rules over periodic rules for resource types supported by AWS Config so compliance evaluation happens promptly on every configuration change rather than on a fixed schedule. [doc](https://aws.amazon.com/blogs/mt/aws-config-best-practices/)
- **[Advanced queries]** Use the Advanced Queries feature to query current resource configuration and compliance state centrally across accounts, Regions, and your organization instead of calling per-service Describe/List APIs individually. [doc](https://aws.amazon.com/blogs/mt/aws-config-best-practices/)
- **[Historical analysis]** Use Amazon Athena against Config history files when you need to query historical (not just current) resource state, since Advanced Queries only covers current state. [doc](https://aws.amazon.com/blogs/mt/aws-config-best-practices/)

## 💰 Cost Optimization
- **[Global resources]** Record global resources such as IAM only in a single Region to avoid redundant configuration items and unnecessary API throttling across every enabled Region. [doc](https://aws.amazon.com/blogs/mt/aws-config-best-practices/)
- **[Global rule scope]** Deploy Config rules and conformance packs that reference global resources (e.g. IAM-related checks) in only one Region, since these rules are duplicated and billed per Region otherwise. [doc](https://aws.amazon.com/blogs/mt/aws-config-best-practices/)
- **[Recording strategy]** Use exclusion- or inclusion-based recording strategies to record only the resource types you actually need instead of "all resources," reducing configuration-item volume and cost. [doc](https://docs.aws.amazon.com/config/latest/developerguide/select-resources-console.html)
- **[Recording frequency]** Switch high-churn or low-priority resource types to daily recording instead of continuous recording when near-real-time capture isn't required, since daily recording only records a snapshot when something changed. [doc](https://aws.amazon.com/blogs/mt/cost-optimization-recommendations-for-aws-config/)
- **[Cost visibility]** Use CloudWatch metrics (and Athena queries against Config history) to identify the resource types and individual resources generating the most configuration items, then target exclusions or frequency changes at those specifically. [doc](https://aws.amazon.com/blogs/mt/cost-optimization-recommendations-for-aws-config/)
- **[ResourceCompliance]** Consider excluding the `AWS::Config::ResourceCompliance` resource type from recording when it dominates your configuration-item volume, since it is generated by rule evaluations rather than actual infrastructure changes. [doc](https://aws.amazon.com/blogs/mt/cost-optimization-recommendations-for-aws-config/)
- **[Ephemeral workloads]** Exclude resource types tied to ephemeral workloads (Spot Instances, EMR jobs, Auto Scaling churn) from recording, or isolate those workloads in a separate account, to avoid a spike in configuration-item costs. [doc](https://docs.aws.amazon.com/config/latest/developerguide/select-resources-console.html)
- **[Rule housekeeping]** Remove Config rules and conformance packs that are no longer needed, and avoid frequent use of "Delete results" / "Re-evaluate" on rules, since each re-evaluation creates a new compliance configuration item that adds to billing. [doc](https://aws.amazon.com/blogs/mt/aws-config-best-practices/)

## ⚙️ Operational Excellence
- **[Conformance packs]** Package Config rules and remediation actions into conformance packs, and deploy them centrally from the management or a delegated administrator account across your AWS Organization, to establish a consistent compliance baseline at scale. [doc](https://aws.amazon.com/blogs/mt/deploying-conformance-packs-across-an-organization-with-automatic-remediation/)
- **[Delegated administration]** Register a delegated administrator account for Config so a dedicated security/compliance account (not just the Organizations management account) can manage organization-wide rules and conformance packs. [doc](https://aws.amazon.com/blogs/mt/deploy-aws-config-rules-and-conformance-packs-using-a-delegated-admin/)
- **[Starter templates]** Start from AWS-provided conformance pack sample templates (industry benchmarks such as CIS, PCI-DSS, HIPAA, FedRAMP) and customize rules, parameters, and remediations for your own environment rather than authoring everything from scratch. [doc](https://aws.amazon.com/blogs/mt/aws-config-best-practices/)
- **[Custom rules]** Use the AWS Config Rule Development Kit (RDK) to author and validate custom rules, and draw on the community AWS Config Rules repository as a starting point. [doc](https://aws.amazon.com/blogs/mt/aws-config-best-practices/)
- **[Event routing]** Route AWS Config configuration and compliance notifications through Amazon CloudWatch Events (in addition to SNS) to filter events and direct specific notification types to the right targets, including external ITSM/CMDB tools. [doc](https://aws.amazon.com/blogs/mt/aws-config-best-practices/)
- **[Change correlation]** Use the CloudTrail LookupEvents API to correlate a configuration change recorded in AWS Config with the specific API call that caused it, speeding up operational and security investigations. [doc](https://aws.amazon.com/blogs/mt/aws-config-best-practices/)
- **[Recording frequency by use case]** Align recording frequency (continuous vs. periodic/daily) with the actual use case — real-time compliance monitoring and incident response need continuous recording, while inventory and routine troubleshooting can often use periodic recording. [doc](https://aws.amazon.com/blogs/mt/best-practices-for-analyzing-aws-config-recording-frequencies/)
- **[Centralized custom rules]** In multi-account setups using custom rules, centralize the underlying Lambda functions in one account and manage deployment via conformance packs for easier maintenance. [doc](https://aws.amazon.com/blogs/mt/aws-config-best-practices/)
