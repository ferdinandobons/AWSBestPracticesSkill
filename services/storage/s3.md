# Amazon S3 — Best Practices

## Common scenarios
- Internet-facing static assets or data lake landing zone -> Security (block public access, least privilege)
- Critical/compliance data needing protection from deletion or ransomware -> Reliability (versioning, Object Lock, MFA delete)
- High-throughput analytics or media workloads reading/writing at scale -> Performance Efficiency (parallelization, byte-range fetches)
- Mixed or unknown access patterns growing over time -> Cost Optimization (Intelligent-Tiering, lifecycle policies)

## 🔒 Security
- **[Public access]** Enable S3 Block Public Access at the account and organization level so buckets are never publicly accessible unless explicitly intended — overrides other permissions and is enforced regardless of how resources are created. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/security-best-practices.html)
- **[Access control]** Disable ACLs with the Bucket owner enforced setting for S3 Object Ownership and manage access exclusively via policies — simplifies permission management and auditing. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/security-best-practices.html)
- **[IAM]** Grant least-privilege permissions in bucket and IAM policies, allowing only the actions required for a task — reduces blast radius from errors or malicious intent. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/security-best-practices.html)
- **[Credentials]** Use IAM roles for applications and AWS services that access S3 instead of embedding long-term keys — supplies auto-rotated temporary credentials and avoids credential leakage. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/security-best-practices.html)
- **[Encryption in transit]** Deny non-HTTPS requests via the `aws:SecureTransport` bucket policy condition — prevents eavesdropping and man-in-the-middle attacks on object traffic. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/security-best-practices.html)
- **[Encryption at rest]** Choose an appropriate server-side encryption mode (SSE-S3, SSE-KMS, or DSSE-KMS) per bucket; all objects are encrypted by default — protects data even if storage media is compromised. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/security-best-practices.html)
- **[Network isolation]** Use VPC endpoints and bucket policies that restrict access to specific VPCs/endpoints — isolates bucket access to your private network. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/network-isolation.html)
- **[Permissions review]** Use IAM Access Analyzer for S3 and AWS Config rules (e.g. `s3-bucket-public-read-prohibited`) as ongoing detective controls — continuously verifies policies grant only required access. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/security-best-practices.html)

## 🛡️ Reliability
- **[Recovery]** Enable S3 Versioning to preserve, retrieve, and restore every object version — lets you recover from unintended user actions and application failures. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/security-best-practices.html)
- **[Immutability]** Use S3 Object Lock (WORM) for critical or compliance data such as CloudTrail logs — prevents accidental or inappropriate deletion or overwrite for a fixed period or indefinitely. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/disaster-recovery-resiliency.html)
- **[Delete protection]** Enable MFA delete on versioned buckets containing critical data — requires a second authentication factor to permanently delete versions, guarding against ransomware, insider threats, and operational errors. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/MultiFactorAuthenticationDelete.html)
- **[Geographic resilience]** Use S3 Replication (Cross-Region or Same-Region) when you must copy data over greater distances or meet compliance and low-latency requirements — provides automatic, asynchronous copies in another bucket or Region. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/disaster-recovery-resiliency.html)

## ⚡ Performance Efficiency
- **[Scaling]** Spread requests horizontally across many concurrent connections and prefixes — each partitioned prefix supports 3,500 writes and 5,500 reads per second, and there is no limit to the number of prefixes. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/optimizing-performance.html)
- **[Large objects]** Use byte-range fetches with the HTTP `Range` header to retrieve object parts in parallel — increases aggregate throughput and improves retry times for interrupted requests. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/optimizing-performance-guidelines.html)
- **[Latency]** Configure aggressive timeouts and retries in the AWS SDKs for latency-sensitive applications — a retried request likely takes a different path and succeeds quickly. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/optimizing-performance-guidelines.html)
- **[Locality]** Access buckets from Amazon EC2 instances in the same Region as the bucket — reduces network latency and data transfer costs. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/optimizing-performance-guidelines.html)
- **[Distance]** Use S3 Transfer Acceleration via CloudFront edge locations for transfers across continents — minimizes latency caused by geographic distance for large or globally distributed uploads. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/optimizing-performance-guidelines.html)
- **[Monitoring]** Monitor 503 (Slow Down) responses via CloudWatch request metrics or S3 Storage Lens — reveals which prefixes, keys, or buckets are being throttled so you can rebalance. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/optimizing-performance-guidelines.html)

## 💰 Cost Optimization
- **[Unknown patterns]** Use S3 Intelligent-Tiering for data with unknown or changing access patterns — automatically moves objects between access tiers to optimize cost without retrieval fees. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/cost-optimization.html)
- **[Lifecycle]** Define S3 Lifecycle rules to transition objects to cheaper storage classes and expire objects no longer needed — automates cost reduction across the data lifecycle for existing and new objects. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/cost-optimization.html)
- **[Right-sizing]** Match storage class to access pattern (Standard-IA / One Zone-IA for infrequent, Glacier tiers for archive) — purpose-built classes provide the lowest cost for each access level. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/cost-optimization.html)
- **[Analysis]** Use S3 Storage Class Analysis, cost allocation tags, and usage reports to study access patterns — informs which objects to transition and where savings exist. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/cost-optimization.html)

## ⚙️ Operational Excellence
- **[Auditing]** Identify and audit all buckets with tools like Trusted Advisor and the `ListBuckets`/`GetBucketPolicy` APIs — ensures every bucket has compliant access controls and configuration. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/security-best-practices.html)
- **[Threat detection]** Enable managed security services such as Amazon GuardDuty for S3 and AWS Security Hub S3 controls — provides continuous monitoring and detective controls over S3 activity. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/security-best-practices.html)
- **[Namespace]** Create buckets in your account regional namespace rather than the shared global namespace — these names are unique to your account and cannot be re-created by another account after deletion. [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/security-best-practices.html)
