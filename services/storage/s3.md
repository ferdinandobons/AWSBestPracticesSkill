# Amazon S3 — Best Practices

## Common scenarios
- Central data lake or analytics storage layer for structured and unstructured data        → Performance Efficiency, Cost Optimization
- Application storage for static assets, backups, logs, and media        → Reliability, Security
- Long-term archival and regulatory/compliance retention of infrequently accessed data        → Cost Optimization, Reliability
- High-throughput object storage backing distributed or high-request-rate workloads        → Performance Efficiency, Reliability

## 🔒 Security
- **[Public access]** Enable S3 Block Public Access at the account level (and on individual buckets) to prevent unintended public exposure of data. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/access-control-block-public-access.html)
- **[Access control]** Use S3 Object Ownership with the bucket owner enforced setting to disable ACLs and manage access solely through bucket policies and IAM policies. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/network-isolation.html)
- **[Least privilege]** Start from AWS managed policies and progressively narrow to customer managed, least-privilege IAM policies scoped to specific S3 actions and resources. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/security_iam_id-based-policy-examples.html)
- **[Policy conditions]** Add IAM policy conditions (for example, requiring SSL/TLS or restricting by source IP or VPC endpoint) to further restrict access to buckets and objects. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/security_iam_id-based-policy-examples.html)
- **[Policy validation]** Use IAM Access Analyzer to validate identity-based and bucket policies for correctness and to identify unintended public or cross-account access. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/security_iam_id-based-policy-examples.html)
- **[Network isolation]** Use VPC endpoints for Amazon S3 and bucket policies that restrict access to specific VPCs or VPC endpoints, isolating access to the AWS network. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/network-isolation.html)
- **[Transport security]** Require TLS 1.2 (and where possible TLS 1.3) for all client connections, since S3 API requests must use Signature Version 4 or Signature Version 2 over TLS. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/network-isolation.html)
- **[Encryption at rest]** Rely on default server-side encryption (SSE-S3), which Amazon S3 applies automatically to all new object uploads. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/bucket-encryption.html)
- **[Encryption at rest]** Configure default bucket encryption with SSE-KMS when you need audit trails of key usage and more granular control over key policies. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/bucket-encryption.html)
- **[Key cost management]** Enable S3 Bucket Keys when using SSE-KMS to reduce the number of calls to AWS KMS for encryption operations. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/bucket-encryption.html)
- **[Asset visibility]** Identify and tag all S3 buckets (using Tag Editor and resource groups) so you have full visibility into your S3 footprint for security assessment. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/security-best-practices.html)
- **[Auditing]** Use S3 Inventory to audit and report on the replication and encryption status of objects for compliance and regulatory needs. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/security-best-practices.html)
- **[Auditing]** Enable AWS CloudTrail data events for bucket-level and object-level API tracking, including identity, IP address, and timestamp of each request. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/cloudtrail-logging.html)
- **[Auditing]** Enable S3 server access logging to get detailed records of requests made to a bucket for security and access audits. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/security-best-practices.html)
- **[Threat detection]** Monitor CloudWatch metrics such as `4xxErrors`, `PutRequests`, and `DeleteRequests`, and configure alarms on sensitive API calls like bucket policy or ACL changes. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/security-best-practices.html)

## 🛡️ Reliability
- **[Versioning]** Enable S3 Versioning on buckets so you can recover from unintended user actions, application failures, or accidental deletions and overwrites. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/disaster-recovery-resiliency.html)
- **[Versioning cost]** Combine versioning with S3 Lifecycle policies to manage and expire older noncurrent versions and control the resulting storage cost growth. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/lifecycle-configuration-examples.html)
- **[Replication]** Use S3 Cross-Region Replication (CRR) or Same-Region Replication (SRR) to maintain a second copy of critical objects in a different bucket, account, or Region. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/replication.html)
- **[Replication SLA]** Use S3 Replication Time Control (RTC) when you need a predictable, SLA-backed replication time of under 15 minutes. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/replication.html)
- **[Immutability]** Use S3 Object Lock in write-once-read-many (WORM) mode to prevent objects from being deleted or overwritten for a fixed period or indefinitely, protecting against ransomware and accidental deletion. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/disaster-recovery-resiliency.html)
- **[Backup]** Use AWS Backup for S3 to automate periodic backups of buckets as an alternative or complement to replication. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/backup-for-s3.html)
- **[Retry handling]** Implement exponential backoff and retry logic (built into the AWS SDKs) to handle transient HTTP 503 (Slow Down) responses during request-rate scaling. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/optimizing-performance-design-patterns.html)

## ⚡ Performance Efficiency
- **[Request distribution]** Distribute high-volume requests across multiple prefixes to take advantage of per-prefix scaling and avoid throttling during rapid scale-up. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/optimizing-performance-design-patterns.html)
- **[Gradual ramp-up]** Gradually ramp up request rates for new or bursty workloads rather than jumping immediately to peak traffic, so Amazon S3 can scale proactively. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/optimizing-performance-design-patterns.html)
- **[Parallelization]** Scale read and write throughput horizontally by using multiple prefixes and multiple concurrent connections rather than relying on a single connection. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/optimizing-performance.html)
- **[Byte-range fetches]** Use byte-range fetches to download specific portions of an object in parallel, improving throughput and enabling retry of only the failed range. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/optimizing-performance-guidelines.html)
- **[Latency-sensitive retries]** Retry requests aggressively for latency-sensitive applications instead of waiting on a single slow request. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/optimizing-performance-guidelines.html)
- **[Co-location]** Combine Amazon S3 storage and Amazon EC2 compute in the same AWS Region to minimize latency and avoid cross-Region data transfer. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/optimizing-performance-guidelines.html)
- **[Transfer Acceleration]** Use S3 Transfer Acceleration to speed up data transport over long geographic distances between clients and a bucket. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/optimizing-performance-guidelines.html)
- **[SDK version]** Use the latest version of the AWS SDKs to benefit from up-to-date retry logic, connection handling, and performance improvements. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/optimizing-performance-guidelines.html)
- **[High-performance tier]** Use S3 Express One Zone for applications that need single-digit millisecond latencies and hundreds of thousands of requests per second. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/optimizing-performance-design-patterns.html)
- **[Monitoring]** Monitor request rates and 5xx error metrics in CloudWatch to detect when an application is approaching current scaling limits. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/optimizing-performance-design-patterns.html)

## 💰 Cost Optimization
- **[Storage class selection]** Match each dataset to the most cost-effective storage class (S3 Standard, Standard-IA, One Zone-IA, Glacier Instant Retrieval, Glacier Flexible Retrieval, or Glacier Deep Archive) based on its access pattern. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/cost-optimization.html)
- **[Unknown access patterns]** Use S3 Intelligent-Tiering for data with unknown, changing, or unpredictable access patterns so objects move automatically between access tiers without operational overhead. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/cost-optimization.html)
- **[Lifecycle automation]** Configure S3 Lifecycle rules to automatically transition objects to lower-cost storage classes or expire them after a defined period, based on prefix, tag, or object size. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/cost-optimization.html)
- **[Granular lifecycle filtering]** Use S3 Batch Operations to tag objects (for example, by file type) so lifecycle rules can apply granular, custom transition criteria beyond native prefix/tag/size filters. — [doc](https://aws.amazon.com/blogs/storage/transition-data-to-cheaper-storage-based-on-custom-filtering-criteria-with-amazon-s3-lifecycle/)
- **[Analysis]** Use S3 Storage Class Analysis, cost allocation tags, and billing/usage reports to analyze cost and usage patterns before choosing lifecycle transitions. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/cost-optimization.html)
- **[KMS cost reduction]** Enable S3 Bucket Keys when using SSE-KMS to reduce the volume of billed AWS KMS requests. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/bucket-encryption.html)

## ⚙️ Operational Excellence
- **[Monitoring strategy]** Define a monitoring plan (goals, resources, frequency, tools, and ownership) before implementing S3 monitoring, since it underpins reliability, availability, and performance. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/monitoring-overview.html)
- **[Metrics and alarms]** Use CloudWatch metrics and alarms for Amazon S3 to track operational health and trigger notifications when thresholds such as error rates are exceeded. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/cloudwatch-monitoring.html)
- **[API auditing]** Use AWS CloudTrail to record bucket-level and object-level API activity, including who made a request, when, and from where. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/cloudtrail-logging.html)
- **[Logging strategy]** Choose CloudTrail for bucket/object-level action logging with cross-account delivery and forwarding to CloudWatch, and use S3 server access logs for HTTP-level detail such as object size and turnaround time. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/logging-with-S3.html)
- **[Event-driven automation]** Configure S3 Event Notifications to trigger Lambda functions or other workflows automatically in response to object changes. — [doc](https://docs.aws.amazon.com/lambda/latest/dg/with-s3.html)
- **[Centralized visibility]** Use S3 Storage Lens to bring visibility into storage usage, activity trends, and data protection posture across accounts and buckets into a single dashboard. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/storage_lens.html)
- **[Best-practice checks]** Use AWS Trusted Advisor to evaluate S3 resources against best-practice checks for security, performance, and cost, and act on the recommendations. — [doc](https://aws.amazon.com/s3/security/)
