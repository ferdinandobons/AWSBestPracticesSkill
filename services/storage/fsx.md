# Amazon FSx — Best Practices

## Common scenarios
- Fully managed Windows file shares for enterprise and SMB workloads        → Security, Reliability, Performance Efficiency
- High-performance parallel file storage for HPC, ML, and media processing        → Performance Efficiency, Cost Optimization
- Shared storage for departmental file shares needing Active Directory integration        → Security, Operational Excellence
- Data protection and disaster recovery for business-critical file data        → Reliability

## 🔒 Security
- **[Identity]** Grant least-privilege IAM permissions to any principal or resource that must integrate with or manage FSx file systems. — [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/well-architected-best-practices-fsx-windows/security.html)
- **[Access control]** Use Windows ACLs for fine-grained share-, file-, and folder-level access control, and migrate existing ACL configurations without modification. — [doc](https://docs.aws.amazon.com/fsx/latest/WindowsGuide/windows-best-practices.html)
- **[Network security]** Don't modify or delete the elastic network interface (ENI) associated with your file system; doing so can permanently disconnect your VPC from the file system. — [doc](https://docs.aws.amazon.com/fsx/latest/WindowsGuide/windows-best-practices.html)
- **[Network security]** Use VPC security groups and network ACLs to restrict access to your file systems, ensuring required ports remain open for clients and, if applicable, Active Directory. — [doc](https://docs.aws.amazon.com/fsx/latest/WindowsGuide/windows-best-practices.html)
- **[Active Directory]** When using a self-managed Active Directory, configure the FSx service account with only the minimum privileges required, and delegate rather than grant broad admin rights. — [doc](https://docs.aws.amazon.com/fsx/latest/WindowsGuide/windows-best-practices.html)
- **[Active Directory]** Use a separate Organizational Unit (OU) for FSx file systems, distinct from other OUs in your Active Directory. — [doc](https://docs.aws.amazon.com/fsx/latest/WindowsGuide/windows-best-practices.html)
- **[Active Directory]** Store Active Directory domain-join service account credentials in AWS Secrets Manager rather than in plaintext application code or configuration files. — [doc](https://docs.aws.amazon.com/fsx/latest/WindowsGuide/windows-best-practices.html)
- **[Active Directory]** Never move or modify the computer objects that Amazon FSx creates and manages in Active Directory, and never change NTFS ACL permissions for the SYSTEM user — either can make the file system inaccessible or misconfigured. — [doc](https://docs.aws.amazon.com/fsx/latest/WindowsGuide/windows-best-practices.html)
- **[Traceability]** Enable file access auditing on FSx for Windows File Server file systems to track access and failed logon attempts. — [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/well-architected-best-practices-fsx-windows/security.html)
- **[Traceability]** Enable AWS CloudTrail to log FSx API calls for administrative activity monitoring. — [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/well-architected-best-practices-fsx-windows/security.html)
- **[Traceability]** Use the Shared Folders tool to monitor connected user sessions and open files on FSx for Windows File Server. — [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/well-architected-best-practices-fsx-windows/security.html)
- **[Encryption in transit]** Limit access to clients that support SMB protocol version 3.0 or higher to enforce in-transit encryption on FSx for Windows File Server. — [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/well-architected-best-practices-fsx-windows/security.html)
- **[Encryption at rest]** Rely on default encryption at rest using AWS KMS keys for both file systems and backups, and use a customer-managed key when you need control over key policies. — [doc](https://docs.aws.amazon.com/fsx/latest/WindowsGuide/what-is.html)
- **[Encryption]** For FSx for Lustre persistent file systems, specify a customer-managed KMS key for data-at-rest encryption when you need more control than the AWS-managed key provides. — [doc](https://docs.aws.amazon.com/fsx/latest/LustreGuide/encryption-at-rest.html)
- **[Automated response]** Automate incident response to security events (for example, unauthorized access attempts) using CloudWatch, EventBridge, Lambda, or Systems Manager. — [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/well-architected-best-practices-fsx-windows/security.html)

## 🛡️ Reliability
- **[High availability]** Use Multi-AZ file systems for production workloads that require high availability; Single-AZ is appropriate only for workloads that can tolerate an AZ-level interruption. — [doc](https://docs.aws.amazon.com/fsx/latest/WindowsGuide/windows-best-practices.html)
- **[Capacity planning]** Ensure IP subnet allocation accounts for future expansion and availability needs before deploying file systems. — [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/well-architected-best-practices-fsx-windows/reliability.html)
- **[Disaster recovery]** Replicate file system data across AWS Regions for cross-Region business continuity. — [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/well-architected-best-practices-fsx-windows/reliability.html)
- **[File-level recovery]** Enable Windows shadow copies so end users can view and restore individual files or folders from earlier versions without administrator intervention. — [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/well-architected-best-practices-fsx-windows/reliability.html)
- **[Backup]** Use the automated daily backups and take additional on-demand backups at key points; back up to AWS Backup for centralized, policy-driven backup plans across accounts. — [doc](https://docs.aws.amazon.com/fsx/latest/WindowsGuide/using-backups.html)
- **[Backup]** Copy backups across AWS Regions and accounts to protect against accidental activity, account compromise, and ransomware, and to meet compliance requirements. — [doc](https://aws.amazon.com/blogs/storage/cross-region-and-cross-account-backups-for-amazon-fsx-using-aws-backup/)
- **[Recovery testing]** Regularly practice restoring file systems from backups, and test Multi-AZ failover (for example by modifying throughput capacity) to validate recovery procedures. — [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/well-architected-best-practices-fsx-windows/reliability.html)
- **[Active Directory health]** Keep the Active Directory configuration current with Amazon FSx (for example after rotating the service account password), and set up notifications for a "Misconfigured" file system status. — [doc](https://docs.aws.amazon.com/fsx/latest/WindowsGuide/windows-best-practices.html)
- **[Active Directory health]** Run the Amazon FSx Active Directory validation tool proactively and on an ongoing basis to detect configuration issues before they cause unavailability. — [doc](https://docs.aws.amazon.com/fsx/latest/WindowsGuide/windows-best-practices.html)
- **[Elastic capacity]** Automatically scale storage and throughput capacity based on utilization metrics instead of statically over-provisioning for peak demand. — [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/well-architected-best-practices-fsx-windows/reliability.html)
- **[Quotas]** Monitor FSx service quotas and maintain sufficient headroom between current usage and account limits to accommodate failover events. — [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/well-architected-best-practices-fsx-windows/reliability.html)
- **[Change management]** Time throughput capacity modifications for idle periods, since updates interrupt Single-AZ availability briefly and trigger failover/failback on Multi-AZ file systems. — [doc](https://docs.aws.amazon.com/fsx/latest/WindowsGuide/windows-best-practices.html)

## ⚡ Performance Efficiency
- **[Monitoring]** Build a monitoring plan using file system CloudWatch metrics to understand storage and performance usage patterns and get ahead of capacity limits. — [doc](https://docs.aws.amazon.com/fsx/latest/WindowsGuide/windows-best-practices.html)
- **[Right-sizing]** Provision sufficient throughput capacity to cover both expected workload traffic and any enabled features (for example, data deduplication needs enough memory, and shadow copies need throughput at least 3x the driven workload to avoid Windows deleting them). — [doc](https://docs.aws.amazon.com/fsx/latest/WindowsGuide/windows-best-practices.html)
- **[Storage headroom]** Maintain at least 20% free storage capacity at all times, and increase throughput capacity by at least 20% before a storage capacity increase to offset performance impact. — [doc](https://docs.aws.amazon.com/fsx/latest/WindowsGuide/windows-best-practices.html)
- **[Scaling order]** Increase throughput capacity before increasing storage capacity, then scale throughput back down afterward, to minimize scaling duration and avoid a temporary write performance reduction (particularly for HDD storage or many-small-file workloads). — [doc](https://docs.aws.amazon.com/fsx/latest/WindowsGuide/windows-best-practices.html)
- **[Horizontal scale-out]** Use Microsoft DFS Namespaces to scale out performance across multiple FSx for Windows File Server file systems in the same namespace, up to tens of Gbps and millions of IOPS. — [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/well-architected-best-practices-fsx-windows/performance.html)
- **[Lustre I/O size]** Use larger average I/O sizes on FSx for Lustre to amortize the per-operation network round-trip latency and increase overall throughput. — [doc](https://docs.aws.amazon.com/fsx/latest/LustreGuide/performance-tips.html)
- **[Lustre write mode]** Enable asynchronous writes on FSx for Lustre for lower latency, understanding the consistency tradeoff versus synchronous writes when multiple EC2 instances are involved. — [doc](https://docs.aws.amazon.com/fsx/latest/LustreGuide/performance-tips.html)
- **[Lustre metadata]** Limit each directory to fewer than 100K files on Persistent 2 FSx for Lustre file systems to keep metadata lock acquisition fast. — [doc](https://docs.aws.amazon.com/fsx/latest/LustreGuide/performance-tips.html)
- **[Lustre compute sizing]** Choose EC2 instance types with enough memory and compute for read/write-intensive workloads accessing FSx for Lustre. — [doc](https://docs.aws.amazon.com/fsx/latest/LustreGuide/performance-tips.html)

## 💰 Cost Optimization
- **[Data reduction]** Enable data deduplication and compression to reduce redundant data storage instead of over-provisioning capacity for uncompressed data. — [doc](https://docs.aws.amazon.com/fsx/latest/WindowsGuide/managing-storage-configuration.html)
- **[Deduplication upkeep]** Schedule deduplication garbage-collection jobs after optimization runs, since space isn't reclaimed until garbage collection executes. — [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/optimize-costs-microsoft-workloads/storage-fsx-deduplication.html)
- **[Right-sizing]** Baseline current performance needs (for example with Windows Performance Monitor counters) before sizing throughput and storage capacity, rather than over-provisioning by default. — [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/optimize-costs-microsoft-workloads/storage-fsx-smb.html)
- **[Storage type selection]** Match storage type to workload needs — use HDD for broad, latency-tolerant workloads (home directories, departmental shares) and SSD for latency-sensitive workloads (databases, media processing) — instead of defaulting to SSD everywhere. — [doc](https://docs.aws.amazon.com/fsx/latest/WindowsGuide/managing-storage-configuration.html)
- **[Data sharding]** For large datasets, consider multiple smaller, purpose-matched file systems (mixing HDD and SSD) instead of one large SSD file system, using DFS Namespaces to unify them; this can materially reduce cost for the same aggregate capacity. — [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/optimize-costs-microsoft-workloads/storage-fsx-data-sharding.html)
- **[Storage capacity]** Be conservative when sizing storage capacity since it can only be scaled up, while throughput capacity can be scaled both up and down as demand changes. — [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/optimize-costs-microsoft-workloads/storage-fsx-smb.html)
- **[Quotas]** Use user-level and folder-level storage quotas to monitor and control consumption, supporting cost allocation and preventing runaway growth. — [doc](https://docs.aws.amazon.com/fsx/latest/WindowsGuide/managing-user-quotas.html)
- **[Migration sizing]** When migrating from on-premises Windows file servers using Windows Deduplication, size the new file system based on actual deduplicated data plus a buffer rather than raw source capacity. — [doc](https://aws.amazon.com/blogs/storage/migration-to-amazon-fsx-for-windows-file-server-cost-optimization-strategies-and-automation/)

## ⚙️ Operational Excellence
- **[Infrastructure as code]** Deploy and manage FSx file systems using IaC so configuration is repeatable and auditable. — [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/well-architected-best-practices-fsx-windows/operational-excellence.html)
- **[Automation]** Automate routine operational procedures, such as turning on user storage quotas in Track mode, enabling data deduplication, and upgrading from Single-AZ to Multi-AZ. — [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/well-architected-best-practices-fsx-windows/operational-excellence.html)
- **[CI/CD]** Store IaC templates in source control and deploy changes through a CI/CD pipeline that tests in a non-production environment before touching production file systems. — [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/well-architected-best-practices-fsx-windows/operational-excellence.html)
- **[Pre-production testing]** Test infrastructure and configuration changes in an environment that mirrors production (same Active Directory, network configuration, file system size, and enabled features) before deploying to production. — [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/well-architected-best-practices-fsx-windows/operational-excellence.html)
- **[Monitoring]** Set up notifications on file system health and Active Directory connectivity validation failures so issues surface proactively rather than after user impact. — [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/well-architected-best-practices-fsx-windows/operational-excellence.html)
- **[Access management]** Mount file shares for the correct set of users through group policies rather than manual, ad hoc configuration. — [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/well-architected-best-practices-fsx-windows/operational-excellence.html)
- **[Lustre monitoring]** Monitor FSx for Lustre client-side metrics (Lustre client, ENA driver, error logs) in addition to AWS-managed file system monitoring to catch workload-side performance issues. — [doc](https://aws.amazon.com/blogs/storage/best-practices-for-monitoring-amazon-fsx-for-lustre-clients-and-file-systems/)
