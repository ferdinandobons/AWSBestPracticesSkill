# AWS Storage Gateway — Best Practices

## Common scenarios
- Extending on-premises file shares into Amazon S3 via NFS/SMB        → Cost Optimization, Performance Efficiency
- Replacing physical tape backup infrastructure with virtual tapes        → Cost Optimization, Reliability
- Providing cloud-backed iSCSI block storage for on-premises servers        → Reliability, Performance Efficiency
- Disaster recovery and data mirroring using EC2-hosted gateways        → Reliability, Security

## 🔒 Security
- **[data in transit]** Require clients to use TLS 1.2 or later (TLS 1.3 recommended) with cipher suites supporting perfect forward secrecy (DHE/ECDHE) when communicating with Storage Gateway. — protects data and credentials from interception on the network. [doc](https://docs.aws.amazon.com/filegateway/latest/files3/infrastructure-security.html)
- **[authentication]** Sign all API requests with credentials tied to an IAM principal, or use AWS STS temporary security credentials instead of long-lived keys. — reduces exposure from leaked long-term credentials. [doc](https://docs.aws.amazon.com/filegateway/latest/files3/infrastructure-security.html)
- **[access control]** Set up individual IAM users or IAM Identity Center identities with MFA rather than sharing root/account credentials. — limits blast radius and enforces accountability per user. [doc](https://docs.aws.amazon.com/storagegateway/latest/vgw/data-protection.html)
- **[access control]** Start from AWS managed policies and progressively narrow to least-privilege customer-managed policies for Storage Gateway actions. — avoids over-permissioned roles while still enabling quick onboarding. [doc](https://docs.aws.amazon.com/storagegateway/latest/tgw/security_iam_id-based-policy-examples.html)
- **[access control]** Add IAM policy conditions (for example, requiring SSL/TLS on requests) to further restrict when and how Storage Gateway actions can be invoked. — narrows the situations in which permissions can be exercised. [doc](https://docs.aws.amazon.com/storagegateway/latest/tgw/security_iam_id-based-policy-examples.html)
- **[access control]** Validate IAM policies with IAM Access Analyzer before deploying them. — catches overly permissive or malformed policies before they take effect. [doc](https://docs.aws.amazon.com/storagegateway/latest/tgw/security_iam_id-based-policy-examples.html)
- **[appliance hardening]** Treat the Storage Gateway virtual appliance as a managed VM: do not install scanning software or patch it outside the standard gateway update mechanism. — out-of-band changes can break the appliance and forfeit AWS support. [doc](https://docs.aws.amazon.com/filegateway/latest/files3/infrastructure-security.html)
- **[logging]** Enable AWS CloudTrail to record API and user activity against Storage Gateway resources. — provides an audit trail for detecting and investigating unauthorized actions. [doc](https://docs.aws.amazon.com/storagegateway/latest/vgw/data-protection.html)
- **[data protection]** Avoid placing confidential or sensitive information in tags or free-form name fields for gateways, volumes, or file shares. — such fields can surface in billing and diagnostic logs. [doc](https://docs.aws.amazon.com/storagegateway/latest/vgw/data-protection.html)
- **[compliance]** Use FIPS endpoints when FIPS 140-3 validated cryptographic modules are required for CLI/API access. — meets regulatory requirements for regulated workloads. [doc](https://docs.aws.amazon.com/storagegateway/latest/vgw/data-protection.html)

## 🛡️ Reliability
- **[high availability]** Deploy gateways in a VMware environment with VMware vSphere High Availability (HA) integration so the gateway automatically recovers from most service interruptions in under 60 seconds. — protects against hardware, hypervisor, network, and software failures. [doc](https://aws.amazon.com/storagegateway/features/)
- **[storage layout]** Provision cache and upload buffer disks on separate underlying physical disks rather than the same data store. — a shared physical disk backing both roles is a common source of poor performance and failure correlation. [doc](https://docs.aws.amazon.com/storagegateway/latest/vgw/Performance.html)
- **[storage layout]** Use striped RAID (for example RAID 10) with a hardware RAID controller for cache and upload buffer disks instead of low-redundancy levels like RAID 1 or RAID 6. — improves both throughput and fault tolerance for locally staged data. [doc](https://docs.aws.amazon.com/storagegateway/latest/vgw/Performance.html)
- **[monitoring]** Enable the recommended CloudWatch alarms (CachePercentDirty, IOWaitPercent, FileSharesUnavailable, health notifications) during gateway activation. — surfaces cache pressure and availability problems before they cause client-facing failures. [doc](https://aws.amazon.com/blogs/storage/monitoring-aws-storage-gateway-health-and-performance-using-amazon-cloudwatch/)
- **[monitoring]** Subscribe CloudWatch alarms to Amazon SNS for notification, or trigger Lambda functions for automated remediation. — shortens time-to-detection and response for gateway health issues. [doc](https://aws.amazon.com/blogs/storage/aws-storage-gateway-provides-simplified-monitoring-for-file-gateway/)
- **[backup]** Use AWS Backup to manage backup and recovery of Volume Gateway volumes. — centralizes and simplifies meeting business and regulatory backup requirements. [doc](https://aws.amazon.com/storagegateway/features/)
- **[disaster recovery]** Consider deploying gateways on Amazon EC2 for disaster recovery and data mirroring scenarios. — provides a cloud-hosted recovery target independent of the on-premises environment. [doc](https://docs.aws.amazon.com/storagegateway/latest/vgw/WhatIsStorageGateway.html)

## ⚡ Performance Efficiency
- **[sizing]** Provision at least 24 dedicated physical CPU cores and RAM scaled to cache size (16 GiB up to 16 TiB cache, up to 48 GiB for 32–64 TiB cache) on the gateway host. — undersized compute/memory is a primary cause of throughput bottlenecks. [doc](https://docs.aws.amazon.com/storagegateway/latest/vgw/Performance.html)
- **[disks]** Back the gateway cache and upload buffer with SSDs behind an NVMe controller rather than standard disks. — disk throughput is a common limiting factor for both upload and download performance. [doc](https://docs.aws.amazon.com/storagegateway/latest/vgw/Performance.html)
- **[disks]** Add multiple upload buffer disks to reduce concurrent I/O contention on any single disk. — increases achievable write throughput to the gateway. [doc](https://docs.aws.amazon.com/storagegateway/latest/vgw/Performance.html)
- **[networking]** Dedicate separate network adapters/VM networks for client ingestion traffic and for gateway-to-AWS traffic, each at 10 Gbps where possible. — avoids contention between local application traffic and cloud upload/download traffic. [doc](https://docs.aws.amazon.com/storagegateway/latest/vgw/Performance.html)
- **[workload isolation]** Place high-throughput applications on a dedicated gateway rather than mixing them with low-throughput workloads on a shared gateway. — prevents one workload's demand from degrading another's observed performance. [doc](https://docs.aws.amazon.com/storagegateway/latest/vgw/Performance.html)
- **[iSCSI tuning]** Tune iSCSI initiator settings, using 256 KiB for MaxReceiveDataSegmentLength and FirstBurstLength and 1 MiB for MaxBurstLength. — aligns initiator behavior with gateway I/O characteristics for higher throughput. [doc](https://docs.aws.amazon.com/storagegateway/latest/vgw/Performance.html)
- **[capacity planning]** Monitor CPU, RAM, network bandwidth (to AWS and from initiator to gateway), and disk throughput as the five key bottleneck dimensions, using CloudWatch ReadBytes/WriteBytes metrics to spot degradation trends. — targeted metric review pinpoints which resource to scale instead of over-provisioning blindly. [doc](https://docs.aws.amazon.com/storagegateway/latest/vgw/Performance.html)

## 💰 Cost Optimization
- **[S3 storage tiering]** Apply S3 Lifecycle policies to transition or expire objects written by an S3 File Gateway, and enable periodic cache refresh (or call RefreshCache) so NFS/SMB clients see the resulting changes. — moves infrequently accessed data to cheaper storage classes without breaking client visibility. [doc](https://aws.amazon.com/storagegateway/faqs/)
- **[object versioning]** Before enabling S3 Versioning on a File Gateway bucket, understand that each file write can create multiple object versions (data upload plus metadata update), and configure lifecycle expiration for noncurrent versions. — unmanaged versioning on File Gateway buckets can significantly inflate S3 storage consumption. [doc](https://docs.aws.amazon.com/filegateway/latest/files3/troubleshooting-file-share-issues.html)
- **[replication]** Measure storage growth from versioning before enabling Same-Region or Cross-Region Replication on a File Gateway bucket. — replication multiplies the extra storage caused by versioning if enabled prematurely. [doc](https://docs.aws.amazon.com/filegateway/latest/files3/troubleshooting-file-share-issues.html)
- **[network cost]** Use schedule-based bandwidth throttling to cap gateway-to-AWS transfer rates during peak business hours and relax them off-peak. — controls network cost and contention without manual daily reconfiguration. [doc](https://aws.amazon.com/blogs/storage/aws-storage-gateway-increases-cache-4x-and-enhances-bandwidth-throttling/)

## ⚙️ Operational Excellence
- **[monitoring]** Use the Storage Gateway console's consolidated CloudWatch view to track cache status, disk performance, and health notifications alongside gateway metrics. — gives a single place to correlate gateway health signals instead of checking multiple dashboards. [doc](https://aws.amazon.com/blogs/storage/aws-storage-gateway-provides-simplified-monitoring-for-file-gateway/)
- **[maintenance]** Apply gateway software updates only through the normal scheduled maintenance window mechanism rather than ad hoc patching. — AWS ships CVE fixes through this channel, and out-of-band changes can cause malfunction or unsupported states. [doc](https://docs.aws.amazon.com/filegateway/latest/files3/infrastructure-security.html)
- **[permissions]** Grant console users only the minimum permissions needed to list and view Storage Gateway resources, and scope CLI/API users to the specific actions they call. — an overly restrictive console policy breaks the UI, while an overly broad one increases risk. [doc](https://docs.aws.amazon.com/storagegateway/latest/tgw/security_iam_id-based-policy-examples.html)
