# AWS Snow Family — Best Practices

## Common scenarios
- Petabyte-scale offline migration of on-premises data into Amazon S3        → Performance Efficiency, Cost Optimization
- Capturing and transferring data from disconnected or remote/edge sites        → Reliability, Security
- Running local compute, storage, and ML inference in disconnected environments        → Reliability, Performance Efficiency
- Physically shipping sensitive data to AWS instead of using network transfer        → Security, Cost Optimization

## 🔒 Security
- **[device handling]** Inspect a device for damage or signs of tampering as soon as it arrives, and don't connect it to your internal network if anything looks suspicious — contact AWS Support for a replacement instead. — a tampered device connected to your network can expose that network to compromise. [doc](https://docs.aws.amazon.com/snowball/latest/developer-guide/BestPractices.html)
- **[credential handling]** Store the unlock code and the job manifest in different locations (for example, manifest on a local server, code sent by email to an authorized user) rather than together on the same workstation. — limits the ability of any single unauthorized party to unlock and access the device's contents. [doc](https://docs.aws.amazon.com/snowball/latest/developer-guide/BestPractices.html)
- **[credential handling]** Treat the local access keys returned by `list-access-keys`/`get-secret-access-key` as job- and device-scoped only, and request a new manifest and unlock code if credentials are lost or suspected compromised. — these keys never map to your AWS account, so re-issuing them via the SSL certificate update process is the correct recovery path. [doc](https://docs.aws.amazon.com/snowball/latest/developer-guide/BestPractices.html)
- **[encryption]** Encrypt data using keys you manage through AWS KMS rather than relying solely on device defaults, and don't store sensitive identifiers (account numbers, PII) in free-form fields such as job names. — KMS-managed keys are never persisted on the device, and free-form fields can be captured in diagnostic logs. [doc](https://docs.aws.amazon.com/snowball/latest/developer-guide/data-protection.html)
- **[network security]** Use only one method at a time for reading and writing to a local bucket on a device, and avoid disconnecting the device or changing its network settings mid-transfer. — concurrent or interrupted access patterns are a common source of data corruption on Snow devices. [doc](https://docs.aws.amazon.com/snowball/latest/developer-guide/BestPractices.html)
- **[access control]** Start from AWS managed policies for Snow Family job management and progressively narrow to least-privilege customer-managed policies scoped to your actual use cases. — avoids leaving broad permissions in place after onboarding is complete. [doc](https://docs.aws.amazon.com/snowball/latest/developer-guide/security_iam_id-based-policy-examples.html)
- **[access control]** Add IAM policy conditions, such as requiring SSL/TLS on all requests, to further restrict when Snow Family actions can be invoked. — narrows the conditions under which granted permissions can actually be used. [doc](https://docs.aws.amazon.com/snowball/latest/developer-guide/security_iam_id-based-policy-examples.html)
- **[access control]** Validate identity-based policies with IAM Access Analyzer before applying them, and require MFA for any IAM users or root user involved in managing jobs. — catches overly permissive or malformed policies and reduces risk from credential theft. [doc](https://docs.aws.amazon.com/snowball/latest/developer-guide/security_iam_id-based-policy-examples.html)
- **[compliance]** Use FIPS-validated endpoints for CLI/API access when FIPS 140-2 validated cryptographic modules are required by policy. — meets regulatory requirements for regulated or government workloads. [doc](https://docs.aws.amazon.com/snowball/latest/developer-guide/data-protection.html)
- **[logging]** Turn on AWS CloudTrail to capture job management API calls, including requester identity, source IP, and timestamp. — provides an audit trail for security analysis and operational troubleshooting of Snow Family activity. [doc](https://docs.aws.amazon.com/snowball/latest/developer-guide/logging-using-cloudtrail.html)

## 🛡️ Reliability
- **[data verification]** For import jobs, keep local copies of transferred data until the import into Amazon S3 has been verified as successful. — protects against data loss if a transfer or import step fails silently. [doc](https://docs.aws.amazon.com/snowball/latest/developer-guide/BestPractices.html)
- **[job lifecycle]** Plan cancellations carefully: a job (or cluster job) can only be canceled while it is in the **Job created** status. — jobs that have progressed past this status cannot be stopped, so validate requirements before submission. [doc](https://docs.aws.amazon.com/snowball/latest/developer-guide/BestPractices.html)
- **[high availability]** For applications running on-device, replicate data and services across multiple devices in an active/passive or active/active configuration using a load balancer or DNS health checks, rather than relying on a single device. — mitigates the risk of losing access to critical local applications if a single device fails. [doc](https://aws.amazon.com/blogs/mt/disaster-recovery-dr-failover-to-the-disconnected-edge/)
- **[high availability]** Cluster databases and use S3-compatible storage clusters (minimum 3, up to 16 devices) across multiple devices so data is replicated and not dependent on one device's disks. — clustered S3-compatible storage on Snow is designed to tolerate the loss of one or two devices depending on cluster size. [doc](https://docs.aws.amazon.com/snowball/latest/developer-guide/s3compatible-on-snow.html)
- **[local site planning]** Plan for failure of supporting local infrastructure (switches, network links) between devices, staging workstations, and data sources, not just the devices themselves. — a single point of failure outside the device can still take down an otherwise redundant deployment. [doc](https://aws.amazon.com/blogs/mt/disaster-recovery-dr-failover-to-the-disconnected-edge/)
- **[job expiry]** Track the 360-day completion window for jobs and renew the SSL certificate before a device becomes locked if you need to retain it longer. — a locked device can no longer be accessed and must be returned, risking work in progress. [doc](https://docs.aws.amazon.com/snowball/latest/developer-guide/limits.html)

## ⚡ Performance Efficiency
- **[file sizing]** Batch files smaller than 1 MB into larger archives before transferring, and keep no more than 500,000 files or directories within a single directory. — small-file overhead from per-object encryption is the largest drag on transfer throughput. [doc](https://docs.aws.amazon.com/snowball/latest/developer-guide/BestPractices.html)
- **[parallelism]** Run multiple simultaneous write operations — multiple terminal sessions, multiple workstations, or multiple target devices — rather than a single sequential copy stream. — this has the single largest positive impact on transfer speed of all tuning options. [doc](https://docs.aws.amazon.com/snowball/latest/developer-guide/BestPractices.html)
- **[transfer hygiene]** Keep files in a static state during transfer; avoid renaming, changing metadata, or writing to files while they are being copied. — modifying files mid-transfer causes read/write conflicts that degrade performance and reliability. [doc](https://docs.aws.amazon.com/snowball/latest/developer-guide/BestPractices.html)
- **[network topology]** Reduce local network contention and eliminate unnecessary network hops by connecting the device, data source, and staging workstation to the same switch. — every intermediate hop and competing traffic flow reduces achievable throughput to the device. [doc](https://docs.aws.amazon.com/snowball/latest/developer-guide/BestPractices.html)
- **[calibration]** Calibrate expected transfer performance by copying a representative data subset first, then progressively add parallel copy operations until returns diminish, before committing to the full migration. — establishes a realistic target transfer rate and avoids under- or over-provisioning devices and workstations. [doc](https://docs.aws.amazon.com/snowball/latest/developer-guide/calibrating-large-transfer.html)
- **[source throughput]** Ensure on-premises source storage has aggregate read throughput at least equal to the sum of the target transfer rates across all devices being written to concurrently. — the source system is frequently the actual bottleneck, not the device or network. [doc](https://aws.amazon.com/blogs/storage/data-migration-best-practices-with-snowball-edge/)
- **[network sizing]** Use a network link of at least 10 Gbps with sub-millisecond latency and zero packet loss between the source, staging workstation, and device, ideally on the same subnet. — matches the device's high-speed interfaces and avoids latency-induced throughput loss. [doc](https://aws.amazon.com/blogs/storage/data-migration-best-practices-with-snowball-edge/)
- **[proof of concept]** Run a proof-of-concept transfer with one or two devices before a large multi-petabyte migration to validate network, tooling, and file-batching assumptions. — surfaces environment-specific gaps (bandwidth, workstation sizing, concurrent write limits) before they affect a production-scale migration. [doc](https://aws.amazon.com/blogs/storage/aws-reinvent-recap-optimize-your-data-migration-with-aws-snow-family/)

## 💰 Cost Optimization
- **[proof of concept]** Apply proof-of-concept learnings (optimal parallelism, file batching, network tuning) before ordering the full fleet of devices for a migration. — since device cost is driven by the number of days each device is retained, faster transfers from a tuned setup directly reduce cost. [doc](https://aws.amazon.com/blogs/storage/data-migration-best-practices-with-snowball-edge/)
- **[migration planning]** Work with your AWS account team and the Snow service team to plan orchestration for large multi-device migrations. — avoids extending device retention (and therefore cost) due to local network or tooling constraints discovered too late. [doc](https://aws.amazon.com/blogs/storage/aws-reinvent-recap-optimize-your-data-migration-with-aws-snow-family/)

## ⚙️ Operational Excellence
- **[monitoring]** Enable AWS CloudTrail across all Regions and deliver logs to a dedicated S3 bucket to maintain an ongoing record of Snow Family job management activity. — supports debugging multi-point failures and provides durable evidence for operational and security investigations. [doc](https://docs.aws.amazon.com/snowball/latest/developer-guide/logging-using-cloudtrail.html)
- **[migration process]** Follow a structured migration process — discovery, proof-of-concept, tooling/optimization, then production migration — rather than shipping devices before validating the environment. — a documented phased process surfaces network, tooling, and file-system issues while they are cheap to fix. [doc](https://aws.amazon.com/blogs/storage/data-migration-best-practices-with-snowball-edge/)
- **[service selection]** For new projects, evaluate AWS DataSync for online transfers, AWS Data Transfer Terminal for physical transfers, or AWS Outposts for edge computing, since AWS Snowball Edge is no longer available to new customers. — ensures new workloads are built on currently supported services rather than a service in wind-down. [doc](https://docs.aws.amazon.com/snowball/latest/developer-guide/BestPractices.html)
