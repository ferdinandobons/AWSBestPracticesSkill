# Amazon S3 Glacier — Best Practices

## Common scenarios
- Long-term compliance and regulatory archives (financial records, medical images)        → Security, Reliability, Cost Optimization
- Backup and disaster-recovery data with infrequent restores        → Reliability, Cost Optimization
- Digital preservation / media archives accessed only a few times a year        → Cost Optimization, Performance Efficiency
- Migrating cold data out of on-premises WORM or tape systems        → Security, Operational Excellence

## 🔒 Security
- **[compliance retention]** Enable S3 Object Lock in Compliance mode (with S3 Versioning) on archive buckets so locked object versions can't be deleted or overwritten by any user, including the root account, for the retention period. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/object-lock.html)
- **[flexible retention]** Use Object Lock Governance mode plus the `s3:BypassGovernanceRetention` permission when you need to test retention settings or allow limited administrative overrides. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/object-lock.html)
- **[legacy vaults]** For accounts still on the original standalone Amazon Glacier vault API, lock a Vault Lock policy only after testing it in the in-progress state, since a locked policy becomes immutable. — [doc](https://docs.aws.amazon.com/amazonglacier/latest/dev/vault-lock-policy.html)
- **[encryption at rest]** Rely on server-side encryption (SSE-S3 by default, or SSE-KMS with a customer managed key) for objects archived to S3 Glacier storage classes to get encryption plus a KMS audit trail. — [doc](https://aws.amazon.com/s3/faqs/)
- **[auditability]** Enable AWS CloudTrail data events on archive buckets to log and retain S3 Glacier storage API activity for audits and compliance certifications (SEC 17a-4, HIPAA, FedRAMP, PCI-DSS). — [doc](https://aws.amazon.com/s3/storage-classes/glacier/)
- **[log integrity]** When CloudTrail logs land in an S3 Glacier-tiered bucket, protect them with S3 Object Lock Compliance mode rather than relying solely on MFA Delete for tamper-proof retention. — [doc](https://aws.amazon.com/blogs/storage/amazon-s3-audit-logging-part-2-centralized-logging-and-analysis-of-s3-data-events-in-aws-cloudtrail-for-security-and-compliance/)

## 🛡️ Reliability
- **[durability]** Design around the 11 nines (99.999999999%) durability of S3 Glacier storage classes, with data redundantly stored across multiple Availability Zones in a Region. — [doc](https://aws.amazon.com/s3/storage-classes/glacier/)
- **[geographic redundancy]** Use S3 Replication (Cross-Region or Same-Region) to replicate archive objects directly into S3 Glacier Flexible Retrieval or S3 Glacier Deep Archive in a second bucket or Region to meet geographic-separation and disaster-recovery requirements. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/replication.html)
- **[restore-in-place]** Remember that restoring an archived object only creates a temporary accessible copy for the duration you specify; create a permanent copy in S3 Standard if you need durable, ongoing access. — [doc](https://docs.aws.amazon.com/sdk-for-kotlin/api/latest/s3/aws.sdk.kotlin.services.s3/restore-object.html)
- **[event-driven recovery]** Use Amazon S3 Event Notifications (e.g., restore-completed events) with SQS or Lambda to automatically detect and process objects once an asynchronous S3 Glacier restore finishes, avoiding broken downstream dependencies. — [doc](https://aws.amazon.com/blogs/storage/automating-retrievals-from-the-amazon-s3-glacier-storage-classes/)

## ⚡ Performance Efficiency
- **[retrieval tier choice]** Match the retrieval tier to urgency: Expedited (1–5 minutes, S3 Glacier Flexible Retrieval only) for occasional urgent needs, Standard (3–5 hours) as the general-purpose default, and Bulk (5–12 hours, up to 48 hours for Deep Archive) for large, non-urgent restores. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/restoring-objects-retrieval-options.html)
- **[provisioned capacity]** Purchase Provisioned Capacity for Expedited retrievals if you have recurring urgent-access needs, since it guarantees retrieval capacity is available when requested. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/restoring-objects-retrieval-options.html)
- **[bulk restores at scale]** Use S3 Batch Operations to issue restore requests for millions of objects in one job, which fully utilizes your account's restore-request rate and can cut restore time for large datasets by up to 90%. — [doc](https://aws.amazon.com/blogs/storage/restoring-archived-objects-at-scale-from-the-amazon-s3-glacier-storage-classes/)
- **[restore ordering]** When restoring large datasets, issue restore requests sequentially in the order objects were originally transitioned into S3 Glacier (often the creation-date order from S3 Inventory) rather than randomly, since S3 Glacier is designed for a limited number of random restore requests per pebibyte per day. — [doc](https://aws.amazon.com/blogs/storage/best-practices-for-archiving-large-datasets-with-aws/)
- **[large object retrieval]** Account for the fact that any single retrieval option delivers up to ~300 MB/s throughput per object, so objects larger than 5 TB (and datasets in the multi-petabyte range) need extra time regardless of tier chosen. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/restoring-objects-retrieval-options.html)
- **[object sizing]** Aggregate many small files into fewer, larger archives before transitioning to S3 Glacier storage classes, since each archived object carries roughly 40 KB of additional metadata overhead that erodes savings for small objects. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/archival-storage.html)

## 💰 Cost Optimization
- **[storage class selection]** Choose among S3 Glacier Instant Retrieval (millisecond access), S3 Glacier Flexible Retrieval (minutes–hours, cheaper), and S3 Glacier Deep Archive (12–48 hours, lowest cost) based on actual access frequency and latency tolerance rather than defaulting to one class. — [doc](https://aws.amazon.com/s3/storage-classes/glacier/)
- **[lifecycle automation]** Use S3 Lifecycle policies to automatically transition objects from S3 Standard/Standard-IA into S3 Glacier storage classes and to expire them afterward, rather than performing manual migrations. — [doc](https://aws.amazon.com/s3/faqs/)
- **[break-even analysis]** Before creating a Lifecycle transition rule, calculate the one-time per-object transition charge against the ongoing storage savings — transitioning very small, numerous objects can take years to break even, while larger objects break even in months. — [doc](https://aws.amazon.com/blogs/storage/identify-cold-objects-for-archiving-to-amazon-s3-glacier-storage-classes/)
- **[minimum storage duration]** Plan retention periods around the minimum storage duration charges — 90 days for S3 Glacier Flexible Retrieval and 180 days for S3 Glacier Deep Archive — since deleting or overwriting objects earlier incurs a pro-rated charge for the remaining minimum period. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/lifecycle-expire-general-considerations.html)
- **[bulk retrieval savings]** Prefer free or low-cost Bulk retrievals for planned, infrequent access patterns (e.g., annual audits) instead of paying premium rates for Expedited retrievals. — [doc](https://aws.amazon.com/solutions/guidance/automated-restore-and-copy-for-amazon-s3-glacier-objects/)
- **[intelligent tiering alternative]** For data with unpredictable or unknown access patterns, consider S3 Intelligent-Tiering's Archive and Deep Archive Access tiers instead of manually managed S3 Glacier lifecycle rules, since it automates the archival transition after 90/180 days without retrieval-fee surprises on access-pattern changes. — [doc](https://aws.amazon.com/blogs/storage/best-practices-for-archiving-large-datasets-with-aws/)

## ⚙️ Operational Excellence
- **[cost/data mapping]** Map your data's access patterns (frequency, latency tolerance) up front before selecting an S3 Glacier storage class or writing Lifecycle rules, instead of retrofitting after storage costs grow. — [doc](https://aws.amazon.com/blogs/storage/best-practices-for-archiving-large-datasets-with-aws/)
- **[async access design]** Build applications to handle the asynchronous nature of S3 Glacier Flexible Retrieval and Deep Archive access (restore-then-fetch) rather than assuming synchronous GET access, using event notifications or a GetObject wrapper to automate the restore-and-retry flow. — [doc](https://aws.amazon.com/blogs/storage/automating-retrievals-from-the-amazon-s3-glacier-storage-classes/)
- **[inventory-driven operations]** Use S3 Inventory reports to track object transition dates and drive orderly, at-scale restore or lifecycle operations instead of listing objects manually. — [doc](https://aws.amazon.com/blogs/storage/best-practices-for-archiving-large-datasets-with-aws/)
- **[throttling awareness]** Design bulk restore workflows for the account-level restore-request quota (1,000 transactions per second) and handle `ThrottlingException` responses gracefully, using S3 Batch Operations to stay within limits. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/restoring-objects-retrieval-options.html)
- **[monitoring locks]** Track access to and changes in Object Lock configurations using S3 Event Notifications with AWS CloudTrail, and raise CloudWatch alarms on unexpected retention or policy changes. — [doc](https://docs.aws.amazon.com/AmazonS3/latest/userguide/object-lock-managing.html)

## 🌱 Sustainability
- **[automatic expiration]** Apply S3 Lifecycle expiration rules so archived objects that are no longer needed are automatically deleted rather than retained indefinitely, reducing unnecessary long-term storage footprint. — [doc](https://aws.amazon.com/solutions/guidance/automated-restore-and-copy-for-amazon-s3-glacier-objects/)
- **[right-tiering]** Transition genuinely cold data into the lowest applicable S3 Glacier storage class (Flexible Retrieval or Deep Archive) instead of leaving it in warmer, more resource-intensive tiers, aligning storage footprint with actual access needs. — [doc](https://aws.amazon.com/blogs/storage/analyze-access-patterns-and-use-the-most-cost-effective-amazon-s3-storage-class/)
