# Amazon Bedrock AgentCore — Best Practices

## Common scenarios
- Deploying and scaling autonomous AI agents in secure, session-isolated runtimes        → Security, Reliability, Performance Efficiency
- Giving agents persistent memory and identity-aware access to AWS and third-party tools        → Security, Reliability
- Exposing APIs, Lambda functions, and MCP servers as agent-callable tools through a gateway        → Security, Performance Efficiency
- Operating agent fleets in production with observability, cost control, and continuous improvement        → Operational Excellence, Cost Optimization, Sustainability

## 🔒 Security
- **[IAM]** Do not use CLI-generated IAM policies in production — they grant broad access for development convenience; create custom least-privilege policies scoped to specific runtime ARNs instead of wildcard resources. [doc](https://docs.aws.amazon.com/bedrock-agentcore/latest/devguide/runtime-security-best-practices.html)
- **[IAM]** Explicitly deny `bedrock-agentcore:InvokeAgentRuntimeForUser` and `GetWorkloadAccessTokenForUserId` for workloads that always have a JWT available, so all user identification is forced through the cryptographically verified JWT path rather than an opaque, spoofable user-id string. [doc](https://docs.aws.amazon.com/bedrock-agentcore/latest/devguide/runtime-security-best-practices.html)
- **[IAM]** Use `bedrock-agentcore:subnets` and `bedrock-agentcore:securityGroups` IAM condition keys to require that all runtimes are deployed into approved VPCs, and run IAM Access Analyzer against these policies to validate least-privilege. [doc](https://docs.aws.amazon.com/bedrock-agentcore/latest/devguide/runtime-security-best-practices.html)
- **[confused deputy]** Add `aws:SourceArn` and `aws:SourceAccount` conditions to execution-role trust policies so only your own AgentCore resources can assume the role — prevents cross-account confused-deputy attacks against the runtime execution role. [doc](https://docs.aws.amazon.com/bedrock-agentcore/latest/devguide/runtime-security-best-practices.html)
- **[gateway]** Front AgentCore Runtime with an AgentCore Gateway and restrict the runtime to accept invocations only from the gateway (resource-based policy for IAM/SigV4, or `allowedWorkloadConfiguration` for OAuth/JWT) — policy enforcement, guardrails, and interceptors only protect you if callers cannot bypass the gateway and reach the runtime directly. [doc](https://docs.aws.amazon.com/bedrock-agentcore/latest/devguide/runtime-security-best-practices.html)
- **[authentication]** Prefer the JWT bearer token path (`GetWorkloadAccessTokenForJWT`) over the UserId path for production, since JWT validates issuer, signature, and expiry, while the UserId header treats the identifier as an unverified opaque string. [doc](https://docs.aws.amazon.com/bedrock-agentcore/latest/devguide/runtime-security-best-practices.html)
- **[credentials]** Use AgentCore Identity for all outbound authentication to third-party services (Slack, GitHub, Zoom) — it stores OAuth tokens, refresh tokens, and API keys in an encrypted token vault instead of leaving credentials exposed in agent code or logs. [doc](https://aws.amazon.com/blogs/security/securing-ai-agents-with-amazon-bedrock-agentcore-identity/)
- **[credentials]** Apply the principle of least privilege to agent identities: grant only the minimum permissions an agent needs, regularly audit and review agent permissions, and separate user-delegated credentials (Authorization Code Grant) from autonomous credentials (Client Credentials Grant). [doc](https://aws.amazon.com/blogs/machine-learning/introducing-amazon-bedrock-agentcore-identity-securing-agentic-ai-at-scale/)
- **[containers]** Configure custom container images to run as a non-root user, and be aware that any code running inside the microVM can read execution-role credentials from the metadata endpoint (MMDS) — scope execution-role permissions accordingly. [doc](https://docs.aws.amazon.com/bedrock-agentcore/latest/devguide/runtime-security-best-practices.html)
- **[network]** Deploy runtimes inside a VPC with private subnets and NAT gateways, use AWS PrivateLink interface VPC endpoints for the AgentCore data and control planes, and scope S3 gateway-endpoint policies to only the ECR layer or code-artifact buckets required. [doc](https://docs.aws.amazon.com/bedrock-agentcore/latest/devguide/runtime-security-best-practices.html)
- **[memory]** Design a hierarchical memory namespace schema (scoped by `{actorId}`/`{sessionId}`) and scope IAM roles to only the namespace prefixes an agent requires — prevents one tenant's agent from reading or corrupting another tenant's long-term memory. [doc](https://docs.aws.amazon.com/wellarchitected/latest/agentic-ai-lens/agentsec01-bp01.html)
- **[memory]** Prevent memory poisoning by never letting users control their own session ID or actor ID, and never embed `ActorID`/`SessionID` values in system prompts where a user could manipulate them; sanitize conversational input with guardrails before it is persisted to memory. [doc](https://docs.aws.amazon.com/bedrock-agentcore/latest/devguide/best-practices.html)
- **[memory]** Encrypt AgentCore Memory, Identity token vaults, Gateway configuration, and related CloudWatch log groups with a customer-managed AWS KMS key for sensitive workloads, rather than relying solely on the AWS-owned default key. [doc](https://docs.aws.amazon.com/bedrock-agentcore/latest/devguide/storage-encryption.html)
- **[command execution]** Restrict which principals can call `InvokeAgentRuntimeCommand` and `InvokeAgentRuntimeCommandShell`, route deterministic operations (tests, git, builds) through these command APIs instead of through the LLM, and set command timeouts to prevent resource waste from runaway processes. [doc](https://docs.aws.amazon.com/bedrock-agentcore/latest/devguide/runtime-security-best-practices.html)
- **[platform server]** Restrict agent code and tools from making unrestricted HTTP calls to the microVM's localhost platform server, and allowlist only the specific ports required for sidecar or container-in-container patterns. [doc](https://docs.aws.amazon.com/bedrock-agentcore/latest/devguide/runtime-security-best-practices.html)
- **[auditing]** Enable AWS CloudTrail for `InvokeAgentRuntime`/control-plane API calls and correlate them with CloudWatch Logs request IDs to build a complete audit trail of who invoked an agent and what commands executed. [doc](https://docs.aws.amazon.com/bedrock-agentcore/latest/devguide/runtime-security-best-practices.html)

## 🛡️ Reliability
- **[memory failover]** Treat AgentCore Memory as a fault-tolerant managed store by default, but add fail-fast logic that routes to an in-process fallback cache for short-term memory and a read-through cache for long-term memory when the store shows elevated error rates. [doc](https://docs.aws.amazon.com/wellarchitected/latest/agentic-ai-lens/agentrel03-bp02.html)
- **[chaos testing]** Use AWS Fault Injection Service to simulate memory-store failures in non-production environments, document the expected degraded-mode behavior for each scenario, and compare observed behavior against expectations on a regular schedule. [doc](https://docs.aws.amazon.com/wellarchitected/latest/agentic-ai-lens/agentrel03-bp02.html)
- **[model fallback]** Decouple the orchestration layer from the underlying model and implement automatic fallback to alternative models when throttling occurs — keeps agents responding transparently instead of failing when a single model is rate-limited. [doc](https://aws.amazon.com/blogs/machine-learning/build-unified-intelligence-with-amazon-bedrock-agentcore/)
- **[context pressure]** Track in-context memory utilization per component (system prompt, retrieved context, conversation history, tool results) and alarm at 80% of the model's context window so summarization or pruning runs before the limit becomes a hard failure. [doc](https://docs.aws.amazon.com/wellarchitected/latest/agentic-ai-lens/agentrel08-bp04.html)
- **[memory growth]** Monitor AgentCore Memory access latency and error rates, and use CloudWatch Metric Math to detect gradual memory growth trends that steady-state threshold alarms would miss. [doc](https://docs.aws.amazon.com/wellarchitected/latest/agentic-ai-lens/agentrel08-bp04.html)
- **[testing]** Run automated test suites to evaluate reasoning quality, task completion, and dialogue coherence pre-deployment, and use fault injection to simulate tool outages, API failures, and data inconsistencies before they reach production users. [doc](https://aws.amazon.com/blogs/architecture/build-resilient-generative-ai-agents/)
- **[isolation]** Rely on AgentCore Runtime's per-session microVM isolation for data protection, but enforce session-to-user mapping and session lifecycle limits (max sessions per user) in your own backend, since AgentCore does not enforce this mapping itself. [doc](https://docs.aws.amazon.com/bedrock-agentcore/latest/devguide/runtime-security-best-practices.html)

## ⚡ Performance Efficiency
- **[tool integration]** Adopt MCP as the standard tool-integration protocol via AgentCore Gateway, and enable built-in semantic tool search (`x_amz_bedrock_agentcore_search`) so agents query for relevant tools by natural-language description instead of receiving the full tool catalog in every prompt. [doc](https://docs.aws.amazon.com/wellarchitected/latest/agentic-ai-lens/agentperf06-bp01.html)
- **[tool integration]** For agents with large tool catalogs, use a two-stage selection pattern — a lightweight pre-filter narrows the catalog to the 5–10 most relevant tools for the current task before they appear in the LLM's prompt — and cache tool results at request, session, and global scope with appropriate TTLs. [doc](https://docs.aws.amazon.com/wellarchitected/latest/agentic-ai-lens/agentperf06-bp01.html)
- **[gateway latency]** Deploy Lambda-backed gateway targets in the same Region as the gateway, optimize them for fast cold starts, use provisioned concurrency where low latency is required, and ensure backing REST APIs have low latency and high availability. [doc](https://docs.aws.amazon.com/bedrock-agentcore/latest/devguide/gateway-advanced-performance.html)
- **[gateway schemas]** Keep MCP tool schemas simple, use appropriate parameter data types, include clear parameter descriptions, and mark required fields — well-designed schemas reduce agent tool-selection errors and retries. [doc](https://docs.aws.amazon.com/bedrock-agentcore/latest/devguide/gateway-advanced-performance.html)
- **[compute elasticity]** Let AgentCore Runtime suspend CPU billing during I/O wait (LLM inference, tool calls) rather than pre-provisioning fixed compute — matches cost and resource consumption to actual work for bursty, variable-resource agent workloads. [doc](https://aws.amazon.com/blogs/machine-learning/securely-launch-and-scale-your-agents-and-tools-on-amazon-bedrock-agentcore-runtime/)
- **[observability-driven tuning]** Instrument agents from day one with AgentCore's automatic OpenTelemetry traces (model invocations, tool calls, reasoning steps) so latency regressions can be attributed to the model, a tool call, or an external API rather than guessed at. [doc](https://aws.amazon.com/blogs/machine-learning/ai-agents-in-enterprises-best-practices-with-amazon-bedrock-agentcore/)

## 💰 Cost Optimization
- **[memory tiering]** Design cost-effective retrieval with tiered memory and apply intelligent compression and pruning of context windows — reduces token consumption driven by unnecessarily large retrieved context or conversation history. [doc](https://docs.aws.amazon.com/wellarchitected/latest/agentic-ai-lens/agentcost03-bp01.html)
- **[tool serving]** Design cost-effective tool selection to minimize unnecessary tool invocations, and cost-optimize tool serving through serverless compute and resource sharing across agents behind the gateway. [doc](https://docs.aws.amazon.com/wellarchitected/latest/agentic-ai-lens/agentcost04-bp02.html)
- **[feedback loop]** Run monthly cost-efficiency reviews covering cost-per-task-completion trends, ranked optimization opportunities, and A/B experiment results — use AgentCore Runtime tracing to decompose spend by reasoning pattern, tool invocation, and memory operation rather than only looking at aggregate cost. [doc](https://docs.aws.amazon.com/wellarchitected/latest/agentic-ai-lens/agentcost07-bp03.html)
- **[experimentation]** Validate cost-reduction changes with AgentCore Evaluations before and after each optimization, and use a cost-quality efficiency ratio (cost reduction % ÷ quality degradation %) to prioritize which optimizations to ship — ratios above 10 indicate strong candidates, below 2 signals the quality hit is too large. [doc](https://docs.aws.amazon.com/wellarchitected/latest/agentic-ai-lens/agentcost07-bp03.html)
- **[artifact management]** Build agent container images with a shared base layer plus agent-specific layers so Amazon ECR deduplicates common dependencies, and define an ECR lifecycle policy to prune old, unreferenced AgentCore Runtime versions rather than letting every configuration change anchor a new image indefinitely. [doc](https://docs.aws.amazon.com/wellarchitected/latest/agentic-ai-lens/agentcost06-bp02.html)
- **[gateway efficiency]** Enable Gateway semantic search and metadata-based tool filtering to reduce the number of tools and context loaded per invocation — fewer irrelevant tool definitions in context means fewer wasted tokens per request. [doc](https://aws.amazon.com/bedrock/agentcore/faqs/)

## ⚙️ Operational Excellence
- **[observability]** Start with AgentCore's default automatic instrumentation (model calls, token usage, tool execution) before adding custom spans, and tune verbosity per environment: high verbosity and full trace capture in development, partial sampling in test, minimal logging with strategic sampling in production. [doc](https://aws.amazon.com/blogs/machine-learning/build-trustworthy-ai-agents-with-amazon-bedrock-agentcore-observability/)
- **[observability]** Establish consistent naming conventions for services, spans, and attributes (for example `agent.` and `business.` prefixes) from the start, and filter sensitive data such as PII out of observability attributes and payloads. [doc](https://aws.amazon.com/blogs/machine-learning/build-trustworthy-ai-agents-with-amazon-bedrock-agentcore-observability/)
- **[observability]** Enable CloudWatch Transaction Search as the one-time setup required to view AgentCore metrics, spans, and traces in the generative AI observability dashboard, and export OpenTelemetry-compatible data to existing tools (Datadog, LangSmith, Langfuse) when your organization already standardizes on them. [doc](https://docs.aws.amazon.com/bedrock-agentcore/latest/devguide/observability-service-provided.html)
- **[CI/CD]** Orchestrate the full agent lifecycle with CI/CD pipelines and infrastructure as code to standardize deployments across environments and reduce manual configuration drift. [doc](https://aws.amazon.com/blogs/architecture/build-resilient-generative-ai-agents/)
- **[incident response]** Maintain playbooks covering both infrastructure-level and agent-specific issues, including seamless handoff of live sessions to fallback agents or human operators without losing conversation context. [doc](https://aws.amazon.com/blogs/architecture/build-resilient-generative-ai-agents/)
- **[continuous improvement]** Use AgentCore Optimization's Insights and Recommendations to analyze production traces for failure patterns and generate improved system prompts or tool descriptions, then validate changes with batch evaluations or live A/B tests before promoting them. [doc](https://aws.amazon.com/blogs/machine-learning/introducing-the-agent-performance-loop-agentcore-optimization-now-in-preview/)
- **[least privilege]** Start from AWS managed policies for common AgentCore use cases and progressively narrow to customer-managed least-privilege policies, adding IAM conditions to restrict which resources a role can act on. [doc](https://docs.aws.amazon.com/bedrock-agentcore/latest/devguide/best-practices.html)

## 🌱 Sustainability
- **[resource boundaries]** Split monolithic agent logic into specialized agents with explicit resource boundaries (timeout, memory ceiling, token budget) sized to each capability's actual workload, and propagate remaining time/token/retry budgets into every delegated child invocation. [doc](https://docs.aws.amazon.com/wellarchitected/latest/agentic-ai-lens/agentsus01-bp01.html)
- **[workload scheduling]** Shift deferrable, non-latency-sensitive agent work (batch enrichment, evaluation runs, knowledge-base indexing) to off-peak periods or Regions with favorable energy profiles, reserving low-latency handling for genuinely user-facing interactions. [doc](https://docs.aws.amazon.com/wellarchitected/latest/agentic-ai-lens/agentsus02-bp04.html)
- **[measurement]** Combine AWS carbon-emissions tracking with AgentCore Observability metrics (tokens per successful task, compute hours per workflow, cache-hit rate) on a 30-day rolling baseline so environmental impact becomes a standing input to engineering reviews rather than an annual afterthought. [doc](https://docs.aws.amazon.com/wellarchitected/latest/agentic-ai-lens/agentsus02-bp04.html)
- **[resource reuse]** Amortize development and compute cost by sharing well-bounded agents (for example a single data-validation agent) across many parent workflows through AgentCore Gateway, instead of duplicating the same capability inside each caller. [doc](https://docs.aws.amazon.com/wellarchitected/latest/agentic-ai-lens/agentsus01-bp01.html)
