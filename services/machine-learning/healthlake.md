# Amazon HealthLake — Best Practices

## Common scenarios
- Storing and querying FHIR R4 clinical data at scale for a healthcare application        → Security, Reliability, Performance Efficiency
- Bulk importing historical clinical data from Amazon S3 into a data store        → Reliability, Performance Efficiency
- Exporting FHIR data to Amazon S3 for downstream analytics with Athena, Redshift, or Glue        → Performance Efficiency, Operational Excellence
- Extracting entities and PHI from unstructured medical text with integrated NLP        → Security, Operational Excellence

## 🔒 Security
- **[access control]** Implement least-privilege IAM policies for HealthLake actions and data stores — limits blast radius if credentials are compromised. [doc](https://docs.aws.amazon.com/healthlake/latest/devguide/best-practices-security.html)
- **[encryption at rest]** Use customer-managed KMS keys (CMKs) instead of AWS-owned keys wherever possible — gives you control over key rotation, access policy, and auditability. [doc](https://docs.aws.amazon.com/healthlake/latest/devguide/best-practices-security.html)
- **[PHI/PII handling]** Use FHIR `search` with `POST` instead of `GET` when queries touch PHI or PII — POST carries parameters in the encrypted request body instead of a loggable URL. [doc](https://docs.aws.amazon.com/healthlake/latest/devguide/best-practices-security.html)
- **[PHI/PII handling]** Keep PHI or PII out of visible fields, logical FHIR IDs (LIDs), data store/job names, and HTTP headers on create, read, update, delete, search, and bulk import/update calls — these values can end up in logs and diagnostics. [doc](https://docs.aws.amazon.com/healthlake/latest/devguide/best-practices-security.html)
- **[auditing]** Enable AWS CloudTrail for HealthLake — provides a record to detect and investigate unexpected activity. [doc](https://docs.aws.amazon.com/healthlake/latest/devguide/best-practices-security.html)
- **[access control]** Limit access to sensitive administrative and auditing functions to a small set of trusted principals — reduces the risk of unauthorized configuration or log tampering. [doc](https://docs.aws.amazon.com/healthlake/latest/devguide/best-practices-security.html)
- **[S3 hygiene]** Apply Amazon S3 security best practices to the buckets used for HealthLake import/export — data transits through S3 in both directions and inherits its access controls. [doc](https://docs.aws.amazon.com/healthlake/latest/devguide/best-practices-security.html)
- **[identity]** Set up individual IAM Identity Center or IAM users/roles with least-privilege permissions and MFA rather than shared account credentials — reduces exposure from lost or reused credentials. [doc](https://docs.aws.amazon.com/healthlake/latest/devguide/data-protection.html)
- **[data in transit]** Require TLS 1.2 or later, preferring TLS 1.3, for connections to HealthLake — HealthLake enforces HTTPS by default, and pinning the minimum version avoids downgrade risk in client configurations. [doc](https://docs.aws.amazon.com/healthlake/latest/devguide/data-protection.html)
- **[data in transit]** Use a FIPS endpoint for CLI or API access when FIPS 140-3 validated cryptographic modules are required — needed for certain regulated or government workloads. [doc](https://docs.aws.amazon.com/healthlake/latest/devguide/data-protection.html)
- **[sensitive data]** Keep confidential or sensitive information, such as customer emails, out of tags and free-form name fields — this data can surface in billing or diagnostic logs. [doc](https://docs.aws.amazon.com/healthlake/latest/devguide/data-protection.html)
- **[network isolation]** Use interface VPC endpoints (AWS PrivateLink) with a restrictive endpoint policy for HealthLake API traffic — keeps traffic off the public internet and scopes access to approved principals and actions. [doc](https://docs.aws.amazon.com/healthlake/latest/devguide/vpc-interface-endpoints.html)
- **[IAM]** Base identity-based policies on the documented HealthLake actions, resources, and condition keys rather than broad wildcard permissions — keeps grants aligned to actual job and data-store needs. [doc](https://docs.aws.amazon.com/healthlake/latest/devguide/security_iam_id-based-policy-examples.html)
- **[encryption]** Choose a customer-managed KMS key at data store creation time if you plan to import or export files — a customer-managed key is required for import/export and the encryption choice cannot be changed after creation. [doc](https://docs.aws.amazon.com/healthlake/latest/devguide/encryption-at-rest.html)
- **[compliance]** Put a Business Associate Addendum (BAA) in place and review the AWS Shared Responsibility Model before processing real PHI — HealthLake is HIPAA-eligible but compliance obligations remain shared. [doc](https://docs.aws.amazon.com/healthlake/latest/devguide/what-is.html)

## 🛡️ Reliability
- **[concurrency control]** Include ETags (`If-Match`/`If-None-Match`) on update and delete operations and retry on `412 Precondition Failed` — prevents unintended overwrites from concurrent modifications. [doc](https://docs.aws.amazon.com/healthlake/latest/devguide/managing-fhir-resources-idempotency.html)
- **[quota planning]** Design import/export pipelines around the default one-concurrent-job-per-account limits for `StartFHIRImportJob` and `StartFHIRExportJob`, requesting increases proactively — avoids job failures or queuing surprises at high volume. [doc](https://docs.aws.amazon.com/healthlake/latest/devguide/reference-healthlake-endpoints-quotas.html)
- **[event-driven ingestion]** Use the Amazon EventBridge integration to track import/export job and FHIR resource-level events — removes the need for custom polling loops that can miss or delay state changes. [doc](https://aws.amazon.com/blogs/industries/amazon-eventbridge-integration-for-aws-healthlake/)
- **[multi-Region design]** Replicate to another AWS Region if you need resilience across greater geographic distances — the built-in Availability Zone architecture covers zonal failure but not Regional events. [doc](https://docs.aws.amazon.com/healthlake/latest/devguide/disaster-recovery-resiliency.html)

## ⚡ Performance Efficiency
- **[search]** Prefer FHIR `search` with `POST` for large or PHI-bearing queries — parameters travel in the encrypted request body instead of being limited by URL length. [doc](https://docs.aws.amazon.com/healthlake/latest/devguide/searching-fhir-resources-post.html)
- **[bulk ingestion]** Use the bulk `$import` API from Amazon S3 for historical/backfill loads, then switch to real-time FHIR create/update APIs for ongoing data — matches ingestion mode to the workload's throughput and latency needs. [doc](https://aws.amazon.com/blogs/industries/amazon-eventbridge-integration-for-aws-healthlake/)
- **[bulk export]** Use the FHIR `$export` operation (System, all-patients, or group-of-patients) to move data to Amazon S3 in bulk `ndjson` format — avoids paginating through individual reads for large downstream analytics loads. [doc](https://docs.aws.amazon.com/healthlake/latest/devguide/reference-fhir-operations-export.html)
- **[write capacity]** Size batch and transaction Bundles with the knowledge that sync write capacity scales per 1 KB of payload and doubles when `x-amz-fhir-history-consistency-level: strong` is set — prevents unexpected throttling under load. [doc](https://docs.aws.amazon.com/healthlake/latest/devguide/reference-healthlake-endpoints-quotas.html)
- **[bundles]** Prefer `batch`-type Bundles over `transaction`-type Bundles when strict all-or-nothing semantics aren't required — batch Bundles process twice the resources per second of transaction Bundles at the same capacity. [doc](https://docs.aws.amazon.com/healthlake/latest/devguide/reference-healthlake-endpoints-quotas.html)
- **[analytics]** Query the Apache Iceberg tables HealthLake populates automatically (governed by AWS Lake Formation) with Amazon Athena, Redshift, or Glue for bulk analysis — offloads heavy SQL-style analytics from the FHIR API path. [doc](https://aws.amazon.com/blogs/industries/building-a-multi-tenant-fhir-server-with-aws-healthlake/)

## ⚙️ Operational Excellence
- **[monitoring]** Track Call Count, Successful Requests, User Errors, Server Errors, Throttled Requests, and Latency in CloudWatch by Operation and data store ID — surfaces throttling and error trends before they affect users. [doc](https://docs.aws.amazon.com/healthlake/latest/devguide/monitoring-cloudwatch.html)
- **[auditing]** Enable ongoing AWS CloudTrail trails, not just event history, for HealthLake API activity — supports operational review and long-term compliance evidence. [doc](https://docs.aws.amazon.com/healthlake/latest/devguide/best-practices-security.html)
- **[infrastructure as code]** Provision HealthLake data stores through AWS CloudFormation — makes encryption and other creation-time settings, which are immutable afterward, consistent and repeatable across environments. [doc](https://docs.aws.amazon.com/healthlake/latest/devguide/security.html)
- **[event-driven operations]** Integrate HealthLake with Amazon EventBridge to trigger automated responses to import/export state changes and resource-level events — replaces manual polling with event-driven operations. [doc](https://aws.amazon.com/blogs/industries/amazon-eventbridge-integration-for-aws-healthlake/)
