# Amazon Forecast — Best Practices

## Common scenarios
- Retail and supply-chain demand planning from historical sales data        → Performance Efficiency, Cost Optimization
- Resource and workforce planning (staffing, energy, server capacity)        → Reliability, Operational Excellence
- Automated, scheduled retraining pipelines for production forecasting        → Operational Excellence, Cost Optimization
- Multi-tenant SaaS forecasting isolated per customer/account        → Security

## 🔒 Security
- **[access control]** Grant Amazon Forecast least-privilege access to Amazon S3 by creating a dedicated IAM role and policy scoped to the buckets Forecast needs, rather than reusing broad or pre-existing roles. [doc](https://docs.aws.amazon.com/forecast/latest/dg/aws-forecast-iam-roles.html)
- **[access control]** Use the `aws:SourceArn` and `aws:SourceAccount` global condition context keys in the IAM role trust policy Forecast assumes, to prevent the cross-service confused deputy problem. [doc](https://docs.aws.amazon.com/forecast/latest/dg/aws-forecast-iam-roles.html)
- **[access control]** Scope identity-based policies to specific Forecast actions and resource ARNs (dataset, dataset group, predictor, forecast) instead of granting `forecast:*`, and reserve the `AWSForecastFullAccess` managed policy for console users who genuinely need full access. [doc](https://docs.aws.amazon.com/forecast/latest/dg/security_iam_id-based-policy-examples.html)
- **[access control]** In multi-tenant designs, enforce a tenant tag on Forecast resources at creation time and condition `list`/`describe`/`delete` actions on a matching tenant tag so one tenant cannot access another's datasets, predictors, or forecasts. [doc](https://aws.amazon.com/blogs/machine-learning/configure-amazon-forecast-for-a-multi-tenant-saas-application/)
- **[data protection]** Supply your own AWS KMS customer managed key and an IAM role in the `EncryptionConfig` of `CreateDataset` and `CreatePredictor` so all downstream operations (import jobs, forecasts, explainability, backtest exports) are encrypted at rest with a key you control instead of the Forecast default service key. [doc](https://docs.aws.amazon.com/forecast/latest/dg/data-protection.html)
- **[data protection]** Encrypt source and export data in your Amazon S3 buckets with your own KMS key (SSE-KMS) rather than relying solely on the default Amazon S3 key, since Forecast reads and writes through S3. [doc](https://docs.aws.amazon.com/forecast/latest/dg/data-protection.html)
- **[data protection]** Monitor AWS KMS `CreateGrant`, `RetireGrant`, and `DescribeKey` calls made by Forecast on your customer managed key via AWS CloudTrail or CloudWatch Logs to verify only expected Forecast operations are using your key. [doc](https://docs.aws.amazon.com/forecast/latest/dg/data-protection.html)
- **[network security]** Use interface VPC endpoints (AWS PrivateLink) for the `forecast` and `forecastquery` services so API traffic from your VPC stays on the AWS network instead of traversing the public internet. [doc](https://docs.aws.amazon.com/forecast/latest/dg/vpc-interface-endpoints.html)
- **[network security]** In regions with strict compliance requirements, use the FIPS 140-2 compliant `forecast-fips` and `forecastquery-fips` VPC endpoints where available. [doc](https://docs.aws.amazon.com/forecast/latest/dg/vpc-interface-endpoints.html)
- **[network security]** Do not rely on Amazon S3 VPC endpoint (gateway) policies to restrict access to Forecast's source buckets, since Forecast cannot reach S3 buckets that only allow VPC-restricted access and such buckets will return `AccessDenied`. [doc](https://docs.aws.amazon.com/forecast/latest/dg/aws-forecast-iam-roles.html)

## 🛡️ Reliability
- **[quotas]** Track dataset, dataset group, and job counts against Forecast's service quotas (for example 500 dataset groups, 1,000 dataset import jobs per Region) and request increases proactively for adjustable quotas before hitting limits in production. [doc](https://docs.aws.amazon.com/forecast/latest/dg/limits.html)
- **[quotas]** Size target time-series datasets with quota ceilings in mind (up to 5,000,000 time series per predictor, 3 billion rows per dataset, with lower ceilings in `ap-south-1`) and plan data granularity accordingly, since datasets over 100,000 items are restricted to yearly/monthly/weekly/daily frequencies. [doc](https://docs.aws.amazon.com/forecast/latest/dg/limits.html)
- **[monitoring]** Enable predictor monitoring on AutoPredictors to detect forecast-accuracy degradation over time (e.g. from shifting customer behavior) instead of assuming a trained predictor remains accurate indefinitely. [doc](https://docs.aws.amazon.com/forecast/latest/dg/predictor-monitoring.html)
- **[validation]** Review backtest accuracy metrics before generating future-dated forecasts, and use multiple backtest windows to reduce selection bias, instead of proceeding straight from training to production forecasts. [doc](https://aws.amazon.com/blogs/machine-learning/measuring-forecast-model-accuracy-to-optimize-your-business-objectives-with-amazon-forecast/)
- **[operations]** Use `CreatePredictor` only for initial training or scheduled retraining, and use `CreateForecast` for routine inference, so production pipelines don't unnecessarily retrain models on every run. [doc](https://docs.aws.amazon.com/whitepapers/latest/time-series-forecasting-principles-with-amazon-forecast/summary-of-forecasting-workflow-and-apis.html)

## ⚡ Performance Efficiency
- **[accuracy]** Start with a simple target time-series-only model to establish a baseline, then iteratively add related time-series and item metadata, measuring the accuracy impact of each change before keeping it. [doc](https://aws.amazon.com/blogs/machine-learning/start-your-successful-journey-with-time-series-forecasting-with-amazon-forecast/)
- **[accuracy]** Analyze item-level accuracy rather than only the top-level/aggregate accuracy statistic, and focus feature engineering effort (price, promotions, seasonality, events) on the outlier items with poor accuracy. [doc](https://aws.amazon.com/blogs/machine-learning/start-your-successful-journey-with-time-series-forecasting-with-amazon-forecast/)
- **[accuracy]** Only add a related time series if backtesting shows it improves or maintains accuracy; drop related time series with missing values or quality issues instead of introducing noise into the model. [doc](https://aws.amazon.com/blogs/machine-learning/making-accurate-energy-consumption-predictions-with-amazon-forecast/)
- **[evaluation]** Compare candidate models using the weighted quantile loss (wQL) metric at the same quantile, since a lower wQL at a fixed quantile is the valid basis for choosing the more accurate model. [doc](https://aws.amazon.com/blogs/machine-learning/measuring-forecast-model-accuracy-to-optimize-your-business-objectives-with-amazon-forecast/)
- **[evaluation]** Choose more than one backtest window (up to 5) when tuning predictors, since relying on a single window can bias model selection toward a window that happens to be easier or harder than average. [doc](https://aws.amazon.com/blogs/machine-learning/measuring-forecast-model-accuracy-to-optimize-your-business-objectives-with-amazon-forecast/)

## 💰 Cost Optimization
- **[training cost]** Use the predictor retrain function (transfer learning from an existing predictor) instead of training a new predictor from scratch when refreshing on incremental data, to reduce training time and cost. [doc](https://aws.amazon.com/blogs/machine-learning/start-your-successful-journey-with-time-series-forecasting-with-amazon-forecast/)
- **[training cost]** Enable predictor monitoring so retraining is triggered only when accuracy has actually degraded, rather than retraining on a fixed schedule regardless of need. [doc](https://aws.amazon.com/blogs/machine-learning/start-your-successful-journey-with-time-series-forecasting-with-amazon-forecast/)
- **[forecast cost]** Don't generate future-dated forecasts until backtest accuracy meets your requirements, since generating forecasts from an inadequately accurate predictor is inefficient spend that must be redone after further tuning. [doc](https://aws.amazon.com/blogs/machine-learning/start-your-successful-journey-with-time-series-forecasting-with-amazon-forecast/)
- **[resource hygiene]** Delete unused Forecast resources (datasets, predictors, forecasts, monitors) on a regular cadence as part of automated pipelines, since retained monitor data persists (and can accumulate) even after underlying forecast data is deleted unless the monitor resource itself is removed. [doc](https://docs.aws.amazon.com/forecast/latest/dg/predictor-monitoring.html)

## ⚙️ Operational Excellence
- **[automation]** Build repeatable, serverless forecasting pipelines (for example with Step Functions, Lambda, and scheduled triggers) rather than running one-off console or ad hoc API workflows, so dataset import, training, and forecast generation run on a reliable, auditable cadence. [doc](https://aws.amazon.com/blogs/machine-learning/start-your-successful-journey-with-time-series-forecasting-with-amazon-forecast/)
- **[automation]** Parameterize retrain-vs-reuse strategy (new predictor, retrain, or reuse without retraining) in your orchestration so the same pipeline can skip unnecessary training when data is stable or monitoring shows no drift. [doc](https://aws.amazon.com/blogs/machine-learning/automate-the-deployment-of-an-amazon-forecast-time-series-forecasting-model/)
- **[logging]** Enable AWS CloudTrail to capture Forecast API calls (console and SDK/CLI) including caller identity and source IP, so you can audit who created, modified, or deleted forecasting resources. [doc](https://docs.aws.amazon.com/forecast/latest/dg/logging-using-cloudtrail.html)
- **[monitoring]** Use Amazon CloudWatch metrics for Forecast alongside CloudTrail to monitor the health, status, and performance of dataset import jobs, predictor training, and forecast generation. [doc](https://docs.aws.amazon.com/forecast/latest/dg/logging-monitoring.html)
- **[planning]** Before building datasets, document inventory/business cadence, forecast granularity and horizon, and the KPIs used to evaluate forecast performance against real-world outcomes, so the forecasting solution is designed around a clear business objective from the start. [doc](https://aws.amazon.com/blogs/industries/aws-service-spotlight-amazon-forecast-for-accurate-demand-forecasting/)
