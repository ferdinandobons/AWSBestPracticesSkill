# Amazon Omics — Best Practices

## Common scenarios
- Running custom bioinformatics pipelines (WDL, Nextflow, CWL) at scale        → Security, Reliability, Performance Efficiency, Cost Optimization
- Storing and managing raw sequence, reference, variant, and annotation data        → Security, Cost Optimization, Performance Efficiency
- Monitoring and troubleshooting production genomics workflow runs        → Reliability, Operational Excellence
- Granting teams and partners least-privilege access to omics data and workflows        → Security, Operational Excellence

## 🔒 Security
- **[IAM roles for runs]** Start from the minimal IAM role that the HealthOmics console generates for a workflow run, then add only the narrowly-scoped permissions the run actually needs, keeping HealthOmics as the sole principal in the role's trust policy. — Limits blast radius if run credentials are ever misused. [doc](https://aws.amazon.com/blogs/industries/secure-your-genomic-workflows-and-data-with-aws-healthomics/)
- **[Identity policies]** Apply least-privilege permissions in identity-based policies, use IAM Access Analyzer to validate them, and require MFA for IAM users with access to HealthOmics resources. — Reduces the risk of over-broad access to stores, workflows, and runs. [doc](https://docs.aws.amazon.com/omics/latest/dev/security_iam_id-based-policy-examples.html)
- **[Cross-account access]** Limit S3 bucket policies and KMS key policies to the minimum set of IAM roles and users when a workflow or store requires cross-account data access. — A bucket or key policy that is broader than necessary is not needed for same-account access and only adds risk in cross-account scenarios. [doc](https://aws.amazon.com/blogs/industries/secure-your-genomic-workflows-and-data-with-aws-healthomics/)
- **[Container images]** Enable continuous vulnerability scanning (ECR enhanced scanning) on workflow container images, reference images by a versioned tag rather than `latest`, and enable ECR tag immutability. — Lets you detect and patch known vulnerabilities quickly and reliably reproduce which image and tools produced a given run for audits and diagnostics. [doc](https://aws.amazon.com/blogs/industries/secure-your-genomic-workflows-and-data-with-aws-healthomics/)
- **[Encryption at rest]** Use customer-managed AWS KMS keys for HealthOmics Storage and Analytics data stores when you need direct control over key policies and access, instead of relying only on the default AWS-owned key. — Customer-managed keys give full control over who can decrypt sensitive omics data; note the encryption choice cannot be changed after a store is created. [doc](https://docs.aws.amazon.com/omics/latest/dev/data-protection.html)
- **[Workflow filesystem]** Rely on HealthOmics's per-run compute, storage, and network isolation (dedicated filesystem destroyed after the run, no shared kernel/CPU/network between tasks, no internet egress except through VPC endpoints) rather than building custom isolation. — Ensures workflow data and intermediate results are not exposed to other runs or the public internet. [doc](https://aws.amazon.com/blogs/industries/secure-your-genomic-workflows-and-data-with-aws-healthomics/)

## 🛡️ Reliability
- **[Task retries]** Configure `errorStrategy`/`maxRetries` (Nextflow) or `preemptible` (WDL) explicitly instead of accepting the language default, and opt out of retries only for tasks that are not idempotent. — HealthOmics automatically retries tasks that fail due to transient 5XX service errors, so tuning retry behavior per task avoids masking real failures while still absorbing intermittent errors. [doc](https://docs.aws.amazon.com/omics/latest/dev/monitoring-runs.html)
- **[Run storage type]** Use dynamic run storage for iterative development, and switch to static (pre-provisioned) storage for long-running, high-I/O, or high-task-concurrency production runs, validating the choice with A/B testing. — Matches provisioning speed and throughput characteristics to the workload instead of defaulting to one storage type for every run. [doc](https://docs.aws.amazon.com/omics/latest/dev/workflows-run-optimize.html)
- **[Data durability]** Store workflow inputs and outputs in Amazon S3 or HealthOmics Storage rather than only in the run's temporary filesystem. — The per-run filesystem is destroyed after the run completes, so anything not persisted to a durable store is lost. [doc](https://aws.amazon.com/blogs/industries/secure-your-genomic-workflows-and-data-with-aws-healthomics/)

## ⚡ Performance Efficiency
- **[Container size]** Keep workflow container images as small as possible, and store large reference datasets in S3 for the workflow to import rather than bundling them into the container. — Smaller containers load faster at the start of each task, improving task launch efficiency across a run. [doc](https://docs.aws.amazon.com/omics/latest/dev/workflows-run-optimize.html)
- **[Run Analyzer]** Use the open-source Run Analyzer tool to evaluate memory/compute bottlenecks, identify over-provisioned tasks, and get recommended instance-size adjustments. — Surfaces concrete, run-specific right-sizing and I/O recommendations instead of guessing at resource allocation. [doc](https://docs.aws.amazon.com/omics/latest/dev/workflows-run-optimize.html)
- **[I/O-bound tasks]** Over-provision static run storage size when a compute task is I/O-bound. — Increasing filesystem size raises maximum filesystem throughput, which can reduce task run time enough to offset the added storage cost. [doc](https://docs.aws.amazon.com/omics/latest/dev/workflows-run-optimize.html)
- **[Sequence storage tiering]** Use HealthOmics Storage's automatic tiering and compression, purpose-built for omics file formats (FASTQ, BAM, CRAM), instead of building custom tiering/compression on generic object storage. — Preserves low-latency access to frequently used data while reducing cost for infrequently accessed sequence data. [doc](https://aws.amazon.com/blogs/industries/store-omics-data-cost-effectively-at-any-scale-with-aws-healthomics/)

## 💰 Cost Optimization
- **[Run cost visibility]** Tag runs by team or project and review costs in Cost Explorer grouped by tag, and use Billing and Cost Management data exports for per-run itemized cost reports. — Enables cost attribution and optimization decisions at the team/project level instead of only aggregate service spend. [doc](https://docs.aws.amazon.com/omics/latest/dev/workflows-run-optimize.html)
- **[Run Analyzer for cost]** Run the Run Analyzer tool to get task-level cost estimates and potential savings from right-sizing recommendations, then test adopted changes before relying on them in production. — Turns manifest log data into actionable, quantified cost-saving opportunities per task. [doc](https://docs.aws.amazon.com/omics/latest/dev/workflows-run-optimize.html)
- **[Storage auto-archival]** Leave HealthOmics Storage's automatic tiering enabled so data not accessed for an extended period moves to the lower-cost storage class automatically. — Reduces total cost of ownership for raw sequence data without requiring manual lifecycle management. [doc](https://aws.amazon.com/blogs/industries/store-omics-data-cost-effectively-at-any-scale-with-aws-healthomics/)
- **[Run storage sizing]** Choose dynamic run storage for development workflows so you don't pay for pre-provisioned capacity you don't need, reserving static storage for production runs that benefit from it. — Avoids paying for fixed high-capacity storage during iterative development and testing. [doc](https://docs.aws.amazon.com/omics/latest/dev/workflows-run-optimize.html)

## ⚙️ Operational Excellence
- **[Centralized logging]** Keep run logging enabled (the default) and review manifest and engine logs in CloudWatch Logs and S3 for task status, timing, and failure reasons. — Provides the diagnostic detail needed to troubleshoot failed or slow runs without re-running the workflow. [doc](https://docs.aws.amazon.com/omics/latest/dev/monitoring-cloudwatch-logs.html)
- **[Metrics and alarms]** Monitor the `AWS/Omics` CloudWatch namespace and set alarms on relevant API and run metrics. — Provides near real-time visibility into run and API activity so issues are caught before they compound. [doc](https://docs.aws.amazon.com/omics/latest/dev/monitoring-cloudwatch.html)
- **[Event-driven alerting]** Subscribe to HealthOmics run status change events on Amazon EventBridge to trigger automated failure notifications and downstream automation. — Notifies operators of run failures as soon as they occur instead of relying on manual console checks. [doc](https://aws.amazon.com/blogs/industries/enhance-monitoring-and-observability-for-aws-healthomics-workflows/)
- **[Audit trail]** Use AWS CloudTrail to capture and review HealthOmics API calls, including which IAM identities invoked them and from where. — Supports troubleshooting, security review, and compliance audit requirements for who accessed or changed workflows and data. [doc](https://docs.aws.amazon.com/omics/latest/dev/monitoring-overview.html)

## 🌱 Sustainability
- **[Serverless-first design]** Rely on HealthOmics's fully managed, serverless orchestration (automatic provisioning, scaling, and teardown of run compute and storage) rather than self-managed compute clusters for bioinformatics pipelines. — Matches provisioned resources to actual workload needs and avoids the idle capacity associated with statically sized, self-managed infrastructure. [doc](https://aws.amazon.com/solutions/guidance/migration-and-storage-of-sequence-data-with-aws-healthomics/)
