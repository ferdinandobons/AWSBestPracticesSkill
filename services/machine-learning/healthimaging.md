# AWS HealthImaging — Best Practices

## Common scenarios
- Storing and streaming DICOM medical images (X-Ray, CT, MRI) at petabyte scale for clinical workflows        → Security, Reliability, Performance Efficiency
- Bulk importing on-premises PACS archives or S3-staged DICOM P10 files into a data store        → Reliability, Performance Efficiency, Cost Optimization
- Building low-latency, zero-footprint web viewers for radiologists using HTJ2K-encoded image frames        → Performance Efficiency
- Running AI/ML inference over a medical imaging archive while auditing all access to PHI        → Security, Operational Excellence

## 🔒 Security
- **[identity]** Set up individual users with IAM Identity Center or IAM, granting each principal only the HealthImaging permissions needed for their role, and require MFA on every account. [doc](https://docs.aws.amazon.com/healthimaging/latest/devguide/data-protection.html)
- **[data in transit]** Require TLS 1.3 with cipher suites that support perfect forward secrecy (DHE or ECDHE) for all client connections, since HealthImaging enforces TLS 1.3+ at the API layer. [doc](https://docs.aws.amazon.com/healthimaging/latest/devguide/infrastructure-security.html)
- **[request signing]** Sign all API requests with an IAM principal's access keys, or use AWS STS to generate short-lived temporary credentials instead of long-lived keys. [doc](https://docs.aws.amazon.com/healthimaging/latest/devguide/infrastructure-security.html)
- **[auditing]** Enable AWS CloudTrail trails to capture every HealthImaging API call (who, when, source IP) for security review and compliance, since all access to PHI-bearing metadata and image data is logged. [doc](https://docs.aws.amazon.com/healthimaging/latest/devguide/logging-using-cloudtrail.html)
- **[encryption at rest]** Use AWS KMS to manage encryption of data stores, choosing customer-managed keys when you need full control and auditability over key policies rather than relying solely on AWS-owned keys. [doc](https://docs.aws.amazon.com/healthimaging/latest/devguide/data-encryption.html)
- **[sensitive data hygiene]** Never place confidential or sensitive information (such as patient identifiers or emails) in tags or free-form name fields, since this data can surface in billing or diagnostic logs. [doc](https://docs.aws.amazon.com/healthimaging/latest/devguide/data-protection.html)
- **[network isolation]** Create interface VPC endpoints (AWS PrivateLink) for the `medical-imaging`, `runtime-medical-imaging`, and `dicom-medical-imaging` service names to keep API, data-plane, and DICOMweb traffic off the public internet, and enable private DNS. [doc](https://docs.aws.amazon.com/healthimaging/latest/devguide/vpc-interface-endpoints.html)
- **[least privilege]** Scope custom IAM policies to specific HealthImaging actions and resource ARNs rather than `*`, and start from the AWS-managed `AWSHealthImagingReadOnlyAccess` policy for read-only use cases. [doc](https://docs.aws.amazon.com/healthimaging/latest/devguide/security-iam-awsmanpol.html)
- **[cross-account import]** In the `ImportJobDataAccessRole` trust policy, use the `aws:SourceArn` and `aws:SourceAccount` global condition keys scoped to your data store ARN to prevent the cross-service confused deputy problem during import jobs. [doc](https://docs.aws.amazon.com/healthimaging/latest/devguide/cross-service-confused-deputy-prevention.html)
- **[compliance]** Treat HealthImaging as HIPAA-eligible: put a Business Associate Addendum (BAA) in place and apply the AWS Shared Responsibility Model before storing or processing real PHI. [doc](https://docs.aws.amazon.com/healthimaging/latest/devguide/what-is.html)
- **[FIPS]** Use a FIPS endpoint for CLI/API access when you require FIPS 140-3 validated cryptographic modules. [doc](https://docs.aws.amazon.com/healthimaging/latest/devguide/data-protection.html)

## 🛡️ Reliability
- **[multi-AZ design]** Rely on the underlying AWS Region/Availability Zone architecture for fault tolerance, and replicate to another AWS Region if your workload needs resilience across greater geographic distances. [doc](https://docs.aws.amazon.com/healthimaging/latest/devguide/disaster-recovery-resiliency.html)
- **[quota planning]** Design import pipelines around per-data-store service quotas (for example, up to 100 concurrent import jobs and 5,000 files per import job in most Regions) and request quota increases proactively for high-volume migrations. [doc](https://docs.aws.amazon.com/general/latest/gr/healthimaging.html)
- **[pixel data integrity]** Verify import correctness using the Image Frame Resolution Checksum (IFRC) in the `job-output-manifest.json` and the `PixelDataChecksumFromBaseToFullResolution` metadata field, confirming lossless transcoding before relying on imported data clinically. [doc](https://docs.aws.amazon.com/healthimaging/latest/devguide/pixel-data-verification.html)
- **[event-driven monitoring]** Use Amazon EventBridge to react to HealthImaging API events (for example import job state changes) instead of building custom polling loops. [doc](https://docs.aws.amazon.com/healthimaging/latest/devguide/monitoring.html)
- **[concurrency limits]** Respect the maximum concurrent `CopyImageSet`, `DeleteImageSet`, and `UpdateImageSetMetadata` requests per data store, and implement retry/backoff logic for throttled requests during bulk operations. [doc](https://docs.aws.amazon.com/general/latest/gr/healthimaging.html)

## ⚡ Performance Efficiency
- **[image codec]** Rely on HTJ2K (High-Throughput JPEG 2000) encoding for pixel data, and decode with SIMD/WASM-SIMD-capable libraries so viewers and AI pipelines get an order-of-magnitude faster decode than legacy JPEG 2000. [doc](https://docs.aws.amazon.com/healthimaging/latest/devguide/reference-libraries.html)
- **[metadata-first access]** Fetch study/series/patient metadata via `GetImageSetMetadata` using developer-friendly keys before loading pixel data, so applications and AI algorithms avoid pulling image frames they don't need. [doc](https://docs.aws.amazon.com/healthimaging/latest/devguide/get-image-frame.html)
- **[frame retrieval]** Use `GetImageFrame` (or `GetDICOMInstanceFrames` for DICOMweb-compatible viewers) to retrieve pixel data in its stored transfer syntax directly, since HealthImaging performs no transcoding at retrieval time to minimize latency. [doc](https://docs.aws.amazon.com/healthimaging/latest/devguide/get-image-frame.html)
- **[parallel ingestion]** Scale ingestion by running multiple import jobs concurrently rather than serializing large archive migrations, taking advantage of HealthImaging's cloud-native parallel processing. [doc](https://aws.amazon.com/blogs/industries/introducing-aws-healthimaging/)
- **[progressive resolution]** Use HealthImaging's support for progressive resolution image access so interactive viewers can render lower-resolution previews first instead of waiting for full-resolution frames. [doc](https://aws.amazon.com/blogs/industries/introducing-aws-healthimaging/)

## 💰 Cost Optimization
- **[intelligent tiering]** Let HealthImaging automatically move image sets to the Archive Instant Access tier after 30 days of inactivity instead of building custom archival logic — both tiers deliver millisecond retrieval, so there is no performance tradeoff for cold data. [doc](https://docs.aws.amazon.com/healthimaging/latest/devguide/cost-optimization.html)
- **[access patterns]** Understand which API actions (`GetImageSetMetadata`, `GetImageFrame`, WADO-RS, `CopyImageSet`, `UpdateImageSetMetadata`, console viewing) promote image sets back to the Frequent Access tier, and which actions (list, search, `GetImageSet`, tagging, delete) do not, to avoid unintended tier promotions that increase cost. [doc](https://docs.aws.amazon.com/healthimaging/latest/devguide/cost-optimization.html)
- **[storage sizing]** Account for the 5 MB minimum billed size per image set and the 30-day minimum storage duration when planning deletions or small-image-set-heavy workloads. [doc](https://docs.aws.amazon.com/healthimaging/latest/devguide/cost-optimization.html)
- **[structured storage]** Estimate indexed structured-data storage costs separately from image-set storage, using the approximate per-resource sizes (study, series, instance) documented for HealthImaging. [doc](https://docs.aws.amazon.com/healthimaging/latest/devguide/cost-optimization.html)
- **[storage footprint]** Take advantage of HTJ2K's efficient lossless compression during import to reduce total storage volume compared to self-managed DICOM archives. [doc](https://aws.amazon.com/blogs/industries/introducing-aws-healthimaging/)

## ⚙️ Operational Excellence
- **[monitoring]** Monitor HealthImaging CloudWatch metrics broken out by `AccountId`, `DatastoreId`, and `StorageTier` dimensions to track storage usage and detect anomalies across Frequent Access and Archive Instant Access tiers. [doc](https://docs.aws.amazon.com/healthimaging/latest/devguide/monitoring-cloudwatch.html)
- **[auditing]** Enable AWS CloudTrail trails (not just default event history) for continuous delivery of HealthImaging API activity to S3, supporting long-term operational review and compliance audits. [doc](https://docs.aws.amazon.com/healthimaging/latest/devguide/logging-using-cloudtrail.html)
- **[event-driven automation]** Build EventBridge rules matching the `aws.medical-imaging` event source to trigger automated workflows on import job completions or other API activity instead of manual polling. [doc](https://docs.aws.amazon.com/eventbridge/latest/ref/events-ref-medical-imaging.html)
- **[infrastructure as code]** Provision HealthImaging data stores through AWS CloudFormation to make configuration consistent and repeatable across environments. [doc](https://docs.aws.amazon.com/healthimaging/latest/devguide/security.html)
- **[import validation]** Review the `job-output-manifest.json` output log after every import job to catch per-image error conditions before treating imported data as production-ready. [doc](https://docs.aws.amazon.com/healthimaging/latest/devguide/importing-imaging-data.html)
