# Amazon Fraud Detector — Best Practices

## Common scenarios
- Real-time scoring of online transactions and payments        → Reliability, Performance Efficiency
- Fraud checks during account registration and new-account sign-up        → Performance Efficiency, Security
- Batch/offline fraud scoring of historical activity via Amazon S3        → Cost Optimization, Operational Excellence
- Business-driven decisioning (approve/review/decline) layered on top of ML scores        → Operational Excellence, Reliability

## 🔒 Security
- **[access control]** Create IAM users or roles with access restricted to the specific Amazon Fraud Detector actions and resources needed, rather than granting broad managed policies to every caller. [doc](https://docs.aws.amazon.com/frauddetector/latest/ug/security_iam_service-with-iam.html)
- **[access control]** Scope any S3 access granted to Fraud Detector or to functions calling it (for example an inline policy limited to `s3:GetObject`/`s3:PutObject` on the specific training/prediction buckets) instead of attaching `AmazonS3FullAccess`. [doc](https://aws.amazon.com/blogs/machine-learning/performing-batch-fraud-predictions-using-amazon-fraud-detector-amazon-s3-and-aws-lambda/)
- **[access control]** Use the `aws:SourceArn` and `aws:SourceAccount` global condition context keys in role trust policies that Fraud Detector assumes, to prevent the cross-service confused deputy problem. [doc](https://docs.aws.amazon.com/frauddetector/latest/ug/confused-deputy-prevention.html)
- **[access control]** Restrict access to specific Fraud Detector resources (or resources matching a tag) with identity-based policies instead of allowing full access, especially in multi-team or multi-application accounts. [doc](https://docs.aws.amazon.com/frauddetector/latest/ug/security_iam_id-based-policy-examples.html)
- **[data protection]** Encrypt Fraud Detector data at rest with a customer managed AWS KMS key via the `PutKMSEncryptionKey` API instead of relying only on the default service-managed key, when you need control over key rotation and access policy. [doc](https://docs.aws.amazon.com/frauddetector/latest/ug/key-management.html)
- **[data protection]** Note that changing or rotating the customer managed KMS key does not retroactively re-encrypt previously stored data, so plan key-rotation and compliance requirements before ingesting sensitive data. [doc](https://docs.aws.amazon.com/frauddetector/latest/ug/key-management.html)
- **[data protection]** Rely on Fraud Detector's default TLS 1.2 (or enforce TLS 1.3) encryption in transit, and use a FIPS endpoint if you require FIPS 140-3 validated cryptographic modules for API/CLI access. [doc](https://docs.aws.amazon.com/frauddetector/latest/ug/data-protection.html)
- **[data protection]** Never put confidential or sensitive values (such as customer email addresses) into tags or free-form text/name fields, since this data can surface in billing or diagnostic logs. [doc](https://docs.aws.amazon.com/frauddetector/latest/ug/data-protection.html)
- **[data protection]** Set up individual users with IAM Identity Center or IAM plus multi-factor authentication rather than sharing root or broad account credentials for teams operating Fraud Detector. [doc](https://docs.aws.amazon.com/frauddetector/latest/ug/data-protection.html)
- **[network security]** Use interface VPC endpoints (AWS PrivateLink) for Fraud Detector so API calls from your VPC do not traverse the public internet. [doc](https://docs.aws.amazon.com/frauddetector/latest/ug/vpc-interface-endpoints.html)
- **[network security]** Apply a restrictive VPC endpoint policy for the Fraud Detector interface endpoint instead of leaving the default full-access policy, if you need to limit which principals or actions can use the endpoint. [doc](https://docs.aws.amazon.com/frauddetector/latest/ug/vpc-interface-endpoints.html)

## 🛡️ Reliability
- **[monitoring]** Use Amazon CloudWatch to track Fraud Detector metrics in real time and set alarms so operational issues (for example prediction errors or throttling) are surfaced automatically. [doc](https://docs.aws.amazon.com/frauddetector/latest/ug/monitoring-overview.html)
- **[quotas]** Track the `GetEventPrediction` throughput quota (200 TPS default, adjustable) against expected production load and request an increase proactively rather than discovering throttling in production. [doc](https://docs.aws.amazon.com/frauddetector/latest/ug/limits.html)
- **[quotas]** Design payloads within the fixed 256 KB per-request and 5,000-input `GetEventPrediction` limits, since these are not adjustable, and batch or trim event variables accordingly. [doc](https://docs.aws.amazon.com/frauddetector/latest/ug/limits.html)
- **[quotas]** Account for the one-concurrent-training-job-per-model and three-concurrent-training-jobs-per-account limits when orchestrating retraining pipelines, to avoid failed or queued training requests. [doc](https://docs.aws.amazon.com/general/latest/gr/fraud-detector.html)
- **[data quality]** Allow event data to mature (weeks to months, depending on your chargeback/investigation cycle) before using it to train, and verify the data distribution across the training period isn't drifting, to avoid biased model evaluation. [doc](https://docs.aws.amazon.com/frauddetector/latest/ug/create-event-dataset.html)
- **[data quality]** Ensure training data is representative of the population the model will actually score in production (for example, include declined as well as approved historical applicants) so evaluated performance reflects real-world accuracy. [doc](https://docs.aws.amazon.com/frauddetector/latest/ug/create-event-dataset.html)
- **[operations]** Enable event ingestion in production so every scored event is retained and available for future model retraining, instead of scoring events without capturing them for continuous improvement. [doc](https://aws.amazon.com/blogs/machine-learning/detect-online-transaction-fraud-with-new-amazon-fraud-detector-features/)
- **[operations]** Continuously evaluate deployed model performance and prediction explanations after launch, and retrain with updated or additional data as fraud patterns evolve, instead of treating a trained model as permanently accurate. [doc](https://docs.aws.amazon.com/frauddetector/latest/ug/frauddetector-workflow.html)

## ⚡ Performance Efficiency
- **[data modeling]** Map event variables to the most specific variable type available (for example `IP_ADDRESS`, `EMAIL_ADDRESS`, `CARD_BIN`, `PHONE_NUMBER`) instead of generic types, so Fraud Detector applies the correct enrichments and derived features. [doc](https://aws.amazon.com/blogs/machine-learning/diagnose-model-performance-before-deployment-for-amazon-fraud-detector/)
- **[data modeling]** Map high-cardinality text fields (such as URLs or free-text descriptions) to `FREE_FORM_TEXT` rather than `CATEGORICAL`, so Fraud Detector extracts text features and embeddings instead of treating each value as an opaque category. [doc](https://aws.amazon.com/blogs/machine-learning/diagnose-model-performance-before-deployment-for-amazon-fraud-detector/)
- **[data modeling]** Include known high-value variable categories (email address, IP, payment type, billing/shipping address, device) when available, since these typically carry the strongest fraud signal. [doc](https://aws.amazon.com/blogs/machine-learning/diagnose-model-performance-before-deployment-for-amazon-fraud-detector/)
- **[model tuning]** Diagnose low AUC or unusable score thresholds by checking for insufficient records/fraud examples, missing values, or incorrect variable type mapping before assuming the use case isn't a good fit for ML. [doc](https://aws.amazon.com/blogs/machine-learning/diagnose-model-performance-before-deployment-for-amazon-fraud-detector/)
- **[model tuning]** Pick your rule score threshold by reviewing the model performance chart's true-positive/false-positive tradeoff at each threshold, rather than choosing a threshold arbitrarily. [doc](https://aws.amazon.com/blogs/machine-learning/catching-fraud-faster-by-building-a-proof-of-concept-in-amazon-fraud-detector/)
- **[cold start]** Use the Cold Start training option when you have limited historical data (as few as 100 events with 50 labeled fraud) instead of waiting to accumulate the standard minimum before launching a model. [doc](https://aws.amazon.com/blogs/machine-learning/overcome-the-machine-learning-cold-start-challenge-in-fraud-detection-using-amazon-fraud-detector/)
- **[evaluation]** Test a trained model against recent, non-overlapping data (up to 30 days old, at least 1,000–5,000 records over roughly two weeks) before deployment, so evaluation reflects real-world effectiveness rather than data already seen during training. [doc](https://aws.amazon.com/blogs/machine-learning/catching-fraud-faster-by-building-a-proof-of-concept-in-amazon-fraud-detector/)

## 💰 Cost Optimization
- **[training data]** Curate and clean training data quality (fix missing values, correct variable type mapping, mature labels) before triggering training, since bad-quality data produces low-performing models that require costly retraining cycles. [doc](https://aws.amazon.com/blogs/machine-learning/diagnose-model-performance-before-deployment-for-amazon-fraud-detector/)
- **[retraining]** Use `UpdateModelVersion` to retrain an existing model with refreshed training data (producing an incremental minor version) instead of always creating a brand-new model from scratch when only the underlying data has changed. [doc](https://docs.aws.amazon.com/frauddetector/latest/api/API_UpdateModelVersion.html)
- **[retraining]** Trigger retraining based on observed business or data drift (for example expansion into new geographies or new sign-up channels) rather than on an arbitrary fixed schedule, to avoid unnecessary training runs. [doc](https://aws.amazon.com/blogs/machine-learning/prevent-fake-account-sign-ups-in-real-time-with-ai-using-amazon-fraud-detector/)
- **[batch scoring]** Use batch prediction pipelines (for example S3 plus Lambda) for offline/non-real-time fraud scoring workloads instead of scaling for peak real-time throughput unnecessarily. [doc](https://aws.amazon.com/blogs/machine-learning/performing-batch-fraud-predictions-using-amazon-fraud-detector-amazon-s3-and-aws-lambda/)

## ⚙️ Operational Excellence
- **[infrastructure as code]** Define detectors, event types, outcomes, variables, labels, and entity types as AWS CloudFormation resources so Fraud Detector configuration is version-controlled and repeatable across environments instead of manually created. [doc](https://aws.amazon.com/blogs/machine-learning/manage-your-amazon-fraud-detector-resources-in-an-automated-and-secure-manner-using-aws-cloudformation/)
- **[rules]** Keep rule expressions simple boolean comparisons per Fraud Detector's rule language (no if-else logic, under 4,000 characters) and use outcomes to represent each distinct business decision (approve, review, decline), rather than encoding complex branching logic in a single rule. [doc](https://docs.aws.amazon.com/frauddetector/latest/ug/rules.html)
- **[logging]** Enable AWS CloudTrail to capture every Fraud Detector API call, including caller identity and source IP, so you can audit who created, modified, or deleted detectors, rules, and models. [doc](https://docs.aws.amazon.com/frauddetector/latest/ug/logging-using-cloudtrail.html)
- **[monitoring]** Combine CloudWatch metrics with CloudTrail logs as the standard operational monitoring pair for Fraud Detector, rather than relying on the console alone to notice issues. [doc](https://docs.aws.amazon.com/frauddetector/latest/ug/monitoring-overview.html)
- **[automation]** Automate model retraining triggers off upstream data events (for example new data landing in Amazon S3) so retraining pipelines run consistently instead of depending on manual initiation. [doc](https://aws.amazon.com/blogs/apn/amazon-fraud-detector-can-accelerate-how-ai-is-embedded-your-business/)
- **[planning]** Note that Amazon Fraud Detector is no longer open to new customers as of November 7, 2025; for new fraud-detection workloads, evaluate Amazon SageMaker, AutoGluon, or AWS WAF, and plan migration timelines for existing workloads accordingly. [doc](https://docs.aws.amazon.com/frauddetector/latest/ug/security-iam.html)
