# AWS IoT Greengrass — Best Practices

## Common scenarios
- Running local compute, messaging, and ML inference on edge/industrial devices        → Reliability, Performance Efficiency
- Managing fleets of edge core devices and pushing OTA component updates        → Operational Excellence, Reliability
- Bridging on-premises sensors/devices to AWS IoT Core and cloud services        → Security, Cost Optimization
- Operating disconnected or intermittently connected edge sites        → Reliability, Sustainability

## 🔒 Security
- **[IAM/component permissions]** Run Greengrass components as unprivileged users and avoid root unless absolutely necessary; also run the Greengrass Core software itself as a non-root user. — Limits blast radius if a component is compromised. [doc](https://docs.aws.amazon.com/greengrass/v2/developerguide/security-best-practices.html)
- **[IAM policies]** Scope the token exchange (device service) role and AWS IoT policies to a finite set of actions and resources instead of wildcards. — Enforces least privilege for cloud calls made from the edge. [doc](https://docs.aws.amazon.com/greengrass/v2/developerguide/security-best-practices.html)
- **[credentials]** Never hardcode credentials in user-defined components; use the secret manager component or the default credential provider chain instead. — Prevents credential leakage from device filesystems or logs. [doc](https://docs.aws.amazon.com/greengrass/v2/developerguide/security-best-practices.html)
- **[logging]** Avoid putting sensitive information or PII in MQTT topic paths, thing names/attributes, component names, or component logs. — Reduces exposure since local logs and CloudWatch Logs are still accessible with the right privileges. [doc](https://docs.aws.amazon.com/greengrass/v2/developerguide/security-best-practices.html)
- **[device clock]** Keep the core device clock synchronized. — X.509 certificate validity checks depend on accurate device time. [doc](https://docs.aws.amazon.com/greengrass/v2/developerguide/security-best-practices.html)
- **[TLS]** Disable legacy/weak cipher suites (e.g., CBC) on the device and rely on the latest TLS cipher suites. — Reduces the risk of downgrade or cryptographic attacks on device-to-cloud TLS sessions. [doc](https://docs.aws.amazon.com/greengrass/v2/developerguide/security-best-practices.html)
- **[device identity]** Protect X.509 private keys with a hardware security module or secure element via Greengrass Hardware Security Integration (HSI) rather than storing them on the filesystem. — Tamper-resistant key storage prevents key exfiltration and device impersonation. [doc](https://aws.amazon.com/blogs/apn/adding-secure-element-support-using-aws-iot-greengrass-hardware-security-integration-hsi/)
- **[incident response]** If you lose control of a core device, remove it from the group and rotate the group CA certificate to stop it from transmitting further data, and use full disk encryption if the device is vulnerable to physical theft. — Contains the impact of a compromised or stolen device. [doc](https://docs.aws.amazon.com/greengrass/v1/developerguide/vulnerability-analysis-and-management.html)
- **[network isolation]** Don't assume Greengrass isolates component network traffic — components can communicate with any other process on the system or network, so isolate untrusted components at the OS/network layer yourself. — Greengrass does not enforce this isolation for you. [doc](https://docs.aws.amazon.com/greengrass/v2/developerguide/vulnerability-analysis-and-management.html)
- **[device auditing]** Use AWS IoT Device Defender to audit core devices against security best practices and detect abnormal behavior. — Greengrass generates predictable client IDs that integrate directly with Device Defender detection. [doc](https://docs.aws.amazon.com/greengrass/v2/developerguide/vulnerability-analysis-and-management.html)

## 🛡️ Reliability
- **[deployments]** Use continuous deployments with a defined `failureHandlingPolicy` (`ROLLBACK` by default) so failed deployments automatically revert. — Prevents a bad rollout from leaving fleets in a broken state. [doc](https://docs.aws.amazon.com/greengrass/v2/APIReference/API_DeploymentPolicies.html)
- **[deployments]** Use component update and configuration validation policies to control when it's safe to update components and how long components get to validate new configuration. — Avoids applying changes while a component is mid-operation or before it can confirm the new config is valid. [doc](https://docs.aws.amazon.com/greengrass/v2/APIReference/API_DeploymentPolicies.html)
- **[offline operation]** Design components to continue local processing and buffering during network outages, since Greengrass core devices are built to operate for extended or indefinite disconnected periods. — Preserves mission-critical functionality when connectivity is unstable. [doc](https://aws.amazon.com/solutions/guidance/tactical-edge-application-deployment-on-aws/)
- **[high availability]** For mission-critical single-node workloads needing zero data loss, use an Active/Passive cluster pattern (e.g., with Pacemaker and block-level replication); for high-throughput workloads, use an Active/Active pattern with independent nodes and load balancing. — Matches the HA pattern to whether the workload needs strict data consistency or horizontal scalability. [doc](https://aws.amazon.com/blogs/iot/high-availability-patterns-for-aws-iot-greengrass-using-pacemaker/)
- **[software updates]** Keep core devices patched via regular OTA software and component updates rather than treating deployed devices as immutable. — Addresses vulnerabilities and maintains reliability even in devices deployed for years in the field. [doc](https://aws.amazon.com/solutions/guidance/tactical-edge-application-deployment-on-aws/)

## ⚡ Performance Efficiency
- **[edge processing]** Process, filter, and preprocess data locally on the core device (e.g., ML inference, video analytics) before sending only necessary results to the cloud. — Cuts latency for time-sensitive decisions and reduces the volume of data transferred back to AWS. [doc](https://aws.amazon.com/solutions/guidance/tactical-edge-application-deployment-on-aws/)
- **[hardware sizing]** Tailor edge hardware and select only the components you actually need for the deployment rather than a one-size-fits-all image. — Right-sizes compute/memory/storage to the workload and avoids wasted resource utilization. [doc](https://aws.amazon.com/solutions/guidance/tactical-edge-application-deployment-on-aws/)
- **[monitoring]** Use AWS Systems Manager and Greengrass telemetry to proactively monitor edge performance and catch issues without shipping raw data back to the cloud. — Enables performance tuning at the edge itself. [doc](https://aws.amazon.com/solutions/guidance/tactical-edge-application-deployment-on-aws/)

## 💰 Cost Optimization
- **[component selection]** Deploy only the components required for a given core device or group instead of a common bundle for the whole fleet. — Minimizes unnecessary compute, storage, and data-transfer consumption per device. [doc](https://aws.amazon.com/solutions/guidance/tactical-edge-application-deployment-on-aws/)
- **[data transfer]** Process and analyze data at the edge so only necessary, pre-processed data is transmitted to the cloud, especially over expensive links (e.g., satellite/cellular). — Directly reduces network egress and cloud ingestion costs. [doc](https://aws.amazon.com/solutions/guidance/tactical-edge-application-deployment-on-aws/)
- **[hardware right-sizing]** Right-size edge hardware platforms to the specific workload rather than over-provisioning. — Avoids paying for unused compute/storage capacity across a fleet of devices. [doc](https://aws.amazon.com/solutions/guidance/tactical-edge-application-deployment-on-aws/)

## ⚙️ Operational Excellence
- **[logging]** Enable CloudWatch Logs for the Greengrass Core software and components, in addition to local file system logs, so you can troubleshoot core devices remotely without physical access. — Centralizes visibility across a distributed fleet. [doc](https://docs.aws.amazon.com/greengrass/v2/developerguide/monitor-logs.html)
- **[monitoring plan]** Before deploying, define a monitoring plan covering goals, which resources to monitor, monitoring frequency, tools, and who is notified on failures. — Ensures multi-point failures across a distributed IoT solution can be debugged systematically. [doc](https://docs.aws.amazon.com/greengrass/v2/developerguide/logging-and-monitoring.html)
- **[log management]** Use the log manager component to control per-component log levels, disk space limits, and CloudWatch upload behavior. — Prevents local disk exhaustion on constrained edge devices while retaining cloud visibility. [doc](https://aws.amazon.com/blogs/iot/developing-a-remote-job-monitoring-application-at-the-edge-using-aws-iot-greengrass-part-2/)
- **[deployment health]** Configure deployment and component health status notifications and check core device status regularly. — Surfaces failed or degraded deployments quickly across large fleets. [doc](https://docs.aws.amazon.com/greengrass/v2/developerguide/logging-and-monitoring.html)
- **[auditing]** Log AWS IoT Greengrass V2 API calls with AWS CloudTrail. — Provides an audit trail for fleet management and deployment actions. [doc](https://docs.aws.amazon.com/greengrass/v2/developerguide/logging-and-monitoring.html)

## 🌱 Sustainability
- **[device longevity]** Use component-based OTA updates to keep core devices flexible and useful for future use cases instead of retiring hardware tied to a single immutable purpose. — Extends device durability and lifespan, reducing e-waste from premature hardware replacement. [doc](https://aws.amazon.com/blogs/architecture/optimizing-your-iot-devices-for-environmental-sustainability/)
- **[edge processing]** Filter and process data locally so only necessary results are transmitted to the cloud. — Reduces network and downstream cloud resource consumption associated with transferring and storing raw data. [doc](https://aws.amazon.com/solutions/guidance/tactical-edge-application-deployment-on-aws/)
