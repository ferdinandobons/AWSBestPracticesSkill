# AWS IoT ExpressLink — Best Practices

## Common scenarios
- Adding secure, pre-certified cloud connectivity to a new microcontroller-based product without in-house crypto/networking expertise        → Security, Operational Excellence
- Onboarding a manufactured device fleet to a customer/OEM AWS account only when the end-user activates the product        → Security, Reliability
- Keeping module firmware and host application firmware current with security patches via OTA        → Reliability, Operational Excellence
- Detecting misbehaving or compromised devices in the field using built-in connectivity/health metrics        → Security, Operational Excellence

## 🔒 Security
- **[Data in transit]** Rely on the module's built-in TLS 1.2+ encrypted channel to AWS IoT Core, and do not attempt to disable or downgrade it — it's the mechanism that protects device-to-cloud traffic and credentials in transit. [doc](https://aws.amazon.com/iot-expresslink/features/)
- **[Host-to-module link]** Add your own end-to-end message encryption between the host processor and the module if sensitive data crosses the UART and an attacker could gain physical access to the device — the serial interface itself is unencrypted even though cloud communication is. [doc](https://docs.aws.amazon.com/iot-expresslink/archive/v1.0/programmersguide/elpg-commands.html)
- **[Identity]** Use the module's pre-provisioned, cryptographically signed unique certificate as the device's root identity rather than generating or injecting your own credentials — this avoids the cost and risk of custom certificate generation, loading, and tracking at manufacturing time. [doc](https://aws.amazon.com/blogs/apn/how-cardinal-peak-harnessed-aws-iot-expresslink-to-speed-implementation-across-the-development-lifecycle/)
- **[Secure boot/updates]** Depend on the module's hardware root of trust, secure boot, and manufacturer-signed OTA firmware updates rather than building a parallel update-integrity mechanism — these are AWS-mandated and qualification-tested for every listed module. [doc](https://aws.amazon.com/blogs/iot/introducing-aws-iot-expresslink-making-it-faster-and-easier-to-develop-secure-iot-devices/)
- **[Host OTA integrity]** When distributing host-processor firmware through the module's OTA/HOTA channel, verify the transferred file's integrity and authenticity with your own signature before applying it — the module transports the file but the host is responsible for validating it. [doc](https://aws.amazon.com/blogs/iot/introducing-aws-iot-expresslink-making-it-faster-and-easier-to-develop-secure-iot-devices/)
- **[Supply chain]** Use the onboarding-by-claim flow (staging account plus late binding at end-user registration) instead of sharing OEM account credentials with contract manufacturers or other supply-chain parties — this removes the need to disclose confidential onboarding secrets to intermediaries. [doc](https://docs.aws.amazon.com/iot-expresslink/latest/oemonboardingguide/oemog.html)
- **[Forced updates]** Reserve the `<force:YES>` module OTA override for genuinely critical security fixes, and design the host application to normally control update timing/acceptance — a forced update wipes the module's OTA buffer, including any host payload occupying it. [doc](https://docs.aws.amazon.com/iot-expresslink/archive/v1.2/programmersguide/elpg-ota-updates.html)

## 🛡️ Reliability
- **[Health monitoring]** Enable AWS IoT Device Defender metric reporting (`DefenderPeriod`/`Defender` configuration parameter) so bytes sent, message counts, reconnect events, and hard-reset events are reported at a regular interval — this gives visibility into connectivity health without custom instrumentation. [doc](https://docs.aws.amazon.com/iot-expresslink/latest/programmersguide/elpg-iot-services.html)
- **[Event-driven design]** Use the module's asynchronous Event output pin (or poll the event queue) to detect disconnects and incoming messages promptly, rather than relying only on synchronous command responses — this is the documented mechanism for surfacing connectivity and message events to the host. [doc](https://docs.aws.amazon.com/iot-expresslink/archive/v1.2/programmersguide/elpg-hardware.html)
- **[Update sequencing]** Follow the documented OTA job flow (prerequisites, then create the firmware update job, then monitor and apply) rather than pushing firmware ad hoc — this keeps host and module firmware versions from drifting into an incompatible state. [doc](https://docs.aws.amazon.com/iot-expresslink/latest/gettingstartedguide/elgsg-setup-ota-update.html)
- **[Device Shadow]** Use AWS IoT Device Shadow to track and reconcile desired vs. reported device state across disconnects — this lets applications read and modify device state even while the device is offline. [doc](https://aws.amazon.com/iot-expresslink/features/)

## ⚙️ Operational Excellence
- **[Fleet onboarding]** Use onboarding-by-claim with just-in-time provisioning (JITP) so devices are only bound to a customer/OEM "thing" when the end-user registers the product — this defers and streamlines onboarding instead of provisioning every unit during manufacturing. [doc](https://docs.aws.amazon.com/iot-expresslink/latest/oemonboardingguide/oemog.html)
- **[Fleet-wide patching]** Use the AWS IoT Device Management console to push OTA updates to both the module firmware and the host processor firmware across the fleet — this centralizes security-patch and feature rollout instead of per-device manual updates. [doc](https://aws.amazon.com/iot-expresslink/features/)
- **[Anomaly response]** Feed Device Defender custom metrics (bytes out, messages sent/received, reconnect and hard-reset flags) into your monitoring/alerting pipeline to detect anomalies and misbehaving devices at fleet scale. [doc](https://docs.aws.amazon.com/iot-expresslink/latest/programmersguide/elpg-iot-services.html)
- **[Command interface]** Build host application logic against the documented AT command set (CONNECT, SEND, SUBSCRIBE, CONF, EVENT, OTA, SHADOW) rather than undocumented behavior — this keeps the host portable across AWS Partner modules that implement the same ExpressLink specification. [doc](https://docs.aws.amazon.com/iot-expresslink/archive/v1.2/programmersguide/elpg.html)
