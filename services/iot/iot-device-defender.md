# AWS IoT Device Defender — Best Practices

## Common scenarios
- Continuously auditing device certificates, IoT policies, and Cognito roles for drift from security best practices        → Security, Operational Excellence
- Detecting compromised or misbehaving devices via cloud-side and device-side behavior metrics        → Security, Reliability
- Automatically remediating findings (quarantine, certificate rotation, policy replacement) at fleet scale        → Security, Operational Excellence
- Feeding audit/detect alerts into CloudWatch and SNS for security dashboards and incident response        → Operational Excellence

## 🔒 Security
- **[Least privilege]** Treat AWS IoT policy overly-permissive and misconfiguration audit findings as action items — replace wildcard `Resource`/`Action` device policies with policies scoped to the specific things, topics, and actions each device needs. [doc](https://aws.amazon.com/blogs/iot/automating-security-remediation-using-aws-iot-device-defender/)
- **[Identity]** Enable the device-certificate-shared check and remediate immediately — multiple devices using the same X.509 identity defeats per-device revocation and traceability. [doc](https://docs.aws.amazon.com/iot-device-defender/latest/devguide/device-defender-audit-checks.html)
- **[Certificate hygiene]** Enable CA and device certificate expiring, key-quality, and revoked-but-still-active checks — weak keys and stale or revoked certificates are otherwise found only after they're exploited. [doc](https://docs.aws.amazon.com/iot-device-defender/latest/devguide/device-defender-audit-checks.html)
- **[Logging]** Enable the logging-disabled audit check and act on it — AWS IoT logging must be on for Device Defender and CloudTrail to have the visibility needed to investigate findings and alarms. [doc](https://docs.aws.amazon.com/iot-device-defender/latest/devguide/device-defender-audit-checks.html)
- **[IAM]** Start from AWS managed policies for Device Defender and Access Analyzer, then narrow to customer-managed policies — this gets least-privilege scoping without hand-authoring every permission from scratch. [doc](https://docs.aws.amazon.com/iot-device-defender/latest/devguide/security_iam_id-based-policy-examples.html)
- **[IAM]** Use IAM Access Analyzer to validate Device Defender-related IAM policies and generate least-privilege permissions for roles (e.g. mitigation-action execution roles) — it derives scoped policies from observed CloudTrail activity instead of guesswork. [doc](https://docs.aws.amazon.com/iot-device-defender/latest/devguide/security_iam_id-based-policy-examples.html)
- **[Defense in depth]** Combine Device Defender audits with a documented device identity/credential strategy (unique identity per device) — audits alone don't establish identity uniqueness, they only detect its absence. [doc](https://aws.amazon.com/blogs/mt/improve-your-iot-security-posture-using-aws/)
- **[Behavior detection]** Group devices with similar expected behavior into thing groups and attach security profiles per group — a single fleet-wide baseline blurs distinct normal-behavior patterns and produces noisier Detect alarms. [doc](https://aws.amazon.com/blogs/iot/use-aws-iot-device-defender-to-detect-statistical-anomalies-and-to-visualize-your-device-security-metrics/)
- **[Behavior detection]** Observe device metrics (via `additionalMetricsToRetain`, no behaviors defined) before authoring alarm thresholds, then validate behaviors with `ValidateSecurityProfileBehaviors` — this avoids attaching untested thresholds that generate false-positive alarms fleet-wide. [doc](https://docs.aws.amazon.com/iot-device-defender/latest/devguide/detect-HowToHowTo.html)

## 🛡️ Reliability
- **[Alerting capacity]** Account for SNS topic quotas when routing Detect alarms — a large burst of simultaneous violations can exceed account SNS limits, so also monitor via CloudWatch metrics as a non-throttled alternative. [doc](https://docs.aws.amazon.com/iot-device-defender/latest/devguide/detect-HowToHowTo.html)
- **[Contextual accuracy]** Rely on Device Defender's integration with AWS IoT Core device-interaction context for audit checks and behavior evaluation — it reduces false positives compared to building parallel custom checks without that context. [doc](https://docs.aws.amazon.com/iot-device-defender/latest/devguide/what-is-device-defender.html)
- **[Mitigation readiness]** Pre-configure mitigation actions (quarantine thing group, deactivate certificate, replace default policy version) before you need them — this lets findings be remediated immediately instead of during an active incident. [doc](https://docs.aws.amazon.com/iot-device-defender/latest/devguide/dd-mitigation-actions.html)

## ⚙️ Operational Excellence
- **[Scheduling]** Run audits on a recurring schedule (daily/weekly/monthly depending on check type) instead of relying only on one-off on-demand audits — this is what catches configuration drift continuously. [doc](https://docs.aws.amazon.com/cdk/api/v2/docs/aws-iot-alpha-readme.html)
- **[Coverage]** Enable the full set of supported audit checks (certificate, policy, Cognito role, role alias, client ID, and logging checks) rather than a partial subset — each check targets a distinct class of misconfiguration that the others don't cover. [doc](https://docs.aws.amazon.com/iot-device-defender/latest/devguide/device-defender-audit-checks.html)
- **[Automated remediation]** Map each audit check to its supported mitigation actions and automate the response (e.g. auto-quarantine on device-certificate-shared, auto-replace policy on overly-permissive-policy) — this scales remediation beyond manual triage of every finding. [doc](https://docs.aws.amazon.com/iot-device-defender/latest/devguide/dd-mitigation-actions.html)
- **[Alert routing]** Publish audit findings and Detect alarms to CloudWatch and SNS, and integrate them into existing security dashboards and incident-response workflows — this surfaces issues faster than checking the console manually. [doc](https://docs.aws.amazon.com/iot-device-defender/latest/devguide/what-is-device-defender.html)
- **[Roadmap awareness]** Note that AWS IoT Device Defender Detect (Rules Detect and ML Detect) will no longer be available to new customers after August 31, 2026 — plan new anomaly-detection designs accordingly, though Audit remains unaffected. [doc](https://docs.aws.amazon.com/iot-device-defender/latest/devguide/detect-HowToHowTo.html)
- **[Program design]** Treat Device Defender auditing and detection as one control within a broader IoT security baseline (unique identities, least-privilege access, logging, certificate rotation, alerting) — it is not a complete security program on its own. [doc](https://aws.amazon.com/blogs/mt/improve-your-iot-security-posture-using-aws/)
