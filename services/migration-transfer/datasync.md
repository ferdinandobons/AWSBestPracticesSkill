# AWS DataSync — Best Practices

## Common scenarios
- One-time or large-scale migration of file/object data from on-premises to S3, EFS, or FSx        → Performance Efficiency, Reliability
- Recurring/incremental replication for hybrid workflows, analytics ingestion, or DR        → Reliability, Operational Excellence
- Scheduled or continuous sync between AWS storage services (S3-to-S3, cross-account, cross-Region)        → Operational Excellence, Cost Optimization
- Bulk transfer of millions of small files or multi-petabyte datasets requiring parallelization        → Performance Efficiency

## 🔒 Security
- **[Network]** Use VPC endpoints powered by AWS PrivateLink for your DataSync agent — this keeps traffic between the agent and the DataSync service off the public internet and removes the need for public IPs. [doc](https://aws.amazon.com/datasync/faqs/)
- **[Network]** Configure source and destination locations with the highest level of security each supports and limit subnet traffic to known protocols — this reduces the attack surface exposed during transfers. [doc](https://docs.aws.amazon.com/datasync/latest/userguide/internetwork-traffic-privacy.html)
- **[Transport]** Require TLS 1.2 at minimum, and TLS 1.3 where possible, with PFS cipher suites (DHE/ECDHE) for any client accessing the DataSync API — this prevents downgrade attacks and protects past sessions if a key is later compromised. [doc](https://docs.aws.amazon.com/datasync/latest/userguide/infrastructure-security.html)
- **[IAM]** Grant IAM identities only the specific DataSync actions and resource ARNs they need instead of broad wildcard access — this limits blast radius if credentials are compromised. [doc](https://docs.aws.amazon.com/datasync/latest/userguide/using-identity-based-policies.html)
- **[IAM]** For an EFS location, create a dedicated IAM role that only DataSync can assume and grant access through the EFS file system policy rather than broad IAM permissions — DataSync mounts EFS as the root user, so scoping access at the file-system layer contains that privilege. [doc](https://aws.amazon.com/blogs/storage/using-available-amazon-efs-security-features-while-migrating-files-with-aws-datasync/)
- **[Secrets]** Store credentials such as SAS tokens or storage-system passwords in AWS Secrets Manager instead of embedding them in configuration or source code — this keeps them encrypted at rest and makes access auditable via CloudTrail. [doc](https://aws.amazon.com/blogs/migration-and-modernization/how-to-migrate-from-azure-blob-storage-to-amazon-s3-using-agentless-aws-datasync/)
- **[Security groups]** Scope the security group used by a DataSync location to only the required ports (2049 for EFS, 445 for SMB/FSx) and restrict source IPs to the agent or VPC endpoint — this prevents unrelated hosts from reaching your storage system through the DataSync path. [doc](https://aws.amazon.com/blogs/storage/automate-online-data-transfers-with-aws-datasync-and-aws-cloudformation/)
- **[Auditability]** Enable CloudTrail for DataSync API calls alongside Secrets Manager and storage-service access — this gives you a complete audit trail for compliance and incident response. [doc](https://aws.amazon.com/blogs/migration-and-modernization/how-to-migrate-from-azure-blob-storage-to-amazon-s3-using-agentless-aws-datasync/)

## 🛡️ Reliability
- **[Availability Zones]** Plan for task failure if the source or destination Availability Zone becomes unavailable — DataSync tasks fail rather than automatically fail over, so build retry/restart logic for critical transfers. [doc](https://docs.aws.amazon.com/datasync/latest/userguide/disaster-recovery-resiliency.html)
- **[Agents]** Don't rely on multiple agents assigned to one location for high availability — all agents for a location must be online before a task can run, so deploy standby agents you can manually activate rather than expecting automatic failover. [doc](https://docs.aws.amazon.com/datasync/latest/userguide/do-i-need-datasync-agent.html)
- **[Task design]** Partition very large datasets across multiple tasks, each with its own agent, instead of one giant task — this contains the blast radius of a failure to a single partition and makes retries faster and cheaper. [doc](https://docs.aws.amazon.com/datasync/latest/userguide/do-i-need-datasync-agent.html)
- **[Verification]** Configure data-integrity verification appropriate to your risk tolerance — this confirms destination data matches the source after every transfer instead of assuming success. [doc](https://aws.amazon.com/datasync/features/)
- **[Proof of concept]** Run a small-scale proof of concept with a representative data subset before a large migration — this validates throughput, source-system impact, and permission/metadata preservation before you commit to the full transfer. [doc](https://docs.aws.amazon.com/datasync/latest/userguide/datasync-large-migration-poc.html)

## ⚡ Performance Efficiency
- **[Scale-out]** Partition large datasets across multiple parallel tasks, one agent per task, when you have millions of files or more bandwidth than a single task can consume — this reduces overall migration time by using more of your available network capacity. [doc](https://aws.amazon.com/blogs/storage/how-to-accelerate-your-data-transfers-with-aws-datasync-scale-out-architectures/)
- **[Scale-out]** Connect up to four agents to a single location when transferring millions of small files — this increases parallelization and throughput, though it doesn't shorten the task's prepare phase. [doc](https://docs.aws.amazon.com/datasync/latest/userguide/do-i-need-datasync-agent.html)
- **[Filtering]** Use include/exclude filters or a manifest to limit what DataSync scans and transfers — this reduces prepare-phase time and unnecessary I/O against source storage. [doc](https://aws.amazon.com/datasync/faqs/)
- **[Bandwidth]** Set or adjust the built-in bandwidth throttle, even on a running task execution — this lets you balance transfer speed against impact on other users of the same network link. [doc](https://docs.aws.amazon.com/datasync/latest/userguide/configure-bandwidth.html)
- **[Monitoring]** Track data throughput, file throughput, and processing rate via CloudWatch metrics or the console during execution — this surfaces bottlenecks and confirms you're using available bandwidth efficiently. [doc](https://docs.aws.amazon.com/help-panel/datasync/latest/console/hp-task-execution-performance-basic.html)
- **[Source impact]** Configure a bandwidth limit when the source storage system also serves production traffic — DataSync's agent reads/writes otherwise compete with other clients for I/O and can degrade their response times. [doc](https://aws.amazon.com/datasync/faqs/)

## 💰 Cost Optimization
- **[S3 storage class]** Choose the S3 storage class deliberately at location creation — new objects use the class you specify, and classes like Standard-IA or Glacier can add retrieval/overwrite charges depending on access patterns. [doc](https://docs.aws.amazon.com/datasync/latest/userguide/create-s3-location.html)
- **[S3 requests]** Disable object-tag copying when tags aren't needed — copying tags generates additional billable S3 API requests with no benefit if you don't use them. [doc](https://docs.aws.amazon.com/datasync/latest/userguide/multipart-upload-policy.html)
- **[S3 multipart uploads]** Configure an S3 lifecycle policy to clean up incomplete multipart uploads — failed or interrupted DataSync uploads can otherwise leave partial uploads that quietly accrue storage charges. [doc](https://docs.aws.amazon.com/datasync/latest/userguide/multipart-upload-policy.html)
- **[Verification]** Configure task verification to check only transferred data rather than doing a full source/destination comparison where appropriate — this minimizes S3 data-retrieval fees incurred during integrity checks. [doc](https://aws.amazon.com/datasync/faqs/)
- **[Egress]** Deploy the DataSync agent in the source public cloud environment rather than in AWS when migrating from another cloud — DataSync compresses data in flight, and this placement can reduce egress fees from the source provider. [doc](https://aws.amazon.com/datasync/faqs/)

## ⚙️ Operational Excellence
- **[Scheduling]** Use DataSync's built-in task scheduling (hourly, daily, weekly, or cron/rate expressions) for recurring transfers — this removes the need to write and maintain custom scripts for repeated transfers. [doc](https://aws.amazon.com/datasync/faqs/)
- **[Monitoring]** Enable DataSync logging to CloudWatch Logs and use CloudWatch Events/EventBridge to trigger notifications on task completion — this alerts operational teams without requiring console polling. [doc](https://aws.amazon.com/blogs/storage/protect-your-file-and-backup-archives-using-aws-datasync-and-amazon-s3-glacier/)
- **[Monitoring]** Use CloudWatch Logs Insights queries against DataSync task execution logs — this lets you analyze transfer patterns and identify failed file-level transfers after a task completes. [doc](https://aws.amazon.com/blogs/storage/monitor-your-aws-datasync-migrations-using-amazon-cloudwatch/)
- **[Monitoring]** Create CloudWatch alarms on task metrics like `BytesTransferred` — this detects when actual throughput falls well below the configured bandwidth limit so you can surface bottlenecks early. [doc](https://aws.amazon.com/blogs/storage/monitor-your-aws-datasync-migrations-using-amazon-cloudwatch/)
- **[Auditing]** Use DataSync task reports and feed them into AWS Glue/Athena/QuickSight — this gives detailed, per-execution auditing of files transferred, skipped, verified, and deleted at scale. [doc](https://aws.amazon.com/datasync/faqs/)
- **[Automation]** Automate agent, location, and task creation with AWS CloudFormation — this makes transfer configurations repeatable and version-controlled instead of relying on manual console setup. [doc](https://aws.amazon.com/blogs/storage/automate-online-data-transfers-with-aws-datasync-and-aws-cloudformation/)
