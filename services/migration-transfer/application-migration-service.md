# AWS Application Migration Service — Best Practices

## Common scenarios
- Lift-and-shift rehosting of physical, virtual, or cloud servers to AWS        → Reliability, Operational Excellence
- Large-scale wave-based datacenter migrations with minimal cutover downtime        → Reliability, Operational Excellence
- Cross-Region or cross-AZ disaster recovery and continuous replication        → Reliability
- Post-migration modernization (OS/license conversion, patching, right-sizing)        → Cost Optimization, Performance Efficiency

## 🔒 Security
- **[Network access]** Restrict replication server access to only the source servers' CIDR range using security group rules — prevents unauthorized access to staging infrastructure during migration. [doc](https://docs.aws.amazon.com/mgn/latest/ug/security.html)
- **[Security groups]** Keep the default "Always use Application Migration Service security group" option enabled — MGN continuously monitors and auto-remediates this group to keep TCP port 1500 open only as needed for replication. [doc](https://aws.amazon.com/blogs/mt/aws-application-migration-service-best-practices/)
- **[Post-migration exposure]** After cutover, expose only the ports required by the application to the public internet and close unused ones — reduces the attack surface of migrated instances. [doc](https://docs.aws.amazon.com/mgn/latest/ug/security.html)
- **[OS hardening]** Keep OS packages and application software up to date and free of known vulnerabilities, and run only necessary services — hardening responsibility sits with the customer under the shared responsibility model. [doc](https://docs.aws.amazon.com/mgn/latest/ug/security.html)
- **[DDoS protection]** Enable AWS Shield on the account used for migration — protects replication servers and migrated instances from denial-of-service risk. [doc](https://docs.aws.amazon.com/mgn/latest/ug/security.html)
- **[IAM]** Grant least-privilege access using the predefined managed policies (e.g., read-only, agent-install-only, EC2-launch-only) instead of full access for every user — limits blast radius of compromised credentials. [doc](https://docs.aws.amazon.com/mgn/latest/ug/mgn-initialize-console.html)
- **[Data in transit]** Rely on MGN's default TLS 1.2 encryption for all data replicated from source servers to AWS, and use private connectivity (VPN, Direct Connect, VPC peering) to avoid traversing the public internet where required — protects data confidentiality during replication. [doc](https://aws.amazon.com/application-migration-service/faqs/)
- **[Data at rest]** Enable default EBS encryption in the target Region so MGN base snapshots and replicated volumes are encrypted at rest, using your own KMS keys where required — ensures encrypted storage without per-server configuration. [doc](https://docs.aws.amazon.com/mgn/latest/ug/AWS-Related-FAQ.html)

## 🛡️ Reliability
- **[Bandwidth planning]** Provision dedicated, constant bandwidth sized from the cutover date backward, accounting for total data volume, source disk write speed, and I/O — insufficient bandwidth risks MGN entering a rescan state that stalls replication. [doc](https://aws.amazon.com/blogs/mt/aws-application-migration-service-best-practices/)
- **[Replication server sizing]** Use a dedicated replication server for write-intensive source servers instead of the shared default — prevents one busy server's replication from degrading others. [doc](https://aws.amazon.com/blogs/mt/aws-application-migration-service-best-practices/)
- **[SLAs and RTO/RPO]** Define SLAs, recovery point objective (RPO), and recovery time objective (RTO) targets for each application before migration starts — anchors the disaster recovery strategy selected for the workload. [doc](https://docs.aws.amazon.com/wellarchitected/latest/migration-lens/best-practices-arranged-by-pillars.html)
- **[Service quotas]** Be aware of MGN service quota limits (such as concurrent replicating servers) and account for them when planning wave size and velocity — avoids migration slippage from unexpected throttling. [doc](https://docs.aws.amazon.com/mgn/latest/ug/MGN-service-limits.html)
- **[Network resilience]** Verify links and equipment connecting on-premises to AWS are highly available, and test network performance and component failure before migration — surfaces weaknesses before they impact a cutover window. [doc](https://docs.aws.amazon.com/wellarchitected/latest/migration-lens/best-practice-arranged-by-migration-phase.html)
- **[Test cutover]** Complete a full test cutover at least two weeks before the actual cutover date — gives time to identify and resolve issues before the real migration event. [doc](https://aws.amazon.com/blogs/mt/aws-application-migration-service-best-practices/)
- **[Wave planning]** Start with a small wave of low-priority applications to build team experience, then increase wave size and criticality in later waves — reduces risk of large-scale failure early in the migration program. [doc](https://aws.amazon.com/blogs/mt/aws-application-migration-service-best-practices/)
- **[Backups]** Avoid replicating database backup instances directly — backup workloads generate heavy write activity that slows replication and can jeopardize data consistency. [doc](https://aws.amazon.com/blogs/mt/aws-application-migration-service-best-practices/)

## ⚡ Performance Efficiency
- **[Replication instance type]** Keep the default replication server instance type unless servers are replicating slowly or experiencing frequent spikes, then move to m5a.xlarge or higher — matches replication capacity to actual demand instead of over-provisioning by default. [doc](https://aws.amazon.com/blogs/mt/aws-application-migration-service-best-practices/)
- **[Bandwidth throttling]** Use network bandwidth throttling per server when replication must share limited bandwidth with production traffic — prevents replication from starving other workloads while still controlling transfer rate over TCP port 1500. [doc](https://docs.aws.amazon.com/mgn/latest/ug/replication-server-settings.html)
- **[Right-sizing]** Let MGN's right-sizing feature select the best-match EC2 instance type for each source server rather than choosing manually per server — matches target compute to actual source server profile at scale. [doc](https://aws.amazon.com/blogs/mt/aws-application-migration-service-best-practices/)
- **[Baseline testing]** Run a control bandwidth test between the target Region and the Region nearest the source workload before migration — establishes a realistic theoretical upper bandwidth limit for planning. [doc](https://aws.amazon.com/blogs/mt/aws-application-migration-service-best-practices/)

## 💰 Cost Optimization
- **[Replication instance cost]** Prefer AMD-based instance types (e.g., m5a) when a larger replication server is needed — keeps replication compute costs lower while meeting performance requirements. [doc](https://aws.amazon.com/blogs/mt/aws-application-migration-service-best-practices/)
- **[Dedicated replication servers]** Reserve the "Use dedicated Replication Server" option for source servers that genuinely need it — this option increases EC2 cost during replication, so apply it selectively rather than account-wide. [doc](https://aws.amazon.com/blogs/mt/aws-application-migration-service-best-practices/)
- **[Right-sizing]** Use MGN's automatic right-sizing for launch instance types instead of over-provisioning target instances to match source hardware — avoids paying for unused capacity post-migration. [doc](https://aws.amazon.com/blogs/mt/aws-application-migration-service-best-practices/)
- **[Lightweight replication]** Rely on MGN's lightweight, auto-scaled replication instances rather than provisioning full source-equivalent compute during replication — eliminates paying for standby capacity that mirrors the source environment. [doc](https://aws.amazon.com/solutions/guidance/migrating-vmware-virtual-machines-to-amazon-ec2/)

## ⚙️ Operational Excellence
- **[Template setup order]** Configure the Replication, Launch, and Post-Launch templates at the account level before installing any replication agents — agents inherit settings only at install time, so changing templates afterward requires updating each server individually. [doc](https://aws.amazon.com/blogs/mt/aws-application-migration-service-best-practices/)
- **[Monitoring and alerting]** Configure Amazon EventBridge rules for MGN lifecycle events (e.g., data replication stalled) with Amazon SNS notifications, and use CloudWatch log groups and metric filters for ongoing visibility — surfaces replication issues automatically instead of relying on manual checks. [doc](https://aws.amazon.com/blogs/mt/aws-application-migration-service-best-practices/)
- **[Application and wave grouping]** Group source servers into Applications and Waves, and avoid scheduling two critical applications from the same application owner in one cutover window — keeps cutover execution manageable for the teams involved. [doc](https://docs.aws.amazon.com/mgn/latest/ug/waves.html)
- **[Post-launch automation]** Define post-launch actions (patching, SSM Agent installation, custom automation runbooks) at the account-level post-launch template before agent installation — automates post-migration configuration consistently across all servers in a wave. [doc](https://aws.amazon.com/blogs/mt/migrating-and-automating-patching-at-scale-with-aws-application-migration-service/)
- **[Runbooks]** Define and automate runbooks for migration operations and communicate them to all involved teams — reduces reliance on tribal knowledge during time-sensitive cutover events. [doc](https://docs.aws.amazon.com/wellarchitected/latest/migration-lens/best-practices-arranged-by-pillars.html)
- **[Test validation]** Launch test instances and validate connectivity and application function before every cutover — confirms the migrated server behaves correctly in AWS prior to the production switchover. [doc](https://aws.amazon.com/blogs/storage/migrate-from-google-cloud-platform-gcp-to-aws-using-aws-application-migration-service/)

## 🌱 Sustainability
- **[Right-sized launches]** Use MGN launch settings to size test and cutover instances to actual source workload requirements rather than over-provisioning — reduces unnecessary compute footprint of the migrated environment. [doc](https://aws.amazon.com/solutions/guidance/migrating-vmware-virtual-machines-to-amazon-ec2/)
- **[Efficient replication]** Rely on MGN's auto-scaled, lightweight replication instances instead of provisioning full source-equivalent compute for the duration of replication — minimizes the environmental impact of the migration process itself. [doc](https://aws.amazon.com/solutions/guidance/migrating-vmware-virtual-machines-to-amazon-ec2/)
