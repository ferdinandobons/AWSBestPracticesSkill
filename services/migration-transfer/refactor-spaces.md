# AWS Migration Hub Refactor Spaces — Best Practices

## Common scenarios
- Incrementally decomposing a monolith into microservices using the Strangler Fig pattern        → Reliability, Operational Excellence
- Extending an existing application with new features built as separate microservices        → Reliability, Operational Excellence
- Bridging networking across multiple AWS accounts (legacy app, new services, environment owner) for gradual cutover        → Security, Reliability
- Routing traffic to newly migrated services via AWS MGN post-launch actions after a lift-and-shift migration        → Reliability, Operational Excellence

## 🔒 Security
- **[account structure]** Deploy Refactor Spaces using a multi-account approach (separate accounts for the legacy monolith, new microservices, and the environment owner) to establish stronger security boundaries, reduce deployment coupling, and limit the blast radius of any one team's service. [doc](https://aws.amazon.com/blogs/apn/using-cast-and-aws-migration-hub-refactor-spaces-to-modernize-legacy-applications/)
- **[least privilege]** Start from the AWS managed policies (`AWSMigrationHubRefactorSpacesFullAccess`) and then define customer managed policies scoped to specific users' actual tasks instead of granting broad access long-term. [doc](https://docs.aws.amazon.com/migrationhub-refactor-spaces/latest/userguide/security_iam_id-based-policy-examples.html)
- **[conditions]** Add IAM policy conditions (for example requiring SSL/TLS, or scoping access to calls made through a specific AWS service like CloudFormation) to further restrict what Refactor Spaces identity-based policies allow. [doc](https://docs.aws.amazon.com/migrationhub-refactor-spaces/latest/userguide/security_iam_id-based-policy-examples.html)
- **[policy validation]** Validate new and existing Refactor Spaces IAM policies with IAM Access Analyzer to catch overly permissive grants and confirm policies adhere to IAM best practices before attaching them. [doc](https://docs.aws.amazon.com/migrationhub-refactor-spaces/latest/userguide/security_iam_id-based-policy-examples.html)
- **[MFA]** Require multi-factor authentication for IAM users and the root user in accounts that manage Refactor Spaces environments, and add MFA conditions to sensitive policies. [doc](https://docs.aws.amazon.com/migrationhub-refactor-spaces/latest/userguide/data-protection.html)
- **[federated access]** Use AWS IAM Identity Center or IAM roles (not long-lived IAM user credentials attached directly) to grant individual users only the permissions needed for their Refactor Spaces duties. [doc](https://docs.aws.amazon.com/migrationhub-refactor-spaces/latest/userguide/data-protection.html)
- **[cross-account access]** When sharing Refactor Spaces resources with other accounts or third parties, use IAM roles with explicit trust policies rather than sharing credentials, and consult the documented patterns for cross-account and third-party access. [doc](https://docs.aws.amazon.com/migrationhub-refactor-spaces/latest/userguide/security_iam_troubleshoot.html)
- **[transport encryption]** Require TLS for all communication with Refactor Spaces and downstream service endpoints — TLS 1.2 is required and TLS 1.3 is recommended. [doc](https://docs.aws.amazon.com/migrationhub-refactor-spaces/latest/userguide/data-protection.html)
- **[private connectivity]** Use AWS PrivateLink (an interface VPC endpoint) to access Refactor Spaces from your VPC without traversing the public internet, removing the need for internet gateways, NAT devices, or public IP addresses on calling instances. [doc](https://docs.aws.amazon.com/migrationhub-refactor-spaces/latest/userguide/vpc-interface-endpoints.html)
- **[sensitive data]** Never put confidential or sensitive information (such as customer email addresses) into Refactor Spaces tags or free-form name fields, since this data can propagate into billing and other downstream systems. [doc](https://docs.aws.amazon.com/migrationhub-refactor-spaces/latest/userguide/data-protection.html)
- **[audit logging]** Enable an AWS CloudTrail trail that captures Refactor Spaces API activity (`CreateEnvironment`, `GetEnvironment`, `ListEnvironments`, and related calls) and deliver logs to a dedicated S3 bucket so you can determine who made each change and whether it used root, IAM, or federated credentials. [doc](https://docs.aws.amazon.com/migrationhub-refactor-spaces/latest/userguide/logging-using-cloudtrail.html)

## 🛡️ Reliability
- **[account topology]** Start production refactoring with at least three accounts — one for the existing application, one for the first new microservice, and one to act as the environment owner — rather than a single account, to keep teams and blast radius isolated. [doc](https://docs.aws.amazon.com/migrationhub-refactor-spaces/latest/userguide/how-it-works.html)
- **[Strangler Fig rollout]** Keep the legacy system live behind the default route while new microservices are added and validated, and cut traffic over route by route only once each replacement service is confirmed to behave as expected — this bounds risk and keeps rollback simple. [doc](https://aws.amazon.com/blogs/apn/safely-modernizing-a-monolith-with-aws-migration-hub-refactor-spaces/)
- **[route activation]** Treat route activation state deliberately: no non-default route can be active unless the default route is also active, and toggling the default route to inactive requires all other routes to be inactive first — plan cutover sequencing around this constraint. [doc](https://docs.aws.amazon.com/AWSCloudFormation/latest/TemplateReference/aws-resource-refactorspaces-route.html)
- **[health checks]** Rely on the one-time health check Refactor Spaces performs when a route transitions to active, but validate service health independently beforehand, since a failing check will keep the route from becoming active. [doc](https://docs.aws.amazon.com/AWSCloudFormation/latest/TemplateReference/aws-resource-refactorspaces-route.html)
- **[CIDR planning]** When connecting your own VPC or transit gateway to a Refactor Spaces environment without a network bridge, choose non-overlapping CIDR ranges for service VPCs and the application proxy VPC up front to avoid connectivity failures. [doc](https://docs.aws.amazon.com/migrationhub-refactor-spaces/latest/userguide/tutorials-using-own-network-infrastucture.html)
- **[dependency mapping]** Analyze the monolith's code and dependencies (for example with an application analysis tool) before carving out each microservice, so services are extracted along real boundaries and Refactor Spaces routing changes don't break hidden dependencies. [doc](https://aws.amazon.com/blogs/apn/using-cast-and-aws-migration-hub-refactor-spaces-to-modernize-legacy-applications/)
- **[migration handoff]** When moving an application with AWS Application Migration Service, use the Refactor Spaces post-launch action to automatically create the environment, application, service, and default route immediately after migration so modernization can continue without a manual setup gap. [doc](https://docs.aws.amazon.com/migrationhub-refactor-spaces/latest/userguide/mgn-tutorial.html)
- **[service availability]** Confirm current service status before planning new work: AWS Migration Hub Refactor Spaces is no longer open to new customers as of November 7, 2025, and AWS points customers with similar needs to AWS Transform. [doc](https://docs.aws.amazon.com/migrationhub-refactor-spaces/latest/userguide/what-is-mhub-refactor-spaces.html)

## ⚙️ Operational Excellence
- **[environment ownership]** Designate a single environment owner account with cross-account visibility into all applications, services, and routes, and share the environment to participant accounts with AWS Resource Access Manager instead of managing resources independently per account. [doc](https://docs.aws.amazon.com/migrationhub-refactor-spaces/latest/userguide/welcome-concepts.html)
- **[infrastructure as code]** Create and manage Refactor Spaces environments, applications, services, and routes with AWS CDK, CloudFormation, or the SDK/CLI (not only the console) so refactor topology changes are versioned, reviewed, and repeatable across environments. [doc](https://aws.amazon.com/blogs/mt/accelerate-end-to-end-application-modernization-with-aws-app2container-and-aws-migration-hub-refactor-spaces/)
- **[dynamic registration]** For services running on Amazon EKS, automate discovery and registration of new microservice endpoints with Refactor Spaces instead of manually updating routes, keeping EKS and Refactor Spaces routing configuration in sync as services are added or removed. [doc](https://aws.amazon.com/blogs/mt/auto-discovery-and-dynamic-route-registration-of-services-running-in-amazon-elastic-kubernetes-service-eks-with-aws-migration-hub-refactor-spaces/)
- **[event-driven automation]** Match Refactor Spaces API activity delivered through CloudTrail into Amazon EventBridge (`source: aws.refactor-spaces`) to trigger automated notifications or remediation workflows on specific actions such as environment or route changes. [doc](https://docs.aws.amazon.com/eventbridge/latest/ref/events-ref-refactor-spaces.html)
- **[resource visibility]** Review the AWS resources Refactor Spaces provisions on your behalf (API Gateway, Network Load Balancer, Transit Gateway, security groups, resource-based IAM policies) in each account so your team can troubleshoot and operate the environment with confidence rather than treating it as an opaque black box. [doc](https://aws.amazon.com/blogs/mt/deep-dive-on-an-aws-migration-hub-refactor-spaces-environment/)
