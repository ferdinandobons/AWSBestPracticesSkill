# AWS AppSync — Best Practices

## Common scenarios
- Real-time GraphQL APIs backed by DynamoDB, Lambda, or RDS        → Security, Performance Efficiency
- Mobile/web apps needing offline sync and real-time subscriptions        → Reliability, Performance Efficiency
- Public or partner-facing GraphQL APIs requiring fine-grained authZ        → Security, Operational Excellence
- Internal GraphQL APIs restricted to a VPC        → Security, Reliability

## 🔒 Security
- **[authentication]** Choose the authorization mode that matches the caller — Amazon Cognito user pools for end users needing fine-grained, attribute-based access; IAM for system-to-system/internal calls; Lambda authorizers for custom logic; OIDC to federate an existing identity provider. — Each mode trades off setup complexity, auditability, and access granularity. [doc](https://docs.aws.amazon.com/appsync/latest/devguide/security-authz.html)
- **[api-keys]** Avoid API keys for production or sensitive data, and rotate them regularly when they must be used. — API keys are best suited to public, unauthenticated, low-risk access and expire after at most 365 days. [doc](https://docs.aws.amazon.com/appsync/latest/devguide/security-authz.html)
- **[secrets]** Store sensitive resolver configuration (API keys, credentials) in AWS Secrets Manager rather than in environment variables or code. — Reduces exposure of credentials embedded in resolver/function configuration. [doc](https://aws.amazon.com/blogs/mobile/best-practices-for-aws-appsync-graphql-apis/)
- **[transport]** Require TLS 1.2 (prefer TLS 1.3) and cipher suites with perfect forward secrecy for all clients and HTTP resolver endpoints. — Protects data in transit between clients, AppSync, and downstream HTTP data sources. [doc](https://docs.aws.amazon.com/appsync/latest/devguide/infrastructure-security.html)
- **[iam]** Grant AppSync service roles and resolver data-source roles the least-privilege permissions needed for each table, function, or endpoint. — Overly broad resolver roles let a single compromised query traverse unrelated resources. [doc](https://docs.aws.amazon.com/appsync/latest/devguide/best-practices.html)
- **[iam]** Use AWS managed policies as a starting point, then narrow to customer-managed least-privilege policies, add IAM Access Analyzer validation, and require MFA for sensitive operations. — Reduces the blast radius of compromised credentials managing AppSync resources. [doc](https://docs.aws.amazon.com/appsync/latest/devguide/best-practices.html)
- **[network]** Use AppSync Private APIs with a VPC interface endpoint (AWS PrivateLink) when an API should only be reachable from inside a VPC or hybrid network. — Keeps GraphQL and real-time traffic entirely on the AWS network, with no internet exposure. [doc](https://docs.aws.amazon.com/appsync/latest/devguide/using-private-apis.html)
- **[network]** Scope VPC endpoint policies to the specific IAM principals that should be allowed to call a Private API. — A shared VPC endpoint otherwise grants reach to every Private API in the account and Region. [doc](https://docs.aws.amazon.com/appsync/latest/devguide/using-private-apis.html)
- **[waf]** Attach an AWS WAF web ACL with rate-based rules to public GraphQL endpoints. — Throttles high-volume source IPs and blocks common web exploits such as SQL injection and XSS before they reach resolvers. [doc](https://aws.amazon.com/blogs/mobile/appsync-waf/)
- **[schema]** Set a query depth limit and disable introspection on production APIs where it isn't needed. — Bounds deeply nested or recursive queries and reduces the amount of schema detail exposed to untrusted clients. [doc](https://aws.amazon.com/blogs/mobile/control-graphql-execution-complexity-query-depth-and-introspection-with-aws-appsync/)
- **[data]** Never put confidential or sensitive information (customer emails, credentials) into free-form fields, tags, or URLs used with AppSync. — Such values can surface in billing records and diagnostic/CloudTrail logs. [doc](https://docs.aws.amazon.com/appsync/latest/devguide/data-protection.html)
- **[deploy]** Account for configuration propagation delay when rolling out authorization or schema changes. — AWS AppSync continues serving the prior configuration until the change fully propagates, so a rollback or dependent change made too quickly can hit inconsistent behavior. [doc](https://docs.aws.amazon.com/appsync/latest/devguide/best-practices.html)

## 🛡️ Reliability
- **[caching]** Use DynamoDB Accelerator (DAX) in front of DynamoDB-backed resolvers when you need high-throughput, low-latency reads and can tolerate eventual consistency. — Offloads read pressure from the base table and accelerates AppSync's DynamoDB resolvers automatically. [doc](https://aws.amazon.com/blogs/mobile/best-practices-for-aws-appsync-graphql-apis/)
- **[caching]** Use `ConsistentRead` in DynamoDB resolvers when strongly consistent data is required alongside DAX. — DAX cache is only eventually consistent with the underlying table. [doc](https://aws.amazon.com/blogs/mobile/best-practices-for-aws-appsync-graphql-apis/)
- **[throttling]** Apply client-side rate limiting and exponential backoff, and raise AppSync/data-source throttling limits deliberately for legitimate high-traffic workloads. — Prevents cascading failures when clients repeatedly retry against a throttled API. [doc](https://aws.amazon.com/blogs/mobile/unlocking-deeper-insights-with-enhanced-monitoring-in-aws-appsync/)
- **[resolvers]** Prefer pipeline resolvers with reusable functions over duplicated unit-resolver logic. — Improves consistency and makes it easier to adapt resolvers when the underlying data structure changes. [doc](https://aws.amazon.com/blogs/mobile/best-practices-for-aws-appsync-graphql-apis/)
- **[iac]** Deploy AppSync APIs, resolvers, and data sources with Infrastructure as Code (CDK or CloudFormation) rather than manual console changes. — Ensures deployments are repeatable and consistent, and removes risk of manual configuration drift. [doc](https://aws.amazon.com/blogs/mobile/best-practices-for-aws-appsync-graphql-apis/)

## ⚡ Performance Efficiency
- **[caching]** Enable server-side caching (full-request, per-resolver, or operation-level) for queries that repeat frequently with the same arguments. — Serves subsequent identical requests from an in-memory cache instead of re-invoking the data source. [doc](https://docs.aws.amazon.com/appsync/latest/devguide/enabling-caching.html)
- **[caching]** Use per-resolver caching with explicit cache keys (`context.arguments`, `context.identity`, `context.source`) when only specific fields or operations should be cached. — Avoids caching data that varies per caller or shouldn't be reused across requests. [doc](https://docs.aws.amazon.com/appsync/latest/devguide/enabling-caching.html)
- **[compression]** Enable API response compression for large payloads. — Reduces response transfer time and the amount of data clients need to download. [doc](https://aws.amazon.com/blogs/mobile/best-practices-for-aws-appsync-graphql-apis/)
- **[resolvers]** Use native/JavaScript resolvers instead of Apache VTL mapping templates for new development. — JavaScript is the recommended resolver language, simplifies logic, and has broader community troubleshooting support than VTL. [doc](https://aws.amazon.com/blogs/mobile/best-practices-for-aws-appsync-graphql-apis/)
- **[resolvers]** Use Lambda resolvers when a data source or logic isn't natively supported, rather than forcing complex logic into VTL mapping templates. — Lets you use a full programming language to connect to services such as RDS/Aurora that need custom handling. [doc](https://aws.amazon.com/graphql/resolvers/)
- **[tracing]** Enable AWS X-Ray tracing on GraphQL APIs to analyze end-to-end latency across resolvers and data sources. — Produces a service map showing where time is spent across the request, including integrated AWS services. [doc](https://docs.aws.amazon.com/appsync/latest/devguide/x-ray-tracing.html)

## 💰 Cost Optimization
- **[caching]** Cache frequently repeated identical queries instead of re-invoking downstream data sources on every request. — Reduces the number of billed requests and read operations against backing data sources like DynamoDB. [doc](https://aws.amazon.com/blogs/mobile/best-practices-for-aws-appsync-graphql-apis/)
- **[api-design]** Consolidate reusable data-fetching logic into pipeline resolver functions rather than duplicating it per resolver. — Cuts down on redundant resolver invocations and simplifies ongoing maintenance cost. [doc](https://aws.amazon.com/blogs/mobile/best-practices-for-aws-appsync-graphql-apis/)
- **[compression]** Turn on response compression for large query results. — Lowers data transfer volume in addition to improving latency. [doc](https://docs.aws.amazon.com/appsync/latest/devguide/enabling-caching.html)

## ⚙️ Operational Excellence
- **[monitoring]** Configure Amazon CloudWatch logging and metrics for every production AppSync API. — Provides the operational visibility needed to detect errors, latency regressions, and usage patterns. [doc](https://docs.aws.amazon.com/appsync/latest/devguide/monitoring.html)
- **[monitoring]** Enable AWS X-Ray tracing to diagnose latency and failures across resolvers and integrated data sources. — Gives a request-level service map instead of relying on aggregate metrics alone. [doc](https://docs.aws.amazon.com/appsync/latest/devguide/x-ray-tracing.html)
- **[ci-cd]** Deploy through a CI/CD pipeline (for example, using AWS Amplify or CDK-based automation) instead of manual, one-off deployments. — Enables frequent, tested, lower-risk releases and reduces the chance of shipping regressions to production. [doc](https://aws.amazon.com/blogs/mobile/best-practices-for-aws-appsync-graphql-apis/)
- **[iac]** Use the native AWS AppSync CDK construct for APIs with non-DynamoDB data sources, or the Amplify data construct when DynamoDB-backed resolvers and auth can be auto-generated from a schema. — Matches the IaC abstraction to how much custom resolver/data-source configuration the API actually needs. [doc](https://aws.amazon.com/blogs/mobile/best-practices-for-aws-appsync-graphql-apis/)
- **[secrets]** Configure automatic rotation for any secrets referenced by resolvers via AWS Secrets Manager. — Limits the exposure window if a stored credential is ever compromised. [doc](https://aws.amazon.com/blogs/mobile/best-practices-for-aws-appsync-graphql-apis/)
