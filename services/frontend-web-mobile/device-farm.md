# AWS Device Farm — Best Practices

## Common scenarios
- Running automated Appium/Espresso/XCTest suites against real mobile devices        → Reliability, Performance Efficiency
- Cross-browser Selenium testing integrated into a CI/CD pipeline        → Operational Excellence, Reliability
- Remote access / manual exploratory testing on physical devices        → Security
- Custom test environments with scripted setup and IAM-based access to AWS resources        → Security, Operational Excellence

## 🔒 Security
- **[IAM]** Grant CI/CD systems that call Device Farm the least privilege possible, and prefer temporary credentials over long-lived access keys for each CI test run. — Limits blast radius if a CI system is compromised. [doc](https://docs.aws.amazon.com/devicefarm/latest/developerguide/security-best-practices.html)
- **[IAM execution roles]** Scope custom test environment execution roles to the minimum permissions and to specific resources (e.g., named S3 buckets) instead of wildcard actions or resources. — Reduces exposure if test code or dependencies are compromised. [doc](https://docs.aws.amazon.com/devicefarm/latest/developerguide/custom-test-environments-iam-roles.html)
- **[IAM execution roles]** Use dedicated test resources and roles separate from production, and periodically review execution roles for continued need. — Prevents accidental impact on production systems and stale over-permissioned roles. [doc](https://docs.aws.amazon.com/devicefarm/latest/developerguide/custom-test-environments-iam-roles.html)
- **[IAM execution roles]** Add IAM condition keys to execution role trust/permission policies where possible. — Further restricts when and how the role can be assumed. [doc](https://docs.aws.amazon.com/devicefarm/latest/developerguide/custom-test-environments-iam-roles.html)
- **[Test spec secrets]** Avoid storing or logging sensitive variables (such as AWS keys) in test spec files or test scripts. — Test spec files, generated scripts, and their logs are downloadable artifacts visible to any account user with read access to the run. [doc](https://docs.aws.amazon.com/devicefarm/latest/developerguide/custom-test-environments-best-practices.html)
- **[Device data hygiene]** Use `adb` commands (or equivalent) in custom test environments to clean up any content your application creates on the device. — Reduces residual data exposure between test sessions on shared device infrastructure. [doc](https://docs.aws.amazon.com/devicefarm/latest/developerguide/security-best-practices.html)
- **[Session hygiene]** Avoid entering sensitive information (account credentials, personal data) during automated tests or remote access sessions. — Device Farm captures video and logs of all activity during device sessions. [doc](https://aws.amazon.com/device-farm/faqs/)
- **[Network/transport]** Ensure clients calling the Device Farm API support TLS 1.2 (TLS 1.3 recommended) and cipher suites with perfect forward secrecy (e.g., ECDHE). — Aligns with Device Farm's infrastructure security requirements for signed API requests. [doc](https://docs.aws.amazon.com/devicefarm/latest/developerguide/infrastructure-security.html)
- **[Authentication]** Sign all Device Farm API requests with IAM principal credentials, or use AWS STS to generate temporary security credentials. — Ensures every API call is authenticated and reduces reliance on long-lived credentials. [doc](https://docs.aws.amazon.com/devicefarm/latest/developerguide/infrastructure-security.html)
- **[Encryption in transit]** Rely on Device Farm's enforced HTTPS (SSL/TLS) for all API calls and S3 upload/download URLs, and independently encrypt any communications your tested application itself makes. — Device Farm secures its own transport, but application-level traffic security remains the customer's responsibility. [doc](https://docs.aws.amazon.com/devicefarm/latest/developerguide/data-protection.html)
- **[Auditability]** Enable and maintain a CloudTrail trail that captures Device Farm API activity (project/run creation, uploads, etc.) and deliver logs to S3 across all Regions. — Provides an ongoing record of who made which Device Farm API calls, from root, IAM, federated, or service identities. [doc](https://docs.aws.amazon.com/devicefarm/latest/developerguide/logging-using-cloudtrail.html)

## 🛡️ Reliability
- **[Run configuration]** Rely on Device Farm–managed software and API features (e.g., the `deviceProxy` parameter, `devicefarm-cli`) for run configuration instead of ad hoc shell commands in the test spec. — Managed configuration is more sustainable and consistent across test hosts and devices as Device Farm updates them. [doc](https://docs.aws.amazon.com/devicefarm/latest/developerguide/custom-test-environments-best-practices.html)
- **[Test portability]** Avoid absolute paths or pinning specific minor software versions in test spec files or test package code. — Device Farm routinely updates test host software; hardcoded paths or minor versions can cause tests to fail to locate executables after an update. [doc](https://docs.aws.amazon.com/devicefarm/latest/developerguide/custom-test-environments-best-practices.html)
- **[Failure detection]** Ensure test commands and frameworks return a non-zero exit code on failure during the test phase, including edge cases like zero tests executed. — Device Farm determines pass/fail from shell exit codes, so silent framework defaults (e.g., some JUnit5 configurations) can mask a failed run as successful. [doc](https://docs.aws.amazon.com/devicefarm/latest/developerguide/custom-test-environments-best-practices.html)
- **[Compatibility]** Review the compatibility of test frameworks (e.g., Appium) with the specific device OS version and test host version before a run. — Some framework features do not work consistently across all OS versions. [doc](https://docs.aws.amazon.com/devicefarm/latest/developerguide/custom-test-environments-best-practices.html)
- **[Execution role session duration]** Configure the IAM execution role's maximum session duration to be at least as long as the Device Farm project's job timeout (default 150 minutes). — A session that expires before the job completes will interrupt in-progress tests. [doc](https://docs.aws.amazon.com/devicefarm/latest/developerguide/custom-test-environments-iam-roles.html)
- **[CI/CD integration]** Use automation and CI/CD pipelines to schedule and run Device Farm tests against a representative set of real hardware, and continually review results to adjust the test plan. — Ensures new features are validated consistently on diverse, real device hardware rather than ad hoc manual runs. [doc](https://docs.aws.amazon.com/wellarchitected/latest/framework/sus_sus_dev_a5.html)

## 💰 Cost Optimization
- **[Managed device farms]** Use a managed device farm like AWS Device Farm instead of procuring and maintaining a private fleet of physical test devices. — Avoids the cost and waste of purchasing, maintaining, and upgrading dedicated test hardware. [doc](https://docs.aws.amazon.com/wellarchitected/latest/framework/sus_sus_dev_a5.html)

## ⚙️ Operational Excellence
- **[Test plan definition]** Define testing requirements up front (test type, target operating systems, schedule) before selecting devices and configuring runs, and use client-side telemetry (e.g., Amazon CloudWatch RUM) to shape the plan. — Produces a targeted, maintainable device/test matrix instead of ad hoc coverage. [doc](https://docs.aws.amazon.com/wellarchitected/latest/framework/sus_sus_dev_a5.html)
- **[Test spec maintainability]** Keep customization in Device Farm–managed features and tooling rather than accumulating many custom shell commands in the test spec file. — Highly customized test specs become difficult to maintain over time; managed features shift that maintenance burden to Device Farm. [doc](https://docs.aws.amazon.com/devicefarm/latest/developerguide/custom-test-environments-best-practices.html)
- **[Artifacts]** Write generated files to the provided temp directory during a run and save execution logs to `$DEVICEFARM_LOG_DIR`. — These are the directories Device Farm guarantees to be accessible and to collect as artifacts; other paths may change over time. [doc](https://docs.aws.amazon.com/devicefarm/latest/developerguide/custom-test-environments-best-practices.html)
- **[Continuous improvement]** Continually review test run results (pass/fail trends, crash reports, logs) and adjust the test plan and device pools accordingly. — Keeps the testing process aligned with actual product and hardware changes over time. [doc](https://docs.aws.amazon.com/wellarchitected/latest/framework/sus_sus_dev_a5.html)

## 🌱 Sustainability
- **[Managed device farms]** Use AWS Device Farm's managed device fleet, including older and less popular hardware, instead of maintaining and refreshing a dedicated private device fleet. — Avoids the environmental impact of unnecessary device upgrades and underused dedicated test hardware. [doc](https://docs.aws.amazon.com/wellarchitected/latest/framework/sus_sus_dev_a5.html)
