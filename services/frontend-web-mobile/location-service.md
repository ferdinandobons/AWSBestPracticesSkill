# Amazon Location Service — Best Practices

## Common scenarios
- Adding maps, place search, or geocoding to public web and mobile apps        → Security, Performance Efficiency, Cost Optimization
- Real-time asset/fleet tracking with geofence entry/exit alerts        → Security, Reliability, Cost Optimization
- Route calculation, isolines, and waypoint optimization for logistics apps        → Reliability, Performance Efficiency
- Unauthenticated (anonymous) end-user access from browsers or mobile clients        → Security, Operational Excellence

## 🔒 Security
- **[access-control]** Use identity federation and IAM roles to control access to Amazon Location resources, following the Principle of Least Privilege so custom policies grant only the permissions a task needs. — Reduces the blast radius if credentials or a role are compromised. [doc](https://docs.aws.amazon.com/location/latest/developerguide/best-practices.html)
- **[unauthenticated-access]** For unauthenticated web/mobile scenarios, choose either read-only API keys or an Amazon Cognito unauthenticated identity pool rather than embedding IAM credentials in client code. — API keys suit simple public access; Cognito identity pools support richer, IAM-policy-based authorization for anonymous users. [doc](https://docs.aws.amazon.com/location/latest/developerguide/security-iam.html)
- **[api-keys]** Restrict each API key to specific domains (via HTTP Referer) or to specific Android/Apple application identifiers (`X-Android-Package`/`X-Android-Cert`, `X-Apple-Bundle-Id`), and to the minimum set of read-only actions and resources it needs. — A leaked but properly scoped key can only be used from the allow-listed app or site. [doc](https://docs.aws.amazon.com/location/latest/developerguide/api-keys-best-practices.html)
- **[api-keys]** Set an expiration on API keys and rotate them on a regular cadence instead of creating keys that work indefinitely. — Expired keys are automatically deactivated, limiting how long a leaked key remains usable. [doc](https://docs.aws.amazon.com/location/latest/developerguide/api-keys-best-practices.html)
- **[network]** For web applications, restrict access to Amazon Location resources using an `aws:referer` IAM condition that allow-lists specific sites. — Prevents other sites from reusing your resource ARNs or credentials in requests. [doc](https://docs.aws.amazon.com/location/previous/developerguide/best-practices.html)
- **[transport]** Always use encrypted `https://` connections when calling Amazon Location Service APIs. — Keeps location data and credentials confidential while in transit between clients and AWS. [doc](https://docs.aws.amazon.com/location/latest/developerguide/best-practices.html)
- **[encryption]** Add a customer managed AWS KMS key as a second layer of encryption on tracker and geofence collection resources when you need to control key policies, rotation, or deletion yourself. — By default Amazon Location encrypts personally identifiable data with AWS owned keys, which you cannot view, manage, or audit. [doc](https://docs.aws.amazon.com/location/latest/developerguide/encryption-at-rest.html)
- **[device-ids]** Use unique, non-secret, non-predictable device IDs (opaque UUIDs) and never embed personally identifiable information such as phone numbers or email addresses in device IDs. — Device IDs can act as foreign keys visible to other systems, so predictable or PII-bearing IDs create an unnecessary data exposure and correlation risk. [doc](https://docs.aws.amazon.com/location/latest/developerguide/best-practices.html)
- **[device-data]** Do not include PII (such as phone numbers or email addresses) in `PositionProperties` when sending device position updates. — Position metadata is stored and can propagate to logs, exports, and downstream consumers, widening the exposure surface for sensitive personal data. [doc](https://docs.aws.amazon.com/location/latest/developerguide/best-practices.html)
- **[secrets]** Store API keys used by production applications in a secrets manager (for example AWS Secrets Manager) rather than hardcoding them in client-side source. — Hardcoded keys are easy to extract from shipped web/mobile bundles and harder to rotate. [doc](https://aws.amazon.com/blogs/mobile/build-a-geospatial-application-with-amazon-location-service-api-keys/)

## 🛡️ Reliability
- **[tracking]** Enable tracker position filtering (accuracy-based, distance-based, or time-based) so noisy or overly frequent device updates don't distort device paths or generate false geofence events. — Filtering removes GPS jitter that would otherwise trigger spurious geofence entry/exit events near boundaries. [doc](https://docs.aws.amazon.com/location/latest/developerguide/trackers.html)
- **[quotas]** Check Amazon Location Service quotas (API keys per account, trackers/geofence collections, geofences per collection, request rate limits per API) before scaling a workload, and request quota increases proactively for adjustable limits. — Exceeding a quota causes Amazon Location Service to throttle or deny additional requests, which can break tracking or routing pipelines under load. [doc](https://docs.aws.amazon.com/location/previous/developerguide/location-quotas.html)
- **[monitoring]** Monitor Amazon Location Service resources with Amazon CloudWatch and define a monitoring plan (what to monitor, how often, who is alerted) before issues occur in production. — Continuous monitoring is what lets you debug a multi-point failure across an AWS solution instead of discovering it after the fact. [doc](https://docs.aws.amazon.com/location/previous/developerguide/monitoring.html)

## ⚡ Performance Efficiency
- **[api-keys]** Prefer API keys over IAM+Cognito for pure Maps/Places/Routes read access from web and mobile clients when low latency matters. — API keys allow direct calls from web/mobile apps and benefit from increased edge caching compared to per-request IAM-signed calls. [doc](https://aws.amazon.com/location/faqs/)
- **[tracking]** Choose the tracker position-filtering strategy that matches device capability: accuracy-based when devices report a horizontal accuracy, distance-based (30 m threshold) when they don't, or time-based to cap updates to one per interval. — Matching the filter to device characteristics avoids either over-filtering real movement or under-filtering GPS noise. [doc](https://docs.aws.amazon.com/location/latest/developerguide/start-create-tracker.html)

## 💰 Cost Optimization
- **[tracking]** Use tracker position filtering (accuracy-based, distance-based, or time-based) to reduce the number of position updates stored and geofence evaluations triggered. — Fewer stored updates and evaluations directly lowers the volume of billed tracker and geofence operations. [doc](https://docs.aws.amazon.com/location/latest/developerguide/trackers.html)
- **[api-keys]** Set expiration dates on API keys and monitor usage with billing alerts, especially for keys tied to metered Maps/Places/Routes actions. — Expired or unused keys that keep accumulating calls (for example from a leaked key) can otherwise drive unexpected charges. [doc](https://aws.amazon.com/blogs/mobile/build-a-geospatial-application-with-amazon-location-service-api-keys/)
- **[api-keys]** Scope each API key's allowed actions to only the specific read-only Maps, Places, or Routes operations the application actually calls. — Narrower action lists reduce the chance of unexpected or unauthorized billed API calls if a key is misused. [doc](https://docs.aws.amazon.com/location/latest/developerguide/api-keys-best-practices.html)

## ⚙️ Operational Excellence
- **[logging]** Enable AWS CloudTrail for Amazon Location Service to record all console and API actions (who made the request, source IP, timestamp) for auditing and incident investigation. — CloudTrail event history is the primary way to establish a security baseline and detect abnormal API activity. [doc](https://docs.aws.amazon.com/location/latest/developerguide/security-logging-and-monitoring.html)
- **[monitoring]** Configure Amazon CloudWatch alarms on Amazon Location Service metrics with notifications routed through Amazon SNS. — Automatically alerts the responsible team when a metric crosses a defined threshold, instead of relying on manual dashboard checks. [doc](https://docs.aws.amazon.com/location/latest/developerguide/security-logging-and-monitoring.html)
- **[resource-management]** Tag Amazon Location Service resources (trackers, geofence collections, maps, place indexes, API keys) to organize them by purpose, owner, environment, or billing relationship. — Tags make it practical to manage, search, and filter resources and attribute costs across a growing set of location resources. [doc](https://aws.amazon.com/location/faqs/)
- **[health]** Use the AWS Health Dashboard to track Amazon Location Service status and any historical events or issues that might affect your environment. — Gives early visibility into AWS-side incidents that could explain degraded application behavior. [doc](https://docs.aws.amazon.com/location/latest/developerguide/security-logging-and-monitoring.html)
