# Amazon Lightsail — Best Practices

## Common scenarios
- Simple VPS hosting for websites, blogs, and small web apps        → Security, Reliability, Cost Optimization
- Migrating a workload from a snapshot to a new instance or larger bundle        → Reliability, Operational Excellence
- Highly available multi-instance application behind a load balancer        → Reliability, Performance Efficiency
- Storing and sharing objects with Lightsail object storage buckets        → Security, Cost Optimization

## 🔒 Security
- **[Instance access]** Restrict SSH/RDP firewall rules to specific source IP addresses or CIDR ranges instead of leaving them open to the internet. — Limits remote-access exposure to only trusted networks. [doc](https://aws.amazon.com/blogs/compute/enhancing-site-security-with-new-lightsail-firewall-features/)
- **[Instance access]** Configure firewall rules per application role (web, database, DNS, SMTP) using the minimum ports required. — Reduces attack surface by only exposing ports the workload actually needs. [doc](https://docs.aws.amazon.com/lightsail/latest/userguide/amazon-lightsail-firewall-rules-reference.html)
- **[Instance access]** Disable or restrict remote access via the browser-based Lightsail console client when it isn't needed. — Prevents an additional unauthenticated path into the instance. [doc](https://docs.aws.amazon.com/lightsail/latest/userguide/understanding-firewall-and-port-mappings-in-amazon-lightsail.html)
- **[SSH keys]** Use custom SSH key pairs per user or team and remove/replace keys on an instance when someone leaves or a key is compromised. — Ensures you can revoke individual access without disabling the whole instance. [doc](https://docs.aws.amazon.com/lightsail/latest/userguide/amazon-lightsail-remove-ssh-key-on-instance.html)
- **[SSH keys]** Store the private key of the default or custom key pair safely, and generate a new key rather than reusing one if it's ever lost or exposed. — Compromised keys allow full shell access to the instance. [doc](https://docs.aws.amazon.com/lightsail/latest/userguide/amazon-lightsail-managing-ssh-keys.html)
- **[Patching]** Regularly update and patch the operating system, applications, and container images yourself after instance creation. — AWS patches the blueprint image at creation time only; ongoing OS/app patching is the customer's responsibility under the shared responsibility model. [doc](https://docs.aws.amazon.com/lightsail/latest/userguide/amazon-lightsail-update-management.html)
- **[Patching]** On Windows Server instances, verify Windows Update is set to automatic and periodically confirm updates are installed. — Keeps known vulnerabilities patched on internet-facing instances. [doc](https://docs.aws.amazon.com/lightsail/latest/userguide/best-practices-for-securing-windows-based-lightsail-instances.html)
- **[IAM]** Grant IAM users and roles least-privilege, task-specific permissions for Lightsail rather than broad account access. — Limits blast radius if credentials are exposed. [doc](https://docs.aws.amazon.com/lightsail/latest/userguide/security_iam_id-based-policy-examples.html)
- **[IAM]** Create a dedicated IAM group/policy scoped to Lightsail for team members who only need Lightsail access, and remove or delete users promptly when they leave. — Keeps access reviewable and revocable as team membership changes. [doc](https://docs.aws.amazon.com/lightsail/latest/userguide/amazon-lightsail-managing-access-for-an-iam-user.html)
- **[Object storage]** Verify Lightsail buckets are not publicly accessible and enable S3 block public access; use least-privilege access and rotate bucket access keys regularly. — Prevents accidental public exposure of stored objects and limits impact of leaked keys. [doc](https://docs.aws.amazon.com/lightsail/latest/userguide/amazon-lightsail-bucket-security-best-practices.html)
- **[Object storage]** Enable access logging, bucket versioning, and use AWS CloudTrail to audit bucket activity. — Provides an audit trail for detecting unauthorized access or accidental deletion. [doc](https://docs.aws.amazon.com/lightsail/latest/userguide/amazon-lightsail-bucket-security-best-practices.html)
- **[EC2 migration]** When exporting a Lightsail snapshot to an EC2 instance, remove residual Lightsail default and system keys (including `lightsail_instance_ca.pub`). — Leftover keys from the Lightsail service could otherwise allow unintended access to the new EC2 instance. [doc](https://docs.aws.amazon.com/lightsail/latest/userguide/amazon-lightsail-securing-linux-unix-amazon-ec2-instances.html)
- **[API access]** Ensure clients connecting to the Lightsail API support TLS 1.2 (TLS 1.3 recommended) and cipher suites with perfect forward secrecy. — Protects API calls and credentials in transit. [doc](https://docs.aws.amazon.com/lightsail/latest/userguide/infrastructure-security.html)

## 🛡️ Reliability
- **[Backups]** Enable automatic daily snapshots for instances and block storage disks in addition to manual snapshots before major changes. — Provides a recent recovery point without relying on manual discipline. [doc](https://docs.aws.amazon.com/lightsail/latest/userguide/amazon-lightsail-configuring-automatic-snapshots.html)
- **[Backups]** Explicitly "keep" important automatic snapshots you want to retain, since automatic snapshots are deleted along with the source resource. — Avoids losing your only backup when an instance or disk is deleted. [doc](https://docs.aws.amazon.com/lightsail/latest/userguide/understanding-snapshots-in-amazon-lightsail.html)
- **[High availability]** Deploy multiple instances across separate Availability Zones and distribute traffic with a Lightsail load balancer. — Protects the application from a failure in a single Availability Zone or data center. [doc](https://docs.aws.amazon.com/lightsail/latest/userguide/understanding-regions-and-availability-zones-in-amazon-lightsail.html)
- **[High availability]** Configure Lightsail load balancer health checks so traffic is only routed to healthy target instances. — Automatically removes failing instances from rotation and reduces downtime. [doc](https://docs.aws.amazon.com/lightsail/latest/userguide/understanding-lightsail-load-balancers.html)
- **[Disaster recovery]** Copy instance and disk snapshots across Regions for critical workloads. — Enables recovery even if an entire Region becomes unavailable. [doc](https://docs.aws.amazon.com/lightsail/latest/userguide/disaster-recovery-resiliency.html)
- **[Monitoring]** Create metric alarms (for example CPUUtilization, StatusCheckFailed, HealthyHostCount) on instances, databases, and load balancers with email/SMS notifications. — Surfaces failures and degraded health early so you can act before an outage impacts users. [doc](https://docs.aws.amazon.com/lightsail/latest/userguide/amazon-lightsail-alarms.html)
- **[Change management]** Take a manual snapshot before upgrading instance plans, applying major updates, or editing configuration/plugins. — Provides a fast rollback point if the change breaks the application. [doc](https://docs.aws.amazon.com/lightsail/latest/userguide/amazon-lightsail-wordpress-tips-best-practices.html)

## ⚡ Performance Efficiency
- **[Scaling]** Choose an instance bundle (general purpose, compute optimized, or memory optimized) that matches the workload's CPU-to-memory profile. — Matches provisioned resources to actual workload characteristics instead of over- or under-provisioning. [doc](https://docs.aws.amazon.com/lightsail/latest/userguide/amazon-lightsail-bundles.html)
- **[Scaling]** When a workload outgrows its current bundle, create a snapshot and launch a larger instance from it rather than trying to resize in place. — Lightsail instances cannot be resized directly; snapshot-and-recreate is the supported upgrade path. [doc](https://docs.aws.amazon.com/lightsail/latest/userguide/lightsail-how-to-create-larger-instance-from-snapshot-using-aws-cli.html)
- **[Scaling]** Use a Lightsail load balancer with multiple target instances to handle traffic spikes and maintain response times under peak load. — Distributes load horizontally instead of relying on a single instance's capacity. [doc](https://docs.aws.amazon.com/lightsail/latest/userguide/amazon-lightsail-faq-load-balancers.html)

## 💰 Cost Optimization
- **[Right-sizing]** Select the smallest bundle that meets current performance needs and monitor CPUUtilization/NetworkIn/NetworkOut metrics to decide when to resize. — Avoids paying for compute, memory, or transfer capacity the workload doesn't use. [doc](https://docs.aws.amazon.com/lightsail/latest/userguide/amazon-lightsail-alarms.html)
- **[Object storage]** Adjust a bucket's storage plan as a long-term strategy based on sustained usage trends, not as a short-term monthly cost-cutting move. — Frequent plan changes reset quotas without refunding excess charges already incurred. [doc](https://docs.aws.amazon.com/lightsail/latest/userguide/amazon-lightsail-changing-bucket-plans.html)
- **[Snapshots]** Regularly review and delete unneeded manual snapshots and disable automatic snapshots on resources that no longer need them. — Snapshot storage is billed continuously, so stale snapshots add ongoing cost. [doc](https://docs.aws.amazon.com/lightsail/latest/userguide/amazon-lightsail-configuring-automatic-snapshots.html)

## ⚙️ Operational Excellence
- **[Static IP]** Attach a static IP to instances that are referenced by DNS records, and re-attach it to a replacement instance during upgrades or migrations. — Avoids reconfiguring external DNS every time the underlying instance changes. [doc](https://aws.amazon.com/blogs/publicsector/building-team-knowledge-base-using-amazon-lightsail/)
- **[Tagging]** Tag Lightsail resources to organize them, allocate costs, and control access through IAM policies based on tags. — Makes it easier to manage and audit resources as the number of instances grows. [doc](https://aws.amazon.com/lightsail/faq/)
- **[Access management]** Use AWS IAM Identity Center permission sets, or IAM users/groups scoped to Lightsail, to formally grant and revoke team access instead of sharing account credentials. — Creates an auditable, revocable access model as your team changes. [doc](https://docs.aws.amazon.com/lightsail/latest/userguide/security_iam_troubleshoot.html)
- **[Monitoring]** Install the CloudWatch agent on Lightsail instances to collect metrics (such as memory usage) beyond the built-in Lightsail metrics. — Built-in Lightsail metrics don't cover memory usage, which is often critical for diagnosing application issues. [doc](https://aws.amazon.com/blogs/compute/14364-2/)
