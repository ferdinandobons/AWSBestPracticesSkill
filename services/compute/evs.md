# Amazon Elastic VMware Service — Best Practices

## Common scenarios
- Lift-and-shift migration of on-premises VMware Cloud Foundation workloads to AWS        → Reliability, Operational Excellence
- Extending on-premises data centers to AWS via Direct Connect/VPN for hybrid VMware operations        → Reliability, Security, Performance Efficiency
- Data center exit or capacity augmentation without refactoring VMware-based applications        → Cost Optimization, Operational Excellence
- Running vSphere/vSAN/NSX workloads natively inside a customer-owned VPC        → Security, Reliability

## 🔒 Security
- **[shared responsibility]** Understand that you are responsible for installing, patching, and hardening the VCF software stack (vCenter Server, NSX, SDDC Manager, ESX) while AWS secures the underlying infrastructure it provisions — prevents gaps where neither side is patching a component. [doc](https://docs.aws.amazon.com/evs/latest/userguide/getting-started.html)
- **[credential rotation]** Rotate the secrets stored in AWS Secrets Manager that Amazon EVS connectors use to access your VCF management appliances — limits the blast radius if a credential is exposed. [doc](https://docs.aws.amazon.com/evs/latest/userguide/getting-started.html)
- **[VCF hardening]** Configure VCF authentication, role-based access control, and password rotation for all VCF management appliances, and harden the VCF management network per Broadcom guidance — reduces the risk of unauthorized access to vCenter, NSX, and SDDC Manager. [doc](https://docs.aws.amazon.com/evs/latest/userguide/getting-started.html)
- **[least privilege]** Start from AWS managed policies and move toward least-privilege customer managed IAM policies scoped to specific Amazon EVS actions and resources — limits what any given identity can do to your environments and hosts. [doc](https://docs.aws.amazon.com/evs/latest/userguide/security-iam-id-based-policy-examples.html)
- **[policy validation]** Use IAM Access Analyzer to validate identity-based policies for Amazon EVS and add conditions (such as requiring SSL/TLS) to further restrict access — catches overly permissive or malformed policies before they are attached. [doc](https://docs.aws.amazon.com/evs/latest/userguide/security-iam-id-based-policy-examples.html)
- **[MFA]** Require multi-factor authentication for IAM users and the root user who can manage Amazon EVS resources — adds a second factor against credential compromise for an environment that can incur significant cost. [doc](https://docs.aws.amazon.com/evs/latest/userguide/security-iam-id-based-policy-examples.html)
- **[encryption in transit]** Use TLS 1.2 (TLS 1.3 recommended) for communication with AWS resources supporting your Amazon EVS environment — protects data moving between your management tools and AWS APIs. [doc](https://docs.aws.amazon.com/evs/latest/userguide/data-protection.html)
- **[activity logging]** Set up AWS CloudTrail trails to capture Amazon EVS API activity (such as CreateEnvironment, GetEnvironment, DeleteEnvironment) across all Regions, and separately configure VMware Aria Operations/Tanzu Observability for VCF-side activity since Amazon EVS does not log non-AWS component activity — gives you a complete audit trail across both the AWS and VMware layers. [doc](https://docs.aws.amazon.com/evs/latest/userguide/data-protection.html)
- **[network segmentation]** Keep the Amazon EVS VPC dedicated to Amazon EVS resources and place other workloads in adjacent VPCs connected through Transit Gateway or AWS Cloud WAN — creates clear blast-radius boundaries and simplifies security auditing. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/building-a-modern-network-for-your-vmware-workloads-using-amazon-elastic-vmware-service/)
- **[traffic inspection]** Use AWS Network Firewall with Transit Gateway integration to centrally inspect egress, ingress, and east-west traffic in and out of the Amazon EVS environment — provides a single point of policy control and centralized logging across hybrid VMware and AWS traffic. [doc](https://aws.amazon.com/blogs/architecture/secure-amazon-elastic-vmware-service-amazon-evs-with-aws-network-firewall/)
- **[micro-segmentation]** Combine NSX Distributed Firewall policies on overlay segments with AWS security controls on the underlay VPC — enforces defense-in-depth across both the VMware software-defined network and the AWS network layer. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/building-a-modern-network-for-your-vmware-workloads-using-amazon-elastic-vmware-service/)
- **[data at rest]** Use AWS encryption solutions and default security controls across the AWS services supporting your Amazon EVS environment, and consider Amazon Macie for discovering sensitive data stored in S3 — protects data outside the VCF environment that AWS resources may hold. [doc](https://docs.aws.amazon.com/evs/latest/userguide/data-protection.html)

## 🛡️ Reliability
- **[quorum and maintenance]** Deploy the minimum four-ESX-host environment that VCF requires and add hosts post-deployment as needed rather than reducing below quorum — maintains vSAN quorum and availability during maintenance operations and host failures. [doc](https://docs.aws.amazon.com/evs/latest/userguide/disaster-recovery-resiliency.html)
- **[NSX Edge resiliency]** Rely on the Active/Standby NSX Edge cluster that Amazon EVS deploys across two different hosts — allows quick failover if an NSX Edge node fails. [doc](https://docs.aws.amazon.com/evs/latest/userguide/disaster-recovery-resiliency.html)
- **[placement strategy]** Use an EC2 partition placement group to spread hosts across logical partitions that do not share underlying hardware, reducing the likelihood of correlated hardware failures for large distributed workloads — do not use spread placement groups, which Amazon EVS does not support. [doc](https://docs.aws.amazon.com/evs/latest/userguide/disaster-recovery-resiliency.html)
- **[disaster recovery]** Use AWS Elastic Disaster Recovery to automate backup and recovery of data in your Amazon EVS environment — provides a tested recovery path instead of ad hoc backup procedures. [doc](https://docs.aws.amazon.com/evs/latest/userguide/disaster-recovery-resiliency.html)
- **[safe host management]** Add or remove hosts only through the Amazon EVS console or AWS CLI, and avoid terminating or modifying the underlying EC2 instances directly — direct EC2 manipulation can cause data loss in the VCF environment. [doc](https://docs.aws.amazon.com/help-panel/evs/latest/console/hp-specify-host.html)
- **[persistent management connectivity]** Ensure the service access subnet and its route tables remain correctly configured, since Amazon EVS requires a persistent connection to your SDDC Manager for operations to succeed — an interrupted connection can cause environment operations to fail. [doc](https://docs.aws.amazon.com/help-panel/evs/latest/console/hp-configure-networks-service-access-subnet.html)
- **[DNS configuration]** Configure primary and secondary DNS servers in the VPC's DHCP option set and create forward/reverse lookup records for every VCF management appliance and host — misconfigured DNS is a common cause of VCF deployment and operational failures. [doc](https://docs.aws.amazon.com/evs/latest/userguide/architecture.html)

## ⚡ Performance Efficiency
- **[CIDR planning]** Size your VPC IPv4 CIDR block and Amazon EVS VLAN subnet CIDR blocks (minimum /28, maximum /24, non-overlapping) appropriately up front — undersized or overlapping ranges force disruptive rework later. [doc](https://docs.aws.amazon.com/evs/latest/userguide/architecture.html)
- **[low-latency clustering]** Use an EC2 cluster placement group when hosts need low-latency, high-throughput networking within the same physical rack — matches placement strategy to workloads that are latency-sensitive rather than fault-tolerance-sensitive. [doc](https://docs.aws.amazon.com/evs/latest/userguide/disaster-recovery-resiliency.html)
- **[private connectivity]** Use AWS PrivateLink for private connectivity from Amazon EVS workloads to AWS and non-AWS services instead of routing through the public internet — reduces latency and avoids unnecessary egress paths. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/building-a-modern-network-for-your-vmware-workloads-using-amazon-elastic-vmware-service/)

## 💰 Cost Optimization
- **[capacity planning]** Plan capacity requirements in advance (including engaging your AWS representative for capacity reservations) before deploying Amazon EVS hosts — avoids over-provisioning the initial four-host minimum or under-provisioning for growth. [doc](https://aws.amazon.com/evs/faqs/)
- **[licensing awareness]** Confirm your eligibility and licensing under your Broadcom agreements before provisioning VCF software through the BYOL model, and maintain valid VCF licenses in SDDC Manager — avoids compliance and service interruption issues tied to licensing. [doc](https://aws.amazon.com/service-terms/)
- **[dedicated hosts for licensing]** Use EC2 Dedicated Hosts where enterprise software licensing (for example Windows or SQL Server) requires it, and Placement Groups otherwise — avoids paying for dedicated tenancy when it isn't required by your licensing terms. [doc](https://docs.aws.amazon.com/help-panel/evs/latest/console/hp-specify-host.html)

## ⚙️ Operational Excellence
- **[infrastructure as code]** Automate Amazon EVS environment provisioning (VPCs, Route 53, Transit Gateway) with infrastructure as code rather than manual configuration — reduces deployment time and configuration errors when standing up production-ready VMware environments. [doc](https://aws.amazon.com/solutions/guidance/automated-setup-for-amazon-elastic-vmware-service/)
- **[centralized audit trail]** Create a CloudTrail trail that applies to all Regions and delivers logs to an S3 bucket for Amazon EVS API activity — supports investigation of who made which environment/host changes and when. [doc](https://docs.aws.amazon.com/evs/latest/userguide/logging-using-cloudtrail.html)
- **[VCF-side observability]** Configure VMware monitoring solutions such as VMware Aria Operations or Tanzu Observability for centralized VCF logging, since Amazon EVS itself does not log activity within vSphere or NSX Manager — closes the observability gap for the VMware control plane. [doc](https://docs.aws.amazon.com/evs/latest/userguide/data-protection.html)
- **[environment health monitoring]** Monitor the health of Amazon EVS connectors and the aggregate environment health that Amazon EVS reports — surfaces connectivity or environment issues early, before they affect workloads. [doc](https://docs.aws.amazon.com/evs/latest/userguide/getting-started.html)
