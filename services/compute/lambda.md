# AWS Lambda — Best Practices

## Common scenarios
- Event-driven data processing (S3, Kinesis, DynamoDB Streams)        → Reliability, Performance Efficiency
- REST/HTTP API backends behind API Gateway        → Performance Efficiency, Cost Optimization
- Asynchronous and scheduled automation (EventBridge, SQS, SNS)        → Reliability, Operational Excellence
- Glue code / orchestration between AWS services        → Security, Operational Excellence

## 🔒 Security
- **[execution role]** Grant the Lambda execution role only the permissions it needs, and use IAM Access Analyzer to derive a least-privilege policy from CloudTrail activity — reduces blast radius if the function or its dependencies are compromised. [doc](https://docs.aws.amazon.com/lambda/latest/dg/lambda-intro-execution-role.html)
- **[resource policy]** Scope resource-based policies (e.g. S3, EventBridge invoke permissions) to specific buckets, accounts, or ARNs rather than granting blanket service access. [doc](https://docs.aws.amazon.com/lambda/latest/dg/concepts-basics.html)
- **[environment variables]** Encrypt sensitive environment variables with a customer managed AWS KMS key instead of the default AWS managed key, for full control over rotation and auditing. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/encryption-best-practices/lambda.html)
- **[environment variables]** Enable client-side encryption helpers on environment variables holding secrets so they are protected in transit as well as at rest. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/encryption-best-practices/lambda.html)
- **[code integrity]** Enable AWS Signer code signing so only functions and layers signed by an approved profile can be deployed. [doc](https://docs.aws.amazon.com/lambda/latest/dg/governance-code-signing.html)
- **[monitoring]** Enable Amazon GuardDuty Lambda Protection to detect anomalous network activity generated by function invocations. [doc](https://docs.aws.amazon.com/lambda/latest/dg/best-practices.html)
- **[posture management]** Use AWS Security Hub CSPM controls for Lambda to continuously evaluate function configurations against security standards. [doc](https://docs.aws.amazon.com/lambda/latest/dg/best-practices.html)
- **[data handling]** Never store user data, event payloads, or other sensitive state in the execution environment across invocations; treat the environment as reusable but not a trusted data store. [doc](https://docs.aws.amazon.com/lambda/latest/dg/best-practices.html)
- **[APIs]** Avoid calling non-documented, non-public Lambda runtime APIs, since AWS may change them in backward-incompatible ways. [doc](https://docs.aws.amazon.com/lambda/latest/dg/best-practices.html)

## 🛡️ Reliability
- **[idempotency]** Write idempotent handler code so that duplicate deliveries (from async retries, at-least-once event sources, or client retries) do not cause inconsistent side effects. [doc](https://docs.aws.amazon.com/lambda/latest/dg/best-practices.html)
- **[async invocations]** Configure a dead-letter queue or an on-failure destination so events that exhaust retries or expire are captured for inspection instead of silently dropped. [doc](https://docs.aws.amazon.com/lambda/latest/dg/invocation-async-retain-records.html)
- **[async invocations]** Tune maximum event age and maximum retry attempts to avoid retrying obsolete events and to control cost from repeated invocations. [doc](https://aws.amazon.com/blogs/compute/implementing-error-handling-for-aws-lambda-asynchronous-invocations/)
- **[throttling]** Build throttle tolerance using timeouts, retries, and backoff with jitter on upstream callers so Lambda can scale up smoothly under load. [doc](https://docs.aws.amazon.com/lambda/latest/dg/best-practices.html)
- **[concurrency]** Use provisioned concurrency for latency-sensitive workloads that need to absorb bursts without relying solely on on-demand scaling. [doc](https://docs.aws.amazon.com/lambda/latest/dg/best-practices.html)
- **[scaling limits]** Know upstream/downstream throughput constraints and use reserved concurrency to cap how high a function can scale when a dependency cannot keep pace. [doc](https://docs.aws.amazon.com/lambda/latest/dg/best-practices.html)
- **[stream processing]** Enable partial batch response for Kinesis/DynamoDB Streams event sources so only failed records are retried instead of the whole batch. [doc](https://docs.aws.amazon.com/lambda/latest/dg/best-practices.html)
- **[queue integration]** When using Amazon SQS as an event source, ensure the queue's visibility timeout is not shorter than the function's expected invocation time to prevent duplicate processing or failed configuration updates. [doc](https://docs.aws.amazon.com/lambda/latest/dg/best-practices.html)
- **[VPC access]** If a function must reach resources in a VPC, configure at least two subnets across separate Availability Zones so VPC access does not reduce the function's inherent multi-AZ availability. [doc](https://aws.amazon.com/blogs/compute/operating-lambda-application-design-part-3/)

## ⚡ Performance Efficiency
- **[execution reuse]** Initialize SDK clients, database connections, and other reusable resources outside the function handler so warm invocations can reuse them instead of recreating them. [doc](https://docs.aws.amazon.com/lambda/latest/dg/best-practices.html)
- **[connections]** Use your runtime's keep-alive directive to maintain persistent connections, since Lambda purges idle connections and reusing a closed one causes connection errors. [doc](https://docs.aws.amazon.com/lambda/latest/dg/best-practices.html)
- **[memory sizing]** Load test and profile functions using the `Max Memory Used` field in CloudWatch REPORT logs (or a tool such as AWS Lambda Power Tuning) to right-size memory, since memory allocation also scales available CPU. [doc](https://docs.aws.amazon.com/lambda/latest/dg/best-practices.html)
- **[timeouts]** Load test to determine an appropriate timeout value, particularly when the function calls downstream services that may not scale at the same rate as Lambda. [doc](https://docs.aws.amazon.com/lambda/latest/dg/best-practices.html)
- **[package size]** Minimize deployment package size and dependency complexity so cold starts spend less time downloading and unpacking code. [doc](https://aws.amazon.com/blogs/compute/optimizing-your-aws-lambda-costs-part-1/)
- **[tracing]** Enable AWS X-Ray active tracing to visualize application components and identify performance bottlenecks in function invocations. [doc](https://aws.amazon.com/blogs/compute/optimizing-your-aws-lambda-costs-part-1/)
- **[stream throughput]** Tune batch size and batching window for stream and queue event sources so polling frequency matches how quickly the function completes its work. [doc](https://docs.aws.amazon.com/lambda/latest/dg/best-practices.html)
- **[Kinesis throughput]** Increase shard count with a well-chosen partition key to raise the maximum concurrent invocations Lambda can sustain against a Kinesis stream. [doc](https://docs.aws.amazon.com/lambda/latest/dg/best-practices.html)
- **[VPC networking]** Provision VPC interface endpoints (AWS PrivateLink) for AWS services your VPC-connected function calls, avoiding extra NAT-related latency and cold-start overhead. [doc](https://aws.amazon.com/blogs/big-data/proactive-monitoring-for-amazon-redshift-serverless-using-aws-lambda-and-slack-alerts/)

## 💰 Cost Optimization
- **[invocation volume]** Cache CloudFront in front of API Gateway for infrequently changing responses to cut the number of Lambda invocations triggered per request. [doc](https://aws.amazon.com/blogs/compute/operating-lambda-performance-optimization-part-3/)
- **[SQS batching]** Increase the SQS event source `BatchSize` to process more messages per invocation and reduce the total number of invocations billed. [doc](https://aws.amazon.com/blogs/compute/operating-lambda-performance-optimization-part-3/)
- **[runtime choice]** Match runtime to workload shape — interpreted languages often minimize total cost for small, simple functions, while compiled languages pay off for compute-heavy workloads or when paired with provisioned concurrency. [doc](https://aws.amazon.com/blogs/compute/operating-lambda-performance-optimization-part-3/)
- **[data transfer]** Minimize the size of data passed between functions and services, since data transfer is billed at standard EC2 data-transfer rates. [doc](https://aws.amazon.com/blogs/compute/operating-lambda-performance-optimization-part-3/)
- **[savings plans]** Apply Compute Savings Plans to your consistent baseline Lambda usage after analyzing historical invocation patterns. [doc](https://aws.amazon.com/blogs/compute/optimizing-your-aws-lambda-costs-part-1/)
- **[recursion]** Avoid recursive invocation patterns, and set reserved concurrency to 0 immediately if unintended invocation volume is detected, to stop runaway costs while you fix the code. [doc](https://docs.aws.amazon.com/lambda/latest/dg/best-practices.html)
- **[unused resources]** Delete Lambda functions that are no longer in use so they stop counting against deployment package size limits and incurring needless overhead. [doc](https://docs.aws.amazon.com/lambda/latest/dg/best-practices.html)
- **[anomaly detection]** Enable AWS Cost Anomaly Detection to catch unusual Lambda usage and spend patterns automatically. [doc](https://docs.aws.amazon.com/lambda/latest/dg/best-practices.html)

## ⚙️ Operational Excellence
- **[configuration]** Pass operational parameters (bucket names, table names, feature flags) via environment variables instead of hard-coding them, to keep deployments environment-agnostic. [doc](https://docs.aws.amazon.com/lambda/latest/dg/best-practices.html)
- **[observability]** Use CloudWatch metrics and alarms (for example, on invocation duration) rather than emitting custom metrics synchronously from within function code, to catch issues earlier and more efficiently. [doc](https://docs.aws.amazon.com/lambda/latest/dg/best-practices.html)
- **[observability]** Emit custom metrics asynchronously using Embedded Metric Format (EMF) through function logs instead of synchronous CloudWatch API calls, to reduce latency. [doc](https://docs.aws.amazon.com/lambda/latest/dg/best-practices.html)
- **[logging]** Use structured JSON logging so logs can be searched, filtered, and correlated more easily across invocations. [doc](https://docs.aws.amazon.com/lambda/latest/dg/best-practices.html)
- **[stream monitoring]** Monitor the `IteratorAge` metric on stream-based event sources to detect stalled shard processing. [doc](https://docs.aws.amazon.com/lambda/latest/dg/best-practices.html)
- **[quotas]** Stay familiar with Lambda quotas — payload size, file descriptors, and `/tmp` space are commonly overlooked limits that cause production incidents. [doc](https://docs.aws.amazon.com/lambda/latest/dg/best-practices.html)
- **[Well-Architected review]** Assess Lambda-based workloads against the Serverless Applications Lens of the AWS Well-Architected Framework to systematically cover operational, security, reliability, performance, and cost concerns. [doc](https://docs.aws.amazon.com/wellarchitected/latest/serverless-applications-lens/definitions.html)
