# Amazon EC2 Auto Scaling — Best Practices

## Common scenarios
- Maintaining a fleet of EC2 instances that self-heals after failures        → Reliability
- Scaling web/app tiers dynamically to match variable traffic        → Performance Efficiency, Cost Optimization
- Running fault-tolerant, cost-efficient fleets with Spot and On-Demand instances        → Cost Optimization, Reliability
- Reducing time-to-scale for applications with long instance boot times        → Performance Efficiency, Operational Excellence

## 🔒 Security
- **[IAM permissions]** Apply least-privilege, tag-scoped IAM policies for Auto Scaling API actions rather than broad `autoscaling:*` grants. — Limits blast radius of compromised credentials or misconfigured automation. [doc](https://docs.aws.amazon.com/autoscaling/ec2/userguide/security_iam_id-based-policy-examples.html)
- **[Launch templates]** Restrict who can modify the launch template/AMI used by the Auto Scaling group and manage it as code under change control. — Prevents unauthorized or unhardened images from being deployed fleet-wide. [doc](https://aws.amazon.com/blogs/publicsector/documenting-the-use-of-amazon-ec2-auto-scaling-groups-in-dod/)
- **[Instance role]** Grant EC2 instances launched by the group an IAM instance profile scoped to only the permissions the application needs. — Avoids over-privileged instances if a host is compromised. [doc](https://docs.aws.amazon.com/autoscaling/ec2/userguide/security-iam.html)
- **[Policy validation]** Validate custom IAM policies for Auto Scaling with IAM Access Analyzer before attaching them. — Catches overly permissive or malformed statements before they reach production. [doc](https://docs.aws.amazon.com/autoscaling/plans/userguide/security_iam_id-based-policy-examples.html)

## 🛡️ Reliability
- **[Availability Zones]** Span each Auto Scaling group across multiple Availability Zones and maintain at least one instance per zone. — Protects the workload from a single zone failure. [doc](https://docs.aws.amazon.com/autoscaling/ec2/userguide/disaster-recovery-resiliency.html)
- **[Load balancing]** Attach a load balancer and keep cross-zone load balancing enabled so traffic is distributed evenly across zones. — Reduces the impact of increased load on remaining instances during a failover. [doc](https://docs.aws.amazon.com/autoscaling/ec2/userguide/disaster-recovery-resiliency.html)
- **[Health checks]** Enable both EC2 status checks and Elastic Load Balancing health checks on the Auto Scaling group, with an appropriate health check grace period. — Ensures unhealthy or unreachable instances are detected and replaced automatically. [doc](https://docs.aws.amazon.com/autoscaling/ec2/userguide/ec2-auto-scaling-health-checks.html)
- **[Health check depth]** Choose shallow or deep ELB health checks based on whether you need to detect only host-level failures or failures of downstream dependencies. — Matches failure detection to the actual causes of application unavailability. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/choosing-the-right-health-check-with-elastic-load-balancing-and-ec2-auto-scaling/)
- **[Graceful termination]** Combine termination lifecycle hooks with an instance lifecycle policy so instances are retained instead of terminated if a graceful shutdown action is abandoned. — Prevents data loss or dropped connections when shutdown automation fails. [doc](https://docs.aws.amazon.com/autoscaling/ec2/userguide/gracefully-handle-instance-termination.html)
- **[Lifecycle hooks]** Use launch and terminate lifecycle hooks to complete configuration or data-preservation steps before an instance goes into service or is terminated. — Ensures instances are fully ready before serving traffic and that critical data is captured before shutdown. [doc](https://docs.aws.amazon.com/autoscaling/ec2/userguide/lifecycle-hooks-overview.html)
- **[Zone rebalancing]** Let Amazon EC2 Auto Scaling rebalance instances across Availability Zones after an outage recovers, rather than manually re-adjusting capacity. — Restores even distribution and resiliency automatically once zones recover. [doc](https://docs.aws.amazon.com/autoscaling/ec2/userguide/as-instance-termination.html)

## ⚡ Performance Efficiency
- **[Scaling policy]** Use target tracking scaling policies tied to a metric that scales proportionally with instance count, instead of manual or simple scaling. — Keeps capacity closely aligned with the actual demand curve. [doc](https://docs.aws.amazon.com/wellarchitected/latest/framework/perf_compute_hardware_scale_compute_resources_dynamically.html)
- **[Custom metrics]** Choose a scaling metric that reflects true utilization for the workload (e.g., queue depth for batch/transcoding jobs) rather than defaulting to CPU utilization. — Avoids scaling decisions that don't correlate with actual load. [doc](https://docs.aws.amazon.com/wellarchitected/latest/framework/perf_compute_hardware_scale_compute_resources_dynamically.html)
- **[Predictive scaling]** Use predictive scaling for workloads with recurring, predictable demand patterns. — Eliminates the need to overprovision capacity ahead of anticipated load. [doc](https://docs.aws.amazon.com/wellarchitected/latest/framework/perf_compute_hardware_scale_compute_resources_dynamically.html)
- **[Boot latency]** Use warm pools to keep pre-initialized instances ready for applications with long boot or configuration times. — Cuts scale-out latency significantly compared to launching and configuring instances from cold. [doc](https://docs.aws.amazon.com/autoscaling/ec2/userguide/ec2-auto-scaling-warm-pools.html)
- **[Metric warm-up]** Enable default instance warmup so new instances finish initializing before contributing data to scaling metrics. — Prevents premature or inaccurate scaling decisions from partially-warmed instances. [doc](https://docs.aws.amazon.com/cdk/api/v2/java/software/amazon/awscdk/services/autoscaling/package-summary.html)

## 💰 Cost Optimization
- **[Purchase options]** Use a mixed instances policy to combine On-Demand and Spot Instances, and multiple instance types, within a single Auto Scaling group. — Lowers costs while Auto Scaling automatically replaces any interrupted Spot Instances. [doc](https://docs.aws.amazon.com/AWSCloudFormation/latest/TemplateReference/aws-properties-autoscaling-autoscalinggroup-mixedinstancespolicy.html)
- **[Allocation strategy]** Select a capacity-optimized allocation strategy for Spot Instances when minimizing interruptions matters more than raw price. — Reduces interruption frequency by favoring instance pools with more available capacity. [doc](https://aws.amazon.com/getting-started/cost-optimization-essentials/)
- **[Dynamic scaling]** Scale capacity automatically with demand instead of maintaining fixed fleet sizes, including scheduled scaling for predictable traffic patterns. — Ensures you run only the capacity actually needed at any given time. [doc](https://aws.amazon.com/getting-started/cost-optimization-essentials/)
- **[Capacity Reservations]** For reservation-dependent workloads (GPU, HPC, ML), use the "reservations then balanced" Availability Zone distribution strategy so Auto Scaling prioritizes reserved capacity before launching on-demand elsewhere. — Maximizes utilization of pre-purchased capacity before incurring on-demand costs. [doc](https://docs.aws.amazon.com/autoscaling/ec2/userguide/ec2-auto-scaling-availability-zone-balanced.html)

## ⚙️ Operational Excellence
- **[Immutable infrastructure]** Bake configuration and security updates into a new AMI and launch template version instead of patching instances in place. — Keeps every instance in the group on a known, consistent, auditable configuration. [doc](https://aws.amazon.com/blogs/publicsector/documenting-the-use-of-amazon-ec2-auto-scaling-groups-in-dod/)
- **[Rolling updates]** Use instance refresh to roll out new launch template versions across an Auto Scaling group instead of manually terminating instances. — Applies updates safely with controlled, incremental replacement of instances. [doc](https://docs.aws.amazon.com/cli/latest/reference/autoscaling/start-instance-refresh.html)
- **[Refresh efficiency]** Enable `SkipMatching` during an instance refresh so instances already running the target launch template and instance type are not needlessly replaced. — Reduces unnecessary churn and speeds up the rollout. [doc](https://docs.aws.amazon.com/cli/latest/reference/autoscaling/start-instance-refresh.html)
- **[Launch templates]** Use launch templates rather than launch configurations for new Auto Scaling groups. — Launch templates support versioning, mixed instances policies, and newer EC2 features that launch configurations cannot. [doc](https://docs.aws.amazon.com/AWSCloudFormation/latest/TemplateReference/aws-properties-autoscaling-autoscalinggroup-mixedinstancespolicy.html)
- **[Scale-in testing]** Test scale-in events explicitly, not just scale-out, to confirm the workload handles instance termination without affecting user experience (e.g., sticky sessions). — Validates real-world resiliency of the full scaling lifecycle, not just growth. [doc](https://docs.aws.amazon.com/wellarchitected/latest/sustainability-pillar/sus_sus_user_a2.html)

## 🌱 Sustainability
- **[Elastic capacity]** Use dynamic scaling so infrastructure shrinks during periods of low demand rather than running fixed, overprovisioned capacity. — Reduces the energy and resource footprint of idle compute. [doc](https://docs.aws.amazon.com/wellarchitected/latest/sustainability-pillar/sus_sus_user_a2.html)
- **[Predictive scaling]** Use predictive scaling to proactively match capacity to anticipated demand instead of overprovisioning as a buffer. — Avoids running excess instances that consume resources without serving load. [doc](https://docs.aws.amazon.com/wellarchitected/latest/sustainability-pillar/sus_sus_user_a2.html)
