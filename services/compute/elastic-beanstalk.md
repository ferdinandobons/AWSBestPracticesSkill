# AWS Elastic Beanstalk — Best Practices

## Common scenarios
- Deploying a web application without managing servers        → Operational Excellence, Cost Optimization
- Running background/worker processing on EC2                 → Reliability, Cost Optimization
- Migrating an app from Heroku or a traditional data center    → Operational Excellence, Security
- Hosting a multi-tier app with a load balancer and database   → Reliability, Security

## 🔒 Security
- **[IAM]** Implement least-privilege access by customizing the Elastic Beanstalk managed policies for instance profiles, service roles, and IAM users instead of using their full default permission sets. — Reduces the blast radius of compromised credentials or errant automation. [doc](https://docs.aws.amazon.com/elasticbeanstalk/latest/dg/security-best-practices.html)
- **[Secrets]** Retrieve credentials and API keys directly from AWS Secrets Manager or AWS Systems Manager Parameter Store from application code rather than embedding them in environment variables or source bundles. — Prevents sensitive data from leaking into logs, config files, or version control. [doc](https://docs.aws.amazon.com/elasticbeanstalk/latest/dg/security-best-practices.html)
- **[EC2 metadata]** Enforce IMDSv2 on environment instances and disable IMDSv1. — Mitigates SSRF-style attacks that could otherwise access instance metadata and steal instance-profile credentials. [doc](https://docs.aws.amazon.com/elasticbeanstalk/latest/dg/security-best-practices.html)
- **[Platform]** Keep environments on a supported, current platform version, ideally via managed platform updates. — Applies OS, runtime, and web/app server security patches automatically without downtime. [doc](https://docs.aws.amazon.com/elasticbeanstalk/latest/dg/security-best-practices.html)
- **[Network]** Place EC2 instances in private subnets and keep only the Application Load Balancer in public subnets, using a NAT gateway for any required outbound internet access. — Removes direct internet exposure of application instances while preserving inbound access through the ALB. [doc](https://aws.amazon.com/blogs/security/hardening-the-security-of-your-aws-elastic-beanstalk-application-the-well-architected-way/)
- **[Data at rest]** Explicitly enable encryption on the Elastic Beanstalk S3 bucket (logs, source bundles) and on any attached RDS database, since neither is encrypted by default. — Protects application data and artifacts from unauthorized access if storage is exposed. [doc](https://aws.amazon.com/blogs/security/hardening-the-security-of-your-aws-elastic-beanstalk-application-the-well-architected-way/)
- **[Data in transit]** Terminate TLS at the Application Load Balancer using an AWS Certificate Manager certificate instead of relying on the default HTTP listener. — Ensures client traffic is encrypted end-to-end instead of using Elastic Beanstalk's HTTP-by-default setup. [doc](https://aws.amazon.com/blogs/security/hardening-the-security-of-your-aws-elastic-beanstalk-application-the-well-architected-way/)
- **[Monitoring]** Enable AWS Config for Elastic Beanstalk resources and track API activity (e.g., `UpdateEnvironment`, `TerminateEnvironment`) with AWS CloudTrail. — Provides detective controls to catch noncompliant configuration or unauthorized environment changes after the fact. [doc](https://docs.aws.amazon.com/elasticbeanstalk/latest/dg/security-best-practices.html)

## 🛡️ Reliability
- **[High availability]** Deploy load-balanced environments across multiple Availability Zones with an Auto Scaling group sized for N+1 capacity. — Keeps the application serving traffic if a single Availability Zone becomes unavailable. [doc](https://docs.aws.amazon.com/elasticbeanstalk/latest/dg/AWSHowTo.html)
- **[Database]** Run RDS in Multi-AZ with automated backups and an appropriate retention period, and decouple the database from the environment lifecycle for production workloads. — Avoids data loss or unwanted database termination when an environment is replaced or terminated. [doc](https://docs.aws.amazon.com/elasticbeanstalk/latest/dg/concepts.concepts.design.html)
- **[Deployments]** Choose a deployment policy (rolling, rolling with additional batch, immutable, or traffic splitting) that matches your downtime and rollback tolerance instead of the default all-at-once policy. — Immutable and traffic-splitting deployments isolate failed new versions and roll back automatically without impacting the running fleet. [doc](https://docs.aws.amazon.com/elasticbeanstalk/latest/dg/using-features.deploy-existing-version.html)
- **[Zero-downtime releases]** Use blue/green deployments with environment URL swap for major or incompatible platform changes, and verify the new environment before terminating the old one. — Enables instant rollback by swapping URLs back if the new version misbehaves. [doc](https://docs.aws.amazon.com/elasticbeanstalk/latest/dg/using-features.CNAMESwap.html)
- **[Health monitoring]** Enable enhanced health reporting so Elastic Beanstalk tracks application-level signals (5XX rates, CPU, memory, disk) in addition to basic EC2/ELB health checks. — Surfaces degraded application behavior that basic health checks would miss. [doc](https://aws.amazon.com/blogs/aws/elastic-beanstalk-update-enhanced-application-health-monitoring/)
- **[Fleet management]** Configure Auto Scaling to maintain a fixed minimum fleet size so unhealthy instances are automatically replaced. — Ensures automated recovery from hardware failure or performance degradation without manual intervention. [doc](https://docs.aws.amazon.com/elasticbeanstalk/latest/dg/concepts.concepts.design.html)

## ⚡ Performance Efficiency
- **[Scaling]** Configure Auto Scaling triggers (e.g., CloudWatch alarms on network traffic, CPU, or custom metrics) appropriate to your application's actual load pattern rather than relying on the default thresholds. — Ensures the environment scales in and out based on real demand instead of generic defaults. [doc](https://docs.aws.amazon.com/elasticbeanstalk/latest/dg/using-features.managing.as.html)
- **[Instance sizing]** Review and adjust the EC2 instance type, EBS storage, and other launch configuration settings for your workload's actual CPU/memory/IO profile. — Right-sized instances avoid both performance bottlenecks and wasted capacity. [doc](https://docs.aws.amazon.com/elasticbeanstalk/latest/dg/using-features.managing.as.html)

## 💰 Cost Optimization
- **[Compute]** Use Spot Instances (mixed with On-Demand via `--on-demand-base-capacity` / `--on-demand-above-base-capacity`) for stateless, fault-tolerant, or dev/test Elastic Beanstalk workloads. — Spot Instances can cut compute costs by up to 90% compared to On-Demand pricing. [doc](https://docs.aws.amazon.com/elasticbeanstalk/latest/dg/environments-cfg-autoscaling-spot.html)
- **[Spot resilience]** Enable Capacity Rebalancing when using Spot Instances so Auto Scaling proactively replaces instances at elevated interruption risk. — Minimizes application impact from Spot interruptions while still capturing the cost savings. [doc](https://docs.aws.amazon.com/elasticbeanstalk/latest/dg/environments-cfg-autoscaling-spot.html)
- **[Non-production environments]** Automate stopping or scaling down test/staging Elastic Beanstalk environments outside business hours. — Test and staging workloads rarely need to run continuously, so scheduling reduces idle spend. [doc](https://aws.amazon.com/blogs/devops/optimizing-the-cost-of-running-aws-elastic-beanstalk-workloads/)

## ⚙️ Operational Excellence
- **[Patching]** Enable managed platform updates with a defined weekly maintenance window so patch and minor platform versions are applied automatically using an immutable, zero-capacity-loss update process. — Keeps environments current on security and bug fixes without manual update operations. [doc](https://docs.aws.amazon.com/elasticbeanstalk/latest/dg/environment-platform-update-managed.html)
- **[Observability]** Forward Elastic Beanstalk enhanced health agent logs and metrics to CloudWatch and set alarms on key environment and custom application metrics. — Gives operators near-real-time visibility into environment and application health for faster incident response. [doc](https://docs.aws.amazon.com/elasticbeanstalk/latest/dg/AWSHowTo.cloudwatch.html)
- **[Governance]** Use AWS Config rules (e.g., the managed rule checking for managed-updates-enabled) to continuously verify Elastic Beanstalk environments meet your operational standards. — Flags noncompliant environments automatically instead of relying on manual audits. [doc](https://docs.aws.amazon.com/awssupport/latest/user/operational-excellence-checks.html)
