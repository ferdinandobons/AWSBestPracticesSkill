# Amazon EC2 — Best Practices

## Common scenarios
- Hosting general-purpose web applications and backend services        → Reliability, Performance Efficiency, Cost Optimization
- Running self-managed databases or stateful workloads on instance storage/EBS        → Reliability, Security
- Auto-scaled fleets behind a load balancer for variable traffic        → Reliability, Cost Optimization, Performance Efficiency
- Batch, dev/test, or fault-tolerant workloads that can use flexible capacity        → Cost Optimization, Sustainability

## 🔒 Security
- **[identity and access]** Use identity federation with an identity provider and IAM roles instead of long-lived credentials to manage access to instances and APIs — reduces the risk of leaked static credentials. [doc](https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/ec2-best-practices.html)
- **[network access]** Implement the least permissive rules for your security groups — minimizes the attack surface exposed by each instance. [doc](https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/ec2-best-practices.html)
- **[instance access]** Use EC2 Instance Connect instead of opening SSH ports or standing up a bastion/jump host — avoids exposing inbound management ports to the internet. [doc](https://aws.amazon.com/getting-started/aws-security-essentials/)
- **[patching]** Regularly patch, update, and secure the guest operating system and applications on your instances — closes known vulnerabilities before they can be exploited. [doc](https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/ec2-best-practices.html)
- **[vulnerability management]** Use Amazon Inspector to automatically discover and scan EC2 instances for software vulnerabilities and unintended network exposure — provides continuous, automated assessment instead of point-in-time scans. [doc](https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/ec2-best-practices.html)
- **[compliance monitoring]** Use AWS Security Hub CSPM controls to monitor EC2 resources against security best practices and standards — surfaces drift from your security baseline in one place. [doc](https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/ec2-best-practices.html)
- **[data protection]** Encrypt EBS volumes and snapshots — protects data at rest and in transit between the instance and the volume. [doc](https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/ec2-best-practices.html)
- **[immutable infrastructure]** Build hardened, immutable AMIs (for example with EC2 Image Builder) and replace instances on update rather than patching in place — keeps fleets consistently secured and simplifies rollback. [doc](https://aws.amazon.com/blogs/security/security-at-multiple-layers-for-web-administered-apps/)
- **[credentials on instances]** Use IAM roles attached to the instance (instance profiles) instead of hard-coded credentials so applications get temporary, least-privilege access to other AWS resources. [doc](https://aws.amazon.com/blogs/security/security-at-multiple-layers-for-web-administered-apps/)

## 🛡️ Reliability
- **[multi-AZ design]** Deploy critical components of your application across multiple Availability Zones and replicate data appropriately — protects against the loss of a single AZ. [doc](https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/ec2-best-practices.html)
- **[Auto Scaling]** Span Auto Scaling groups across multiple Availability Zones and maintain at least one instance in each — preserves capacity if one AZ becomes impaired. [doc](https://docs.aws.amazon.com/autoscaling/ec2/userguide/disaster-recovery-resiliency.html)
- **[load balancing]** Attach a load balancer to distribute traffic across Availability Zones, keeping cross-zone load balancing enabled for Application Load Balancers — limits the blast radius of an AZ failover and improves traffic distribution. [doc](https://docs.aws.amazon.com/autoscaling/ec2/userguide/disaster-recovery-resiliency.html)
- **[health checks]** Configure Elastic Load Balancing health checks correctly and enable them on the Auto Scaling group — ensures unhealthy instances are removed from traffic and replaced automatically. [doc](https://docs.aws.amazon.com/autoscaling/ec2/userguide/disaster-recovery-resiliency.html)
- **[failover readiness]** Design applications to handle dynamic IP addressing on instance restart, and prepare an attach path for network interfaces or Elastic IPs to a replacement instance — avoids hard dependencies on ephemeral addressing. [doc](https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/ec2-best-practices.html)
- **[capacity flexibility]** Use EC2 Fleet with multiple instance types/attribute-based instance selection and implement retries for `InsufficientInstanceCapacity`/`InternalError` errors — improves the odds of getting capacity when On-Demand supply is constrained. [doc](https://aws.amazon.com/blogs/compute/how-to-prepare-your-application-to-scale-reliably-with-amazon-ec2/)
- **[scaling technique]** Scale out in smaller, more frequent batches rather than large infrequent jumps, starting as soon as load increases are detected — improves reliability and reduces the risk of insufficient capacity during spikes. [doc](https://aws.amazon.com/blogs/compute/how-to-prepare-your-application-to-scale-reliably-with-amazon-ec2/)
- **[backup and recovery]** Regularly back up EBS volumes with snapshots, create AMIs as recovery templates, and regularly test instance/volume recovery — validates that your recovery process actually works before you need it. [doc](https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/ec2-best-practices.html)
- **[geographic resilience]** Use Local Zones or additional AWS Regions/Availability Zones as failover targets when replicating data over greater distances — extends resiliency beyond a single Region or AZ. [doc](https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/disaster-recovery-resiliency.html)

## ⚡ Performance Efficiency
- **[instance selection]** Use AWS Compute Optimizer to analyze utilization history and get instance type recommendations, including moves to newer generations or different families — removes guesswork from matching instance type to workload. [doc](https://docs.aws.amazon.com/autoscaling/ec2/userguide/asg-getting-recommendations.html)
- **[validation]** Test recommended instance type changes thoroughly before applying them in production — confirms the new configuration actually meets the workload's performance needs. [doc](https://docs.aws.amazon.com/help-panel/compute-optimizer/latest/console/ec2-instances-details.html)
- **[storage placement]** Use instance store for temporary data and separate EBS volumes for data that must persist, understanding that instance store data is lost on stop/hibernate/terminate — prevents accidental data loss and matches storage type to durability needs. [doc](https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/ec2-best-practices.html)

## 💰 Cost Optimization
- **[rightsizing]** Identify idle or low-utilization instances with Cost Explorer resource optimization and AWS Compute Optimizer, then stop, schedule, or rightsize them — eliminates spend on unused or over-provisioned capacity. [doc](https://aws.amazon.com/blogs/compute/10-things-you-can-do-today-to-reduce-aws-costs/)
- **[purchase options]** Use Savings Plans, Reserved Instances, or Spot Instances for steady-state or fault-tolerant/ephemeral workloads instead of only On-Demand — matches purchase model to workload predictability for lower effective cost. [doc](https://docs.aws.amazon.com/wellarchitected/latest/management-and-governance-guide/implementation-priorities-6.html)
- **[continuous review]** Periodically review historic spend to detect cost spikes and re-run right-sizing analysis as workload characteristics change — keeps cost efficiency aligned with actual usage rather than a one-time decision. [doc](https://docs.aws.amazon.com/wellarchitected/latest/management-and-governance-guide/implementation-priorities-6.html)
- **[instance generation]** Review new EC2 instance types and generations as they are released and migrate to take advantage of better price-performance — avoids paying more for older-generation hardware. [doc](https://docs.aws.amazon.com/wellarchitected/latest/management-and-governance-guide/implementation-priorities-6.html)
- **[automation]** Automate stopping and starting non-production instances outside business hours (for example with an instance scheduler) — cuts spend on environments that don't need to run continuously. [doc](https://docs.aws.amazon.com/whitepapers/latest/tagging-best-practices/automated-infrastructure-activities.html)
- **[monitoring recommendations]** Use AWS Trusted Advisor to inspect your environment for cost-saving, performance, and security opportunities — surfaces optimization opportunities you might otherwise miss. [doc](https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/ec2-best-practices.html)

## ⚙️ Operational Excellence
- **[resource tracking]** Use instance metadata and custom resource tags to track and identify EC2 resources — makes it possible to filter, automate, and attribute resources at scale. [doc](https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/ec2-best-practices.html)
- **[quota management]** View current Amazon EC2 service quotas and request increases well in advance of need — prevents launch failures caused by hitting account limits unexpectedly. [doc](https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/ec2-best-practices.html)
- **[monitoring and response]** Monitor EC2 resources and configure alerting so operators can respond to events promptly — reduces mean time to detect and mean time to recover from incidents. [doc](https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/ec2-best-practices.html)
- **[alarm design]** Use tags to organize and automate CloudWatch alarms per resource/team, and avoid over-alerting that can overwhelm operators — keeps alerts actionable and focused on real upstream issues. [doc](https://docs.aws.amazon.com/whitepapers/latest/tagging-best-practices/operational-observability.html)
- **[automation]** Use AWS Systems Manager to run patching, automation documents, and runbooks against tagged fleets of instances — standardizes operational tasks instead of relying on manual, per-instance actions. [doc](https://docs.aws.amazon.com/whitepapers/latest/tagging-best-practices/automated-infrastructure-activities.html)
