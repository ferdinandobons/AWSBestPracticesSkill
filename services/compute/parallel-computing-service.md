# AWS Parallel Computing Service — Best Practices

## Common scenarios
- Traditional HPC workloads (CFD, weather, genomics) run with Slurm on managed clusters        → Reliability, Performance Efficiency
- GPU/EFA-accelerated simulation and ML training clusters        → Performance Efficiency, Cost Optimization
- Bursty scientific/engineering compute that scales node groups up and down        → Cost Optimization, Reliability
- Multi-tenant HPC environments needing isolation, auditing, and compliance        → Security, Operational Excellence

## 🔒 Security
- **[AMIs]** Avoid using AWS PCS sample AMIs for production workloads — they are unsupported and intended only for testing. [doc](https://docs.aws.amazon.com/pcs/latest/userguide/security-best-practices.html)
- **[AMIs]** Regularly update the operating system and software in compute node group AMIs, using only authenticated official AWS PCS packages from official AWS sources — mitigates known vulnerabilities. [doc](https://docs.aws.amazon.com/pcs/latest/userguide/security-best-practices.html)
- **[AMIs]** Automate periodic rebuilds and rollout of updated AMIs to compute node groups — minimizes the window of exposure to unpatched vulnerabilities. [doc](https://docs.aws.amazon.com/pcs/latest/userguide/security-best-practices.html)
- **[Slurm]** Restrict access to Slurm control and compute nodes to trusted users and systems, and rely on Slurm's built-in authentication for job submissions and management commands. [doc](https://docs.aws.amazon.com/pcs/latest/userguide/security-best-practices.html)
- **[Slurm]** Keep clusters on a supported Slurm version — a cluster running a version that reaches end of support life (EOSL) is stopped immediately, so track the AWS PCS Slurm versions RSS feed and upgrade proactively. [doc](https://docs.aws.amazon.com/pcs/latest/userguide/security-best-practices.html)
- **[Slurm]** Regularly rotate cluster secrets to maintain security compliance and remediate potential compromises — required for HIPAA and FedRAMP compliance. [doc](https://docs.aws.amazon.com/pcs/latest/userguide/security-best-practices.html)
- **[IAM]** Start from AWS managed policies and move toward least-privilege customer-managed policies scoped to specific AWS PCS actions and resources. [doc](https://docs.aws.amazon.com/pcs/latest/userguide/security_iam_id-based-policy-examples.html)
- **[IAM]** Add IAM policy conditions (for example, requiring SSL/TLS or scoping actions to calls made through a specific AWS service) to further restrict when and how AWS PCS permissions can be used. [doc](https://docs.aws.amazon.com/pcs/latest/userguide/security_iam_id-based-policy-examples.html)
- **[IAM]** Validate identity-based policies with IAM Access Analyzer before attaching them to catch overly permissive or non-functional statements. [doc](https://docs.aws.amazon.com/pcs/latest/userguide/security_iam_id-based-policy-examples.html)
- **[IAM]** Require multi-factor authentication (MFA) for IAM users or the root user, and add MFA conditions to sensitive AWS PCS API calls. [doc](https://docs.aws.amazon.com/pcs/latest/userguide/security_iam_id-based-policy-examples.html)
- **[Network]** Deploy AWS PCS clusters in a dedicated VPC to isolate the HPC environment from other network traffic. [doc](https://docs.aws.amazon.com/pcs/latest/userguide/security-best-practices.html)
- **[Network]** Use security groups and network ACLs to tightly control inbound and outbound traffic to AWS PCS instances and subnets. [doc](https://docs.aws.amazon.com/pcs/latest/userguide/security-best-practices.html)
- **[Network]** Use an AWS PrivateLink interface endpoint for AWS PCS so API traffic between clusters and other AWS services stays on the AWS network instead of the public internet. [doc](https://docs.aws.amazon.com/pcs/latest/userguide/vpc-interface-endpoints.html)
- **[Network]** Attach a custom endpoint policy to the AWS PCS interface endpoint to scope which principals, actions, and clusters can be reached through it — the default policy allows full access. [doc](https://docs.aws.amazon.com/pcs/latest/userguide/vpc-interface-endpoints.html)
- **[Data protection]** Use a customer managed KMS key (instead of only the AWS owned key) to encrypt EBS volumes when you need control over key policy and rotation. [doc](https://docs.aws.amazon.com/pcs/latest/userguide/data-protection.html)
- **[Data protection]** Never place confidential or sensitive information, such as customer email addresses, in tags or free-form name fields — this data can surface in billing or diagnostic logs. [doc](https://docs.aws.amazon.com/pcs/latest/userguide/data-protection.html)
- **[Data protection]** Connect to cluster nodes using SSH or AWS Systems Manager rather than exposing management access directly over the internet. [doc](https://docs.aws.amazon.com/pcs/latest/userguide/data-protection.html)
- **[Monitoring/audit]** Enable AWS CloudTrail to capture AWS PCS API calls for troubleshooting and auditing who created, modified, or deleted clusters and node groups. [doc](https://docs.aws.amazon.com/pcs/latest/userguide/security-best-practices.html)

## 🛡️ Reliability
- **[Cluster sizing]** Choose a cluster size (Small/Medium/Large) based on the expected number of managed instances and concurrently tracked jobs, and plan for a new cluster if requirements grow — cluster size can't be changed after creation. [doc](https://docs.aws.amazon.com/pcs/latest/userguide/working-with_clusters_size.html)
- **[Quotas]** Check and request increases for relevant Amazon EC2 quotas (Spot requests, On-Demand instances, launch templates) before scaling, since AWS PCS scaling is bound by these underlying EC2 quotas. [doc](https://docs.aws.amazon.com/pcs/latest/userguide/service-endpoints-quotas.html)
- **[Availability]** Design clusters to take advantage of multiple Availability Zones in a Region so workloads can continue if one zone is impaired. [doc](https://docs.aws.amazon.com/pcs/latest/userguide/disaster-recovery-resiliency.html)
- **[Scaling behavior]** When launching large batches of instances at once, account for the default limit of up to 500 instances launched per batch, and consider scaling in batches of 500 or fewer to avoid partial-capacity stalls. [doc](https://aws.amazon.com/blogs/hpc/minimize-hpc-compute-costs-with-all-or-nothing-instance-launching/)
- **[Scaling behavior]** Space out rapid job submissions by roughly a minute where possible — Slurm batches instance requests together, and back-to-back submissions can be chunked into one oversized request that risks failing on capacity limits. [doc](https://aws.amazon.com/blogs/hpc/minimize-hpc-compute-costs-with-all-or-nothing-instance-launching/)
- **[Cluster updates]** Follow the documented Slurm version update procedure — update compute node group AMIs and restore scaling limits methodically — to upgrade schedulers without disrupting running capacity. [doc](https://docs.aws.amazon.com/pcs/latest/userguide/working-with_clusters_version_update_procedure.html)

## ⚡ Performance Efficiency
- **[Node groups]** Organize compute node groups by instance type and workload profile (for example, separate CPU, single-GPU, and multi-GPU node groups mapped to distinct queues) so job scheduling matches jobs to the right hardware. [doc](https://aws.amazon.com/blogs/hpc/scalable-cryo-em-on-aws-parallel-computing-service-pcs/)
- **[Instance selection]** Base instance type choices for each node group on workload-specific benchmarking rather than defaults — different HPC and AI/ML stages often perform best on different instance families. [doc](https://aws.amazon.com/blogs/hpc/scalable-cryo-em-on-aws-parallel-computing-service-pcs/)
- **[Networking]** Use Elastic Fabric Adapter (EFA) enabled instances and monitor EFA-specific metrics (device details, RDMA reads/writes, dropped packets) for tightly-coupled, latency-sensitive HPC workloads. [doc](https://aws.amazon.com/blogs/hpc/the-complete-picture-unified-monitoring-for-aws-parallel-computing-service/)
- **[Observability]** Track CapacityUtilization and IdleCapacity CloudWatch metrics for each cluster to right-size node groups and identify underutilized or bottlenecked capacity. [doc](https://docs.aws.amazon.com/pcs/latest/userguide/monitoring-cloudwatch_metrics.html)

## 💰 Cost Optimization
- **[Node scaling]** Set compute node group minimum instance counts to 0 so idle capacity scales down fully between jobs rather than remaining provisioned. [doc](https://docs.aws.amazon.com/pcs/latest/userguide/getting-started_create-cng_workers.html)
- **[Purchase options]** Use Amazon EC2 Spot Instances for flexible, interruption-tolerant HPC compute node groups to substantially lower cost versus On-Demand. [doc](https://docs.aws.amazon.com/whitepapers/latest/financial-services-grid-computing/instance-procurement.html)
- **[Purchase options]** Reserve Savings Plans or On-Demand for long-running, steady-state components such as HPC controller-adjacent baseline capacity, and use Spot for elastic burst capacity — matches pricing model to each workload's flexibility. [doc](https://docs.aws.amazon.com/whitepapers/latest/financial-services-grid-computing/instance-procurement.html)
- **[Capacity reservation]** Use AWS Capacity Blocks for ML to reserve latest-generation GPU instances for a defined future window when running time-bound training or fine-tuning jobs — avoids paying for idle reserved GPU capacity outside that window. [doc](https://docs.aws.amazon.com/whitepapers/latest/financial-services-grid-computing/instance-procurement.html)
- **[Observability]** Use CapacityUtilization and IdleCapacity CloudWatch metrics to identify underutilized node groups and adjust scaling configuration or instance mix to reduce idle spend. [doc](https://aws.amazon.com/blogs/hpc/the-complete-picture-unified-monitoring-for-aws-parallel-computing-service/)

## ⚙️ Operational Excellence
- **[Monitoring]** Use Amazon CloudWatch to monitor AWS PCS cluster health and performance, including ActualCapacity, DesiredCapacity, IdleCapacity, and UtilizedCapacity metrics published under the AWS/PCS namespace. [doc](https://docs.aws.amazon.com/pcs/latest/userguide/monitoring-cloudwatch_metrics.html)
- **[Monitoring]** Configure the CloudWatch agent via the EC2 launch template on compute nodes to persist metrics and logs after instance termination — CloudWatch data is otherwise typically lost once ephemeral instances terminate. [doc](https://docs.aws.amazon.com/pcs/latest/userguide/monitoring-cloudwatch.html)
- **[Logging]** Send Slurm, job, and instance logs to Amazon CloudWatch Logs, and use AWS CloudTrail to capture API activity, so operators can trace who performed cluster and node group actions and when. [doc](https://docs.aws.amazon.com/pcs/latest/userguide/monitoring-overview.html)
- **[Observability]** Build unified dashboards (for example with Amazon Managed Grafana and Amazon Managed Service for Prometheus, sourced from CloudWatch Logs, Slurm Exporter, EFA Exporter, and DCGM Exporter) for real-time visibility into jobs, nodes, GPUs, and partitions — speeds diagnosis of long-running job degradation. [doc](https://aws.amazon.com/blogs/hpc/the-complete-picture-unified-monitoring-for-aws-parallel-computing-service/)
- **[IAM permissions]** Verify the IAM principal used for cluster setup has the required permissions for AWS PCS, service-linked roles, AWS CloudFormation, and VPC resources, and perform related setup steps as the same principal to avoid permission mismatches. [doc](https://docs.aws.amazon.com/pcs/latest/userguide/required-iam-permissions.html)
