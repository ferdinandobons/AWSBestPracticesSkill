# AWS Outposts — Best Practices

## Common scenarios
- Running low-latency, on-premises workloads that need local data processing        → Performance Efficiency, Reliability
- Meeting data residency or sovereignty requirements without a local AWS Region        → Security, Reliability
- Extending a VPC on-premises for consistent hybrid operations and migration        → Operational Excellence, Reliability
- Hosting regulated workloads (gaming, healthcare, finance) requiring physical data control        → Security, Cost Optimization

## 🔒 Security
- **[IAM policies]** Start from AWS managed policies and move toward least-privilege customer-managed policies for Outposts actions — limits which identities can create, modify, or delete Outposts resources. [doc](https://docs.aws.amazon.com/outposts/latest/userguide/security_iam_id-based-policy-examples.html)
- **[IAM conditions]** Add policy conditions (for example requiring SSL, or scoping actions to specific AWS services like CloudFormation) — further restricts how and when Outposts permissions can be used. [doc](https://docs.aws.amazon.com/outposts/latest/userguide/security_iam_id-based-policy-examples.html)
- **[policy validation]** Validate IAM policies with IAM Access Analyzer before attaching them — catches overly permissive or malformed policies before they grant unintended access. [doc](https://docs.aws.amazon.com/outposts/latest/userguide/security_iam_id-based-policy-examples.html)
- **[authentication]** Require multi-factor authentication (MFA) for IAM users and the root user, and add MFA conditions to sensitive Outposts API calls — reduces the impact of credential compromise. [doc](https://docs.aws.amazon.com/outposts/latest/userguide/security_iam_id-based-policy-examples.html)
- **[physical access control]** Restrict physical access to the location where Outposts hardware is deployed using controls such as keyed access and biometrics — prevents unauthorized personnel from tampering with the rack or server, which is the customer's responsibility under the shared responsibility model. [doc](https://docs.aws.amazon.com/wellarchitected/latest/data-residency-hybrid-cloud-services-lens/drhcsec06-bp02.html)
- **[encryption at rest]** Rely on default encryption at rest for EBS volumes and S3 on Outposts (data keys are wrapped by the removable Nitro Security Key), and additionally enable Amazon EBS encryption with AWS KMS keys where supported — protects data even if the physical media is removed. [doc](https://docs.aws.amazon.com/outposts/latest/userguide/data-protection.html)
- **[encryption in transit]** Ensure Outposts service-link traffic to the parent Region stays encrypted, and use Direct Connect private virtual interfaces instead of the public internet where possible — keeps management and data traffic off the public internet. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/hybrid-cloud-best-practices/networking.html)
- **[secure remote access]** Use an interface VPC endpoint (AWS PrivateLink) for AWS Systems Manager so instance management traffic stays private, and use Systems Manager Session Manager instead of opening inbound SSH/RDP — reduces the network attack surface on Outposts instances. [doc](https://aws.amazon.com/blogs/compute/managing-and-securing-aws-outposts-instances-using-aws-systems-manager-amazon-inspector-and-amazon-guardduty/)
- **[threat detection]** Enable Amazon GuardDuty and Amazon Inspector for accounts running Outposts workloads — continuously monitors CloudTrail, VPC Flow Logs, and DNS logs for malicious activity and surfaces vulnerabilities on Outposts instances. [doc](https://aws.amazon.com/blogs/compute/managing-and-securing-aws-outposts-instances-using-aws-systems-manager-amazon-inspector-and-amazon-guardduty/)
- **[network segmentation]** Use multiple local gateway (LGW) routing domains to isolate network segments and combine customer-owned IP (CoIP) and direct VPC routing (DVR) modes as needed — limits the blast radius if one segment is compromised and helps meet regulated-industry segmentation requirements. [doc](https://aws.amazon.com/blogs/compute/simplify-network-segmentation-for-aws-outposts-racks-with-multiple-local-gateway-routing-domains/)
- **[data residency guardrails]** Use AWS Control Tower landing zone guardrails and Organizations service control policies to restrict data snapshots, prohibit unintended data creation in the Region, and limit data transfer off the Outpost — enforces data residency and sovereignty requirements programmatically rather than relying on manual process. [doc](https://aws.amazon.com/blogs/compute/architecting-for-data-residency-with-aws-outposts-rack-and-landing-zone-guardrails/)
- **[compliance verification]** Verify that your Outposts facility meets the physical, environmental, and regulatory requirements for the workloads you plan to run before deployment — avoids discovering compliance gaps after the Outpost is installed. [doc](https://docs.aws.amazon.com/wellarchitected/latest/data-residency-hybrid-cloud-services-lens/prepare.html)

## 🛡️ Reliability
- **[multi-Outpost design]** Deploy applications across two or more Outposts, each connected to a different Availability Zone, using an active-active design — protects workloads from the loss or maintenance of a single Outpost. [doc](https://docs.aws.amazon.com/outposts/latest/server-userguide/updates-service-link.html)
- **[placement groups]** Use EC2 placement groups on Outposts racks to spread instances across distinct racks — reduces the impact of a single hardware failure on a highly available application. [doc](https://aws.amazon.com/outposts/rack/features/)
- **[service-link connectivity]** Provision redundant, highly available connectivity of at least 500 Mbps (1 Gbps or higher recommended) between the Outpost and its parent Region, ideally using Direct Connect — service-link loss can disrupt management and Region-dependent operations. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/hybrid-cloud-best-practices/networking.html)
- **[protect service-link resources]** Apply a service control policy that restricts deletion of the VPC/subnets, Transit Gateway/route tables, and virtual gateway resources that carry the service link — prevents an accidental configuration change from silently dropping Outposts connectivity to the Region. [doc](https://aws.amazon.com/blogs/networking-and-content-delivery/introducing-aws-outposts-private-connectivity/)
- **[capacity headroom]** Plan spare N+M capacity per instance family (N assets needed plus M for failure/maintenance headroom) and spread instances across as many assets as possible — ensures Auto Scaling and failover can still launch replacement instances during a hardware or maintenance event. [doc](https://aws.amazon.com/blogs/compute/control-instance-placement-using-asset-level-capacity-management-for-aws-outposts/)
- **[capacity planning]** Because Outposts compute, storage, and network capacity is finite, plan ahead for required resources rather than assuming Region-like elasticity — proactive planning is required to sustain availability and data residency requirements under failure conditions. [doc](https://docs.aws.amazon.com/wellarchitected/latest/data-residency-hybrid-cloud-services-lens/drhcrel04-bp01.html)
- **[disaster recovery]** Define clear RTO/RPO targets and build a matching DR solution — for example EBS snapshots to the parent Region for lower-cost recovery, or AWS Elastic Disaster Recovery for near-continuous replication and faster recovery of critical workloads. [doc](https://docs.aws.amazon.com/wellarchitected/latest/data-residency-hybrid-cloud-services-lens/drhcops03-bp01.html)
- **[network path resilience]** Prefer the local gateway path over the service-link path for on-premises and internet-bound traffic, provision redundant network paths to critical on-premises resources, and use dynamic routing to reroute around on-premises network failures — reduces dependence on a single network path for workload availability. [doc](https://docs.aws.amazon.com/wellarchitected/latest/data-residency-hybrid-cloud-services-lens/drhcops04-bp02.html)
- **[no local Region]** For jurisdictions without a local AWS Region, use multiple Outposts across locations (or Local Zones with redundancy) plus services like Amazon S3 on Outposts for resilient local data management — meets data residency needs while retaining redundancy against a single-site failure. [doc](https://docs.aws.amazon.com/wellarchitected/latest/data-residency-hybrid-cloud-services-lens/drhcrel03-bp01.html)

## ⚡ Performance Efficiency
- **[dedicated hosts and recovery]** Use Dedicated Hosts on Outposts where license or isolation requirements apply, and configure instance recovery so instances are automatically restarted on healthy hardware — keeps compute optimization strategies working the same way they do in the Region. [doc](https://docs.aws.amazon.com/outposts/latest/userguide/outposts-optimizations.html)
- **[MTU and bandwidth]** Ensure the network between the Outpost and the service-link endpoints in the parent Region supports the required MTU (1,500 bytes) and sufficient bandwidth — undersized or misconfigured links degrade both management operations and application performance. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/hybrid-cloud-best-practices/networking.html)
- **[direct VPC routing]** Use direct VPC routing where Outpost instances need to reach resources in Regional VPCs by private IP — avoids the extra hop and latency of routing through a VPN or Direct Connect for intra-VPC traffic. [doc](https://docs.aws.amazon.com/wellarchitected/latest/data-residency-hybrid-cloud-services-lens/drhcops04-bp02.html)
- **[service selection]** Use in-Region AWS services (identity, automation, monitoring) for components of the application that are not subject to data residency restrictions, reserving Outposts capacity for the parts that must stay local — keeps the architecture from over-provisioning finite on-premises capacity for workloads that don't need it. [doc](https://aws.amazon.com/blogs/architecture/announcing-the-well-architected-data-residency-with-hybrid-cloud-services-lens/)

## 💰 Cost Optimization
- **[right-size before ordering]** Engage an AWS Outposts specialist to validate that the capacity you plan to order matches actual workload requirements before purchase — Outposts capacity cannot be scaled down after deployment, so overprovisioning is a sunk cost. [doc](https://docs.aws.amazon.com/wellarchitected/latest/data-residency-hybrid-cloud-services-lens/drhcsus02-bp02.html)
- **[capacity monitoring]** Continuously monitor Outposts capacity and utilization with CloudWatch metrics and alarms, and use capacity tasks to reconfigure instance sizes/quantities as demand changes — avoids paying for stranded, unused capacity or hitting unexpected capacity limits. [doc](https://docs.aws.amazon.com/wellarchitected/latest/data-residency-hybrid-cloud-services-lens/drhccost02-bp01.html)
- **[dynamic reconfiguration]** Use capacity tasks to dynamically reconfigure instance types and sizes on existing Outposts assets as workload requirements change, rather than ordering additional hardware — gets more value from already-purchased capacity. [doc](https://aws.amazon.com/blogs/compute/dynamically-reconfigure-your-aws-outposts-capacity-using-capacity-tasks/)

## ⚙️ Operational Excellence
- **[centralized monitoring]** Use Amazon CloudWatch metrics, the AWS Health Dashboard, and AWS Health Aware together with centralized notifications and dashboards across accounts — gives operators a single view of Outposts health, capacity, and service-link status. [doc](https://aws.amazon.com/blogs/mt/monitoring-best-practices-for-aws-outposts/)
- **[logging]** Enable CloudTrail logs, VPC Flow Logs, and Traffic Mirroring for Outposts workloads — provides the audit trail and traffic visibility needed to investigate operational and security incidents. [doc](https://docs.aws.amazon.com/outposts/latest/server-userguide/monitor-outposts.html)
- **[patch management]** Manage Outposts instances with AWS Systems Manager, including Patch Manager baselines and maintenance windows, the same way you patch Regional EC2 fleets — keeps on-premises instances current without a separate, manual patching process. [doc](https://aws.amazon.com/blogs/compute/managing-and-securing-aws-outposts-instances-using-aws-systems-manager-amazon-inspector-and-amazon-guardduty/)
- **[maintenance resilience]** Design applications to tolerate brief latency or packet loss during service-link maintenance windows, since traffic on the local network interface is unaffected — avoids treating expected maintenance events as outages. [doc](https://docs.aws.amazon.com/outposts/latest/server-userguide/updates-service-link.html)
- **[capacity change timing]** Schedule capacity tasks for periods when you don't expect to launch new instances, since assets can be temporarily isolated while a task runs — prevents capacity reconfiguration from disrupting active deployments. [doc](https://docs.aws.amazon.com/outposts/latest/server-userguide/modify-instance-capacity.html)

## 🌱 Sustainability
- **[avoid overprovisioning]** Size Outposts orders to actual workload requirements with help from an AWS Outposts specialist instead of provisioning for worst-case headroom — minimizes the energy footprint of fixed, on-premises hardware that can't be scaled down later. [doc](https://docs.aws.amazon.com/wellarchitected/latest/data-residency-hybrid-cloud-services-lens/drhcsus02-bp02.html)
- **[maximize utilization]** Monitor workload resource utilization and EC2 instance consumption on the Outpost, and migrate portable workloads from the Region onto underutilized Outpost capacity where appropriate — makes fuller use of already-deployed fixed resources instead of running duplicate capacity in the Region. [doc](https://docs.aws.amazon.com/wellarchitected/latest/data-residency-hybrid-cloud-services-lens/drhcsus03-bp02.html)
