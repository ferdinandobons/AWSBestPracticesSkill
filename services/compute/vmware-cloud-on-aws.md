# VMware Cloud on AWS — Best Practices

## Common scenarios
- Lift-and-shift migration of on-premises vSphere workloads to AWS without re-architecting        → Reliability, Operational Excellence
- Running mission-critical VMs that require high availability across Availability Zones        → Reliability
- Establishing hybrid connectivity between an on-premises data center and an SDDC        → Reliability, Security
- Using VMware Cloud on AWS as a disaster recovery target for on-premises vSphere environments        → Reliability, Cost Optimization

## 🔒 Security
- **[Shared responsibility]** Understand that AWS and VMware jointly manage the underlying SDDC infrastructure and hypervisor, while you remain responsible for guest OS hardening, VM-level security policies, and workload configuration. — Misunderstanding the boundary leaves gaps in patching and access control that neither party covers by default. [doc](https://aws.amazon.com/blogs/apn/how-to-leverage-the-aws-cloud-adoption-framework-for-vmware-cloud-on-aws/)
- **[Native service integration]** Integrate VMware Cloud on AWS workloads with AWS Security Hub, AWS Config, and Amazon EventBridge for centralized security findings and governance rather than operating a separate on-premises-style toolchain. — SDDC-hosted VMs aren't covered by agentless services like GuardDuty/Inspector, so findings must be aggregated deliberately. [doc](https://aws.amazon.com/blogs/apn/how-to-leverage-the-aws-cloud-adoption-framework-for-vmware-cloud-on-aws/)
- **[Web application protection]** Front internet-facing applications hosted in the SDDC with AWS WAF (and an AWS load balancer) rather than assuming on-premises-style firewalls behave the same way in this environment. — Traditional on-prem WAF/load-balancer patterns don't translate directly, leaving web workloads unprotected against common exploits and bots. [doc](https://aws.amazon.com/blogs/apn/provide-enhanced-security-for-web-servers-in-vmware-cloud-on-aws-using-aws-waf/)
- **[Identity]** Use least-privilege vSphere role-based access control (RBAC) for SDDC administrators and combine it with IAM policies that scope AWS-side account linking and cross-account access. — Limits blast radius on both the VMware management plane and the connected AWS account. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/aws-storage-for-vmware-professionals/security-features.html)
- **[Compliance]** Map VMware-provided compliance documentation (via the VMware Trust Center) and AWS Artifact reports to your required compliance frameworks instead of assuming either party's certifications cover the whole stack. — Compliance responsibility is split, so gaps only surface when you explicitly reconcile both sets of attestations. [doc](https://aws.amazon.com/blogs/apn/how-to-leverage-the-aws-cloud-adoption-framework-for-vmware-cloud-on-aws/)
- **[Account governance]** Restrict the AWS Organizations OU/guardrails used for the VMware Cloud on AWS-linked account so they don't block the CloudFormation stack that performs SDDC-to-VPC account linking. — Overly restrictive Control Tower/SCP guardrails can silently break the IAM roles and Lambda functions the linking stack must create. [doc](https://docs.aws.amazon.com/whitepapers/latest/sddc-deployment-and-best-practices/account-requirements.html)

## 🛡️ Reliability
- **[Cluster resiliency]** Configure an appropriate vSAN Failures to Tolerate (FTT) policy for the cluster size, and move to FTT ≥ 2 (RAID-1/RAID-6) once a cluster grows to six or more hosts. — Continuing to run FTT-1 policies on larger clusters means VMware no longer guarantees availability per the service definition. [doc](https://aws.amazon.com/blogs/apn/resiliency-design-considerations-and-best-practices-for-vmware-cloud-on-aws/)
- **[Multi-AZ resiliency]** Deploy mission-critical workloads on a stretched cluster spanning two Availability Zones with synchronous replication, rather than a standard single-AZ cluster. — Provides near-zero RPO failover and automatic VM restart via vSphere HA if an entire Availability Zone fails. [doc](https://aws.amazon.com/blogs/apn/resiliency-design-considerations-and-best-practices-for-vmware-cloud-on-aws/)
- **[Connectivity]** Use AWS Direct Connect (ideally dual virtual interfaces over separate physical connections and locations) for production traffic instead of relying solely on IPsec VPN over the internet. — VPN alone lacks the consistent bandwidth and low latency production workloads need, and provides no location redundancy against a single connection failure. [doc](https://aws.amazon.com/blogs/apn/resiliency-design-considerations-and-best-practices-for-vmware-cloud-on-aws/)
- **[Connectivity]** When using VPN as a backup to Direct Connect, size expectations accordingly and avoid relying on VPN backup for workloads exceeding roughly 1 Gbps. — VPN throughput and latency are not comparable to Direct Connect, so failover under VPN may not sustain production load. [doc](https://aws.amazon.com/blogs/apn/resiliency-design-considerations-and-best-practices-for-vmware-cloud-on-aws/)
- **[Multi-SDDC connectivity]** Use VMware Transit Connect (a VMware-managed AWS Transit Gateway) to interconnect multiple SDDCs within an SDDC group and to on-premises, instead of manual VPN mesh configurations. — Simplifies operations at scale through automatic route propagation and eliminates per-connection VPN management overhead. [doc](https://aws.amazon.com/blogs/apn/vmware-cloud-on-aws-hybrid-network-design-patterns/)
- **[Backup]** Protect SDDC virtual machines with AWS Backup's VMware support, using a centrally managed policy and cross-Region copy of recovery points. — Provides application-consistent backups and a single policy-driven recovery strategy across on-premises, VMware Cloud on AWS, and native AWS resources, with resilience against ransomware and Regional failure. [doc](https://aws.amazon.com/backup/features/vmware/)
- **[Disaster recovery]** Design DR using a minimally sized or on-demand VMware Cloud on AWS environment as the recovery site, sized to meet RTO/RPO rather than mirroring full production capacity at all times. — Keeps a warm or cold standby site ready for failover without paying to run full-scale duplicate infrastructure continuously. [doc](https://aws.amazon.com/blogs/apn/vmware-cloud-on-aws-disaster-recovery-with-amazon-fsx-for-netapp-ontap-and-snapmirror/)
- **[Region/AZ planning]** Place the SDDC in the same Region and Availability Zone as related native AWS workloads whenever possible. — Avoids cross-AZ data transfer charges and unnecessary latency between the SDDC and dependent AWS services. [doc](https://docs.aws.amazon.com/whitepapers/latest/sddc-deployment-and-best-practices/account-requirements.html)

## ⚡ Performance Efficiency
- **[Storage placement]** Use vSAN Storage DRS-style balancing and configure disk striping, cache policies, and per-VM IOPS limits to match workload I/O profiles. — Prevents individual VMs or datastores from becoming performance bottlenecks as workloads scale. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/aws-storage-for-vmware-professionals/optimizing-cost-and-performance.html)
- **[Sizing]** Choose Large-sized NSX Edge and vCenter Server appliances at deployment time for SDDCs expected to exceed roughly 30 hosts or 3,000 VMs, since this cannot be changed after deployment. — Undersized management appliances can become oversubscribed and degrade performance as the SDDC scales, and resizing later isn't supported. [doc](https://docs.aws.amazon.com/whitepapers/latest/sddc-deployment-and-best-practices/deploying-vmware-cloud-on-aws-sddc.html)
- **[Host placement]** Rely on the automatic AWS Partition Placement Groups used by VMware Cloud on AWS for host placement, and pair this with an appropriate FTT setting. — Ensures a single hardware or rack-level failure impacts only one host rather than cascading across the cluster. [doc](https://aws.amazon.com/blogs/apn/resiliency-design-considerations-and-best-practices-for-vmware-cloud-on-aws/)
- **[Network scaling]** Use Transit VIF with a Direct Connect Gateway rather than multiple private VIFs when connecting several SDDCs or VPCs at scale. — A single transit VIF and BGP session provides more scalable, transitive routing than provisioning a private VIF per connection. [doc](https://aws.amazon.com/blogs/apn/vmware-cloud-on-aws-hybrid-network-design-patterns/)

## 💰 Cost Optimization
- **[Capacity planning]** Right-size SDDC host count and cluster resource pools to actual sustained utilization instead of provisioning for peak or projected capacity upfront. — VMware Cloud on AWS still requires capacity planning discipline since scaling is not as automatically elastic as native AWS services. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/aws-storage-for-vmware-professionals/optimizing-cost-and-performance.html)
- **[Multi-tenant environments]** For service-provider or multi-tenant use cases, use VMware Cloud Director service to carve SDDC capacity into smaller resource pools and expand to dedicated pools only as tenants grow. — Reduces the initial infrastructure investment and avoids over-provisioning capacity for tenants that haven't yet grown into it. [doc](https://aws.amazon.com/blogs/apn/managing-msp-costs-with-vmware-cloud-director-service-multi-tenancy/)
- **[Disaster recovery]** Use an on-demand or minimally-running VMware Cloud on AWS environment as the DR target, scaling up only during a failover event or test. — Avoids paying to run a full duplicate production-sized environment continuously when it's only needed during a DR event. [doc](https://aws.amazon.com/blogs/apn/vmware-cloud-on-aws-disaster-recovery-with-amazon-fsx-for-netapp-ontap-and-snapmirror/)
- **[Connectivity costs]** Account for AWS Transit Gateway and VMware Transit Connect data transfer charges when designing multi-SDDC or multi-VPC network topologies. — These charges apply per the standard Transit Gateway price list and can be significant at scale if not factored into the network design. [doc](https://aws.amazon.com/blogs/apn/vmware-cloud-on-aws-disaster-recovery-with-amazon-fsx-for-netapp-ontap-and-snapmirror/)

## ⚙️ Operational Excellence
- **[Monitoring]** Integrate Amazon CloudWatch with VMware Cloud on AWS VM-level metrics to get a unified, centralized monitoring view alongside native AWS services, rather than relying only on vCenter/vSphere tools in isolation. — Native VMware tools alone don't provide the centralized, cross-environment visibility needed to correlate SDDC and AWS-side issues. [doc](https://aws.amazon.com/blogs/apn/amazon-cloudwatch-monitoring-for-workloads-hosted-on-vmware-cloud-on-aws/)
- **[Account management]** Use AWS Organizations (optionally with AWS Control Tower) to govern the AWS account linked to the SDDC, placing it in a dedicated organizational unit with policies that permit the required CloudFormation-based account linking. — Centralizes governance across accounts while ensuring guardrails don't inadvertently break the linking process. [doc](https://docs.aws.amazon.com/whitepapers/latest/sddc-deployment-and-best-practices/account-requirements.html)
- **[Access administration]** Assign the Organization Owner role in the VMware Cloud Services Portal only to a small set of trusted administrators, since only that role can invite or manage other users. — Limits who can grant access, manage billing, or file support requests for the SDDC organization. [doc](https://docs.aws.amazon.com/whitepapers/latest/sddc-deployment-and-best-practices/account-requirements.html)
- **[Backup automation]** Automate VM backup scheduling and lifecycle (warm-to-cold tiering) through AWS Backup's policy-driven approach instead of managing ad hoc snapshot schedules per VM. — Reduces manual operational overhead while still producing auditor-ready reports for backup, copy, and restore operations. [doc](https://aws.amazon.com/blogs/apn/getting-started-with-aws-backups-for-vmware-cloud-on-aws/)
