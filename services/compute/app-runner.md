# AWS App Runner — Best Practices

## Common scenarios
- Deploying a containerized web app or API without managing servers      → Operational Excellence, Cost Optimization
- Connecting a service to a private database or backend in a VPC        → Security, Reliability
- Restricting an internal API so it's reachable only from inside a VPC  → Security
- Running a production service that must survive a regional disruption → Reliability

## 🔒 Security
- **[IAM]** Scope both the access role and the instance role to least privilege, customizing the App Runner managed policies rather than granting broad default permissions. — Reduces blast radius if application code or build credentials are compromised. [doc](https://docs.aws.amazon.com/apprunner/latest/dg/security-best-practices.html)
- **[IAM]** Create a dedicated instance role with a trust policy scoped to `tasks.apprunner.amazonaws.com`, and only attach the specific AWS API permissions your application code actually calls. — App Runner has no way to infer which APIs your code needs, so an overly broad instance role is an unnecessary standing privilege. [doc](https://docs.aws.amazon.com/apprunner/latest/dg/security_iam_service-with-iam.html)
- **[Container images]** Scan container images for vulnerabilities using Amazon ECR image scanning before deploying them to App Runner. — Catches known CVEs in base images and dependencies before they reach a running service. [doc](https://docs.aws.amazon.com/apprunner/latest/dg/security-best-practices.html)
- **[Networking]** Use a VPC connector with outbound-only security group rules when a service needs to reach resources like RDS, ElastiCache, or internal services in a VPC, and select private subnets (never public subnets) for the connector. — Keeps outbound application traffic under your VPC's routing and security-group controls instead of the public internet. [doc](https://docs.aws.amazon.com/apprunner/latest/dg/network-vpc.html)
- **[Networking]** Ensure the destination resource's security group explicitly allows inbound traffic from the VPC connector's security group. — The VPC connector only governs outbound traffic; without a matching inbound rule on the destination, connectivity fails or is unintentionally open. [doc](https://aws.amazon.com/blogs/containers/deep-dive-on-aws-app-runner-vpc-networking/)
- **[Private ingress]** For internal-only services, enable a Private endpoint (VPC Ingress Connection backed by AWS PrivateLink) instead of exposing the service on the public internet. — Restricts inbound access to traffic originating from within a specified VPC. [doc](https://docs.aws.amazon.com/apprunner/latest/dg/network-pl.html)
- **[Private ingress]** If source-IP/CIDR based access control is required for a private service, enforce it with VPC interface endpoint security groups rather than AWS WAF. — App Runner private services don't forward client source IP to WAF, so IP-based WAF rules silently fail to apply. [doc](https://docs.aws.amazon.com/apprunner/latest/dg/network-pl-manage.html)
- **[Data at rest]** Provide your own AWS KMS customer-managed key in the encryption configuration when creating a service, instead of relying solely on the AWS-managed key. — Gives you direct control over key rotation and access policy for encrypted source images/bundles and logs. [doc](https://docs.aws.amazon.com/apprunner/latest/dg/security-data-protection-encryption.html)
- **[Data in transit]** Rely on App Runner's managed HTTPS endpoint and, for custom domains, its automatically managed TLS certificate rather than terminating TLS elsewhere. — Ensures client traffic to the service is encrypted without operating your own certificate lifecycle. [doc](https://aws.amazon.com/blogs/containers/architecting-for-resiliency-on-aws-app-runner/)
- **[Detective controls]** Track App Runner control-plane actions such as `PauseService` and `DeleteConnection` with AWS CloudTrail, and alarm on key CloudWatch metrics. — Surfaces potentially disruptive or unauthorized changes after they occur. [doc](https://docs.aws.amazon.com/apprunner/latest/dg/security-best-practices.html)

## 🛡️ Reliability
- **[Availability Zones]** Let App Runner's automatic multi-AZ placement handle single-region resiliency, and when using a VPC connector, select subnets spread across at least three Availability Zones (or all available AZs if fewer exist). — Ensures Fargate tasks behind the VPC connector aren't concentrated in a single AZ. [doc](https://aws.amazon.com/blogs/containers/deep-dive-on-aws-app-runner-vpc-networking/)
- **[Multi-region]** For workloads with strict resiliency requirements, deploy App Runner services in two or more regions behind Route 53 with health checks and a weighted, latency, or geolocation routing policy. — Lets traffic automatically shift away from a region experiencing issues. [doc](https://aws.amazon.com/blogs/containers/architecting-for-resiliency-on-aws-app-runner/)
- **[Custom domains]** Associate the same custom domain across regional App Runner services so all endpoints share one SAN certificate managed by App Runner. — Enables consistent, trusted TLS regardless of which regional endpoint serves a given request. [doc](https://aws.amazon.com/blogs/containers/architecting-for-resiliency-on-aws-app-runner/)
- **[Health checks]** Configure a `HealthCheckConfiguration` (protocol, path, thresholds) that reflects genuine application health rather than a static 200 response. — Ensures App Runner and any front-door health checks (e.g., Route 53) detect real degradation and route around or replace unhealthy instances. [doc](https://docs.aws.amazon.com/apprunner/latest/api/API_UpdateService.html)
- **[Data tier]** Choose a data store that itself supports multi-region replication (such as DynamoDB global tables) when running App Runner in an active-active multi-region configuration. — The database tier is typically the hardest component to make resilient across regions, and using a multi-active managed store avoids building custom replication. [doc](https://aws.amazon.com/blogs/containers/architecting-for-resiliency-on-aws-app-runner/)
- **[Auto scaling]** Set a `MinSize` greater than the default in the auto scaling configuration for production services. — A higher minimum spreads instances across more Availability Zones, improving resilience at a modest cost tradeoff. [doc](https://docs.aws.amazon.com/apprunner/latest/api/API_AutoScalingConfiguration.html)

## ⚡ Performance Efficiency
- **[Concurrency]** Tune the max concurrency setting in the auto scaling configuration to match how many simultaneous requests a single instance can actually handle well. — Concurrency set too high causes request queuing/latency on overloaded instances; set too low triggers unnecessary scale-out. [doc](https://aws.amazon.com/apprunner/faqs/)
- **[Auto scaling]** Set `MaxSize` based on realistic peak demand rather than leaving it unconstrained. — A lower `MaxSize` controls cost but trades off some responsiveness during unexpected demand spikes, so size it to your actual traffic profile. [doc](https://docs.aws.amazon.com/apprunner/latest/api/API_AutoScalingConfiguration.html)
- **[Deployments]** Prefer container image deployments and keep images reasonably small. — App Runner's image-pull path benefits from smaller images, and image-based deployments avoid the build-time overhead of source-based deployments. [doc](https://aws.amazon.com/blogs/containers/aws-app-runner-improves-performance-for-image-based-deployments/)
- **[Egress path]** When a service must call other AWS APIs (S3, DynamoDB, etc.) over a VPC connector, use VPC endpoints (AWS PrivateLink) for those services instead of routing through a NAT gateway. — Keeps traffic on the AWS network backbone and avoids unnecessary NAT hops that add latency and cost. [doc](https://docs.aws.amazon.com/apprunner/latest/dg/network-vpc.html)

## 💰 Cost Optimization
- **[Idle cost]** Pause services that aren't actively needed (e.g., dev/test environments) via the console, CLI, or API instead of leaving them running continuously. — You're billed only while a service is running, so pausing idle services eliminates unnecessary provisioned-instance charges. [doc](https://aws.amazon.com/apprunner/pricing/)
- **[Right-sizing]** Match provisioned vCPU and memory to the application's actual resource profile rather than over-provisioning by default. — App Runner bills for the vCPU and memory configured for active instances, so oversized instances directly inflate cost. [doc](https://aws.amazon.com/apprunner/pricing/)
- **[Auto scaling]** Cap `MaxSize` in the auto scaling configuration to a level consistent with budget constraints. — A lower `MaxSize` directly bounds the maximum concurrent compute spend during traffic spikes. [doc](https://docs.aws.amazon.com/apprunner/latest/api/API_AutoScalingConfiguration.html)
- **[Concurrency tuning]** Increase the per-instance concurrency setting where application latency budgets allow, so fewer instances are needed to serve the same request volume. — Higher concurrency per instance reduces the number of billed active instances for a given load. [doc](https://aws.amazon.com/apprunner/faqs/)

## ⚙️ Operational Excellence
- **[Monitoring]** Set CloudWatch alarms on key App Runner service and instance metrics, and stream application logs to CloudWatch Logs for centralized troubleshooting. — Provides the visibility needed to detect and diagnose issues quickly instead of relying on customer reports. [doc](https://docs.aws.amazon.com/apprunner/latest/dg/security-monitoring.html)
- **[Tracing]** Enable X-Ray tracing through App Runner's observability configuration, using AWS Distro for OpenTelemetry (ADOT) if the application isn't already instrumented. — Gives an end-to-end view of request flow across the application and downstream AWS resources without managing sidecars yourself. [doc](https://aws.amazon.com/blogs/containers/tracing-an-aws-app-runner-service-using-aws-x-ray-with-opentelemetry/)
- **[Auditing]** Enable AWS CloudTrail for App Runner API activity and review it alongside CloudWatch metrics and logs as part of routine operations. — Creates an audit trail of control-plane changes (deployments, pauses, deletions) to speed up incident investigation. [doc](https://docs.aws.amazon.com/apprunner/latest/dg/security-monitoring.html)
- **[Infrastructure as code]** Define App Runner services, VPC connectors, and auto scaling configurations with an IaC tool (CloudFormation, CDK, or Terraform) instead of manual console changes. — Makes service configuration reviewable, repeatable, and consistent across environments and regions. [doc](https://aws.amazon.com/blogs/containers/architecting-for-resiliency-on-aws-app-runner/)
- **[Auto scaling changes]** Use a named, versioned `AutoScalingConfiguration` resource (updating or creating new revisions as needed) rather than embedding one-off scaling values per service. — Revisioned, shareable scaling configurations make it easier to see and manage scaling behavior across multiple services. [doc](https://aws.amazon.com/about-aws/whats-new/2023/09/aws-app-runner-auto-scaling-management/)
