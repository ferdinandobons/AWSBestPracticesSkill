# Red Hat OpenShift Service on AWS (ROSA) — Best Practices

## Common scenarios
- Migrating on-premises Red Hat OpenShift workloads to AWS        → Reliability, Operational Excellence
- Running enterprise Kubernetes platforms with joint Red Hat/AWS support        → Operational Excellence, Reliability
- Building secure, scalable containerized applications integrated with native AWS services        → Security, Performance Efficiency
- Running regulated or compliance-sensitive workloads (FedRAMP, HIPAA) on managed OpenShift        → Security, Cost Optimization

## 🔒 Security
- **[Cluster access]** Deploy private clusters using AWS PrivateLink so API server and application route traffic stays within your VPC. — Keeps management-plane traffic off the public internet and reduces exposure. [doc](https://docs.aws.amazon.com/rosa/latest/userguide/infrastructure-security.html)
- **[Credentials]** Create ROSA clusters with AWS STS (`--sts`) and use IAM roles for service accounts (IRSA) via the cluster's OIDC provider instead of long-lived credentials. — Grants pods only the temporary, least-privileged credentials they need and improves auditability via CloudTrail. [doc](https://aws.amazon.com/blogs/containers/fine-grained-iam-roles-for-red-hat-openshift-service-on-aws-rosa-workloads-with-sts/)
- **[Identity]** Configure a supported external identity provider (GitHub, LDAP, Google, OpenID Connect, etc.) for cluster OAuth instead of relying on HTPasswd for general use. — HTPasswd is intended only for a single static admin account, not general-purpose authentication. [doc](https://docs.aws.amazon.com/rosa/latest/userguide/getting-started-classic-cli.html)
- **[IAM]** Scope ROSA operator and account roles to least privilege, and prefer ROSA with HCP's AWS-managed, narrowly-scoped policies where possible. — Reduces blast radius from compromised credentials or misconfigured roles. [doc](https://aws.amazon.com/blogs/containers/diving-into-red-hat-openshift-service-on-aws-rosa-with-hosted-control-planes-hcp/)
- **[Data at rest]** Use AWS KMS customer-managed keys to encrypt persistent volumes and node storage when default AWS-managed encryption doesn't meet compliance requirements. — Gives you control over key rotation and access policies for sensitive workload data. [doc](https://docs.aws.amazon.com/rosa/latest/userguide/data-protection-key-management.html)
- **[Data in transit]** Enforce TLS 1.2 (prefer TLS 1.3) for all communication with AWS resources and use secured OpenShift routes with appropriate TLS termination. — Protects data moving between clients, the cluster, and AWS services. [doc](https://docs.aws.amazon.com/rosa/latest/userguide/data-protection.html)
- **[Account hygiene]** Enable multi-factor authentication (MFA) on AWS accounts and set up individual IAM users/roles rather than shared credentials. — Limits impact of credential compromise and supports least-privilege access. [doc](https://docs.aws.amazon.com/rosa/latest/userguide/data-protection.html)
- **[Auditing]** Set up AWS CloudTrail for API and user activity logging across the account hosting the ROSA cluster. — Provides an audit trail for security investigations and compliance reporting. [doc](https://docs.aws.amazon.com/rosa/latest/userguide/data-protection.html)
- **[Egress control]** Route cluster-wide traffic through an HTTP/HTTPS proxy when using an existing VPC without direct internet access for core components. — Adds a controlled layer for inspecting and securing outbound cluster traffic. [doc](https://docs.aws.amazon.com/rosa/latest/userguide/infrastructure-security.html)
- **[Shared responsibility]** Understand and plan around the ROSA shared responsibility model: Red Hat manages cluster infrastructure and platform security, while you remain responsible for securing applications, workloads, and data deployed to the cluster. — Prevents gaps where neither party is enforcing controls you assumed were covered. [doc](https://docs.aws.amazon.com/rosa/latest/userguide/rosa-responsibilities.html)

## 🛡️ Reliability
- **[Topology]** Deploy Multi-AZ clusters spreading control plane, infrastructure, and worker nodes across at least three Availability Zones for production workloads. — Ensures API requests reroute automatically if a control plane node or AZ fails. [doc](https://docs.aws.amazon.com/rosa/latest/userguide/disaster-recovery-resiliency.html)
- **[Worker recovery]** Enable cluster autoscaling (min/max replicas) so failed or scaled-down worker nodes are automatically replaced. — Reduces manual intervention and restores capacity for rescheduled pods after node failure. [doc](https://docs.aws.amazon.com/rosa/latest/userguide/disaster-recovery-resiliency.html)
- **[Managed infrastructure]** Never manually stop, terminate, or modify the EC2 instances backing ROSA control plane, infrastructure, or worker nodes from the console or CLI. — Manual intervention on Red Hat SRE-managed infrastructure can cause customer data loss. [doc](https://docs.aws.amazon.com/rosa/latest/userguide/disaster-recovery-resiliency.html)
- **[Architecture choice]** Consider ROSA with Hosted Control Planes (HCP) for improved resiliency, since the control plane is managed in a Red Hat-owned account and decoupled from customer-account failures. — Isolates control plane availability from actions or issues in your own AWS account. [doc](https://aws.amazon.com/blogs/containers/diving-into-red-hat-openshift-service-on-aws-rosa-with-hosted-control-planes-hcp/)
- **[Workload placement]** Distribute application replicas across all Availability Zones used by the cluster rather than concentrating them in one AZ. — Matches application-level resilience to the multi-AZ infrastructure design. [doc](https://aws.amazon.com/blogs/containers/red-hat-openshift-service-on-aws-architecture-and-networking/)

## ⚡ Performance Efficiency
- **[Compute provisioning]** Use Karpenter for dynamic, workload-aware compute provisioning alongside or instead of the cluster autoscaler, using pod placement (taints/tolerations, nodeSelectors) to control the migration path. — Matches node shape to actual pending-pod requirements faster than static machine pools. [doc](https://aws.amazon.com/blogs/ibm-redhat/cut-costs-scale-smarter-with-rosa-karpenter-automates-compute-provisioning/)
- **[Instance selection]** Right-size EC2 instance types for machine pools based on actual workload CPU/memory requirements, using tools like the EC2 Instance Type Explorer or Instance Type Finder. — Avoids both throttling from underprovisioned nodes and wasted capacity from oversized ones. [doc](https://aws.amazon.com/blogs/ibm-redhat/maximizing-the-value-of-red-hat-openshift-on-aws/)
- **[Dedicated pools]** Create separate machine pools for specialized workloads (e.g., GPU-backed instance types) rather than mixing them into general-purpose pools. — Ensures workloads land on nodes with the hardware capabilities they actually need. [doc](https://aws.amazon.com/blogs/containers/build-rosa-clusters-with-terraform/)

## 💰 Cost Optimization
- **[Architecture choice]** Evaluate ROSA with HCP for new clusters, since it reduces the minimum EC2 footprint (as few as two instances vs. seven-plus for classic) and lowers associated infrastructure fees. — Fewer required control-plane/infrastructure EC2 instances directly lowers baseline run cost. [doc](https://docs.aws.amazon.com/rosa/latest/userguide/rosa-architecture-models.html)
- **[Right-sizing]** Configure precise CPU/memory requests and limits per container to minimize "slack cost" between requested and actually utilized resources. — Over-provisioned requests silently inflate compute spend across a cluster. [doc](https://aws.amazon.com/blogs/ibm-redhat/maximizing-the-value-of-red-hat-openshift-on-aws/)
- **[Compute pricing]** Use Karpenter to mix Spot and On-Demand capacity, prioritizing Spot for fault-tolerant and interruption-tolerant workloads, with a diverse set of instance types to reduce interruption risk. — Spot pricing can substantially cut compute costs for workloads that tolerate node interruption. [doc](https://aws.amazon.com/blogs/ibm-redhat/cut-costs-scale-smarter-with-rosa-karpenter-automates-compute-provisioning/)
- **[Processor choice]** Evaluate AWS Graviton (ARM-based) instance types for supported workloads to improve price-performance. — Graviton instances can offer meaningfully better cost-per-performance versus comparable x86 instances. [doc](https://aws.amazon.com/blogs/ibm-redhat/maximizing-the-value-of-red-hat-openshift-on-aws/)
- **[Elastic capacity]** Enable autoscaling on machine pools so compute capacity tracks real demand instead of running fixed, statically-sized pools continuously. — Avoids paying for idle capacity during low-demand periods. [doc](https://aws.amazon.com/solutions/guidance/computer-vision-for-quality-insights-with-ibm-maximo-visual-inspection-on-aws/)

## ⚙️ Operational Excellence
- **[Responsibility model]** Map out which operational tasks are owned by Red Hat SRE, AWS, and your own team before go-live, using the published shared responsibility guidance. — Clarifies who acts on upgrades, node replacement, logging, and application-layer incidents. [doc](https://docs.aws.amazon.com/rosa/latest/userguide/rosa-responsibilities.html)
- **[Observability]** Deploy the Cluster Logging Operator and centralize ROSA cluster logs and metrics into Amazon CloudWatch alongside other AWS service telemetry. — Enables unified monitoring, longer retention for compliance, and correlated incident investigation across ROSA and AWS services. [doc](https://aws.amazon.com/blogs/ibm-redhat/security-auditing-in-rosa-using-aws-observability-services/)
- **[Troubleshooting]** Use `rosa describe cluster --debug` and review ROSA CLI debug logs when diagnosing cluster creation or configuration failures. — Surfaces the underlying error messages needed to root-cause failed operations quickly. [doc](https://docs.aws.amazon.com/rosa/latest/userguide/troubleshooting-rosa.html)
- **[Upgrades]** With ROSA HCP, take advantage of independent upgrade scheduling for the control plane and each machine pool rather than requiring a single all-at-once cluster upgrade. — Reduces upgrade blast radius and allows staged rollout across environments. [doc](https://docs.aws.amazon.com/rosa/latest/userguide/rosa-architecture-models.html)
- **[Infrastructure as code]** Provision ROSA clusters, VPCs, and machine pools using Terraform rather than ad hoc console steps. — Makes cluster and network configuration repeatable, versioned, and auditable. [doc](https://aws.amazon.com/blogs/containers/build-rosa-clusters-with-terraform/)
