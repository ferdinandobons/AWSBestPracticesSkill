# Amazon EKS — Best Practices

## Common scenarios
- Running production-grade, highly available Kubernetes clusters        → Reliability, Security
- Migrating self-managed Kubernetes workloads to a managed control plane        → Operational Excellence, Reliability
- Scaling containerized microservices with dynamic compute demand        → Performance Efficiency, Cost Optimization
- Running regulated or multi-tenant workloads on Kubernetes        → Security, Cost Optimization

## 🔒 Security
- **[Cluster endpoint]** Configure the EKS cluster endpoint for private (or public-with-restricted-CIDR) access instead of leaving it open to the entire internet. — Keeps node-to-control-plane and administrative traffic off the public internet while still requiring IAM/RBAC authentication and authorization. [doc](https://docs.aws.amazon.com/eks/latest/best-practices/identity-and-access-management.html)
- **[Endpoint mode]** Prefer public-and-private cluster endpoint mode so node-to-control-plane traffic stays inside the VPC while limiting the public endpoint to known CIDR blocks. — Reduces exposure while still allowing controlled external `kubectl`/CI access. [doc](https://docs.aws.amazon.com/eks/latest/best-practices/subnets.html)
- **[Identity]** Use IAM roles for service accounts (IRSA) or EKS Pod Identity to grant AWS permissions to specific Pods instead of sharing the node's IAM role. — Scopes credentials to least privilege per workload and prevents Pods from accessing other Pods' AWS permissions via the node role. [doc](https://docs.aws.amazon.com/eks/latest/userguide/iam-roles-for-service-accounts.html)
- **[Access management]** After cluster creation, stop relying on the original cluster-creator principal's implicit `system:masters` access; manage ongoing user and service access centrally through Kubernetes RBAC (or EKS access entries) tied to an identity provider. — Keeps access auditable and revocable instead of depending on a single privileged bootstrap identity. [doc](https://docs.aws.amazon.com/eks/latest/best-practices/identity-and-access-management.html)
- **[Least privilege]** Apply role-based access control (RBAC) to scope user and service permissions to specific namespaces and resources rather than cluster-wide access. — Limits blast radius from a compromised identity or misconfigured application. [doc](https://docs.aws.amazon.com/eks/latest/best-practices/security.html)
- **[Network isolation]** Use Kubernetes Network Policies to restrict Pod-to-Pod traffic, isolating system namespaces (like `kube-system`) and security tooling from application workloads. — Reduces the risk of lateral movement if a workload is compromised. [doc](https://docs.aws.amazon.com/eks/latest/best-practices/network-security.html)
- **[Secrets]** Integrate AWS Secrets Manager (or Parameter Store) with the Secrets Store CSI Driver instead of storing sensitive values as native Kubernetes Secrets or environment variables. — Keeps credentials and API keys out of plaintext manifests and centralizes rotation and access control. [doc](https://docs.aws.amazon.com/eks/latest/best-practices/data-encryption-and-secrets-management.html)
- **[Threat detection]** Enable Amazon GuardDuty EKS Protection to continuously monitor cluster and workload activity for malicious or anomalous behavior. — Provides real-time detection of suspicious API calls and runtime activity you would otherwise have to build yourself. [doc](https://docs.aws.amazon.com/eks/latest/best-practices/runtime-security.html)
- **[Image security]** Scan container images for vulnerabilities before deployment and use trusted, minimal base images from a secured registry such as Amazon ECR. — Prevents known-vulnerable or tampered images from reaching production clusters. [doc](https://docs.aws.amazon.com/eks/latest/best-practices/image-security.html)
- **[Shared responsibility]** Understand the EKS shared responsibility model: AWS secures the managed control plane (API servers, etcd), while you are responsible for data-plane configuration, node OS patching, container configuration, and network controls. — Prevents gaps where neither party assumes ownership of a control. [doc](https://docs.aws.amazon.com/eks/latest/userguide/security.html)

## 🛡️ Reliability
- **[Control plane]** Rely on EKS's built-in multi-AZ control plane (at least two API server instances and three etcd instances across three Availability Zones) rather than building custom HA tooling for the control plane itself. — AWS automatically scales, detects, and replaces unhealthy control plane instances, backed by an API server availability SLA. [doc](https://docs.aws.amazon.com/eks/latest/userguide/disaster-recovery-resiliency.html)
- **[Data plane spread]** Spread managed node groups, self-managed nodes, or Karpenter-provisioned nodes across multiple Availability Zones rather than a single AZ. — Ensures workloads can reschedule and continue running if an entire AZ becomes unavailable. [doc](https://docs.aws.amazon.com/eks/latest/best-practices/reliability.html)
- **[Managed data plane]** Prefer EKS Auto Mode, Fargate, or managed node groups over self-managed nodes where possible, since AWS assumes more of the provisioning, patching, and lifecycle responsibility as you move up that stack. — Reduces the operational burden and risk of stale, unpatched worker nodes. [doc](https://docs.aws.amazon.com/eks/latest/best-practices/reliability.html)
- **[Application resilience]** Use Pod Disruption Budgets, readiness/liveness probes, and pod topology spread constraints to keep critical workloads available during node updates, scaling events, or zonal disruptions. — Prevents voluntary disruptions (like node drains) from taking down all replicas of a service simultaneously. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/ha-resiliency-amazon-eks-apps/ha-resilience-design.html)
- **[AMI/patch currency]** Establish a regular process (or use EKS Auto Mode's automatic AMI updates with disruption controls) to keep node AMIs current with CVE fixes and security patches. — Stale AMIs on managed or self-managed nodes accumulate unpatched vulnerabilities over time. [doc](https://docs.aws.amazon.com/eks/latest/best-practices/reliability.html)
- **[Version upgrades]** Use EKS upgrade insights to check cluster and workload compatibility before every Kubernetes minor version upgrade, and upgrade control plane, nodes, add-ons, and clients in sequence. — Surfaces deprecated APIs and known upgrade-impacting issues before they cause outages. [doc](https://docs.aws.amazon.com/eks/latest/best-practices/cluster-upgrades.html)
- **[Autoscaling]** Deploy Karpenter or Cluster Autoscaler so unschedulable Pods automatically trigger new node capacity, and configure autoscaler min/max bounds deliberately. — Ensures capacity keeps pace with demand and unhealthy or scaled-in nodes are replaced without manual intervention. [doc](https://docs.aws.amazon.com/eks/latest/best-practices/cost-opt-compute.html)

## ⚡ Performance Efficiency
- **[Autoscaling engine]** Prefer Karpenter over the traditional Cluster Autoscaler for new deployments, since it provisions right-sized nodes directly per pending-Pod requirements without pre-defined node groups. — Improves bin-packing and reduces the operational overhead of managing many fixed-shape node groups. [doc](https://docs.aws.amazon.com/eks/latest/best-practices/cost-opt-compute.html)
- **[Right-sizing]** Profile production and pre-production workloads with Amazon CloudWatch Container Insights (or equivalent tooling) to set accurate CPU/memory requests rather than guessing. — Ensures Pods get the resources they need without over- or under-provisioning nodes. [doc](https://docs.aws.amazon.com/eks/latest/best-practices/cost-opt.html)
- **[Processor choice]** Evaluate AWS Graviton-based instance types for supported workloads running on EKS. — Graviton processors typically deliver better price-performance for compatible workloads. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/scaling-amazon-eks-infrastructure/cost-optimization.html)
- **[Specialized workloads]** Use dedicated node groups or NodePools for workloads with specific hardware needs (for example, GPU instance types) instead of mixing them into general-purpose pools. — Ensures scheduling places specialized workloads only on nodes with the required hardware, avoiding wasted capacity elsewhere. [doc](https://docs.aws.amazon.com/eks/latest/userguide/gpu-ami.html)

## 💰 Cost Optimization
- **[Infrastructure flexibility]** Design workloads to be independent of a specific EC2 instance type or family (except where hardware like GPUs is genuinely required) so they can run on the least expensive suitable infrastructure. — Widens the pool of eligible instance types and purchasing options, improving cost efficiency. [doc](https://docs.aws.amazon.com/eks/latest/best-practices/cost-opt.html)
- **[Purchasing options]** Combine On-Demand, Savings Plans, and Spot Instances based on workload criticality and interruption tolerance. — Matches purchasing commitment to workload predictability, lowering overall compute spend. [doc](https://docs.aws.amazon.com/eks/latest/best-practices/cost-opt.html)
- **[Node provisioning]** Use Karpenter to bin-pack Pods efficiently across diverse instance types and to scale Spot-backed capacity for fault-tolerant workloads. — Reduces idle/wasted capacity compared with statically sized node groups. [doc](https://docs.aws.amazon.com/eks/latest/best-practices/cost-opt-compute.html)
- **[Right-sizing]** Monitor CPU/memory utilization with CloudWatch Container Insights and adjust requests/limits and node group sizing over time rather than a one-time setup. — Prevents "slack cost" from persistently over-provisioned resources. [doc](https://docs.aws.amazon.com/eks/latest/best-practices/cost-opt.html)
- **[Expenditure visibility]** Use AWS Cost Explorer, the Billing and Cost Management console, and AWS Organizations for granular cost visibility across clusters and teams. — Enables cost allocation and identification of unexpected spend before it accumulates. [doc](https://docs.aws.amazon.com/eks/latest/best-practices/cost-opt.html)
- **[Networking cost]** Weigh the inter-AZ data transfer cost of spreading nodes across three Availability Zones against your high-availability requirements when designing cluster topology. — Multi-AZ designs improve reliability but can increase cross-AZ traffic costs if not architected carefully. [doc](https://docs.aws.amazon.com/eks/latest/best-practices/cost-opt-framework.html)

## ⚙️ Operational Excellence
- **[Upgrade planning]** Maintain a regular, planned cadence for Kubernetes minor version upgrades rather than falling multiple versions behind. — In-place upgrades proceed one minor version at a time, so multi-version gaps make upgrades more complex and risky. [doc](https://aws.amazon.com/blogs/containers/planning-kubernetes-upgrades-with-amazon-eks/)
- **[Upgrade readiness]** Review the EKS console's Upgrade Insights (or API) before every version upgrade to catch deprecated API usage and other known upgrade-impacting issues. — Reduces the risk of workloads breaking after a control plane upgrade. [doc](https://docs.aws.amazon.com/eks/latest/userguide/cluster-insights.html)
- **[Infrastructure as code]** Provision clusters, node groups, and networking with `eksctl`, CloudFormation, CDK, or Terraform instead of manual console steps. — Makes cluster configuration repeatable, versioned, and easier to reproduce across environments. [doc](https://docs.aws.amazon.com/eks/latest/userguide/update-workers.html)
- **[Centralized logging]** Aggregate control plane and application logs into a centralized system such as Amazon CloudWatch Logs, using structured (for example JSON) log formats and DaemonSet-based collectors like Fluent Bit. — Simplifies troubleshooting and gives a single point of access across all cluster components. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/amazon-eks-observability-best-practices/logging-best-practices.html)
- **[Monitoring]** Deploy comprehensive monitoring (for example Amazon Managed Service for Prometheus and Amazon Managed Grafana, or CloudWatch Container Insights) to track cluster and workload health rather than relying on ad hoc checks. — Enables proactive detection of anomalies and data-driven scaling and reliability decisions. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/amazon-eks-observability-best-practices/monitoring.html)
- **[Node lifecycle]** For self-managed nodes, use a migration (taint-and-drain) or in-place upgrade strategy driven by infrastructure-as-code rather than ad hoc manual node replacement. — Ensures workloads are gracefully rescheduled instead of being abruptly terminated during node updates. [doc](https://docs.aws.amazon.com/eks/latest/userguide/migrate-stack.html)
