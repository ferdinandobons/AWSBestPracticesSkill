# Amazon ECR Public — Best Practices

## Common scenarios
- Hosting open-source or vendor container images for global, anonymous distribution        → Security, Reliability
- Publishing official application images consumed by CI/CD pipelines and other AWS accounts        → Security, Operational Excellence
- Mirroring third-party public images into a private registry via pull-through cache        → Reliability, Cost Optimization
- Building a discoverable public gallery listing for open-source project images        → Operational Excellence

## 🔒 Security
- **[IAM permissions]** Grant only `ecr-public:GetAuthorizationToken` and `sts:GetServiceBearerToken` plus the minimal push/describe actions needed, rather than broad ECR Public access. — Amazon ECR Public requires these specific calls to authenticate, so scoping beyond them follows least privilege. [doc](https://docs.aws.amazon.com/AmazonECR/latest/public/public-images.html)
- **[IAM policies]** Start from AWS managed policies for common use cases, then author customer-managed policies scoped to least privilege for your actual repositories and actions. — Reduces the chance of over-broad grants that let principals create, modify, or delete public resources unnecessarily. [doc](https://docs.aws.amazon.com/AmazonECR/latest/public/security_iam_id-based-policy-examples.html)
- **[Conditions]** Add IAM policy conditions (for example requiring SSL, or restricting by source service) to further restrict when ECR Public actions are allowed. — Limits the circumstances under which sensitive actions like repository creation or tagging can succeed. [doc](https://docs.aws.amazon.com/AmazonECR/latest/public/security_iam_id-based-policy-examples.html)
- **[Policy validation]** Run new and existing IAM policies through IAM Access Analyzer before deploying them. — Catches overly permissive or non-functional policies before they're attached to users or roles. [doc](https://docs.aws.amazon.com/AmazonECR/latest/public/security_iam_id-based-policy-examples.html)
- **[MFA]** Require multi-factor authentication for IAM users or root-account access used to manage ECR Public resources. — Reduces the impact of credential compromise on a registry whose contents are world-readable. [doc](https://docs.aws.amazon.com/AmazonECR/latest/public/security_iam_id-based-policy-examples.html)
- **[Tag-based access]** Use resource tags with IAM condition keys (e.g. `aws:RequestTag/environment`) to control who can create or modify repositories in specific environments. — Lets you separate permissions for dev versus prod repositories without maintaining separate policies per repository. [doc](https://docs.aws.amazon.com/AmazonECR/latest/public/ecr-supported-iam-actions-tagging.html)
- **[Trusted sources]** Base container images on trusted, well-maintained sources (official images, verified publishers) rather than arbitrary public repositories. — Reduces the amount of unvetted code and the vulnerability surface you inherit into production images. [doc](https://aws.amazon.com/blogs/containers/building-better-container-images/)
- **[Image pinning]** Reference images by explicit version tags, or by SHA256 digest for production deployments, instead of mutable tags like `latest`. — Prevents silently deploying a different image than what was tested, since tags in a public registry can be updated by the publisher at any time. [doc](https://aws.amazon.com/blogs/containers/building-better-container-images/)
- **[Third-party dependency risk]** Mirror any production-critical public images into a registry you control instead of pointing workloads directly at a public registry you don't own. — Protects you from upstream image removal, unexpected changes, or availability issues outside your control. [doc](https://aws.amazon.com/blogs/containers/bitnami-image-removal-from-ecr-public/)
- **[Repository policy limits]** Understand that repository policies in Amazon ECR Public cannot be used to deny view or pull access, since all public repositories are visible on the ECR Public Gallery. — Avoid designing a security model that assumes pull-access restriction is possible on public repositories. [doc](https://docs.aws.amazon.com/AmazonECR/latest/public/public-repository-policy-examples.html)
- **[Auditing]** Enable AWS CloudTrail and review the event history for Amazon ECR Public API activity, including who authenticated, pushed, or modified repositories. — Provides the audit trail needed to investigate unexpected changes to publicly distributed images. [doc](https://docs.aws.amazon.com/AmazonECR/latest/public/logging-using-cloudtrail.html)

## 🛡️ Reliability
- **[Upstream dependency]** Mirror public images your production workloads depend on into a registry you control rather than pulling directly from ECR Public or other public registries at runtime. — Removes a hard runtime dependency on an external registry's availability and content stability. [doc](https://aws.amazon.com/blogs/containers/bitnami-image-removal-from-ecr-public/)
- **[Pull-through cache]** Configure an Amazon ECR pull-through cache rule for `ecr-public` as a long-term resiliency layer for any public images your workloads consume. — Automatically syncs and caches upstream images so pulls continue succeeding even if the upstream registry is degraded or images are removed. [doc](https://aws.amazon.com/blogs/containers/bitnami-image-removal-from-ecr-public/)
- **[Deployment pinning]** Deploy using image digests (SHA256) rather than tags. — Guarantees you always deploy the exact, previously-tested image content even if an upstream tag is retagged or removed. [doc](https://aws.amazon.com/blogs/containers/bitnami-image-removal-from-ecr-public/)
- **[Source auditing]** Periodically audit deployment configurations to identify dependencies on third-party public registries. — Surfaces hidden single points of failure before an upstream change or removal causes an outage. [doc](https://aws.amazon.com/blogs/containers/bitnami-image-removal-from-ecr-public/)

## ⚙️ Operational Excellence
- **[Catalog data]** Populate repository catalog data (description, architecture/OS compatibility tags, About and Usage sections) for public repositories. — Helps consumers of your images find, understand, and correctly use them from the ECR Public Gallery. [doc](https://docs.aws.amazon.com/AmazonECR/latest/public/public-repository-catalog-data.html)
- **[Resource organization]** Apply a consistent tagging scheme (owner, team, environment) to public repositories. — Makes it easier to track ownership and filter resources as the number of public repositories grows. [doc](https://docs.aws.amazon.com/AmazonECR/latest/public/ecr-public-using-tags.html)
- **[Namespacing]** Organize public repositories using namespaces that reflect team or project structure. — Keeps a large public registry navigable and aligns repository layout with existing team workflows. [doc](https://aws.amazon.com/ecr/features/)
- **[Credential automation]** Use the Amazon ECR credential helper to authenticate `docker push`/`docker pull` against ECR Public instead of manually managing tokens. — Removes manual `docker login` and token-refresh steps from build scripts and CI/CD pipelines. [doc](https://aws.amazon.com/blogs/containers/amazon-ecr-credential-helper-now-supports-amazon-ecr-public/)
- **[Auditing access]** As an administrator, use IAM Identity Center permission sets or explicit IAM policies to grant ECR Public access, rather than sharing credentials. — Keeps access provisioning traceable to individual users or applications. [doc](https://docs.aws.amazon.com/AmazonECR/latest/public/security_iam_troubleshoot.html)
- **[Image freshness]** Regularly update base and application images to their latest secure versions and tag them explicitly (e.g. `v1`, `v2`) rather than relying solely on `latest`. — Keeps published images current and avoids ambiguity about which version is actually being distributed. [doc](https://aws.amazon.com/blogs/containers/building-better-container-images/)

## 💰 Cost Optimization
- **[Pull-through cache]** Use Amazon ECR pull-through cache rules to sync frequently used public images into your private registry instead of repeatedly pulling from the public internet. — Reduces redundant external data transfer and keeps a local, cost-predictable copy of images you depend on. [doc](https://docs.aws.amazon.com/AmazonECR/latest/userguide/pull-through-cache-creating-rule.html)
- **[Authenticated pulls]** Authenticate to Amazon ECR Public with an AWS account before pulling images instead of pulling anonymously. — Authenticated pulls raise the free monthly data bandwidth allowance well above the anonymous tier. [doc](https://aws.amazon.com/blogs/containers/amazon-ecr-credential-helper-now-supports-amazon-ecr-public/)
