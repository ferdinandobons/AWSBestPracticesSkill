# Amazon EKS Anywhere — Best Practices

## Common scenarios
- Running Kubernetes consistently across on-premises data centers, vSphere, bare metal, Nutanix, or Snow        → Reliability, Operational Excellence
- Operating disconnected or air-gapped clusters for data residency or regulatory requirements        → Security, Reliability
- Modernizing on-premises applications without taking on full self-managed Kubernetes patching burden        → Operational Excellence, Cost Optimization
- Building hybrid or edge platforms that need AWS-consistent tooling and optional AWS service integration        → Reliability, Performance Efficiency

## 🔒 Security
- **[AWS access]** Use AWS IAM Roles Anywhere with a private certificate authority (e.g., AWS Private CA) to grant on-premises EKS Anywhere workloads temporary AWS credentials instead of long-lived access keys. — Avoids storing static AWS credentials on-premises and reduces exposure if a node or pod is compromised. [doc](https://aws.amazon.com/blogs/security/connect-your-on-premises-kubernetes-cluster-to-aws-apis-using-iam-roles-anywhere/)
- **[Credential lifecycle]** Automate X.509 certificate issuance and rotation for IAM Roles Anywhere with cert-manager and a short-lived certificate mode backed by AWS Private CA. — Keeps credential lifetimes aligned with dynamic, short-lived pod lifecycles instead of relying on long-lived static keys. [doc](https://aws.amazon.com/blogs/security/connect-your-on-premises-kubernetes-cluster-to-aws-apis-using-iam-roles-anywhere/)
- **[Networking]** Use Cilium as the cluster CNI with direct network interfaces where supported so pod-to-pod traffic avoids NAT. — Avoiding NAT improves both network performance and security posture for inter-pod communication. [doc](https://aws.amazon.com/blogs/apn/building-production-grade-kubernetes-clusters-with-amazon-eks-anywhere-on-nutanix/)
- **[Support boundary]** Track the Amazon EKS Anywhere version support policy and keep clusters on a supported Kubernetes version, since EKS Anywhere is customer-managed and you are responsible for patching and lifecycle operations. — Running unsupported versions leaves clusters without access to security patches and vendor support. [doc](https://aws.amazon.com/eks/eks-anywhere/faqs/)

## 🛡️ Reliability
- **[Control plane HA]** Use kube-vip (or an equivalent control-plane load balancer) running across multiple control plane nodes with leader election for the cluster endpoint virtual IP. — Provides automatic failover of the Kubernetes API endpoint if a control plane node goes down. [doc](https://aws.amazon.com/blogs/apn/building-production-grade-kubernetes-clusters-with-amazon-eks-anywhere-on-nutanix/)
- **[Upgrade strategy]** Perform in-place cluster upgrades one minor Kubernetes version at a time, upgrading control plane components before worker nodes, and run `eksctl anywhere upgrade plan cluster` to preview changes before upgrading. — Skipping versions or upgrading nodes out of order is unsupported and risks cluster instability. [doc](https://aws.amazon.com/blogs/containers/blue-green-kubernetes-upgrades-for-amazon-eks-anywhere-using-flux/)
- **[Upgrade risk reduction]** For production clusters, consider a blue/green upgrade pattern using Flux (bundled with EKS Anywhere) to stand up a new cluster on the target version and shift traffic once validated, instead of always upgrading in place. — Enables easy rollback by shifting traffic back and lets you validate a new Kubernetes version safely before committing production traffic. [doc](https://aws.amazon.com/blogs/containers/blue-green-kubernetes-upgrades-for-amazon-eks-anywhere-using-flux/)
- **[Backup]** Take regular backups of cluster state (etcd) and application data before upgrades, using a tool such as Velero, and validate restore procedures. — Kubernetes upgrades cannot be downgraded, so a tested backup/restore path is the primary recovery mechanism if an upgrade fails. [doc](https://docs.aws.amazon.com/eks/latest/best-practices/cluster-upgrades.html)
- **[GitOps]** Manage cluster and workload configuration declaratively in Git using the bundled Flux controller so changes are version-controlled and applied consistently. — Reduces accidental configuration drift and provides an audit trail and rollback path for cluster changes. [doc](https://aws.amazon.com/blogs/containers/blue-green-kubernetes-upgrades-for-amazon-eks-anywhere-using-flux/)
- **[Storage]** Use a supported CSI driver for your infrastructure provider (e.g., Nutanix CSI Volume Driver, vSphere CSI) with snapshot support for stateful workloads. — Ensures persistent volumes are properly provisioned and recoverable via snapshots on customer-managed infrastructure. [doc](https://aws.amazon.com/blogs/apn/building-production-grade-kubernetes-clusters-with-amazon-eks-anywhere-on-nutanix/)

## ⚙️ Operational Excellence
- **[Add-ons]** Use Amazon EKS Anywhere Curated Packages (with an active Enterprise Subscription) for load balancing, observability, and autoscaling add-ons instead of independently sourcing and supporting each component. — Curated Packages are pre-validated and AWS-supported, reducing the number of separate vendor support agreements you need. [doc](https://aws.amazon.com/eks/eks-anywhere/faqs/)
- **[Observability]** Deploy the AWS Distro for OpenTelemetry (ADOT) curated package to export on-premises cluster metrics and traces to Amazon Managed Service for Prometheus and other AWS observability services. — Gives a unified, centralized view of on-premises Kubernetes telemetry alongside AWS-hosted workloads. [doc](https://aws.amazon.com/blogs/mt/using-curated-packages-and-aws-managed-open-source-services-to-observe-your-on-premise-kubernetes-environment/)
- **[Governance at scale]** When rolling out EKS Anywhere across an organization, define a formal engagement model (roles, RACI matrix, communications plan) and a shared program tracker for issues, upgrades, and stability work before broad adoption. — Structured engagement and tracking materially reduce risk and rework when scaling from a pilot to organization-wide Kubernetes adoption. [doc](https://aws.amazon.com/blogs/containers/organizational-best-practices-for-implementing-eks-anywhere-at-scale/)
- **[Enablement]** Invest in dedicated technical education and skill-development programs for platform and application teams as part of the rollout plan, not just infrastructure setup. — On-premises Kubernetes operations place more operational burden on customer teams than a fully managed service, making team readiness a key success factor. [doc](https://aws.amazon.com/blogs/containers/organizational-best-practices-for-implementing-eks-anywhere-at-scale/)
- **[CI validation]** Validate Kubernetes manifest and cluster configuration changes in a CI pipeline before they are synced to the cluster via GitOps. — Minimizes accidental configuration changes reaching production clusters and impacting workload availability. [doc](https://aws.amazon.com/blogs/containers/blue-green-kubernetes-upgrades-for-amazon-eks-anywhere-using-flux/)

## 💰 Cost Optimization
- **[Support consolidation]** Consolidate support needs by adopting Amazon EKS Anywhere Curated Packages and Enterprise Subscriptions instead of purchasing separate support contracts for each third-party Kubernetes tool. — Reduces the total number of vendor relationships and support costs required to run production on-premises Kubernetes. [doc](https://aws.amazon.com/eks/eks-anywhere/faqs/)
- **[Subscription term]** Choose a 3-year Enterprise Subscription term over repeated 1-year terms when long-term on-premises Kubernetes usage is committed. — 3-year terms are offered at a discounted per-year rate compared to 1-year terms. [doc](https://docs.aws.amazon.com/help-panel/eks/latest/console/hp-anywhere-create.html)
- **[No lock-in]** Evaluate EKS Anywhere without upfront commitment, since the software itself has no license fee and Enterprise Subscriptions are optional and tied only to support and Curated Packages access. — Lets teams pilot on-premises Kubernetes before committing spend to paid support and add-ons. [doc](https://aws.amazon.com/blogs/apn/building-production-grade-kubernetes-clusters-with-amazon-eks-anywhere-on-nutanix/)
