# AWS Fargate — Best Practices

## Common scenarios
- Running containerized web APIs/microservices without managing servers        → Security, Reliability, Operational Excellence
- Batch and event-driven container jobs with variable load        → Cost Optimization, Performance Efficiency
- Multi-tenant or regulated workloads needing strong workload isolation        → Security, Reliability
- Bursty or unpredictable container workloads that must scale automatically        → Reliability, Cost Optimization

## 🔒 Security
- **[secrets]** Retrieve sensitive values (API keys, DB passwords) via Secrets Manager or SSM Parameter Store integration in the task definition instead of embedding them in images or environment variables — avoids leaking credentials in source control or config files. [doc](https://aws.amazon.com/blogs/compute/securing-credentials-using-aws-secrets-manager-with-aws-fargate/)
- **[storage]** Rely on Fargate's default AES-256 ephemeral storage encryption, and use platform version 1.4.0+ with AWS KMS customer managed keys when you need control over the encryption key — protects data written to task ephemeral storage. [doc](https://docs.aws.amazon.com/AmazonECS/latest/developerguide/security-fargate.html)
- **[runtime protection]** Enable Amazon GuardDuty Runtime Monitoring for Fargate tasks to continuously analyze on-host behavior (file access, process execution, network connections) — detects privilege escalation, exposed credentials, and malware. [doc](https://docs.aws.amazon.com/AmazonECS/latest/developerguide/security-fargate.html)
- **[capabilities]** Avoid architectures that assume privileged containers, CAP_SYS_ADMIN, or CAP_NET_ADMIN; only CAP_SYS_PTRACE can be added on Fargate — design workloads around this locked-down runtime rather than around EC2-only container features. [doc](https://docs.aws.amazon.com/AmazonECS/latest/developerguide/fargate-security-considerations.html)
- **[image supply chain]** Scan container images for vulnerabilities before deployment (e.g., with Amazon ECR image scanning) and block deployment of images with known critical CVEs or default root credentials — reduces risk of privilege-escalation and resource-exhaustion attacks at runtime. [doc](https://aws.amazon.com/blogs/apn/automating-aws-fargate-image-scanning-to-block-container-threats/)
- **[networking]** Place Fargate tasks in private subnets and use security groups and network ACLs to restrict inbound/outbound traffic at the elastic network interface level — each task gets its own ENI so controls apply per task. [doc](https://docs.aws.amazon.com/AmazonECS/latest/developerguide/fargate-security-considerations.html)
- **[private connectivity]** Use AWS PrivateLink / VPC endpoints for Amazon ECR and other AWS service calls from tasks in private subnets, and attach least-privilege endpoint policies — avoids routing image pulls and API calls over the public internet. [doc](https://aws.amazon.com/blogs/industries/fsi-service-spotlight-amazon-elastic-container-service-ecs-with-aws-fargate/)
- **[debugging access]** Use ECS Exec instead of building custom SSH or bastion access into task containers — provides an audited path to get a shell or run diagnostic commands without exposing the underlying host. [doc](https://docs.aws.amazon.com/AmazonECS/latest/developerguide/fargate-security-considerations.html)

## 🛡️ Reliability
- **[multi-AZ]** Distribute Fargate tasks/services across at least three Availability Zones for workloads that require high availability — reduces the capacity headroom needed to absorb an AZ failure compared to two-AZ deployments. [doc](https://aws.amazon.com/blogs/containers/amazon-ecs-availability-best-practices/)
- **[deployment safety]** Keep the service's minimumHealthyPercent at 100% (the default) for stateless service tasks so Amazon ECS starts a replacement task and confirms it is running before retiring an old one — minimizes disruption from routine Fargate task retirement and maintenance. [doc](https://docs.aws.amazon.com/AmazonECS/latest/developerguide/task-maintenance.html)
- **[task retirement]** Design services to run multiple task replicas rather than a single long-running task, since AWS periodically retires and replaces individual Fargate tasks for maintenance — a single-replica design creates unnecessary downtime windows. [doc](https://docs.aws.amazon.com/AmazonECS/latest/developerguide/task-maintenance.html)
- **[capacity headroom]** Over-provision capacity and target 60-80% steady-state utilization for autoscaled services — covers the latency inherent in CloudWatch metric aggregation and task provisioning, and leaves room to absorb an AZ failure. [doc](https://docs.aws.amazon.com/AmazonECS/latest/developerguide/capacity-availability-best-practice.html)
- **[resilience validation]** Use AWS Resilience Hub checks for Amazon ECS to verify tasks/services are actually running across multiple AZs (and in a secondary Region if Regional-disruption coverage is required) — catches configurations that look multi-AZ but aren't. [doc](https://docs.aws.amazon.com/resilience-hub/latest/userguide/resilience-checks.html)

## ⚡ Performance Efficiency
- **[task sizing]** Set task-level CPU and memory to the rounded-up sum of each container's declared resource reservations rather than guessing, and review actual application performance metrics rather than reusing EC2-instance sizing habits — prevents oversized tasks that waste provisioned capacity. [doc](https://docs.aws.amazon.com/AmazonECS/latest/developerguide/fargate-task-size-best-practice.html)
- **[processor choice]** Use AWS Graviton-powered Fargate where your images support arm64 — delivers better price-performance than comparable x86 Fargate for typical application server, web service, and media-processing workloads. [doc](https://aws.amazon.com/fargate/faqs/)
- **[image startup]** Optimize container image size and structure (and consider Seekable OCI indexes) using the task lifecycle timestamps published during the task's image-pull phase to identify slow-starting images — reduces task startup latency. [doc](https://docs.aws.amazon.com/AmazonECS/latest/developerguide/fargate-tasks-services.html)

## 💰 Cost Optimization
- **[right-sizing]** Use AWS Compute Optimizer to identify oversized Fargate tasks and right-size CPU/memory based on real utilization — right-sizing long-running tasks can cut costs 30-70% versus unreviewed defaults. [doc](https://docs.aws.amazon.com/prescriptive-guidance/latest/optimize-costs-microsoft-workloads/optimizer-ecs-fargate.html)
- **[pricing model]** Use Fargate Spot for interruption-tolerant tasks and Compute Savings Plans for steady-state baseline usage — Spot can save up to 70% versus On-Demand Fargate pricing for workloads that tolerate interruption. [doc](https://aws.amazon.com/fargate/faqs/)
- **[processor choice]** Run Fargate on AWS Graviton where compatible — up to 40% better price-performance at lower cost than comparable x86-based Fargate. [doc](https://aws.amazon.com/fargate/faqs/)
- **[visibility]** Tag ECS/Fargate services, task definitions, tasks, and clusters, then use Cost Explorer's Usage Type filters (e.g., Fargate-vCPU-Hours, Fargate-GB-Hours) to track spend per workload and spot untagged or idle resources — enables granular cost allocation and savings identification. [doc](https://aws.amazon.com/blogs/containers/cost-optimization-checklist-for-ecs-fargate/)

## ⚙️ Operational Excellence
- **[logging]** Configure the awslogs (or awsfirelens/splunk) log driver in the task definition to ship container logs to CloudWatch Logs or a partner destination — each container gets logs routed automatically without agents on a host you don't control. [doc](https://docs.aws.amazon.com/AmazonECS/latest/developerguide/fargate-tasks-services.html)
- **[observability]** Enable CloudWatch Container Insights to collect and aggregate cluster/task/service metrics and diagnostic data (including restart failures) for Fargate workloads — gives a single operational dashboard without deploying host-level agents. [doc](https://aws.amazon.com/fargate/faqs/)
- **[event-driven ops]** Consume Amazon ECS events via EventBridge to get near real-time notifications on cluster/service/task state changes and automate remediation — reduces reliance on manual polling for operational state. [doc](https://docs.aws.amazon.com/AmazonECS/latest/developerguide/fargate-tasks-services.html)
- **[change tracking]** Use AWS Config with Amazon ECS to track how task definitions, services, and cluster configurations change over time — supports compliance auditing and operational troubleshooting. [doc](https://aws.amazon.com/ecs/faqs/)
- **[maintenance prep]** Monitor for Fargate task retirement notifications and prepare services to tolerate individual task replacement gracefully — Fargate periodically retires tasks for underlying maintenance regardless of workload. [doc](https://docs.aws.amazon.com/AmazonECS/latest/developerguide/task-maintenance.html)
