# AWS Step Functions — Best Practices

## Common scenarios
- Orchestrating multi-step business processes across Lambda, ECS, Batch, and other AWS services        → Reliability, Operational Excellence
- High-volume, short-duration event processing (IoT ingestion, streaming transforms, mobile backends)        → Performance Efficiency, Cost Optimization
- Long-running, auditable workflows with human approval or manual review steps        → Reliability, Security
- Large-scale parallel data processing over S3 objects, CSV, or JSON datasets        → Performance Efficiency, Cost Optimization

## 🔒 Security
- **[IAM policies]** Start from AWS managed policies and then define customer managed policies scoped to the specific actions and resources a state machine needs, moving toward least privilege rather than granting broad access. [doc](https://docs.aws.amazon.com/step-functions/latest/dg/security_iam_id-based-policy-examples.html)
- **[IAM conditions]** Add IAM policy conditions (for example requiring SSL, or restricting calls to when they originate from Step Functions) to further scope down what the execution role and callers can do. [doc](https://docs.aws.amazon.com/step-functions/latest/dg/security_iam_id-based-policy-examples.html)
- **[policy validation]** Validate identity-based policies with IAM Access Analyzer to catch overly permissive or non-functional statements before they reach production. [doc](https://docs.aws.amazon.com/step-functions/latest/dg/security_iam_id-based-policy-examples.html)
- **[encryption at rest]** Enable encryption at rest for state machine data using an AWS KMS customer managed key when you need direct control over key rotation, access policy, and auditing. [doc](https://aws.amazon.com/blogs/compute/strengthening-data-security-in-aws-step-functions-with-a-customer-managed-aws-kms-key/)
- **[cross-account KMS access]** When using a customer managed key across accounts, scope the key policy with a `kms:ViaService` condition restricted to the Step Functions service endpoint in the relevant Region, rather than granting broad key usage. [doc](https://aws.amazon.com/blogs/compute/strengthening-data-security-in-aws-step-functions-with-a-customer-managed-aws-kms-key/)
- **[sensitive data]** Avoid passing sensitive data such as tokens or passwords directly in state machine input or output — retrieve secrets at execution time from AWS Systems Manager Parameter Store or AWS Secrets Manager instead. [doc](https://aws.amazon.com/blogs/database/migrating-data-from-oracle-to-amazon-aurora-dsql/)
- **[resource-based access]** Restrict who can start, stop, or describe executions of a state machine using IAM tag-based policies on the execution resources, which inherit the tags of their state machine, rather than relying solely on network controls. [doc](https://docs.aws.amazon.com/step-functions/latest/dg/tag-based-policies.html)
- **[private networking]** Use AWS PrivateLink VPC endpoints to invoke Step Functions from VPC-enabled Lambda functions and other resources without traversing the public internet. [doc](https://docs.aws.amazon.com/step-functions/latest/dg/vpc-endpoints.html)
- **[tag-based authorization]** Tag state machines and activities so execution resources inherit the tags, then use those tags in IAM conditions to allow or deny access to executions at the state machine level. [doc](https://docs.aws.amazon.com/step-functions/latest/dg/sfn-best-practices.html)

## 🛡️ Reliability
- **[timeouts]** Set an explicit `TimeoutSeconds` on every `Task` state, since without one a stuck activity worker or unresponsive integration can leave an execution hanging indefinitely. [doc](https://docs.aws.amazon.com/step-functions/latest/dg/sfn-best-practices.html)
- **[callback heartbeats]** When using `.waitForTaskToken`, set `HeartbeatSeconds` below the task timeout so a missed heartbeat is reported as a heartbeat failure rather than being confused with a long-running but healthy task. [doc](https://docs.aws.amazon.com/step-functions/latest/dg/sfn-best-practices.html)
- **[retry/catch]** Use built-in `Retry` and `Catch` on Task, Parallel, and Map states to handle transient errors and route to designated cleanup or recovery states instead of failing the whole execution. [doc](https://docs.aws.amazon.com/step-functions/latest/dg/concepts-error-handling.html)
- **[transient Lambda errors]** Add retriers for transient Lambda service exceptions so temporary throttling or service errors don't cause an otherwise-healthy execution to fail. [doc](https://docs.aws.amazon.com/step-functions/latest/dg/sfn-best-practices.html)
- **[execution history quota]** Use the Map state in Distributed mode, or split work across nested/child executions via `StartExecution`, for long-running or high-iteration workflows so you don't hit the 25,000-event execution history quota. [doc](https://docs.aws.amazon.com/step-functions/latest/dg/sfn-best-practices.html)
- **[workflow model choice]** Choose Standard workflows for non-idempotent actions that need an exactly-once execution model (for example starting a cluster or processing a payment), and only use Express workflows when the logic is idempotent and safe under at-least-once or at-most-once execution. [doc](https://docs.aws.amazon.com/wellarchitected/latest/serverless-applications-lens/step-functions-workflows.html)
- **[large payloads]** Store large data in Amazon S3 and pass the object ARN between states instead of passing payloads that might exceed 256 KiB, since oversized payloads can terminate an execution. [doc](https://docs.aws.amazon.com/step-functions/latest/dg/sfn-best-practices.html)
- **[polling avoidance]** Prefer Callbacks (`.waitForTaskToken`) or the `.sync` service integration pattern over building a manual polling loop with a Task state, since polling adds unnecessary state transitions and points of failure. [doc](https://docs.aws.amazon.com/wellarchitected/latest/serverless-applications-lens/step-functions-workflows.html)

## ⚡ Performance Efficiency
- **[workflow type selection]** Match the workflow type to the throughput profile: Standard for durable, auditable, longer-running workflows (up to a year, ~2K executions/sec starts), Express for high-volume short-duration workloads (up to five minutes, over 100K executions/sec). [doc](https://docs.aws.amazon.com/step-functions/latest/dg/welcome.html)
- **[state transition throttling]** Estimate expected state transitions and compare against the account's token-bucket quota for Standard workflows, since sustained high-transition workloads may need a quota increase or a switch to Express workflows. [doc](https://docs.aws.amazon.com/wellarchitected/latest/serverless-applications-lens/aws-step-functions.html)
- **[large-scale parallelism]** Use the Map state in Distributed mode instead of Inline mode when you need more than 40 concurrent iterations, since Distributed Map supports up to 10,000 concurrent child workflow executions and can read directly from S3. [doc](https://docs.aws.amazon.com/wellarchitected/latest/serverless-applications-lens/step-functions-workflows.html)
- **[downstream concurrency limits]** Cap Distributed Map `maxConcurrency` to match the burst and sustained concurrency quotas of downstream services (for example Lambda's account concurrency limit) rather than defaulting to maximum parallelism. [doc](https://aws.amazon.com/blogs/aws/step-functions-distributed-map-a-serverless-solution-for-large-scale-parallel-data-processing/)
- **[synchronous Express invocation]** Use synchronous Express Workflow invocation (via API Gateway, Lambda, or `StartSyncExecution`) when a caller needs the result inline and prefers not to build separate retry/parallel-handling logic. [doc](https://docs.aws.amazon.com/wellarchitected/latest/serverless-applications-lens/aws-step-functions.html)
- **[log analysis]** Use CloudWatch Logs Insights to interactively search and analyze workflow execution logs rather than building custom log-parsing tooling. [doc](https://docs.aws.amazon.com/wellarchitected/latest/serverless-applications-lens/aws-step-functions.html)

## 💰 Cost Optimization
- **[Express migration]** Migrate workflows to Express when they complete within five minutes, tolerate at-least-once execution, and don't use `.waitForTaskToken` or `.sync` integrations, since Express is billed per execution/duration/memory instead of per state transition. [doc](https://docs.aws.amazon.com/step-functions/latest/dg/sfn-best-practices.html)
- **[nested Express in Standard]** Combine workflow types by nesting an Express workflow for idempotent, high-volume steps inside a Standard workflow that handles the non-idempotent or long-running parts, balancing cost and durability guarantees. [doc](https://docs.aws.amazon.com/step-functions/latest/dg/sfn-best-practices.html)
- **[avoid polling loops]** Replace polling-loop patterns for asynchronous jobs with Callbacks or `.sync` service integrations, since Standard workflow billing is per state transition and polling multiplies transitions unnecessarily. [doc](https://docs.aws.amazon.com/wellarchitected/latest/serverless-applications-lens/step-functions-workflows.html)
- **[cost allocation tagging]** Tag state machines with cost-center and application identifiers to attribute spend accurately in AWS Cost Allocation Reports. [doc](https://docs.aws.amazon.com/step-functions/latest/dg/sfn-best-practices.html)

## ⚙️ Operational Excellence
- **[observability]** Enable CloudWatch metrics, CloudWatch Logs, and AWS CloudTrail for state machines so execution history, API activity, and performance can all be monitored from one place. [doc](https://docs.aws.amazon.com/step-functions/latest/dg/monitoring-logging.html)
- **[distributed tracing]** Enable AWS X-Ray tracing on state machines to visualize integrations, identify latency bottlenecks, and troubleshoot failed requests across the services a workflow calls. [doc](https://docs.aws.amazon.com/step-functions/latest/dg/concepts-xray-tracing.html)
- **[log level selection]** Choose an appropriate execution history log level for Express workflows, since publishing logs to CloudWatch Logs doesn't block or slow executions but does affect what you can see and its cost. [doc](https://docs.aws.amazon.com/wellarchitected/latest/serverless-applications-lens/aws-step-functions.html)
- **[async failure handling]** For failures that aren't caught inside the state machine (such as top-level execution timeouts), listen for `TIMED_OUT` and other execution-status events on an EventBridge bus and invoke a handler, since Step Functions catchers don't cover top-level execution failures. [doc](https://docs.aws.amazon.com/step-functions/latest/dg/concepts-error-handling.html)
- **[resource organization]** Tag state machines and activities and manage them with AWS Resource Groups to keep large numbers of workflows organized and auditable. [doc](https://docs.aws.amazon.com/step-functions/latest/dg/sfn-best-practices.html)
