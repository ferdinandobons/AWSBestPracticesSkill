- **[IAM]** Grant only the specific EventBridge actions needed (e.g. scope `events:PutEvents` to a named event bus) rather than broad `events:*` permissions, using identity-based policies for callers and resource-based policies for cross-account access. [doc](https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-manage-iam-access.html)
- **[IAM]** Use event bus resource policies to let event-consuming teams manage their own rules across accounts/Regions instead of sharing broad IAM roles, and scope permissions with condition keys such as `events:detail-type`, `events:source`, and `events:creatorAccount`. [doc](https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-rules-best-practices.html)
- **[IAM]** Prevent the confused-deputy problem on target invocation roles by adding `aws:SourceArn` (scoped to the rule ARN) and `aws:SourceAccount` conditions to the role trust policy used by `events.amazonaws.com`. [doc](https://docs.aws.amazon.com/eventbridge/latest/userguide/cross-service-confused-deputy-prevention.html)
- **[Encryption]** Use customer managed KMS keys instead of the AWS owned key for event buses, archives, and connections when you need audit control over the encryption key protecting event data, authorization parameters, and rule/target configuration. [doc](https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-data-protection.html)
- **[Encryption]** Grant EventBridge the minimum required KMS permissions (`kms:DescribeKey`, `kms:GenerateDataKey`, `kms:Decrypt`) in the key policy, scoped with condition keys, when using customer managed keys for connections or archives. [doc](https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-encryption-key-policy.html)
- **[Data protection]** Never place confidential or sensitive information in event bus names, rule names, or tags â€” these fields are not encrypted at rest even when a customer managed key is configured. [doc](https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-data-protection.html)
- **[Event patterns]** Scope resource policies and event patterns with explicit `account` and `region` fields to limit which cross-account or cross-Region events and API calls are authorized, rather than relying on broad wildcard principals. [doc](https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-patterns-best-practices.html)

## đź›ˇď¸Ź Reliability
- **[Error handling]** Configure a dead-letter queue (SQS) on every rule target so events that exhaust retries or fail immediately (e.g. missing permissions, deleted target) are retained for later reprocessing instead of being dropped. [doc](https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-rule-dlq.html)
- **[Retry policy]** Tune per-target `MaximumEventAgeInSeconds` and `MaximumRetryAttempts` to match target recovery characteristics â€” the default of 24 hours and 185 attempts can overload an already-degraded downstream target instead of letting it recover. [doc](https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-rule-retry-policy.html)
- **[Rule design]** Write event patterns as precisely as possible (matching `source`, `detail-type`, and specific `detail` values) to avoid infinite loops where a rule's triggered action re-fires the same rule. [doc](https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-patterns-best-practices.html)
- **[Rule design]** Scope event patterns to account for event source evolution â€” new event types added by a publisher can unexpectedly match an overly broad pattern and cause processing failures or backlogs. [doc](https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-patterns-best-practices.html)
- **[Global endpoints]** Enable event replication for global endpoints and design consumers to be idempotent, since event IDs can change across Regions and replay/failover can cause events to be processed in both the primary and secondary Region. [doc](https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-ge-best-practices.html)
- **[Global endpoints]** Keep `PutEvents` and target invocation quotas consistent across the primary and secondary Region of a global endpoint to prevent throttling after failover. [doc](https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-ge-best-practices.html)
- **[Global endpoints]** Avoid basing Route 53 health checks for a global endpoint on subscriber-side metrics, since a single unhealthy subscriber can trigger an unwanted failover even though the primary Region is otherwise healthy. [doc](https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-ge-best-practices.html)
- **[Monitoring]** Alarm on the `ThrottledRules` and `FailedInvocations`/`InvocationsSentToDLQ` metrics to detect when rules are being throttled by account-level invocation quotas or when target delivery is failing. [doc](https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-rules-best-practices.html)
- **[Recovery]** Use archives with event pattern filters to capture events for a source event bus, and replay them (waiting at least 10 minutes after ingestion to ensure completeness) to recover from downstream bugs or outages. [doc](https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-archive.html)

## âšˇ Performance Efficiency
- **[Rule design]** Set a single target per rule rather than packing multiple targets into one rule â€” this makes it easier to evolve each target's event pattern and configuration independently as needs diverge. [doc](https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-rules-best-practices.html)
- **[Filtering]** Use precise event patterns, including content-based filtering (prefix, suffix, numeric ranges, `$or`), so only relevant events reach targets and unnecessary invocations are avoided. [doc](https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-patterns-best-practices.html)
- **[Input transformation]** Use input transformers to extract and reshape only the needed fields from an event before invoking a target, reducing payload size and downstream processing overhead. [doc](https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-transform-target-input.html)
- **[Validation]** Validate event patterns with the console sandbox or the CLI `test-event-pattern` command before deploying rules, to confirm they match only the intended events. [doc](https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-patterns-best-practices.html)
- **[Monitoring]** Monitor `IngestionToInvocationStartLatency`/`IngestionToInvocationSuccessLatency` for low-latency use cases â€” sustained latency above 30 seconds can indicate a service disruption or rule throttling. [doc](https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-rules-best-practices.html)
- **[Monitoring]** Track `InvocationAttempts`, `SuccessfulInvocationAttempts`, and `RetryInvocationAttempts` per rule to detect when a target is under-scaled and rate-limiting event delivery. [doc](https://aws.amazon.com/blogs/compute/monitoring-best-practices-for-event-delivery-with-amazon-eventbridge/)

## đź’° Cost Optimization
- **[Filtering]** Filter events precisely at the rule level so only events of genuine interest are processed and billed, instead of routing broad event volumes to targets and filtering downstream. [doc](https://aws.amazon.com/eventbridge/features/)
- **[Rule design]** Avoid overly broad event patterns that unintentionally match new event types as publishers add features â€” this drives unwanted invocation volume and billing charges. [doc](https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-patterns-best-practices.html)
- **[Loop prevention]** Guard against infinite-loop rule configurations, which can rapidly generate excess invocations and unexpectedly high charges; use budgets and alerts to catch cost spikes early. [doc](https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-patterns-best-practices.html)
- **[Archives]** Scope archive event patterns and retention periods to only the events you actually need to replay, since archive processing and storage are billed based on volume and duration. [doc](https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-archive.html)

- **[Rule design]** Keep one target per rule so rule and target lifecycle changes can be made independently, simplifying maintenance as consumer requirements diverge over time. [doc](https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-rules-best-practices.html)
