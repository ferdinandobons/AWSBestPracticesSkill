# Amazon MQ — Best Practices

## Common scenarios
- Migrating on-premises ActiveMQ or RabbitMQ brokers to a managed service        → Reliability, Operational Excellence
- Decoupling producers and consumers in enterprise integration patterns (JMS/AMQP/STOMP/MQTT)        → Reliability, Performance Efficiency
- Connecting legacy applications that require standard messaging protocols        → Security, Reliability
- Priority- or task-based message processing pipelines with compute services (Lambda, App Runner, ECS)        → Reliability, Cost Optimization

## 🔒 Security
- **[Network]** Prefer brokers without public accessibility so they can't be reached from outside your VPC, reducing exposure to DDoS and unauthorized access. [doc](https://docs.aws.amazon.com/amazon-mq/latest/developer-guide/using-amazon-mq-securely.html)
- **[ActiveMQ authorization]** Always configure an authorization map restricting permissions by group, since ActiveMQ has no authorization map by default and any authenticated user can otherwise perform any action. [doc](https://docs.aws.amazon.com/amazon-mq/latest/developer-guide/using-amazon-mq-securely.html)
- **[Network]** Block unnecessary protocols and ports with VPC security groups, allowing only the specific protocol ports in use (for example AMQP 5671, MQTT 8883, OpenWire 61617, STOMP 61614, WebSocket 61619). [doc](https://docs.aws.amazon.com/amazon-mq/latest/developer-guide/using-amazon-mq-securely.html)
- **[Identity]** Set up individual IAM users or IAM Identity Center identities with least-privilege permissions instead of sharing broad account credentials. [doc](https://docs.aws.amazon.com/amazon-mq/latest/developer-guide/data-protection.html)
- **[Authentication]** Enforce TLS for all client connections to the broker (TLS 1.2 required, TLS 1.3 recommended) rather than allowing unencrypted transport. [doc](https://docs.aws.amazon.com/amazon-mq/latest/developer-guide/data-protection.html)
- **[Encryption]** Rely on Amazon MQ encryption at rest and in transit, and use AWS KMS customer-managed keys where compliance requires control over key policies. [doc](https://docs.aws.amazon.com/amazon-mq/latest/migration-guide/concept-chapter-about.html)
- **[Data handling]** Avoid putting confidential or sensitive information (such as customer emails or credentials) into broker tags or other free-form text fields, since these may surface in billing or diagnostic logs. [doc](https://docs.aws.amazon.com/amazon-mq/latest/developer-guide/data-protection.html)
- **[Auditability]** Enable AWS CloudTrail logging to capture and continuously monitor Amazon MQ API activity for security and compliance review. [doc](https://docs.aws.amazon.com/amazon-mq/latest/developer-guide/data-protection.html)
- **[RabbitMQ accounts]** Do not rely on or recreate a "guest" account; Amazon MQ deletes the default guest account and periodically removes any customer-created account named "guest". [doc](https://docs.aws.amazon.com/amazon-mq/latest/developer-guide/best-practices-broker-setup.html)
- **[Network isolation]** Deploy production brokers in private subnets within your VPC and restrict access using security groups instead of using the publicly-accessible option. [doc](https://aws.amazon.com/blogs/big-data/migrate-jms-applications-to-amazon-mq-for-rabbitmq-with-minimal-changes/)

## 🛡️ Reliability
- **[ActiveMQ HA]** Use active/standby broker deployments across two Availability Zones so that a broker or AZ failure triggers automatic failover to the standby instance. [doc](https://docs.aws.amazon.com/amazon-mq/latest/developer-guide/amazon-mq-broker-architecture.html)
- **[RabbitMQ HA]** Use cluster deployments (three nodes across three Availability Zones) instead of single-instance brokers for production workloads to remove single points of failure and ensure SLA-backed availability. [doc](https://docs.aws.amazon.com/amazon-mq/latest/developer-guide/best-practices-broker-setup.html)
- **[RabbitMQ queues]** Use quorum queues as the default replicated queue type for production RabbitMQ brokers (3.13+), since they use Raft consensus to elect a new leader automatically and keep delivering messages if a node or AZ is unavailable. [doc](https://docs.aws.amazon.com/amazon-mq/latest/developer-guide/best-practices-broker-setup.html)
- **[Client connectivity]** Use the Failover Transport to let clients connect across multiple broker endpoints, since multi-AZ brokers can fail over during maintenance windows and restarts. [doc](https://docs.aws.amazon.com/amazon-mq/latest/developer-guide/best-practices-activemq.html)
- **[Network of brokers]** Enable persistent mode, keep advisory messages enabled, and avoid multicast broker discovery (unsupported) when configuring a network of ActiveMQ brokers. [doc](https://docs.aws.amazon.com/amazon-mq/latest/developer-guide/best-practices-activemq.html)
- **[XA transactions]** Resolve prepared XA transactions promptly with commit() or rollback(), since unresolved transactions are replayed on every restart and can significantly slow broker startup and failover. [doc](https://docs.aws.amazon.com/amazon-mq/latest/developer-guide/best-practices-activemq.html)
- **[RabbitMQ durability]** Declare durable queues and persistent delivery mode so messages survive broker crashes or restarts. [doc](https://docs.aws.amazon.com/amazon-mq/latest/developer-guide/best-practices-message-reliability.html)
- **[RabbitMQ delivery guarantees]** Configure publisher confirms and consumer delivery acknowledgements so applications know messages were reliably stored and processed instead of silently dropped. [doc](https://docs.aws.amazon.com/amazon-mq/latest/developer-guide/best-practices-message-reliability.html)
- **[RabbitMQ queue health]** Keep queues short and apply TTL/auto-delete policies, since large backlogs slow node synchronization during maintenance and can trigger memory alarms that pause the broker. [doc](https://docs.aws.amazon.com/amazon-mq/latest/developer-guide/best-practices-message-reliability.html)
- **[Client resiliency]** Enable automatic network recovery in client libraries and add retry logic for the initial connection attempt, since automatic recovery does not cover the first failed connection. [doc](https://docs.aws.amazon.com/amazon-mq/latest/developer-guide/best-practices-network-resilience.html)
- **[Infrastructure]** Never modify or delete the Amazon MQ–managed elastic network interface in your VPC, as doing so can permanently break connectivity between your VPC and the broker. [doc](https://docs.aws.amazon.com/amazon-mq/latest/developer-guide/best-practices-activemq.html)

## ⚡ Performance Efficiency
- **[Connection management]** Use connection pooling (for example PooledConnectionFactory) for producers instead of creating a new connection per request; never pool connections for consumers. [doc](https://docs.aws.amazon.com/amazon-mq/latest/developer-guide/best-practices-activemq.html)
- **[Message routing]** Avoid JMS message selectors, which fill the broker's filter buffer, and prefer virtual destinations over durable subscriptions for better scalability. [doc](https://docs.aws.amazon.com/amazon-mq/latest/developer-guide/best-practices-activemq.html)
- **[Storage type]** Choose Amazon EFS storage for high durability and cross-AZ replication, or Amazon EBS for lower latency and higher throughput, based on workload needs. [doc](https://docs.aws.amazon.com/amazon-mq/latest/developer-guide/best-practices-activemq.html)
- **[Slow consumers]** Disable concurrent store and dispatch (set `concurrentStoreAndDispatchQueues` to false) for queues with slow consumers to avoid throughput degradation. [doc](https://docs.aws.amazon.com/amazon-mq/latest/developer-guide/best-practices-activemq.html)
- **[Instance sizing]** Test different broker instance types against your real message size and consumer speed, since larger instances only help specific patterns (non-persistent mode, fast consumers, batched transactions). [doc](https://docs.aws.amazon.com/amazon-mq/latest/developer-guide/best-practices-activemq.html)
- **[RabbitMQ connections]** Use one connection per process and one channel per thread rather than a 1:1 connection-to-channel ratio, to avoid connection churn and channel leaks. [doc](https://docs.aws.amazon.com/amazon-mq/latest/developer-guide/best-practices-broker-setup.html)
- **[RabbitMQ message size]** Keep messages under 1 MB and use the claim-check pattern (store payload in Amazon S3, pass only a reference) for large payloads to avoid memory alarms and slower recovery. [doc](https://docs.aws.amazon.com/amazon-mq/latest/developer-guide/best-practices-performance.html)
- **[RabbitMQ consumption]** Use long-lived consumers with `basic.consume` instead of polling with `basic.get`, and always set an explicit pre-fetch value to prevent brokers or consumers from running out of memory. [doc](https://docs.aws.amazon.com/amazon-mq/latest/developer-guide/best-practices-performance.html)
- **[Celery workloads]** Disable unnecessary control queues (`worker_enable_remote_control`, task events) and use quorum queues with Celery 5.5+ to reduce non-critical broker activity that can trigger memory alarms. [doc](https://docs.aws.amazon.com/amazon-mq/latest/developer-guide/best-practices-performance.html)

## 💰 Cost Optimization
- **[Right-sizing]** Start with the smallest broker instance type for testing (for example `mq.t2.micro`/`m7g.medium`) and size up only after measuring real throughput needs, since larger instance types don't always improve throughput. [doc](https://docs.aws.amazon.com/amazon-mq/latest/developer-guide/best-practices-activemq.html)
- **[Architecture choice]** Reserve RabbitMQ cluster deployments and ActiveMQ active/standby for production, and use single-instance brokers only for development or testing to avoid paying for unneeded redundancy. [doc](https://aws.amazon.com/blogs/compute/migrating-message-driven-applications-to-amazon-mq-for-rabbitmq/)
- **[Resource efficiency]** Keep RabbitMQ queues short and apply TTL/auto-delete policies so resource overutilization doesn't force larger instance types or additional storage. [doc](https://docs.aws.amazon.com/amazon-mq/latest/developer-guide/best-practices-message-reliability.html)

## ⚙️ Operational Excellence
- **[Monitoring]** Regularly monitor CloudWatch metrics and set alarms for broker resource limits (for example `RabbitMQMemUsed`/`RabbitMQMemLimit`, `RabbitMQDiskFree`/`RabbitMQDiskFreeLimit`, or `JournalFilesForFastRecovery` for ActiveMQ) to catch degradation before it affects applications. [doc](https://docs.aws.amazon.com/amazon-mq/latest/developer-guide/best-practices-network-resilience.html)
- **[Logging]** Enable general and audit logging to CloudWatch Logs so broker activity and management actions are captured for troubleshooting and compliance. [doc](https://docs.aws.amazon.com/amazon-mq/latest/developer-guide/configure-logging-monitoring-activemq.html)
- **[Resource naming]** Use a standard, restricted character set for RabbitMQ vhost, queue, exchange, and policy names even though RabbitMQ permits arbitrary UTF-8, to avoid tooling and operational issues. [doc](https://docs.aws.amazon.com/amazon-mq/latest/developer-guide/best-practices-resource-naming.html)
- **[Permissions]** Grant the `logs:CreateLogGroup` permission to the principal creating or rebooting brokers, and configure the required resource-based CloudWatch Logs policy, or broker logs will silently fail to appear. [doc](https://docs.aws.amazon.com/amazon-mq/latest/developer-guide/troubleshooting-activemq.html)
